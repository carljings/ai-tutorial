AI 产品设计
从"加上聊天框"到"为 AI 而生"的思维跃迁

AI 产品的每次迭代都该用 A/B 测试验证，而不是靠'感觉变好了'。但 AI 指标往往方差大、样本量不足，容易得出假阳性结论。下面这段代码做正确的比例检验并算出所需样本量，避免被噪声误导。

# A/B 测试: 两版本成功率差异的显著性检验 + 样本量估算
import math
def two_proportion_z(c1, n1, c2, n2):
    """c1,c2: 两版本成功数; n1,n2: 样本量。返回 z 值。"""
    p1, p2 = c1/n1, c2/n2
    p  = (c1 + c2) / (n1 + n2)            # 合并比例
    se = math.sqrt(p*(1-p)*(1/n1 + 1/n2))
    if se == 0: return 0
    return (p1 - p2) / se
def required_n(p_base, mde, alpha=0.05, power=0.8):
    """算检测出最小可观测效应(mde)所需每组样本量。"""
    z_a = 1.96; z_b = 0.84               # alpha=5%, power=80%
    p = p_base + mde/2
    n = ((z_a*math.sqrt(2*p_base*(1-p_base)) +
          z_b*math.sqrt(p*(1-p)+(p_base+mde)*(1-p_base-mde))) / mde)**2
    return math.ceil(n)
# 案例: 任务成功率从 40% 提升到 43%, 需要多少样本?
n = required_n(0.40, 0.03)
print(f"检测 40%->43% 需要每组约 {n} 样本")
z = two_proportion_z(430, 1000, 400, 1000)
print(f"实测 z={z:.2f} (>1.96 显著)" if z>1.96 else f"实测 z={z:.2f} (不显著)")

★ AI 产品的度量指标体系实战

前面讲了 AI Native 和 AI Featured 的分野、六条设计原则，但产品上线后最现实的问题是：怎么度量它"好不好"？传统互联网产品有一套成熟的指标（DAU、留存、转化），但 AI 产品多了一个维度——模型质量本身。一个 AI 产品可能留存很高但幻觉率也高（用户觉得新鲜但被骗），也可能延迟极低但任务完成率低（快但没用）。所以 AI 产品的度量必须是三层结构：北极星指标（用户价值）、护栏指标（模型质量约束）、反指标（警惕过度优化）。下面这套 dataclass 把这三层显式编码，让产品的健康度可计算、可对比。

# -*- coding: utf-8 -*-
"""AI 产品的北极星与护栏指标体系：一个可跑的度量脚手架。
把"用户价值"和"模型质量"拆成可观测的指标族。
"""
from dataclasses import dataclass, field
from typing import List

@dataclass
class AIMetrics:
    # ---- 北极星(用户价值) ----
    task_success_rate: float = 0.0   # 用户任务完成率(核心)
    time_to_value: float = 0.0       # 首次获得有效帮助的耗时(秒)
    retention_d7: float = 0.0        # 7日留存
    # ---- 模型质量(护栏) ----
    hallucination_rate: float = 0.0  # 幻觉/编造比例
    latency_p50: float = 0.0         # 中位延迟
    latency_p99: float = 0.0         # 尾部延迟
    cost_per_session: float = 0.0    # 每会话成本(美元)
    # ---- 反指标(警惕过度优化) ----
    override_rate: float = 0.0       # 用户手动改写AI输出的比例

    def health(self):
        """综合健康分: 北极星为主, 护栏为约束, 反指标为告警。"""
        star = (self.task_success_rate * 0.6 +
                min(self.retention_d7, 0.5) / 0.5 * 0.4)
        guard_penalty = (1.0 if self.hallucination_rate < 0.05 else 0.7)
        return round(star * guard_penalty, 3)

这套指标体系的关键设计在于反指标这一层——它专门盯那些"看起来在变好、其实在变差"的信号。比如 override_rate（用户手动改写 AI 输出的比例）就是个极有价值的反指标：如果模型升级后成功率提升了，但 override_rate 也飙升，说明模型在"蒙对"——用户其实不信任它的输出，事后还要大改。这种情况下单纯的成功率会误导你。把反指标纳入健康分计算，能防止团队在错误的指标上狂奔。下面这段代码模拟两个版本的指标对比，演示如何用健康分做版本决策：

# 埋点采集 -> 指标计算的最小流水线
import numpy as np

def simulate_sessions(n=1000, seed=0):
    rng = np.random.default_rng(seed)
    return AIMetrics(
        task_success_rate=rng.beta(8, 2),         # 多数会话成功
        time_to_value=rng.gamma(2, 3),
        retention_d7=rng.beta(3, 5),
        hallucination_rate=rng.beta(1, 30),
        latency_p50=rng.gamma(3, 0.3),
        latency_p99=rng.gamma(5, 0.6),
        cost_per_session=rng.gamma(2, 0.01),
        override_rate=rng.beta(2, 8),
    )

# 对比两个版本(A: 精度优先, B: 速度优先)
import copy
A = simulate_sessions(0); A.hallucination_rate = 0.02; A.latency_p99 = 4.0
B = copy.copy(A);        B.hallucination_rate = 0.08; B.latency_p99 = 1.2
for name, m in [("v1.0(精度优先)", A), ("v1.1(速度优先)", B)]:
    print(f"{name:18s} 健康分={m.health()}  "
          f"成功率={m.task_success_rate:.2f} "
          f"幻觉={m.hallucination_rate:.2f} "
          f"P99={m.latency_p99:.1f}s")

把这套度量体系用到真实产品里，有三条落地经验。① 北极星只能有一个：不要同时优化三个"都很重要"的指标，那等于没有北极星；task_success_rate 通常是最稳的选择。② 护栏指标设硬阈值：幻觉率超过 5% 就回滚、P99 超过 5 秒就告警，这些是红线，不能用"均值好看"来掩盖。③ 反指标每周 review：把它单独拎出来看趋势，override_rate、负面反馈率一旦上升，就要追问"我们是不是在过度优化某个指标"。这套体系的本质是把产品决策从"老板拍脑袋"变成"数据说话"，而 AI 产品因为模型行为的不可预测性，比传统产品更需要这种纪律。

★ AI Native vs AI Featured：一道决定生死的分水岭

市场上 90% 的"AI 产品"其实是 AI Featured——AI 只是一个附加功能，去掉 AI 产品依然能运转。比如一个加了智能客服的电商平台，把 AI 摘掉只是回到手动客服。这类产品的 AI 是锦上添花，而非核心引擎。

真正的 AI Native 产品则截然不同：没有 AI，产品根本无法存在。Notion AI 去掉 AI 仍然是一个笔记软件，所以它是 AI Featured；但 Cursor 去掉 AI 就退化成一个普通编辑器，几乎失去存在意义，所以它是 AI Native。Midjourney 去掉 AI 就什么都不是。判断标准很简单：如果你把 AI 模块完全移除，产品还有独立价值吗？

这个区分为什么重要？因为 AI Native 产品享有更高的壁垒、更深的护城河、更强的用户黏性。AI Featured 产品容易被"原生 AI 玩家"降维打击——当 Notion 内嵌的笔记遇到 Cursor，当传统客服遇到 Intercom Fin，胜负在基因层面就已注定。设计 AI 产品的第一步，就是诚实地问自己：我做的是 AI Native 还是 AI Featured？如果是后者，我有没有可能重新设计成前者？

★ 六大产品设计原则

设计 AI Native 产品没有标准答案，但以下六条原则是从大量成功与失败案例中提炼出来的。它们不是教条，而是帮你避免重蹈覆辙的检查清单。

原则一：默认 AI 在场，而非藏在按钮后。用户不会主动点击"召唤 AI"。Cursor 默认在 Tab 键里植入 AI 补全，Perplexity 默认用 AI 组织搜索结果。AI 应该是产品的"默认态"，用户需要主动选择"不用 AI"，而不是反过来。如果你的 AI 入口是一个二级菜单的按钮，基本意味着它不会被用到。

原则二：拥抱不确定性，设计优雅的降级。AI 会犯错——这是不可避免的。好产品设计不是假装 AI 不会错，而是让错误变得可被发现、可被纠正。GitHub Copilot 的建议是灰色的（视觉上区分），用户 Tab 接受、Esc 拒绝；Cursor 的 Agent 修改前会展示 diff，用户逐行审批。设计降级路径：当 AI 不确定时，宁可问用户，也不要自信地给出错误答案。

原则三：让 AI 的思考过程可见。用户不信任黑箱。ChatGPT 的"正在思考..."动画、Cursor Agent 的"正在读取文件 → 正在修改 → 正在测试"步骤展示、Perplexity 的引用来源——这些"透明度设计"极大提升了信任。不要让 AI 神秘地给出结果，要让用户看到它"在想什么"。

原则四：数据飞轮从第一天开始转。AI 产品的核心竞争力是数据闭环——用户使用产生数据，数据改善模型，模型提升体验，体验吸引更多用户。设计时就要考虑：哪些用户行为是有价值的反馈信号？如何设计 UI 让反馈收集自然发生？（比如 Copilot 的"接受/拒绝"就是天然的偏好数据。）

# 原则四的数据飞轮示意：收集隐式反馈改善模型
def feedback_pipeline(user_action: str, suggestion: str, context: dict):
    """每一次用户的接受/拒绝/修改，都是训练信号"""
    signal = {
        "suggestion": suggestion,
        "context": context,
        # 三类反馈标签
        "accepted": user_action == "tab",          # 完全接受
        "rejected": user_action == "esc",          # 完全拒绝
        "modified": user_action == "edit",         # 接受但修改了
    }
    # 隐式偏好数据飞轮：越多人用，模型越懂你的代码风格
    send_to_finetune_queue(signal)
    return signal

# Perplexity 的"点赞/踩"、Cursor 的"接受率"都是同样原理

原则五：人机协作的节奏感。AI 太快会让人焦虑（结果瞬间出现但无法验证），太慢会让人不耐烦。好的设计是"渐进式呈现"——先给一个粗略结果让用户有掌控感，再逐步细化。这就像导航软件先显示总路线，再在每个路口提示转弯。

原则六：AI 不是万能的，明确边界。告诉用户 AI 擅长什么、不擅长什么。Perplexity 在健康、金融类查询会加免责声明，Cursor 建议用户审查安全相关代码。过度承诺 AI 能力，只会透支信任。

★ 案例拆解一：Perplexity 如何重新定义搜索

Perplexity 是 AI Native 搜索的标杆。它没有像 Google 那样返回"10 个蓝色链接"让你自己挑，而是直接用 AI 综合多个来源，生成一个带引用的答案。表面上看这只是"搜索 + 摘要"，但底层是三重创新：

第一，查询理解层。Perplexity 会先用 LLM 重写你的查询，判断是否需要联网、需要几轮搜索。复杂问题（"对比 A100 和 H100 的推理性能"）会被拆成多个子查询并行搜索。第二，检索-生成一体化。它不是先搜索再让 LLM 总结，而是在生成过程中动态决定是否需要补充搜索（类似 Agent 的工具调用）。第三，引用透明度。每个关键句后面都有 [1][2] 角标，点击跳转到原始来源。这个设计让用户既能快速获得答案，又能随时溯源验证。

Perplexity 的启示是：AI Native 不是"AI + 旧产品"，而是用 AI 重构整个用户旅程。Google 的搜索框 + 链接列表是 Web 时代的交互范式，Perplexity 的"问题→答案→引用"是 AI 时代的新范式。这个转变足以支撑一家百亿美元的公司。

★ 案例拆解二：Cursor 如何靠 AI 重做编辑器

Cursor（Anysphere 出品）是 2024 年增长最快的 AI 产品之一。它表面上是一个 VS Code 的 fork，但内核完全围绕 AI 重构。三个设计决策让它脱颖而出：

第一，代码库级别的上下文索引。Cursor 在后台对整个仓库构建语义索引，当你问"用户认证的逻辑在哪"，它能精确定位跨文件的代码块，而不是只看当前文件。第二，Tab 模型。Cursor 训练了专门的"光标位置预测"模型，它不仅预测下一个 token，还能预测你接下来要做的多行编辑（如重命名一个变量并更新所有引用）。这种"预测意图"比简单的补全高出几个层级。第三，Agent 模式的 diff 审查。AI 修改文件时，不是直接覆盖，而是展示一个可编辑的 diff 视图，你可以逐行接受、拒绝或手动调整。

Cursor 的成功证明了一个道理：AI Native 产品不是把 AI 加到现有产品里，而是为 AI 重新设计每一个交互细节。VS Code 加一个 Copilot 插件是 AI Featured，Cursor 从底层重写编辑器才是 AI Native。

★ 案例拆解三：Character.AI 与"情感 AI"的陷阱

Character.AI 曾是增长最快的 AI 应用之一——用户可以创建和扮演各种角色（动漫人物、历史名人、虚拟伴侣），与 AI 进行沉浸式对话。它的高留存率一度令人惊叹。但这也是一个关于"产品-伦理边界"的警示案例。

Character.AI 的设计亮点在于人格定制：用户可以设定角色的性格、语气、背景故事，模型会严格保持角色一致性。这种"情感连接"带来了极高的用户黏性。但问题也随之而来——部分青少年用户对虚拟角色产生了过度依赖，甚至出现了悲剧事件。这促使 Character.AI 在 2024 年大幅加强了未成年人保护措施（独立的青少年模型、安全过滤、时间限制）。

这个案例的教训是：AI 产品设计不能只追求"有用"和"好用"，必须前置考虑伦理边界。特别是涉及情感、社交、心理健康的场景，过度优化"用户黏性"可能走向有害的方向。负责任的 AI 产品设计需要在 KPI（留存率、使用时长）之外，加入"用户福祉"作为第一优先级指标。

★ 本章小结

AI 产品设计的核心分野是 AI Native 与 AI Featured——前者把 AI 作为产品存在的唯一理由，后者把 AI 作为附加功能。六条原则可以概括为：默认在场、优雅降级、思考可见、数据飞轮、节奏掌控、明确边界。Perplexity 重新定义了搜索（答案优先于链接），Cursor 重构了编辑器（AI 原生交互），Character.AI 警示了情感 AI 的伦理红线。设计 AI 产品不是"给旧产品加 AI"，而是"为 AI 重新想象产品"。最高境界是：用户感觉不到自己在"用 AI"，因为 AI 已经融化在每一个交互细节里。
