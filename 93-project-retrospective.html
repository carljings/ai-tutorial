★ 项目背景

某中型科技公司（500 人），内部知识散落在 Confluence、飞书文档、Slack 讨论里。新员工入职要花 2 周才能找到基本信息。CTO 想：「能不能做个 AI 问答系统，员工直接问，系统基于公司文档回答？」

这就是我们的项目——「AskCorp」企业智能问答系统（与第 13 章的项目呼应，但这里做完整复盘）。

POC 验证（第 2 周）

用最简方案快速验证「AI 能不能回答公司问题」：

poc.py —— 最小验证（30 分钟搭完）
"""
poc.py —— 30 分钟验证：GLM 能否回答公司文档问题？
"""
from zhipuai import ZhipuAI
client = ZhipuAI()

# 手动挑 10 个真实问题 + 对应文档片段
test_cases = [
    ("年假怎么算？", "工龄1-3年年假7天..."),
    ("报销流程？", "填写报销单-直属领导审批-财务..."),
    # ... 10 个
]

correct = 0
for q, doc in test_cases:
    resp = client.chat.completions.create(model="glm-5.2", messages=[
        {"role": "system", "content": "根据资料回答。"},
        {"role": "user", "content": f"资料：{doc}\n问题：{q}"}
    ])
    answer = resp.choices[0].message.content
    print(f"Q: {q}")
    print(f"A: {answer[:100]}")
    # 人工判断是否正确

print(f"准确率: {correct}/10")
# 结果：9/10 正确 → 可行！进入正式开发
💡 POC 的价值

30 分钟验证了「GLM + 文档片段 = 能回答公司问题」。如果 POC 失败（准确率 <50%），就别花 2 个月开发了。先验证再投入是 AI 项目的第一原则。

03 第三阶段：开发（第 4-6 周）

文档解析与索引（第 4 周）

ingest.py —— 多源文档解析+索引（简化版）
"""
ingest.py —— 从 Confluence/飞书/Slack 解析文档，分块，索引
"""
from langchain_community.document_loaders import ConfluenceLoader, UnstructuredMarkdownLoader
from langchain_text_splitters import RecursiveCharacterTextSplitter
import chromadb
from sentence_transformers import SentenceTransformer

embedder = SentenceTransformer("BAAI/bge-m3")
splitter = RecursiveCharacterTextSplitter(chunk_size=400, chunk_overlap=60)
client = chromadb.PersistentClient(path="./vectordb")
collection = client.get_or_create_collection("docs")

# 1. 从各源加载
def load_all_sources():
    docs = []
    # Confluence
    try:
        conf_docs = ConfluenceLoader(url="https://corp.atlassian.net",
            token="xxx").load()
        docs.extend([(d.page_content, {"source": "confluence", "title": d.metadata.get("title", "")}) for d in conf_docs])
    except Exception as e:
        print(f"Confluence 加载失败: {e}")
    # 飞书导出的 Markdown
    import glob
    for f in glob.glob("./data/feishu/*.md"):
        text = open(f).read()
        docs.append((text, {"source": "feishu", "title": f}))
    return docs

# 2. 分块 + 索引
def index_all():
    docs = load_all_sources()
    total_chunks = 0
    for text, meta in docs:
        chunks = splitter.split_text(text)
        if chunks:
            embs = embedder.encode(chunks).tolist()
            ids = [f"{meta['title']}_{i}" for i in range(len(chunks))]
            metas = [{**meta, "chunk_idx": i} for i in range(len(chunks))]
            collection.add(ids=ids, documents=chunks, embeddings=embs, metadatas=metas)
            total_chunks += len(chunks)
    print(f"✅ 索引完成：{total_chunks} 个块")

index_all()

踩坑记录：文档解析阶段

坑现象解法
Confluence 表格丢失表格内容被解析成乱码用 HTML 模式而非 Wiki 模式导出
飞书文档格式导出的 Markdown 格式不统一写正则预处理，统一标题层级
分块切断了表格一个表格被切成两半分块前用特殊标记保护表格边界
Slack 噪音太多大量闲聊被索引，降低检索质量只索引 pinned 消息 + 有 thread 的消息
重复文档同一文档多版本都被索引用文档 hash 去重

检索 + 生成 Pipeline（第 5 周）

qa_pipeline.py —— 核心问答逻辑
"""
qa_pipeline.py —— 查询改写 → 检索 → Rerank → 生成 → 带引用
"""
from zhipuai import ZhipuAI
from sentence_transformers import CrossEncoder
import chromadb

client = ZhipuAI()
reranker = CrossEncoder("BAAI/bge-reranker-v2-m3")
collection = chromadb.PersistentClient(path="./vectordb").get_collection("docs")

PROMPT = """你是公司内部知识助手。严格根据资料回答。
规则：
1. 只用资料信息，没有就说"根据现有资料无法回答"
2. 在关键信息处标注 [来源编号]
3. 简洁、准确

资料：
{context}

问题：{question}"""

def answer(question, chat_history=[]):
    # 1. 查询改写（消解指代）
    if chat_history:
        question = rewrite_query(question, chat_history)

    # 2. 向量检索
    q_emb = collection.query(query_embeddings=embedder.encode([question]).tolist(), n_results=20)
    candidates = q_emb["documents"][0]
    metas = q_emb["metadatas"][0]

    # 3. Rerank 精排
    scores = reranker.predict([(question, c) for c in candidates])
    ranked = sorted(zip(scores, candidates, metas), key=lambda x: -x[0])[:5]

    # 4. 组装 prompt
    context_parts = []
    for i, (s, c, m) in enumerate(ranked, 1):
        context_parts.append(f"[{i}] 来源：{m.get('title','')}（{m.get('source','')}）\n{c}")
    context = "\n\n".join(context_parts)

    # 5. 生成
    resp = client.chat.completions.create(model="glm-5.2", messages=[
        {"role": "user", "content": PROMPT.format(context=context, question=question)}
    ], stream=True)

    refs = [{"source": m.get("title", ""), "type": m.get("source", "")} for _, _, m in ranked]
    return resp, refs  # 流式返回

踩坑记录：检索阶段

坑现象解法
指代不清用户说"它的流程是什么"，"它"指代不明加查询改写，用对话历史消解
检索结果跑偏向量检索返回了语义相近但不相关的加 Rerank（Cross-Encoder 精排）
Lost in Middle中间位置的信息被忽略Rerank 后最重要的放前面
幻觉模型编造文档里没有的内容系统提示强制"只用资料"+引用编号
部门权限市场部员工查到了财务部机密检索时按部门 metadata 过滤

关键经验总结

✅ 做对了的
  • 先 POC 再开发——30 分钟省了 2 个月
  • 建评估集——否则优化全凭感觉
  • Rerank 是关键——投入产出比最高
  • 流式输出——用户体验立竿见影
  • 灰度发布——提前发现大量问题
  • 持续优化——bad case 驱动迭代
⚠️ 踩过的坑
  • 文档解析——格式不统一花了一周
  • 权限隔离——差点泄露跨部门信息
  • Slack 噪音——闲聊被索引，降低质量
  • 评估集太晚建——应该第一周就建
  • 成本预估偏低——实际超了 30%
  • 用户培训不足——很多人不会用

如果重来会怎么做

第 1 天就建评估集

第一天就用 Rerank

不要先试朴素检索再「优化」加 Rerank。一开始就加。

文档解析花更多时间

数据质量决定一切。清洗花一周值得。

第一天就做权限设计

不是上线前才加。从架构层面就考虑。

更早做用户培训

用户不会自然就会用。做引导+示例。
