RAG 架构设计
Naive → Advanced → Modular 演进

RAG（Retrieval-Augmented Generation）从 2023 年爆火到现在，已经从"一个 embedding + 一个 LLM"的玩具，演化成有检索前、检索中、检索后、生成后多个可插拔模块的复杂系统。理解 Naive RAG → Advanced RAG → Modular RAG 这条演进路线，能帮你判断自己的系统处于哪个阶段、下一步该往哪里优化。

Naive RAG 就是 RAG 教程里第一个 demo：用户问 query → embedding → 向量库召回 top-k → 拼到 prompt 里 → LLM 生成答案。它结构清晰，50 行代码能跑通，是 RAG 的"hello world"。

Naive RAG 流程
用户 query ──▶ embed ──▶ 向量库 top-k ──▶ prompt 拼接 ──▶ LLM ──▶ 答案

它的问题在真实场景里迅速暴露：

  • 召回质量差：用户 query 经常口语化、有错别字、指代不清（"那个怎么用"），直接 embed 召回质量低。
  • 只走语义：精确关键词（产品名、错误码）召回不到。
  • 无重排：召回的 top-k 顺序粗糙，最相关的可能排在第 5 位。
  • 上下文利用率低：把 top-k 一股脑塞进 prompt，LLM 容易"Lost in the Middle"，中间的文档被忽略。
  • 无答案质量保障：LLM 可能幻觉、可能不基于上下文乱编，没有校验。

Naive RAG 适合 demo 和小规模玩具。一旦上生产、用户多了、问题复杂了，它就扛不住。这就是 Advanced RAG 出场的原因。

★ Advanced RAG：检索前/中/后全优化

Advanced RAG 的核心思想是"把检索流程拆成三段，每段都加优化模块"。检索前（pre-retrieval）优化查询，检索中（retrieval）提升召回，检索后（post-retrieval）精炼结果。

Advanced RAG 流程
                                                              ┌─ rerank 重排
用户 query ─┐                                ┌─ 混合检索 BM25+向量 ─┤
            ├─ 查询改写 ─┐                  │                     └─ 压缩/过滤
多轮上下文 ─┘   HyDE 等  ├─ 多路召回 ──────┤
                          └─ 元数据过滤 ───┘
                                                              └─ 上下文打包
                                                                       │
                                                                  LLM 生成

检索前（Pre-retrieval）

  • 查询改写（见 138 篇）：HyDE 生成假设答案再 embed、Query Decomposition 把复杂问题拆成子问题、Step-back 把具体问题抽象成原理问题。
  • 多轮上下文压缩：把多轮对话历史压缩成独立 query（"那个怎么用" → "Acme Corp API 怎么调用"）。
  • 查询路由：根据 query 类型路由到不同索引（FAQ 库 vs 文档库 vs SQL）。

检索中（Retrieval）

  • 混合检索（见 132 篇）：BM25 + 向量 + RRF 融合。
  • 元数据过滤（见 131 篇）：用属性过滤缩小搜索范围。
  • parent-child 分块（见 136 篇）：小块检索大块返回。

检索后（Post-retrieval）

  • 重排序（见 133 篇）：cross-encoder 精排 top-k。
  • 上下文压缩：用小模型或 LLM 把冗长文档压缩成关键句。
  • 上下文打包（见 139 篇）：重排顺序解决 Lost in the Middle。
  • 去重：相似文档去重，避免重复信息。
# Advanced RAG 的骨架代码（伪代码 + 关键调用）
def advanced_rag(query, chat_history):
    # === 检索前 ===
    standalone = rewrite_query(query, chat_history)   # 多轮压缩成独立 query
    sub_queries = decompose(standalone)               # 拆子问题（可选）
    if not sub_queries: sub_queries = [standalone]

    all_candidates = []
    for sq in sub_queries:
        hyde_answer = hyde(sq)                        # 生成假设答案用于 embed
        vec = embed(hyde_answer)

        # === 检索中：混合检索 + 过滤 ===
        bm25_hits = bm25_search(sq, top_k=50)
        vec_hits  = vector_store.search(vec, top_k=50, filter={"year": 2024})
        fused = rrf_fusion([bm25_hits, vec_hits], top_n=50)   # RRF 融合

        # === 检索后：重排 + 压缩 ===
        ranked = reranker.rerank(sq, fused, top_k=10)
        compressed = [compress_context(sq, c) for c in ranked]
        all_candidates.extend(compressed)

    # 去重 + 最终打包
    final = dedup_and_pack(all_candidates, max_tokens=3000)

    # === 生成 ===
    answer = llm.generate(prompt=RAG_PROMPT.format(context=final, question=query))
    return answer

Advanced RAG 把每个环节都优化，效果比 Naive RAG 提升显著（典型 RAGAS 综合分从 0.5 涨到 0.75）。但它仍然是"一条固定的流水线"，所有 query 走同样的路径。下一步是让架构模块化、可路由。

★ Modular RAG：可插拔、可路由、可迭代

Modular RAG 把 Advanced RAG 的每个优化都抽象成独立模块，然后用一个编排层（orchestrator）根据 query 类型动态组合模块。不同 query 走不同路径——简单事实查询走轻量路径（向量+rerank），复杂推理查询走重量路径（多跳检索+自反思），闲聊直接走 LLM 不检索。

核心模块

  • Router：判断 query 类型，决定走哪条路径（FAQ / 文档 / SQL / 计算器 / 不检索）。
  • Retriever：可插拔的检索器（向量/BM25/SPLADE/Graph），按需调用。
  • Reranker：可切换的 rerank 模型。
  • Memory：对话历史 + 长期记忆（用户偏好、过往查询）。
  • Generator：可切换的 LLM（小问题用 Haiku，难问题用 Sonnet/Opus）。
  • Critic / Self-RAG：生成后自评，质量不够就重新检索或重写答案。
  • Tool use：可调用外部工具（SQL、计算器、API）补足检索盲区。
Modular RAG：模块化 + 路由 + 自反思
                      ┌─ Router 判断 query 类型
                      │
   query ──▶ Router ──┼─▶ 简单事实 ──▶ 向量+rerank ─────────────┐
                      │                                          │
                      ├─▶ 复杂推理 ──▶ 多跳检索 ──▶ 自反思循环 ─┤
                      │                                          ├─▶ Generator ──▶ Critic
                      ├─▶ 结构化数据 ──▶ Text2SQL ──▶ DB ────────┤     (质量不够 → 回流)
                      │                                          │
                      └─▶ 闲聊 ──▶ 直接 LLM ─────────────────────┘
# Modular RAG 的编排层（LangGraph 风格）
from langgraph.graph import StateGraph, END

class RAGState(dict):
    query: str
    query_type: str
    context: list
    answer: str
    quality_ok: bool

def route(state):
    if state["query_type"] == "factual":
        return "factual_path"
    elif state["query_type"] == "reasoning":
        return "reasoning_path"
    elif state["query_type"] == "structured":
        return "sql_path"
    return "chitchat"

def factual_retrieve(state):
    vec = embed(state["query"])
    hits = vector_store.search(vec, top_k=20)
    state["context"] = reranker.rerank(state["query"], hits, top_k=5)
    return state

def reasoning_retrieve(state):
    # 多跳：第一跳检索 + LLM 生成追问 + 第二跳检索
    state["context"] = multi_hop_retrieve(state["query"], hops=3)
    return state

def critic(state):
    # 自评：答案是否基于 context、是否完整
    state["quality_ok"] = ragas_quick_check(state["answer"], state["context"])
    return state

def should_retry(state):
    return "regenerate" if not state["quality_ok"] else END

# 用 LangGraph 编排
g = StateGraph(RAGState)
g.add_node("classify", classify_query)
g.add_node("factual_path", factual_retrieve)
g.add_node("reasoning_path", reasoning_retrieve)
g.add_node("sql_path", text2sql_path)
g.add_node("generate", generate_answer)
g.add_node("critic", critic)
g.add_node("regenerate", regenerate)
g.add_conditional_edges("classify", route)
g.add_edge("factual_path", "generate")
g.add_edge("reasoning_path", "generate")
g.add_edge("generate", "critic")
g.add_conditional_edges("critic", should_retry)
app = g.compile()

★ 我的系统处于哪个阶段

  • 还在 Naive：单一向量检索 + 直接 LLM 生成；RAGAS 综合分通常 <0.6。优化点：先加 rerank（最便宜的提升）、再上混合检索。
  • 进入 Advanced：有查询改写、混合检索、rerank、上下文压缩中的至少两三个；RAGAS 综合分 0.65-0.8。优化点：按失败 case 分析缺哪个模块，逐个补。
  • 走向 Modular：有 Router 按类型分流、有 Self-RAG 自反思、有 Tool use 补足盲区；RAGAS >0.8。优化点：模块的精细调优、A/B 测试、成本控制。

不要一上来就堆 Modular——多数业务系统在 Advanced 阶段就能达到不错的效果，Modular 的复杂度只有在 query 类型高度多样、精度要求极高的场景才划算。先 Naive 跑通，按失败 case 增量演进，是更务实的路径。

★ RAG 系统的典型失败模式与诊断

了解了 RAG 架构演进，看真实系统最常见的失败模式，以及怎么诊断它们。这能帮你在自己的系统出问题时快速定位。

失败模式一：检索没召回正确文档。症状是 LLM 答非所问或说"不知道"。诊断：把检索回的 top-k 打印出来，看正确文档在不在里面。不在 → 检索问题（embedding 差、分块烂、过滤错了）；在但排序靠后 → 重排问题。这是最常见的失败，占 60%。

失败模式二：召回了但 LLM 没用。症状是正确文档在 top-k 里，但 LLM 答案没用到它。诊断：看 LLM 的 prompt 里文档排在哪个位置——如果在中间，可能是 Lost in the Middle；如果在末尾但没用，可能是 prompt 没要求基于文档答。这是第二大失败，占 20%。

失败模式三：LLM 幻觉。症状是 LLM 编造了文档里没有的信息。诊断：用 RAGAS 的 faithfulness 指标量化，低分说明幻觉严重。对策是改 prompt（加引用要求、强调"不知道就说不知道"）、换更强模型、或加 Self-RAG 自反思。

失败模式四：查询理解错。症状是用户问 A，系统理解成 B。诊断：把查询改写后的结果打印出来看。多轮对话场景特别容易出现指代消解失败（"那个"指代错）。

失败模式五：延迟过高。症状是用户等不及。诊断：拆各阶段耗时——embedding、检索、rerank、LLM 生成。RAG 的延迟主要在 rerank 和 LLM 生成，优化重点在这里。

# RAG 失败诊断工具：把每一步的中间结果记录下来
def rag_with_diagnosis(query, history):
    log = {}
    log["original_query"] = query
    # 1. 查询改写
    rewritten = rewrite(query, history)
    log["rewritten"] = rewritten
    # 2. 检索
    candidates = retrieve(rewritten, k=50)
    log["retrieval_candidates"] = [c.id for c in candidates[:10]]
    log["retrieval_scores"] = [c.score for c in candidates[:10]]
    # 3. 重排
    ranked = rerank(rewritten, candidates, k=5)
    log["rerank_order"] = [c.id for c in ranked]
    # 4. 生成
    answer = generate(query, ranked)
    log["answer"] = answer
    # 5. 自动评分
    log["faithfulness"] = quick_faithfulness(answer, ranked)
    return answer, log
# 失败时分析 log，定位是哪一步出问题

这个诊断思维——"先定位是哪一步失败，再针对性优化"——比盲目加模块有效得多。很多团队 RAG 效果差，不是缺模块，是没诊断清楚问题在哪。

★ RAG 系统的成本与延迟优化

Advanced/Modular RAG 模块多，成本和延迟容易失控。一个复杂查询可能调 3-5 次 LLM（改写、HyDE、生成、自反思）、跑 2-3 路检索、一次 rerank，端到端延迟几秒、成本几美分。生产化必须优化。

延迟优化的几个杠杆：

  • 并行化：多路检索并行、查询改写和 embedding 预热并行、rerank 和 prompt 构建并行。把串行改并行能砍掉 30-50% 延迟。
  • 缓存：查询改写结果、检索结果、甚至 LLM 生成都可缓存（相同查询直接返回）。RAG 场景缓存命中率 20-40%。
  • 模型分级：简单任务用小模型（Haiku 改写、Haiku 生成简单答案），复杂任务才上大模型。Router 按难度分流。
  • 流式输出：LLM 生成用流式，用户看到第一个 token 的延迟从几秒降到几百毫秒，体感快很多。

成本优化的几个杠杆：

  • 按需触发模块：不是每个查询都需要 HyDE、Decomposition、Self-RAG。用分类器判断查询复杂度，简单的走轻量路径。
  • Prompt Caching：system prompt 和 few-shot 示例缓存，省 50-90% 的输入 token 费用。
  • 批处理：非实时场景（如离线索引、批量评估）用 batch API，成本是实时的 1/2。
  • 开源模型替代：部分环节（embedding、rerank、简单生成）用开源模型本地部署，省 API 费。
# RAG 的成本延迟监控
class RAGCostMonitor:
    def __init__(self): self.costs = []
    def track(self, step, llm_calls, tokens, latency_ms):
        cost = tokens * 0.00001   # 简化计价
        self.costs.append({"step": step, "calls": llm_calls,
                           "tokens": tokens, "cost": cost, "latency_ms": latency_ms})
    def summary(self):
        total_cost = sum(c["cost"] for c in self.costs)
        total_lat = sum(c["latency_ms"] for c in self.costs)
        print(f"总成本: ${total_cost:.4f}  总延迟: {total_lat:.0f}ms")
        for c in self.costs:
            print(f"  {c['step']:15s} ${c['cost']:.4f} {c['latency_ms']:.0f}ms")
# 用监控定位成本和延迟的大头，针对性优化

优化原则：先监控定位大头，再针对性优化。不要盲目优化所有模块——80% 的成本和延迟通常集中在 20% 的环节（往往是 LLM 生成和 rerank），优化这些收益最大。

★ Modular RAG 的核心模块深入：Router 与 Self-RAG

Modular RAG 里最关键的两个模块是 Router 和 Self-RAG，它们让系统从"固定流水线"变成"动态智能体"。

Router 的设计。Router 决定一个查询走哪条路径，是模块化的入口。实现方式：① 规则路由——用关键词/长度/正则判断（如含"对比"走多路检索），简单但死板；② LLM 路由——让小 LLM 分类查询类型，灵活但有延迟和成本；③ 学习路由——训练一个小分类器（如 FastText、小 BERT），快且准但要标注数据。生产中常用 LLM 路由（用 Haiku 这类便宜模型）+ 规则兜底的组合。

Router 的关键决策点：要不要检索（闲聊不检索）、检索哪类（文档/FAQ/SQL）、检索深度（简单 top-5 vs 复杂多跳）、要不要 rerank、用哪个 LLM 生成。一个好的 Router 能把"对所有查询一视同仁"变成"按需分配资源"，整体成本和延迟大幅优化。

Self-RAG 的自反思。Self-RAG 让 LLM 在生成后自评：这个答案基于 context 吗？完整吗？如果不好，触发重新检索或重写。这把 RAG 从"一次检索+一次生成"变成"检索-生成-评估-必要时重来"的闭环。

# Router + Self-RAG 的完整编排
def modular_rag(query, chat_history):
    # 1. Router 判断
    route = router.classify(query, chat_history)
    # route = {"need_retrieval": True, "type": "factual", "depth": "simple", "llm": "haiku"}
    if not route["need_retrieval"]:
        return llm.generate(query, model="haiku")
    # 2. 按路由检索
    if route["type"] == "multi_hop":
        context = multi_hop_retrieve(query, hops=3)
    elif route["type"] == "factual":
        context = simple_retrieve(query, k=5)
    else:
        context = hybrid_retrieve(query, k=10)
    # 3. 生成
    answer = llm.generate(query, context, model=route["llm"])
    # 4. Self-RAG 自反思
    critique = self_rag.critique(query, answer, context)
    if critique.needs_retry:
        if critique.issue == "low_faithfulness":
            # 幻觉：强化 prompt 重生成
            answer = llm.generate(query, context, model="sonnet",
                                  extra_instruction="严格基于 context")
        elif critique.issue == "insufficient_context":
            # 上下文不足：扩大检索重来
            context = retrieve(query, k=20)
            answer = llm.generate(query, context, model=route["llm"])
    return answer

Self-RAG 的 critique 设计。自反思要让 LLM 输出结构化判断：faithfulness（忠实度）、completeness（完整性）、needs_retry（要不要重来）。用专门的 critique prompt，让 LLM 逐项评估。要注意 critique 本身也可能错——过度自反思会让系统陷入"一直不满意一直重来"的死循环。要设最大重试次数（如 2 次）和置信阈值。

成本控制。Router 和 Self-RAG 都增加 LLM 调用，要控成本。Router 用最便宜的模型（Haiku/规则），Self-RAG 只对"边缘答案"（faithfulness 在 0.5-0.8 之间）触发，明确的答案（>0.9 或 <0.3）不 critique。这样把额外开销控制在 20-30%。

这两个模块让 Modular RAG 真正"智能"——不是固定的 if-else，而是按查询动态决策。但智能也带来复杂度和不可预测性，要配合完善的监控和评估（140 篇）才能稳定运行。

★ RAG 系统的成本建模与优化

RAG 系统的成本不透明，建个成本模型才能看清钱花在哪、怎么省。

成本构成。一次查询的成本 = embedding 编码 + 向量检索 + rerank + LLM 生成。其中 LLM 生成占大头（80-95%），rerank 次之，检索和 embedding 很少。优化重点在 LLM 生成。

LLM 生成的成本杠杆。① 模型分级——简单查询用 Haiku，复杂用 Sonnet/Opus，Router（137 篇）动态选；② 上下文压缩——rerank 后只传 top-3 而非 top-20，输入 token 减半；③ 缓存——相同查询的答案缓存，命中率 20-40% 能省大头；④ 批处理——非实时场景批量生成。

# RAG 查询的成本计算与优化对比
def query_cost(model, input_tokens, output_tokens, cached=False):
    prices = {"haiku": (0.25, 1.25), "sonnet": (3, 15), "opus": (15, 75)}
    ip, op = prices[model]
    if cached: ip *= 0.1   # prompt caching
    return input_tokens/1e6 * ip + output_tokens/1e6 * op
# 朴素 RAG：Sonnet + 5000 token 上下文 + 300 token 答案
naive = query_cost("sonnet", 5000, 300)
# 优化 RAG：Router 分级（70% Haiku）+ rerank 压缩到 1500 token + 30% 缓存
opt = 0.7 * query_cost("haiku", 1500, 300) + 0.3 * query_cost("sonnet", 1500, 300)
opt_cached = opt * 0.7 + opt * 0.3 * 0.1   # 30% 查询命中缓存
print(f"朴素: ${naive*1000:.2f}/千查询  优化: ${opt_cached*1000:.2f}/千查询")
print(f"降本: {(1-opt_cached/naive)*100:.0f}%")

rerank 成本。rerank 用 cross-encoder，10 个候选 × 100ms 延迟，如果自部署 GPU 成本可控。用 API（如 Cohere rerank）按调用收费，高频要算账。优化：只对 Router 判定为"需要精排"的查询 rerank，简单查询跳过。

检索与 embedding 成本。自建向量库主要是服务器成本，按 QPS 算。embedding 编码用 CPU ONNX 最省。这部分通常占总成本 <10%，不是优化重点，但大规模下也要盯。

成本/质量权衡。降本会降质量，要找甜区。通常 Router 分级 + 上下文压缩能降 50-60% 成本而质量掉 <5%，是高 ROI 优化。再往下（如全用 Haiku、砍 rerank）质量掉明显，要按业务容忍度。建成本模型 + 质量监控（140 篇），持续找甜区。

★ 本章小结

  • Naive RAG 是三段式（embed→检索→生成），简单但召回差、无重排、上下文利用率低，只适合 demo。
  • Advanced RAG 拆成检索前（查询改写/路由）、检索中（混合检索/过滤）、检索后（重排/压缩/打包）三段全优化，RAGAS 通常从 0.5 涨到 0.75。
  • Modular RAG 把每个优化抽象成可插拔模块，用 Router 按 query 类型动态编排，加 Self-RAG 自反思和 Tool use 补足盲区。
  • 选型：先 Naive 跑通，按失败 case 增量演进；多数业务在 Advanced 阶段够用；只有 query 高度多样、精度要求极高才上 Modular。

import numpy as np
from dataclasses import dataclass
from typing import List, Optional

@dataclass
class PerformanceMetrics:
    latency_ms: float
    throughput_ops: float
    memory_mb: float
    accuracy: float

def benchmark(fn, iterations=100):
    """通用性能基准测试"""
    import time
    t0 = time.time()
    for _ in range(iterations):
        result = fn()
    elapsed = time.time() - t0
    return PerformanceMetrics(
        latency_ms=elapsed * 1000 / iterations,
        throughput_ops=iterations / elapsed,
        memory_mb=0,  # 实际中用 psutil 获取
        accuracy=0.0  # 根据具体任务计算
    )

# 示例：对比两种方案的性能
def approach_a(data):
    """方案A：直接计算"""
    return np.softmax(data @ data.T / np.sqrt(data.shape[-1]), axis=-1)

def approach_b(data):
    """方案B：分块计算（省内存）"""
    chunk_size = 64
    results = []
    for i in range(0, data.shape[0], chunk_size):
        chunk = data[i:i+chunk_size]
        results.append(np.softmax(chunk @ data.T / np.sqrt(data.shape[-1]), axis=-1))
    return np.vstack(results)

data = np.random.randn(512, 768)
metrics_a = benchmark(lambda: approach_a(data))
metrics_b = benchmark(lambda: approach_b(data))
print(f"方案A: {metrics_a.latency_ms:.1f}ms, {metrics_a.throughput_ops:.0f} ops/s")
print(f"方案B: {metrics_b.latency_ms:.1f}ms, {metrics_b.throughput_ops:.0f} ops/s")

常见问题FAQ

Q: 如何选择合适的方案？ 先用方案A跑通验证正确性，再根据实际瓶颈（内存/延迟/精度）切换到方案B或C。永远不要过早优化。

Q: 生产环境最大的坑是什么？ 过度工程化——在需求不明确时投入大量精力做复杂架构。建议MVP先行，收集反馈后迭代。

Q: 如何评估效果？ 定义明确的成功指标（准确率/延迟/成本），用golden set做回归测试，A/B测试验证改进效果。至少100个样本才有统计显著性。
