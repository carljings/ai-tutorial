向量检索评估基准
VectorDBBench / ANN-benchmark 与 recall@k

"我这个向量库性能怎么样？"——这个问题没有真实测过就没法回答。ANN-benchmarks 和 VectorDBBench 是当前最权威的两个向量检索评估基准，前者测算法本身的 recall-QPS 曲线，后者测完整向量库产品（Milvus/Qdrant/Weaviate）的端到端性能。本文拆解 recall@k 指标的定义、两个基准的用法，并跑一遍实操。

阅读约 16 分钟 难度 ★★★

★ recall@k：最核心也最易被滥用的指标

recall@k 是评估 ANN 算法的基石指标。定义：对每个查询 q，先用暴力检索算出真正的 top-k（ground truth），再用 ANN 算法算出近似 top-k，recall@k = |近似 top-k ∩ 真实 top-k| / k。对一组查询取平均，就是系统的 recall@k。

这个定义看似简单，实操中却有几个易被滥用的细节：

  • k 必须一致：暴力 top-10 和 ANN top-10 比，不能暴力 top-100 和 ANN top-10 比（后者会虚高）。
  • 距离度量要一致：暴力用 L2，ANN 也必须用 L2，不能一个 L2 一个余弦。
  • 查询集要独立：查询不能出现在数据库里（不然 ANN 一定能找到自己，recall 虚高）。
  • 样本量要够：至少几百到几千个查询，否则方差大。

另一个进阶指标是 recall@k vs QPS 曲线——单点 recall 没意义，要和速度一起看（120 篇讲过）。

# 手算 recall@k —— 理解每个细节
import numpy as np

def compute_recall_at_k(ann_results, gt_results, k):
    """
    ann_results: list of list，每个查询的 ANN top-k（id 列表）
    gt_results:  list of list，每个查询的真实 top-k（id 列表）
    k:          top-k
    return:     平均 recall@k
    """
    assert len(ann_results) == len(gt_results)
    recalls = []
    for ann, gt in zip(ann_results, gt_results):
        ann_set = set(ann[:k])
        gt_set = set(gt[:k])
        recall = len(ann_set & gt_set) / k
        recalls.append(recall)
    return np.mean(recalls)

# 模拟：1000 查询，每个查 top-10
N_q = 1000; k = 10
gt = [list(np.random.choice(100000, k, replace=False)) for _ in range(N_q)]
# ANN 大部分对、少部分错
ann = [list(set(g[:8]) | set(np.random.choice(100000, 2))) for g in gt]
print(f"recall@10 = {compute_recall_at_k(ann, gt, 10):.3f}")
# 输出约 0.8 —— 因为每个查询 10 个里命中 8 个

★ ANN-benchmarks：算法层面的标杆

ANN-benchmarks（ann-benchmarks.com）是 Erik Bernhardsson 等人维护的算法评测，覆盖 HNSW、Annoy、FAISS-IVF、NGT、ScaNN 等几乎所有主流 ANN 算法。它用标准数据集（如 SIFT 1M、GIST 1M、NYTimes、Fashion-MNIST），在每个算法的多个参数档位下跑 recall@10 vs QPS 曲线，画在同一张图上比较。算法越靠右上（高 recall + 高 QPS）越好。

它的价值在于可比性：同一硬件、同一数据集、同一指标下，所有算法公平对比。这是你看"HNSW 和 ScaNN 谁快"这种问题的权威答案来源。但它只测算法（库级别），不测完整产品（不含网络、并发、持久化）。

# 在本地跑 ANN-benchmarks（Docker）
git clone https://github.com/erikbern/ann-benchmarks
cd ann-benchmarks

# 1. 安装
pip install -r requirements.txt

# 2. 下载一个数据集（如 SIFT 1M）
python install.py --dataset sift-128-euclidean

# 3. 跑某个算法（如 hnswlib）的评测
python run.py --algorithm hnswlib --dataset sift-128-euclidean

# 4. 跑多个算法对比
python run.py --algorithm hnswlib faiss annory --dataset sift-128-euclidean

# 5. 生成 recall-QPS 曲线图
python plot.py --dataset sift-128-euclidean --output plot.png
# 输出的图里每条曲线是一个算法，越靠右上越好

★ VectorDBBench：产品层面的标杆

VectorDBBench（zilliztech 给的）是评估完整向量库产品的基准——Milvus、Qdrant、Weaviate、Chroma、pgvector、Elasticsearch 等。它不只是测算法 recall-QPS，还测：

  • 加载/构建时间：1 亿向量插入 + 建索引要多久。
  • 并发 QPS：多客户端并发查询的真实吞吐。
  • 延迟分布：P50/P95/P99 延迟（不只看平均）。
  • 过滤场景性能：带属性过滤时 recall 和 QPS 怎么变化。
  • 容量规划：不同数据规模下的内存/磁盘占用。

选向量库产品时，VectorDBBench 是最权威的数据来源。它的结果反映了真实生产场景（含网络、序列化、并发），比 ANN-benchmarks 的算法对比更贴近实际，但也更复杂、跑一次更久。

★ 标准数据集：测什么很重要

不同数据集特性差异巨大，同一个算法在不同数据集上表现可能反转。常见的标准数据集：

  • SIFT-128-euclidean：图像特征，128 维 L2，1M 规模，最经典的 baseline。
  • GIST-960-euclidean：图像 GIST 特征，960 维，更高维的挑战。
  • NYTimes-256-angular：新闻 embedding，256 维余弦。
  • Glove-25/100-angular：词向量，不同维度。
  • Cohere-768-angular：Cohere embedding，768 维，更接近现代 RAG 场景。

选数据集的原则：尽量选和你业务相近的（维度、距离度量、规模、分布）。如果业务是 768 维余弦的文本 embedding，Glove-25 这种低维数据测出来的结果参考价值有限。

★ 评测常见误区

  • 只看 QPS 不看 recall：调小 efSearch 能让 QPS 翻倍，但 recall 掉到 0.5，单看 QPS 没意义。
  • 用训练数据当查询：查询和库重叠，recall 虚高。
  • k 不匹配：暴力算 top-100，ANN 算 top-10，recall 虚高。
  • 硬件不一致对比：在自己机器测 HNSW，在论文里抄 ScaNN 数据，对比不公平。
  • 忽略 P99 延迟：平均延迟 5ms 但 P99 200ms，用户体验仍差。
  • 忽略过滤场景：很多库在无过滤时很快，加了属性过滤就崩，不测过滤等于没测。

★ 自己搭一套评测流程

除了跑现成基准，多数团队需要测自己业务数据的性能。流程：

# 自建评测：用自己的 embedding 数据测向量库性能
import numpy as np
import time
import faiss

# 1. 准备数据：用真实 embedding（这里用随机模拟）
d = 768; N = 10_000_000
xb = np.random.randn(N, d).astype(np.float32); faiss.normalize_L2(xb)

# 拆分：95% 入库，5% 做查询（保证查询不在库里）
split = int(N * 0.95)
db_vecs, query_vecs = xb[:split], xb[split:]

# 2. 构建 ground truth（暴力，只对查询集算）
flat = faiss.IndexFlatIP(d); flat.add(db_vecs)
k = 10
_, gt = flat.search(query_vecs, k)    # 这个会慢，但只算一次

# 3. 构建待测索引
def build_index(index_type="hnsw"):
    if index_type == "hnsw":
        idx = faiss.IndexHNSWFlat(d, 32)
        idx.hnsw.efConstruction = 200
    elif index_type == "ivfpq":
        q = faiss.IndexFlatIP(d)
        idx = faiss.IndexIVFPQ(q, d, 4096, 64, 8, faiss.METRIC_INNER_PRODUCT)
        idx.train(db_vecs[:50000])
    idx.add(db_vecs)
    return idx

# 4. 评测函数：扫参数档位，画 recall-QPS 曲线
def benchmark(idx, param_name, param_values, set_param_fn):
    results = []
    for v in param_values:
        set_param_fn(idx, v)
        t0 = time.time()
        _, I = idx.search(query_vecs, k)
        qps = len(query_vecs) / (time.time() - t0)
        # 算 recall@k
        hits = sum(len(set(I[i]) & set(gt[i])) for i in range(len(query_vecs)))
        recall = hits / (len(query_vecs) * k)
        results.append((v, recall, qps))
        print(f"{param_name}={v}  recall@{k}={recall:.4f}  QPS={qps:.0f}")
    return results

idx = build_index("hnsw")
benchmark(idx, "efSearch", [16, 32, 64, 128, 256],
          lambda i, v: i.hnsw.__setattr__("efSearch", v))
# 输出一张表，按业务需要的 recall 下限选最优 efSearch

★ 设计自己的 benchmark：数据集与场景设计

自建 benchmark 不是随便跑跑，要设计得能反映真实场景。几个设计要点。

数据集的真实性。最理想是用你自己的业务 embedding 和真实查询。退而求其次，用和业务相近的公开数据集：文本检索用 BEIR 的子集、代码检索用 CodeSearchNet、多语言用 MIRACL。不要用 SIFT/GIST 这种图像特征数据集测文本 RAG——分布差太远，结果没参考价值。

查询集的多样性。查询要覆盖真实场景的各种类型：事实查询、对比查询、模糊查询、长尾词查询、多跳查询。只测事实查询会让 benchmark 偏向某类算法（如精确匹配强的）。建议按业务真实查询的分布抽样。

规模梯度。不要只测一个规模。测 10 万、100 万、1000 万三档，看算法在不同规模下的表现变化。有些算法小规模强、大规模崩（如 HNSW 的内存），有些相反。

过滤场景。如果你的业务有属性过滤（绝大多数有），一定要测过滤场景下的性能。很多算法无过滤时很强，加了过滤就崩。

# 设计覆盖多场景的 benchmark
def design_benchmark(business_data, query_types):
    scenarios = []
    # 1. 不同规模
    for scale in [100_000, 1_000_000, 10_000_000]:
        data = business_data.sample(scale)
        # 2. 不同查询类型
        for qtype, queries in query_types.items():
            # 3. 有/无过滤
            for filter_ratio in [1.0, 0.1, 0.01]:   # 无过滤、10%、1% 命中
                scenarios.append({
                    "scale": scale, "query_type": qtype,
                    "filter_ratio": filter_ratio,
                    "data": data, "queries": queries,
                })
    return scenarios
# 跑所有场景，画矩阵图，看算法在哪个场景强/弱

评估指标的全面性。不只看 recall@k 和 QPS，还要看：P99 延迟（长尾体验）、内存占用（成本）、构建时间（迭代速度）、索引大小（存储成本）。一个算法 recall 高但 P99 抖，可能不适合在线服务。

可复现性。benchmark 要能复现：固定随机种子、记录硬件配置、版本号、参数档位。否则结果不可比较，调优就失去依据。建议把 benchmark 脚本和数据集版本一起管理（git + DVC）。

★ benchmark 的常见误读与陷阱

即便跑了 benchmark，结果也可能误导。几个常见陷阱要警惕。

陷阱一：单点比较。只比一个参数档位（如 HNSW efSearch=64 vs IVF nprobe=16）就下结论。正确做法是扫参数画曲线比，因为不同算法的最优档位不同，单点比可能恰好比的 unfavorably。

陷阱二：忽略硬件差异。在自己机器测 HNSW，在论文里抄 ScaNN 数据，对比不公平。ScaNN 对 AVX-512 优化深，老 CPU 上表现差；HNSW 对缓存敏感，大内存机器上更强。必须同一硬件对比。

陷阱三：数据分布偏差。用均匀随机数据测，结果和真实业务差很远。真实 embedding 往往高度聚簇（同类内容挤在一起），这种分布下不同算法表现可能反转。一定要用真实数据测。

陷阱四：忽略查询分布。benchmark 的查询如果都是"容易的"（明确、有明确答案），会高估所有算法。真实查询里有大量"难的"（模糊、长尾、多跳），这些查询才是区分算法的关键。查询集要包含足够比例的难查询。

陷阱五：过度优化 benchmark。在 benchmark 上调到最优，上线后效果没变好——因为 benchmark 和线上分布有差异。对策：定期用线上真实流量回放更新 benchmark，保持它和线上一致。

# 避免 benchmark 陷阱的检查清单
benchmark_checklist = [
    "□ 扫参数画曲线，而非单点比较",
    "□ 同一硬件、同一数据集对比",
    "□ 用真实业务数据，非均匀随机",
    "□ 查询集含足够比例的难查询",
    "□ 测了过滤场景（如业务有过滤）",
    "□ 看了 P99 延迟，不只平均",
    "□ 考虑了内存/构建时间，不只 recall/QPS",
    "□ benchmark 定期用线上流量更新",
]
for item in benchmark_checklist: print(item)

正确态度。benchmark 是工具不是真理。它的作用是"排除明显差的选项"和"量化权衡"，而非"给出唯一最优解"。最终选型要结合 benchmark 数据 + 业务约束 + 团队能力综合判断。一个 benchmark 第三名但团队熟悉的算法，可能比 benchmark 第一名但没人会维护的算法更适合你。

这种务实态度能让 benchmark 真正发挥作用，而非成为"为了好看的数字而调参"的形式主义。评估的最终目的是决策，不是数字本身。

★ 主流向量库产品的 benchmark 对比与解读

VectorDBBench 跑出来的产品对比怎么解读？看几个关键维度和常见误读。

关键对比维度。看向量库 benchmark 不要只看 recall 和 QPS，要综合：

  • 加载/构建时间：亿级数据建索引要多久。Milvus 较快，Qdrant 中等，DiskANN 慢。
  • 并发 QPS：多客户端并发下的真实吞吐。Qdrant 单机并发强，Milvus 分布式扩展好。
  • 延迟分布：P99/P999 长尾。Weaviate P99 抖动较大，Milvus/Qdrant 较稳。
  • 过滤性能：带属性过滤时表现。这是产品差异最大的维度——有些库过滤后崩，有些稳。
  • 资源效率：每 QPS 的 CPU/内存消耗。影响成本。

常见误读：

  • "QPS 最高的最好"——错。QPS 高但 P99 抖，在线服务体验差。要看延迟分布。
  • "recall 最高的最好"——错。recall 99% 但构建要 10 小时，迭代慢。要综合构建时间。
  • "benchmark 第一的就选它"——错。benchmark 用通用数据，你的业务分布可能不同。要自测。
  • "过滤 QPS 不重要"——大错。绝大多数业务有过滤，过滤性能差的产品上线就崩。

按场景选型：

  • 大规模 + 分布式 + 团队能运维 → Milvus
  • 单机高性能 + 强过滤 + 简单部署 → Qdrant
  • 已有 PostgreSQL + 中等规模 → pgvector（最少引入新组件）
  • 已有 ES + 关键词为主 → Elasticsearch
  • 原型/小规模 + 快速迭代 → Chroma（轻量）
  • 混合检索 + 模块化 → Weaviate

自测的重要性。VectorDBBench 的结果用通用数据，你的业务数据分布可能让排名反转。务必用自己数据在 2-3 个候选库上跑同样的 benchmark，再决策。这比看任何榜单都靠谱。144 篇前面讲了自建 benchmark 的方法，这里强调：选向量库一定要自测，榜单只是起点。

长期考量。选向量库不只是选当下的性能，还要看：① 社区活跃度（Milvus/Qdrant 活跃，bug 修复快）；② 生态成熟度（连接器、监控、文档）；③ 升级路径（数据迁移成本）。一个当下第二但生态成熟的库，可能比当下第一但小众的库更适合长期投入。

★ Benchmark 数据集的准备与质量管理

benchmark 的质量很大程度取决于数据集。怎么准备和管理高质量的 benchmark 数据集？

查询集的代表性。benchmark 用的查询要反映真实业务分布。来源：① 真实查询日志（去敏后）；② 人工构造（覆盖各种查询类型）；③ LLM 合成（量大但可能偏离真实）。最佳是真实查询日志采样 + 人工补充边缘 case。几百到几千条查询够用，贵精不贵多。

ground truth 的质量。recall@k 依赖准确的 ground truth。获取方法：① 全量暴力检索（最准但慢，大库不可行）；② 人工标注（贵）；③ 多算法交叉（用多个 ANN 算法各自召回 top-k，并集作为近似 ground truth）。第三种实用——HNSW + IVF + 暴力的并集，漏真近邻的概率低。

# 多算法交叉生成近似 ground truth
def build_ground_truth(data, queries, k=100):
    # 三个不同算法各自召回
    gt_hnsw = hnsw_search(data, queries, k*3)
    gt_ivf = ivf_search(data, queries, k*3, nprobe=128)
    gt_brute = brute_force_sample(data, queries, k*3, sample_ratio=0.1)
    # 并集 + 暴力精排
    gt = []
    for i in range(len(queries)):
        union = set(gt_hnsw[i]) | set(gt_ivf[i]) | set(gt_brute[i])
        # 对并集做暴力精排取 top-k
        dists = [(dist(data[q_idx], queries[i]), idx) for idx in union]
        dists.sort()
        gt.append([idx for _, idx in dists[:k]])
    return gt
# 这样漏真近邻的概率远低于单算法

数据集的版本管理。数据集要和代码一起 git，每次改动记录原因。数据集变了（加查询、改 ground truth），历史 benchmark 结果不能直接比——要在同一数据集版本上对比。用数据集 hash 标记版本。

评估集的隔离。benchmark 数据集不能用于调参（否则过拟合 benchmark）。要有独立的调参集和评估集。调参集用来选参数，评估集只用来报告最终指标。这是机器学习的 ABC，但 RAG 实践里常被忽略。

数据分布监控。业务数据会变（新术语、新内容），benchmark 数据集要定期更新以反映新分布。但不频繁更新（成本高、历史不可比）。折中：核心查询集稳定，定期加新查询反映变化。

★ 本章小结

  • recall@k = |ANN top-k ∩ 真实 top-k| / k，是 ANN 评估的基石。注意 k/距离/查询独立/样本量四个细节，否则数字虚高。
  • ANN-benchmarks 测算法本身（recall-QPS 曲线，库级别），VectorDBBench 测完整向量库产品（端到端 QPS/延迟/过滤/容量）。
  • 标准数据集（SIFT/GIST/NYTimes/Cohere）特性差异大，选和业务相近的。
  • 误区：只看 QPS 不看 recall、查询和库重叠、k 不匹配、硬件不一致、忽略 P99、忽略过滤场景。
  • 自建评测：用真实 embedding，拆查询集，先暴力算 ground truth，再扫参数档位画 recall-QPS 曲线。
