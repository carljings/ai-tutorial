LangChain

agent 框架的鼻祖与生态核心

深度进阶★★★

LangChain 的地位与演进

LangChain 是最早也是最知名的 LLM 应用框架，2022 年底发布。它最初把"LLM + 提示词 + 工具 + 记忆"封装成可组合的组件，极大降低了 agent 开发门槛。随着生态发展，LangChain 演进为一个框架家族：核心 LangChain（组件库）、LangGraph（有状态 agent 图）、LangSmith（可观测性）。

核心抽象：Chain 和 Agent

LangChain 最核心的抽象是"链"（Chain）——把多个步骤串起来。早期用 AgentExecutor 来跑 agent 循环，但它在复杂场景下不够灵活（难以自定义控制流）。0.1 版本引入了 LCEL（LangChain Expression Language），用 | 管道符把组件组合成链，更声明式也更灵活。

from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI
from langchain_core.output_parsers import StrOutputParser

# LCEL：用管道符组合组件
prompt = ChatPromptTemplate.from_template("解释 {topic} 给一个10岁小孩")
model = ChatOpenAI(model="gpt-4")
parser = StrOutputParser()

# 这就是一个 "chain"：prompt | model | parser
chain = prompt | model | parser
result = chain.invoke({"topic": "量子计算"})

工具与 Agent 的构建

from langchain_community.tools import DuckDuckGoSearchRun
from langchain.agents import create_tool_calling_agent, AgentExecutor

# 定义工具
tools = [DuckDuckGoSearchRun()]

# 创建 agent（现代方式：基于模型的原生 function calling）
agent = create_tool_calling_agent(model, tools, prompt)
executor = AgentExecutor(agent=agent, tools=tools, verbose=True)

# 运行
executor.invoke({"input": "GLM-5.2 是什么时候发布的？"})
💡 要点
LangChain 的价值在于"组件丰富 + 生态庞大"——几百个预置工具集成、几十个向量库支持、成熟的文档检索链。代价是抽象层较多、学习曲线陡、早期版本 API 频繁变动。现在官方推荐用 LangGraph 替代 AgentExecutor 做复杂 agent。

LCEL 链式语法：管道符组合的深度解析

第一节展示了 `prompt | model | parser` 的基本用法，这里深入 LCEL（LangChain Expression Language）的设计原理和进阶用法。LCEL 不只是语法糖——它统一了同步/异步/流式/批处理的调用接口，这是它比传统 Chain 类强大的根本原因。

from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI
from langchain_core.output_parsers import StrOutputParser, JsonOutputParser
from langchain_core.runnables import RunnablePassthrough, RunnableParallel

# 基础链：prompt | model | parser
prompt = ChatPromptTemplate.from_template("用一句话解释 {topic}")
model = ChatOpenAI(model="gpt-4o-mini", temperature=0)
chain = prompt | model | StrOutputParser()

# LCEL 的强大：同一套接口支持四种调用模式
# 1. 同步
result = chain.invoke({"topic": "量子计算"})
# 2. 异步
result = await chain.ainvoke({"topic": "量子计算"})
# 3. 批量（自动并行）
results = chain.batch([{"topic": "量子计算"}, {"topic": "区块链"}])
# 4. 流式
for chunk in chain.stream({"topic": "量子计算"}):
    print(chunk, end="", flush=True)

# 进阶 1：RunnablePassthrough（透传输入到输出）
# 场景：RAG 链里要把原始 question 传给 prompt，同时检索 context
rag_chain = (
    {"context": retriever, "question": RunnablePassthrough()}
    | ChatPromptTemplate.from_template("基于 {context} 回答 {question}")
    | model
    | StrOutputParser()
)
# RunnablePassthrough() 把输入原样传给 prompt 的 question 占位符
# retriever 接收输入，检索结果传给 context 占位符

# 进阶 2：RunnableParallel（并行执行多个链）
# 场景：同时生成中文和英文摘要
parallel = RunnableParallel(
    chinese=ChatPromptTemplate.from_template("用中文总结: {text}") | model | StrOutputParser(),
    english=ChatPromptTemplate.from_template("Summarize in English: {text}") | model | StrOutputParser(),
)
result = parallel.invoke({"text": "...长文本..."})
# result = {"chinese": "...", "english": "..."}  两个版本同时生成

# 进阶 3：带 JSON 解析和重试的链
json_parser = JsonOutputParser(pydantic_object=Summary)
robust_chain = (
    ChatPromptTemplate.from_template("分析并输出 JSON: {text}\n{format_instructions}")
    | model
    | json_parser
    .with_retry(stop_after_attempt=3)   # 解析失败自动重试 3 次
)

LCEL 的设计哲学是"所有组件都是 Runnable"——有统一的 invoke/ainvoke/batch/stream 接口。这意味着任何 Runnable 都能用 `|` 组合，组合后的链本身也是 Runnable，可以继续组合。这种"可组合性"让复杂逻辑（RAG + 并行 + 重试 + 流式）能用几行声明式代码表达，而不需要手写控制流。`RunnablePassthrough` 和 `RunnableParallel` 是两个最实用的原语：前者解决"输入要同时传给下游多个位置"的接线问题（RAG 必备），后者解决"并行执行"的性能问题。相比传统 AgentExecutor，LCEL 链的每一步都是显式的、可单独测试的。

💡 要点
LCEL 的 `|` 管道符底层是 Python 的 `__or__` 魔术方法。每个 Runnable 实现了 `__or__`，返回一个新的 RunnableSequence。这就是为什么 `prompt | model` 能返回一个可调用的链——它本质是 `RunnableSequence([prompt, model])`。理解这个，就不会觉得 LCEL 是"魔法"，而是一种优雅的 DSL 设计。

RunnablePassthrough 与数据流控制

RunnablePassthrough 看似简单（就是透传嘛），但在复杂链里它是"数据流路由"的关键。理解它和 RunnableParallel、RunnableLambda 的配合，才能构建出复杂的数据管线。下面用一个完整的 RAG 链展示数据流控制。

from langchain_core.runnables import (
    RunnablePassthrough, RunnableLambda, RunnableParallel
)
from langchain_core.output_parsers import StrOutputParser

# 完整的 RAG 链：展示数据流如何被路由
def format_docs(docs):
    return "\n\n".join(d.page_content for d in docs)

# 场景：用户问问题 → 检索 → 同时做"问答"和"追问生成"
# 两个输出都基于检索结果，但用不同 prompt

# 第一步：并行准备数据
prep = RunnableParallel(
    # 把检索结果格式化
    context=retriever | RunnableLambda(format_docs),
    # 原始问题透传
    question=RunnablePassthrough(),
    # 派生信息：问题长度（展示 RunnableLambda 做转换）
    question_len=RunnableLambda(lambda q: len(q)),
)

# 第二步：两个并行分支
answer_branch = (
    ChatPromptTemplate.from_template(
        "基于以下资料回答问题。\n资料: {context}\n问题: {question}"
    )
    | model | StrOutputParser()
)

followup_branch = (
    ChatPromptTemplate.from_template(
        "基于资料，生成 3 个相关的追问。\n资料: {context}\n问题: {question}"
    )
    | model | StrOutputParser()
)

# 组合：prep → 并行两个分支
full_chain = prep | RunnableParallel(
    answer=answer_branch,
    followups=followup_branch,
    # 透传元数据
    metadata=RunnablePassthrough(),
)

result = full_chain.invoke("GLM-5.2 的上下文长度是多少？")

#   "answer": "GLM-5.2 支持 128K 上下文...",
#   "followups": "1. 它的推理速度如何?\n2. ...",
#   "metadata": {"context": "...", "question": "...", "question_len": 18}
# }

这段代码展示了 LCEL 数据流控制的精髓：`RunnableParallel` 在每一步同时处理多个字段，`RunnablePassthrough` 把上游数据原样透到下游，`RunnableLambda` 做任意转换（如格式化、计算）。三者组合，能表达任意复杂的数据路由——相当于声明式的数据管道。注意 `prep | RunnableParallel(...)` 这个结构：prep 阶段准备好 {context, question, question_len} 字典，然后两个分支各自从这个字典取需要的字段。这种"先准备数据，再并行消费"的模式，比把所有逻辑塞进一个巨型 prompt 清晰得多，也更易调试（每个 Runnable 可单独 invoke 测试）。

Agent Executor：从传统到现代的演进

第一节给了 AgentExecutor 的基本用法，这里讲它的局限和演进。AgentExecutor 是 LangChain 早期的 agent 运行器，封装了 ReAct 循环。但它在复杂场景下不够灵活——难以自定义控制流、无法精确控制工具调用顺序、错误处理粗糙。这就是为什么官方现在推荐迁移到 LangGraph。

# 传统 AgentExecutor（适合简单场景）
from langchain.agents import create_tool_calling_agent, AgentExecutor
from langchain_core.prompts import ChatPromptTemplate

tools = [search_tool, calculator_tool]
prompt = ChatPromptTemplate.from_messages([
    ("system", "你是一个有用的助手，可以用工具。"),
    ("human", "{input}"),
    ("placeholder", "{agent_scratchpad}"),   # 工具调用历史占位符
])

agent = create_tool_calling_agent(model, tools, prompt)
executor = AgentExecutor(
    agent=agent,
    tools=tools,
    verbose=True,
    max_iterations=10,        # 最大循环次数
    handle_parsing_errors=True,  # 解析错误自动恢复
    early_stopping_method="generate",  # 超时前让模型生成最终答案
)

# 传统方式的局限：
# 1. 控制流固定（ReAct 循环），无法加条件分支
# 2. 无法在中间步骤插入人工审批
# 3. 错误处理只有"重试或停止"两选项
# 4. 无法表达"先并行做 A/B，再汇总"这种结构

result = executor.invoke({"input": "查 GLM-5.2 的发布信息并算它距今天数"})

# 现代方式：迁移到 LangGraph（支持复杂控制流）
from langgraph.graph import StateGraph, END
from langgraph.prebuilt import ToolNode

# LangGraph 能做 AgentExecutor 做不到的事：
# - 条件分支（if 复杂度 > X，走人工审核）
# - 并行节点（同时调多个工具）
# - 循环 + 状态（带 memory 的多轮）
# - 中断恢复（human-in-the-loop）

def build_modern_agent(model, tools):
    graph = StateGraph(AgentState)
    graph.add_node("agent", call_model)
    graph.add_node("tools", ToolNode(tools))
    graph.add_node("review", human_review_node)   # 人工审核节点

    graph.set_entry_point("agent")
    # 条件路由：简单直接答，复杂走审核，要工具走 tools
    graph.add_conditional_edges("agent", route_decision,
                                {"tools": "tools", "review": "review", "end": END})
    graph.add_edge("tools", "agent")   # 工具结果回 agent
    graph.add_edge("review", "agent")  # 审核后回 agent
    return graph.compile()

# 迁移建议：
# - 简单 agent（固定 ReAct）：AgentExecutor 够用，不必迁移
# - 复杂 agent（需控制流/审批/并行）：迁移到 LangGraph
# - 新项目：直接用 LangGraph，跳过 AgentExecutor

从 AgentExecutor 到 LangGraph 的演进，反映了 agent 工程从"封装好的黑盒"到"显式图结构"的趋势。AgentExecutor 的优点是简单——三行代码就能跑一个 agent；缺点是"黑盒"——一旦需要定制控制流，就要深入源码 hack。LangGraph 把控制流显式化成图，每个节点、每条边都可见可改，代价是代码量增加。迁移判断标准很简单：如果你的 agent 逻辑能用"循环调工具直到回答"描述，AgentExecutor 够用；一旦你需要"条件分支""人工审批""并行节点"中的任何一个，就该上 LangGraph。新项目建议直接用 LangGraph——它的 prebuilt ToolNode 和 create_react_agent 已经覆盖了 AgentExecutor 的大部分简单场景，没有理由再从旧 API 开始。

特性AgentExecutorLangGraph
控制流固定 ReAct 循环任意图结构
条件分支不支持conditional_edges
人工审批不支持interrupt_before
并行不支持多节点并行
状态持久化checkpointer（弱）内置 checkpointer
代码量少（3 行起步）多（需定义图）
学习曲线平缓陡峭
💡 要点
不要因为 LangGraph 更"先进"就全盘迁移。很多 agent 场景就是简单的 ReAct 循环，AgentExecutor 跑得好好的，迁移到 LangGraph 反而增加复杂度。技术选型的原则是"用刚好够的工具解决当前问题"——AgentExecutor 够用就用它，需求超出它的能力边界时再上 LangGraph。LangChain 生态的价值恰恰是提供了从简单到复杂的完整梯度。

本章小结

LangChain 是 agent 框架的奠基者：Chain 抽象、LCEL 管道、丰富的工具/向量库集成。它的生态最大、文档最全，是入门 agent 开发的首选。复杂 agent 场景推荐迁移到同家族的 LangGraph。
