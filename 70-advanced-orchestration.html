Agent 编排
用 LangGraph 构建状态图、条件分支与人在回路

单个 LLM 调用能做的事有限，真正的生产力来自 Agent——能自主规划、调用工具、根据条件分支、在关键节点请人确认的复合系统。这一章用 LangGraph 这个当前最成熟的 Agent 编排框架，手把手构建一个带状态管理、条件路由、人在回路（Human-in-the-Loop）的完整 Agent。看完你就能设计出能处理真实业务流程的 AI Agent。

阅读约 31 分钟 难度 ★★★★

★ 一、什么是 Agent 编排

Agent 和普通 LLM 调用的区别在于「自主性」。普通调用是你给 prompt、模型给回答，一次结束。Agent 是你给一个目标，模型自己决定：调用哪些工具、按什么顺序、什么时候需要更多信息、什么时候任务完成。这需要一个「编排系统」来管理状态、路由、工具调用。

Agent 编排要解决四个核心问题：状态管理（记住中间结果）、流程控制（条件分支和循环）、工具调用（与外部系统交互）、人工介入（关键决策需要人确认）。LangGraph 用「状态图」（StateGraph）优雅地解决了这四个问题。

★ 二、LangGraph 的核心概念

LangGraph 把 Agent 建模为一张「状态图」。图里有节点（Node，执行具体动作）和边（Edge，定义流转方向）。每个节点接收当前状态、返回状态更新。这种设计让 Agent 的行为变得可视、可调试、可回放。

# LangGraph 的核心概念演示
from langgraph.graph import StateGraph, END
from typing import TypedDict, Annotated, List
import operator

# 第 1 步：定义状态（State）——整个 Agent 共享的数据结构
class AgentState(TypedDict):
    messages: Annotated[List[str], operator.add]  # 对话历史，自动累加
    current_task: str                              # 当前任务
    retrieved_docs: List[str]                      # 检索到的文档
    needs_human_review: bool                       # 是否需要人工审核
    final_answer: str                              # 最终答案

# 第 2 步：定义节点（Node）——每个节点是一个函数
def retrieve_node(state: AgentState) -> dict:
    """检索相关文档。"""
    query = state["current_task"]
    docs = vector_db.search(query, top_k=3)
    return {"retrieved_docs": docs, "messages": ["检索完成"]}

def generate_node(state: AgentState) -> dict:
    """基于检索结果生成回答。"""
    context = "\n".join(state["retrieved_docs"])
    answer = llm.generate(f"基于以下内容回答：{context}\n问题：{state['current_task']}")
    return {"final_answer": answer, "messages": ["生成完成"]}

# 第 3 步：定义条件路由
def should_review(state: AgentState) -> str:
    """判断是否需要人工审核。"""
    answer = state["final_answer"]
    # 如果回答涉及高风险主题（如医疗、法律），需要人工审核
    if any(kw in answer for kw in ["医疗建议", "法律意见", "投资建议"]):
        return "human_review"
    return "done"

# 第 4 步：组装状态图
workflow = StateGraph(AgentState)
workflow.add_node("retrieve", retrieve_node)
workflow.add_node("generate", generate_node)
workflow.add_node("human_review", human_review_node)

workflow.set_entry_point("retrieve")
workflow.add_edge("retrieve", "generate")
workflow.add_conditional_edges("generate", should_review)
workflow.add_edge("human_review", END)
workflow.add_edge("done", END)

app = workflow.compile()

★ 三、条件分支：让 Agent 会做选择

真实业务流程不是线性的，需要根据中间结果走不同路径。LangGraph 的 conditional_edges 让你能动态决定下一步去哪个节点。

# 完整案例：智能客服 Agent 的条件路由
from langgraph.graph import StateGraph, END
from typing import TypedDict

class CustomerServiceState(TypedDict):
    user_query: str
    intent: str             # 意图：refund/technical/faq/complaint
    confidence: float       # 意图识别置信度
    response: str
    escalated: bool         # 是否升级到人工

def classify_intent(state: CustomerServiceState) -> dict:
    """识别用户意图。"""
    query = state["user_query"]
    # 用 LLM 做意图分类
    result = llm.classify(query, categories=["refund", "technical", "faq", "complaint"])
    return {"intent": result.category, "confidence": result.confidence}

def route_by_intent(state: CustomerServiceState) -> str:
    """根据意图和置信度路由到不同处理节点。"""
    # 置信度低 → 升级人工
    if state["confidence"] < 0.7:
        return "escalate"
    # 高风险意图（投诉）→ 升级人工
    if state["intent"] == "complaint":
        return "escalate"
    # 其他 → 对应处理节点
    return state["intent"]

def handle_refund(state: CustomerServiceState) -> dict:
    """处理退款请求。"""
    response = llm.generate(f"处理退款请求：{state['user_query']}")
    return {"response": response}

def handle_technical(state: CustomerServiceState) -> dict:
    """处理技术问题。"""
    docs = kb.search(state["user_query"])
    response = llm.generate(f"基于文档回答技术问题：{docs}")
    return {"response": response}

def escalate_to_human(state: CustomerServiceState) -> dict:
    """升级到人工客服。"""
    return {"escalated": True, "response": "已为您转接人工客服..."}

# 构建工作流
workflow = StateGraph(CustomerServiceState)
workflow.add_node("classify", classify_intent)
workflow.add_node("refund", handle_refund)
workflow.add_node("technical", handle_technical)
workflow.add_node("escalate", escalate_to_human)

workflow.set_entry_point("classify")
workflow.add_conditional_edges("classify", route_by_intent, {
    "refund": "refund",
    "technical": "technical",
    "escalate": "escalate",
})
for node in ["refund", "technical", "escalate"]:
    workflow.add_edge(node, END)

app = workflow.compile()

# 这个 Agent 的行为：
# 1. 先识别用户意图
# 2. 如果意图明确 → 走对应自动处理
# 3. 如果意图模糊或涉及投诉 → 升级人工
# 这就是「条件分支」的价值——Agent 会做判断

★ 四、人在回路（Human-in-the-Loop）

AI 不应该在所有事情上都自主决策——涉及金钱、安全、不可逆操作时，必须有人工确认环节。这就是 Human-in-the-Loop（HITL）。LangGraph 原生支持这种模式。

# 人在回路的实现：Agent 在关键节点暂停，等待人工确认
from langgraph.graph import StateGraph, END

class OrderAgentState(TypedDict):
    order_details: dict
    needs_approval: bool
    approved: bool
    result: str

def process_order(state: OrderAgentState) -> dict:
    """处理订单，涉及大额时需要人工确认。"""
    order = state["order_details"]
    # 金额超过 10000 的订单需要人工确认
    if order["amount"] > 10000:
        return {"needs_approval": True}
    # 小额订单直接处理
    return {"needs_approval": False, "result": execute_order(order)}

def check_approval(state: OrderAgentState) -> str:
    """路由：需要确认就走审批，否则直接完成。"""
    if state["needs_approval"]:
        return "await_approval"
    return "complete"

def await_human_approval(state: OrderAgentState) -> dict:
    """暂停，等待人工审批结果。"""
    # LangGraph 的 interrupt 机制：
    # Agent 在这里暂停，把控制权交给人类
    # 人类审批后，恢复执行
    approval = interrupt(
        prompt=f"订单 {state['order_details']} 需要审批，是否批准？",
        choices=["approve", "reject"]
    )
    if approval == "approve":
        return {"approved": True, "result": execute_order(state["order_details"])}
    else:
        return {"approved": False, "result": "订单已取消"}

# 构建带 HITL 的工作流
workflow = StateGraph(OrderAgentState)
workflow.add_node("process", process_order)
workflow.add_node("await_approval", await_human_approval)

workflow.set_entry_point("process")
workflow.add_conditional_edges("process", check_approval, {
    "await_approval": "await_approval",
    "complete": END,
})
workflow.add_edge("await_approval", END)

# 关键：compile 时启用 interrupt，让 Agent 能在指定节点暂停
app = workflow.compile(checkpointer=MemorySaver(), interrupt_before=["await_approval"])

# HITL 的典型场景：
# 1. 大额交易（金额超过阈值）
# 2. 不可逆操作（删除数据、发送邮件）
# 3. 高风险决策（医疗建议、法律文书）
# 4. 模型不确定时（置信度低）

★ 五、工具调用：让 Agent 连接真实世界

Agent 的强大在于它能调用外部工具——查数据库、调 API、执行代码。LangGraph 让工具调用成为图中的一个节点，可以和条件分支组合。

# Agent 工具调用的完整实现
from langgraph.prebuilt import ToolNode
from langchain.tools import tool

# 定义 Agent 可用的工具
@tool
def search_database(query: str) -> str:
    """搜索公司数据库。"""
    return db.execute(query)

@tool
def send_email(to: str, subject: str, body: str) -> str:
    """发送邮件（高风险操作，需配合 HITL）。"""
    email_service.send(to, subject, body)
    return "邮件已发送"

@tool
def calculate(expression: str) -> str:
    """执行数学计算。"""
    return str(eval(expression))  # 生产环境用安全的 eval

tools = [search_database, send_email, calculate]

def agent_node(state: AgentState) -> dict:
    """Agent 决策节点：决定调用哪个工具。"""
    messages = state["messages"]
    # 让 LLM 决定下一步用什么工具
    response = llm_with_tools.invoke(messages)
    return {"messages": [response]}

def should_use_tool(state: AgentState) -> str:
    """判断 Agent 是否需要调用工具，还是直接回答。"""
    last_message = state["messages"][-1]
    if last_message.tool_calls:  # LLM 决定调用工具
        return "tools"
    return END  # 没有工具调用，任务完成

# 构建带工具的 Agent
workflow = StateGraph(AgentState)
workflow.add_node("agent", agent_node)
workflow.add_node("tools", ToolNode(tools))  # 工具执行节点

workflow.set_entry_point("agent")
# 条件路由：需要工具 → tools，不需要 → 结束
workflow.add_conditional_edges("agent", should_use_tool)
# 工具执行完，回到 agent 做下一步决策（循环）
workflow.add_edge("tools", "agent")

app = workflow.compile()

# 这个 Agent 的循环逻辑：
# 1. agent 节点：LLM 看消息，决定调工具还是回答
# 2. 如果调工具 → tools 节点执行 → 回到 agent（循环）
# 3. 如果直接回答 → 结束
# 这就是 ReAct（Reason + Act）模式的图实现

★ 六、错误处理与回退

真实世界的 Agent 必须能处理工具调用失败、超时、异常输入。LangGraph 支持在图里加入错误处理节点。

# Agent 的错误处理机制
def safe_tool_call(tool_func, *args, max_retries=3):
    """带重试和降级的工具调用。"""
    for attempt in range(max_retries):
        try:
            return tool_func(*args)
        except TimeoutError:
            if attempt < max_retries - 1:
                time.sleep(2 ** attempt)  # 指数退避
                continue
            return {"error": "工具调用超时，请稍后重试"}
        except Exception as e:
            return {"error": f"工具调用失败：{str(e)}"}

def fallback_node(state: AgentState) -> dict:
    """错误降级节点：工具失败时的备选方案。"""
    return {
        "response": "抱歉，处理过程中遇到问题。"
                    "我已为您转接人工客服，请稍候。"
    }

# 在工作流里加入错误处理路径
workflow.add_node("fallback", fallback_node)
# 工具节点失败时路由到 fallback
workflow.add_conditional_edges("tools", check_tool_success, {
    "success": "agent",
    "failure": "fallback",
})

★ 七、Agent 的可观测性

Agent 的调试比普通应用难得多——因为它会自主决策，行为不可预测。必须做好可观测性：记录每一步的状态、工具调用、LLM 输出。LangGraph 的 checkpointer 能持久化每个节点的状态，支持回放和调试。

# Agent 的可观测性配置
from langgraph.checkpoint.memory import MemorySaver

# 启用状态持久化（生产环境用 PostgresSaver）
checkpointer = MemorySaver()
app = workflow.compile(checkpointer=checkpointer)

# 每次执行都能回溯完整的状态轨迹
config = {"configurable": {"thread_id": "session-123"}}
result = app.invoke(initial_state, config=config)

# 获取完整的状态历史
states = list(app.get_state_history(config))
for state in states:
    print(f"节点: {state.next}, 状态: {state.values}")
# 这让你能：
# 1. 调试：看每一步的状态变化，定位哪里出了问题
# 2. 回放：从任意节点重新执行
# 3. 分析：统计 Agent 的行为模式，优化流程
# 4. 审计：记录关键决策，满足合规要求

★ 八、常见 Agent 设计模式

  • ReAct 模式：推理 → 行动 → 观察的循环，最经典的 Agent 模式，适合工具调用密集的场景。
  • Plan-and-Execute：先规划完整步骤，再逐步执行，适合复杂任务，减少中途偏差。
  • 多 Agent 协作：多个专职 Agent 分工（如一个负责搜索、一个负责写作、一个负责审查），适合大型任务。
  • 人在回路：关键节点暂停等人确认，适合高风险场景。

深 LangGraph 状态图：复杂 Agent 的可控编排

复杂 Agent 的工作流不是线性的——有条件分支、循环、人在回路。LangGraph 用状态图来编排这类流程，比纯链式 Agent 可控得多。下面给出核心模式和避坑点。

条件分支与人在回路

下面用 LangGraph 实现一个带“质量检查 + 人工兜底”的 Agent：低置信度时转入人工审核节点，而非硬输出。

from typing import TypedDict, Literal
# 概念性伪代码(展示状态图思路)
class AgentState(TypedDict):
    query: str
    draft: str
    confidence: float
    approved: bool

def generate(state):           # 生成节点
    state["draft"] = llm_generate(state["query"])
    state["confidence"] = score_confidence(state["draft"])
    return state

def quality_gate(state) -> Literal["ok","review"]:   # 条件路由
    return "ok" if state["confidence"] > 0.8 else "review"

def human_review(state):       # 人在回路节点
    state["approved"] = ask_human(state["draft"])
    if not state["approved"]: state["draft"] = llm_revise(state["draft"])
    return state

def finalize(state): return {"answer": state["draft"]}

# graph: generate --(confidence>0.8?)--> finalize | review -> human -> finalize
# 状态图把"何时转人工"从硬编码变成可声明、可观测的路由规则
# 比在 prompt 里让模型自己决定"要不要找人"可控得多

状态图的核心价值是把控制流从 prompt 里拿出来——不再让模型自己决定“要不要循环、要不要转人工”，而是用代码声明式地定义路由规则，可测试、可观测、可回滚。

编排模式适用场景优势风险
线性链简单流程直观不能处理分支
状态图有分支/循环可控、可观测设计复杂

循环与防死循环

状态图能表达循环（如“生成→自检→不通过则重生成”），这是它比线性链强大的地方，但也引入死循环风险——模型可能无限“重生成”却一直不满足退出条件。防死循环的三道保险：第一，硬性迭代上限，无论质量如何，循环超过 N 次强制退出（转入人工或返回当前最优）；第二，退出条件要可量化，用具体阈值（如质量分 > 0.8）而非模糊判断（如“满意”），否则模型永远“不满意”；第三，每轮成本累加，超预算自动停。另一个要点是状态最小化：状态图的状态只存真正需要跨节点共享的数据，不要把所有中间结果都塞进状态——状态膨胀会导致每轮 prompt 变长、成本上升、且难以调试。可观测性对状态图尤其重要：每个节点的输入输出、路由决策都要记录，因为流程复杂，出问题时必须能重放看到“在哪一步、因什么条件、走了哪条边”。最后，复杂度要克制——能用线性链解决的别上状态图，能用 3 个节点解决的别搞 10 个，编排复杂度越高，维护和调试成本越高，收益要配得上复杂度。

深 Agent 编排 FAQ 与可控性

复杂 Agent 编排的核心是「可控」，这些问题最关键。

Q1：Agent 进入死循环（一直重试不退出），怎么办？

三道防死循环保险：硬性迭代上限（超 N 次强制退出，转人工或返回当前最优）；退出条件可量化（用具体阈值如质量分>0.8，而非模糊「满意」）；每轮成本累加（超预算停）。还要监控循环次数，异常多轮触发告警。死循环根因常是退出条件设太严或模型永远「不满意」，调松阈值或加「够好就停」的兜底。循环能用但必须可控——状态图的好处就是把路由规则从 prompt 拿到代码里，可设硬上限、可观测、可干预。

Q2：状态图什么时候用，什么时候过度设计？

有分支/循环/人在回路才用，否则线性链够。判断：流程是否条件分支？是否需要循环重试？是否有高风险动作要人工？都没有就别上状态图，线性链更简单。复杂度要克制——能用 3 节点解决别搞 10 个，编排越复杂维护调试越难。状态图的价值是把控制流从 prompt 拿出（可测试可观测），但简单流程用它反而增加负担。先从最简方案开始，证明不够再加复杂度，是工程美德。多数 Agent 任务用「线性链 + 几个条件判断」就够，真正需要复杂状态图的是少数。

★ 本章小结

这一章我们用 LangGraph 构建了完整的 Agent 系统。核心要点：

  • 状态图：LangGraph 用 StateGraph 管理 Agent 状态，节点是动作，边是流转，让 Agent 行为可视、可调试。
  • 条件分支：conditional_edges 让 Agent 根据中间结果动态路由，处理非线性业务流程。
  • 人在回路：高风险操作用 interrupt 暂停等人确认，Agent 不在所有事上都自主决策。
  • 工具调用：ToolNode + 条件路由实现 ReAct 循环（推理→行动→观察），让 Agent 连接真实世界。
  • 错误处理：重试、退避、降级，Agent 必须能优雅处理工具失败。
  • 可观测性：checkpointer 持久化状态历史，支持调试、回放、分析、审计。
