思维链 CoT 模式
让大模型学会"想一想再回答"

你问一个模型"17 乘以 24 是多少",它脱口而出"408"。错了,正确答案是 408——这次它蒙对了,但下次换个数可能就错。问题在于模型是"直觉式"回答,没有展开推理过程。思维链(Chain-of-Thought, CoT)的核心洞察是:让模型把推理步骤写出来,准确率会大幅提升。本章深入剖析 CoT 的三种形态,并给出可直接复用的 prompt 和代码。

阅读约 20 分钟 难度 ★★★

★ CoT 为什么有效:直觉与推理的差别

人类面对复杂问题时不直接给答案,而是"想一下":拆解、推理、验证,最后才说出来。这个思考过程对正确率至关重要。而大语言模型默认的生成方式是"next token prediction"——逐 token 生成,没有显式的"思考缓冲"。这导致模型在需要多步推理的数学、逻辑、规划任务上表现不稳定,因为每一步的小误差会在后续步骤中放大。

CoT(Chain-of-Thought)的洞察来自 2022 年 Google 的论文《Chain-of-Thought Prompting Elicits Reasoning in Large Language Models》。它发现,只要让模型在给答案前先输出推理步骤,复杂推理任务的准确率会显著提升。背后的原理可以从几个角度理解。

第一,推理过程本身就是计算。神经网络每生成一个 token 都做了一次前向计算,生成一串推理步骤相当于展开了一个"程序",每一步的计算结果被写进上下文,供下一步使用。这打破了单次前向的表达力限制,本质上是用"生成"做"计算"。

第二,训练数据里有推理样本。预训练语料里有大量带过程的解题、论证文本,模型见过"先分析再结论"的模式。CoT 提示唤醒了这种模式,让模型调用它在预训练中学到的推理能力。

第三,分解降低了任务难度。复杂问题一步做对很难,但拆成几步,每步都不难。CoT 把困难的整体任务分解为简单的子任务,每步的错误率低,整体准确率自然上升。

★ Zero-shot CoT:一句咒语的力量

CoT 最简单的形态是 zero-shot CoT,来自同一时期的论文《Large Language Models are Zero-Shot Reasoners》。它的做法极其简洁:在原始问题后加一句"Let's think step by step"(让我们一步一步思考),仅此而已。这句"咒语"在没有示例的情况下,就能让模型展开推理,在多个推理 benchmark 上提升两位数百分点。

这听起来像魔法,但原理清楚:模型在预训练中见过大量"step by step"开头的推理文本,这句话把模型引导到"推理模式",激活了它已具备的推理能力。这印证了 CoT 的本质是"释放"而非"赋予"推理能力——大模型本身就有推理潜力,只是默认没被激活。

下面是 zero-shot CoT 的对比示例。同一个数学问题,不加提示和加提示的效果天壤之别:

from anthropic import Anthropic
client = Anthropic()

question = "食堂里有 23 个苹果,用了 20 个做午餐,又买了 6 个,问现在有多少个苹果?"

# 普通 prompt:模型可能直接给错误答案
plain_prompt = f"{question}\n\n答案:"
# 典型输出: "29" (可能正确,但不稳定,复杂题易错)

# Zero-shot CoT:加一句引导
cot_prompt = f"{question}\n\nLet's think step by step."
# 典型输出:
# "1. 初始有 23 个苹果。
#  2. 做午餐用了 20 个,所以还剩 23 - 20 = 3 个。
#  3. 又买了 6 个,所以现在有 3 + 6 = 9 个。
#  4. 答案是 9。"

resp = client.messages.create(
    model="claude-3-5-sonnet-20241022",
    max_tokens=1024,
    messages=[{"role": "user", "content": cot_prompt}],
)
print(resp.content[0].text)

Zero-shot CoT 的优势是零成本、通用、无需构造示例。它的局限是效果不如 few-shot CoT 稳定,对某些需要特定推理范式的任务(如几何证明)引导不够具体。但作为"默认加在推理任务后的一句话",它的性价比无可匹敌——几乎没有任何理由不用。

★ Few-shot CoT:用示例定义推理范式

Few-shot CoT 通过在提示里给出几个"问题 + 完整推理过程 + 答案"的示例,让模型学会按你期望的推理范式解题。它的效果通常优于 zero-shot,尤其适用于有特定推理格式的领域(数学要用方程、法律要用法条)。

构造 few-shot 示例的关键是示范的推理过程要规范、清晰。模型会模仿示例的推理风格——如果示例跳步,模型也会跳步;如果示例严谨,模型也会严谨。下面是一个数学推理的 few-shot CoT 模板,注意推理过程的规范度:

Q: 小明有 15 元,买了 3 支铅笔,每支 2 元,又买了一本笔记本 6 元。他还剩多少钱?
A: 让我们一步步分析。
   第一步,计算铅笔的总花费:3 支 × 2 元/支 = 6 元。
   第二步,计算总花费:铅笔 6 元 + 笔记本 6 元 = 12 元。
   第三步,计算剩余:15 元 - 12 元 = 3 元。
   所以,小明还剩 3 元。

Q: 一辆车以 60 公里/小时的速度行驶,行驶了 2.5 小时后休息 30 分钟,
   然后以 80 公里/小时继续行驶 1.5 小时。总共行驶了多少公里?
A: 让我们一步步分析。
   第一阶段:60 × 2.5 = 150 公里。休息阶段不产生行驶距离。
   第二阶段:80 × 1.5 = 120 公里。
   总行驶距离:150 + 120 = 270 公里。
   所以,总共行驶了 270 公里。

Q: [你的实际问题]
A:

给模型看了这两个示例后,它会按"分阶段计算 + 汇总"的范式解决新的行程问题。Few-shot CoT 的示例数量通常 2-5 个足够,关键不是多而是精——每个示例都要展示一种典型的推理模式。示例选择上,应覆盖问题的不同难度和题型,让模型学会泛化而非死记。

一个进阶技巧是自洽(self-consistency):让模型对同一个问题用 CoT 生成多个不同推理路径的答案,取多数投票作为最终答案。因为每条推理路径的误差独立,投票能显著降低出错概率。我们下一节专门讲。

★ Self-Consistency:投票提升可靠性

CoT 有一个绕不开的问题:推理路径不唯一,模型可能选错路径。Self-Consistency(SC)是 Google 提出的改进方案,核心思想简单却有效:对同一问题采样多条不同的推理路径,看哪条结论出现最多,取多数投票结果。

为什么有效?假设对某道数学题,正确的推理路径有 60% 的概率被采样到,错误路径各占 10%、10%、20%。单次 CoT 准确率 60%。但采样 5 次后,正确答案获得多数票的概率会大幅提升(因为错误路径分散,正确路径集中)。这是"集成学习"思想在 LLM 推理上的应用。

实现 SC 的关键是用较高 temperature 采样(如 0.7-1.0),让模型生成不同的推理路径;如果 temperature 为 0,所有路径都一样,投票就失去意义。下面是完整实现:

import re
from collections import Counter

def self_consistent_answer(question, n_samples=5):
    """对同一问题采样多条 CoT,投票选最终答案"""
    cot_prompt = f"""{question}

Let's think step by step. 最后用 "The answer is X." 给出答案。"""

    answers = []
    for _ in range(n_samples):
        resp = client.messages.create(
            model="claude-3-5-sonnet-20241022",
            max_tokens=1024,
            temperature=0.8,            # 关键:高温度产生多样路径
            messages=[{"role": "user", "content": cot_prompt}],
        )
        text = resp.content[0].text
        # 抽取最终答案
        m = re.search(r"answer is\s*([^\.\n]+)", text, re.IGNORECASE)
        if m:
            answers.append(m.group(1).strip())

    # 多数投票
    counter = Counter(answers)
    best, count = counter.most_common(1)[0]
    print(f"投票分布: {dict(counter)}")
    return best

print(self_consistent_answer("一个数加上 7,再乘以 3,等于 36。这个数是多少?"))
# 投票分布: {'9': 4, '8': 1}
# 9

SC 的代价是推理成本翻倍:采样 N 次意味着 N 倍的 token 消耗。所以它适合对准确率要求高、对延迟和成本不敏感的场景(如数学竞赛、代码生成、关键决策)。日常对话用 zero-shot CoT 足矣。

★ CoT 的进阶变体

CoT 的成功催生了一系列变体,针对不同需求做优化。了解它们能让你在合适场景用对工具。

Zero-shot CoT 的中文变体:中文场景下"让我们一步一步思考"同样有效。也可以用"请仔细分析"或"请逐步推理"。关键是引导模型展开过程,而非直接给答案。

Tree of Thoughts(ToT):把推理组织成一棵搜索树,每个节点是一个推理状态,允许回溯和多路径探索。ToT 适合需要前瞻和试错的复杂任务(如填字游戏、24 点)。它的效果好但成本极高,因为要做树搜索,通常只在极难任务上才划算。

Graph of Thoughts(GoT):把推理扩展成图,允许不同推理分支合并。比 ToT 更灵活,适合需要综合多种思路的任务。

Least-to-Most Prompting:先把复杂问题分解为一系列由易到难的子问题,逐个解决。适合需要复杂规划的任务。

ReAct(Reason + Act):把推理和行动交替进行——推理一步,执行一个工具调用(搜索、计算),看到结果再推理下一步。这是 Agent 的核心范式,我们会在 Agent 章节深入。

选择建议:简单推理用 zero-shot CoT,需要稳定用 few-shot CoT,需要高准确率用 self-consistency,需要复杂规划用 ToT 或 ReAct。不要一上来就用最复杂的,先试最简单的能否达标。

★ 什么时候该用 CoT,什么时候不该

CoT 不是万能的,误用反而可能帮倒忙。判断标准是任务是否真的需要"多步推理"。

适合 CoT 的场景:数学计算、逻辑推理、多约束规划、因果分析、需要分解的复杂问答、代码生成中的算法设计。这些任务的特点是"一步想不清楚,需要拆解"。

不适合 CoT 的场景:事实性问答(首都、生日)、简单分类(情感分析)、创意写作(写诗)、翻译。这些任务直接给答案更好,加 CoT 反而可能引入错误(比如让它"推理"法国首都,它可能绕一圈还答错)。一个经验法则:如果人类做这个任务也需要一步步想,就用 CoT;如果人类是直觉反应,就不用。

CoT 可能带来风险的场景:涉及安全或合规的决策。CoT 会暴露模型的推理过程,如果推理过程包含偏见或不当内容,即使最终答案没问题,过程也可能引发问题。这类场景要审慎,或对推理过程做过滤。

★ 模型大小与 CoT 效果的关系

一个重要发现:CoT 对大模型有效,对小模型可能无效甚至有害。原始 CoT 论文显示,CoT 的收益在模型规模超过某个阈值(约 60B 参数)后才显著出现,小模型用 CoT 反而不如直接答。

原因不难理解:小模型的推理能力本来就弱,让它展开推理反而会暴露更多错误。这就像让一个数学不好的小学生"写出详细解题过程",过程错了,答案更糟。所以用 CoT 前,先确认你用的模型足够大——现代主流模型(7B 以上,尤其 70B+ 和 GPT-4 级别)都能从 CoT 获益。

这也解释了为什么 prompt 工程没有"放之四海皆准"的方案:同一个提示,在 7B 模型上有效,在 70B 上未必;在 GPT-4 上有效,在小模型上未必。永远要在你的目标模型上实测,别照搬论文结论。

CoT 的质量退化与稳定性治理

Chain-of-Thought（CoT）是让 LLM "想清楚再说"的核心技巧，但它远非万能——同样的 prompt，模型有时推理精彩、有时胡说八道，质量波动巨大。生产里用 CoT，稳定性治理比"会不会用"更重要。理解 CoT 质量退化的根源，才能针对性地稳住它。

CoT 退化的第一个根源是"推理漂移"。模型在长链条推理中，前几步还紧扣问题，越往后越容易跑偏——被中间某个无关细节带偏、或累积了小错误最终放大。表现是开头分析对、结论却错。缓解是"分段检查点"——让模型在推理中途自我核对（"目前得到的中间结论是否合理"），及时纠偏。第二个根源是"伪推理"。模型有时不是真在推理，而是生成"看起来像推理"的文本——先给结论再编造支持理由。表现是推理过程自洽但其实在合理化一个错误结论。缓解是"先推理后结论"的 prompt 结构（强制结论在最后），并核查推理是否真的支撑结论。

第三个根源是"分布敏感"。CoT 对 prompt 措辞极敏感——换个同义词、调下顺序，结果可能天差地别。这让 CoT 难以稳定复现。缓解是多次采样 + 投票——同一问题让模型推理多次（temperature>0），取多数结论。这牺牲成本换稳定性，对关键决策值得。第四个根源是"领域错觉"。模型在它不熟的领域强行 CoT，用通用套路套专业问题，得出看似合理实则错误的结果。缓解是注入领域知识（few-shot 示例、检索增强），别让模型在陌生领域裸跑 CoT。

退化类型表现根因治理手段
推理漂移开头对结尾错长链累积误差分段检查点
伪推理编理由支持结论结论先行强制结论在后
分布敏感换措辞结果变对 prompt 过拟合多次采样投票
领域错觉套路套错领域缺乏领域知识注入示例/RAG
过度推理简单问题复杂化不分场景按难度路由
# CoT 稳定性治理：分段检查 + 多次采样投票 + 难度路由
import statistics
class StableCoT:
    def __init__(self, model, samples=5):
        self.model = model
        self.samples = samples

    def solve(self, question, difficulty="auto"):
        # 难度路由：简单题不必 CoT，省成本
        if difficulty == "auto":
            difficulty = self._estimate_difficulty(question)
        if difficulty == "easy":
            return self.model.direct_answer(question)
        # 多次采样推理（temperature > 0 增多样性）
        candidates = []
        for _ in range(self.samples):
            reasoning = self.model.generate(
                self._checkpoint_prompt(question), temperature=0.7)
            if self._midpoint_check(reasoning):   # 中间结论自检
                candidates.append(self._extract_answer(reasoning))
        if not candidates:
            return self.model.direct_answer(question)
        return self._majority_vote(candidates)

    def _checkpoint_prompt(self, question):
        return f"""{question}
请一步步推理。在关键中间步骤后用【自检】标注：当前结论是否合理。
最后用【结论】给出答案。必须先完成全部推理，再写结论。"""

    def _majority_vote(self, answers):
        try:
            return statistics.median([float(a) for a in answers])
        except ValueError:
            return max(set(answers), key=answers.count)

    def _estimate_difficulty(self, q):
        if len(q) < 30 and not any(w in q for w in ["为什么","推导","证明"]):
            return "easy"
        return "hard"

CoT 的成本效益分析与场景路由

CoT 不是免费的——它显著增加 token 消耗（推理过程本身占大量输出 token）和延迟（要等推理完才有答案）。对所有问题都用 CoT 是浪费：简单问题用 CoT 既慢又贵还没收益。成本效益分析决定"哪些问题值得 CoT"，场景路由则把决策落地。

CoT 的收益与问题类型强相关。高收益场景：多步数学/逻辑推理、需要分解的复杂规划、因果分析、权衡决策——这些问题不分解根本做不对，CoT 的额外成本换来了正确率的大幅提升。低收益场景：事实检索（知道就知道，推理没用）、简单分类、格式转换、单步计算——这些问题直答即可，CoT 反而增加出错面（推理过程可能引入错误）。负收益场景：模型不熟的领域强行 CoT（用通用套路套专业问题）、极简单问题复杂化（过度思考）。

场景路由的实现可以是规则 + 模型的混合。规则层：基于问题特征（长度、是否含推理关键词、领域）做粗筛。模型层：用一个轻量分类器判断"这题需不需要 CoT"。还可以动态切换——先用直答试，置信度低再升级到 CoT；或先 CoT 试，发现是简单题中途切换到直答。这种"按需升级"的策略在成本和质量间取得最佳平衡。

问题类型CoT 收益典型场景推荐策略
多步推理高数学/逻辑/规划必用 CoT
因果分析高诊断/归因用 CoT
权衡决策中高方案选择用 CoT + 多视角
事实检索低查信息直答 + RAG
简单分类低情感/意图直答
陌生领域负专业问题注入知识再 CoT
# CoT 场景路由：按需升级，平衡成本与质量
class CoTRouter:
    def __init__(self, model, cot_model, classifier):
        self.model = model          # 快速直答模型
        self.cot = cot_model        # CoT 推理模型
        self.clf = classifier

    def answer(self, question):
        if self._trivial(question):
            return self.model.direct_answer(question), "trivial_direct"
        direct = self.model.direct_answer(question)
        confidence = self.model.confidence(direct)
        if confidence >= 0.9:
            return direct, "confident_direct"
        need_cot = self.clf.needs_cot(question)
        if need_cot or confidence < 0.5:
            return self.cot.solve(question), "upgraded_cot"
        return direct, "medium_direct"

    def _trivial(self, q):
        return (len(q) < 20 and not any(w in q for w in ["为什么","如何","推导","比较"]))
# 统计：medium_direct 正确率最低 → 该类应升级 CoT

★ 本章小结

CoT 的核心洞察是:让模型写出推理过程,而非直接给答案,能显著提升复杂推理任务的准确率。它的原理在于"生成即计算"、"激活预训练中的推理模式"、"分解降低难度"。

三种形态各有定位:Zero-shot CoT 加一句"Let's think step by step",零成本,是推理任务的默认选项;Few-shot CoT 用规范示例定义推理范式,效果更稳;Self-Consistency 采样多条路径投票,适合高准确率需求但成本翻倍。进阶变体如 ToT、ReAct 适合复杂规划和 Agent 场景。

使用判断:需要多步推理的任务用 CoT,直觉性任务不用;大模型受益于 CoT,小模型未必;安全敏感场景要审视推理过程。记住 CoT 是工具不是目的——目标是把任务做对,CoT 只是达到目标的一条路径。在简单方法够用时,不要为了用 CoT 而用 CoT。
