乘积量化 PQ 与 ADC
对称距离表、非对称距离表与压缩比推导

乘积量化（Product Quantization, PQ）是向量检索里把"省内存"做到极致的经典技术。它能把一个 768 维 float32 的向量从 3072 字节压到 16-48 字节，压缩比高达 64:1 到 192:1，还能直接在压缩表示上算距离。本文拆解 PQ 的编码原理、对称距离表 SDT 与非对称距离表 ADC 的区别，并手推那个让人困惑的压缩比公式。

量化（quantization）的本质是把一个连续空间离散化：把无穷多个可能的向量映射到有限个"代表向量"（码字）上，存储时只存代表向量的编号。最朴素的量化是 K-means：把整个 768 维空间聚成 K 个簇，存簇心编号。问题在于 K 要大到覆盖所有细节，比如 K=2^16=65536 时，每个码字仍是 768 维 float32，光码本就要 65536×768×4 = 192MB，且 65536 个簇心对 768 维空间只是杯水车薪，量化误差巨大。

PQ 的巧思是分而治之：把 d 维向量切成 m 段，每段 d/m 维，对每段独立做 K-means 量化。假设每段聚成 256 个簇心（用 1 字节即 8 比特编码，记作 nbits=8），那么一个向量编码成 m 字节，每段一个簇心编号。整个码本是 m 个独立的子码本，每个子码本 256 个簇心。

关键的数学魔法在于组合爆炸：m=8、每段 256 个簇心，看起来只有 8×256=2048 个子簇心，但它们能组合出 256^8 = 2^64 ≈ 1.8×10^19 个不同的"等效全向量"。也就是说，PQ 用 2048 个子码字，等价表达了 10^19 量级的码本——这正是它能兼顾低存储和高表达力的原因。

压缩比推导

原始向量：d 维 float32 = 4d 字节。PQ 编码后：m 段，每段 nbits 比特 = m·nbits/8 字节。压缩比 = 4d / (m·nbits/8) = 32d / (m·nbits)。

以 d=768、m=64、nbits=8 为例：压缩比 = 32×768/(64×8) = 48。原始 3072 字节 → 64 字节。如果 m=192（每段只 4 维），压缩比 = 32×768/(192×8) = 16，编码后 192 字节，精度更高。常见配置 m=16/d=768（每段 48 维，编码 16 字节，压缩比 192:1）到 m=64（编码 64 字节，压缩比 48:1）之间。

# 手写一个 PQ 编码器，看清每一步
import numpy as np
from sklearn.cluster import KMeans

class ProductQuantizer:
    def __init__(self, m, nbits=8):
        self.m = m
        self.k = 2 ** nbits          # 每段子码本的簇心数
        self.codebooks = []          # m 个子码本，每个 (k, d/m)

    def train(self, X):
        d = X.shape[1]
        assert d % self.m == 0
        sub_d = d // self.m
        for i in range(self.m):
            sub_X = X[:, i*sub_d:(i+1)*sub_d]
            km = KMeans(n_clusters=self.k, n_init=3, random_state=0)
            km.fit(sub_X)
            self.codebooks.append(km.cluster_centers_)     # (k, sub_d)
        return self

    def encode(self, X):
        """X: (N, d) → codes: (N, m) 每个元素是 0..k-1 的簇心编号"""
        d = X.shape[1]; sub_d = d // self.m
        codes = np.zeros((X.shape[0], self.m), dtype=np.uint8)
        for i in range(self.m):
            sub_X = X[:, i*sub_d:(i+1)*sub_d]
            codes[:, i] = np.argmin(
                np.linalg.norm(sub_X[:, None] - self.codebooks[i], axis=2), axis=1)
        return codes

# 验证压缩比
d = 768
pq = ProductQuantizer(m=16, nbits=8)
X = np.random.randn(10000, d).astype(np.float32)
pq.train(X)
codes = pq.encode(X)
print(f"原始每向量: {d*4} 字节")
print(f"PQ 编码每向量: {codes.shape[1]} 字节")
print(f"压缩比: {d*4 / codes.shape[1]:.0f}:1")

★ SDT vs ADC：两种距离计算方式

PQ 把向量压成编码后，怎么在编码上算距离？论文给出两种方案，理解它们的区别是掌握 PQ 的关键。

对称距离表 Symmetric Distance Table (SDT)

查询向量 q 也先编码成 PQ 码 q'，然后距离 d(x, q) 近似为 d(x', q')——两个码字之间的距离。因为码字组合有限（每个子段 256 个），可以预计算一个 m × 256 × 256 的距离表 table[i][a][b]，表示第 i 段、码字 a 和 b 之间的距离。查询时查表 m 次求和即可，完全不用解码。

SDT 的优点是查询编码也可缓存复用（同 q 查多次只编码一次），缺点是查询也被量化了，引入了双重量化误差（x 和 q 都量化），精度更低。

非对称距离表 Asymmetric Distance Computation (ADC)

ADC 不编码查询 q，保留 q 的原始浮点形式，距离近似为 d(x', q)——码字到原始查询的距离。预计算一个 m × 256 的表 table[i][a]，表示第 i 段、码字 a 到 q 的第 i 段的距离。这个表是每次查询动态算的（依赖 q），算好后查表 m 次求和。

ADC 只有一次量化误差（只量化了 x），精度比 SDT 高，是实际系统里几乎都采用的方式。代价是每次查询要花 m×k 次子向量距离计算来建表（对 768 维、m=16、k=256，约 16×256=4096 次 48 维点积，微秒级，可忽略）。

# ADC 距离计算 —— 实战中的标准做法
def adc_distance(pq, codes_x, q):
    """
    codes_x: (N, m) 数据库向量的 PQ 编码
    q:       (d,) 查询原始向量（不编码）
    return:  (N,) 每个数据库向量到 q 的近似距离
    """
    d = q.shape[0]; sub_d = d // pq.m
    # 1. 为本次查询预计算查表: table[i][a] = 第i段码字a 到 q第i段 的距离平方
    table = np.zeros((pq.m, pq.k), dtype=np.float32)
    for i in range(pq.m):
        sub_q = q[i*sub_d:(i+1)*sub_d]                       # (sub_d,)
        # codebooks[i]: (k, sub_d)
        table[i] = np.sum((pq.codebooks[i] - sub_q) ** 2, axis=1)   # (k,)

    # 2. 查表求和得到每个 x 的距离
    N = codes_x.shape[0]
    dists = np.zeros(N, dtype=np.float32)
    for i in range(pq.m):
        dists += table[i][codes_x[:, i]]     # 高级索引，向量化查表
    return dists

# 对比 PQ 近似距离 vs 真实距离
q = np.random.randn(d).astype(np.float32)
true_dists = np.linalg.norm(X - q, axis=1) ** 2
pq_dists = adc_distance(pq, codes, q)
mae = np.mean(np.abs(true_dists - pq_dists))
print(f"PQ ADC 距离平均绝对误差: {mae:.4f}")
print(f"真实距离量级: {np.mean(true_dists):.4f}")
# 误差通常在 10-20% 左右，这就是 PQ 召回略低于 Flat 的根源

★ IVF + PQ：工业级的标准组合

单独的 PQ 要对全库 N 个编码查表，复杂度仍是 O(N·m)，没省多少计算。实战中 PQ 几乎总是和 IVF 组合成 IVFPQ：先用 IVF 把搜索范围从 N 剪到 nprobe 个倒排列表（比如 1 亿 → 1 万），再在这 1 万个编码上做 ADC 查表。这样计算量从 N·m 降到 (N/nlist)·nprobe·m，提速上千倍。

IVFPQ 的标准调参流程：先按 IVF 的方法定 nlist 和 nprobe，再定 PQ 的 m 和 nbits。一个重要技巧是重排序（rerank）：IVFPQ 召回 top-200（用 ADC 近似距离），再对这 200 个用原始向量算精确距离取 top-10。因为重排只对 200 个算，开销小，但能把召回从 0.92 拉到 0.98。

# FAISS IVFPQ 完整流程 + 重排序
import faiss, numpy as np, time

d = 768; N = 1_000_000
xb = np.random.randn(N, d).astype(np.float32); faiss.normalize_L2(xb)
xq = np.random.randn(1000, d).astype(np.float32); faiss.normalize_L2(xq)

# 1. 构建 IVFPQ
quantizer = faiss.IndexFlatIP(d)
nlist, m, nbits = 4096, 64, 8
index = faiss.IndexIVFPQ(quantizer, d, nlist, m, nbits, faiss.METRIC_INNER_PRODUCT)
index.train(xb[:50000]); index.add(xb)
index.nprobe = 32

# 2. 直接用 PQ 距离召回
t0 = time.time()
D, I = index.search(xq, 10)
print(f"IVFPQ 直接召回: {1000/(time.time()-t0):.0f} QPS")

# 3. 召回更多 + 原始向量重排序
index.nprobe = 16                                   # 重排前可适当降 nprobe
D_raw, I_raw = index.search(xq, 200)               # 召回 200
# 用原始向量对 200 重排（这里用 Flat 的精确距离）
def rerank(I_candidates, k=10):
    out = []
    for i, cand in enumerate(I_candidates):
        cand = cand[cand >= 0]
        sims = xb[cand] @ xq[i]                    # 精确内积
        out.append(cand[np.argpartition(-sims, k)[:k]])
    return np.array(out)
I_reranked = rerank(I_raw)
# 对比 ground truth
flat = faiss.IndexFlatIP(d); flat.add(xb)
_, gt = flat.search(xq, 10)
recall_raw = sum(len(set(I[i]) & set(gt[i])) for i in range(1000)) / 10000
recall_rr  = sum(len(set(I_reranked[i]) & set(gt[i])) for i in range(1000)) / 10000
print(f"无重排 recall@10: {recall_raw:.4f}")
print(f"重排后 recall@10: {recall_rr:.4f}")

★ 进阶：OPQ 旋转预处理

PQ 的一个隐含假设是"各段子向量独立"，但真实 embedding 的各维度往往强相关，直接切片会让某些段方差大、某些段方差小，方差大的段量化误差主导整体。优化乘积量化 OPQ 在 PQ 前加一个学习的正交旋转矩阵，把向量旋转到各段方差均衡的子空间，能再压低 20-40% 的量化误差。FAISS 里用 RQ 或 OPQ matrix 包装即可启用。

★ PQ 训练的工程细节与坑

PQ 的训练（K-means 聚码本）有几个工程细节直接决定最终召回质量，论文里往往一笔带过，实战却很关键。

第一个是训练样本量。每段 K-means 要聚 256 个簇心（nbits=8），每个簇心至少需要 30-50 个样本才能统计稳定，所以每段至少要 256×40 ≈ 1 万样本，m 段独立训练但可共用同批样本，所以总训练样本至少 1-2 万。样本太少会出现"空簇"（某些簇心没有样本分配），导致那段的量化误差爆炸。FAISS 默认要求训练样本数 ≥ nlist×39（IVFPQ 场景），不够会报警。

第二个是各段方差的均衡。PQ 假设各段独立同分布，但真实 embedding 的各维度方差往往差异巨大——某些主成分方向方差大、其它方向方差小。方差小的段量化误差小（本来信息就少），方差大的段量化误差大（信息多但比特不够）。这导致量化误差被方差大的段主导。OPQ（优化乘积量化）通过学习一个正交旋转矩阵把方差均衡到各段，能再压 20-40% 误差。

第三个是码本的复用与刷新。线上 embedding 分布会随业务漂移（新内容、新话题），用旧码本量化新分布会引入额外误差。建议定期（如每月）用最新数据重训码本。重训成本不高（几万样本 K-meas 几分钟），但能保持量化质量。

# 用 OPQ 包装 IVFPQ，再压量化误差
import faiss
# OPQ 旋转矩阵 + IVFPQ
index_opq = faiss.IndexPreTransform(
    faiss.OPQMatrix(d, m),   # 学习的正交旋转
    faiss.IndexIVFPQ(faiss.IndexFlatL2(d), d, nlist, m, nbits)
)
index_opq.train(xb[:50000])
index_opq.add(xb)
# 对比召回：OPQ 通常比裸 IVFPQ 高 3-8 个点

第四个坑是归一化与距离度量的配合。PQ 默认优化 L2 距离，算内积/余弦要先归一化。但如果归一化后某段方差极小（因为归一化压缩了模长），量化误差会异常。常见做法是归一化后用 L2，配合 IP metric 的 quantizer，效果稳定。

★ PQ 在实际系统里的应用模式

了解了原理，看 PQ 在真实系统里怎么用。最常见的三种模式：

模式一：IVFPQ 作为大库的主力索引。亿级向量 + 内存预算 100GB 以内的场景，IVFPQ 几乎是唯一选择。典型配置 nlist=16384、m=32、nbits=8，配合重排序，能跑出 recall@10=0.93、QPS 几千的成绩。这是大规模推荐系统、搜索引擎召回层的标配。

模式二：HNSW+PQ 作为内存优化的 HNSW。HNSW 速度无敌但内存重，把图里的向量换成 PQ 编码（FAISS 的 IndexHNSWPQ），内存省 8-32 倍，QPS 掉 20-40%，召回掉 2-5 个点。适合"HNSW 太贵但又想要 HNSW 的查询模式"的场景。

模式三：PQ 作为多阶段检索的粗排器。前面 122 篇讲的多阶段架构里，IVFPQ 常做第一段粗筛，top-N 召回后用原始向量重排。这种组合兼顾了 PQ 的内存优势和原始向量的精度。

# HNSW + PQ 组合：内存优化版 HNSW
index_hnsw_pq = faiss.IndexHNSWPQ(d, m=16, M=16)
index_hnsw_pq.hnsw.efConstruction = 200
index_hnsw_pq.train(xb[:50000])
index_hnsw_pq.add(xb)
# 内存：约 HNSWFlat 的 1/8，QPS 掉约 30%，召回掉 3-5 点
# 适合"HNSW 太贵但纯 IVFPQ 又太慢"的中间地带

这三种模式覆盖了 90% 的 PQ 用法。选哪种取决于你的瓶颈是内存、延迟还是召回，按瓶颈对号入座。

★ PQ 的常见疑问与误区澄清

PQ 是个有点反直觉的方法，学习时常见的几个疑问这里集中澄清。

疑问一：PQ 为什么不直接对全向量做 K-meas？全向量 K-means 聚 K 个簇心，要存 K×d 个浮点。K=65536、d=768 时码本 192MB，且 65536 个簇心对 768 维空间远远不够（维度灾难）。PQ 分段后，m×k 个子簇心能组合出 k^m 个等效全向量，用极少存储表达极多码字——这是 PQ 的核心数学魔法。

疑问二：PQ 的距离是精确的还是近似的？近似的。PQ 用码字之间的距离近似原向量距离，有量化误差。误差大小取决于码本训练质量、m 和 nbits。ADC 的误差通常让 recall 掉 5-10 个点（相比 Flat），重排序能补回来。

疑问三：PQ 适合所有距离度量吗？默认优化 L2。算内积/余弦要先归一化。最大内积搜索（MIPS）不归一化时，PQ 的 L2 优化不直接适用——要用支持 IP 的量化（如 FAISS 的 IndexIVFPQ 配 METRIC_INNER_PRODUCT，内部做了适配）。

疑问四：m 越大越好吗？不是。m 大意味着每段维度小、量化细，但存储增加、查表次数增加。且 m 太大时每段维度太小（如 d/m=4），K-means 聚 256 个簇心在 4 维空间不稳定。经验：每段 8-64 维为宜，即 m ∈ [d/64, d/8]。

# 验证：m 对精度和存储的影响
for m in [4, 8, 16, 32, 64, 96]:
    sub_d = d // m
    idx = faiss.IndexIVFPQ(faiss.IndexFlatL2(d), d, 4096, m, 8)
    idx.train(xb[:50000]); idx.add(xb); idx.nprobe=32
    D, I = idx.search(xq, 10)
    recall = sum(len(set(I[i])&set(gt[i])) for i in range(len(xq)))/(len(xq)*10)
    storage = N * m * 1 / 1e6   # m 字节/向量
    print(f"m={m:3d} sub_d={sub_d:3d}: recall={recall:.3f} storage={storage:.0f}MB")
# m=16-32 通常是甜区，再大存储涨但 recall 提升小

疑问五：PQ 能用于实时插入吗？能，但要小心。新向量用已有码本编码即可（K-means 码本不变）。问题是分布漂移——新数据的分布偏离训练码本时的分布，量化误差增大。定期重训码本能缓解。

这些疑问澄清后，对 PQ 的理解就从"会用 API"上升到"知道每个参数的意义和边界"，调参时心里有数。

★ 标量量化的补充：从 PQ 到 SQ8

PQ 之外还有标量量化（SQ），它和 PQ 互补，常组合使用。

标量量化（Scalar Quantization, SQ）。把每个 float32 维度独立量化到 int8（SQ8）或 int4（SQ4）。SQ8：每维用 8 比特表示 [-max, max] 的均匀分桶。存储 768 维从 3KB 降到 768 字节（4:1 压缩），精度损失小（recall 掉 1-3%）。比 PQ 简单（不用训练码本）、精度更高，但压缩比不如 PQ（PQ 能 16:1+）。

SQ vs PQ 的选择。SQ：压缩比中等（4-8:1）、精度高、无需训练、实现简单。适合"想省存储但不想牺牲太多精度"。PQ：压缩比高（16-32:1）、精度中等、要训练。适合"存储极敏感、可接受精度损失"。很多场景 SQ8 就够了，不必上 PQ。

# FAISS 里 SQ 和 PQ 的对比
def compare_sq_pq(xb, xq, gt):
    # SQ8
    sq = faiss.IndexScalarQuantizer(d, faiss.ScalarQuantizer.QT_8bit, faiss.METRIC_L2)
    sq.add(xb)
    D, I = sq.search(xq, 10); r_sq = recall(I, gt)
    # PQ (m=32, nbits=8)
    pq = faiss.IndexPQ(d, 32, 8); pq.train(xb[:50000]); pq.add(xb)
    D, I = pq.search(xq, 10); r_pq = recall(I, gt)
    # 存储对比
    print(f"SQ8: recall={r_sq:.3f} 存储={N*768/1e6:.0f}MB")
    print(f"PQ:  recall={r_pq:.3f} 存储={N*32/1e6:.0f}MB")
    # SQ8 recall 通常高 3-5%，PQ 存储省 8 倍

SQ + PQ 组合（IVFPQ + SQ）。FAISS 支持先 SQ 再 PQ：SQ8 降精度到 int8，再对 int8 向量做 PQ。这能进一步压缩（SQ8 的 768 字节再 PQ 到 32 字节），且 PQ 在 int8 上训练更稳。适合"既要高压缩又要尽量保精度"。

FP16 半精度。最简单的"量化"——float32 直接转 float16。存储减半、精度几乎无损（recall 掉 <0.5%）、GPU 原生支持。很多向量库默认就支持 FP16。如果只想要 2:1 压缩且无损，FP16 是首选，不必上 SQ/PQ。

量化选择决策：无损省一半 → FP16；省 4-8 倍、精度可接受 → SQ8；省 16+ 倍、可接受明显精度损失 → PQ；极致压缩 → SQ8+PQ 组合。按存储预算和精度容忍度选，别一上来就 PQ——很多人其实 SQ8 或 FP16 就够了。

★ 本章小结

  • PQ 把 d 维向量切成 m 段独立 K-means 量化，每段 nbits 比特，存储 m·nbits/8 字节，压缩比 = 32d/(m·nbits)。768 维 m=16 时约 192:1。
  • 组合魔法：m 个子码本能表达 k^m 个等效全向量，m=8/k=256 即 2^64 个，这是 PQ 高表达力的根源。
  • SDT（对称）编码查询、双重量化误差、精度低；ADC（非对称）不编码查询、单次量化误差、精度高，是实战默认。
  • ADC 流程：每次查询先算 m×k 的子段距离表，再对 N 个编码查表求和，复杂度 O(N·m)。
  • 工业标准组合是 IVFPQ：IVF 剪枝到小范围，PQ 在小范围做 ADC；务必配合重排序（召回 top-200 → 原始向量精确重排 top-10）补足量化误差。
  • OPQ 加正交旋转预处理能再压 20-40% 量化误差，是追求 SOTA 的标配。

import numpy as np
from dataclasses import dataclass
from typing import List, Optional

@dataclass
class PerformanceMetrics:
    latency_ms: float
    throughput_ops: float
    memory_mb: float
    accuracy: float

def benchmark(fn, iterations=100):
    """通用性能基准测试"""
    import time
    t0 = time.time()
    for _ in range(iterations):
        result = fn()
    elapsed = time.time() - t0
    return PerformanceMetrics(
        latency_ms=elapsed * 1000 / iterations,
        throughput_ops=iterations / elapsed,
        memory_mb=0,  # 实际中用 psutil 获取
        accuracy=0.0  # 根据具体任务计算
    )

# 示例：对比两种方案的性能
def approach_a(data):
    """方案A：直接计算"""
    return np.softmax(data @ data.T / np.sqrt(data.shape[-1]), axis=-1)

def approach_b(data):
    """方案B：分块计算（省内存）"""
    chunk_size = 64
    results = []
    for i in range(0, data.shape[0], chunk_size):
        chunk = data[i:i+chunk_size]
        results.append(np.softmax(chunk @ data.T / np.sqrt(data.shape[-1]), axis=-1))
    return np.vstack(results)

data = np.random.randn(512, 768)
metrics_a = benchmark(lambda: approach_a(data))
metrics_b = benchmark(lambda: approach_b(data))
print(f"方案A: {metrics_a.latency_ms:.1f}ms, {metrics_a.throughput_ops:.0f} ops/s")
print(f"方案B: {metrics_b.latency_ms:.1f}ms, {metrics_b.throughput_ops:.0f} ops/s")

常见问题FAQ

Q: 如何选择合适的方案？ 先用方案A跑通验证正确性，再根据实际瓶颈（内存/延迟/精度）切换到方案B或C。永远不要过早优化。

Q: 生产环境最大的坑是什么？ 过度工程化——在需求不明确时投入大量精力做复杂架构。建议MVP先行，收集反馈后迭代。

Q: 如何评估效果？ 定义明确的成功指标（准确率/延迟/成本），用golden set做回归测试，A/B测试验证改进效果。至少100个样本才有统计显著性。
