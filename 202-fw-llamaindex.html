LlamaIndex：从 RAG 到 Agent

RAG 框架的王者——LlamaIndex 的架构与使用

深度进阶★★★

LlamaIndex：RAG 领域的第一框架

如果说 LangChain 是"通用 Agent 框架"，LlamaIndex 就是"RAG 专精框架"——它的核心能力是把各种数据源（PDF/数据库/API/网页）高效地索引、检索、喂给 LLM。如果你的应用核心是 RAG，LlamaIndex 比 LangChain 更顺手。~50K GitHub stars。

LlamaIndex 的核心抽象

概念是什么类比
Document一个文档（PDF/网页/文本）一本书
Node文档切块后的最小单元一个段落
Index索引结构（向量/列表/树/图）图书馆的目录
Retriever从索引里检索图书管理员
QueryEngine检索+LLM生成问答服务
Agent能调工具的自主循环研究员

从 RAG 到 Agent：LlamaIndex 的进化

LlamaIndex 最初是纯 RAG 框架（文档→索引→检索→问答）。2024 年后它进化成了"Agentic Document Workflows"——不只检索，还能让 Agent 自主决定查什么、查几次、怎么综合。这和 Agentic RAG（第207章）的理念一致。

# LlamaIndex 最简 RAG
from llama_index.core import VectorStoreIndex, SimpleDirectoryReader

# 1. 加载文档
documents = SimpleDirectoryReader("./data").load_data()

# 2. 建索引（自动切块+嵌入+存储）
index = VectorStoreIndex.from_documents(documents)

# 3. 查询（自动检索+生成）
query_engine = index.as_query_engine()
response = query_engine.query("这本书讲了什么？")
print(response)

# 进阶：用 Agent 替代 QueryEngine（能多步检索）
from llama_index.core.agent import ReActAgent
agent = ReActAgent.from_tools([query_tool], llm=llm)
response = agent.chat("对比第三章和第五章的核心观点")

LlamaIndex vs LangChain 怎么选

场景推荐理由
纯 RAG（文档问答）LlamaIndexRAG 抽象更完善
复杂 Agent（多工具/多步）LangGraph状态图更灵活
快速原型LlamaIndex5行代码跑通RAG
生产级可控LangGraph错误处理/可观测更好
两者结合LlamaIndex 做检索 + LangGraph 做 Agent各取所长
💡 要点
最佳实践：**LlamaIndex 做"数据层"（索引/检索），LangGraph 做"控制层"（Agent循环/工具调度）**。这是当前生产级 RAG-Agent 系统的主流架构——LlamaIndex 的 Document/Node/Index 抽象处理数据最好，LangGraph 的 StateGraph 处理复杂流程最好。

索引构建：从原始文档到可检索的向量库

LlamaIndex 最简 RAG 只需三行代码，但生产环境的索引构建远比这复杂。真实的索引构建是一条完整管道：文档加载 → 清洗 → 切块 → 嵌入 → 存储。每个环节都有坑：PDF 要处理表格和图片、切块策略影响召回率、嵌入模型选型决定语义理解上限。下面拆解这条管道的每一步。

# LlamaIndex 生产级索引构建管道
from llama_index.core import (
    SimpleDirectoryReader, VectorStoreIndex,
    StorageContext, Settings
)
from llama_index.core.node_parser import SemanticSplitterNodeParser
from llama_index.core.schema import MetadataMode
from llama_index.embeddings.huggingface import HuggingFaceEmbedding
from llama_index.vector_stores.qdrant import QdrantVectorStore

# 1. 配置全局设置（嵌入模型 + LLM）
Settings.embed_model = HuggingFaceEmbedding(
    model_name="BAAI/bge-large-zh-v1.5"  # 中文场景首选
)
Settings.llm = None  # 建索引时不需要 LLM，省成本

# 2. 文档加载：支持 PDF/网页/Notion/API 等 20+ 数据源
documents = SimpleDirectoryReader(
    input_dir="./knowledge_base",
    required_exts=[".pdf", ".md", ".docx", ".txt"],
    recursive=True,  # 递归读子目录
).load_data()

# 给每个文档打元数据标签（后续可按标签过滤检索）
for doc in documents:
    doc.metadata.update({
        "source": "internal_docs",
        "department": classify_department(doc.text),  # 自动分类
        "updated_at": doc.metadata.get("last_modified"),
    })

# 3. 语义切块：按语义边界切，而非固定字数（召回率更高）
splitter = SemanticSplitterNodeParser(
    embed_model=Settings.embed_model,
    breakpoint_percentile_threshold=95,  # 语义断层阈值
)
nodes = splitter.get_nodes_from_documents(documents, chunk_size=512)

print(f"加载 {len(documents)} 个文档，切成 {len(nodes)} 个节点")

# 4. 嵌入 + 存储：用 Qdrant 做向量库（生产级）
vector_store = QdrantVectorStore(
    client=qdrant_client,
    collection_name="company_kb",
)
storage_context = StorageContext.from_defaults(vector_store=vector_store)

# 5. 建索引（自动嵌入所有节点并存入向量库）
index = VectorStoreIndex(
    nodes=nodes,
    storage_context=storage_context,
    show_progress=True,
)
# 持久化（下次直接 load，不用重建）
index.storage_context.persist(persist_dir="./index_storage")

这条管道的几个关键决策点：① 切块用语义切而非固定大小——固定 512 字符可能把一个完整论点切成两半，语义切能保持片段的完整性，召回率和答案质量都更好；② 嵌入模型要匹配语言——中文用 bge-large-zh，英文用 text-embedding-3，混用会导致语义匹配失准；③ 元数据要打好——后续检索可以按部门、时间、来源过滤，大幅提升精准度（"只查 2024 年财务部的文档"）。

Query Engine 配置：让检索又准又全

建好索引只是第一步，怎么查同样重要。LlamaIndex 的 QueryEngine 提供了大量可配置项：检索 top-k 数量、是否加 reranker 精排、相似度阈值、是否做查询改写、返回内容窗口大小。默认配置往往不是最优的，需要根据任务调参。下面是一个针对"复杂问答"优化的 QueryEngine 配置。

# 生产级 QueryEngine：检索 + 精排 + 查询改写
from llama_index.core.query_engine import RetrieverQueryEngine
from llama_index.core.retrievers import VectorIndexRetriever
from llama_index.core.postprocessor import (
    SentenceTransformerRerank, MetadataReplacementPostProcessor
)
from llama_index.core.response_synthesizers import ResponseSynthesizer

# 1. 检索器：召回 top-20（先广撒网）
retriever = VectorIndexRetriever(
    index=index,
    similarity_top_k=20,
    filters=MetadataFilters(exact_filters=[  # 元数据过滤
        ExactMatchFilter(key="department", value="财务部"),
    ]),
)

# 2. Reranker：用 cross-encoder 精排到 top-5（再精选）
reranker = SentenceTransformerRerank(
    model="BAAI/bge-reranker-large",  # 中文精排模型
    top_n=5,
)

# 3. 上下文扩展：检索到片段后，取其前后窗口（保留完整上下文）
postprocessor = MetadataReplacementPostProcessor(
    target_metadata_key="window"  # 用相邻节点扩展上下文
)

# 4. 组装 QueryEngine
query_engine = RetrieverQueryEngine(
    retriever=retriever,
    node_postprocessors=[reranker, postprocessor],
    response_synthesizer=ResponseSynthesizer.from_args(
        response_mode="compact",  # 紧凑模式：合并片段再喂 LLM
        streaming=True,            # 流式输出
    ),
)

# 测试检索质量
response = query_engine.query("2024 年 Q3 的营收增长主要来自哪些业务？")
for src in response.source_nodes:  # 每个答案都带来源溯源
    print(f"[{src.score:.2f}] {src.node.metadata['file_name']}")
print(response.response)

这个配置体现了一个重要的 RAG 优化理念：两阶段检索（召回 + 精排）。第一阶段用向量检索快速召回 top-20（recall 优先，宁多勿漏），第二阶段用 cross-encoder reranker 精排到 top-5（precision 优先，只留最相关的）。为什么要两阶段？因为向量检索（双塔模型）快但不够精确，cross-encoder 精确但慢——两者结合既快又准。这是生产级 RAG 的标配。

Data Ingestion 管道：让知识库自动更新

知识库不是建一次就完事——文档会更新、新增、删除。手动重建索引既低效又容易遗漏。LlamaIndex 的 IngestionPipeline 解决这个问题：它把"加载→处理→嵌入→存储"封装成可定时执行的管道，配合增量更新机制，只处理变化的文档，省时省算力。这是知识库"持续保鲜"的关键。

# Data Ingestion 管道：增量更新 + 定时执行
from llama_index.core.ingestion import IngestionPipeline, docstore  # 简化示例
from llama_index.core.extractors import TitleExtractor, KeywordExtractor
import hashlib

class KnowledgeBaseIngester:
    """知识库自动更新管道：只处理变化的文档。"""
    def __init__(self, vector_store, doc_store_path="./docstore"):
        self.pipeline = IngestionPipeline(
            transformations=[
                # 切块
                SemanticSplitterNodeParser(...),
                # 抽取元数据（用 LLM 自动生成标题和关键词）
                TitleExtractor(nodes=5),
                KeywordExtractor(keywords=5),
                # 嵌入
                Settings.embed_model,
            ],
            vector_store=vector_store,
            docstore=doc_store,  # 记录已处理文档的哈希
        )

    def _file_hash(self, path):
        """计算文件内容哈希，用于判断是否变化。"""
        return hashlib.md5(open(path, "rb").read()).hexdigest()

    def sync(self, docs_dir: str):
        """同步：只处理新增/修改的文档。"""
        current_files = {f: self._file_hash(f) for f in glob(f"{docs_dir}/**")}
        previous = self.doc_store.load_hashes()  # 上次的哈希记录

        to_add, to_update, to_delete = [], [], []
        for path, h in current_files.items():
            if path not in previous:           # 新文件
                to_add.append(path)
            elif previous[path] != h:          # 内容变了
                to_update.append(path)
        for path in previous:
            if path not in current_files:       # 文件被删了
                to_delete.append(path)

        # 增量处理
        if to_add or to_update:
            docs = SimpleDirectoryReader(input_files=to_add + to_update).load_data()
            self.pipeline.run(documents=docs, show_progress=True)
        for path in to_delete:  # 删除向量库中对应节点
            self.vector_store.delete(ref_doc_id=path)

        self.doc_store.save_hashes(current_files)
        print(f"新增 {len(to_add)}，更新 {len(to_update)}，删除 {len(to_delete)}")

# 定时任务：每天凌晨同步知识库
# （用 cron 或 Airflow 调度）
# 0 2 * * * python -c "from kb import ingester; ingester.sync('./kb')"
💡 生产建议
LlamaIndex 的 Data Ingestion 管道解决的是知识库的"保鲜"问题——过时的知识库比没有知识库更危险（用户得到错误答案）。关键设计：① 增量更新——用文件哈希判断变化，只重新处理变化的文档（全量重建太慢太贵）；② 定时同步——配合 cron/Airflow 每日同步，保持知识库与源文档一致；③ 文档溯源——每个节点保留来源文件和时间戳，答案能追溯到具体文档。这三个能力加起来，才能支撑一个可信的生产级 RAG 系统。

本章小结

LlamaIndex 是 RAG 专精框架（50K stars），核心抽象：Document→Node→Index→Retriever→QueryEngine→Agent。从纯 RAG 进化为 Agentic Document Workflows。选型：纯 RAG 用 LlamaIndex，复杂 Agent 用 LangGraph，生产级最佳组合是 LlamaIndex（数据层）+ LangGraph（控制层）。
