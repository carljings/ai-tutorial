Continue 的 Embeddings
本地 vs 云端，以及 Jina / BGE / OpenAI 的选择

embedding 模型是 @codebase 检索的基础——它把代码块变成向量，向量间的距离代表语义相似度。选哪个 embedding 模型，直接影响检索质量。Continue 支持本地（transformers.js 跑 all-MiniLM、BGE、Jina）和云端（OpenAI text-embedding-3）两大类。本章我们对比这些模型在代码场景下的表现，并讨论「本地还是云端」这个对很多团队敏感的决策。

阅读约 17 分钟 难度 ★★★★

补充说明：embedding模型选型建议是快速发展的领域，建议定期关注Hugging Face博客、arXiv最新论文和主流开源社区动态，保持知识的持续更新。实践比理论更重要——每学一个新概念，就动手写一个小demo验证。

★ 一、embedding 在代码检索里扮演什么角色

回顾上一章，Continue 的检索是向量 + tf-idf 的混合方案。embedding 模型负责把代码块和查询都映射到同一个向量空间，让「语义相似」的代码在这个空间里距离更近。比如「用户登录验证」这个查询，理想情况下应该和包含 def authenticate(user, password) 的代码块向量接近——虽然字面只有「用户」二字相关，但语义上都是「登录验证」。

embedding 模型的质量，直接决定了向量检索这一半的召回质量。一个差的 embedding 模型可能把「登录」和「注销」映射得很近（因为都是认证相关），但你要找的是 login 代码不是 logout 代码——这就会产生噪声。好的 embedding 模型能在「相关」和「精确」之间找到平衡。

对代码场景，embedding 模型有两个特殊挑战：第一，代码是「结构化+自然语言」的混合（标识符是结构化、注释是自然语言）；第二，代码的语义强依赖上下文（同一段代码在不同上下文里语义不同）。通用文本 embedding 模型不一定能处理好这些。

★ 二、本地 vs 云端：不只是速度问题

Continue 默认用本地的 transformers.js 跑 all-MiniLM-L6-v2，但允许切换到 OpenAI 等云端服务。这个选择涉及多个维度：

维度本地（transformers.js）云端（OpenAI 等）
代码隐私代码不出本机代码上传到服务器
成本免费按 token 计费
延迟~10-50ms/块（首次加载慢）~100-300ms/块（含网络）
索引速度中等（受 CPU 限制）快（GPU 集群）
模型质量中等（小模型）高（大模型）
离线可用是否

对很多企业团队，代码隐私是决定性因素。把整个代码库上传到第三方 embedding 服务，可能违反公司的数据安全政策。本地 embedding 让代码不出本机，这是 Continue 默认本地的核心原因。

但本地有它的代价——transformers.js 在浏览器/Node 里跑 ONNX 模型，速度不如原生 GPU。一个大仓库首次索引可能要几分钟（CPU 模式）。Continue 用 Web Worker 并行化，缓解了这个问题但没根除。

★ 三、all-MiniLM-L6-v2：Continue 的默认选择

all-MiniLM-L6-v2 是 sentence-transformers 系列的一个小模型——只有 22M 参数、384 维输出。它之所以是 Continue 的默认，是因为三个优点：体积小（80MB，下载快）、推理快（CPU 上几十毫秒）、质量在通用文本任务上够用。

// 本地 embedding provider（简化）
import { pipeline } from "@xenova/transformers";

export class TransformersEmbedder implements EmbeddingsProvider {
  private extractor: any;

  async init() {
    // 加载模型（首次会从 hub 下载）
    this.extractor = await pipeline(
      "feature-extraction",
      "Xenova/all-MiniLM-L6-v2",
      { device: "cpu" }   // 浏览器/Node 默认 CPU
    );
  }

  async embed(texts: string[]): Promise<Float32Array[]> {
    const results: Float32Array[] = [];
    for (const text of texts) {
      const output = await this.extractor(text, {
        pooling: "mean",      // 对 token 向量做平均池化
        normalize: true,      // L2 归一化，方便直接算点积
      });
      results.push(output.data as Float32Array);
    }
    return results;
  }
}

注意 pooling: "mean" 和 normalize: true 两个选项。Transformer 模型输出的是每个 token 的向量序列，要把它变成「整段文本的向量」，最简单的是平均池化（mean pooling）。归一化让向量长度为 1，这样两个向量的点积就等于 cosine 相似度，省一次开方。这些是 embedding 工程的标准操作。

★ 四、代码专用模型：Jina 与 BGE

all-MiniLM 是通用文本模型，对代码不一定最优。Continue 也支持代码场景更擅长的模型，主要有两家：Jina 和 BGE。

Jina embeddings（如 jina-embeddings-v2-base-code）是 Jina AI 专门为代码训练的 embedding 模型。它的训练语料包含大量 GitHub 代码，对代码标识符、函数语义有更好的理解。在 CodeSearchNet 等代码检索 benchmark 上，它的表现通常优于通用模型。

BGE（BAAI General Embedding，如 bge-large-en-v1.5）是智源研究院开源的 embedding 模型，在 MTEB（Massive Text Embedding Benchmark）排行榜上长期名列前茅。BGE 不是代码专用，但因为模型大（335M 参数）、训练数据广，在代码任务上也表现不错。

// 配置 Jina 代码 embedding（本地）
{
  "embeddingsProvider": {
    "provider": "transformers.js",
    "model": "Xenova/jina-embeddings-v2-base-code"  // 代码专用
  }
}

// 或配置 BGE
{
  "embeddingsProvider": {
    "provider": "transformers.js",
    "model": "Xenova/bge-large-en-v1.5"
  }
}

这两个模型的代价是体积更大（BGE-large 约 1.3GB）、推理更慢（CPU 上几百毫秒一次）。对于大型代码库，首次索引可能从几分钟变成几十分钟。这是「质量 vs 速度」的典型权衡——团队需要根据自己的场景选择。

★ 五、OpenAI text-embedding-3：云端的高质量选项

如果不在乎成本和隐私，OpenAI 的 text-embedding-3-large 是目前公开可用的最强 embedding 之一。它在 MTEB 上的分数高于大多数开源模型，且支持自定义维度（可以通过 Matryoshkin 降维到 256/512/1024 等）。

// 云端 embedding provider（简化）
export class OpenAIEmbedder implements EmbeddingsProvider {
  constructor(private apiKey: string, private model: string = "text-embedding-3-small") {}

  async embed(texts: string[]): Promise<Float32Array[]> {
    // 批量调用（OpenAI 支持 2048 个输入/请求）
    const batches = chunk(texts, 2048);
    const allEmbeddings: Float32Array[] = [];
    for (const batch of batches) {
      const resp = await fetch("https://api.openai.com/v1/embeddings", {
        method: "POST",
        headers: {
          "Authorization": `Bearer ${this.apiKey}`,
          "Content-Type": "application/json",
        },
        body: JSON.stringify({
          model: this.model,
          input: batch,
        }),
      });
      const data = await resp.json();
      for (const item of data.data) {
        allEmbeddings.push(new Float32Array(item.embedding));
      }
    }
    return allEmbeddings;
  }
}

// 配置
{
  "embeddingsProvider": {
    "provider": "openai",
    "model": "text-embedding-3-large",
    "apiKey": "{OPENAI_API_KEY}",
    "dimensions": 1536  // 可选，降维节省存储
  }
}

注意 dimensions 参数——text-embedding-3 支持输出降维后的向量（如把原生的 3072 维降到 1536 维），存储空间减半但质量损失很小（这是 Matryoshkin representation learning 的功劳）。对大型代码库，这个优化能显著节省向量数据库的空间。

★ 六、如何选择：一个决策框架

面对这么多选项，团队该怎么选？Continue 没有强制推荐，但我们可以梳理一个决策框架：

  1. 代码能否上传第三方？ 不能 → 只能本地（MiniLM / Jina / BGE）。
  2. 仓库多大？ 几百文件 → MiniLM 够用；几万文件 → 考虑 BGE 或云端提升召回质量。
  3. 对检索质量多敏感？ 代码检索是核心场景 → 投资更高质量模型；偶尔用 → MiniLM 够了。
  4. 能否接受云端成本？ 接受且代码可上传 → OpenAI text-embedding-3 是最省心的选择。
  5. 首次索引延迟容忍度？ 低 → 云端（GPU 快）；高 → 本地大模型可接受。

一个常见的实战组合是：开发阶段用本地 MiniLM 快速迭代，正式部署换 OpenAI 或 BGE 提升质量。Continue 的配置让这种切换只需要改 config.json 一行，这是它设计上对「可插拔」的重视。

★ 七、切换 embedding 模型的代价

一个容易被忽视的点：切换 embedding 模型意味着所有已索引的向量都要重新计算。不同模型的向量空间不兼容——MiniLM 的 384 维向量和 BGE 的 1024 维向量无法比较，即使都叫「向量」。

Continue 在检测到 embeddingsProvider 配置变更时，会自动触发全量重建索引。对大仓库这可能意味着几分钟到几十分钟的等待。所以选择 embedding 模型最好在项目初期就定下来，避免频繁切换。如果必须换，建议在低峰时段进行，或后台重建不影响当前使用。

★ 本章小结

embedding 模型是检索质量的根基。要点回顾：

  • 角色：把代码和查询映射到向量空间，让语义相似的代码距离近，质量直接决定向量检索的召回。
  • 本地 vs 云端：隐私、成本、延迟、质量、离线可用是五大维度，代码隐私常是决定性因素。
  • all-MiniLM-L6-v2：22M 参数、384 维，体积小速度快，通用文本质量够用，是 Continue 默认。
  • Jina 代码专用：在代码语料上训练，对标识符和函数语义理解更好，CodeSearchNet 表现优。
  • BGE：335M 参数，MTEB 排行榜领先，模型大质量高但 CPU 推理慢。
  • OpenAI text-embedding-3：云端最强，支持 Matryoshkin 降维节省存储，但代码上传和成本是门槛。
  • 决策框架：隐私 → 本地；大仓库 → 投资质量；可上传 → OpenAI 省心。
  • 切换代价：模型变更需全量重建索引，最好初期定下来。
