生成策略全景
从贪心解码到核采样的权衡之道

同一个模型，greedy 输出干瘪重复，采样却能写出灵动的句子——差别全在「怎么从概率分布里挑下一个词」这件小事上。本章把 greedy、beam search、top-k、top-p、temperature 五个旋钮全部拧开，用真实代码和对比实验，让你看清每一种策略在质量、多样性、速度之间的取舍。

阅读约 24 分钟 难度 ★★★★

★ 一、解码问题：从概率分布里挑一个词

Decoder-Only 模型每前进一步，都会输出一个覆盖整个词表的概率分布。比如当前上下文是「今天天气真」，模型可能给出这样的概率：「好」0.6、「不错」0.2、「冷」0.1、「棒」0.05……以及剩下几万个词瓜分那点残值。

问题来了：这个分布给出来了，我们到底该选哪个词作为下一个输出？这就是解码策略要回答的核心问题。它看似是个工程小细节，却直接决定了你拿到的回答是机灵还是呆板、是丰富还是重复。

所有解码策略本质上都在回答同一个问题：在「确定性」（质量、连贯）和「随机性」（多样性、创造性）之间，你想要哪一个端点，或者哪个中间点？

1.1 两大家族：确定性 vs 采样

解码策略可以分成两大阵营。第一类是确定性策略，代表是 greedy 和 beam search。给定相同的概率分布，它们永远输出相同的结果。优点是稳定、可复现、质量下限有保证；缺点是容易陷入重复，缺乏创造力。

第二类是采样策略，代表是 top-k、top-p、temperature。它们引入随机性，每次输出都可能不同。优点是输出丰富多样、更像人类写作；缺点是不可复现，偶尔会生成无意义的句子。现代对话系统几乎都用采样策略，因为没人希望每次问同一个问题都得到一模一样的回答。

★ 二、贪心解码：最快但最容易翻车

greedy 是最朴素的策略：每一步都选概率最大的那个词。它不需要任何额外计算，速度最快，实现也最简单。但它的致命伤是「目光短浅」——只看眼前最优，可能错过全局最优。

2.1 贪心的陷阱：局部最优 ≠ 全局最优

举个通俗例子。假设模型当前预测「我」后面跟「爱」的概率是 0.6，跟「去」的概率是 0.3。greedy 会选「爱」。但如果真实语料里更常见的其实是「我去北京」，而「我爱」后面跟着的可能是冷僻词，那么 greedy 就会因为第一步贪了 0.6 的那个词，把整条路径带偏。

更严重的是重复问题。greedy 在开放式生成里很容易陷入「我我我我」或者「是是是是」这种死循环，因为一旦某个词的概率稍高，它就被反复选中，而后文又强化了这个概率。

2.2 手写 greedy 解码

下面这段代码完全从零实现 greedy，不用 Hugging Face 的 generate，让你看清每一步在干什么。

import torch
from transformers import AutoModelForCausalLM, AutoTokenizer

model_name = "gpt2"
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForCausalLM.from_pretrained(model_name)
model.eval()

def greedy_decode(model, tokenizer, prompt, max_new_tokens=20):
    ids = tokenizer(prompt, return_tensors="pt").input_ids
    generated = ids
    for _ in range(max_new_tokens):
        with torch.no_grad():
            logits = model(generated).logits[0, -1]   # 取最后一个位置的 logits
        next_id = logits.argmax().unsqueeze(0).unsqueeze(0)  # 选概率最大的词
        generated = torch.cat([generated, next_id], dim=-1)
    return tokenizer.decode(generated[0], skip_special_tokens=True)

print(greedy_decode(model, tokenizer, "Once upon a time", max_new_tokens=30))

跑几次你会发现结果完全一样——这就是确定性的含义。如果 prompt 足够开放，你很可能会看到模型陷入循环，这是 greedy 的典型病症。

★ 三、Beam Search：保留多条候选路径

beam search 是 greedy 的升级版。它不只保留一个最优词，而是同时保留 beam_width（束宽）条候选路径，每一步都扩展所有候选，然后只保留累计概率最高的 beam_width 条。这样即使某一步贪错了，别的路径还有机会翻盘。

打个比方：greedy 像一个人蒙着眼只走眼前最宽的路，beam search 像是同时派 5 个侦察兵分头探路，每到一个路口都汇报，指挥官保留最有希望的 5 条继续探索。虽然多花几倍计算，但能找到整体更优的句子。

3.1 beam search 的评分公式

beam search 用累计对数概率来给整条路径打分。对一条生成长度为 T 的序列，分数定义为所有 token 对数概率之和：score = log P(w1) + log P(w2) + ... + log P(wT)。之所以用对数而不是直接乘概率，是为了避免多个小于 1 的概率相乘导致数值下溢。

这里有个细节：直接用累计对数概率会偏好短序列（因为越乘越小）。所以工程上常用长度归一化，用 score 除以 T 或者 T^alpha 来抵消这个偏差，让长句子不被天然惩罚。

3.2 beam search 实现

import torch
import torch.nn.functional as F
from transformers import AutoModelForCausalLM, AutoTokenizer

model_name = "gpt2"
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForCausalLM.from_pretrained(model_name)
model.eval()

def beam_search(model, tokenizer, prompt, beam_width=4, max_new_tokens=20):
    ids = tokenizer(prompt, return_tensors="pt").input_ids          # (1, L)
    beams = [(0.0, ids)]    # (累计对数概率, 序列)
    for _ in range(max_new_tokens):
        candidates = []
        for score, seq in beams:
            with torch.no_grad():
                logits = model(seq).logits[0, -1]
            log_probs = F.log_softmax(logits, dim=-1)
            # 取对数概率最高的 beam_width 个词，避免遍历整个词表
            topk_logp, topk_ids = log_probs.topk(beam_width)
            for logp, tid in zip(topk_logp.tolist(), topk_ids.tolist()):
                new_seq = torch.cat([seq, torch.tensor([[tid]])], dim=-1)
                candidates.append((score + logp, new_seq))
        # 按累计对数概率排序，保留前 beam_width 条
        beams = sorted(candidates, key=lambda x: x[0], reverse=True)[:beam_width]
    return tokenizer.decode(beams[0][1][0], skip_special_tokens=True)

print(beam_search(model, tokenizer, "The capital of France is", beam_width=4, max_new_tokens=15))

你会看到 beam search 的输出通常比 greedy 更通顺，也更聚焦于「高概率」的事实性答案（比如直接说出「Paris」）。但它的代价是计算量是 greedy 的 beam_width 倍。

3.3 beam search 的适用边界

beam search 在翻译、摘要这种「有标准答案」的任务上表现极佳，因为这些任务的输出空间相对受限，最优解确实存在。但在开放式对话、创意写作上，它反而会让输出变得「安全但无聊」——因为它总倾向于选高概率词，结果就是一堆陈词滥调。这是 OpenAI 在 GPT-2 时代就发现的现象，也是现代对话模型转向采样的根本原因。

★ 四、Temperature：调节概率的「陡峭度」

temperature（温度）是采样策略的基础旋钮。它的作用是在 softmax 之前，先把 logits 除以一个温度值 T，再计算概率。T 越小，分布越尖锐（概率越集中）；T 越大，分布越平缓（概率越均匀）。

用类比理解：temperature 像是情绪的「冷静程度」。T 接近 0 时，模型变得绝对理性，几乎只选概率最高的词（趋近 greedy）；T 很大时，模型变得「头脑发热」，连低概率词都可能被选中，输出变得天马行空。

4.1 温度的数学含义

设原始 logits 为 z，温度为 T，则施加温度后的概率变为 p_i = exp(z_i / T) / sum(exp(z_j / T))。当 T 趋近 0 时，argmax 项的指数远大于其他项，分布退化为 one-hot，等同于 greedy；当 T 趋近无穷大时，所有 z_i / T 都趋近 0，分布趋近均匀分布，模型变成「随机掷骰子」。

工程上，T 通常取 0.7 到 1.0 之间。低于 0.5 会显得机械，高于 1.2 会开始胡言乱语。需要精确回答（数学、事实）时调低，需要创意（写诗、头脑风暴）时调高。

★ 五、Top-k 与 Top-p：动态收缩候选集

纯随机采样会从整个词表里采样，这显然不靠谱——大部分词根本不该出现在当前位置。top-k 和 top-p 都是为了「只在合理的候选里采样」，但收缩候选集的方式不同。

5.1 Top-k：只看概率最高的 k 个词

top-k 的逻辑很直白：把概率最高的 k 个词拿出来，重新归一化成新的概率分布，然后在这个小集合里采样。比如 k=50，就是只在前 50 个候选词里挑。这过滤掉了长尾的「噪声词」，让输出更安全。

top-k 的局限是 k 是固定的。在概率分布很集中时（比如「法国的首都是」后面「巴黎」概率 0.9），k=50 会引入 49 个没必要的噪声；在概率分布很分散时（比如开放续写），k=50 又可能太窄，砍掉了合理选项。

5.2 Top-p（核采样）：按累积概率截断

top-p（又叫 nucleus sampling，核采样）解决了这个问题。它不看词的个数，而是看累积概率：把词按概率从高到低排序，一直累加，直到累积概率第一次达到 p（比如 0.9），就截断，只在这些词里采样。

当分布集中时，可能只要 2-3 个词就累计到 0.9，候选集很小；当分布分散时，可能要 100 个词才到 0.9，候选集很大。它是动态自适应的，比 top-k 更聪明。这也是当前对话模型最常用的采样方式，典型设置是 top_p=0.9。

5.3 一个完整的对比实验

下面这段代码用同一个 prompt 和同一个模型，对比五种策略的输出，你可以直接运行观察差异。

from transformers import AutoModelForCausalLM, AutoTokenizer, set_seed

model_name = "gpt2"
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForCausalLM.from_pretrained(model_name)
model.eval()

prompt = "In the year 2050, humanity will"

# 1. greedy
print("=== Greedy ===")
print(tokenizer.decode(
    model.generate(**tokenizer(prompt, return_tensors="pt"),
                   max_new_tokens=25, do_sample=False)[0],
    skip_special_tokens=True))

# 2. beam search
print("\n=== Beam Search (width=5) ===")
print(tokenizer.decode(
    model.generate(**tokenizer(prompt, return_tensors="pt"),
                   max_new_tokens=25, num_beams=5, do_sample=False,
                   early_stopping=True)[0],
    skip_special_tokens=True))

# 3. temperature sampling
print("\n=== Temperature=0.7 ===")
set_seed(42)
print(tokenizer.decode(
    model.generate(**tokenizer(prompt, return_tensors="pt"),
                   max_new_tokens=25, do_sample=True, temperature=0.7)[0],
    skip_special_tokens=True))

# 4. top-k
print("\n=== Top-k=50 ===")
set_seed(42)
print(tokenizer.decode(
    model.generate(**tokenizer(prompt, return_tensors="pt"),
                   max_new_tokens=25, do_sample=True, top_k=50)[0],
    skip_special_tokens=True))

# 5. top-p
print("\n=== Top-p=0.9 ===")
set_seed(42)
print(tokenizer.decode(
    model.generate(**tokenizer(prompt, return_tensors="pt"),
                   max_new_tokens=25, do_sample=True, top_p=0.9)[0],
    skip_special_tokens=True))

运行这个对比，你会清晰地看到：greedy 和 beam search 输出稳定但偏「官方腔」，而 temperature、top-k、top-p 给出的句子更有想象力。top-p 通常是这几种里在「合理」和「多样」之间平衡得最好的。

★ 六、实战：组合使用才是王道

实际工程里，这些策略很少单独使用，而是组合起来。最经典的组合是 temperature + top-p：先用 temperature 调节整体分布的陡峭度，再用 top-p 截断长尾。Hugging Face 的 generate 函数天然支持这种组合。

6.1 推荐配置矩阵

不同任务的最佳配置差异很大。下面给出一份基于工程经验的配置参考，你可以作为起点再微调。

# 不同场景的推荐解码配置
configs = {
    "事实问答（要准确）":   {"do_sample": False, "num_beams": 1},                # greedy
    "数学推理（要严谨）":   {"do_sample": False, "num_beams": 4, "early_stopping": True},
    "摘要翻译（要通顺）":   {"do_sample": False, "num_beams": 5, "early_stopping": True},
    "日常对话（要自然）":   {"do_sample": True,  "temperature": 0.7, "top_p": 0.9},
    "创意写作（要灵感）":   {"do_sample": True,  "temperature": 0.9, "top_p": 0.95},
    "头脑风暴（要发散）":   {"do_sample": True,  "temperature": 1.1, "top_p": 0.98},
}
for scene, cfg in configs.items():
    print(f"{scene:20s} -> {cfg}")

记住一个原则：越是需要「正确答案」的任务越往确定性一端调，越是需要「新颖性」的任务越往随机性一端调。翻译、数学用 beam search；闲聊、写作用 top-p 采样。没有银弹，按场景选择。

★ 七、如何评估不同策略的好坏

比较解码策略不能只靠主观感觉，需要量化指标。这里介绍两个常用的自动化指标，帮你客观评估。

  • 多样性指标：通常用 distinct-1、distinct-2 来衡量，即生成文本里 unigram 和 bigram 的去重比例。比例越高说明用词越丰富，重复越少。采样策略的 distinct 通常远高于 greedy。
  • 重复率：统计生成文本里重复 n-gram 的比例。greedy 在长文本上重复率可能高达 30% 以上，而合理配置的 top-p 能压到 5% 以下。

当然，自动化指标只是参考，最终还是要靠人工评估或者训练一个奖励模型（这就是 RLHF 的思路）。但作为快速调参的依据，diversity 和 repetition 已经很实用了。

★ 本章小结

这一章我们把生成策略的全家桶梳理了一遍。要点回顾：

  • 两大阵营：确定性策略（greedy、beam search）稳定但易重复；采样策略（temperature、top-k、top-p）多样但不可复现。
  • Greedy：每步选概率最高的词，速度最快但目光短浅，开放生成里容易陷入循环。
  • Beam Search：保留多条候选路径，适合翻译摘要等有标准答案的任务，但会让开放对话变得平庸。
  • Temperature：softmax 前把 logits 除以 T，T 越小越确定、越大越随机，常用区间 0.7 到 1.0。
  • Top-k：只在概率最高的 k 个词里采样，固定数量，不够灵活。
  • Top-p（核采样）：按累积概率截断，动态自适应候选集大小，是当前对话模型的主流选择。
  • 组合使用：实际工程里 temperature + top-p 是黄金组合，按任务类型在确定性和随机性之间找平衡。

掌握了生成策略，你就能解释为什么同一个模型在不同配置下表现天差地别。但采样本身也会带来一个新问题：模型有时会反复输出同一个词或短语。下一章我们就来攻克这个顽疾——重复惩罚机制。
