Skills 技能体系
让 AI 拥有可复用的肌肉记忆

从 Prompt 到 Skill，是把「一次性指令」沉淀成「可复用能力」的进化。本章讲清 Skills 的设计哲学，以及它背后的统一协议 MCP（Model Context Protocol）——号称「AI 工具的 USB-C」，2026 年下载量破亿的事实标准。

阅读约 30 分钟 难度 ★★★★ 前置：第 03-05 章

★ 什么是 Skill

想象你每次让 AI 做某件事，都要重新解释一遍「怎么做」——繁琐、不一致、容易出错。Skill（技能）就是把「怎么做某类事」的完整知识封装成一个可加载、可复用的单元。

Skill 的定义：一个自包含的能力单元，包含「什么时候用、怎么用、用什么工具、注意什么」。模型按需加载它，就像人调用「肌肉记忆」一样，不必每次从头思考。

01 一个 Skill 的解剖

一个完整的 Skill 通常包含四部分：

一个 Skill 的结构（SKILL.md 风格）
name: pdf-report-generator
description: 把 Markdown 内容生成精美的 PDF 报告，支持封面、目录、图表
trigger:                              # 什么时候触发这个技能
  - "生成 PDF 报告"
  - "做一份精美的文档"
tools:                                # 这个技能需要哪些工具
  - pandoc
  - latex
  - file_writer
workflow:                              # 怎么做（流程）
  1. 确认报告主题、受众、风格
  2. 收集/生成内容
  3. 选择模板
  4. 渲染 PDF
  5. 自检排版问题并修正
tips:                                  # 经验性注意事项
  - 中文报告优先用 xeCJK
  - 图表宽度别超过文本宽度

关键洞察：Skill 不只是「一段提示」，而是提示 + 触发条件 + 工具依赖 + 执行流程 + 经验的完整封装。这让 AI 从「需要被指挥」变成「自己知道怎么干」。

03 MCP：工具调用的 USB-C

如果说 Skill 是「能力」，那 MCP（Model Context Protocol）就是让能力跨平台流通的「标准接口」。它由 Anthropic 在 2024 年提出，到 2026 年已成为事实标准——下载量破亿，号称「AI 工具的 USB-C」。

MCP 解决了什么问题

MCP 之前，每个模型/框架都有自己的工具定义格式：

  • OpenAI 的 function calling 是一套格式；
  • 智谱的是另一套（虽兼容 OpenAI）；
  • LangChain 又是一套抽象；
  • 你给 Claude 写的工具，换到 GLM 就跑不了。

这导致工具生态碎片化——开发者要为每个平台重复适配。MCP 的出现统一了这一切：

一个用 MCP 写的工具/技能，任何支持 MCP 的客户端（Claude、GLM、Cursor、各种 Agent SDK）都能直接用。

MCP 的核心概念

概念含义
Tools（工具）可被模型调用的函数（如读文件、查数据库）
Resources（资源）可被读取的数据（如文件内容、配置）
Prompts（提示模板）预定义的提示模板
Server（服务端）提供上述能力的进程，可以是本地或远程
Client（客户端）AI 应用，连接 Server 获取能力

写一个最简 MCP Server

用官方 SDK 写一个 MCP 工具服务
# weather_server.py —— 一个极简 MCP Server
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("weather")

@mcp.tool()
def get_weather(city: str) -> str:
    """查询指定城市的天气"""
    # 实际调用天气 API
    return f"{city} 今天晴，25°C"

@mcp.tool()
def get_forecast(city: str, days: int = 3) -> str:
    """查询未来几天的天气预报"""
    return f"{city} 未来 {days} 天多云为主"

if __name__ == "__main__":
    mcp.run()   # 任何 MCP 客户端都能连上来用这两个工具

就这么简单——用装饰器声明工具，SDK 自动处理协议细节。任何支持 MCP 的客户端（Claude Desktop、Cursor、智谱 Agent 等）连上这个 Server，就能用 get_weather 和 get_forecast。写一次，到处用。

💡 MCP 的生态规模

截至 2026 年，MCP 已有 97 百万下载，50+ 框架支持。GitHub 上有大量现成的 MCP Server（GitHub、Slack、数据库、文件系统、浏览器……），拿来即用。Claude Agent SDK 有最深的 MCP 集成，智谱 Agent SDK 也全面支持。这意味着你写的工具能被整个生态复用。

04 怎么设计一个好 Skill

不是所有工作流都该做成 Skill。好的 Skill 有几个特征：

高频复用

同样的流程会反复用到。一次性任务不必封装。

边界清晰

能明确说清「这个 Skill 做什么、不做什么」。模糊的 Skill 会和别的重叠。

自包含

尽量少依赖外部状态。带上自己的工具、提示、流程。

有触发条件

明确什么场景该激活它，避免误触发。

可测试

能写测试用例验证 Skill 行为正确。

带经验

把踩过的坑、最佳实践写进去，让 Skill 自带「老司机」属性。

05 技能发现：让模型自己选

当 Skill 库变大（几十上百个），不可能全塞进上下文。需要技能发现机制：根据当前任务，动态选出最相关的几个 Skill 加载。

  • 基于描述匹配：把任务和所有 Skill 的 description 做匹配（可用 Embedding）；
  • 基于触发词：Skill 声明自己的触发关键词，命中即加载；
  • 基于路由模型：训一个小模型专门做「任务→Skill」路由；
  • 分层加载：先加载 Skill 摘要，确定要用时再加载完整内容。

这正是 Claude Skills、智谱 Skills 等系统在做的事。智能体的「智能」，很大程度体现在能不能选对 Skill。

09 代码实战：Skill 加载器与技能发现

第 5 节讲了技能发现的理念。这里实现一个基于相关性的技能发现系统——从一堆 Skill 里动态选出当前任务最相关的：

skill_loader.py —— 技能发现 + 渐进式加载
"""
skill_loader.py —— 技能发现：根据任务动态选出相关 Skill
避免把所有 Skill 都塞进上下文
"""
import os, yaml
from sentence_transformers import SentenceTransformer
embedder = SentenceTransformer("BAAI/bge-m3")

class SkillRegistry:
    def __init__(self, skills_dir="./skills"):
        self.skills = {}     # name → skill 数据
        self.desc_embs = {}  # name → 描述向量
        for f in os.listdir(skills_dir):
            if f.endswith(".md") or f.endswith(".yaml"):
                data = yaml.safe_load(open(os.path.join(skills_dir, f)))
                self.skills[data["name"]] = data
                self.desc_embs[data["name"]] = embedder.encode([data["description"])[0]

    def discover(self, task, top_k=3):
        # 用任务与所有 Skill 描述算相似度，选 top_k
        task_emb = embedder.encode([task])[0]
        scored = [(cosine(task_emb, emb), name) for name, emb in self.desc_embs.items()]
        scored.sort(reverse=True)
        return [self.skills[name] for _, name in scored[:top_k]]

    def load_progressive(self, task):
        # 渐进式披露：先只加载 name + description（省 token）
        candidates = self.discover(task, top_k=5)
        brief = [{"name": s["name"], "desc": s["description"]} for s in candidates]
        # 让模型从 brief 里选真正要用的，再加载完整 workflow
        chosen = llm(f"任务：{task}\n候选技能：{brief}\n选 1-2 个最相关的（只给 name）：")
        return [self.skills[n] for n in chosen if n in self.skills]

这个「向量初筛 → 模型精选 → 完整加载」的三步渐进式披露，是 Claude Skills 的核心设计。它让拥有上百个 Skill 的系统也能高效运行——上下文里永远只装当前需要的。

10 代码实战：完整的 MCP Server（带资源 + 工具）

第 3.3 节给了极简版，这里是一个更完整的 MCP Server，包含工具、资源、错误处理：

db_mcp_server.py —— 数据库查询 MCP Server
"""
db_mcp_server.py —— 一个数据库查询的 MCP Server
提供「查表结构、执行 SQL、导出结果」三个工具
任何 MCP 客户端（Claude/GLM/Cursor）连上即可用
"""
import sqlite3, json
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("database")
db = sqlite3.connect("app.db", check_same_thread=False)

@mcp.tool()
def list_tables() -> str:
    """列出数据库所有表名"""
    cur = db.execute("SELECT name FROM sqlite_master WHERE type='table'")
    return json.dumps([r[0] for r in cur.fetchall()], ensure_ascii=False)

@mcp.tool()
def describe_table(table: str) -> str:
    """查看某张表的字段结构"""
    if not table.isidentifier():          # 防 SQL 注入
        return "[错误] 非法表名"
    cur = db.execute(f"PRAGMA table_info({table})")
    return json.dumps([{"col": r[1], "type": r[2]} for r in cur.fetchall()])

@mcp.tool()
def query(sql: str, limit: int = 100) -> str:
    """执行只读 SELECT 查询（自动加 LIMIT）"""
    sql = sql.strip()
    if not sql.upper().startswith("SELECT"):   # 只允许查，不允许改
        return "[错误] 仅支持 SELECT"
    if "LIMIT" not in sql.upper():
        sql += f" LIMIT {limit}""
    try:
        cur = db.execute(sql)
        rows = cur.fetchmany(limit)
        cols = [d[0] for d in cur.description]
        return json.dumps([dict(zip(cols, r)) for r in rows], ensure_ascii=False)
    except Exception as e:
        return f"[错误] {e}"

if __name__ == "__main__":
    mcp.run()
    # 现在任何 MCP 客户端都能用这三个工具操作你的数据库
    # Claude Desktop / Cursor / 智谱 Agent 连上后，自然语言就能查库

这个 Server 展示了 MCP 的安全设计：表名校验防注入、只允许 SELECT 防误删、自动 LIMIT 防全表扫描。写 MCP Server 时，把不可信输入当作攻击来防——因为模型的查询可能不完美。

📌 MCP 的网络效应

你写的这个数据库 Server，今天就能被 Claude、GLM、Cursor、Cline、任何 MCP 客户端使用。写一次，全生态可用——这就是标准协议的力量。GitHub 上已有几百个现成 MCP Server（GitHub/Slack/Notion/各种数据库/浏览器…），你的智能体可以瞬间获得这些能力。
