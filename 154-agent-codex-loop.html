Codex Agent 循环
从 model call 到 approval 到 continue

所有 Agent 的核心都是一个循环：调模型 → 模型说要调工具 → 执行工具 → 把结果喂回模型 → 继续。听起来简单，但真正的工程实现里藏着无数细节——流式中断怎么办、审批超时怎么办、工具失败要不要重试、循环什么时候算结束。本章我们把 Codex 的 Agent 循环拆到每一圈，并和 OpenCode 的实现做正面对比，看清同一种抽象下的不同工程取舍。

Q1: 这个工具的核心设计决策是什么？为什么这么选？

每个Agent工具都有一个"灵魂设计决策"。OpenCode的灵魂是"配置即代码"——所有行为（权限、工具、技能）都通过声明式配置定义。Codex的灵魂是"安全第一"——Rust内核+双层沙箱+Guardian审批。Aider的灵魂是"Git原生"——每次修改自动commit，diff格式编辑。Continue的灵魂是"双模式"——autocomplete和chat共享同一套模型配置但走不同的推理路径。理解灵魂决策，就理解了整个工具的架构方向。

Q2: 生产环境部署时需要注意什么？

三个关键注意事项：(1)API密钥安全——永远不要把密钥写在代码里，用环境变量或密钥管理服务（如HashiCorp Vault）。(2)速率限制——实现指数退避重试，避免429错误。建议设置每分钟最大请求数（如60/min）。(3)成本控制——设置每日token预算上限，超限自动降级到更便宜的模型或拒绝请求。

# 生产级成本控制示例
import time
from collections import defaultdict

class BudgetGuard:
    def __init__(self, daily_limit=1_000_000):
        self.daily_limit = daily_limit
        self.usage = defaultdict(int)
        self.day = time.strftime("%Y-%m-%d")
    
    def check_and_deduct(self, tokens, model="default"):
        today = time.strftime("%Y-%m-%d")
        if today != self.day:  # 新的一天，重置
            self.usage = defaultdict(int)
            self.day = today
        
        if self.usage[today] + tokens > self.daily_limit:
            # 降级到更便宜的模型
            if model == "glm-5":
                return "glm-5-flash", tokens // 2  # flash模型更便宜
            raise BudgetExceeded(f"日预算{self.daily_limit}已用尽")
        
        self.usage[today] += tokens
        return model, tokens

guard = BudgetGuard(daily_limit=500_000)
model, tokens = guard.check_and_deduct(8000, "glm-5")

Q3: 与竞品相比，优势和劣势分别是什么？

客观对比是选型的关键。以Agent工具为例：Claude Code在推理深度上最强但只支持Anthropic模型；Cursor在IDE集成上最好但是闭源且价格高；OpenCode在开放性和可定制性上最优但文档不如商业产品；Codex在安全性上最严格但配置复杂。选型时先确定你的第一优先级（安全/体验/开放/成本），再匹配工具。

Q4: 有哪些常见的错误用法和踩坑经验？

五个常见坑：(1)过度信任Agent输出——Agent会"自信地犯错"，生产中必须有人审核。(2)忽略token成本——一个复杂任务可能消耗50K+token，不控制成本很快烧光预算。(3)上下文管理不当——把所有历史都塞进context window，导致重要信息被"稀释"。应该用摘要+检索策略。(4)错误处理缺失——工具调用失败后Agent可能进入异常状态，必须有try-catch+重试+降级。(5)安全边界模糊——给Agent过大的权限（如root执行），一次prompt injection就可能导致数据泄露。

Q5: 未来的演进方向是什么？

Agent工具的三个演进趋势：(1)多模态Agent——不只处理文本，还能理解截图、UI布局、视频流（如Computer Use）。(2)长时自治Agent——从"完成单轮任务"到"持续运行、自我管理、定期报告"（如 Devin 的自主开发模式）。(3)Agent协作网络——多个专业化Agent通过A2A协议协作，每个Agent负责自己擅长的领域（前端/后端/测试/部署），像人类团队一样分工。

关键洞察：Agent工具的核心竞争力正在从"模型能力"转移到"工程能力"。同一个模型，套上不同的Harness/工具链/权限模型，效果可以差出10倍。这跟01-03章讲的Harness理论完全一致——"模型是引擎，工程是底盘"。
核心概念与实现★★★★

一个成熟的 Agent 循环，单轮（turn）内部其实可以切成五段。我们用伪代码标出边界，再逐段拆解。

// 一轮 turn 的五段结构
async fn run_turn(session: &mut Session) {
    // 【段 1】准备：拼装 system prompt、工具列表、历史消息
    let request = session.build_request();

    // 【段 2】调用：流式调用 LLM，边收边向前端推 delta
    let response = session.call_model_streaming(request, |delta| {
        emit(Event::OutputDelta { text: delta });
    }).await;

    // 【段 3】判停：LLM 有没有请求工具？没有就结束本轮
    let tool_calls = match response.tool_calls {
        Some(c) => c,
        None => { emit(Event::TurnComplete); return; }
    };

    // 【段 4】执行：对每个 tool_call 审批 → 沙箱执行 → 收集结果
    for call in tool_calls {
        let approved = request_approval(&call).await;
        if !approved { continue; }
        let result = execute_in_sandbox(&call).await;
        session.add_tool_result(call.id, result);
    }

    // 【段 5】递归：把工具结果喂回去，开始下一圈
    // 实际用 loop 实现，这里递归示意
    Box::pin(run_turn(session)).await;
}

这五段不是 Codex 独创的——OpenCode、Claude Code、Aider 的循环骨架都长这样。差异在于每一段的细节处理。我们重点看 Codex 在段 2（流式调用）、段 3（判停策略）、段 4（审批与执行）上的独到设计。

★ 二、段 2：流式调用与中断的协作

Codex 的 LLM 调用是流式的——每收到一个 token 片段，立即发一个 OutputDelta 事件给前端，前端就能实时渲染。但这带来一个问题：如果用户中途按了 Esc 想中断，怎么办？

朴素的方案是直接 abort HTTP 请求，但这会让 LLM 已经生成但还没送达的内容全部丢失。Codex 的做法是 协作式中断：设置一个 cancellation_token，流式回调每次检查它，如果被设置就停止向 event 发 delta，但让底层 HTTP 请求自然读完（或优雅关闭），保证已生成的内容能落盘到会话日志。

// 流式调用的中断处理（概念示意）
use tokio_util::sync::CancellationToken;

async fn call_model_streaming<F>(request: Request, on_delta: F, token: CancellationToken)
    -> Response
where F: Fn(String) {
    let mut stream = http_client.stream_post(&request).await;

    let mut full_text = String::new();
    let mut tool_calls = vec![];

    while let Some(chunk) = stream.next().await {
        // 【关键】每次收到 chunk 先检查取消信号
        if token.is_cancelled() {
            // 不立即 break，先把当前 chunk 处理完
            // 但停止向前端推 delta，标记本次为「被中断」
            full_text.push_str(&chunk.text);
            break;
        }

        if !chunk.text.is_empty() {
            full_text.push_str(&chunk.text);
            on_delta(chunk.text.clone());  // 实时推给前端
        }
        if let Some(tc) = chunk.tool_call {
            tool_calls.push(tc);
        }
    }

    Response { text: full_text, tool_calls: if tool_calls.is_empty() { None } else { Some(tool_calls) } }
}

注意这里有一个微妙的设计：中断后已经收到的文本仍然保留在 full_text 里，会被写入会话历史。这意味着如果用户中断后又想继续，Agent 知道「上次说到哪了」，可以从断点续上，而不是从零开始。这种「优雅中断」的体验，远好过粗暴 abort。

★ 三、段 3：判停策略——什么时候算一轮结束

Agent 循环最怕两种情况：提前结束（任务还没做完就停了）和 永不结束（陷入死循环反复调同一个工具）。段 3 的判停逻辑就是在这两者间找平衡。

Codex 的判停条件是：LLM 这一轮没有返回任何 tool_call。这是一个简单但有效的启发式——如果 LLM 觉得还需要调工具，它会继续请求；如果它认为任务完成或需要用户输入，它会输出文本而不请求工具。这种「把判停权交给模型」的策略，依赖模型本身的判断力，但避免了 Agent 框架层的过度干预。

但这带来死循环风险：如果 LLM 反复请求同一个工具（比如一直 read_file 同一个文件），循环永远不结束。Codex 的防线是 最大轮数限制：

// 防死循环的最大轮数兜底
const MAX_TURNS_PER_USER_REQUEST: usize = 50;

async fn run_agent_session(session: &mut Session) {
    for turn_idx in 0..MAX_TURNS_PER_USER_REQUEST {
        // 检测重复工具调用
        if detect_repetitive_tool_calls(session, 3) {
            emit(Event::Error {
                message: "检测到重复的工具调用模式，已停止以防死循环".into()
            });
            break;
        }
        run_turn(session).await;
        if session.last_turn_had_no_tool_calls() {
            break;  // 正常结束
        }
    }
    if turn_idx == MAX_TURNS_PER_USER_REQUEST - 1 {
        emit(Event::Error { message: "达到最大轮数限制".into() });
    }
}

// 检测连续 N 次相同的工具调用
fn detect_repetitive_tool_calls(session: &Session, n: usize) -> bool {
    let recent = session.recent_tool_calls(2 * n);
    if recent.len() < 2 * n { return false; }
    // 检查是否成对重复（call → result → call → result 模式）
    recent.chunks(2).all(|pair| pair.len() == 2 && pair[0].same_call_as(&pair[1]))
}

这里的 detect_repetitive_tool_calls 是一个简单但实用的启发式：如果连续多次调用完全相同的工具 + 相同参数，很可能陷入了死循环。我们在第 175 章讲错误恢复时会专门讨论各种死循环检测策略。

★ 四、段 4：审批的异步等待

不是所有工具调用都需要审批——read_file 这种只读操作通常自动放行。但 execute（执行命令）这类有副作用的操作，Codex 默认要求用户审批。审批是一个异步等待：发请求事件给前端，挂起当前 tool_call 的执行，直到前端回一个 Op::Approve。

// 审批流程（概念示意）
async fn request_approval(call: &ToolCall) -> bool {
    let call_id = call.id.clone();
    let (tx, rx) = tokio::sync::oneshot::channel();

    // 把等待回调注册到 session，前端 Approve 时会触发它
    session.register_approval_waiter(&call_id, tx);

    // 发事件给前端
    emit(Event::ExecApprovalRequest {
        call_id: call_id.clone(),
        command: call.args.to_string(),
        cwd: session.cwd.clone(),
    });

    // 阻塞等待用户决定
    match timeout(Duration::from_secs(300), rx).await {
        Ok(Ok(ApprovalDecision::Allow)) => true,
        Ok(Ok(ApprovalDecision::AlwaysAllow)) => {
            session.add_allow_rule(&call);  // 学习规则，下次不问
            true
        }
        _ => false,  // 拒绝或超时
    }
}

三个细节。第一，oneshot channel：审批是「一次性」的，用 oneshot 而不是 mpsc，语义更精确。第二，超时兜底：5 分钟没回应就当拒绝，避免 Agent 永远挂起。第三，AlwaysAllow 学习：和 OpenCode 的 allow-always 一样，用户可以选择「以后都允许」把规则固化，减少后续打断。这种「交互式学习权限」已经成为现代 Agent 的标配。

★ 五、段 4 续：沙箱执行的隔离

Codex 的工具执行不是直接 spawn 进程，而是在沙箱里执行。沙箱的具体实现（macOS 用 seatbelt，Linux 用 Landlock）我们会在 156、157 章详述，这里只点出它的存在对循环设计的影响。

沙箱执行意味着 每次工具调用都有独立的文件系统视图和网络权限。Agent 跑 npm install 时，沙箱可能限制它只能写 node_modules 目录、禁止访问网络（或只允许特定域名）。这种隔离让 Agent 即使「乱跑命令」也不会污染宿主系统。

对循环的影响是：工具执行失败的处理更复杂了。一次失败可能是命令本身的问题，也可能是沙箱限制（比如没网络权限导致 install 失败）。Codex 在返回 tool_result 时会把沙箱的审计信息（哪些路径被拒、哪些网络请求被拦）一并附上，让 LLM 能区分「命令错误」和「权限错误」。

★ 六、与 OpenCode 循环的对比

把两个项目的循环放在一起对比，能看出不同的工程侧重。

  • 判停策略：两者都依赖「LLM 不再请求工具」作为正常结束条件，但 Codex 多了最大轮数兜底和重复调用检测，OpenCode 更依赖模型的自觉。
  • 中断处理：Codex 用 CancellationToken 做协作式中断，保留已生成内容；OpenCode 用 AbortController，行为类似但实现在 TypeScript 层。
  • 审批模型：两者都支持 allow/ask/deny 三态和 allow-always 学习。Codex 的审批通过 Op/Event 协议异步完成，OpenCode 通过 EventBus 的 emitAndWait。
  • 工具执行：Codex 强制沙箱（seatbelt/Landlock），OpenCode 可选（路径白名单 + 进程直接 spawn）。Codex 的隔离更强，OpenCode 的开销更低。
  • 错误反馈：两者都把工具失败当正常返回（isError=true），让 LLM 自我修正，不中断循环。

总体来说，Codex 的循环更「重」——更多防护、更强隔离、更复杂的失败处理；OpenCode 的循环更「轻」——更依赖模型判断、更少的框架干预、更快的迭代。这和它们的语言选择（Rust vs TypeScript）、定位（引擎 vs 产品）一脉相承。

D1 深度剖析：Turn 模型与推理持久化的协同

本章介绍了 Codex 的 Turn 模型与推理持久化。这一节我们把 Turn 抽象成数据结构，并展示推理状态如何在 Turn 间持久化以支持中断恢复。

// Turn 的数据模型与持久化
interface Turn {
  id: string;
  parentTurnId?: string;        // 支持分支（fork）
  status: "running" | "paused" | "done" | "aborted";
  userMessage: Message;         // 触发本 Turn 的用户输入
  assistantSteps: Step[];       // LLM 的多步推理
  reasoningState?: {            // 持久化的推理状态（用于恢复）
    partialOutput: string;      // 中断时的部分输出
    pendingToolCall?: ToolCall; // 待执行的工具调用
    tokenCount: number;
  };
  startedAt: string;
  finishedAt?: string;
}

// 中断恢复的核心：从 reasoningState 重建 LLM 上下文
function resumeTurn(turn: Turn): Turn {
  if (turn.status !== "paused" || !turn.reasoningState) return turn;
  // 把已完成的 steps 作为历史，从中断点继续
  const rs = turn.reasoningState;
  if (rs.pendingToolCall) {
    // 恢复时优先执行未完成的工具调用
    return { ...turn, status: "running" };
  }
  // 否则把部分输出作为已知，让 LLM 续写
  return { ...turn, status: "running" };
}

Turn 模型与推理持久化的协同，本质上是在解决"长任务的可靠性"。一个复杂任务可能跨越多个 Turn、调用几十次工具，任何一步都可能因网络/超时/用户中断而暂停。持久化推理状态后，重启能从中断点无缝继续，而不是从头重跑——这对用户体验和成本控制都至关重要。

Turn 状态触发转移持久化内容恢复行为
runningLLM/工具完成或中断实时增量—
paused用户中断/超时reasoningState 快照从中断点续
done正常结束完整 steps只读回放
aborted用户取消部分 steps不恢复

D2 生产避坑：Turn 持久化的难点

难点 1：工具副作用的幂等性

恢复 Turn 时，若中断发生在工具执行之后、结果回传之前，恢复后重跑工具可能造成重复副作用（如文件写两次、命令执行两次）。必须在持久化时记录"已执行未回传"的工具，恢复时跳过执行、用缓存结果。对非幂等工具（如发邮件），需幂等键保护。

难点 2：LLM 上下文重建的一致性

恢复时重建的 messages 数组必须与中断时完全一致，否则 LLM 的续写会"接不上"。任何 token 化、截断、格式化都要可重现。建议持久化时存原始消息（而非 token 化后），恢复时重新打包。

难点 3：并发 Turn 的隔离

用户可能同时开多个 Turn（如分屏调试）。它们的文件系统副作用若不隔离，会互相覆盖。Codex 用工作目录隔离或 git 分支隔离，每个 Turn 在独立空间操作，完成后才合并。

★ 本章小结

Codex 的 Agent 循环是工业级工程的样本。要点回顾：

  • 五段结构：准备 → 流式调用 → 判停 → 执行（审批+沙箱）→ 递归，是所有 Agent 循环的通用骨架。
  • 协作式中断：用 CancellationToken 检查，中断后保留已生成内容，支持断点续传。
  • 判停靠模型：LLM 不请求工具即结束，但用最大轮数 + 重复调用检测兜底防死循环。
  • 审批异步等待：oneshot channel + 超时兜底 + AlwaysAllow 学习，平衡安全与体验。
  • 沙箱执行：强制隔离，工具失败时附带沙箱审计信息，帮 LLM 区分错误类型。
  • 与 OpenCode 对比：Codex 重防护强隔离，OpenCode 轻量快迭代，是不同定位下的合理取舍。
