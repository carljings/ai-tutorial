知识蒸馏进阶：从 logits 到 agent 行为

大模型怎么"教"小模型——蒸馏的三层境界

深度进阶★★★

蒸馏的本质：老师教学生

知识蒸馏（Knowledge Distillation）的核心思想：用一个大的"老师"模型指导一个小的"学生"模型。学生不只学"正确答案"（硬标签），还学老师对每个选项的"概率分布"（软标签）——这个软标签包含了老师"觉得哪些错误选项更像正确答案"的"暗知识（dark knowledge）"。

💡 要点
软标签的价值：老师输出 [0.7, 0.2, 0.1]（猫/狗/鸟），比硬标签 [1,0,0]（猫）信息量大得多——它告诉学生"虽然答案是猫，但这个图也有点像狗"。这些"类间关系"是硬标签里没有的暗知识。

三层蒸馏

层级学生学什么代表压缩比
L1: logit 蒸馏老师的输出概率分布DistilBERT2×
L2: 回复蒸馏老师生成的回答（SFT 数据）Alpaca/Vicuna10-25×
L3: 行为蒸馏老师的 agent 行为（工具调用/推理链）Agent 蒸馏10-50×

L2 回复蒸馏：当前主流

对于 LLM，最实用的蒸馏是 L2——用 GPT-4 生成大量高质量问答对，用这些数据 SFT 一个小模型。Alpaca（7B，用 5.2 万条 GPT-3.5 数据）和 Vicuna 就是这种路线。成本低（只需 API 调用）、效果好（7B 能达到 GPT-3.5 的 80%+ 能力）。

# L2 回复蒸馏流程
# 1. 用老师模型（GPT-4）生成训练数据
prompts = generate_diverse_prompts(n=50000)  # 多样化问题
train_data = [{"prompt": p, "response": gpt4.generate(p)} for p in prompts]

# 2. 用这些数据 SFT 学生模型（7B）
student_model.sft(train_data, epochs=3)

# 3. 评估学生 vs 老师的差距
# MMLU: GPT-4=86%, 学生7B=68% → 达到老师 79% 水平
# 成本: GPT-4 推理 $0.06/1K token, 学生7B 本地推理 ~$0.001/1K token

L3 行为蒸馏：Agent 时代的新前沿

当老师是一个 Agent（能调工具、做多步推理），蒸馏就不只是"学回答"，而是"学行为"——学生在什么时候该调什么工具、推理链怎么展开。这是 2025 年的前沿方向，让小模型也能做 Agent 任务。

KL 散度损失：logit 蒸馏的数学核心

L1 logit 蒸馏的目标是让学生模仿老师的输出概率分布。衡量两个分布差异的标准工具是 KL 散度（Kullback-Leibler divergence）。完整的蒸馏损失是"硬标签的交叉熵（学正确答案）+ 软标签的 KL 散度（学老师的暗知识）"的加权组合。理解这个公式的每一项，才能调好温度参数。

知识蒸馏的损失函数

  总损失 L = α · L_hard + (1-α) · L_soft

  ① L_hard：学生输出 vs 真实标签的交叉熵（标准分类损失）
     L_hard = CE(z_s, y)      # z_s 是学生 logit，y 是真实硬标签

  ② L_soft：学生软分布 vs 老师软分布的 KL 散度
     L_soft = KL( p_T^τ ‖ p_S^τ )
     其中 p^τ = softmax(z / τ)   # τ 是温度参数

  关键：温度 τ 的作用
     τ=1: softmax 原样（峰值尖锐，暗知识弱）
     τ→∞: 分布趋平（所有类别概率接近 1/V）
     τ=4~8: 经验最佳区间——软化到能看清类间关系，又不至于平到无信息

  直觉：τ 越大，softmax 输出越平，那些"错误类别的小概率"
         被放大，学生能学到"老师觉得哪个错误答案更像对的"。
         这正是暗知识（dark knowledge）的来源。

  α 的经验值：0.5~0.9（软标签权重通常更大，因为信息量更高）
  τ 的经验值：分类任务 4，检索/排序任务 1~2

这个公式里最容易忽视的是温度 τ 对"暗知识可见度"的放大作用。原始 logits 经 softmax 后，正确类别的概率往往高达 0.99，其他类别都是 0.001 量级——梯度几乎为零，学生学不到类间关系。引入温度 τ 把分布拉平后，原本 0.001 和 0.002 的两个错误类别的差异被放大成可学习的梯度信号。这就是 Hinton 原论文里说的"teachers know more than they reveal"——温度蒸馏就是为了逼老师把隐藏的知识交出来。

教师-学生模型 PyTorch 实现

把上面的损失函数落成可跑的 PyTorch 代码。这段实现了一个通用的蒸馏训练循环：老师和学生都是分类模型，损失由 CE（硬）和 KL（软）两部分加权。关键是用 `F.kl_div` 时注意它的输入顺序和 `log_target` 参数——PyTorch 这里的 API 坑不少。

import torch
import torch.nn as nn
import torch.nn.functional as F

class DistillationTrainer:
    def __init__(self, teacher, student, temperature=4.0, alpha=0.7):
        """
        teacher: 已训练好的老师模型（eval 模式，参数冻结）
        student: 待训练的学生模型
        temperature: 软化温度 τ
        alpha: 硬标签损失权重，软标签权重 = 1 - alpha
        """
        self.teacher = teacher.eval()
        for p in self.teacher.parameters():
            p.requires_grad = False       # 老师不更新
        self.student = student
        self.T = temperature
        self.alpha = alpha

    def distillation_loss(self, student_logits, teacher_logits, labels):
        # 1) 硬标签损失：学生 vs 真实标签（用学生原始 logits）
        loss_hard = F.cross_entropy(student_logits, labels)

        # 2) 软标签损失：学生软分布 vs 老师软分布（KL 散度）
        #    注意：温度 T 同时作用于两边，且 KL 要除以 T^2 保持梯度尺度
        teacher_probs = F.softmax(teacher_logits / self.T, dim=1)
        student_log_probs = F.log_softmax(student_logits / self.T, dim=1)
        loss_soft = F.kl_div(
            student_log_probs, teacher_probs,
            reduction='batchmean', log_target=False
        ) * (self.T ** 2)   # 关键：补偿温度对梯度的缩放

        return self.alpha * loss_hard + (1 - self.alpha) * loss_soft

    def train_step(self, inputs, labels, optimizer):
        with torch.no_grad():
            teacher_logits = self.teacher(inputs)   # 老师前向（不算梯度）
        student_logits = self.student(inputs)       # 学生前向

        loss = self.distillation_loss(student_logits, teacher_logits, labels)
        optimizer.zero_grad()
        loss.backward()
        optimizer.step()
        return loss.item()

# 使用示例
trainer = DistillationTrainer(teacher=resnet50, student=resnet18,
                              temperature=4.0, alpha=0.7)
optimizer = torch.optim.AdamW(resnet18.parameters(), lr=1e-3)
for epoch in range(20):
    for imgs, labels in train_loader:
        loss = trainer.train_step(imgs, labels, optimizer)

这段代码有两个工程细节不读论文容易踩坑：① 软标签损失必须乘以 T²——因为温度 softmax 改变了梯度尺度，乘 T² 是 Hinton 原论文给出的梯度补偿，否则软标签梯度会被温度压小，与硬标签权重失衡；② `F.kl_div` 第一个参数要传 log 概率、第二个传概率，且 `reduction='batchmean'` 才是数学正确的 KL 散度（默认的 'mean' 会除以类别数，低估损失）。老师前向要用 `torch.no_grad()` 且冻结参数，避免无谓的反向传播。

💡 要点
老师的 logits 不一定要现算——可以离线把老师在训练集上的 logits 缓存下来，训练时直接读。当训练集很大、老师模型很慢时，这一步能把蒸馏训练成本降一个数量级。代价是存储：每条样本存 V 个类的 logits（V 是词表大小，对 LLM 可能是 5 万+），需要量化或只存 top-k。

温度参数调优：从经验到实验

温度 T 是蒸馏里最重要的超参，但没有万能值。Hinton 原论文推荐 T=4，但不同任务差异很大。正确做法是用小规模实验扫描：固定其他参数，在 {1, 2, 4, 8, 16} 里选让验证集指标最高的 T。下面给出温度选择的决策逻辑和常见误区。

温度 T软标签分布暗知识强度适用场景
1（不软化）接近 one-hot几乎无退化为普通训练
2-4轻微软化弱-中类别多、类间差异大
4-8明显软化强（推荐区间）大多数分类任务
8-16高度软化很强但噪声也增类间关系复杂的检索任务
>20趋近均匀分布梯度被压垮不推荐

温度调优的一个反直觉现象：T 太大时学生性能反而下降。原因是当 T 趋近无穷大时，softmax 输出趋近均匀分布，老师实际上"什么都没教"——所有类别的概率都一样，KL 散度梯度趋近零。所以温度不是越大越好，而是存在一个"信息量最大化"的甜点区间。实践中，先在 T=4 跑一个 baseline，再在 {2, 4, 8} 里做网格搜索，通常 3-4 次实验就能定下来。对于 LLM 的 logit 蒸馏（如 DistilBERT），T 通常取 4-8，alpha 取 0.5-0.9，这是社区验证过最稳的配置。

⚠️ 落地提醒
对 LLM 而言，L1 logit 蒸馏的前提是你能拿到老师的 logits（即老师是你自己的模型或开源模型）。如果你只能调闭源 API（GPT-4/Claude），拿不到 logits，就只能走 L2 回复蒸馏。这也是为什么 L2 是当前主流——绝大多数团队没有自研大模型，蒸馏的是 API 的"行为"而非"logits"。

本章小结

知识蒸馏让大模型"教"小模型，三层境界：L1 logit 蒸馏（学软标签，2×）、L2 回复蒸馏（学老师生成的回答，10-25×，当前主流）、L3 行为蒸馏（学 agent 行为，前沿）。L2 是当前性价比最高的路线——用 GPT-4 生成数据 SFT 小模型，成本降低 10 倍。

import numpy as np
from dataclasses import dataclass
from typing import List, Optional

@dataclass
class PerformanceMetrics:
    latency_ms: float
    throughput_ops: float
    memory_mb: float
    accuracy: float

def benchmark(fn, iterations=100):
    """通用性能基准测试"""
    import time
    t0 = time.time()
    for _ in range(iterations):
        result = fn()
    elapsed = time.time() - t0
    return PerformanceMetrics(
        latency_ms=elapsed * 1000 / iterations,
        throughput_ops=iterations / elapsed,
        memory_mb=0,  # 实际中用 psutil 获取
        accuracy=0.0  # 根据具体任务计算
    )

# 示例：对比两种方案的性能
def approach_a(data):
    """方案A：直接计算"""
    return np.softmax(data @ data.T / np.sqrt(data.shape[-1]), axis=-1)

def approach_b(data):
    """方案B：分块计算（省内存）"""
    chunk_size = 64
    results = []
    for i in range(0, data.shape[0], chunk_size):
        chunk = data[i:i+chunk_size]
        results.append(np.softmax(chunk @ data.T / np.sqrt(data.shape[-1]), axis=-1))
    return np.vstack(results)

data = np.random.randn(512, 768)
metrics_a = benchmark(lambda: approach_a(data))
metrics_b = benchmark(lambda: approach_b(data))
print(f"方案A: {metrics_a.latency_ms:.1f}ms, {metrics_a.throughput_ops:.0f} ops/s")
print(f"方案B: {metrics_b.latency_ms:.1f}ms, {metrics_b.throughput_ops:.0f} ops/s")

常见问题FAQ

Q: 如何选择合适的方案？ 先用方案A跑通验证正确性，再根据实际瓶颈（内存/延迟/精度）切换到方案B或C。永远不要过早优化。

Q: 生产环境最大的坑是什么？ 过度工程化——在需求不明确时投入大量精力做复杂架构。建议MVP先行，收集反馈后迭代。

Q: 如何评估效果？ 定义明确的成功指标（准确率/延迟/成本），用golden set做回归测试，A/B测试验证改进效果。至少100个样本才有统计显著性。
