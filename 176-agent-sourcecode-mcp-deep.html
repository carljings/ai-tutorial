MCP 协议深度解析
JSON-RPC、四类能力与传输层

我们在第 151 章从 OpenCode 的角度看过 MCP 集成。本章我们从协议本身出发，深入 MCP 的细节——JSON-RPC 消息格式、initialize 握手、tools/resources/prompts/sampling 四类能力的语义差异、stdio 与 SSE（以及 streamable HTTP）三种传输的取舍。理解了这些，你不仅能用好 MCP，还能自己写一个 MCP server。

阅读约 22 分钟 难度 ★★★★★

★ 一、MCP 基于 JSON-RPC 2.0

MCP 的消息格式基于 JSON-RPC 2.0——一个轻量级的远程过程调用协议。JSON-RPC 定义了三种消息类型：request（请求，期待响应）、response（响应，成功或失败）、notification（通知，不期待响应）。

// Request（client -> server）
{
  "jsonrpc": "2.0",
  "id": 1,                          // 整数或字符串，用于匹配 response
  "method": "tools/call",
  "params": {
    "name": "search_code",
    "arguments": { "query": "authenticate" }
  }
}

// Response - 成功（server -> client）
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": [{ "type": "text", "text": "找到 3 处..." }]
  }
}

// Response - 失败
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -32602,                 // JSON-RPC 标准错误码
    "message": "Invalid params: missing 'query'"
  }
}

// Notification（不期待响应）
{
  "jsonrpc": "2.0",
  "method": "notifications/initialized",
  "params": {}
}

JSON-RPC 的核心是 id 的匹配机制：每个 request 有唯一 id，对应的 response 用相同 id。这让 client 能把异步收到的 response 和它发的 request 配对——即使 response 乱序到达（在 SSE 等异步传输上可能发生），也能正确匹配。notification 没有 id，因为不需要匹配。

MCP 在 JSON-RPC 之上定义了自己的 method 命名空间，如 initialize、tools/list、tools/call、resources/read 等。这些 method 构成了 MCP 协议的 API 表面。

★ 二、Initialize 握手：版本与能力协商

MCP 连接建立后的第一件事是 initialize 握手。双方交换协议版本、声明各自支持的能力（capabilities），确保兼容。这一步至关重要——MCP 协议在演进（从 2024-11-05 到 2025-03-26 等版本），client 和 server 可能支持不同版本。

// Client -> Server: initialize request
{
  "jsonrpc": "2.0",
  "id": 0,
  "method": "initialize",
  "params": {
    "protocolVersion": "2024-11-05",
    "capabilities": {
      "roots": { "listChanged": true },      // client 能提供 roots（工作目录）
      "sampling": {},                         // client 支持 sampling 代理
      "elicitation": {}                       // client 支持 elicit 用户输入
    },
    "clientInfo": {
      "name": "opencode",
      "version": "1.2.0"
    }
  }
}

// Server -> Client: initialize response
{
  "jsonrpc": "2.0",
  "id": 0,
  "result": {
    "protocolVersion": "2024-11-05",          // 协商后的版本（可能降级）
    "capabilities": {
      "tools": { "listChanged": true },       // server 提供工具
      "resources": { "subscribe": true },     // server 提供资源，支持订阅
      "prompts": { "listChanged": true },     // server 提供 prompt 模板
      "logging": {}                            // server 支持发日志
    },
    "serverInfo": {
      "name": "github-mcp-server",
      "version": "0.3.0"
    }
  }
}

// Client -> Server: initialized notification（无 id，无响应）
{
  "jsonrpc": "2.0",
  "method": "notifications/initialized"
}

握手的关键是 capabilities 的双向声明。client 声明自己能提供什么（roots、sampling），server 声明自己提供什么（tools、resources、prompts）。握手后双方都知道对方的边界——client 不会向不支持 resources 的 server 发 resources/read 请求。这种「能力发现」机制让 MCP 协议既灵活（server 可以只实现部分能力）又安全（不会发不被支持的请求）。

★ 三、tools：可执行的能力

tools 是 MCP 最常用的能力——server 暴露的可执行函数，client 的 LLM 可以主动调用。tools 的发现通过 tools/list，调用通过 tools/call。

// Client -> Server: 列出工具
{ "jsonrpc": "2.0", "id": 1, "method": "tools/list" }

// Server -> Client: 工具列表
{
  "jsonrpc": "2.0", "id": 1,
  "result": {
    "tools": [
      {
        "name": "create_issue",
        "description": "在 GitHub 仓库创建 issue",
        "inputSchema": {
          "type": "object",
          "properties": {
            "title": { "type": "string", "description": "issue 标题" },
            "body": { "type": "string", "description": "issue 正文" },
            "labels": { "type": "array", "items": { "type": "string" } }
          },
          "required": ["title"]
        }
      }
    ]
  }
}

// Client -> Server: 调用工具
{
  "jsonrpc": "2.0", "id": 2, "method": "tools/call",
  "params": {
    "name": "create_issue",
    "arguments": { "title": "登录按钮在 Safari 下错位", "labels": ["bug"] }
  }
}

// Server -> Client: 调用结果
{
  "jsonrpc": "2.0", "id": 2,
  "result": {
    "content": [
      { "type": "text", "text": "已创建 issue #142: 登录按钮..." }
    ],
    "isError": false
  }
}

tools 的 inputSchema 用 JSON Schema 描述参数——这和我们第 148 章讲的工具 schema 完全一致。这种格式对齐意味着 MCP server 的工具可以几乎零成本地映射成 LLM 的 function calling 格式，client 把它们直接塞进 tools 字段发给 LLM 即可。

★ 四、resources：可读取的数据源

resources 是 MCP 的第二类能力——server 暴露的 可读取数据，类似文件但更泛化。一个 resource 用 URI 标识（如 github://repo/user/project/issues/142），client 可以按需读取。

// 列出 resources
{ "jsonrpc": "2.0", "id": 3, "method": "resources/list" }

// 返回
{
  "jsonrpc": "2.0", "id": 3,
  "result": {
    "resources": [
      {
        "uri": "github://issues/142",
        "name": "Issue #142",
        "description": "登录按钮在 Safari 下错位",
        "mimeType": "text/markdown"
      },
      {
        "uri": "db://users/schema",
        "name": "Users 表结构",
        "mimeType": "application/json"
      }
    ]
  }
}

// 读取一个 resource
{
  "jsonrpc": "2.0", "id": 4, "method": "resources/read",
  "params": { "uri": "github://issues/142" }
}

// 返回内容
{
  "jsonrpc": "2.0", "id": 4,
  "result": {
    "contents": [
      {
        "uri": "github://issues/142",
        "mimeType": "text/markdown",
        "text": "# Issue #142: 登录按钮...\n\n## 描述\n..."
      }
    ]
  }
}

resources 和 tools 的关键区别是 触发主体：tools 由 LLM 主动调用（模型决定什么时候需要），resources 由用户或 client 主动引用（用户在对话里 @github://issues/142）。这个区别决定了 client 怎么把它们暴露给交互——tools 进 LLM 的 tools 列表，resources 进 @引用 系统。

resources 还支持 订阅（如果 server 声明了 subscribe 能力）：client 可以订阅某个 resource 的变更，当 resource 内容变化时 server 主动推送通知。这对实时数据（如数据库状态、日志流）很有用。

★ 五、prompts：预定义的提示词模板

prompts 是第三类能力——server 提供的 提示词模板，接受参数，展开成完整的 prompt。这适合「有固定套路」的任务，如代码审查、commit message 生成。

// 列出 prompts
{ "jsonrpc": "2.0", "id": 5, "method": "prompts/list" }

// 返回
{
  "jsonrpc": "2.0", "id": 5,
  "result": {
    "prompts": [
      {
        "name": "code-review",
        "description": "审查一个 PR 的代码",
        "arguments": [
          {
            "name": "pr_url",
            "description": "PR 的 URL",
            "required": true
          },
          {
            "name": "focus",
            "description": "审查重点：security/performance/style/all",
            "default": "all"
          }
        ]
      }
    ]
  }
}

// 获取（展开）一个 prompt
{
  "jsonrpc": "2.0", "id": 6, "method": "prompts/get",
  "params": {
    "name": "code-review",
    "arguments": { "pr_url": "https://github.com/.../pull/142", "focus": "security" }
  }
}

// 返回展开后的 messages
{
  "jsonrpc": "2.0", "id": 6,
  "result": {
    "description": "Security review of PR #142",
    "messages": [
      {
        "role": "user",
        "content": {
          "type": "text",
          "text": "请对以下 PR 进行安全审查...\n\nPR diff:\n..."
        }
      }
    ]
  }
}

prompts 的价值在于 封装领域知识。一个 GitHub MCP server 的 code-review prompt，内部知道怎么拉取 PR diff、怎么组织审查指令、关注哪些安全点——这些复杂逻辑被封装在 server 端，client 只需要传参数就能得到一个专业的审查 prompt。这降低了「写好 prompt」的门槛。

★ 六、sampling：server 反向请求 client 调 LLM

sampling 是 MCP 里最特别的能力——它让 server 反向请求 client 帮它调一次 LLM。这听起来奇怪（client 不是调 server 的吗？），但有真实价值。

场景：一个 GitHub MCP server 在处理「总结这个 issue 的所有评论」时，需要 LLM 来做摘要。但 server 自己没有 LLM API key（也不应该有——key 应该由 client 持有）。sampling 让 server 发一个 sampling/createMessage 请求给 client，client 用自己配置的 LLM 生成回复，返回给 server。

// Server -> Client: 请求 sampling
{
  "jsonrpc": "2.0", "id": 7, "method": "sampling/createMessage",
  "params": {
    "messages": [
      { "role": "user", "content": { "type": "text", "text": "总结：..." } }
    ],
    "modelPreferences": {
      "hints": [{ "name": "claude-sonnet" }],   // 偏好但不强制
      "intelligencePriority": 0.8,               // 重视质量
      "speedPriority": 0.2
    },
    "maxTokens": 200
  }
}

// Client -> Server: 返回 LLM 生成结果
{
  "jsonrpc": "2.0", "id": 7,
  "result": {
    "role": "assistant",
    "content": { "type": "text", "text": "评论主要讨论了..." },
    "model": "claude-sonnet-4",
    "stopReason": "end_turn"
  }
}

sampling 的精妙在于 它保持了「client 是 LLM 访问的唯一入口」这一架构。所有的 LLM 调用都通过 client 发起（client 持有 key、控制成本、记录审计），server 不直接接触 LLM API。这种设计让企业能集中管控 LLM 的使用，而不是让每个 MCP server 各自配 key。

★ 七、三种传输的深度对比

MCP 支持三种传输，各有适用场景：

维度stdioSSE (deprecated)Streamable HTTP
运行模型本地子进程远程长连接HTTP 请求/响应
消息方向stdin/stdout 双向SSE 下行 + POST 上行单端点双向
状态保持进程活着就有连接活着就有session ID 关联
断线恢复重启进程，状态丢重连，状态丢带 session ID 可恢复
部署复杂度低（本地跑）中（要 SSE 支持）低（标准 HTTP）
多客户端一进程一客户端多连接多 session

SSE 传输在较新的 MCP 规范里被标记为 deprecated，推荐迁移到 streamable HTTP。后者用单个 HTTP 端点处理双向消息——client POST 请求，server 可以在响应里返回 SSE 流（如果需要 server 主动推送）。这种设计更兼容现代基础设施（CDN、代理、负载均衡），且支持 session 恢复。

★ 本章小结

MCP 是 Agent 接入外部能力的开放标准。要点回顾：

  • JSON-RPC 2.0：request/response/notification 三种消息，id 匹配机制支持异步乱序。
  • initialize 握手：协商协议版本、双向声明 capabilities、initialized 通知后才正式可用。
  • tools：LLM 主动调用的函数，inputSchema 用 JSON Schema，和 function calling 格式对齐。
  • resources：用户引用的可读数据，URI 标识，支持订阅变更，区别于 tools 的触发主体。
  • prompts：预定义模板，封装领域知识，接受参数展开成 messages。
  • sampling：server 反向请求 client 调 LLM，保持 client 作为 LLM 唯一入口，集中管控。
  • 三种传输：stdio（本地子进程）、SSE（已 deprecated）、streamable HTTP（推荐，支持 session 恢复）。
