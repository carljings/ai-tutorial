System Prompt 设计原则
角色、工具、约束与 few-shot

System prompt 是 Agent 的「宪法」——它定义了 Agent 是谁、能做什么、不能做什么、应该怎么思考。一个精心设计的 system prompt 能让模型的行为稳定可控，而一个糟糕的 prompt 会导致 Agent 跑偏、乱用工具、忽略安全约束。本章我们通过对比 OpenCode、Codex、Claude Code 等项目的 system prompt，总结出四类核心设计原则。

把 system prompt 当代码管理后，下一步是做版本化 A/B 测试：让不同用户随机命中不同 prompt 版本，用下游指标决定哪个更好。这是 prompt 工程走向数据驱动的关键一环。

# Prompt 版本管理与流量分配
import hashlib, random
class PromptRegistry:
    def __init__(self):
        self.versions = {}   # version_id -> prompt_text
        self.traffic = {}    # version_id -> 流量百分比
    def register(self, vid, prompt, pct):
        self.versions[vid] = prompt; self.traffic[vid] = pct
    def assign(self, user_id):
        """用用户id哈希稳定分流(同一用户总命中同版本)。"""
        h = int(hashlib.md5(user_id.encode()).hexdigest(), 16) % 100
        cum = 0
        for vid, pct in self.traffic.items():
            cum += pct
            if h < cum:
                return vid, self.versions[vid]
        return vid, self.versions[vid]
reg = PromptRegistry()
reg.register("v1", "你是一个有用的助手...", 50)
reg.register("v2", "你是一名专家级助手，先思考再回答...", 50)
# 模拟 1000 用户分流, 验证均匀性
from collections import Counter
c = Counter(reg.assign(f"u{i}")[0] for i in range(1000))
print("分流结果:", dict(c))
# 实际使用: 把各版本的 task_success_rate 汇总, 显著更优的胜出。

★ System Prompt 模板生成器

前面通过对比 OpenCode、Codex、Claude Code 的 system prompt 总结出四类设计原则，但读别人的 prompt 和能产出自己的 prompt 是两回事。生产级 system prompt 的最大痛点是太长、太散、难维护——动辄上千字，角色、能力、约束、风格、安全混在一起，改一处怕牵动别处。解决之道是把 prompt 结构化：用一个 dataclass 把它拆成"角色 / 核心能力 / 行为约束 / 可用工具 / 回答风格 / 输出格式 / 安全红线"七段，每段独立维护，最后用一个 render 函数拼装。这样 prompt 就变成了可版本化、可 diff、可单测的代码资产，而不是一段谁都不敢动的散文。

# -*- coding: utf-8 -*-
"""System Prompt 模板生成器：把"角色/能力/约束/风格/安全"
五段式拼装成一份结构化、可版本化的 system prompt。
"""
from dataclasses import dataclass, field
from typing import List

@dataclass
class SystemPromptSpec:
    role: str                          # 角色身份
    expertise: List[str] = field(default_factory=list)   # 能力域
    constraints: List[str] = field(default_factory=list) # 硬约束
    style: str = ""                    # 语气风格
    safety: List[str] = field(default_factory=list)      # 安全红线
    tools: List[str] = field(default_factory=list)       # 可用工具
    format_rules: str = ""             # 输出格式

    def render(self) -> str:
        parts = [f"# 角色\n你是{self.role}。"]
        if self.expertise:
            parts.append("# 核心能力\n" + "\n".join(f"- {e}" for e in self.expertise))
        if self.constraints:
            parts.append("# 行为约束\n" + "\n".join(f"- {c}" for c in self.constraints))
        if self.tools:
            parts.append("# 可用工具\n" + ", ".join(self.tools))
        if self.style:
            parts.append(f"# 回答风格\n{self.style}")
        if self.format_rules:
            parts.append(f"# 输出格式\n{self.format_rules}")
        if self.safety:
            parts.append("# 安全红线\n" + "\n".join(f"- {s}" for s in self.safety))
        return "\n\n".join(parts)

下面用这个生成器产出一个"代码评审 Agent"的完整 system prompt。你会看到，结构化之后每一段都职责清晰：核心能力界定它能干什么、行为约束界定它该怎么干、安全红线界定它不能干什么。这种"能力-约束-红线"的三层结构，是工业级 prompt 的标配。注意一个细节：输出格式段非常关键——明确要求"按文件分节、每条问题标注行号和级别"，能让 Agent 的输出直接被下游工具解析，这是从"聊天机器人"升级到"工程组件"的关键。

# 用模板生成一个"代码评审 Agent"的 system prompt
spec = SystemPromptSpec(
    role="一名严谨的资深代码评审工程师",
    expertise=[
        "精通 Python/TypeScript/Go 的代码规范与常见陷阱",
        "能识别并发、内存、安全漏洞",
        "熟悉 SOLID 原则与设计模式",
    ],
    constraints=[
        "先理解意图再点评，不要为风格细节过度纠缠",
        "每个问题给出具体修改建议和理由，不要泛泛而谈",
        "按严重程度分级: Blocker / Major / Minor / Nit",
    ],
    tools=["read_file", "search_code", "run_tests"],
    style="直接、专业、就事论事；可以指出问题但保持尊重。",
    format_rules="输出 Markdown，按文件分节，每条问题: 行号 - 级别 - 建议。",
    safety=[
        "不输出可被直接利用的攻击代码",
        "涉及密钥/凭证时用 *** 占位",
    ],
)
print(spec.render())

把这种模板化方法用到团队里，能收获三个工程收益。第一，prompt 可 review：每次改 prompt 都像改代码一样提 PR，reviewer 能看清改了哪一段、为什么改，避免"偷偷改了一句导致 Agent 行为漂移"。第二，prompt 可复用：不同 Agent 共享同一段安全红线、同一段风格规范，改一处全量生效，降低维护成本。第三，prompt 可测试：针对每一段写一个"探针用例"（比如专门测安全红线的越狱 prompt），纳入回归测试，确保 prompt 迭代不引入退化。这套方法本质上是把"prompt 工程"升级成"prompt 软件工程"，是 Agent 从 demo 走向生产必经的一步。

★ 一、System Prompt 的四个组成部分

分析多个开源 Agent 的 system prompt，会发现它们虽然措辞各异，但都包含四个核心部分：角色定义、能力描述、行为约束、风格示范。我们逐一拆解。

一个典型的编程 Agent system prompt 结构：

【角色定义】
你是一位资深的软件工程师，正在协助用户开发软件。
你熟悉多种编程语言和框架，能阅读、理解、修改代码。

【能力与工具】
你可以使用以下工具：
- read_file：读取文件内容
- write_file：写入或修改文件
- execute：执行 shell 命令
- search_codebase：语义检索代码
调用工具时，明确说明你为什么要调用它。

【行为约束】
- 修改代码前先阅读相关文件，理解上下文
- 每次改动后运行测试验证
- 不要猜测 API 的行为，不确定时查文档或读源码
- 涉及破坏性操作（删除文件、git push）时先确认
- 失败时分析原因，不要盲目重试

【输出风格】
- 简洁直接，先说结论再展开
- 代码改动用 diff 格式展示
- 解释关键决策的理由
- 不确定时明确说「不确定」

这四部分不是孤立的，它们共同塑造 Agent 的行为。角色定义影响「思维方式」，能力描述影响「工具使用」，行为约束影响「决策边界」，风格示范影响「输出质量」。缺任何一部分，Agent 的行为都会出现盲区。

★ 二、角色定义：锚定 Agent 的「自我认知」

角色定义看似形式化，但它对模型行为有真实影响。LLM 是在大量人类文本上训练的，文本里隐含着「不同角色怎么说话」的模式。当你告诉模型「你是一位资深工程师」，它会激活和「资深工程师」相关的语言模式——更专业、更谨慎、更关注工程实践。

对比两种角色定义的效果差异：

// 弱角色定义
你是一个 AI 助手。帮助用户完成任务。

// 强角色定义（参考 Claude Code 的风格）
你是一位务实的软件工程师，在用户的代码库里工作。
你的首要目标是帮用户解决真实的工程问题，而不是展示知识。
你偏好简单直接的方案，警惕过度设计。
你像一位可靠的同事：守时（不啰嗦）、负责（改完验证）、
坦诚（不确定就说）。

强角色定义有几个特点：具体（不是泛泛的「助手」，而是「工程师」）、有价值倾向（「简单直接」「警惕过度设计」）、有人格化比喻（「可靠的同事」）。这些细节让模型在生成回答时有明确的行为参照，而不是按「通用助手」的模糊默认行事。

一个常被忽视的点：角色定义要和工具能力匹配。如果你给 Agent 的工具是只读的（像 ask 模式），角色定义说「你是资深工程师，负责修改代码」就会造成矛盾——模型会试图改代码但没有工具。Roo Code 的多模式（参见 171 章）之所以有效，正是因为每种模式的角色定义和工具集是协调的。

★ 三、行为约束：用「不要」划定边界

行为约束是 system prompt 里最关键的安全层。它告诉模型 什么不能做、什么要谨慎做。好的约束有几个特征：

第一，具体而非抽象。「要安全」太抽象；「涉及 rm -rf、git push --force、DROP TABLE 时必须先确认用户意图」才具体可用。模型需要可执行的规则，不是原则。

第二，解释理由。不只说「不要」，还要说为什么。「不要在没读文件前就改——你可能改错地方或引入冲突」比单纯的「不要」更有效，因为模型理解了理由后能在边缘情况自行判断。

第三，正向替代。告诉模型「不要做什么」的同时，告诉它「应该做什么」。比如「不要猜测 API 行为——不确定时用 search_codebase 或 read_file 查源码」。

// Claude Code 风格的约束（参考）
约束：
1. 修改前先读。不要基于假设改代码，先 read_file 看实际内容。
   理由：代码可能和你记忆中的不同，特别是大型项目。

2. 小步前进。每次只做一个逻辑改动，改完验证再继续。
   理由：大改动难调试，小改动易回滚。

3. 诚实面对失败。命令失败时读错误信息，分析根因，不要盲目重试。
   理由：盲目重试浪费 token 且可能加重问题。

4. 保护用户。涉及删除、覆盖、推送等不可逆操作时，停下来确认。
   理由：Agent 的错误操作可能造成数据丢失。

5. 不假装全知。不确定时说「我不确定，让我查一下」，然后真的去查。
   理由：编造的答案比承认无知更危险。

★ 四、工具描述：在 prompt 里 vs 在 tools 字段里

现代 LLM API 有专门的 tools 字段，每个工具有 name + description + schema（参见 148 章）。那 system prompt 里还要不要再描述工具？答案取决于 Agent 的复杂度。

简单 Agent（工具少、用途明确）可以完全依赖 tools 字段，system prompt 里不重复。但复杂 Agent 往往会在 system prompt 里加一段 工具使用指南——不是重复每个工具的描述，而是讲工具之间的协作策略：

工具使用指南：

- 优先用 search_codebase 找相关代码，而不是盲目 read_file 试错。
  这能快速定位到正确的文件，省 token。

- 修改文件时，先 read_file 看完整内容，确保你理解了现有结构。
  不要只看片段就改，容易遗漏上下文。

- 改完代码后，立即用 execute 跑相关测试。不要等所有改动都做完再跑——
  早跑早发现问题。

- 当测试失败时，先读完整的错误输出，再决定怎么修。
  不要看到第一个错误就动手，可能有多个相关问题。

- 需要确认 API 用法时，优先 search_codebase 找项目里的现有用法，
  其次 @docs 查官方文档，最后才考虑 @web 搜索。
  项目里的现有用法最贴合当前代码风格。

这段指南的核心是 工具间的优先级和协作，这是单个工具的 description 无法表达的。它教会模型「先做什么再做什么」，而不是孤立地理解每个工具。

★ 五、Few-shot 示例：用范例校准输出

对于复杂的输出格式或行为模式，纯文字描述不如具体范例有效。这就是 few-shot 示例的价值——在 system prompt 里放一两个「正确的输入-输出范例」，让模型模仿。

对于使用 SEARCH/REPLACE 格式的 Agent（Aider、Cline），few-shot 几乎是必须的：

代码修改格式示例：

当你要修改代码时，使用以下格式：

<<<<<<< SEARCH
def greet(name):
    return "Hello"
=======
def greet(name):
    return f"Hello, {name}"
>>>>>>> REPLACE

注意事项（从示例中学习）：
- SEARCH 块要包含足够的上下文（前后各 1-2 行），确保唯一匹配
- 不要包含行号，工具会自己定位
- 一次只改一处，多处改动用多个 SEARCH/REPLACE 块
- REPLACE 块是替换后的完整内容，不是增量

这种「范例 + 注意事项」的组合，比纯文字描述「SEARCH 块的格式是...」有效得多。模型看到具体范例后，能直接模仿格式，减少格式错误。Aider 的实践表明，加了 few-shot 后 SEARCH/REPLACE 的格式正确率从 ~70% 提升到 ~95%。

★ 六、Prompt 版本管理与迭代

System prompt 是会演进的——发现 Agent 在某类任务上表现差，往往要改 prompt。所以 prompt 应该被 版本化管理，像代码一样迭代。一些最佳实践：

  • prompt 独立于代码：把 system prompt 放在单独的文件（如 prompts/system.md），而不是硬编码在源码字符串里。这样非工程师（如产品经理）也能参与 prompt 迭代。
  • 版本号：每个 prompt 版本打标签，配合评估集衡量版本间的效果差异。
  • 评估集：维护一组「标准任务 + 预期行为」的测试用例，每次改 prompt 后跑一遍，确保改进没有引入回归。
  • 差异化：不同模型可能需要不同的 prompt 措辞（Claude 对 XML 敏感，GPT 对 Markdown 敏感），支持按模型切换 prompt 变体。

★ 七、常见反模式

最后总结几个 system prompt 的常见反模式，避免踩坑：

  • 过度啰嗦：prompt 写了几千字，模型反而抓不住重点。研究表明 prompt 越长，模型对中间部分的遵循度越低（lost-in-the-middle）。保持 prompt 简洁，核心约束放前面。
  • 矛盾指令：一会儿说「要高效」一会儿说「要谨慎」，模型无所适从。确保指令间不冲突。
  • 空洞口号：「你要做到最好」「追求卓越」这种没有可操作性的话，对模型行为无影响。换成具体的「每次改动后跑测试」。
  • 忽略安全：只讲「要做什么」，不讲「不能做什么」。安全约束是 prompt 的必备部分。
  • 不可测试：写了 prompt 但没有评估集，改完不知道是变好还是变坏。prompt 迭代必须配合评估。

★ 本章小结

System prompt 是 Agent 行为的源头。要点回顾：

  • 四部分：角色定义（自我认知）、能力描述（工具）、行为约束（边界）、风格示范（few-shot），缺一不可。
  • 角色要具体：有价值观、有人格化比喻、和工具能力匹配，避免泛泛的「助手」。
  • 约束要可执行：具体而非抽象、解释理由、给正向替代，「不要猜测，去查源码」优于「要准确」。
  • 工具指南讲协作：不只重复 tools 字段，而是讲工具间的优先级和流程。
  • few-shot 校准格式：复杂输出格式用范例比描述有效，正确率显著提升。
  • 版本化管理：prompt 独立成文件、打版本号、配评估集、支持模型差异化。
  • 避免反模式：啰嗦、矛盾、空洞、忽略安全、不可测试。
