★ 借鉴人脑：记忆的认知模型

AI 记忆系统的设计大量借鉴认知科学，把人脑的记忆机制映射到工程：

认知科学AI 工程对应特征
工作记忆（Working Memory）上下文窗口容量小（7±2）、当前任务、易失
短期记忆（Short-term）会话历史/摘要几分钟~几小时、滚动更新
长期记忆（Long-term）向量库/知识库/LLM Wiki持久、需检索、容量大
情景记忆（Episodic）历史交互日志「上次发生了什么」
语义记忆（Semantic）事实/偏好库「用户喜欢简洁」
程序记忆（Procedural）Skills（第 8 章）「怎么做某事」

02 记忆的读写循环

memory_system.py —— 完整的读写记忆循环
"""
memory_system.py —— 带重要性判断、去重、检索的记忆系统
"""
class SmartMemory:
    def __init__(self, embedder, llm):
        self.embedder = embedder
        self.llm = llm
        self.store = []   # (content, embedding, importance, timestamp)

    def recall(self, query, top_k=5):
        """检索相关记忆（相关性 × 重要性 × 时效性）"""
        q_emb = self.embedder.encode([query])[0]
        scored = []
        import time
        now = time.time()
        for content, emb, imp, ts in self.store:
            relevance = cosine(q_emb, emb)
            recency = 1.0 / (1 + (now - ts) / 86400)  # 越新分越高
            score = relevance * 0.5 + imp * 0.3 + recency * 0.2
            scored.append((score, content))
        scored.sort(reverse=True)
        return [c for _, c in scored[:top_k]]

    def remember(self, interaction):
        """从交互中抽取值得记的，去重后存入"""
        # ① 让 LLM 判断什么值得记
        facts = self.llm(f"从这次交互抽取值得长期记住的事实（JSON 数组，空则[]）：{interaction}")
        import json
        for fact in json.loads(facts):
            emb = self.embedder.encode([fact])[0]
            # ② 去重：相似度高的更新而非新增
            is_dup = False
            for i, (_, e, _, _) in enumerate(self.store):
                if cosine(emb, e) > 0.9:
                    self.store[i] = (fact, emb, 0.8, time.time())  # 更新
                    is_dup = True; break
            if not is_dup:
                self.store.append((fact, emb, 0.7, time.time()))

03 记忆的四大挑战

📦

记忆爆炸

记太多 → 检索慢、噪音多。解法：重要性过滤、定期压缩、遗忘。

⚔️

记忆冲突

「用户喜欢简洁」vs「用户要详细」。解法：置信度、时间戳、冲突检测。

🕐

时效性

旧记忆可能过时。解法：时效衰减、主动更新、TTL 过期。

🔒

隐私

不该记的（敏感/临时）被记住。解法：选择性记忆、用户可删、遗忘权。

04 遗忘的艺术

好的记忆系统和人脑一样，需要遗忘：

  • 重要性衰减：长期不被检索的记忆，重要性降低；
  • 主动清理：定期删除低重要性、过时、重复的记忆；
  • 压缩合并：把多条细碎记忆合并成一条摘要；
  • 用户控制：用户可以查看、删除自己的记忆（隐私权）。

06 持久化长期记忆：从内存到向量库

前面的 SmartMemory 把记忆存在进程内的列表里，进程一退出就全没了。生产环境必须把长期记忆落到独立存储。常见三种方案各有取舍：

存储后端写入语义检索典型场景
Redis（带 RediSearch）极快支持向量 + 全文高并发会话记忆、在线 Agent
PostgreSQL + pgvector快向量 + 结构化过滤需要事务、要 join 用户表
Milvus / Qdrant中亿级向量专用全局共享记忆、大规模召回

下面给出一份基于 pgvector 的可落地的长期记忆存储抽象，把上一节的「相关性×重要性×时效」直接写进 SQL 检索，由数据库一次性算分。

long_term_memory.py —— 基于 pgvector 的持久化记忆
import time, json, asyncio
import asyncpg            # pip install asyncpg

CREATE_SQL = """
CREATE EXTENSION IF NOT EXISTS vector;
CREATE TABLE IF NOT EXISTS memory (
    id          BIGSERIAL PRIMARY KEY,
    user_id     TEXT NOT NULL,
    content     TEXT NOT NULL,
    embedding   vector(1024) NOT NULL,
    importance  REAL DEFAULT 0.5,
    confidence  REAL DEFAULT 0.5,       -- 置信度，解决冲突用
    created_at  TIMESTAMPTZ DEFAULT now(),
    last_used   TIMESTAMPTZ DEFAULT now(),
    hit_count   INT DEFAULT 0,
    ttl_days    INT                     -- 为 NULL 表示永久
);
CREATE INDEX ON memory USING hnsw (embedding vector_cosine_ops);
CREATE INDEX ON memory (user_id, importance DESC);
"""

class PgVectorMemory:
    def __init__(self, dsn, embedder):
        self.dsn = dsn
        self.embedder = embedder
        self.pool = None

    async def init(self):
        self.pool = await asyncpg.create_pool(self.dsn)
        async with self.pool.acquire() as conn:
            await conn.execute(CREATE_SQL)

    async def write(self, user_id, content, importance=0.5, confidence=0.5):
        """写一条记忆。语义相近且置信度更高的旧记忆会被更新而非新建"""
        emb = self.embedder.encode([content])[0].tolist()
        async with self.pool.acquire() as conn:
            # 检查近似重复（cosine 距离 < 0.15）
            dup = await conn.fetchval("""
                SELECT id FROM memory
                WHERE user_id=$1 AND embedding <> $2 < 0.15
                ORDER BY embedding <> $2 LIMIT 1
            """, user_id, str(emb))
            if dup:
                await conn.execute("""
                    UPDATE memory SET content=$1, embedding=$2,
                        importance=GREATEST(importance,$3),
                        confidence=GREATEST(confidence,$4),
                        created_at=now()
                    WHERE id=$5
                """, content, str(emb), importance, confidence, dup)
                return dup
            return await conn.fetchval("""
                INSERT INTO memory(user_id, content, embedding, importance, confidence)
                VALUES ($1,$2,$3,$4,$5) RETURNING id
            """, user_id, content, str(emb), importance, confidence)

    async def retrieve(self, user_id, query, top_k=5):
        """相关性×重要性×时效性，一次 SQL 搞定"""
        q = self.embedder.encode([query])[0].tolist()
        rows = await self.pool.fetch("""
            SELECT content,
                   (1 - (embedding <> $1)) * 0.5                  AS rel,
                   importance * 0.3                                AS imp,
                   (1.0 / (1 + EXTRACT(EPOCH FROM now()-created_at)/86400.0)) * 0.2 AS rec
            FROM memory
            WHERE user_id=$2
              AND (ttl_days IS NULL OR created_at > now() - (ttl_days||' days')::INTERVAL)
            ORDER BY (rel + imp + rec) DESC
            LIMIT $3
        """, str(q), user_id, top_k)
        # 顺手更新 last_used / hit_count
        return [r["content"] for r in rows]

注意 <> 是 pgvector 的余弦距离算符；我们用 1 - 距离 把它转成相似度。GREATEST(importance, $3) 保证了「同一事实被多次提及，重要性只升不降」，这是 Generative Agents 论文里采用的策略。

07 检索策略：不止是 top-k

「记忆爆炸」之所以难治，是因为检索的难点不在于召回多少条，而在于召回后怎么用。三种进阶策略，按复杂度从低到高：

策略做法适用
分数阈值只保留得分 > 0.6 的记忆记忆量小、噪音少
重排（rerank）向量召回 50 条，用 cross-encoder 精排到 5 条记忆量大、对精度敏感
分层检索先查语义记忆（事实），再查情景记忆（事件），最后合并结构化记忆系统

下面是分层检索的一个典型陷阱：用户问「上次提到的那个项目怎么样了」，事实层（语义记忆）查不到，必须去情景层找历史事件。把两类记忆混在一个表里、用同一套 embedding 检索，恰恰是新手最常踩的坑。

hybrid_retrieval.py —— 语义 + 情景双通道
async def hybrid_recall(memory, user_id, query, llm, top_k=5):
    """先做意图分类，决定从哪个通道召回，再合并"""
    intent = llm("判断查询属于 fact（问事实/偏好）还是 event（问上次/那次发生的事），"
                 "只回 fact 或 event：" + query).strip().lower()

    # 通道 1：语义记忆 —— 高置信度、稳定的事实
    facts = []
    if intent == "fact":
        facts = await memory.retrieve(user_id, query, top_k=top_k,
                                       table="semantic_memory")

    # 通道 2：情景记忆 —— 带时间戳的事件日志
    events = []
    if intent == "event":
        # 事件检索要重排时间，越近越优先
        events = await memory.fetch_recent_events(user_id, query, limit=10)

    # 通道 3：兜底 —— 两个通道都空，做一次宽松检索避免「失忆」
    if not facts and not events:
        facts = await memory.retrieve(user_id, query, top_k=top_k)

    return {"facts": facts, "events": events}

08 主动遗忘机制

第 04 节讲了「为什么要遗忘」，这里讲「怎么具体实现」。生产系统通常同时跑三种遗忘：被动衰减、定期压缩、显式 TTL。压缩这一步尤其关键——它不是简单删除，而是把 N 条旧记忆用 LLM 摘成 1 条，把信息密度提上去。

forget.py —— 衰减 + 压缩 + TTL 三重遗忘
async def passive_decay(memory):
    """每周跑一次：长期未命中的记忆 importance 衰减"""
    await memory.pool.execute("""
        UPDATE memory SET importance = importance * 0.85
        WHERE last_used < now() - INTERVAL '7 days'
    """)
    # importance 太低就清理，但保留永久标记的
    await memory.pool.execute("""
        DELETE FROM memory
        WHERE importance < 0.05 AND ttl_days IS NULL
    """)

async def compress_old_memories(memory, user_id, llm, batch=30):
    """把同一用户的老旧记忆批量摘要合并，降低条目数"""
    rows = await memory.pool.fetch("""
        SELECT id, content FROM memory
        WHERE user_id=$1 AND created_at < now() - INTERVAL '30 days'
        ORDER BY created_at LIMIT $2
    """, user_id, batch)
    if len(rows) < 5:
        return 0
    joined = "\n- ".join(r["content"] for r in rows)
    summary = llm("把以下记忆条目合并为一条不超过 200 字的高密度摘要，保留所有事实和偏好：\n- "
                  + joined)
    # 写入新摘要，删掉旧条目
    await memory.write(user_id, summary, importance=0.8)
    ids = ",".join(str(r["id"]) for r in rows)
    await memory.pool.execute(f"DELETE FROM memory WHERE id IN ({ids})")
    return len(rows)

# 主动遗忘的 cron 作业
async def nightly_forget_job(memory, llm):
    await memory.pool.execute("DELETE FROM memory WHERE ttl_days IS NOT NULL "
                              "AND created_at < now() - (ttl_days||' days')::INTERVAL")
    await passive_decay(memory)
    for uid in await memory.active_users():
        await compress_old_memories(memory, uid, llm)

这三种机制互为补充：TTL 处理显式声明的临时记忆（如「这个会议纪要保留 7 天」）；衰减 处理自然老化；压缩 把历史细节降维成高密度摘要，既节省存储又提升检索精度。三者都默认跳过 ttl_days IS NULL 的永久记忆，保证用户设定的核心偏好不会被误删——这是隐私合规的红线。

深 长期记忆架构：写入、遗忘与一致性

给 Agent 加“长期记忆”听起来美好，落地却处处是坑：记忆写多了检索噪声大、写少了等于没记；用户改了偏好，旧记忆却不更新；多轮冲突的事实并存，模型无所适从。下面把记忆系统的三个核心难题拆开讲。

记忆的写入策略：不是什么都该记

最差的记忆系统是把每轮对话都存进向量库——检索时全是噪声，有用信息被淹没。正确做法是写入前先判断“是否值得记”：只提取事实、偏好、决策这类稳定信息，过滤掉寒暄、临时查询。

import json
from openai import OpenAI
client = OpenAI(base_url="https://api.z.ai/api/paas/v4", api_key="YOUR_KEY")

EXTRACT_PROMPT = '''分析对话，提取值得长期记住的事实/偏好，输出JSON数组。
只提取稳定信息(姓名、偏好、重要决策)，忽略临时问题和寒暄。
无值得记忆的内容则返回 []。示例: [{"type":"preference","k":"diet","v":"素食"}]'''

def extract_memories(dialogue):
    r = client.chat.completions.create(model="glm-4-plus", temperature=0.0,
        messages=[{"role":"system","content":EXTRACT_PROMPT},
                  {"role":"user","content":dialogue}])
    try: return json.loads(r.choices[0].message.content)
    except: return []

def should_write(memory, existing):
    """去重与冲突检测：新记忆与已有的是否冲突。"""
    for e in existing:
        if e["k"] == memory["k"]:
            if e["v"] != memory["v"]:
                return ("CONFLICT", e)     # 冲突，需更新而非新增
            return ("DUPLICATE", e)        # 重复，丢弃
    return ("NEW", None)

new = extract_memories("我叫张三，我对花生过敏，最近在减肥。")

这套“提取 + 去重 + 冲突检测”是可用记忆系统的最小骨架。关键是写入是有门槛的：经过价值判断、去重、冲突处理后，才落库，确保记忆库干净。

遗忘机制：会记也要会忘

人类记忆之所以有用，很大程度因为会遗忘——把过时、低频的信息淡忘，才能让重要的凸显。Agent 记忆也应如此。实现“遗忘”有三种机制：一是TTL 过期，临时信息（如“今天天气”）设较短过期时间；二是访问频次衰减，长期未被检索命中的记忆降低权重，最终归档；三是显式更新覆盖，检测到冲突时新值覆盖旧值（用户说“我现在不减肥了”就应更新 goal 字段）。三者结合，记忆库才能长期保持“新鲜且相关”。一个常被忽略的点是记忆也要做版本管理——记录每条记忆的来源对话和时间戳，出问题时能追溯是哪次交互写错了，而不是面对一锅无来源的“记忆汤”无从排查。

Q1：记忆越积越多，检索反而变差了，怎么办？

典型的记忆膨胀导致噪声淹没信号。解法是主动遗忘：TTL 过期（临时信息短命）、访问频次衰减（长期未命中的降权归档）、重要度评分（低价值记忆淘汰）。还要定期整理（合并重复、更新过时、提炼摘要）。关键认知：好的记忆系统会忘，像人一样——把不重要的淡忘，重要的才凸显。无脑全存的记忆库，不如精心管理的少量高质量记忆。

Q2：用户改了偏好（如「我现在不减肥了」），旧记忆怎么更新？

必须做冲突检测与覆盖：写入新记忆前，检索同 key 的旧记忆，发现值不同则更新而非新增（避免新旧并存模型困惑）。更稳的做法是时序版本化——保留历史但标注「最新」，检索时优先最新值。还要处理隐式偏好变化：用户最近的行为若和记忆矛盾（记忆说喜欢简洁，但最近总要求详细），应触发偏好重判。冲突处理是记忆系统从「能记」到「记准」的关键，不做就是一锅矛盾的记忆汤。
