OpenCode 工具系统
从 JSON Schema 到 Tool.execute 的完整链路

Agent 和普通聊天机器人的本质区别，在于它能「动手」——调用工具来读写文件、执行命令、检索信息。OpenCode 把这种能力抽象成了一套统一的 Tool 接口：每个工具用 JSON Schema 声明参数，用 async 函数执行逻辑，返回结构化结果。本章我们从 read_file、write_file、execute 这三个核心工具的源码出发，看清一个工具从「LLM 想调用」到「真正执行完」走过了哪些环节。

★ 一、Tool 接口：一个工具长什么样

在 OpenCode 里，所有工具——无论是内置的还是 MCP 提供的还是插件注册的——都遵循同一个接口契约。这个契约最关键的部分是三样东西：name（LLM 用它来引用工具）、description（告诉 LLM 这个工具能干嘛）、schema（参数的 JSON Schema，约束 LLM 传什么进来）。加上一个 execute 函数，就构成了一个完整工具。

// packages/opencode-core/src/tool/types.ts（简化）
export interface Tool<TParams = any, TResult = any> {
  name: string;                  // 如 "read_file"，LLM 在 tool_call 里引用
  description: string;           // 给 LLM 看的说明，影响调用决策
  schema: {                      // JSON Schema，描述参数
    type: "object";
    properties: Record<string, JSONSchema>;
    required?: string[];
  };
  execute: (params: TParams, ctx: ToolContext) => Promise<ToolResult<TResult>>;
}

export interface ToolContext {
  session: Session;              // 当前会话，可以读 messages、cwd
  signal: AbortSignal;           // 中断信号，长任务要检查
  // ...日志、权限检查回调等
}

export interface ToolResult<T> {
  content: Array<
    | { type: "text"; text: string }
    | { type: "image"; data: string; mimeType: string }
  >;
  isError?: boolean;             // true 表示执行失败，但仍按正常返回给 LLM
}

注意 isError 这个字段的设计哲学：工具执行失败（比如文件不存在）不被抛成异常，而是被包装成一个 isError=true 的正常返回。这样 LLM 能读到错误信息，自行决定下一步——重试、换个路径、还是告诉用户。如果抛异常，整个 Agent 循环会中断，LLM 没机会自我修正。这是 Agent 工具设计的一条重要经验：把错误也当成一种返回值。

★ 二、read_file：最基础也最讲究的工具

read_file 看起来简单——读个文件嘛——但它是 LLM 获取代码上下文的最高频工具，所以 OpenCode 在它身上花了不少心思。我们看一个接近真实实现的版本：

// packages/opencode-core/src/tool/read_file.ts（简化）
export const readFileTool: Tool = {
  name: "read_file",
  description: `读取本地文件内容。支持指定起始行和读取行数。
注意：路径相对于会话工作目录（cwd）。二进制文件会返回提示而非乱码。`,
  schema: {
    type: "object",
    properties: {
      filePath: {
        type: "string",
        description: "要读取的文件的绝对或相对路径",
      },
      startLine: {
        type: "integer",
        description: "起始行号（从 1 开始），默认 1",
      },
      lineCount: {
        type: "integer",
        description: "读取的行数，默认读整个文件",
      },
    },
    required: ["filePath"],
  },
  execute: async (params, ctx) => {
    const absPath = resolve(ctx.session.cwd, params.filePath);

    // 1. 路径校验：禁止跳出 cwd（安全护栏）
    if (!absPath.startsWith(ctx.session.cwd)) {
      return {
        content: [{ type: "text", text: `错误：路径越界，禁止读取 ${absPath}` }],
        isError: true,
      };
    }

    // 2. 读取文件（用 Bun 的 API，二进制安全）
    const bytes = await Bun.file(absPath).arrayBuffer();

    // 3. 检测是否为二进制
    if (isBinary(bytes)) {
      return {
        content: [{ type: "text", text: `这是一个二进制文件（${bytes.length} 字节），无法以文本展示。` }],
        isError: false,
      };
    }

    // 4. 按行切片
    const lines = new TextDecoder().decode(bytes).split("\n");
    const start = (params.startLine ?? 1) - 1;
    const end = params.lineCount ? start + params.lineCount : lines.length;
    const selected = lines.slice(start, end);

    // 5. 带行号格式化输出（LLM 很依赖行号来定位编辑位置）
    const numbered = selected
      .map((line, i) => `${String(start + i + 1).padStart(6)}\t${line}`)
      .join("\n");

    return { content: [{ type: "text", text: numbered }] };
  },
};

这里有两个设计细节值得细品。第一是行号注入：输出是 1\t第一行内容 这种带 tab 分隔的行号格式。这不是为了好看，而是因为 read_file 通常和后面要讲的 write_file（它用行号定位 SEARCH/REPLACE）配合使用，稳定的行号让 LLM 能精确指定编辑位置。

第二是二进制检测：如果直接把一个 .png 或 .exe 的字节当 UTF-8 解码，会得到一堆乱码字符，不仅浪费 token 还可能触发模型的 tokenization 异常。OpenCode 用一个简单的启发式（检测前 N 字节里的 NULL 或非打印字符比例）来判断二进制，提前返回友好提示。

★ 三、write_file：从覆盖到 patch 的演进

write_file 是 Agent 修改代码的主要手段。最朴素的实现就是「给定路径 + 完整内容，覆盖写入」。但这有个严重问题：LLM 必须在每次编辑时把整个文件重写一遍，对于几千行的文件，光输出 token 就是巨大的浪费，而且容易在某一行抄错导致整个文件损坏。

所以 OpenCode 的 write_file 实际上支持两种模式：完整覆盖和基于 SEARCH/REPLACE 的补丁。后者借鉴自 Aider 的 diff 格式——LLM 只需要给出「要替换的旧代码块」和「新代码块」，工具在本地做字符串匹配替换。

// write_file 的 patch 模式核心逻辑（简化）
async function applyPatch(absPath: string, patch: string): Promise<ToolResult> {
  let original = await Bun.file(absPath).text();

  // patch 格式：
  //   <<<<<<< SEARCH
  //   要被替换的旧代码
  //   =======
  //   替换后的新代码
  //   >>>>>>> REPLACE
  const SEARCH_RE = /<<<<<<<\s*SEARCH\n([\s\S]*?)\n=======\n([\s\S]*?)\n>>>>>>>\s*REPLACE/g;

  let match;
  let applied = 0;
  while ((match = SEARCH_RE.exec(patch)) !== null) {
    const [, oldText, newText] = match;
    if (!original.includes(oldText)) {
      return {
        content: [{ type: "text",
          text: `替换失败：找不到以下代码块（可能缩进或空格不一致）：\n${oldText}` }],
        isError: true,
      };
    }
    original = original.replace(oldText, newText);
    applied++;
  }

  await Bun.write(absPath, original);
  return {
    content: [{ type: "text", text: `成功应用 ${applied} 处修改到 ${absPath}` }],
  };
}

这段代码揭示了一个关键的容错策略：当 SEARCH 块匹配不到时，不抛异常，而是把匹配失败的那段代码原样返回给 LLM。LLM 看到这个反馈后，通常会重新读文件、修正缩进后重试。这种「失败 → 反馈 → 重试」的循环是 Agent 自我修正能力的基础，我们在第 175 章会专门讨论。

★ 四、execute：执行 shell 命令的工具

execute（有些版本叫 bash 或 run_command）让 Agent 能跑任意 shell 命令——跑测试、装依赖、git 操作。这是威力最大的工具，也是最危险的，所以它的实现里安全和超时是重点。

// packages/opencode-core/src/tool/execute.ts（简化）
export const executeTool: Tool = {
  name: "execute",
  description: "在工作目录执行 shell 命令，返回合并的 stdout/stderr。",
  schema: {
    type: "object",
    properties: {
      command: { type: "string", description: "要执行的命令" },
      timeout: { type: "integer", description: "超时毫秒数，默认 120000" },
    },
    required: ["command"],
  },
  execute: async (params, ctx) => {
    const proc = Bun.spawn(["bash", "-c", params.command], {
      cwd: ctx.session.cwd,
      stdout: "pipe",
      stderr: "pipe",
    });

    // 用 AbortController 实现超时
    const timer = setTimeout(() => proc.kill(), params.timeout ?? 120000);
    ctx.signal.addEventListener("abort", () => proc.kill());

    const [stdout, stderr, exitCode] = await Promise.all([
      new Response(proc.stdout).text(),
      new Response(proc.stderr).text(),
      proc.exited,
    ]);
    clearTimeout(timer);

    const output = stdout + (stderr ? `\n[stderr]\n${stderr}` : "");
    return {
      content: [{
        type: "text",
        text: `[exit ${exitCode}]\n${output.slice(0, 50000)}`  // 截断防 token 爆炸
      }],
      isError: exitCode !== 0,
    };
  },
};

三个细节：输出截断（防止一条 find / 爆掉 context window）、外部 abort 信号（用户按 Ctrl+C 时 ctx.signal 触发，进程被 kill）、exit code 作为 isError（非零退出码让 LLM 知道命令失败了）。这些细节叠加起来，才让 execute 在「强大」和「可控」之间取得平衡。

★ 五、为什么 schema 的措辞决定工具是否被正确调用

很多开发者写工具描述时随便写一句「读取文件」就完事，然后纳闷为什么 LLM 总是用错参数。OpenCode 的工具描述有一个明确的原则：把约束、默认值、注意事项都写进 description。因为 LLM 选择工具和填参数，完全依赖它读到的文字描述——schema 里的 description 字段，本质上是给 LLM 看的「使用说明书」。

举个例子，read_file 的 schema 里 startLine 的描述是「起始行号（从 1 开始），默认 1」。这个「从 1 开始」至关重要——如果不写，有些模型会以为是 0-based，导致读取偏移一行。这种细节就是工具系统的「暗坑」，OpenCode 在源码里把每个字段都描述得很啰嗦，正是为了减少 LLM 的歧义。

★ 六、tools 列表：动态拼装与发送给 LLM

一个 Session 里可用的工具，不是写死的。它是三部分的并集：内置工具（read_file 等）、当前 session 加载的 MCP server 提供的工具、以及已启用插件注册的工具。每次发起 LLM 请求前，OpenCode 会把这些工具的 name + description + schema 汇总成一个数组，塞进请求的 tools 字段。

// 收集 session 可用工具（简化）
function collectTools(session: Session): Tool[] {
  const builtin = [readFileTool, writeFileTool, executeTool, listDirTool, /* ... */];
  const fromMcp = session.mcpServers.flatMap(s => s.tools);
  const fromPlugins = pluginRegistry.getTools();
  return [...builtin, ...fromMcp, ...fromPlugins];
}

// 转成 LLM API 需要的格式
function toLLMTools(tools: Tool[]) {
  return tools.map(t => ({
    type: "function",
    function: {
      name: t.name,
      description: t.description,
      parameters: t.schema,
    },
  }));
}

注意名字冲突的处理：如果两个来源都有同名工具，OpenCode 会给 MCP 工具加前缀（如 mcp__filesystem__read_file），避免覆盖内置工具。这种前缀策略后来也被 Claude Code 等其他 Agent 采用，成了事实约定。

★ 本章小结

OpenCode 的工具系统是一套高度统一的抽象。核心要点回顾：

  • 统一接口：所有工具遵循 name + description + JSON Schema + async execute 的契约，错误也走正常返回（isError=true），不中断循环。
  • read_file 讲究：注入行号方便后续编辑、二进制检测避免乱码、路径越界检查是安全护栏。
  • write_file 双模式：完整覆盖适合小文件，SEARCH/REPLACE 补丁省 token 且匹配失败时反馈原文让 LLM 重试。
  • execute 三重防护：输出截断、超时 kill、外部 abort 信号，让强大命令执行工具保持可控。
  • 描述即文档：schema 里的 description 是给 LLM 看的使用说明，措辞精度直接影响调用正确率。
  • 动态工具列表：内置 + MCP + 插件三者并集，名字冲突时给 MCP 加前缀。

下一章（149 章）我们会继续深入工具之上的一层——权限模型。工具定义好了不等于能随便调用，OpenCode 用 allow/ask/deny 三态把工具的每一次执行都纳入了用户掌控。
