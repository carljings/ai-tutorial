资源治理

把向量库塞进 200MB 内存预算的工程艺术

深度进阶★★★

端侧资源约束：一组完全不同的数字

在云端，你管向量库的口头禅是"加机器""加内存"。在端侧——手机、IoT、车载、可穿戴——这套失效了。设备出厂那一刻，内存、CPU、磁盘、电量就定死了，你的向量库只能在这个硬预算内工作。理解这组数字，是端侧资源治理的起点：

设备可用内存CPU存储电量约束
旗舰手机1-2 GB（给你的 App）4-8 核 ARM几十 GB后台查询不能频繁
中端手机300-800 MB4-6 核几十 GB严格
IoT/树莓派256-512 MB2-4 核几 GB常电
智能音箱/可穿戴64-128 MB1-2 核< 1 GB极严
车载 IVI1-4 GB4-8 核几十 GB常电

对比一下云端：一台向量库服务器通常 64-256 GB 内存、32+ 核、NVMe SSD。端侧旗舰手机给你的预算可能只有云端的 1/100。所以"在云端能跑"的方案，到了端侧往往直接 OOM（内存溢出）或卡顿。zvec 在资源治理上的设计，本质就是"在这组硬预算内，最大化能容纳的向量数和查询质量"——这也是为什么 zvec.init() 直接暴露 memory_limit_mb 和 query_threads 参数：它把自己当成进程里的一个"资源租户"来管，而不是无节制地抢内存。

💡 要点
端侧资源治理的核心命题：内存是硬上限、CPU 是软约束、电量是隐性成本、存储看场景。四者里"内存"几乎永远是第一约束——所有优化最终都在回答"怎么用更少的内存装下更多/更准的向量"。zvec 的 memory_limit_mb 参数就是这条约束的入口。

内存预算分配：一个典型 App 的内存账本

假设一个手机 App 分到 500 MB 内存预算（iOS/Android 对单个 App 的常见上限），这套预算要在 App 的所有组件间分配，向量库只占一部分。一份现实的账本：

组件内存占用说明
UI / WebView / 框架150 MB系统框架、渲染、JS 引擎
LLM 运行时（7B Q4 量化）200 MBQwen2.5-7B int4 量化
嵌入模型100 MBbge-small-zh 权重
向量库（zvec）50 MB← 你能用的就这么点
缓冲/临时余量查询候选、WAL 缓冲

是的，向量库在端侧的真实预算常常只有几十 MB。在这个预算下，要装下"用户的知识库"——可能是几千到几十万个 chunk。50 MB 装 10 万个 768 维向量？原始 float32 一个就要 3KB，10 万个 = 300 MB，根本装不下。这就是为什么端侧向量库必须重度依赖量化——而 zvec.init(memory_limit_mb=...) 的存在，就是让你明确告诉它这个预算，它据此选择压缩策略。

量化到极限：把内存压到 1/4 甚至 1/8

在上面的预算下，zvec 必须用量化把向量压下来。一个 768 维向量的"内存账"在不同存储格式下的变化：

存储格式单向量大小10 万向量总占用50MB 预算能装多少
float (FP32)3072 B (3 KB)300 MB1.6 万
float161536 B (1.5 KB)150 MB3.3 万
int8（标量量化）768 B75 MB6.5 万
int4（标量量化）384 B38 MB13 万
PQ（乘积量化）~32-96 B3-9 MB50-150 万

这张表是端侧量化的决策依据，也和 zvec 官方 benchmark 的选择一致——1M 和 10M benchmark 默认都用 int8 存储。如果你要在 50 MB 里装 10 万向量，float 根本不行（要 300 MB），float16 也不行（150 MB），int8 勉强（75 MB，略超预算），int4 刚好（38 MB），PQ 轻松（几 MB）。所以端侧的默认选择通常是 int8 或 int4，绝少用 float。

💡 要点
端侧量化口诀：内存紧 → int8（4× 压缩、几乎无损，官方 benchmark 默认）；内存极紧 → int4（8× 压缩、中损失）；追求极致 → PQ + refiner 精排（压缩比之王，但要 rerank）。float 在端侧是奢侈品，几乎不用。

两级检索（refiner）在端侧的内存经济学

激进量化（int4/PQ）配合 zvec 的 refiner（src/db/reranker/），在端侧有特殊的内存经济学：粗筛索引（量化后的码）极小，可以全驻内存；refiner 用的更高精度向量量大，可以放磁盘/闪存 mmap 按需加载。这样"热数据"（常被查到的）自然留在 page cache 里，冷数据落盘不占内存。这是 zvec 在内存极紧设备上的核心策略，也是官方 10M benchmark 开 --is-using-refiner 的原因。

端侧两级检索的内存布局（配合 zvec.init(memory_limit_mb=...)）

[内存]  约 50 MB 预算（memory_limit_mb=50）
   ├── int8 粗筛索引（10万向量 × 768B）  ≈ 75 MB → 压到预算内
   ├── HNSW 图结构                       ≈ 8 MB   ← 全驻内存
   └── page cache（热数据）              ≈ 余量

[磁盘/闪存]  kb.zv 数据文件
   └── 完整向量表（WAL + checkpoint 落盘）  ← mmap 按需 page-in

查询时：
  int8 粗筛（内存）→ 200 候选
  → refiner 精排（从磁盘 mmap 取这 200 条）→ top-k
  热查询的 200 条大概率已在 page cache，几乎零磁盘 I/O

CPU 与 SIMD：让 ARM 也能飞起来

端侧 CPU 是 ARM 架构，主频和核数都不如服务器 x86。但 ARM 的 NEON SIMD 单元很强大——一条指令能并行处理 4-8 个 float32 或 16-32 个 int8。zvec 内核（继承自 Proxima）的距离计算（L2、内积、余弦）在 src/core/metric/ 里有专门的 SIMD 实现，覆盖 AVX2/AVX-512（x86）和 NEON（ARM）。这让 int8 的距离计算能在 ARM 上跑到很高吞吐。

一个端侧特有的优化：尽量用 int8 而不是 float 距离计算。int8 点积用 NEON 可以一条指令算 16 维，float32 只能算 4 维，差 4 倍。这就是为什么 int8 量化在端侧不仅省内存，还省 CPU——计算也快了。这是量化在端侧的"双重红利"，也是 Proxima "扁平索引 + 硬件优化"哲学的具体落地。

距离类型NEON 吞吐（理论）端侧适用
float32 点积4 维/周期精度优先、内存够
float16 点积8 维/周期内存中等
int8 点积16 维/周期端侧默认（又快又省）
二值汉明128 位/周期粗筛阶段

写入限流与背压：保护设备体验

端侧还有一个云端没有的问题：你的向量库不能"独占"设备资源。用户可能在用 App 的同时还在刷微信、看视频，你的索引构建如果占满 CPU，整个手机会卡。zvec 应对的方式：① zvec.init(query_threads=..., optimize_threads=...) 显式限定线程数，留出核给前台；② 把大的 ingest 拆成小批次，在设备空闲时（安卓 WorkManager、iOS BGTask）执行；③ 利用 memory_limit_mb 让 zvec 在系统内存压力高时主动退让。

# 端侧友好的后台摄入
def background_ingest(new_docs, batch=50):
    """分批、限线程、可中断的后台摄入。"""
    for batch_docs in chunks(new_docs, batch):
        if system.memory_pressure_high():
            yield "wait"          # 让出，等系统压力降下来
            continue
        if not device.is_idle():  # 用户在用设备，延后
            schedule_later(batch_docs)
            continue
        # 低优先级线程执行（zvec.init 时已限定 optimize_threads）
        run_low_priority(lambda: ingest_batch(batch_docs))
        yield "done"
💡 要点
端侧资源治理的三条铁律：① 内存靠量化（float 是奢侈品，用 zvec.init(memory_limit_mb=...) 明确预算）；② CPU 靠 SIMD + 限线程（int8 NEON + query_threads/optimize_threads 让出前台）；③ 写入靠分时（设备空闲时做，充电+息屏是黄金窗口）。违反任何一条，App 都会被用户卸载——卡顿和耗电是端侧的死刑。

zvec 资源管理：内存池与降级调度

向量引擎要扛住高 QPS 和大数据量，资源管理是底层绕不开的硬骨头。前面讲了 zvec 的索引和查询，但当十万级并发打过来时，真正决定稳定性的不是索引算法，而是内存怎么分配、线程怎么调度、显存不够时怎么降级。zvec 这类工业引擎的资源管理有三个核心抽象：内存池（避免频繁 malloc/free 的碎片）、线程池（控制并发防止过载）、资源感知调度（水位高时自动降级保活）。我们用 Python 实现一个最小但忠实于原理的版本，帮你建立"资源是被管理的、不是无限"的工程直觉。

# -*- coding: utf-8 -*-
"""zvec 资源管理：内存池 + 线程池 + 显存调度的统一抽象。
下面的 ResourcePool 演示如何在向量引擎里管理多类资源。
"""
import threading
from dataclasses import dataclass, field
from typing import Optional

@dataclass
class MemoryPool:
    """定长块内存池：预分配，避免频繁 malloc/free。"""
    total_bytes: int
    block_size: int = 4096
    _free: list = field(default_factory=list)
    _lock: threading.Lock = field(default_factory=threading.Lock)

    def __post_init__(self):
        n = self.total_bytes // self.block_size
        self._free = [bytearray(self.block_size) for _ in range(n)]

    def alloc(self) -> Optional[bytearray]:
        with self._lock:
            return self._free.pop() if self._free else None

    def free(self, blk):
        with self._lock:
            self._free.append(blk)

    @property
    def usage(self):
        used = (self.total_bytes // self.block_size) - len(self._free)
        return used / (self.total_bytes // self.block_size)

内存池的核心收益是消除分配抖动。向量查询每次都要申请临时缓冲区存放候选集，如果用系统 malloc，高并发下会产生大量碎片、延迟剧烈波动；内存池预分配一大块、切成定长块按需借用/归还，延迟稳定且可预测。注意实现里用了锁保护 free 列表——在真实引擎里会进一步用无锁队列（lock-free queue）或分片锁降低争用。下面这个资源感知的查询调度器，演示了"内存不够就降级"的工程实践：

# 资源感知的查询调度：内存不够时降级(如把HNSW换成IVF+PQ)
class QueryScheduler:
    def __init__(self, mem_pool: MemoryPool):
        self.pool = mem_pool

    def plan(self, query_batch_size):
        """根据当前内存水位选择执行计划。"""
        plans = []
        for _ in range(query_batch_size):
            blk = self.pool.alloc()
            if blk is None:
                # 内存吃紧 -> 降级策略
                plans.append(("degraded", "IVF_PQ", None))
            else:
                plans.append(("full", "HNSW", blk))
        return plans

    def release(self, plans):
        for kind, _, blk in plans:
            if blk is not None:
                self.pool.free(blk)

# 演示
pool = MemoryPool(total_bytes=4096 * 100, block_size=4096)
sched = QueryScheduler(pool)
# 先占掉 95% 内存, 制造压力
hog = [pool.alloc() for _ in range(95)]
plans = sched.plan(10)
full = sum(1 for k, *_ in plans if k == "full")
print(f"水位={pool.usage:.0%}  10个查询中 full={full} degraded={10-full}")
for blk in hog: pool.free(blk)

这种降级策略是生产系统稳定性的关键。当内存水位超过 90%，调度器不再硬撑着跑全精度 HNSW（每个查询要借大块内存），而是自动降级成 IVF+PQ（内存占用小一个数量级），虽然召回率略降，但保证了系统不 OOM、不雪崩。这种"宁可降级也不宕机"的哲学，是高可用服务的铁律。把它推广开，zvec 的资源管理还有两个进阶点：一是显存与内存的统一调度——GPU 显存不够时把部分索引换出到 CPU 内存（类似 vLLM 的 preempt），用 PCIe 传输换可用性；二是多租户隔离——给不同租户分配独立的内存池配额，防止一个大租户的突发流量挤垮其他租户。把这些资源管理做扎实，向量引擎才能从"能跑"升级到"扛得住生产"。

本章小结

端侧资源治理是在"几十 MB 内存、ARM CPU、严格电量"的硬预算下做向量检索。zvec 用 zvec.init(memory_limit_mb=..., query_threads=..., optimize_threads=...) 把自己当成进程里的"资源租户"。核心手段：① 内存靠量化（int8/int4，float 是奢侈品，官方 benchmark 默认 int8）；② CPU 靠 NEON/AVX SIMD（src/core/metric/，int8 比 float 快 4 倍）+ 限线程；③ 写入靠分时（设备空闲窗口）；④ 持久化靠 WAL + checkpoint（热数据驻内存、冷数据落盘）。两级检索（refiner）是内存极紧时的标配。Dart/Flutter FFI 绑定让 zvec 能直接进手机 App。下一章我们做横向对比，看 zvec 在整个向量库生态里的位置。
