投机解码
用小模型猜测，大模型批量验证的加速魔法

大模型推理慢，瓶颈在于每生成一个 token 都要跑一次完整的前向。但如果我们能让一个轻量小模型先「投机」猜出后面好几个 token，再让大模型一次性批量验证，就能把多次串行调用压缩成一次并行调用。这就是 Speculative Decoding——它在数学上保证输出和纯大模型完全一致，却能带来 2 到 3 倍的实际加速。本章拆解它的原理、接受拒绝采样的概率推导，以及从零的实现。

阅读约 25 分钟 难度 ★★★★★

★ 一、为什么需要投机解码

先理解大模型推理慢在哪。Decoder-Only 模型生成时，每吐出一个 token 都要跑一次完整的前向传播，而前向里最贵的是 attention 的 KV 计算和 FFN 的矩阵乘。假设要生成 100 个 token，就要串行跑 100 次前向，延迟和 token 数严格成正比。

更关键的是，现代 GPU 的并行计算能力在「只生成一个 token」时严重浪费——一次前向只算一个 query，但 attention 的 key/value 和 FFN 的矩阵都铺满了显存带宽。这就像用一辆大卡车一次只运一个快递，运力利用率极低。

投机解码的核心洞察是：大模型并行处理多个 token 的成本，和处理单个 token 几乎一样。因为前向的主要开销是读取权重和 KV cache（memory-bound），多算几个 query 的增量很小。于是，如果我们能让小模型先猜出一串 token，再让大模型一次性验证这串 token，就能用「一次大模型前向」替代「多次小步前进」，把延迟大幅压缩。

1.1 一个直观的生活类比

把大模型想象成一位严谨的资深编辑，校对一个字要 10 分钟。把小模型想象成一位实习编辑，写草稿很快但偶尔出错。如果你让资深编辑一个字一个字地校，写 100 个字要 1000 分钟。但如果你让实习编辑先一口气写 8 个字，再把这 8 个字交给资深编辑一次性审——资深编辑审 8 个字大概只要 12 分钟（因为审 1 个和审 8 个的边际成本很低），如果前 5 个字对、第 6 个字错，资深编辑划掉第 6 个字并改正，然后让实习编辑从改正处继续。这样平均下来，资深编辑的时间利用效率大大提升。

这里的「时间利用效率」就是投机解码的加速比。加速多少取决于小模型猜得有多准——猜对的越多，大模型一次验证通过的越多，加速越大。

★ 二、投机解码的完整流程

把上面的类比翻译成算法。设大模型为 M（目标模型，target），小模型为 D（草稿模型，draft）。当前已经生成的序列是 x，我们要往后生成若干 token。一次投机解码的迭代包括四步。

2.1 四步走：起草、验证、接受、修正

  • 第一步 起草（Draft）：用小模型 D 自回归地生成 k 个候选 token，记为 d1, d2, ..., dk。同时记录 D 在每一步的概率分布 q1, ..., qk。
  • 第二步 验证（Verify）：把 [x, d1, d2, ..., dk] 一次性喂给大模型 M 做一次前向（因为是一次输入多个 token，能并行计算）。M 会输出在每个位置的概率分布 p1, ..., p(k+1)。
  • 第三步 接受/拒绝（Accept/Reject）：从前往后逐个检查 d_i 是否被 M 接受。对每个 d_i，从均匀分布 U(0,1) 采样一个数 r；如果 r < min(1, p_i(d_i) / q_i(d_i))，则接受 d_i，继续检查下一个；否则拒绝 d_i 及其之后的所有草稿。
  • 第四步 修正（Correct）：如果接受了前 j 个 token（d1 到 dj 被接受，d(j+1) 被拒绝），则从修正分布重新采样一个 token 替代 d(j+1)；如果所有 k 个都接受了，则额外从 p(k+1) 里采样一个 bonus token。然后把接受的 token 加到序列里，开始下一轮投机。

这里的 accept/reject 规则和修正分布不是随便定的，它们背后有严格的数学保证——保证最终输出的分布和大模型 M 单独生成的分布完全一致。下面我们就来推导这个关键性质。

★ 三、接受拒绝采样的数学保证

投机解码最精妙的地方，是它能保证输出分布和纯大模型采样完全一致。这意味着：无论小模型猜得多烂，投机解码的输出质量都不会比大模型单独生成差，区别只是速度。这个保证来自拒绝采样（rejection sampling）的理论。

3.1 单个 token 的接受概率

考虑草稿 token d_i。大模型给它的概率是 p_i(d_i)，小模型给的是 q_i(d_i)。我们设定的接受规则是：以概率 min(1, p_i(d_i) / q_i(d_i)) 接受 d_i。这看起来随意，但配合修正分布就能保证正确性。

修正分布的设定是：如果拒绝 d_i，则从修正分布 p'_i(x) = max(0, p_i(x) - q_i(x)) / Z 里重新采样，其中 Z 是归一化常数，Z = sum_x max(0, p_i(x) - q_i(x))。这个修正分布的物理含义是「大模型概率超出小模型的那部分」。

3.2 为什么最终分布等于大模型分布

现在证明：经过「以 min(1, p/q) 接受 d，否则从 (p-q)+/Z 采样」这套流程，最终输出某个 token x 的总概率，等于大模型概率 p(x)。分两种情况讨论。

当草稿恰好等于 x（即 d = x）时，输出 x 的概率是「接受 x 的概率」= min(1, p(x)/q(x)) * q(x)。当草稿不等于 x（即 d != x）且最终修正到 x 时，输出 x 的概率是「拒绝后修正到 x」= sum_{d != x} q(d) * (1 - min(1, p(d)/q(d))) * max(0, p(x)-q(x))/Z。

经过代数化简（过程略，关键是用到 Z 的定义和 max(0, p-q) 的性质），两种情况加起来恰好等于 p(x)。这就是投机解码的理论核心：拒绝采样保证了输出分布的精确还原。所以小模型越准，接受率越高，加速越大；小模型再烂，也只是接受率低、速度退回到和纯大模型一样，质量绝不会下降。

★ 四、从零实现投机解码

理论讲透了，现在动手实现。为了可在 CPU 上跑通，我们用两个不同大小的 GPT-2 模型（标准版做 target，蒸馏版做 draft），实现一个简化但完整的投机解码循环。

4.1 核心实现

import torch
import torch.nn.functional as F

@torch.no_grad()
def draft_generate(draft_model, input_ids, num_draft, temperature=1.0):
    """用小模型自回归生成 num_draft 个草稿 token，同时记录每步概率分布 q。"""
    draft_tokens = []
    draft_probs = []    # q_i 分布
    seq = input_ids
    for _ in range(num_draft):
        logits = draft_model(seq).logits[0, -1] / temperature
        q = F.softmax(logits, dim=-1)
        next_id = torch.multinomial(q, num_samples=1)
        draft_tokens.append(next_id.item())
        draft_probs.append(q)
        seq = torch.cat([seq, next_id.unsqueeze(0)], dim=-1)
    return draft_tokens, draft_probs

@torch.no_grad()
def speculative_step(target_model, draft_model, input_ids, num_draft=4):
    """一轮投机解码。返回 (新生成的 token 列表, 更新后的 input_ids)。"""
    # 1. 起草
    draft_tokens, draft_probs = draft_generate(draft_model, input_ids, num_draft)
    # 2. 验证：把 [input_ids, draft...] 一次性喂给 target
    full = torch.cat([input_ids,
                      torch.tensor([draft_tokens]).T], dim=-1)
    target_logits = target_model(full).logits[0]   # (L+k, vocab)
    # 我们关心的是每个草稿位置 i 对应的 target 分布 p_i
    # 位置：input_ids 长度 L，草稿从 L 开始，target_logits[L-1] 预测第 1 个草稿
    L = input_ids.shape[1]
    target_probs = [F.softmax(target_logits[L-1+i], dim=-1) for i in range(num_draft)]

    # 3. 接受/拒绝
    accepted = []
    for i in range(num_draft):
        d = draft_tokens[i]
        p = target_probs[i][d].item()
        q = draft_probs[i][d].item()
        r = torch.rand(1).item()
        if r < min(1.0, p / max(q, 1e-8)):
            accepted.append(d)
        else:
            # 拒绝：从修正分布采样一个 token 替代
            correction = torch.clamp(target_probs[i] - draft_probs[i], min=0)
            if correction.sum() > 0:
                correction = correction / correction.sum()
                fixed = torch.multinomial(correction, num_samples=1).item()
                accepted.append(fixed)
            break
    else:
        # 所有草稿都接受：bonus token 从 target_probs[-1] 的下一步采样
        bonus_logits = target_model(full).logits[0, -1]
        bonus = torch.multinomial(F.softmax(bonus_logits, dim=-1), num_samples=1).item()
        accepted.append(bonus)

    new_ids = torch.tensor([accepted]).T
    return accepted, torch.cat([input_ids, new_ids], dim=-1)

这段代码完整实现了四步流程：起草、验证、接受拒绝、修正加 bonus。注意关键细节：接受判断用 min(1, p/q) 的拒绝采样；拒绝时从 (p-q)+ 的修正分布重采样；全接受时额外送一个 bonus token。这些都对应前面的数学推导。

4.2 完整的投机解码循环

from transformers import AutoModelForCausalLM, AutoTokenizer

target_name = "gpt2"              # 大模型
draft_name = "distilgpt2"         # 小模型（草稿）
tokenizer = AutoTokenizer.from_pretrained(target_name)
target = AutoModelForCausalLM.from_pretrained(target_name).eval()
draft = AutoModelForCausalLM.from_pretrained(draft_name).eval()

def speculative_decode(prompt, max_new_tokens=30, num_draft=4):
    ids = tokenizer(prompt, return_tensors="pt").input_ids
    total = 0
    rounds = 0
    while total < max_new_tokens:
        accepted, ids = speculative_step(target, draft, ids, num_draft)
        total += len(accepted)
        rounds += 1
    return tokenizer.decode(ids[0], skip_special_tokens=True), rounds

text, rounds = speculative_decode("The future of AI is", max_new_tokens=30, num_draft=4)
print("输出：", text)
print(f"投机轮数: {rounds}（如果纯 target 需要 30 次前向，这里只用了 {rounds} 次 target 前向）")

运行后你会看到一个惊人的事实：生成同样多的 token，投机解码只用了一小部分大模型前向调用。在 distilgpt2 猜得比较准的简单 prompt 上，30 个 token 可能只要 8 到 10 轮投机，即 8 到 10 次大模型前向——相比纯 target 的 30 次前向，节省了三分之二。

★ 五、加速效果分析

投机解码能加速多少，取决于三个因素：草稿长度 k、小模型的接受率 alpha、大小模型的前向耗时比。我们用一个简化模型来分析。

5.1 接受率与加速比的关系

设小模型每步的接受概率为 alpha（0 到 1），草稿长度为 k。则一轮投机平均接受的 token 数为 (1 - alpha^(k+1)) / (1 - alpha)（几何级数）。一轮投机的时间成本是 1 次大模型前向 + k 次小模型前向。假设大模型前向耗时 T_t，小模型前向耗时 T_d（通常 T_d 远小于 T_t）。

于是平均每生成一个 token 的有效时间为 (T_t + k*T_d) / 平均接受数。而纯大模型每生成一个 token 需要 T_t。加速比就是 T_t / 这个有效时间。把 alpha 提高或把 T_d 压低，都能提升加速比。

下面这段代码帮你直观感受不同接受率下的加速比。

def speculative_speedup(alpha, k, Tt_Td_ratio=10):
    """
    估算投机解码相对纯 target 的加速比。
    alpha: 接受率; k: 草稿长度; Tt_Td_ratio: 大小模型前向耗时比。
    """
    if alpha == 1.0:
        accepted = k + 1   # 全接受，加 bonus
    else:
        accepted = (1 - alpha**(k+1)) / (1 - alpha)
    Tt = 1.0
    Td = 1.0 / Tt_Td_ratio
    time_per_round = Tt + k * Td
    time_per_token_spec = time_per_round / accepted
    time_per_token_pure = Tt
    return time_per_token_pure / time_per_token_spec

print(f"{'接受率':>8} {'k=4 加速比':>12} {'k=8 加速比':>12}")
for alpha in [0.5, 0.7, 0.85, 0.95]:
    s4 = speculative_speedup(alpha, 4)
    s8 = speculative_speedup(alpha, 8)
    print(f"{alpha:>8.2f} {s4:>12.2f}x {s8:>12.2f}x")

你会看到：接受率 0.7、k=4 时大约 2 倍加速；接受率 0.85 时接近 3 倍。接受率越高，加速越显著；但 k 不是越大越好——k 太大时如果末尾几个草稿老被拒，反而浪费了小模型的前向。实践中 k 通常取 4 到 8，需要根据具体模型动态调整。

★ 六、投机解码的变体与工程实践

基础的投机解码已经很有用，研究社区还发展出几个值得关注的变体，进一步提升了适用性。

  • Self-Speculative / Layer Skipping：不引入独立的小模型，而是跳过目标模型的若干层来当 draft（如 LayerSkip）。省去了维护第二个模型的开销。
  • Medusa：在目标模型上额外训练几个「头」，每个头并行预测未来第 2、3、4 个位置，相当于让目标模型自己当 draft，省去小模型。
  • Lookahead Decoding：用 Jacobi 迭代并行生成多个 token，不需要任何 draft 模型，纯靠并行化加速。
  • EAGLE / EAGLE-2：训练一个轻量 draft 模型，不仅输入 token 还输入隐状态，接受率大幅提高，是目前实际效果最好的方案之一。

工程上，投机解码已经在 vLLM、TensorRT-LLM、TGI 等主流推理框架里落地。它的主要适用场景是低并发、长输出的交互式推理（比如聊天助手），因为这种场景下单次请求延迟的降低能直接提升用户体验。在高并发的批处理场景，收益会打折扣，因为 GPU 已经被多个请求填满了。

★ 本章小结

这一章我们把投机解码从原理到实现完整过了一遍。要点回顾：

  • 核心动机：大模型推理是 memory-bound，一次前向算一个 token 和算多个 token 的成本接近，所以可以让小模型先猜多个再让大模型批量验证。
  • 四步流程：起草（draft model 生成 k 个候选）、验证（target model 一次前向）、接受拒绝（min(1, p/q) 规则）、修正（从修正分布重采样或 bonus token）。
  • 数学保证：拒绝采样保证最终输出分布和纯 target 模型完全一致，小模型越准加速越大，再烂也不影响质量。
  • 加速因素：接受率越高、草稿长度适中、大小模型耗时比越大，加速越显著，典型加速 2 到 3 倍。
  • 工程落地：vLLM、TensorRT-LLM 已支持；变体包括 self-speculative、Medusa、EAGLE 等，适用于交互式低并发场景。

投机解码是「不改模型、只改解码算法」就能加速的优雅方案。接下来我们转向另一个方向的优化——模型量化，它通过把权重从 FP16 压到 INT8 甚至 INT4，直接砍掉显存和计算开销。这是让大模型跑在消费级显卡上的关键技术。
