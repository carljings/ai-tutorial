SKILL.md 是什么：技能的入口契约

在 Claude、ZCode、以及任何支持 Skills 的 agent 里，一个"技能"本质上是一个文件夹，里面装着指令、模板、参考文件。而这个文件夹的灵魂是一个叫 SKILL.md 的文件——它是 agent 认识这个技能的唯一入口。agent 启动时不会把整个技能文件夹塞进上下文（那样太占 token），而是只读每个 SKILL.md 的头部元信息（name + description），等真正需要用这个技能时才加载完整内容。这就是"渐进式披露"（progressive disclosure）的第一层。

可以把 SKILL.md 理解成"技能的名片 + 使用说明书"。名片部分（YAML frontmatter）告诉 agent"我叫什么、我擅长什么"，让 agent 在遇到相关任务时能想起我；说明书部分（正文）告诉 agent"具体该怎么做"。前者必须精简（占用最少 token），后者可以详尽（只在需要时加载）。

YAML frontmatter：技能的元信息

SKILL.md 的开头是一个 YAML frontmatter 块，包含技能的身份信息。这部分是 agent 启动时扫描所有技能时唯一读取的内容，所以必须做到"看一眼就知道这个技能能干什么、什么时候该调用"。

---
name: pdf-report-generator
description: 生成专业的 PDF 报告。适用于月度总结、项目汇报、数据分析报告、简历等需要排版精美的文档输出场景。当用户需要把内容做成 PDF、提到"报告""汇报""简历""海报"时使用此技能。
---

# PDF 报告生成技能

## 何时使用
当用户需要生成格式化的 PDF 文档时...

## 使用流程
1. 确认报告类型和受众
2. 收集必要内容素材
3. 选择对应模板
4. 生成并预览
...

注意 description 的写法：它不只是描述技能"是什么"，更要描述"什么时候该用"。好的 description 像一个精准的触发器——当用户的请求匹配时，agent 能毫不犹豫地选中这个技能；写得含糊的话，agent 要么漏选、要么误选。

💡 要点
description 是 SKILL.md 里最关键的一行。它决定了 agent 能否在正确的时机激活这个技能。写法秘诀：包含"做什么"+"适用场景关键词"，让语义匹配既精准又宽覆盖。

正文的分层结构

SKILL.md 的正文是技能的"操作手册"。它的写法直接影响 agent 执行技能的质量。好的正文应该分层：先给决策指引（什么时候用/不用），再给操作步骤（怎么做），最后给参考资料（遇到细节去哪查）。

一个常见的误区是把所有细节都堆进 SKILL.md 正文。但这样会让技能在加载时占用大量 token。更好的做法是正文只放"决策级"和"流程级"的内容，把"细节级"的模板、示例、参数表拆到同目录的独立文件里，正文里用"详见 TEMPLATES.md"来引用。这就是渐进式披露的第二层：按需深入。

技能文件夹的完整结构

my-skill/
├── SKILL.md            ← 入口：元信息 + 操作指引（agent 启动时只读头部）
├── TEMPLATES.md        ← 模板集合（需要时才加载）
├── examples/
│   ├── basic.txt       ← 示例输入
│   └── advanced.txt
├── reference/
│   ├── params.md       ← 完整参数说明
│   └── troubleshooting.md
└── assets/
    └── logo.png        ← 技能用到的资源

这个结构让 agent 可以"逐层打开"技能：先读 SKILL.md 决定是否用、怎么用；执行中需要模板就读 TEMPLATES.md；遇到报错就去 troubleshooting.md。每一层只在需要时加载，避免上下文爆炸。

完整 SKILL.md 模板：拿来即用的标准结构

前面拆解了 SKILL.md 的 frontmatter 和正文结构，现在给一个完整的、可直接套用的模板。这个模板涵盖了生产级技能应该具备的所有要素——元信息、触发条件、使用流程、边界声明、辅助文件引用。你可以把它当成写技能的"脚手架"。

---
# ═══════════ frontmatter（agent 启动时扫描，必须精简）═════════════
name: api-doc-generator           # 技能唯一标识（kebab-case）
description: >-
  生成专业的 API 技术文档。适用于为 REST/GraphQL 接口自动生成
  Markdown 或 HTML 格式的 API 文档，包含端点说明、参数表、
  请求/响应示例、错误码。当用户提到"生成 API 文档""写接口文档"
  "API 说明""接口文档"" documenting API"时使用此技能。
  注意：不用于生成用户手册或产品文档（那是 user-guide 技能的职责）。

# 可选字段（按需添加）
version: 1.2.0
author: dev-tools-team
enabled: true
permissions:                     # 声明需要的工具权限
  - read_file                    # 读代码文件
  - write_file                   # 写生成的文档
  - execute_command              # 运行代码提取注释
requires_agent: ">=0.5.0"        # 兼容的最低 agent 版本
tags: [documentation, api, code]
---

# API 文档生成技能

## 何时使用
- 用户要把代码里的 API 接口导出成文档
- 需要为 REST/GraphQL 接口生成规范说明
- 代码缺少文档，需要从代码注释反向生成

## 何时不使用（边界）
- 生成用户手册/产品文档 → 用 user-guide 技能
- 只是想解释某段代码 → 直接回答，不用技能
- 生成 OpenAPI/Swagger 规范 → 用 openapi-spec 技能

## 使用流程
1. **确认范围**：问用户要为哪些接口生成（全部/指定模块/指定路径）
2. **提取接口**：从代码扫描路由定义、参数、返回类型
3. **选格式**：确认输出格式（Markdown / HTML / Confluence wiki）
4. **生成初稿**：按模板填充，每个端点含：说明、参数表、示例、错误码
5. **补充示例**：为每个端点生成请求/响应示例（可从测试代码提取）
6. **审查完善**：检查完整性，补充遗漏的边界情况说明

## 关键规范
- 参数表必须含：名称、类型、必填、默认值、说明
- 每个端点至少一个成功示例和一个错误示例
- 错误码要列全，标注含义和处理建议

## 辅助资源（按需加载，详见对应文件）
- 模板：详见 `TEMPLATES.md`（Markdown/HTML 模板）
- 参数详表格式：详见 `reference/params-format.md`
- 常见问题：详见 `reference/troubleshooting.md`

## 变更日志
- v1.2.0: 支持 GraphQL 接口文档生成
- v1.1.0: 增加 HTML 格式输出
- v1.0.0: 初始版本，支持 REST API + Markdown

frontmatter 字段详解：每个字段的作用

上面的模板用了很多 frontmatter 字段，每个都有特定作用。理解它们能让你的技能被更精准地匹配、更安全地执行、更好地管理版本。下面逐一详解关键字段。

SKILL.md frontmatter 字段详解

【必填字段】
─────────────────────────────────────────────
name (string)
  技能的唯一标识，kebab-case 命名。
  作用：技能的 ID，用于引用、加载、卸载。
  示例：api-doc-generator
  注意：全局唯一，改 name 等于新建技能（旧的找不到）。

description (string)
  技能做什么 + 什么时候用 + 不做什么（边界）。
  作用：技能匹配的唯一依据（agent 据此决定是否调用）。
  写法：见前面"description 怎么写"章节。
  这是最重要的字段，决定了技能被发现和正确使用的概率。

【推荐字段】
─────────────────────────────────────────────
version (string)
  语义化版本号 major.minor.patch。
  作用：版本管理、兼容性检查、回滚。
  规则：修 bug → patch+1；加功能 → minor+1；破坏性变更 → major+1。

permissions (list)
  声明技能需要的工具权限（读文件/写文件/执行命令/网络）。
  作用：agent 据此审批——运行时只授予声明过的权限。
  安全意义：技能不能越权（比如只声明了 read 却想写，会被拦）。

enabled (bool)
  是否启用此技能。
  作用：临时禁用技能（不删除），用于调试或 A/B 测试。

requires_agent (string)
  兼容的最低 agent 版本（语义化版本）。
  作用：防止旧技能在不兼容的新 agent 上出错。
  示例：">=0.5.0" 表示需要 agent 0.5.0 及以上。

【可选字段】
─────────────────────────────────────────────
author / tags / category
  元信息，用于技能市场的搜索、分类、展示。
  示例：tags: [documentation, api, code]

渐进式披露三层：省 token 的精妙设计

渐进式披露（Progressive Disclosure）是 Skills 体系的核心设计哲学——它让 agent 在不同阶段只加载需要的内容，避免一次性把整个技能塞进上下文。这套机制分三层，每层加载的信息量和时机都不同。理解这三层，你就能设计出既功能完整又 token 高效的技能。

渐进式披露的三层架构

第一层：索引层（INDEX）—— agent 启动时
─────────────────────────────────────────────
加载内容：只读每个 SKILL.md 的 frontmatter（name + description）
Token 占用：每个技能 ~30-50 token（30 个技能 ~1500 token）
时机：agent 启动时，扫描所有技能目录
目的：让 agent"知道有哪些技能可用"，用于匹配判断
类比：图书馆的目录卡片（只看书名和摘要，决定找不找）

  ┌─────────────────────────────────────┐
  │ 索引层（常驻上下文）                 │
  │  [api-doc-generator] 生成 API 文档   │
  │  [pdf-report] 生成 PDF 报告          │
  │  [code-review] 代码审查              │
  │  ...（每个技能一行）                 │
  └─────────────────────────────────────┘

第二层：指令层（INSTRUCTIONS）—— 匹配到技能时
─────────────────────────────────────────────
加载内容：匹配到的技能的 SKILL.md 正文（使用流程、规范）
Token 占用：~500-2000 token（只有被选中的技能）
时机：用户任务匹配到某技能时，按需加载
目的：让 agent 知道"具体怎么做这个任务"
类比：从书架取下书，翻开目录和正文

  用户："生成 API 文档"
       ↓ 匹配到 api-doc-generator
  ┌─────────────────────────────────────┐
  │ 指令层（按需加载）                   │
  │  # API 文档生成                      │
  │  ## 使用流程                         │
  │  1. 确认范围  2. 提取接口...         │
  │  ## 关键规范                         │
  │  参数表必须含：名称、类型、必填...   │
  └─────────────────────────────────────┘

第三层：细节层（DETAILS）—— 执行中需要时
─────────────────────────────────────────────
加载内容：辅助文件（TEMPLATES.md / reference/ / examples/）
Token 占用：按需，可能 ~1000-5000 token
时机：执行过程中，遇到需要模板/参数详表/示例时
目的：提供"细节级"参考，处理边界情况
类比：查阅书的附录、索引、参考资料

  执行到"选格式"步骤
       ↓ 需要 Markdown 模板
  ┌─────────────────────────────────────┐
  │ 细节层（按需加载）                   │
  │  ## Markdown 模板                    │
  │  ## {{endpoint}}                     │
  │  ### 参数                            │
  │  | 名称 | 类型 | 必填 | ...          │
  └─────────────────────────────────────┘
# 渐进式披露的实现：控制每层何时加载
class ProgressiveSkillLoader:
    def load_index(self):
        """第一层：启动时加载所有技能的索引（常驻）。"""
        for skill_dir in self.skills_path.iterdir():
            meta = read_frontmatter(skill_dir / "SKILL.md")
            self.index[meta["name"]] = meta["description"]
        # 这 30 个技能的索引常驻上下文，只占 ~1500 token

    def load_instructions(self, skill_name):
        """第二层：匹配到技能时加载正文（按需）。"""
        body = read_body(f"{self.skills_path}/{skill_name}/SKILL.md")
        self.context.inject(body)
        # 只有被选中的技能正文进入上下文，~1000 token

    def load_details(self, skill_name, file_name):
        """第三层：执行中需要时加载辅助文件（按需）。"""
        path = f"{self.skills_path}/{skill_name}/{file_name}"
        content = read_file(path)
        self.context.inject(content)
        # 只有真正需要的辅助文件加载，避免全加载

# 三层披露的 token 经济学（30 个技能的场景）：
# 全加载（无渐进式）：30 × (50+1500+3000) = 136,500 token ❌ 爆炸
# 渐进式披露：
#   第一层：30 × 50 = 1,500 token（常驻）
#   第二层：1 × 1500 = 1,500 token（用到的）
#   第三层：1 × 3000 = 3,000 token（偶尔用）
#   总计：~6,000 token ✓ 可控（省了 95%+）
💡 设计心法
渐进式披露的本质是"用多少加载多少"——这和操作系统的虚拟内存分页思想一致：不可能把所有数据都装进内存（上下文），而是按需调入需要的页。设计技能时要主动分层：把"决策级"信息放第一层（什么时候用）、"流程级"放第二层（怎么做）、"细节级"放第三层（模板/参数/示例）。这样既保证功能完整，又把 token 占用控制在可接受范围。判断标准：如果某信息不是每次执行都需要，就别放前两层——下沉到辅助文件按需加载。
