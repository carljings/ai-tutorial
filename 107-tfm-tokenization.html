分词 Tokenizer
文字如何变成模型能吃的 Token

在 token 进入 embedding 之前，首先要经历分词——把「我喜欢 Transformers」切成模型认识的单元。BPE、WordPiece、Unigram、SentencePiece 四大算法各有门道。中文没有空格，怎么分？emoji 和代码怎么处理？词表大小又怎么权衡推理成本？这一章把分词这件「看似简单实则水深」的事讲透，附带可运行的 BPE 实现和 tiktoken 实战。

阅读约 28 分钟 难度 ★★★★ 前置：第 108 章概念

★ 为什么需要分词

神经网络只能吃数字。所以文本要变成一串数字 ID，才能喂给 embedding 层。这个「文本 → ID 序列」的过程就是分词（tokenization），中间的单元叫 token。

分词看似简单，实则水深。它直接决定了：①模型怎么理解语言（粒度）；②推理成本（同样的话切多少 token 决定费用）；③对多语言、代码、特殊字符的友好度。一个糟糕的分词器能让模型又贵又差。

三种朴素的极端方案

先看三种最朴素的分词思路，理解它们的缺陷，就知道为什么需要 BPE 这些算法：

  • 按字符（char-level）：「hello」→ ['h','e','l','l','o']。词表小（几十个），但序列太长，模型要学「h+e+l+l+o=hello」的组合，学习负担重。
  • 按整词（word-level）：「I love NLP」→ ['I','love','NLP']。序列短，但词表爆炸（几百万词），且遇到没见过的词（OOV，out-of-vocabulary）就抓瞎。
  • 按子词（subword）：「unhappiness」→ ['un','happiness'] 或 ['un','happy','ness']。折中：常见词整块，罕见词拆成有意义的片段。这是现代 LLM 的选择。

子词分词的精髓：用数据驱动地学出一套「词表」，覆盖从字符到整词的所有粒度。常见组合整个是一个 token，罕见词拆成若干已知子词。既不爆炸也不太长，还能处理任何新词。

01 BPE：字节对编码

BPE（Byte Pair Encoding）是 GPT 系列用的算法。它的训练过程是贪心地合并最高频的相邻字节对：

初始化

把所有词拆成单个字符（或字节）。统计每个词的出现频率。

找最高频对

扫描所有词，找出相邻字符对里出现频率最高的那一对。比如 ('t','h') 出现了 100 次。

合并

把这对合并成一个新 token「th」，加进词表。所有出现 ('t','h') 的地方替换成「th」。

重复

循环 2~3，直到词表达到目标大小（如 5 万）。每次合并都产生一个新 token。

结果：词表里既有单字符（'a','b'...），也有高频组合（'th','tion'...），还有整词（'the'...）。高频组合被合并成单 token，低频的保持拆分——这就是「数据驱动」的子词粒度。

推理时的分词

训练好 BPE 词表（一个有序的合并规则列表）后，对新文本分词：①先拆成单字符；②按训练时的合并顺序，依次尝试应用每条合并规则；③所有能合并的都合并完，剩下的就是 token 序列。

💡 类比：搭积木

BPE 像搭积木。单字符是最小的积木块。训练时观察哪些积木经常挨在一起，就把它们粘成一个大积木。越常一起出现的，粘得越大。最后你有一盒从大到小的积木。搭新模型（分新词）时，能用大积木就用大的，搭不上的用小块拼。

02 代码：从零实现 BPE

bpe_from_scratch.py —— 极简 BPE 训练器
from collections import Counter

def train_bpe(corpus, vocab_size=100):
    """在语料上训练 BPE，返回合并规则列表（按学习顺序）。"""
    # 初始化：每个词拆成字符，统计词频
    word_freqs = Counter(corpus.split())
    splits = {w: list(w) for w in word_freqs}

    merges = []
    # 初始词表：所有出现过的字符
    vocab = set(c for w in splits for c in splits[w])

    while len(vocab) < vocab_size:
        # 统计所有相邻对的出现次数
        pairs = Counter()
        for w, freq in word_freqs.items():
            chars = splits[w]
            for i in range(len(chars) - 1):
                pairs[(chars[i], chars[i+1])] += freq
        if not pairs:
            break
        # 找最高频对
        best = pairs.most_common(1)[0][0]
        merges.append(best)
        vocab.add(''.join(best))
        # 在所有词里合并这个对
        for w in splits:
            chars = splits[w]
            new_chars = []
            i = 0
            while i < len(chars):
                if i < len(chars) - 1 and (chars[i], chars[i+1]) == best:
                    new_chars.append(chars[i] + chars[i+1])
                    i += 2
                else:
                    new_chars.append(chars[i])
                    i += 1
            splits[w] = new_chars
    return merges, vocab

# 在小语料上训练
corpus = "low low low low low lower lower newest newest newest newest newest newest widest widest widest"
merges, vocab = train_bpe(corpus, vocab_size=30)
print("学到的合并规则（前 10 条）:")
for m in merges[:10]:
    print(f"  {m[0]} + {m[1]} → {m[0]+m[1]}")
# 你会看到 ('e','s')、('es','t')、('est','_') 等高频后缀被合并

运行后你会看到 BPE 学到了 'e'+'s'='es'、'es'+'t'='est' 这类高频组合——因为语料里 'newest'、'widest' 都包含 'est'，它被高频合并了。这就是数据驱动的子词学习。

03 Byte-level BPE：GPT 的做法

原始 BPE 在字符层面工作，但字符层面有 Unicode 问题：中文、emoji、各种语言的字符成千上万，词表会被这些罕见字符塞满。GPT-2 的解决方案是 Byte-level BPE：

不在 Unicode 字符层面做 BPE，而是先把文本编码成 UTF-8 字节（只有 256 种），再在字节层面做 BPE。

好处：基础词表只有 256 个字节，覆盖任何语言的任何字符（因为任何字符都能用 UTF-8 字节表示）。中文「我」是 3 个字节，emoji 可能是 4 个字节。BPE 会在字节层面学合并，常见的中文字符序列会被合并成单 token。

这是 GPT-2/3/4、Llama、GLM 等几乎所有现代 LLM 的做法。tiktoken 是 OpenAI 开源的 GPT 系列 BPE 分词器实现。

tiktoken_demo.py —— 用 tiktoken 看 GPT 怎么分词

运行后你会看到：「我」是一个 token（中文友好），「 Transformers」被切成几个子词，emoji 「🤗」是几个字节 token。不同模型的分词器对中文的友好度差异很大，直接影响中文场景的 token 成本。

04 WordPiece：BERT 的选择

WordPiece（Google，2016）和 BPE 很像，也是贪心合并，但合并标准不同。BPE 选「频率最高的对」，WordPiece 选「让语言模型似然提升最大的对」：

WordPiece 的合并标准：选能让 score = freq(xy) / (freq(x)·freq(y)) 最大的对
（类似点互信息 PMI，衡量 x、y 共现的「意外程度」）

直观说：BPE 关心「这对有多常出现」，WordPiece 关心「这对的出现是不是特别显著（相对于各自单独出现的频率）」。BERT、DistilBERT 用 WordPiece。标记上 WordPiece 用「##」表示非词首的子词（如 "playing" → ["play", "##ing"]）。

实践中 BPE 和 WordPiece 效果接近，BPE 因为实现简单、和 byte-level 结合好，在生成式 LLM 里更流行。WordPiece 主要在 BERT 系的 encoder 模型里用。

05 Unigram：概率视角

Unigram Language Model（SentencePiece 的一种模式）走完全不同的路：它不「合并」，而是「从一个大词表里逐步删词」。

初始化大词表

用所有可能的子串（或 BPE 结果）作为初始词表，可能有几百万个候选。

用 EM 训练概率

给每个 token 一个概率，让整个语料的「分词似然」最大。用期望最大化（EM）迭代。

删低贡献词

计算每个 token 对总似然的贡献，删掉贡献最低的（比如删 20%）。重复训练。

直到目标大小

循环到词表达到目标大小。最终词表里每个 token 都有概率，分词时选概率最高的切分。

Unigram 的好处：分词不是唯一的——同一个词可能有多种切分，每种有不同概率。这让模型能学到「这个切分更可能」的软选择。而且因为它基于概率，对未见过的词有更好的泛化。T5、mBART、ALBERT 用 Unigram（通过 SentencePiece）。

06 SentencePiece：跨语言的工程实现

SentencePiece（Google）不是一个新算法，而是 BPE 和 Unigram 的工程实现框架。它解决了几个实际痛点：

  • 语言无关：不依赖空格分词（英文有空格，中文日文没有）。直接在原始文本或字节上工作。
  • 把空格当普通字符：用特殊符号 ▁（U+2581）表示空格。这样「 hello」和「hello」是不同的 token，空格信息不丢失。
  • 支持 BPE 和 Unigram：用户可以选算法。
  • 可逆：分词后能无损还原原文（包括空格）。

Llama、T5、GLM 的分词器都用 SentencePiece（或它的变体）。对中文这种无空格语言，SentencePiece 的「空格当字符」设计特别重要——它让模型能学到中文词边界。

07 中文分词的特殊问题

中文没有空格，这给分词带来独特挑战。同一个模型，中文的 token 效率（每 token 承载的信息）往往低于英文，因为：

  • 词表偏向英文：很多开源模型的词表英文 token 占比高，中文常见字可能没进词表，要拆成 UTF-8 字节（3 字节/字），token 数膨胀。
  • 词边界模糊：「自然语言处理」是「自然/语言/处理」还是「自然语言/处理」？不同切分语义不同。BPE 是按频率学的，可能切得不「语义」。
  • 「的字了」等高频字：这些字单独成 token，但很多短语被切碎。

对比同样意思的中英文在 GPT-4 分词器下的 token 数：

语言句子字符数Token 数效率
英文"I love natural language processing"35~75 字符/token
中文"我喜欢自然语言处理"10~61.7 字符/token

中文的 token 效率明显低——同样意思的话，中文要花更多 token，也就是更多钱、更慢。这就是为什么国产大模型（GLM、Qwen、DeepSeek）都专门优化中文词表：在预训练前，把中文常见字、常见词加入词表，让中文切得更粗。GLM-4 的中文 token 效率比 GPT-4 高约 2~3 倍。

⚠️ Token 效率直接影响成本

API 按 token 计费。如果你的应用主要处理中文，选一个中文友好的模型（如 GLM）能省一大笔钱。反之用英文优化的模型，中文成本可能翻 2~3 倍。这是选型时常被忽视的因素。

08 词表大小怎么选

词表大小是个 trade-off：

  • 太小（~1 万）：常见词也被切碎，序列变长，推理慢、成本高。
  • 适中（3~10 万）：平衡。GPT-2 是 5 万，Llama 2 是 3.2 万，GPT-4 是 10 万。
  • 太大（50 万+）：embedding 层参数爆炸（词表 × d_model），且很多 token 训练数据不足，学不好。

embedding 层的参数量 = vocab_size × d_model。Llama-2-7B：32000 × 4096 = 1.3 亿（占总参数 7B 的约 2%）。GPT-4 假设词表 10 万、d_model=12288：10 万 × 12288 = 12 亿——光 embedding 就 12 亿参数。所以词表越大，embedding 越占参数，这也是为什么词表不能无限大。

多语言模型的词表通常更大（要覆盖多种语言的字符和常见词），Llama 3 把词表从 3.2 万扩到 12.8 万就是为了更好的多语言和代码支持。

09 特殊 token 和代码、emoji 处理

除了从语料学的子词，分词器还有一批特殊 token：

  • BOS/EOS（Begin/End of Sequence）：序列的起止标记。
  • PAD：填充 token，用于 batch 内对齐长度。
  • UNK（Unknown）：理论上 byte-level 不会有 UNK（总能拆到字节），但有些实现保留。
  • 系统/用户/助手标记：聊天模型的对话结构标记（<|im_start|> 等）。

代码和 emoji 的处理：byte-level BPE 天然支持（都是字节），但效果取决于训练数据。如果训练语料代码少，代码会被切得很碎（每个符号一个 token），序列膨胀。这就是为什么编程能力强的模型（如 GLM、Claude）都在训练语料里混入大量代码——让 BPE 学到代码的常见结构（如 "def"、"function"、"(){}"）作为单 token。

10 四种算法总结

算法思路代表模型工具
BPE / Byte-level BPE贪心合并最高频对GPT 全系列, Llama, GLMtiktoken, SentencePiece
WordPiece合并似然提升最大的对BERT, DistilBERTtokenizers (HF)

📌 现代 LLM 的主流选择

2024+ 的开源大模型几乎清一色用 Byte-level BPE（通过 tiktoken 或 SentencePiece 实现）。原因：①byte-level 无 OOV、语言无关；②BPE 实现简单、效果好；③和 GPT 生态兼容。WordPiece 和 Unigram 主要在 BERT 系和老模型里用。

11 本章小结

分词的本质
  • 文本 → token ID 序列，子词粒度
  • BPE：贪心合并高频对
  • WordPiece：合并似然提升最大的对
  • Unigram：从大词表删低贡献词
工程要点
  • Byte-level BPE 是现代 LLM 主流
  • 词表大小 3~13 万，影响 embedding 参数
  • 中文 token 效率低于英文，国产模型优化了
  • token 数直接影响推理成本

分词把文本变成了 token ID。但这些 ID 只是无意义的数字——「猫」是 234、「狗」是 567，数字大小不代表语义。怎么把这些 ID 变成有语义的向量？这就是下一章词嵌入（Embedding）的主题。我们会讲清 embedding 的查表本质、维度选择、位置 embedding 的组合，以及 tied embedding（输入输出权重共享）这个省参数又提效果的技巧。
