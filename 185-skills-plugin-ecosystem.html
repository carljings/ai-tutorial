技能生态与插件市场

技能生态与插件市场

Skills 技能体系源码深挖★★★ 进阶

从个人技能到技能生态

前面几章我们关注"怎么写一个技能"。但当技能数量多了，就出现了平台级问题：怎么让技能被发现、被分享、被复用？怎么让不同人写的技能能协同？怎么建立一个可持续的生态？这就是"技能生态"要解决的——它把技能从个人工具升级为可交易、可组合的平台能力。

插件市场：技能的分发中枢

技能生态的核心基础设施是"插件市场"（Plugin Marketplace）——一个集中式的技能注册、发现、分发平台。类比手机应用商店：开发者上架技能，用户搜索安装，平台负责质量把控和分发。

一个成熟的插件市场需要这些能力：

能力作用类比
搜索与分类用户能按关键词/类别找到技能App Store 搜索
评分与评论社区反馈帮用户筛选质量应用评分
版本管理技能可多版本并存、回滚软件版本控制
依赖解析技能 A 依赖技能 B，自动装上npm/pip 依赖
安全审核上架前扫描恶意指令应用安全审查
使用统计下载量、活跃度帮开发者决策应用数据分析

技能的组合与依赖

高级的技能生态支持"组合"——一个技能可以依赖或调用另一个技能。比如"全栈开发"技能可能依赖"代码审查"技能和"测试生成"技能。这要求技能系统支持依赖声明和自动安装。

---
name: fullstack-dev
description: 全栈开发辅助：编码、审查、测试一条龙
dependencies:
  - team-code-review    # 依赖代码审查技能
  - test-generator      # 依赖测试生成技能
  - git-workflow        # 依赖 Git 工作流技能
---

依赖管理带来力量也带来复杂性：版本冲突（A 要 review v1，B 要 review v2）、循环依赖、安全传递（依赖的技能是否可信）。这些是生态规模化的必经挑战。

社区与治理

技能生态的长期活力来自社区。需要解决的治理问题：谁来审核技能的安全性？标准由谁定？纠纷怎么处理？利益怎么分配？参考开源软件生态的经验，通常的模式是：核心团队维护标准和官方市场，社区贡献第三方技能，通过评分和"官方认证"标识建立信任梯度。

💡 要点
技能生态成败的关键不是技术，而是治理。再好的技术平台，没有质量把控和激励机制，也会被低质或恶意内容淹没。信任是生态的货币。

plugin.json 结构详解：插件的元数据契约

前面的 YAML 是技能依赖声明的简化版。真实的插件清单（manifest）要丰富得多——它不仅是依赖列表，更是插件与宿主之间的完整契约：声明权限、暴露的工具、配置项、兼容性。下面是一个贴近真实标准的 plugin.json 完整结构。

{
  "name": "fullstack-dev",
  "version": "2.1.0",
  "description": "全栈开发辅助：编码、审查、测试、Git 工作流一条龙",
  "author": {
    "name": "DevTools Inc",
    "email": "dev@example.com",
    "url": "https://example.com"
  },
  "license": "MIT",
  "homepage": "https://example.com/fullstack-dev",
  "repository": "https://github.com/example/fullstack-dev",

  "entry": "./index.js",
  "main_skill": "./skills/SKILL.md",

  "permissions": {
    "filesystem": {
      "read": ["./**", "../shared/**"],
      "write": ["./src/**", "./tests/**"]
    },
    "network": ["api.github.com", "registry.npmjs.org"],
    "shell": ["git", "npm", "node"],
    "env": ["GITHUB_TOKEN"]
  },

  "tools": [
    {
      "name": "run_tests",
      "description": "运行项目测试套件并返回结果摘要",
      "parameters": {"suite": {"type": "string", "default": "all"}}
    },
    {
      "name": "format_code",
      "description": "用项目配置的 formatter 格式化代码"
    }
  ],

  "config": {
    "linter": {
      "type": "string",
      "default": "eslint",
      "description": "使用的代码检查工具"
    },
    "auto_fix": {
      "type": "boolean",
      "default": true,
      "description": "是否自动修复可修复的问题"
    }
  },

  "dependencies": {
    "team-code-review": "^1.0.0",
    "test-generator": "^2.1.0",
    "git-workflow": "^1.2.0"
  },

  "compatibility": {
    "host_version": ">=1.5.0 <3.0.0",
    "platforms": ["macos", "linux", "windows"]
  }
}

这个 manifest 里有几个关键字段值得注意：① `permissions` 是插件安全的基石——它显式声明了插件能访问哪些文件、网络、命令、环境变量，宿主据此做沙箱限制（后面详述）；② `config` 让插件可被用户定制（如选 linter），无需改代码；③ `compatibility.host_version` 用 semver 范围声明兼容的宿主版本，防止插件在不兼容的宿主上崩溃。这三个字段是把"个人技能"升级为"可分发插件"的分水岭——没有它们，插件就只是个脚本，谈不上生态。

插件加载机制：从清单到运行时

有了 manifest，宿主怎么把它加载进运行时？核心流程是：发现 → 校验 → 依赖解析 → 沙箱实例化 → 注册工具。下面这段 Python 代码实现了一个最小可用的插件加载器，展示了整个生命周期。

import json
import importlib.util
import semver
from pathlib import Path

class PluginLoader:
    def __init__(self, host_version="2.0.0"):
        self.host_version = host_version
        self.loaded = {}      # name → plugin instance
        self.registry = {}    # name → manifest

    def load(self, plugin_dir: str):
        """从目录加载一个插件。"""
        manifest_path = Path(plugin_dir) / "plugin.json"
        manifest = json.loads(manifest_path.read_text())

        # 1) 校验清单完整性
        for field in ("name", "version", "entry", "permissions"):
            if field not in manifest:
                raise ValueError(f"插件清单缺少必填字段: {field}")

        # 2) 兼容性检查（semver 范围）
        compat = manifest.get("compatibility", {}).get("host_version")
        if compat and not self._version_match(self.host_version, compat):
            raise RuntimeError(
                f"插件 {manifest['name']} 要求宿主 {compat}，"
                f"当前 {self.host_version} 不兼容"
            )

        # 3) 依赖解析（递归加载依赖）
        for dep, version_range in manifest.get("dependencies", {}).items():
            if dep not in self.loaded:
                dep_dir = self._resolve_dependency(dep, version_range)
                self.load(dep_dir)   # 递归

        # 4) 在沙箱里实例化插件
        sandbox = Sandbox(manifest["permissions"])
        entry = Path(plugin_dir) / manifest["entry"]
        plugin = self._instantiate(entry, sandbox, manifest)

        # 5) 注册到全局表
        self.loaded[manifest["name"]] = plugin
        self.registry[manifest["name"]] = manifest
        return plugin

    def _version_match(self, current: str, range_str: str) -> bool:
        """简化的 semver 范围匹配：'>=1.0.0 <3.0.0'。"""
        # 真实实现用 semver 库的 satisfies()
        return semver.match(current, range_str)

    def _instantiate(self, entry_path, sandbox, manifest):
        """动态导入插件入口模块，注入沙箱。"""
        spec = importlib.util.spec_from_file_location(
            manifest["name"], entry_path
        )
        module = importlib.util.module_from_spec(spec)
        module.sandbox = sandbox        # 注入权限沙箱
        module.config = manifest.get("config", {})
        spec.loader.exec_module(module)
        return module

    def get_tools(self):
        """汇总所有已加载插件暴露的工具。"""
        tools = []
        for name, plugin in self.loaded.items():
            if hasattr(plugin, "export_tools"):
                tools.extend(plugin.export_tools())
        return tools

# 沙箱：根据 permissions 限制插件能力
class Sandbox:
    def __init__(self, permissions):
        self.perms = permissions

    def can_read(self, path):
        return self._match_path(path, self.perms.get("filesystem", {}).get("read", []))

    def can_write(self, path):
        return self._match_path(path, self.perms.get("filesystem", {}).get("write", []))

    def can_network(self, host):
        return host in self.perms.get("network", [])

    def _match_path(self, path, patterns):
        from fnmatch import fnmatch
        return any(fnmatch(path, p) for p in patterns)

这个加载器揭示了插件系统的几个关键工程点：① `compatibility` 检查在加载时就拦下不兼容的插件，避免运行时崩溃——这是生态质量的基本保障；② 依赖是递归加载的（A 依赖 B，加载 A 时自动先加载 B），但要小心循环依赖，真实实现要加 visited 集合防环；③ Sandbox 把 permissions 清单变成运行时强制约束——插件尝试读未授权文件时，sandbox.can_read() 返回 False 直接拒绝。这套机制让"第三方插件"可以被安全地加载运行，是整个生态信任的基础。

💡 要点
动态加载（importlib）是有安全风险的——执行了第三方代码。生产环境绝不能直接 exec 不可信的 Python 文件，要用进程隔离（子进程 + 能力受限的 IPC）或 WASM 沙箱。上面代码展示的是信任环境（如企业内部插件）的简化模型，对外部分发的插件，进程级隔离是底线。

权限模型：最小权限原则的工程化

权限模型是插件生态安全的最后一道防线。核心原则是"最小权限"——插件只拿到它声明且用户批准的最小能力集合。下面这个表梳理了常见的权限维度和它们的危险等级，是设计审批界面和风险提示的依据。

权限维度风险等级典型场景防护措施
只读文件低代码分析、文档检索路径白名单
写文件中代码生成、格式化路径白名单 + 回收站
网络访问中API 调用、搜索域名白名单 + 流量审计
Shell 命令高构建、测试、git命令白名单 + 审批
环境变量高读取 token显式声明 + 用户确认
任意代码执行极高代码插件进程隔离 + 超时 + 资源限制

这个权限模型要和 UI 配合：当用户安装一个请求"Shell 命令"权限的插件时，安装界面必须高亮提示风险（红色警告 + 具体命令列表），而非和其他低危权限混在一起默认同意。这就是前面 plugin.json 里 `permissions.shell: ["git", "npm", "node"]` 的价值——它不只说"我要 shell 权限"，而是精确到"我只要这三个命令"。用户能据此判断"这个代码插件要 git/npm 很合理"，而一个天气插件要 shell 就明显可疑。粒度越细的权限声明，越能建立用户信任——这是生态长期健康的关键。
