LLM 测试驱动开发实践
给不确定性带来确定性

传统软件有单元测试。LLM 输出不确定，怎么测试？本章讲清LLM 测试方法论——从 assert 到 LLM-as-Judge 到评估集，让你的 AI 系统像传统软件一样可测、可靠、可回归。

阅读约 35 分钟难度 ★★★★前置：第 24、74 章

01 LLM 测试的挑战

传统 assert 对 LLM 几乎没用——assert output == "expected" 永远失败（模型不会每次输出一模一样）。LLM 测试需要不同的策略：

传统测试LLM 测试
精确匹配 assert x == y模糊匹配/包含/语义相似
确定性（同输入同输出）概率性（同输入可能不同输出）
pass/fail 二元评分（0-10/多维度）
快（毫秒级）慢（秒级，要调模型）
免费贵（消耗 token）

02 六种 LLM 测试策略

📝

1. 包含/排除测试

检查输出是否包含/不包含特定关键词。最简单。

📐

2. 格式测试

JSON schema 校验/正则/长度/结构。

🔢

3. 数值测试

涉及数字的：范围/精度/单位。
🤖

4. LLM-as-Judge

用强模型当裁判打分。规模化但需校准。

👥

5. 人工评审

金标准，贵但可靠。适合关键测试。

📊

6. 评估集回归

一批标注样本，每次改动跑，统计通过率。

03 LLM 的 TDD 流程

llm_tdd.py —— LLM 测试驱动开发
"""
llm_tdd.py —— LLM 应用的测试驱动开发
先写评估集，再开发，每次改动回归测试
"""
import pytest

# ① 评估集（你的"测试用例"）
EVAL_SET = [
    {"input": "GLM-5.2 上下文多长", "must_contain": ["1M", "百万"], "must_not_contain": ["128K"]},
    {"input": "写一个 hello world", "must_contain": ["print"], "format": "code"},
    {"input": "今天天气", "semantic_check": "回答应该提到无法获取实时天气"},
]

# ② 测试函数
@ pytest.mark.parametrize("case", EVAL_SET)
def test_llm_response(case):
    output = my_llm_app(case["input"])

    # 包含检查
    for keyword in case.get("must_contain", []):
        assert keyword in output, f"应包含 '{keyword}'，实际: {output[:100]}"

    # 排除检查
    for keyword in case.get("must_not_contain", []):
        assert keyword not in output, f"不应包含 '{keyword}'"

    # 格式检查
    if case.get("format") == "code":
        assert "```" in output or "def " in output

    # 语义检查（LLM-as-Judge）
    if "semantic_check" in case:
        score = llm_judge(output, case["semantic_check"])
        assert score >= 0.7, f"语义检查不通过: {case['semantic_check']}"

# ③ 运行：pytest test_llm.py -v
# 每次改 prompt/模型/RAG 后都跑，确保不退化

04 评估集设计原则

🎯

覆盖多样

覆盖正常/边界/异常/安全场景。

📏

分层难度

简单/中等/难各占合理比例。

🔄

持续更新

把线上 bad case 持续加入。

📊

量化指标

通过率/平均分/分布，不是主观判断。

07 AI 辅助 TDD 完整流程

前三节讲了「为什么测」「测什么」「怎么放进 CI」，但都默认测试是人写的。实际上 2026 年的 TDD 已经是「AI 写测试、人评审」的双人舞。这里给出一套完整的 AI 辅助 TDD 工作流：从一个需求描述出发，到产出可运行的测试和实现，全程可复现。

关键原则只有一条：红 → 绿 → 重构的顺序不变，但每一步都用 AI 加速。AI 写的测试，人必须读懂、必须能跑红；AI 写的实现，必须先跑过测试才能合并。

ai_tdd.py —— AI 辅助 TDD 的端到端循环
"""
ai_tdd.py —— 从需求到通过测试的自动化 TDD 循环
步骤：生成测试 → 跑红 → 生成实现 → 跑绿 → 重构
"""
import subprocess, json, textwrap
from zhipuai import ZhipuAI
client = ZhipuAI()

def gen_tests(spec, target_file, test_file):
    """① 用 AI 从需求生成 pytest 测试，强制失败断言"""
    prompt = textwrap.dedent(f"""\
        需求：{spec}
        目标函数文件：{target_file}
        请按以下规则生成 pytest 测试文件 {test_file}：
        1) 只写测试，不要写实现；
        2) import 来自 {target_file} 的函数；
        3) 至少覆盖：正常值、边界值、异常值各一；
        4) 数值断言用 pytest.approx 容忍浮点误差；
        5) 直接输出代码，不要解释。""")
    code = client.chat.completions.create(
        model="glm-5.2",
        messages=[{"role":"user","content":prompt}]).choices[0].message.content
    # 去掉可能包裹的 ```python 围栏
    code = code.replace("```python", "").replace("```", "")
    open(test_file, "w").write(code)

def run_pytest(test_file):
    """跑测试，返回 (passed, 失败摘要)"""
    r = subprocess.run(["pytest", test_file, "-q", "--tb=line"],
                      capture_output=True, text=True)
    passed = r.returncode == 0
    return passed, r.stdout + r.stderr

def gen_impl(spec, target_file, test_file, failure_msg):
    """② 看到失败后，用 AI 写最小实现去通过测试"""
    prompt = textwrap.dedent(f"""\
        需求：{spec}
        测试文件内容：
        {open(test_file).read()}
        当前失败信息：
        {failure_msg}
        请写 {target_file} 的最小实现，要求：
        1) 只写让测试通过的代码，不要过度设计；
        2) 不要修改测试文件；
        3) 直接输出代码。""")
    code = client.chat.completions.create(
        model="glm-5.2",
        messages=[{"role":"user","content":prompt}]).choices[0].message.content
    code = code.replace("```python", "").replace("```", "")
    open(target_file, "w").write(code)

def ai_tdd(spec, target_file, test_file, max_attempts=3):
    """主循环：红→绿→（绿则结束）"""
    gen_tests(spec, target_file, test_file)
    for attempt in range(1, max_attempts + 1):
        ok, msg = run_pytest(test_file)
        if ok:
            print(f"✅ 全部通过（第 {attempt} 轮）")
            return True
        print(f"🔴 仍有失败（第 {attempt}/{max_attempts} 轮），让 AI 修实现...")
        gen_impl(spec, target_file, test_file, msg)
    print("⚠ 达到最大尝试次数仍未通过，需人工介入")
    return False
⚠ AI 生成测试的反模式

AI 经常会「顺手」把实现也写进测试文件里，或者为了让测试好看而降低断言强度（例如把精确值改成 assert result is not None）。必须在 prompt 里硬性禁止这两点，否则 TDD 就退化成了「AI 自己出题自己答」，红绿循环失去意义。

08 测试生成 Prompt 模板库

第 07 节的 gen_tests 是通用模板。实际工程里不同类型的代码需要不同的测试模板。下面这张表给出六类常见代码的测试生成 prompt 骨架，每个都针对该类代码最常被忽略的测试维度。

代码类型prompt 关键约束常被漏掉的维度
纯函数（数学/转换）「覆盖 0、负数、极大值、NaN」边界值、精度
LLM 调用封装「mock 模型，断言调用次数与异常分支」异常、超时
RAG 检索「构造可控语料，断言召回顺序」空结果、多语言

# 模板 1：LLM 调用封装 —— 必须 mock 模型，不能真调
PROMPT_LLM_WRAPPER = """
你正在为函数 {fn_name} 写 pytest 测试。
这个函数内部会调用 LLM。请：
1) 用 unittest.mock.patch 把 LLM 调用 mock 掉；
2) 构造三种返回：正常文本、空字符串、抛 TimeoutError；
3) 断言每种情况下函数的行为符合预期；
4) 不要真的发起网络请求。直接输出测试代码。
"""

# 模板 2：结构化输出 —— 强制 schema 校验
PROMPT_STRUCT = """
为函数 {fn_name} 写测试，该函数返回一个 dict。
schema：{schema_json}
请：
1) 用 jsonschema.Draft7Validator 校验返回值；
2) 至少 2 个 case 应该是合法返回；
3) 至少 1 个 case 构造非法输入（缺字段/类型错），断言函数内部抛出明确异常。
直接输出代码。
"""

# 模板 3：Agent 工具 —— 权限与降级是核心
PROMPT_AGENT_TOOL = """
为工具函数 {fn_name} 写测试。该工具会被 Agent 调用。
请覆盖：
1) 权限：未授权用户调用应抛 PermissionError；
2) 超时：模拟工具执行超过 3 秒，断言走降级分支；
3) 滥用：连续调用超过 10 次，断言被限流；
4) 参数：缺少必填参数应抛 ValueError。
不要写实际业务逻辑的测试，只测这些守卫逻辑。直接输出代码。
"""

def pick_template(code_kind):
    return {
        "llm_wrapper": PROMPT_LLM_WRAPPER,
        "structured": PROMPT_STRUCT,
        "agent_tool": PROMPT_AGENT_TOOL,
    }[code_kind]

用模板库的好处是一致性：同一个团队生成的测试都覆盖相同维度，review 时只需要看「是否漏了 schema 校验」「是否真调了模型」这种清单项，而不是逐行读懂 AI 写的测试逻辑。

深 LLM 的测试驱动：给不确定性加护栏

传统 TDD 在 LLM 上不能直接套用——输出不确定，断言“等于某值”会随机失败。但测试的思路依然有价值，只需把“精确断言”换成“属性断言”。下面给出 LLM 版的 TDD 方法。

属性测试：断言“性质”而非“精确值”

核心思路：不测“输出是 X”，而测“输出满足某些性质”（是合法 JSON、长度在范围、不含敏感词、语义相关）。下面是属性测试框架。

import json, re

def property_test(output, checks):
    """对 LLM 输出做属性断言，而非精确值断言。"""
    failures = []
    for name, fn in checks.items():
        try:
            if not fn(output):
                failures.append(f"{name} 不满足")
        except Exception as e:
            failures.append(f"{name} 异常: {e}")
    return failures

# 例：测一个'返回JSON格式产品信息'的LLM功能
def is_valid_json(s):
    try: json.loads(s); return True
    except: return False

checks = {
    "是合法JSON":   is_valid_json,
    "含name字段":   lambda s: "name" in json.loads(s),
    "name非空":     lambda s: len(json.loads(s).get("name","")) > 0,
    "无敏感词":     lambda s: "密码" not in s and "token" not in s.lower(),
    "长度合理":     lambda s: 10 < len(s) < 500,
}
out = '{"name": "智能音箱", "price": 299}'
print(property_test(out, checks) or "全部通过")
# 属性测试让 LLM 输出'虽不确定但满足约束'，可进 CI 而不随机崩

这套属性测试把“不确定输出”纳入了可测试范畴：不要求精确匹配，只要求满足关键性质，这样测试不会因采样随机而 flaky，又能守住质量底线。

测试类型传统断言LLM 属性断言适用
格式等于某字符串是合法JSON/XML结构化输出
长度等于N在[a,b]区间摘要/生成
内容含某词语义相关(相似度>阈值)问答/抽取
安全无不含敏感/有害所有输出
鲁棒无输入扰动后仍合格稳定性

评估集即“测试”，CI 即“回归防护”

在 LLM 语境下，评估集就是测试集，CI 跑评估就是回归测试。把 TDD 的“先写测试”迁移为“先建评估集”：开发新功能前，先准备一批预期输入和“合格标准”（属性断言），开发过程中持续跑这批测试，合格率达标才算完成。这比“开发完再想怎么测”健康得多。golden set 是核心资产：它既是 CI 门禁，也是质量基线，要持续维护、覆盖正常/边界/对抗场景。防 flaky 是 LLM 测试的关键——属性断言比精确断言稳，但仍可能因采样偶尔失败，解法是允许少量容差（如合格率 ≥95% 而非 100%）、或多次采样取多数。测试要分层：单元级（单个 prompt 的属性）、集成级（端到端流程的属性）、回归级（golden set 整体准确率），各层关注点不同。失败要可定位：测试挂了要能看出是哪个 case、什么属性不满足、什么输入触发的，而不是只给个“合格率 80%”。最后，把对抗测试纳入 CI：定期跑 Prompt Injection、越狱、边界输入测试，确保安全护栏不被绕过。LLM 的 TDD 本质是用属性和评估给不确定性加护栏——无法消灭不确定性，但能让它落在可接受的边界内，这才是 LLM 工程化“可靠性”的真正含义。

深 LLM 测试 FAQ 与属性断言

LLM 测试的核心是「属性断言」，这些问题最实用。

Q1：LLM 输出不确定，怎么写不会 flaky 的测试？

断言属性而非精确值。不测「输出等于 X」，测「输出是合法 JSON、长度在 [a,b]、含必要字段、语义相关（相似度>阈值）、无敏感词」。属性断言容忍采样随机，不 flaky，又守质量底线。多次采样取多数进一步稳。允许少量容差（合格率≥95% 而非 100%）。temperature=0 调试时复现，但生产测试要接受微小波动。分层测试：单元级（单 prompt 属性）、集成级（端到端）、回归级（golden set 整体准确率）。属性断言是把不确定输出纳入可测试范畴的关键。

Q2：怎么测 LLM 的鲁棒性（输入扰动不崩）？

对抗/边界测试集。准备一组扰动输入（同义改写、错别字、超长、空、特殊字符、多语言、注入尝试），跑模型看输出是否仍合格（属性断言）。不变性测试：同义改写不应改变分类结果。边界测试：空/超长/特殊字符不崩、有兜底。安全测试：注入/越狱不成功。把这些纳入 CI 定期跑，防退化。鲁棒性测试暴露的是「偶发但致命」的问题（如某类输入必崩），上线前必须覆盖。记住 LLM 测试目标不是「证明对」（不可能），而是「守住边界」（不崩、合规、满足关键属性）。

★ TDD 在 AI 与复杂系统中的适配

测试驱动开发（TDD）在传统软件中已有成熟实践，但在 AI 系统中需要做重要的适配。核心困难在于：AI 模型的输出本质上是概率性的，"正确答案"往往不是一个确定值，而是一个可接受的分布范围。因此传统的"断言相等"在很多场景下失效，需要替换为"统计性断言"——例如断言模型在测试集上的准确率不低于某阈值，或断言输出与参考答案的语义相似度在某区间。这种断言天然带有噪声，需要合理设置容忍度和样本量，否则测试会频繁误报。

更实用的分层策略是：把系统拆成"确定性部分"和"概率性部分"分别测试。数据预处理、特征工程、后处理、业务规则、API 契约这些确定性逻辑，应当严格用 TDD 覆盖；而模型推理本身则用"回归测试集 + 阈值"来守护，且接受它偶尔的波动。混淆两者的测试粒度，会导致测试套件既脆弱又缓慢，最终被团队放弃。一个经验法则是：越靠近模型核心，测试越偏向"基准与阈值"；越远离模型，越偏向"严格的单元测试"。

此外，AI 系统的"数据漂移"使得测试用例本身会过期。一个今天通过的测试集，半年后可能因为真实数据分布变化而不再有代表性。因此 TDD 在 AI 场景下必须配合"测试集的持续更新机制"——定期用线上真实流量回灌测试集，剔除过时样本，补充新出现的边缘 case。没有这种维护，再完善的测试套件也会逐渐沦为"绿色的虚假安全感"。把测试数据当作与代码同等重要的资产来治理，是 AI 时代质量保障的关键认知。
