Harness 工程
让模型可靠跑起来的那圈「外壳」

2026 年 AI 圈最热的词之一就是 Harness（线束/外壳）。它指的是包裹在模型外面、让模型变成「可用产品」的整套工程系统。本章拆解 Claude Code 的五段式管线，讲清权限模型、Hook 系统、反馈循环、上下文管理这些把模型「驯化」成可靠助手的关键工程。

阅读约 35 分钟 难度 ★★★★ 前置：第 02 章

★ 到底什么是 Harness

先讲个反直觉的真相：同一个模型，套不同的 Harness，能力可以差出一个数量级。

你拿一个裸 GPT-4/GLM-5 的 API，问它「帮我把这个仓库重构一下」，它只能给你一段建议文字。但同样是这个模型，套上 Claude Code / Cursor / Copilot Workspace 这样的 Harness 后，它能自己读文件、改代码、跑测试、修 bug、提交 PR。能力仿佛凭空多了十倍。

这「凭空多出来的能力」，就是 Harness 给的。

Harness 的定义：围绕一个（或多个）大模型构建的工程系统，负责把「模型预测下一个 token」这件事，转变成「完成用户的一个真实任务」。它包括：上下文编排、工具接入、权限控制、反馈循环、记忆管理、错误恢复、可观测性。

一个类比：CPU 与主板

把大模型想象成 CPU——计算能力再强，没有主板（供电、总线、内存控制器、I/O）也跑不起来。Harness 就是 AI 时代的「主板」：它给模型供电（数据）、连总线（工具）、管内存（上下文）、做 I/O（与用户/系统交互）。选对 CPU 很重要，但设计好主板同样关键——这正是 2026 年 Harness 工程火热的原因。

01 为什么 2026 年 Harness 突然爆火

📈

模型趋同，工程显威

顶级模型（GPT、Claude、GLM）能力越来越接近，benchmark 差距缩小。真正的差异在 Harness。同样的 GLM-5.2，不同团队能做出天差地别的产品。

🔓

智能体能力解锁

模型原生支持长上下文、工具调用、多步推理后，具备了当智能体的潜质。但潜质需要 Harness 激活——否则就像有四肢却不会走路。

🏗️

开源基础设施成熟

MCP、Agent SDK、awesome-harness-engineering 等开源资源让「自己造 Harness」门槛大幅降低，社区开始大规模实践。

💰

商业价值兑现

Claude Code、Cursor 这类产品证明：好的 Harness 能把模型变成几十亿美元生意。资本和人才涌入。

2026 年 4 月，有人逆向工程了 Claude Code 的架构，发现它是一个五段式（five-stage）管线。这份分析（收录在 GitHub 的 awesome-harness-engineering）成了行业必读。下面我们对照它来拆解一个现代 Harness。

入口（Ingress）：理解用户意图

用户的一句话往往信息严重不足。「帮我重构这个文件」——哪个文件？重构到什么程度？保留什么风格？入口段要补全这些：

  • 意图解析：是问答、编辑、生成、还是执行任务？
  • 上下文补充：当前在哪个文件、选了哪段代码、最近的操作历史；
  • 权限检查：用户有没有权限做这件事？（避免越权）
  • 技能路由：这个任务匹配哪个 Skill？（详见第 8 章）

装配（Context Assembly）：给模型备好「弹药」

这是 Harness 最硬核的工程之一。模型只能看到你喂给它的上下文，喂什么、怎么喂、喂多少，直接决定输出质量。装配段要做的事：

收集候选上下文

系统提示词、技能说明、用户消息、相关文件、代码符号、RAG 检索结果、历史对话、工具定义……动辄上百 KB。

优先级排序

哪些是必须的（如当前文件）、哪些是相关的（如被引用的文件）、哪些是可选的（如全局规则）？给每个片段打分。

裁剪到预算

上下文窗口是有限的（即便是 1M 也不能塞所有东西，太长反而降低质量并增加成本）。按优先级塞，超出预算的低优先级片段丢弃。

结构化封装

用清晰的标签/分隔符组织上下文，让模型「看得懂」每一段是什么。例如 <file path="...">...</file>。

⚠️ 上下文不是越长越好

很多人误以为「上下文越长越准」。但研究表明：上下文越长，模型对中间内容的注意力越分散（Lost in the Middle），且推理变慢、成本飙升。聪明的 Harness 会精准裁剪，只放真正相关的——这就是为什么 RAG（第 6 章）和上下文压缩如此重要。

推理（Generation）：调用模型

把装配好的上下文喂给模型，拿到输出。这一段看似简单，实际有很多技巧：

  • 采样参数：temperature、top-p、top-k 的选择影响创造性与确定性；
  • 结构化输出：强制模型输出 JSON/特定格式，便于后续解析（GLM、Claude 都支持）；
  • 流式处理：边生成边处理，提升用户体验；
  • 多模型协作：复杂任务用强模型，简单子任务用快模型省钱；
  • 重试与回退：超时/限流时自动重试或切备用模型。

执行（Execution）：让输出落地

模型输出「我要读文件 X」「我要运行命令 Y」后，谁来真正执行？这就是执行段。它要：

  • 解析模型输出：从文本里抽出工具调用意图；
  • 权限二次校验：这个动作允许吗？（关键安全闸门）
  • 调用工具：读文件、写文件、跑命令、调 API；
  • 捕获结果：成功/失败、输出、耗时；
  • 结果格式化：把执行结果转成模型能理解的观察。

出口（Egress）：反馈与收尾

任务完成后，要做的事：

  • 把结果呈现给用户：diff、终端输出、总结；
  • 更新记忆：把这次会话的关键信息存下来（短期/长期记忆）；
  • 记录遥测：耗时、token 用量、工具调用次数，用于优化和计费；
  • 触发后续动作：比如自动跑测试、提交 PR。

03 权限模型：Harness 的安全闸门

让 AI 能读写文件、执行命令，意味着它有能力搞破坏。一个误删 rm -rf / 就能让系统报废。所以权限模型是 Harness 的生命线。

Claude Code、Claude Agent SDK 都内置了精细的权限模型，核心思路是分级授权 + 显式确认：

权限级别典型操作策略
只读读文件、列目录、查 git 状态默认允许
本地写改文件、新建文件、本地构建需确认或白名单
系统装软件、改系统配置、起服务必须显式确认
外联发网络请求、调外部 API、推送代码高风险，严格确认 + 审计
破坏删文件、强推、格式化默认禁止，需多重确认
🚨 权限模型三条铁律
  • 最小权限：默认只给完成任务必需的最小权限，不「图省事」全开；
  • 显式确认：高风险操作必须用户点击确认，不能自动执行；
  • 可审计：每一次工具调用都留日志，可追溯、可回放。

权限的配置方式

成熟的 Harness 通常支持多层配置：

权限配置示例（rules 文件，YAML 风格）
# .agent/rules.yaml —— 项目级权限规则
permissions:
  allow:
    - "read:./**"           # 读本项目所有文件
    - "shell:npm test"       # 允许跑测试
    - "shell:git status"
    - "shell:git diff:*"
  ask:                       # 需要确认
    - "write:./src/**"        # 改 src 下的代码要问
    - "shell:git commit:*"
  deny:                       # 绝对禁止
    - "write:./.env*"         # 不准碰环境变量
    - "shell:rm:*"            # 不准删
    - "shell:git push:*"      # 不准推（防止误推）
    - "net:*"                # 默认禁网

这种 allow / ask / deny 三态模型，比单纯的「允许/禁止」更灵活，能精确控制「哪些自动做、哪些要问、哪些绝不做」。Augment、Cursor 等都有类似的 rules 文件机制。

04 Hook 系统：Harness 的可扩展点

Hook（钩子）是 Harness 在关键节点暴露的「插拔点」，让你能在不修改核心逻辑的前提下注入自定义行为。这是把 Harness 从「黑盒」变成「可编程平台」的关键。

Hook 能干什么

🔍

审计与合规

记录每次工具调用，满足合规要求。所有 AI 操作留痕。

🛡️

安全沙箱

PreToolUse Hook 拦截危险操作（如外发敏感数据），实现二次防御。

🔄

自动反馈

PostToolUse Hook 在写文件后自动跑 lint/format，把结果反馈给模型自我修正。

📊

质量门禁

提交前自动跑测试，不通过就阻止提交。

一个 PreToolUse Hook 示例：阻止删除 .env 文件
# hooks/pre_tool_use.py
import sys, json, re

def main():
    event = json.load(sys.stdin)          # Harness 传入的事件
    tool  = event["tool_name"]
    args  = event["tool_input"]

    # 拦截危险操作
    if tool == "Bash" and re.search(r"rm\s+.*\.env", args.get("command", "")):
        # 返回非零退出码 + stderr 即可阻止该工具调用
        print("BLOCKED: 禁止删除 .env 文件", file=sys.stderr)
        sys.exit(2)

    # 放行
    sys.exit(0)

if __name__ == "__main__":
    main()

这个 Hook 被配置成「每次工具调用前执行」，它读 stdin 拿到事件，决定放行（exit 0）或阻止（exit 2）。Claude Code 的 Hook 系统就是这种设计——任何脚本/命令都能作为 Hook，门槛极低。

05 PEV 反馈环：让模型自我进化

Augment 的 Harness 工程指南里反复提到一个核心概念：PEV 环（Prompt-Execute-Verify）。这是让 AI 编程 Agent 从「瞎写」变成「可靠」的关键机制。

P: Prompt
▶
E: Execute
▶
V: Verify
↺
图 3-4 · PEV 环：生成 → 执行 → 验证 → 反馈 → 再生成

核心思想：不要相信模型的第一次输出，要让它「写完就验证，错了就改」。具体流程：

  1. P（Prompt）：模型生成一段代码/动作；
  2. E（Execute）：Harness 真的执行它（跑测试、起服务、调 API）；
  3. V（Verify）：检查执行结果——测试通过了吗？报错了吗？输出对吗？
  4. 反馈：把验证结果（尤其是失败信息）送回模型，让它修正后重试。

这个循环可以嵌套：外层 PEV 负责整个任务（功能完成了吗），内层 PEV 负责每一步（这一行代码对吗）。GLM-5.2 的 Agentic Coding 能力，本质上就是在用户给的 Harness 里跑这个 PEV 环，跑到任务完成。

💡 PEV 的本质：把「思考」变成「实验」

纯语言模型只能「想」，不能「试」。PEV 环给模型装上了「手」——它可以写代码、跑一下、看结果、再改。这把模型从「一次猜对」的赌博，变成了「反复试错直到成功」的工程。可靠性因此大幅提升。这也是为什么编程 Agent 比纯问答 Agent 成熟得快——代码可以自动验证。

06 上下文管理：Harness 的内存系统

长任务里，对话历史会越来越长，最终撑爆上下文窗口（即便是 1M 也有极限）。Harness 必须主动管理上下文，这叫上下文管理或记忆管理。

三层记忆模型

层级类比内容生命周期
工作记忆RAM当前对话/任务的上下文会话内
短期记忆SSD 缓存最近的会话摘要、近期决策数小时~数天
长期记忆硬盘 / 知识库用户偏好、项目规范、历史经验永久（需 RAG）

上下文压缩与摘要

当对话变长，常用技巧：

  • 滚动摘要（Rolling Summary）：把早期对话压缩成一段摘要，只保留近期原文；
  • 选择性保留：只保留关键决策、工具调用结果，丢弃寒暄和冗余；
  • 外部化：把不需要即时的信息写到文件/数据库，需要时再 RAG 取回；
  • 分块会话：把超长任务拆成多个子会话，每个子会话独立管理上下文。

Claude Code 的一个聪明设计是子智能体（Sub-agent）：主 Agent 把一个子任务（如「研究这个库的 API」）派给一个全新的子 Agent，子 Agent 用自己的独立上下文窗口完成，最后只把结论返回给主 Agent。这样主 Agent 的上下文不会被研究过程的细节撑爆。

07 可观测性：Harness 的仪表盘

生产级 Harness 必须能「看见自己」。一旦 AI 自主跑了 50 步然后失败，你必须知道是哪一步、为什么。可观测性包括：

📝

Trace（轨迹）

完整记录每一步：模型的思考、工具调用、参数、结果、耗时。像分布式追踪一样。

💰

Token / 成本追踪

每次调用消耗多少 token、多少钱。支持按用户/项目/任务计费。

📈

指标监控

成功率、平均步数、重试次数、工具调用分布。用于发现瓶颈。

🔁

回放与调试

能基于 Trace 重放某次执行，定位问题。这是排查「为什么 AI 这么干」的利器。

08 质量门禁：守住底线

Harness 要保证 AI 产出的东西至少不比人差太多。质量门禁是一系列自动检查：

  • 代码：lint、类型检查、单元测试、安全扫描（如密钥泄露检测）；
  • 文档：格式校验、链接有效性、术语一致性；
  • 操作：变更范围检查（别一次改 100 个文件）、危险命令拦截；
  • 输出：JSON schema 校验、必填字段检查。

门禁失败时，Harness 把失败信息喂回模型，让它在 PEV 环里自我修正。这套机制让 AI 从「偶尔惊艳」变成「持续可用」。

09 动手：一个最小可用 Harness

讲这么多，不如写一个。下面是一个不到 100 行的最小 Harness，它包含：系统提示、工具定义、权限确认、PEV 环。用 GLM-5.2 的 OpenAI 兼容 API：

mini_harness.py —— 一个可运行的最小 Harness
"""
mini_harness.py —— 最小可用 AI Harness
特性：系统提示 / 工具调用 / 权限确认 / PEV 反馈环
依赖：pip install openai
"""
import json, subprocess
from openai import OpenAI

client = OpenAI(
    api_key="你的智谱 API Key",
    base_url="https://open.bigmodel.cn/api/paas/v4",
)
MODEL = "glm-5.2"

# ---------- 工具定义 ----------
TOOLS = [{
    "type": "function",
    "function": {
        "name": "run_shell",
        "description": "在沙箱里执行 shell 命令并返回输出",
        "parameters": {
            "type": "object",
            "properties": {"command": {"type": "string"}},
            "required": ["command"],
        },
    },
}]

# ---------- 权限确认 ----------
DANGER = ("rm ", "sudo ", "git push", "> /", "curl ", "wget ")
def approve(cmd):
    if any(d in cmd for d in DANGER):
        return input(f"⚠️ 危险命令，确认执行? [y/N] {cmd}: ").lower() == "y"
    return True

# ---------- 工具执行 ----------
def execute(cmd):
    if not approve(cmd):
        return "[BLOCKED] 用户拒绝执行"
    try:
        r = subprocess.run(cmd, shell=True, capture_output=True, text=True, timeout=30)
        return (r.stdout + r.stderr)[:2000] or "[no output]"
    except Exception as e:
        return f"[ERROR] {e}"

# ---------- 系统提示（Harness 的「性格」） ----------
SYSTEM = """你是一个终端助手。完成任务时：
1. 先思考需要哪些信息，用 run_shell 工具获取；
2. 每次只执行安全的只读命令，涉及修改必须先说明；
3. 命令失败时，分析错误并重试（PEV 思维）；
4. 任务完成后给出简短总结。"""

# ---------- 主循环（就是下一章要讲的 Agent Loop！） ----------
def run(task):
    messages = [{"role": "system", "content": SYSTEM}, {"role": "user", "content": task}]
    for step in range(12):                       # 最多 12 步，防止失控
        resp = client.chat.completions.create(model=MODEL, messages=messages, tools=TOOLS)
        msg = resp.choices[0].message
        messages.append(msg)
        if not msg.tool_calls:                     # 模型不再调工具 → 给最终答案
            print(f"\n🤖 {msg.content}")
            return
        for tc in msg.tool_calls:               # E: 执行每个工具调用
            args = json.loads(tc.function.arguments)
            print(f"\n🔧 执行: {args['command']}")
            out = execute(args["command"])      # V: 拿到结果
            print(f"📋 结果: {out[:200]}...")
            messages.append({"role": "tool", "tool_call_id": tc.id, "content": out})
    print("⏰ 达到最大步数，停止。")

if __name__ == "__main__":
    run(input("任务: "))

这个 90 行的程序已经具备了一个 Harness 的所有核心要素：系统提示（性格）、工具定义（能力）、权限确认（安全）、PEV 环（可靠性）、步数限制（防失控）。把它跑起来，你就理解了 Claude Code 在最底层做的事。

11 代码实战：一个完整的权限引擎

第 3 节讲了权限模型的理念，这里给一个真正可用的权限引擎——支持 allow/ask/deny 三态、glob 模式匹配、多规则优先级合并。它是任何生产级 Harness 的安全基石。

permissions.py —— 生产级权限引擎
"""
permissions.py —— Harness 权限引擎
支持 allow/ask/deny 三态 + glob 匹配 + 规则合并
"""
import fnmatch
from dataclasses import dataclass
from enum import Enum

class Decision(Enum):
    ALLOW = "allow"     # 直接放行
    ASK   = "ask"       # 需要用户确认
    DENY  = "deny"      # 拒绝

@dataclass
class Rule:
    action: str          # 如 "write:./src/**" "shell:rm:*"
    decision: Decision
    reason: str = ""

class PermissionEngine:
    def __init__(self, rules):
        # deny 优先级最高，ask 其次，allow 最低——保证安全默认
        self.rules = sorted(rules, key=lambda r: {Decision.DENY:0, Decision.ASK:1, Decision.ALLOW:2}[r.decision])

    def check(self, requested_action) -> Rule:
        # 逐条匹配，返回第一个命中的规则
        for rule in self.rules:
            if fnmatch.fnmatch(requested_action, rule.action):
                return rule
        # 默认拒绝（最小权限原则）
        return Rule(requested_action, Decision.DENY, "无匹配规则，默认拒绝")

# ---------- 使用示例 ----------
rules = [
    Rule("read:**",             Decision.ALLOW, "允许读所有文件"),
    Rule("shell:git status",   Decision.ALLOW, "允许查 git 状态"),
    Rule("shell:npm test",     Decision.ALLOW, "允许跑测试"),
    Rule("write:./src/**",     Decision.ASK,   "改源码要确认"),
    Rule("shell:git commit:*", Decision.ASK,   "提交要确认"),
    Rule("write:./.env*",      Decision.DENY,  "禁止碰环境变量"),
    Rule("shell:rm:*",         Decision.DENY,  "禁止删除"),
    Rule("shell:git push:*",   Decision.DENY,  "禁止推送"),
    Rule("net:*",             Decision.DENY,  "默认禁网"),
]
engine = PermissionEngine(rules)

for act in ["read:./src/app.py", "write:./src/app.py", "write:./.env", "shell:rm -rf /"]:
    r = engine.check(act)
    print(f"{r.decision.value:6} | {act:25} | {r.reason}")
# 输出：
# allow  | read:./src/app.py       | 允许读所有文件
# ask    | write:./src/app.py      | 改源码要确认
# deny   | write:./.env            | 禁止碰环境变量
# deny   | shell:rm -rf /          | 禁止删除

注意几个设计要点：默认拒绝（无规则命中时拒绝，符合最小权限原则）、deny 优先匹配（即使有 allow 也先看 deny）、规则可来自多源合并（全局默认 + 项目级 + 用户级，层层覆盖）。

12 代码实战：上下文装配算法

第 2.2 节讲了上下文装配的理念。这里给一个带优先级和预算控制的装配器——它是 Harness 把「上百 KB 候选内容」裁剪到「模型窗口预算」内的核心算法。

context_assembler.py —— 优先级 + 预算的上下文装配
"""
context_assembler.py —— 把候选内容按优先级装进有限预算
"""
from dataclasses import dataclass

@dataclass
class ContextPiece:
    role: str          # "system" / "user" / "file" / "skill" / "history"
    content: str
    priority: int       # 数字越小优先级越高（0 最重要）
    tokens: int = 0      # 估算的 token 数

def estimate_tokens(text):
    return len(text) // 3   # 粗估：1 token ≈ 3 字符（中英文混合）

class ContextAssembler:
    def __init__(self, token_budget=120000):
        self.budget = token_budget
        self.pieces = []

    def add(self, role, content, priority):
        self.pieces.append(ContextPiece(role, content, priority, estimate_tokens(content)))

    def assemble(self):
        # 按优先级排序：priority 小的先装
        ordered = sorted(self.pieces, key=lambda p: p.priority)
        result, used = [], 0
        for p in ordered:
            if used + p.tokens <= self.budget:
                result.append(p); used += p.tokens
            else:
                # 高优先级溢出也要装（如系统提示），低优先级丢弃
                if p.priority == 0:
                    result.append(p); used += p.tokens
        return result, used

# ---------- 使用 ----------
asm = ContextAssembler(token_budget=1000)   # 故意设小，演示裁剪
asm.add("system",  "你是编程助手。",            priority=0)   # 必装
asm.add("user",    "帮我修这个函数的 bug",     priority=1)   # 必装
asm.add("file",    "def f(x): ..." * 50,       priority=2)   # 当前文件
asm.add("history", "之前的对话..." * 100,       priority=5)   # 历史，优先级低
asm.add("skill",   "调试技能说明...",          priority=3)

pieces, used = asm.assemble()
print(f"装入了 {len(pieces)} 块，共 {used} tokens（预算 1000）")
# 低优先级的 history 会被裁掉，保证 system/user/file 装得下

这个算法的精髓：不是简单的「先到先得」，而是按重要性智能裁剪。当预算紧张时，丢掉的是历史和次要内容，留下的是系统提示、用户问题、核心文件。生产级 Harness 还会加上「相关性打分」（用 Embedding 算 query 与各片段的相似度），让裁剪更精准。

13 代码实战：Hook 系统的完整实现

第 4 节讲了 Hook 的理念。这里实现一个支持多种 Hook 类型、可插拔的 Hook 管理器：

hooks.py —— 可插拔的 Hook 管理器
"""
hooks.py —— Harness 的 Hook 系统
支持 PreToolUse / PostToolUse / OnError / OnStep 等多种钩子
"""
from enum import Enum
from typing import Callable, List

class HookEvent(Enum):
    PRE_TOOL  = "pre_tool"     # 工具调用前：可拦截
    POST_TOOL = "post_tool"    # 工具调用后：可注入反馈
    ON_ERROR  = "on_error"     # 出错时
    ON_STEP   = "on_step"      # 每步循环

class HookManager:
    def __init__(self):
        self._hooks = {e: [] for e in HookEvent}

    def register(self, event: HookEvent, handler: Callable):
        self._hooks[event].append(handler)

    def run(self, event: HookEvent, ctx: dict) -> dict:
        # ctx 是上下文，handler 可修改它；返回 None 表示拦截
        for handler in self._hooks[event]:
            result = handler(ctx)
            if result is None:        # 拦截信号
                return None
            ctx = result or ctx
        return ctx

# ---------- 注册各种 Hook ----------
hooks = HookManager()

# Hook 1：审计日志（记录所有工具调用）
def audit_logger(ctx):
    print(f"\033[90m[审计] {ctx['tool']}({ctx.get('args')})\033[0m")
    return ctx
hooks.register(HookEvent.PRE_TOOL, audit_logger)

# Hook 2：危险命令拦截
def block_danger(ctx):
    if ctx["tool"] == "shell" and "rm -rf" in ctx.get("args", ""):
        print("\033[31m[拦截] 危险命令！\033[0m")
        return None      # 返回 None 阻止执行
    return ctx
hooks.register(HookEvent.PRE_TOOL, block_danger)

# Hook 3：写文件后自动格式化（PostToolUse）
def auto_format(ctx):
    if ctx["tool"] == "write_file" and ctx["path"].endswith(".py"):
        print("[Hook] 自动 black 格式化")
        ctx["feedback"] = "文件已自动格式化"
    return ctx
hooks.register(HookEvent.POST_TOOL, auto_format)

# Hook 4：步数计数与预算检查
state = {"step": 0}
def budget_check(ctx):
    state["step"] += 1
    if state["step"] > 15:
        ctx["force_stop"] = "达到步数上限"
    return ctx
hooks.register(HookEvent.ON_STEP, budget_check)

# 测试
hooks.run(HookEvent.PRE_TOOL, {"tool": "shell", "args": "ls -la"})      # 放行
hooks.run(HookEvent.PRE_TOOL, {"tool": "shell", "args": "rm -rf /"})      # 拦截

这个设计的好处是完全可插拔：加新功能（如密钥泄露检测、成本告警、自动测试）只需 hooks.register(...)，不动核心循环。Claude Code 的 Hook 系统就是这种思路——任何脚本都能作为 Hook，让 Harness 成为可编程平台。

14 综合实战：一个带 PEV 的生产级 Harness

把权限、上下文装配、Hook、PEV 环组合成一个完整的 Harness。这是本章所有理念的集大成：

production_harness.py —— 集大成的 Harness
"""
production_harness.py —— 生产级 Harness：权限 + 上下文 + Hook + PEV + 子任务
这是本章所有理念的集成。约 120 行，可直接跑。
"""
import json, subprocess
from openai import OpenAI

client = OpenAI(base_url="https://open.bigmodel.cn/api/paas/v4", api_key="YOUR_KEY")
MODEL = "glm-5.2"

# ---------- 复用前面的 PermissionEngine / ContextAssembler / HookManager ----------
# （此处省略导入，假设已实例化为 perm / asm / hooks）

SYSTEM = """你是资深工程师的 AI 助手。遵循 PEV 工作法：
1. 先规划，再执行；
2. 每次改动后立即验证（跑测试/lint）；
3. 验证失败时分析错误，修正后重试；
4. 工具失败必须如实报告，不得臆测；
5. 涉及写操作前说明意图。"""

def run_harness(task, max_steps=20):
    messages = [{"role": "system", "content": SYSTEM}, {"role": "user", "content": task}]
    last_actions = []

    for step in range(max_steps):
        # Hook: 步数检查
        ctx = hooks.run(HookEvent.ON_STEP, {})
        if ctx and ctx.get("force_stop"):
            return ctx["force_stop"]

        try:
            resp = client.chat.completions.create(model=MODEL, messages=messages, tools=TOOLS)
            msg = resp.choices[0].message
            messages.append(msg)

            if not msg.tool_calls:
                return msg.content

            for tc in msg.tool_calls:
                action = f"{tc.function.name}:{json.loads(tc.function.arguments)}"

                # ① 权限检查
                decision = perm.check(action)
                if decision.decision == Decision.DENY:
                    obs = f"[权限拒绝] {decision.reason}"
                elif decision.decision == Decision.ASK:
                    if input(f"确认 {action}? [y/N]: ").lower() != "y":
                        obs = "[用户拒绝]"
                    else:
                        obs = execute_tool(tc)      # E: 执行
                else:
                    # ② Hook: PreToolUse
                    pre = hooks.run(HookEvent.PRE_TOOL, {"tool": tc.function.name, "args": tc.function.arguments})
                    if pre is None:
                        obs = "[Hook 拦截]"
                    else:
                        obs = execute_tool(tc)      # E: 执行
                        # ③ Hook: PostToolUse（V: 验证 + 反馈）
                        post = hooks.run(HookEvent.POST_TOOL, {"tool": tc.function.name, "result": obs})
                        if post and post.get("feedback"):
                            obs += f"\n[反馈] {post['feedback']}"

                messages.append({"role": "tool", "tool_call_id": tc.id, "content": obs})

        except Exception as e:
            hooks.run(HookEvent.ON_ERROR, {"error": str(e)})
            messages.append({"role": "system", "content": f"系统错误：{e}，请调整。"})

    return "达到最大步数"

def execute_tool(tc):
    # 实际执行工具调用，带异常处理与重试
    for attempt in range(3):
        try:
            return REGISTRY[tc.function.name](**json.loads(tc.function.arguments))
        except Exception as e:
            if attempt == 2: return f"[错误] {e}"
    return "[未知错误]"

这 120 行包含了本章讲的所有核心机制：权限三态校验、上下文装配（通过 SYSTEM + messages）、PreToolUse/PostToolUse Hook、PEV 环（执行→验证→反馈→重试）、步数预算、错误恢复。把它和 Claude Code 的架构对照，你会发现骨架惊人地相似——区别只在工程细节的打磨程度。

15 成本工程：让 Harness 经济可行

Harness 跑起来后，最大的运营成本是模型调用费。一个复杂任务可能调几十次模型、消耗几十万 token。成本工程是生产化的必修课：

策略怎么做节省
模型分级简单子任务用小模型，难任务才用大模型50-80%
上下文裁剪精准装配，不塞无关内容30-60%
缓存相同 prompt 缓存结果（prompt caching）重复查询 90%
批量处理多个独立请求合并成一批20-50%
提前终止检测到已完成就停，不浪费步数变长任务显著
本地小模型路由/分类等简单任务用本地模型边际为零
💡 Prompt Caching 是 2026 的省钱利器

很多 Harness 的系统提示、技能说明、规则文件是固定不变的。现代 API（GLM、Claude）支持 prompt caching：把这些固定前缀缓存，后续请求只算「新增部分」的 token，缓存命中部分大幅打折。对于长系统提示的智能体，这能省下大部分成本。

16 测试 Harness：怎么验证它靠谱

Harness 本身是软件，也需要测试。关键测试维度：

🧪

任务回归集

建一批标准任务，每次改 Harness/Skill/Prompt 都跑一遍，确保成功率不退化。

🛡️

安全测试

专门测越权、注入、危险命令——确保权限引擎和 Hook 拦得住。

💥

混沌测试

故意让工具失败、网络超时，看 Harness 的错误恢复是否健壮。

📊

成本测试

测每个任务的平均 token/成本，防止某次改动导致成本暴涨。
