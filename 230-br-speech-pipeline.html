语音Pipeline
从VAD到ASR到TTS的全链路实时对话

让用户对着设备说一句话，几秒后听到流利的语音回复——这条看似简单的体验背后，是一条VAD→ASR→LLM→TTS的四段式实时pipeline。每一段都有自己的延迟、流式处理、错误处理挑战，把它们拼成一个首字响应在1秒以内的系统是真正的工程艺术。本文用Whisper做ASR、CosyVoice做TTS，给你一套能跑的完整Python实现。

阅读约 26 分钟 难度 ★★★★ 前置：第 40 章LLM应用

★ 四段式pipeline：为什么是这个顺序

主题要点难度
四段式pipeline：为什么是这个顺序核心概念与实现★★★★
流式chunk处理：把延迟藏在流水线里核心概念与实现★★★★
生产化要点：从能跑到稳定跑核心概念与实现★★★★
VAD 简单实现：能量阈值法核心概念与实现★★★★

语音对话系统的标准架构是 VAD → ASR → LLM → TTS → 播放。VAD（Voice Activity Detection，端点检测）负责判断"用户开始说话了"和"用户说完了"，这是触发后续流程的开关。没有VAD，系统要么需要用户手动按按钮（体验差，不像对话），要么持续把环境噪声送进ASR（浪费算力且增加误识别）。VAD的核心任务是区分语音和静音/噪声，常用算法有基于能量的WebRTC VAD（轻量、延迟低、但抗噪差）、基于神经网络的Silero VAD（更鲁棒、对低信噪比场景友好），生产环境两者经常组合使用——WebRTC做初步过滤，神经网络做精细判定。

ASR（Automatic Speech Recognition，语音识别）把音频转成文本。Whisper是OpenAI开源的语音识别模型，支持99种语言，在噪声环境、口音、专业术语上都表现出色，是目前开源ASR的事实标准。它的流式版本Whisper-streaming能在用户说话的同时持续识别，不用等整句说完，这对降低端到端延迟至关重要。除了Whisper，国内场景还常用阿里FunASR（Paraformer模型，中文识别精度高、支持热词定制）、字节SeamlessM4T（支持语音到语音的直接翻译）。选型主要看目标语言、热词定制需求、以及是否能本地部署（隐私场景必须本地）。

LLM拿到文本后做理解和生成回复，这部分和文本对话完全一样，可以直接复用任何Agent框架——LangGraph、AutoGen、或自己写的orchestrator。关键设计是让LLM输出支持流式（streaming），即逐token返回而非整句返回，因为下游TTS可以提前消费这些token开始合成。TTS（Text-to-Speech）把LLM的文本回复合成成语音，CosyVoice是2024年开源的高质量TTS，支持声音克隆（几秒参考音频即可复刻音色）、情感控制（让合成语音带开心/悲伤/严肃语气）、流式合成（不等整句合成完就能开始播放）。商业方案还有Azure TTS、ElevenLabs，效果更稳但成本高。

这个顺序的关键设计点在于"流式串联"。如果每一段都等上一段完全结束才开始，端到端延迟会累加：VAD停顿检测通常500ms，ASR整句识别1-3秒，LLM生成首token 300-800ms，TTS合成第一段1秒——加起来可能3-5秒才出第一个字，用户体验是灾难。真正可用的系统必须把每一段都做成流式：VAD用更短的静音阈值提前判定句尾、ASR边收音边出partial结果、LLM收到几个token就开始生成、TTS收到LLM的第一个短语就开始合成。理想情况下首字响应能压到800毫秒以内，这就逼近了人类对话的自然节奏。

★ 流式chunk处理：把延迟藏在流水线里

流式pipeline的核心抽象是"chunk"——一个固定大小的数据单元在各个阶段间流动。音频以100-300毫秒的chunk进入VAD，VAD标记每个chunk是否含语音；连续若干个语音chunk累积到一个窗口后送ASR；ASR对每个窗口产出partial transcription（部分识别结果，会随音频增加而修订），等到检测到句尾（VAD判定静音）产出final transcription；这个final文本送进LLM，LLM逐token流式输出；每当LLM输出遇到标点（逗号、句号）或字数达到阈值，把这段文本送给TTS；TTS合成出音频chunk立即送播放器，不等整句合成完。

这套设计的精妙之处在于"延迟重叠"。TTS合成第一段的1秒钟里，LLM正在生成第二段；LLM生成第二段时，ASR可能已经开始处理用户的下一句话。每个阶段的处理时间被后续阶段的并行执行"吃掉"了大部分，用户感知到的只是pipeline的"深度"而非"总长度"。这和CPU指令流水线是同一个思想——通过流水线把串行的多阶段操作变成并行执行，把整体延迟从"各阶段之和"降到"最慢阶段的时间"。一个设计良好的四级语音pipeline，能比朴素的"录完整句→识别→生成→合成"串行流程快3-5倍。

# 完整的实时语音对话 pipeline（VAD → ASR → LLM → TTS → 播放）
import asyncio
import numpy as np
from queue import Queue
import sounddevice as sd
import webrtcvad
import openai
# from cosyvoice.cli.cosyvoice import CosyVoice   # TTS（按需启用）

SAMPLE_RATE = 16000
FRAME_MS = 30                         # WebRTC VAD 标准 30ms 帧
FRAME_SIZE = SAMPLE_RATE * FRAME_MS // 1000

class VoicePipeline:
    def __init__(self):
        self.vad = webrtcvad.Vad(2)    # 0-3，越大越激进（易截断）
        self.client = openai.AsyncOpenAI()
        self.audio_in = Queue()        # 麦克风 → pipeline
        self.tts_out = Queue()         # pipeline → 扬声器
        self.silence_frames = 0
        self.SILENCE_LIMIT = 25        # ~750ms 静音判定句尾

    async def vad_loop(self):
        """阶段1：VAD，把语音帧累积成完整的一句话"""
        buffer = []
        while True:
            frame = self.audio_in.get()
            is_speech = self.vad.is_speech(frame.tobytes(), SAMPLE_RATE)
            if is_speech:
                buffer.append(frame)
                self.silence_frames = 0
            elif buffer:               # 已开始说话，遇静音计数
                self.silence_frames += 1
                if self.silence_frames > self.SILENCE_LIMIT:
                    # 句尾确认，把整句送 ASR
                    asyncio.create_task(self.asr_step(np.concatenate(buffer)))
                    buffer = []

    async def asr_step(self, audio: np.ndarray):
        """阶段2：ASR（用 Whisper），转完送 LLM"""
        resp = await self.client.audio.transcriptions.create(
            model="whisper-1",
            file=("in.wav", audio.tobytes(), "audio/wav"),
            language="zh",
        )
        text = resp.text
        print(f"[ASR] {text}")
        if text.strip():
            asyncio.create_task(self.llm_step(text))

    async def llm_step(self, user_text: str):
        """阶段3：LLM 流式生成，遇到断句即送 TTS"""
        stream = await self.client.chat.completions.create(
            model="gpt-4o-mini",
            messages=[{"role": "user", "content": user_text}],
            stream=True,
        )
        buf = ""
        async for chunk in stream:
            tok = chunk.choices[0].delta.content or ""
            buf += tok
            # 流式切分：遇到标点或字数阈值就送 TTS，不等整句
            if any(p in tok for p in "，。！？,.!?") or len(buf) >= 15:
                asyncio.create_task(self.tts_step(buf))
                buf = ""

    async def tts_step(self, text: str):
        """阶段4：TTS 合成，音频 chunk 立即送扬声器"""
        # audio = self.cosyvoice.inference_sft(text, "播报员")
        # self.tts_out.put(audio.samples)
        self.tts_out.put(text.encode())     # demo 占位

但流式也带来一致性挑战。最常见的是"修订"问题：ASR的partial结果是"今天天气"，用户继续说"怎么样"，partial变成"今天天气怎么样"——如果前面已经把"今天天气"送给LLM开始生成就错了。解决方案要么在LLM侧维护一个可撤销的buffer（partial来了先暂存，final来了再确认提交），要么干脆只在final才送LLM（牺牲一点延迟换正确性，适合对准确性要求高的场景如客服）。另一个挑战是TTS对LLM输出的切分：切太碎合成质量差（TTS需要上下文做韵律和重音），切太大延迟高。常用启发式是"遇到逗号/句号/问号即切，或字数达15字即切"，既保证每个TTS单元有足够上下文，又控制首字延迟。

# 第二个示例：处理 barge-in（用户打断）和 ASR partial 修订
import asyncio

class BargeInController:
    """协调各阶段，支持用户随时打断当前回复"""
    def __init__(self, pipeline):
        self.pipe = pipeline
        self.current_task: asyncio.Task | None = None
        self.cancel = asyncio.Event()

    def on_user_speech_detected(self, audio_chunk):
        """VAD 检测到新语音 → 立即打断正在播放的 TTS"""
        self.cancel.set()                       # 通知所有阶段停止
        if self.current_task and not self.current_task.done():
            self.current_task.cancel()          # 取消正在跑的 LLM/TTS
        # 清空待播放的音频队列，避免继续播旧回复
        while not self.pipe.tts_out.empty():
            self.pipe.tts_out.get_nowait()
        self.cancel.clear()
        # 重新进入 ASR 接收新输入
        self.current_task = asyncio.create_task(
            self.pipe.asr_step(audio_chunk)
        )

    def on_asr_partial(self, partial_text):
        """ASR partial 结果：只更新 UI 显示，不触发 LLM"""
        # 用户的屏幕上实时看到识别中的文字，但还没送 LLM
        update_ui_caption(partial_text)

    def on_asr_final(self, final_text):
        """ASR final 结果：才真正提交给 LLM 生成"""
        self.current_task = asyncio.create_task(
            self.pipe.llm_step(final_text)
        )

# 关键设计：partial 与 final 分离
#   partial  → 只做 UI 反馈（让用户知道"它在听"）
#   final    → 才进入 LLM（避免被修订污染）
# barge-in  → cancel event 传播，立即停止当前所有异步任务

★ 生产化要点：从能跑到稳定跑

上面的pipeline能跑通只是第一步，生产化还有几个关键问题。第一是中断处理——用户在TTS播放过程中又开口说话了（barging-in打断），系统必须立即停止TTS播放、清空待播放队列、重新进入ASR。这要求TTS播放器支持立即停止（不能阻塞），且所有阶段之间有可撤销的信号通道（一个"cancel"事件能传播到整个pipeline）。第二是回声消除——如果用扬声器外放，TTS的输出会被麦克风收回去造成"自言自语"的循环，必须用AEC（Acoustic Echo Cancellation）算法或硬件把扬声器声音从麦克风信号里减掉。WebRTC自带AEC模块，手机平台有系统级AEC，但调参需要音频工程经验。

第三是多轮上下文管理。语音对话和文本不同，用户不会显式说"我们继续上一轮的话题"，但会默认系统记得。所以LLM侧需要维护对话历史，且要根据ASR的final文本（而非partial）更新历史，否则历史会被噪声和修订污染。还要处理"指代消解"——用户说"那个怎么样"，系统要从上下文推断"那个"指代什么。第四是错误兜底——ASR识别置信度低（用户口音重或环境嘈杂）、LLM生成超时、TTS合成失败时，要有明确的降级路径，比如回复"我没听清，请再说一遍"，而不是静默或循环重试让用户干等。

性能层面，这套pipeline在合理的硬件上（一块A10或4090 GPU）能做到首字响应800毫秒到1.2秒，吞吐支持10-20路并发。瓶颈通常在ASR和TTS的GPU占用，LLM反而是最轻的（因为流式token生成算力消耗小）。优化方向是把VAD/ASR/TTS搬到本地模型（Whisper-large-v3本地部署、CosyVoice本地推理），减少网络往返，这能把延迟再降一半。更激进的做法是为ASR和TTS各部署一个模型实例池，避免它们争抢同一块GPU造成排队——GPU推理的排队延迟往往比计算本身还大，分离部署能让每个阶段独立横向扩展。

延迟预算的拆解是性能调优的核心工具。把端到端800毫秒的目标分解到每个阶段：VAD句尾判定300毫秒（可调短静音阈值降到200毫秒）、ASR final输出200毫秒（用更小的Whisper-turbo模型）、LLM首token 200毫秒（用流式+ KV cache预热）、TTS首音频块100毫秒（流式合成）。任何一个阶段超标都会拖垮整体，所以要做"分段SLA监控"——每个阶段记录自己的耗时分布，超标就告警。这种细粒度监控比只看端到端延迟更能定位瓶颈，也更容易做容量规划：知道ASR是瓶颈，就扩ASR实例数；知道LLM是瓶颈，就考虑换更快的推理引擎如TensorRT-LLM。

语音pipeline是LLM应用里工程复杂度最高的形态之一，但也是体验最"像人"的形态——当用户对着设备自然地说话、几秒内听到流利的回复，那种交互的魔力是任何文本界面都给不了的。这也是为什么智能音箱、车载助手、语音输入法、AI通话助手都在重金投入这条pipeline。掌握VAD的端点判定、ASR的流式修订、LLM的流式生成、TTS的chunk合成这四个核心机制，以及它们之间的并发协调，你就具备了搭建任何实时语音交互系统的工程基础。

★ VAD 简单实现：能量阈值法

VAD（Voice Activity Detection）是语音 pipeline 的第一道关卡，决定"什么时候用户说完了"。理解它最好从最朴素的能量阈值法开始。下面是一个可运行的最小实现：

import numpy as np

class EnergyVAD:
    """最简单的语音端点检测(VAD)：基于短时能量 + 静音帧计数。
    生产级会用 WebRTC VAD 或 Silero VAD(神经网络)，但能量法是理解原理的最佳起点。
    """
    def __init__(self, frame_ms: int = 30, sr: int = 16000,
                 energy_thresh: float = 0.02, silence_frames: int = 25):
        self.frame_size = int(sr * frame_ms / 1000)   # 30ms 一帧
        self.sr = sr
        self.energy_thresh = energy_thresh             # 能量阈值(归一化后)
        self.silence_limit = silence_frames            # 连续多少静音帧判句尾

    def _frame_energy(self, frame: np.ndarray) -> float:
        # 短时能量 = RMS，归一化到 [0,1]
        rms = np.sqrt(np.mean(frame.astype(np.float32) ** 2))
        return float(rms)

    def detect(self, audio: np.ndarray) -> list[tuple[int, int]]:
        """返回 [(start_sample, end_sample), ...] 每个语音段。"""
        segments, start = [], None
        silence_count = 0
        n = len(audio)
        for i in range(0, n, self.frame_size):
            frame = audio[i:i + self.frame_size]
            if len(frame) < self.frame_size:
                break
            e = self._frame_energy(frame)
            if e > self.energy_thresh:
                if start is None:
                    start = i              # 语音段开始
                silence_count = 0
            else:
                if start is not None:
                    silence_count += 1
                    if silence_count >= self.silence_limit:
                        segments.append((start, i))   # 静音足够长，判句尾
                        start = None
        if start is not None:
            segments.append((start, n))
        return segments

# 造一段模拟音频：前1s静音 + 2s语音 + 1s静音
rng = np.random.default_rng(0)
sr = 16000
silence = (rng.standard_normal(sr) * 0.001).astype(np.float32)
speech = (rng.standard_normal(2 * sr) * 0.1).astype(np.float32)
audio = np.concatenate([silence, speech, silence])

vad = EnergyVAD(frame_ms=30, sr=sr, energy_thresh=0.02, silence_frames=25)
for s, e in vad.detect(audio):
    print(f"语音段: {s/sr:.2f}s ~ {e/sr:.2f}s  (时长 {(e-s)/sr:.2f}s)")
# 应输出约 1.0s ~ 3.0s 的语音段
# 工程升级路径：能量法 → WebRTC VAD(GMM) → Silero VAD(轻量神经网络)
# Silero VAD 在噪声环境下 F1 可达 0.95+，是当前语音 pipeline 的事实标准
