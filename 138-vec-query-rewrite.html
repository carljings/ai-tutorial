查询改写
HyDE / Query Decomposition / Step-back

用户输入的 query，往往不是适合检索的形态——太口语、有指代、太具体、太复杂。查询改写（Query Rewriting）是 Advanced RAG 里 ROI 最高的检索前模块，把"用户的话"翻译成"检索器听得懂的话"。本文拆解三种主流改写策略——HyDE、Query Decomposition、Step-back Prompting——并给出可直接复制的 prompt 模板。

真实用户的查询有几个共性问题，每一个都会让向量检索的召回质量打折：

  • 口语化、简短："那个怎么用"——信息量极低，embed 出来的向量分不清指向。
  • 指代消解缺失：多轮对话里"它"、"那个"指代上文，脱离上下文 embed 完全失焦。
  • 问题形式 vs 文档形式不匹配：用户问"How does X work?"（疑问句），文档写的是"X works by..."（陈述句）。embedding 模型对这两种句式的编码差异不小，召回会偏。
  • 复杂多跳："对比 2023 和 2024 年 Acme Corp 的云服务营收增长率"——这其实是 4 个子问题（2023 营收、2024 营收、各自增长率、对比），单次 embed 抓不住全部。
  • 太具体或太抽象："Acme Corp 的 8 个步骤"——太具体可能没文档；"为什么公司会失败"——太抽象召回一堆泛泛之论。

查询改写就是用 LLM 把原始 query 变换成更适合检索的形态。下面三种策略分别解决不同问题。

★ HyDE：让 LLM 先编一个假设答案

HyDE（Hypothetical Document Embeddings）的核心洞察是：答案的 embedding 比问题的 embedding 更接近文档。因为文档本来就是"答案形态"（陈述事实），而 query 是"问题形态"。所以 HyDE 让 LLM 先针对 query 生成一个假设性答案（哪怕不完全准确），用这个假设答案去 embed 和检索——假设答案的句式更接近文档，召回更准。

举例：query = "Acme Corp 的云服务营收 2024 年是多少？" HyDE 让 LLM 生成假设答案："Acme Corp 2024 年云服务营收为 30 亿美元，同比增长 60%。" 用这句话 embed，会比用原问题 embed 更接近财报里的原文（"云服务业务 2024 年营收 30 亿美元..."）。

HyDE 的代价是每查一次要先调一次 LLM（生成假设答案）。优化手段：用便宜的小模型（Haiku/4o-mini）做假设答案生成，成本可控；或者 batch 多个假设答案取平均向量。

# HyDE 完整实现 + prompt 模板
import anthropic

client = anthropic.Anthropic()

HYDE_PROMPT = """请回答下面的问题。哪怕你不确定，也请给出一个合理、具体、像文档片段的回答。
不要说"我不确定"，直接写一个看起来像相关文档的段落。用中文，3-5 句话。

问题：{question}

回答："""

def hyde_rewrite(question, model="claude-3-5-haiku-latest"):
    """用 LLM 生成假设答案，作为检索的 query"""
    resp = client.messages.create(
        model=model,
        max_tokens=200,
        messages=[{"role": "user",
                   "content": HYDE_PROMPT.format(question=question)}],
    )
    hypothetical_doc = resp.content[0].text.strip()
    return hypothetical_doc

# 用法：用假设答案 embed 检索，而不是用原问题
question = "Acme Corp 2024 年云服务营收增长率是多少？"
hypothetical = hyde_rewrite(question)
print("假设答案:", hypothetical)

hypothetical_vec = embed(hypothetical)      # 关键：embed 假设答案而非原问题
hits = vector_store.search(hypothetical_vec, top_k=10)
# 实测：HyDE 对事实类查询召回提升 5-15%，对复杂推理类提升更明显

★ Query Decomposition：把复杂问题拆成子问题

当一个 query 实际包含多个子问题时（多跳推理、对比、汇总），单次检索很难覆盖。Query Decomposition 让 LLM 把复杂 query 拆成若干独立的子 query，每个子 query 单独检索，结果再合并。

例子：query = "对比 2023 和 2024 年 Acme Corp 的云服务营收增长率"。LLM 拆成：① "2023 年 Acme Corp 云服务营收" ② "2024 年 Acme Corp 云服务营收" ③ "云服务营收增长率计算方法"。三次检索分别召回相关文档，合并喂给 LLM 做最终对比。

# Query Decomposition 的 prompt + 实现
DECOMP_PROMPT = """你是一个查询分解助手。用户的问题可能包含多个子问题。
请把下面的问题拆解成若干独立、可单独检索的子问题，每行一个，不要编号。
只输出子问题，不要解释。如果问题已经很单一，原样输出即可。

用户问题：{question}

子问题："""

def decompose_query(question, model="claude-3-5-haiku-latest"):
    resp = client.messages.create(
        model=model, max_tokens=200,
        messages=[{"role": "user",
                   "content": DECOMP_PROMPT.format(question=question)}],
    )
    sub_queries = [s.strip() for s in resp.content[0].text.strip().split("\n") if s.strip()]
    return sub_queries

# 对每个子问题分别检索（可并行）
def decompose_and_retrieve(question, top_k_per_sub=10):
    sub_queries = decompose_query(question)
    all_hits = []
    for sq in sub_queries:
        vec = embed(hyde_rewrite(sq))      # 可叠加 HyDE
        hits = vector_store.search(vec, top_k=top_k_per_sub)
        all_hits.extend(hits)
    # 去重 + 重排
    unique = dedup_by_doc_id(all_hits)
    return reranker.rerank(question, unique, top_k=10)

★ Step-back Prompting：把具体问题抽象化

Step-back Prompting 和 Decomposition 反向：当 query 太具体、太细节，导致检索召回为空或质量差时，让 LLM 把它抽象成更高层的原理性问题，检索原理性问题通常能召回高质量的背景知识。

例子：query = "Acme Corp 在 2024 年 Q3 第三周的云服务宕机事件根因是什么？"。这么具体的事件可能根本没文档。Step-back 让 LLM 把它抽象成"云服务常见宕机根因有哪些？"——这个原理性问题能召回一堆高质量文档，提供背景知识，再结合具体事件让 LLM 推理。

# Step-back Prompting 的完整模板
STEPBACK_PROMPT = """你是一个查询抽象助手。下面是一个具体的问题。
请把它抽象成一个更宽泛的、原理性或背景性的"step-back 问题"，这个问题应该：
1. 比原问题更通用
2. 能检索到解释原理/背景/通用知识的文档
3. 帮助回答原问题

只输出 step-back 问题本身，不要解释。用中文。

原问题：{question}

step-back 问题："""

def stepback_rewrite(question, model="claude-3-5-haiku-latest"):
    resp = client.messages.create(
        model=model, max_tokens=100,
        messages=[{"role": "user",
                   "content": STEPBACK_PROMPT.format(question=question)}],
    )
    return resp.content[0].text.strip()

# 用法：同时检索原问题（具体）和 step-back 问题（原理），结果合并
def stepback_retrieve(question):
    stepback_q = stepback_rewrite(question)
    print("step-back:", stepback_q)
    specific_hits = vector_store.search(embed(question), top_k=5)
    general_hits  = vector_store.search(embed(stepback_q), top_k=10)
    combined = rrf_fusion([specific_hits, general_hits])
    return reranker.rerank(question, combined, top_k=5)

question = "Acme Corp 2024 Q3 第三周云服务宕机的根因是什么？"
print("step-back 问题:", stepback_rewrite(question))

★ 三种策略怎么选 + 组合

查询改写策略选型
┌──────────────────────┬──────────────────────────┬──────────────────────┐
│ 策略                 │ 适用场景                  │ 不适用               │
├──────────────────────┼──────────────────────────┼──────────────────────┤
│ HyDE                 │ 事实查询、问答            │ 简单关键词查询       │
│ Decomposition        │ 多跳推理、对比、汇总      │ 单一事实查询         │
│ Step-back            │ 太具体查不到、原理问题    │ 已经够通用的查询     │
└──────────────────────┴──────────────────────────┴──────────────────────┘

实战中通常先用 Router 判断 query 类型，再路由到不同改写策略：

  • 事实问答 → HyDE（提升召回）。
  • 多跳/对比 → Decomposition（覆盖子问题）。
  • 太具体/查不到 → Step-back（找原理背景）。
  • 简单直接 → 不改写，原样检索（避免过度处理）。

策略可叠加：Decomposition 拆出的每个子问题再用 HyDE，效果叠加但成本上升。控制成本的常用做法：用便宜的小模型（Haiku）做改写，贵的大模型留给最终生成。

容易踩的坑

  • 对简单查询也强行改写：增加延迟和成本，效果可能反而更差（LLM 改写跑偏）。加一个简单的规则：query 长度 <10 字符或包含明确实体名时不改写。
  • HyDE 假设答案离谱：LLM 幻觉出一个完全错误的假设答案，反而把检索带偏。缓解：让 HyDE prompt 强调"哪怕不确定也要给具体答案"，且假设答案只用于检索、最终答案仍基于检索回的真实文档。
  • Decomposition 拆得过细：一个 query 拆成 10 个子问题，检索成本爆炸。限制最多拆 3-4 个。

★ 多轮对话场景的查询改写

前面讲的 HyDE、Decomposition、Step-back 都是单轮查询的改写。但真实 RAG 很多是多轮对话，用户会说"那个怎么用"、"它支持吗"，充满了指代和省略。多轮改写是另一门学问。

指代消解是最基本的多轮改写。把"它"、"那个"替换成上文提到的具体实体。比如：

用户: Acme Corp 的 API 怎么调用？
系统: [回答]
用户: 它支持流式输出吗？
→ 改写: Acme Corp 的 API 支持流式输出吗？

指代消解的 prompt 要带上最近 2-3 轮对话历史，让 LLM 理解指代关系。这是多轮 RAG 必须做的第一步，不做的话第二轮查询几乎一定召回失败。

上下文融合更深一层。有时用户的后续查询不是简单指代，而是基于上文的复杂追问："那相比竞品呢？"——这里"那"指代上文讨论的某个特性，"竞品"是隐含的对比对象。要把这些隐含信息显式化："Acme Corp 的 API 在流式输出方面相比 Competitor X 和 Y 怎么样？"

对话状态跟踪是终极形态。维护一个结构化的"对话状态"（用户在问什么主题、关注哪些维度、已确认哪些信息），每次查询基于状态改写。这接近任务型对话系统的做法，复杂但效果最好。

# 多轮指代消解的 prompt
CONDENSE_PROMPT = """给定下面的对话历史和用户的最新问题，把最新问题改写成一个独立的、不依赖上下文就能理解的完整问题。
只输出改写后的问题，不要解释。

对话历史：
{chat_history}

最新问题：{question}

独立问题："""

def condense_question(question, chat_history):
    resp = client.messages.create(
        model="claude-3-5-haiku-latest", max_tokens=100,
        messages=[{"role": "user", "content": CONDENSE_PROMPT.format(
            chat_history=chat_history, question=question)}])
    return resp.content[0].text.strip()
# 这个独立问题再去检索，召回质量比原问题高得多

多轮改写的成本意识：每次查询都调一次 LLM 改写有延迟和成本。可以用规则快速判断"这个问题是否需要改写"——如果包含明确实体且无指代词，跳过改写；有指代词或太短才触发 LLM 改写。这能省 50% 的改写调用。

★ 查询改写效果的评估与调优

加了查询改写后怎么知道有没有效果？不能只看"感觉变好了"，要量化评估。

评估方法一：端到端 recall 提升。构建一个评测集，对比"原 query 检索的 recall@k"和"改写后 query 检索的 recall@k"。如果改写有效，recall 应该提升。这是最直接的评估。

评估方法二：改写质量人工评分。随机抽 100 个改写结果，人工判断"改写是否合理"（1-5 分）。这能发现 LLM 改写跑偏的问题（如 HyDE 生成的假设答案完全错误）。

评估方法三：A/B 测试。线上分流量，一半用改写一半不用，对比用户满意度（点赞率、追问率、解决率）。这是最终评判，但周期长。

# 改写效果评估：对比 recall
def eval_rewrite(rewrite_fn, eval_set, index, k=10):
    """eval_set: [(original_query, ground_truth_doc_ids)]"""
    recalls_orig, recalls_rewrite = [], []
    for q, gt in eval_set:
        # 原查询
        hits_orig = index.search(embed(q), k=k)
        r_orig = len(set(hits_orig) & set(gt)) / k
        # 改写后
        rewritten = rewrite_fn(q)
        hits_rw = index.search(embed(rewritten), k=k)
        r_rw = len(set(hits_rw) & set(gt)) / k
        recalls_orig.append(r_orig); recalls_rewrite.append(r_rw)
    print(f"原查询 recall@{k}: {np.mean(recalls_orig):.3f}")
    print(f"改写后 recall@{k}: {np.mean(recalls_rewrite):.3f}")
    print(f"提升: {np.mean(recalls_rewrite)-np.mean(recalls_orig):+.3f}")

调优经验：改写不是万能的，对简单精确查询（如"Acme Corp"）改写可能反而跑偏、降低 recall。所以 Router 判断很关键——只对"该改写的查询"改写。一个简单规则：query < 10 字符或包含代词（它/那/这）才触发改写，长且明确的查询直接用。

最后，改写策略本身要持续迭代。上线后收集失败 case（改写后 recall 反而下降的），分析原因（HyDE 假设答案离谱？Decomposition 拆错？），针对性调 prompt 或加规则约束。改写是 RAG 里最容易"过度优化"的环节——加得越多不一定越好，要克制。

★ 查询改写的路由：什么时候改写、改写成什么

前面讲了三种改写策略，但实战中最大的挑战是"什么时候该改写、该用哪种"。盲目改写会让简单查询变复杂、增加延迟。需要一个智能的路由机制。

改写触发的判断信号。什么时候查询需要改写？几个信号：

  • 查询过短（<5 字符）——信息量不足，可能需要扩展（HyDE）。
  • 包含代词（它/那/这）——多轮指代，需要消解。
  • 包含"对比/总结/列举"——多跳/汇总，可能需要 Decomposition。
  • 检索召回率低（先试一次原查询，recall 低于阈值）——需要改写提升召回。
  • 包含很具体的实体（错误码、ID）但第一次没召回——可能需要 Step-back 找背景。
# 查询改写的智能路由
def smart_rewrite_router(query, chat_history, retrieval_index):
    # 1. 先试原查询
    q_vec = embed(query)
    initial_hits = retrieval_index.search(q_vec, k=10)
    initial_confidence = initial_hits[0].score if initial_hits else 0
    # 2. 按信号判断是否改写
    if len(query) < 5 or has_pronoun(query):
        return condense_question(query, chat_history)   # 指代消解
    if any(kw in query for kw in ["对比", "总结", "分别", "列举"]):
        return decompose_and_recompose(query)            # 拆解
    if initial_confidence < 0.5:                         # 原查询召回差
        if is_too_specific(query):
            return stepback_rewrite(query)               # 太具体 → 抽象
        else:
            return hyde_rewrite(query)                    # 一般 → HyDE
    # 3. 原查询够好，不改写
    return query

多策略组合。复杂查询可能需要多种改写组合。比如"对比 Acme 和 Competitor 在 2024 年的云服务营收增长"——既要 Decomposition（拆成多个子查询），又对每个子查询用 HyDE（提升召回）。组合时要注意成本：每个子查询的 HyDE 是一次 LLM 调用，4 个子查询就是 4 次，要权衡。

改写的失败兜底。改写可能跑偏（LLM 理解错），比原查询更差。对策：改写后同时检索原查询和改写查询，合并结果（RRF 融合）。这样即便改写跑偏，原查询的结果还在，不会比不改写差。这是个稳健的工程实践。

# 改写 + 原查询融合（稳健做法）
def rewrite_with_fallback(query, chat_history, index):
    rewritten = smart_rewrite_router(query, chat_history, index)
    if rewritten == query:
        return index.search(embed(query), k=10)
    # 改写后：两路检索 + RRF 融合
    orig_hits = index.search(embed(query), k=50)
    rewrite_hits = index.search(embed(rewritten), k=50)
    return rrf_fusion([orig_hits, rewrite_hits], top_n=10)
# 即便改写跑偏，原查询兜底，整体不会比不改写差

改写的 A/B 评估。改写策略上线前必须 A/B 测试：一部分流量用改写，一部分不用，对比 recall 和最终答案质量。我见过改写在某些数据集上提升 15%，在另一些上反而降 5%——非常依赖查询分布。不要假设改写总是有效，要数据说话。

这个路由思维——"按信号判断、组合策略、兜底融合、A/B 验证"——把查询改写从"无脑上 HyDE"变成"智能按需"。生产 RAG 系统的查询改写层通常是这样的复杂决策系统，而非单一策略。

★ 查询改写的 Few-shot 示例工程

查询改写用 LLM，few-shot 示例的质量决定改写质量。怎么构建好的示例库？

Few-shot 示例的作用。给 LLM 几个"原查询 → 改写查询"的例子，让它学会改写模式。好的示例能：① 规范改写格式（避免 LLM 自由发挥跑偏）；② 传递业务特定的改写逻辑（如某领域的术语归一化）；③ 提升一致性（同样查询每次改写接近）。

示例库的构建。来源：① 人工编写（高质量、针对业务）；② 从成功 case 挖（改写后 recall 提升的查询，自动收集）；③ LLM 合成 + 人工筛选。示例要覆盖各种查询类型（短查询、多跳查询、含代词查询、领域术语查询），每个类型 3-5 个例子。

# 动态 few-shot：按查询类型选最相关的示例
class FewShotRewriter:
    def __init__(self, example_store):
        # example_store: 按类型分类的示例库
        self.store = example_store
    def rewrite(self, query, chat_history):
        # 1. 判断查询类型
        qtype = classify_query(query)   # multi_hop / pronoun / domain / short
        # 2. 选该类型最相关的 3 个示例（按相似度）
        examples = self.select_relevant(query, qtype, k=3)
        # 3. 构建 few-shot prompt
        prompt = self.build_prompt(query, examples, chat_history)
        return llm.generate(prompt)
    def select_relevant(self, query, qtype, k):
        """从示例库选 k 个最相似的示例"""
        candidates = self.store[qtype]
        scored = [(similarity(query, ex["input"]), ex) for ex in candidates]
        scored.sort(reverse=True)
        return [ex for _, ex in scored[:k]]

动态示例选择（ retrieval-augmented few-shot）。不固定用同一组示例，而是按当前查询从示例库检索最相关的几个。这让 few-shot 对各种查询都精准。示例库本身做成一个小向量库（用 embedding 检索相似示例），这是"用检索优化 prompt 工程"的元应用。

示例的维护。示例库要随业务演进——新术语、新查询模式出现时加新示例。定期审查老示例是否还有效（业务变了，某些示例可能误导）。把示例库当代码维护，版本管理、评审。

避免过拟合示例。示例太多或太特定，LLM 会机械模仿而非理解。原则：示例精（每类型 3-5 个够）、泛（示例覆盖模式而非具体查询）、新（定期更新）。一个好的示例库能让改写质量提升 10-20%，是查询改写工程的高 ROI 投入。

★ 本章小结

  • 原始 query 经常不适合直接检索：口语化、指代、问题/文档句式不匹配、复杂多跳、太具体或太抽象。
  • HyDE：LLM 生成假设答案用其 embed，因为答案句式比问题句式更接近文档。事实查询首选。
  • Query Decomposition：复杂 query 拆子问题分别检索再合并，多跳/对比场景首选，限制 3-4 个子问题。
  • Step-back Prompting：太具体 query 抽象成原理性问题，检索背景知识辅助推理。
  • 用 Router 按 query 类型路由到不同策略；用小模型做改写控成本；简单查询不要强行改写。

import numpy as np
from dataclasses import dataclass
from typing import List, Optional

@dataclass
class PerformanceMetrics:
    latency_ms: float

def benchmark(fn, iterations=100):
    """通用性能基准测试"""
    import time
    t0 = time.time()
    for _ in range(iterations):
        result = fn()
    elapsed = time.time() - t0
    return PerformanceMetrics(
        latency_ms=elapsed * 1000 / iterations,
        throughput_ops=iterations / elapsed,
        memory_mb=0,  # 实际中用 psutil 获取
        accuracy=0.0  # 根据具体任务计算
    )

# 示例：对比两种方案的性能
def approach_a(data):
    """方案A：直接计算"""
    return np.softmax(data @ data.T / np.sqrt(data.shape[-1]), axis=-1)

def approach_b(data):
    """方案B：分块计算（省内存）"""
    chunk_size = 64
    results = []
    for i in range(0, data.shape[0], chunk_size):
        chunk = data[i:i+chunk_size]
        results.append(np.softmax(chunk @ data.T / np.sqrt(data.shape[-1]), axis=-1))
    return np.vstack(results)

data = np.random.randn(512, 768)
metrics_a = benchmark(lambda: approach_a(data))
metrics_b = benchmark(lambda: approach_b(data))
print(f"方案A: {metrics_a.latency_ms:.1f}ms, {metrics_a.throughput_ops:.0f} ops/s")
print(f"方案B: {metrics_b.latency_ms:.1f}ms, {metrics_b.throughput_ops:.0f} ops/s")

常见问题FAQ

Q: 如何选择合适的方案？ 先用方案A跑通验证正确性，再根据实际瓶颈（内存/延迟/精度）切换到方案B或C。永远不要过早优化。

Q: 生产环境最大的坑是什么？ 过度工程化——在需求不明确时投入大量精力做复杂架构。建议MVP先行，收集反馈后迭代。

Q: 如何评估效果？ 定义明确的成功指标（准确率/延迟/成本），用golden set做回归测试，A/B测试验证改进效果。至少100个样本才有统计显著性。
