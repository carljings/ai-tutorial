Proxima 是什么：达摩院自主研发的向量检索引擎

zvec 的算法内核不是从零写的，它的底层是 Proxima（github.com/alibaba/proxima）——阿里巴巴达摩院（DAMO Academy）自主研发、在阿里集团内部使用了多年的向量检索引擎。Proxima 不是一个"学术原型"，它在阿里最核心的几个业务里跑了很久：淘宝主搜索的商品召回、蚂蚁的人脸支付特征比对、优酷的视频推荐、菜鸟的图像检索……据阿里云官方技术博客《Proxima: A Vector Retrieval Engine independently developed by Alibaba DAMO Academy》的口径，这种"在真实大规模业务里反复打磨"的引擎，和实验室里跑 benchmark 的项目，工程成熟度完全是两个层级。

zvec 的做法是：把 Proxima 的核心算法（索引构建、量化压缩、距离计算、检索）抽出来，重新封装成一个嵌入式、轻量、面向端侧的库。从 zvec 的源码目录能看到这条血缘：src/core/ 下有 algorithm / framework / metric / quantizer / plugin / mixed_reducer 等子目录——这就是从 Proxima 一脉相承下来的内核分层。所以理解 zvec 的能力边界和性能特性，必须先理解 Proxima 的设计哲学。

💡 要点
类比：Proxima 之于阿里，就像 FAISS 之于 Meta、ScaNN 之于 Google——都是大厂内部沉淀、扛过核心业务的向量检索基础设施。zvec 把 Proxima 的内核以嵌入式形态开放出来。

Proxima 的设计哲学：扁平索引 + 硬件优化

翻开阿里云官方 Proxima 博客，达摩院团队给出的核心设计哲学和很多人想象的"复杂索引结构"不同：他们强调的是"相对扁平的索引结构 + 极致的硬件优化"。换句话说，Proxima 没有去发明花哨的新索引数据结构，而是把工业界验证过的几种（FLAT / IVF / HNSW / 量化）做到极致，然后把主要精力放在"让这些算法在现代硬件上跑得最快"——SIMD 向量化、多线程并发、内存布局对齐、缓存友好。这个判断是有道理的：在大规模检索里，"算法本身的渐进复杂度"通常不是瓶颈，"内存带宽和 CPU 流水线利用"才是。

这条哲学直接解释了 zvec 在 benchmark 上的表现：在官方公布的 VectorDBBench 数据里，zvec 在 1M 和 10M 数据集上都报出了 8000+ QPS 量级的吞吐，这个数字不是靠更巧的算法，而是靠"扁平结构 + SIMD + 内存布局"把每个查询的常数项压到极低。同样的逻辑也解释了为什么 FAISS 在裸算法性能上往往领先——Meta 团队也是走"极致硬件优化"路线的标杆。

设计维度Proxima / zvec 的选择为什么
索引结构扁平为主（FLAT/HNSW/IVF）复杂结构的常数项开销在端侧不划算
距离计算SIMD（AVX2/AVX-512/ARM NEON）内存带宽是瓶颈，SIMD 是杠杆
并发模型多线程 + 无锁友好数据结构现代 CPU 核数多，并发是免费午餐
内存布局按 SIMD lane 对齐、连续存储避免 cache miss 和跨 lane 访问
量化int8 / int4 / float 多档让算法在"精度-内存"间连续可调

从 zvec 源码看内核分层

理论说够了，直接看 zvec 的源码目录（src/core/ 下的实际分层），这比任何二手描述都准：

zvec/src/core/  —— 从 Proxima 一脉相承的内核分层
├── algorithm/      # 索引算法实现：HNSW / IVF / FLAT 等
├── framework/      # 算法调度框架：把算法、量化、距离计算串起来
├── metric/         # 距离度量：L2 / InnerProduct / Cosine
├── quantizer/      # 量化器：float / int8 / int4 等
├── mixed_reducer/  # 混合归约：dense + sparse 向量融合
├── plugin/         # 插件机制：可替换的算法/存储组件
├── interface/      # 对外 C++ 接口
└── utility/        # 工具：内存池、线程池、SIMD 封装

zvec/src/db/       —— 数据库语义层（在 core 之上）
├── collection.cc  # Collection：schema + 索引 + WAL 的统一句柄
├── index/         # 索引管理：构建/合并/查询调度
├── reranker/      # ★ refiner：两级检索的精排层
├── sqlengine/     # 标量过滤 + SQL 语义执行
├── common/        # WAL、checkpoint、文件管理
└── proto/         # 序列化协议（持久化用）

两个值得专门拎出来的目录：① src/core/mixed_reducer/——这是 zvec 支持"dense + sparse 混合检索"的地方（把稠密向量召回和稀疏关键词召回的结果合并），是它区别于纯向量库的关键能力；② src/db/reranker/——这是 zvec 的 refiner（精排器），对应官方 benchmark 里反复出现的 --is-using-refiner 参数，实现"粗筛 + 精排"两级检索。这两个目录的存在，本身就证明了 zvec 不是"只支持单级 HNSW"的玩具，而是把 Proxima 的多级检索能力完整搬过来了。

WAL 与持久化：嵌入式库的"崩溃恢复"

嵌入式库最大的工程问题之一是"崩溃了怎么办"。你往 zvec 插了 10 万条向量，进程崩了——下次打开，这 10 万条还在吗？zvec 的答案是 WAL（Write-Ahead Log，预写日志）+ 周期性 checkpoint，这套机制和 SQLite 的 journal、PostgreSQL 的 WAL 是同一类思路：

WAL + Checkpoint 工作流

写入路径：
  col.insert(records)
       │
       ▼
  [1] 追加写到 WAL 日志（顺序写，极快）
       │   —— 先落盘 WAL，再改内存索引
       │   —— 保证"日志先于数据"（write-ahead）
       ▼
  [2] 更新内存索引（HNSW 图 / 倒排）
       │   —— 此时查询已能看到新数据
       ▼
  返回成功

后台周期性 checkpoint：
  [3] 把当前内存索引快照完整写到一个新数据文件
  [4] 截断旧的 WAL（已 checkpoint 的部分不再需要）

崩溃恢复（下次 zvec.open）：
  [5] 加载最近的 checkpoint 数据文件
  [6] replay WAL 里 checkpoint 之后的部分
       —— 把还没落盘的写入重做一遍
  [7] 索引恢复到崩溃前的一致状态

这个设计的精妙之处：正常路径（写入）只需要一次顺序的 WAL 追加写（磁盘顺序写几乎和内存写一样快），不需要每次插入都把整个索引 flush（那是灾难级的写放大）。真正昂贵的"全量索引序列化"由后台 checkpoint 异步做，不阻塞前台写入和查询。崩溃恢复时，因为 WAL 是顺序的、确定性的，replay 是幂等的——恢复后索引状态严格一致。这就是为什么嵌入式库也能做到"断电不丢数据"。

💡 要点
WAL 是嵌入式向量库持久化的基石：写路径只追加顺序日志（快），全量落盘由后台 checkpoint 异步做（不阻塞），崩溃靠 replay WAL 恢复（一致）。这套机制让 zvec 既有"内存级写入速度"，又有"数据库级持久性"——和 SQLite 用同一套思路。

与 FAISS 的关系：引擎 vs 库

很多人会问：Proxima 和 FAISS 是什么关系？它们不是竞争关系，而是"层级不同"：

FAISS 是一个算法库——它提供 HNSW/IVF/PQ 这些算法的高效实现，但不管"数据怎么组织、怎么写入、怎么持久化、怎么并发"。你要自己把 FAISS 包成一个"服务"或"数据库"。

Proxima 是一个引擎——它在算法（部分算法实现借鉴 FAISS 的思路）之上，还管并发调度、SIMD 优化、量化、混合检索、持久化。换句话说，Proxima 是"用 FAISS 那类算法 + 一套数据库级的工程外壳"。zvec 则是把这个引擎再封装一层，变成嵌入式 API，并加上 schema、SQL 过滤、WAL 这些"数据库语义"。

层级代表提供什么不管什么
算法库FAISS / ScaNN / hnswlib索引算法实现写入/持久化/并发/服务化
引擎Proxima算法 + 并发/SIMD/量化/混合检索API 形态/部署模式/schema
嵌入式库zvec引擎 + schema + WAL + SQL 过滤 + 进程内 API多客户端/分布式
服务化数据库Milvus / Qdrant引擎 + 服务 + 分布式 + 多租户（更重）
💡 要点
层级关系：算法库（FAISS）⊂ 引擎（Proxima）⊂ 嵌入式库（zvec）⊂ 服务化数据库（Milvus）。zvec 站在 Proxima 之上，Proxima 站在 FAISS 那类算法之上。理解这个"叠加"，就能解释为什么 zvec 体积小但能力强——它复用了达摩院沉淀多年的引擎内核。

本章小结

zvec 的算法内核来自达摩院的 Proxima 引擎——一个在淘宝搜索、人脸支付、视频推荐里扛了多年的基础设施。Proxima 的核心设计哲学是"扁平索引 + 极致硬件优化"（SIMD/多线程/内存布局），而不是发明花哨的新结构——这解释了它在 benchmark 上的高吞吐。zvec 源码 src/core 下的分层（algorithm/framework/metric/quantizer/mixed_reducer）直接继承自 Proxima，src/db 下的 reranker（refiner）和 sqlengine 则证明它带过来了多级检索和过滤能力。持久化靠 WAL + checkpoint，崩溃靠 replay 恢复，和 SQLite 同源。与 FAISS 相比，Proxima 多了"数据库级工程"那一层；与 Milvus 相比，zvec 少了"服务化和分布式"那一层。下一章我们看 zvec 暴露给用户的索引类型和参数。
