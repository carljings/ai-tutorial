Agent可观测性实战
追踪、成本、告警与LangSmith集成

一个 Agent 系统在跑起来之后，最让工程师头疼的不是"它会不会出错"，而是"它为什么出错"。一次失败可能涉及提示词、工具调用、检索召回、模型回退——任何一环都可能是元凶。可观测性（Observability）就是给黑盒系统装上仪表盘和行车记录仪：每一次 LLM 调用都被追踪，每一分 token 成本都被记账，每一次异常都会触发告警。本章带你搭建一套覆盖追踪、成本、告警、调试的可观测性体系。

阅读约 32 分钟 难度 ★★★★

★ 为什么 Agent 比普通服务更需要可观测性

传统微服务的可观测性聚焦于"请求耗时、错误率、资源占用"这三板斧，因为服务的逻辑是确定的——同样的输入几乎一定走同样的代码路径。但 Agent 完全不同：它是模型驱动的，每一次执行可能走完全不同的工具调用序列，模型输出本身具有随机性，错误既可能是代码异常，也可能是模型生成了不合规的内容或陷入了无限循环。这种"非确定性 + 多步组合"让传统监控手段力不从心。

举个具体场景：一个客服 Agent 在某次对话中给用户报了错误订单号。要定位根因，你需要回看完整的调用链——它先调用了 search_user，再调了 get_order，最后调了 reply。是 search_user 返回了错误的用户？是 get_order 取错了订单？还是模型在 reply 里编造了号？没有完整的 trace，你只能靠日志一行行翻，而生产环境的日志量让你绝望。

Agent 可观测性的三大支柱因此被重新定义：第一是调用追踪（Tracing），记录每次 LLM 调用、工具调用、子 Agent 调用的输入输出和耗时，组织成树状的执行轨迹；第二是成本监控（Cost），按用户、按租户、按场景统计 token 消耗和折算金额；第三是质量监控（Quality），追踪幻觉率、拒答率、人工反馈分布等业务指标。下面我们逐个展开。

★ LLM 调用追踪与 OpenTelemetry

追踪的基础协议是 OpenTelemetry（OTel），它是 CNCF 旗下的可观测性标准，已成为云原生监控的事实规范。OTel 的核心概念是 Span：一个有起止时间、带属性（attributes）和事件（events）的工作单元。多个 Span 通过 parent-child 关系组成 Trace。把这套模型套用到 Agent 上：一次用户对话是一个 Trace，里面的每次 LLM 调用、工具调用、检索调用都是子 Span，层层嵌套形成树。

OTel 为生成式 AI 定义了一套语义约定（Semantic Conventions for GenAI），规定了应该记录哪些标准字段：gen_ai.system（模型供应商）、gen_ai.request.model（模型名）、gen_ai.usage.prompt_tokens（输入 token）、gen_ai.usage.completion_tokens（输出 token）、gen_ai.response.finish_reason（结束原因）等。遵循这套约定，意味着你的 trace 可以被任何兼容 OTel 的后端（Jaeger、Tempo、Datadog、Honeycomb）正确解析。下面这段代码演示如何用 OpenLLMetry 库自动给 LLM 调用打 span。

# pip install opentelemetry-sdk opentelemetry-exporter-otlp openllmetry
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
from openllmetry import auto_instrument  # 自动 patch openai/anthropic/llama-index 等

# 1) 初始化 Tracer，把 span 推到 OTLP 收集器（如 Jaeger / Tempo）
provider = TracerProvider()
provider.add_span_processor(
    BatchSpanProcessor(
        OTLPSpanExporter(endpoint="http://otel-collector:4317", insecure=True)
    )
)
trace.set_tracer_provider(provider)

# 2) 自动给主流 LLM SDK 装上 instrumentation
auto_instrument()

# 3) 业务代码无需改动，每次调用都会自动生成符合 GenAI 语义约定的 span
import openai
resp = openai.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "用一句话解释什么是 Agent"}],
)
print(resp.choices[0].message.content)
# 在 Jaeger 里你会看到一条 trace，包含 model、token usage、latency 等字段

OTel 的强大之处在于它跨语言、跨后端、跨框架通用。你用 Python 写 Agent、用 Go 写中间件、用 Node 写前端 BFF，所有 span 都能汇到同一条 trace 里。这种"全链路追踪"在排查"为什么这一笔请求慢了 3 秒"时极其关键——可能是 Python 端的 LLM 调用慢，也可能是 Go 端的鉴权慢，没有统一 trace 你永远定位不到。

★ Token 成本监控：从计费到预算控制

Token 就是钱。一次 GPT-4o 调用，1000 输入 token 大约 0.0025 美元，1000 输出 token 约 0.01 美元。一个日活 10 万的产品，每个用户平均 5 轮对话、每轮 2000 token，一天的 LLM 成本可能就是数千美元。没有精细的成本监控，你的产品可能"做得很火但越做越亏"。成本监控的目标有三个：知道钱花在哪、知道谁在花钱、能在超预算时及时止损。

实现上分三层。第一层是调用级记账：每次 LLM 调用从 response 里取出 usage.prompt_tokens 和 usage.completion_tokens，按当前模型价目表折算金额，写入 trace 的 span 属性里。第二层是聚合统计：把调用级数据按用户、租户、场景、模型维度聚合，每小时/每天滚动，存到时序数据库（如 Prometheus、ClickHouse）。第三层是预算控制：给每个租户或用户设置月度预算，剩余额度低于阈值时降级（改用便宜模型）、为负时拒绝服务。下面是一段实现骨架。

from dataclasses import dataclass
from datetime import datetime
from opentelemetry import trace

# 模型价目表：美元 / 1K token，生产环境应从配置中心动态加载
PRICE_TABLE = {
    "gpt-4o":        {"input": 0.0025, "output": 0.010},
    "gpt-4o-mini":   {"input": 0.00015, "output": 0.0006},
    "claude-3.5-sonnet": {"input": 0.003, "output": 0.015},
}

@dataclass
class CostRecord:
    user_id: str
    model: str
    input_tokens: int
    output_tokens: int
    cost_usd: float
    ts: datetime

async def chat_with_cost(user_id: str, messages, model="gpt-4o-mini"):
    # 1) 预算预检：剩余额度低于阈值则降级到更便宜的模型
    remaining = await budget_repo.remaining(user_id)
    if remaining < 0.5:   # 低于 0.5 美元
        model = "gpt-4o-mini"   # 降级

    # 2) 调用并记账
    resp = await llm_client.chat(model=model, messages=messages)
    price = PRICE_TABLE[model]
    cost = (resp.usage.prompt_tokens * price["input"]
            + resp.usage.completion_tokens * price["output"]) / 1000

    await cost_repo.append(CostRecord(
        user_id=user_id, model=model,
        input_tokens=resp.usage.prompt_tokens,
        output_tokens=resp.usage.completion_tokens,
        cost_usd=cost, ts=datetime.utcnow(),
    ))
    # 3) 把 cost 写到当前 OTel span 上，便于在 trace 里直接看
    span = trace.get_current_span()
    span.set_attribute("gen_ai.cost.usd", cost)
    return resp

# 定时任务：每小时把 cost_repo 数据推到 Prometheus / Grafana 做大盘

一个实战经验：成本异常往往比错误更隐蔽。某个用户的 prompt 模板里不小心塞了 5 万 token 的文档，单次调用成本就可能翻 50 倍。建议在聚合层加一个"单次调用成本突增告警"——当某次调用成本超过历史 P99 的 5 倍时立刻告警，能帮你抓住这类隐性浪费。

★ 错误率告警：把被动救火变成主动响应

可观测性的价值不在于"事后能查"，而在于"事前能防"。告警就是把监控指标转化为主动响应的关键环节。Agent 系统需要监控的错误类型比传统服务丰富得多，至少包括四类：调用异常（HTTP 5xx、超时、限流）、模型异常（输出为空、触发内容过滤、finish_reason 为 length/content_filter）、业务异常（工具调用失败、Agent 陷入循环、人审拒绝率升高）、成本异常（单位时间成本突增）。每一类都要配独立的告警规则。

告警设计的核心原则是"信噪比优先"。一条好的告警应该是可操作的——收到的人知道该干什么；反之，每天响 100 次却没人响应的告警，等于没有告警，只会导致告警疲劳。实践中通常用"多级阈值"：P95 延迟超过 2 秒发 Warning 通知到值班群，超过 5 秒发 Critical 并触发 PagerDuty 电话告警。错误率从 1% 涨到 3% 发 Warning，涨到 10% 立刻 Critical。

# Prometheus 告警规则示例（alertmanager.yml 配套使用）
# 保存为 agent_alerts.yml，由 prometheus 加载
groups:
- name: agent_errors
  rules:
  - alert: LLMErrorRateHigh
    # 过去 5 分钟 LLM 调用错误率
    expr: |
      sum(rate(llm_calls_total{status="error"}[5m]))
      / sum(rate(llm_calls_total[5m])) > 0.05
    for: 2m              # 持续 2 分钟才告警，避免抖动
    labels:
      severity: critical
    annotations:
      summary: "LLM 错误率超过 5%"
      description: "模型 {{ $labels.model }} 错误率达 {{ $value }}，请检查供应商状态。"

  - alert: AgentCostSpike
    # 过去 1 小时成本相比昨天同期涨 3 倍
    expr: |
      sum(rate(llm_cost_usd_total[1h]))
      / sum(rate(llm_cost_usd_total[1h] offset 1d)) > 3
    for: 10m
    labels:
      severity: warning
    annotations:
      summary: "LLM 成本异常上涨"

  - alert: AgentLoopDetected
    # 同一个 trace 内工具调用次数超过 15，疑似死循环
    expr: trace_tool_call_count > 15
    for: 0m
    labels:
      severity: warning
    annotations:
      summary: "Agent 可能陷入循环，trace={{ $labels.trace_id }}"

这三条规则覆盖了成本、错误率、循环三个最典型的 Agent 故障模式。部署后建议先用"混沌测试"验证告警是否真的会响——人为注入一次高错误率或死循环，看告警链路是否完整打通。很多团队的告警在出事时才发现"规则没生效"或"通知渠道配错了"，根因都是缺乏演练。

★ LangSmith 集成：调试与评测一体化

OpenTelemetry 解决了"通用追踪"，但在 LLM 专项调试上略显单薄——它不会帮你做提示词版本对比、不会自动给 trace 打质量分、不会让你在 UI 里直接重放某次失败调用并改 prompt 重跑。这些正是 LangSmith 这类 LLM 专属可观测平台的强项。LangSmith 是 LangChain 团队推出的产品，深度集成 LangGraph / LangChain，能可视化 Agent 的执行图、支持 trace 级别的标注和评测，是当下 Agent 开发调试的事实首选。

LangSmith 的工作流是：你在代码里设置环境变量启用追踪，所有 LangChain / LangGraph 的执行都会自动上报到 LangSmith 后台；在后台你可以按用户、按场景筛选 trace，展开看每一步的输入输出；对失败 trace 可以直接点"Playground"修改 prompt 后重跑，定位是不是 prompt 的问题；还可以把一批 trace 标注成数据集，用它们做回归评测，衡量 prompt 修改是否带来整体提升。下面这段代码演示一个带 LangSmith 追踪的 LangGraph Agent。

# pip install langgraph langchain-openai langsmith
# 环境变量（也可写在 .env）：
#   export LANGSMITH_TRACING=true
#   export LANGSMITH_API_KEY=lsv2_***
#   export LANGSMITH_PROJECT=agent-prod
import os
os.environ["LANGSMITH_TRACING"] = "true"

from langgraph.prebuilt import create_react_agent
from langchain_openai import ChatOpenAI
from langchain_core.tools import tool

@tool
def search_order(order_id: str) -> str:
    """根据订单号查询订单状态。"""
    # 真实实现会查数据库
    return f"订单 {order_id} 已发货，预计明日送达。"

@tool
def refund_order(order_id: str) -> str:
    """为指定订单发起退款。"""
    return f"订单 {order_id} 退款已受理，3-5 工作日到账。"

model = ChatOpenAI(model="gpt-4o-mini")
agent = create_react_agent(model, [search_order, refund_order])

# 这次调用会自动在 LangSmith 后台生成一条 trace，
# 包含每一步的 thought / action / observation，便于回放调试。
result = agent.invoke({
    "messages": [{"role": "user", "content": "我的订单 A123 状态怎样？如果没到就退款。"}]
})
print(result["messages"][-1].content)

# 把失败 trace 转成评测数据集，做 prompt 迭代后的回归测试
from langsmith import Client
client = Client()
# 在 UI 里勾选若干失败 trace，导出为 dataset "agent-regression"
# 之后每次改 prompt 都跑一遍：
# for ex in client.list_examples(dataset_name="agent-regression"):
#     ...

LangSmith 的真正威力在于把"调试、评测、监控"三件事拧成一股绳：线上 trace 可以一键变成评测样本，评测样本可以驱动 prompt 迭代，迭代结果又通过 trace 验证。这种闭环让 Agent 系统的演进有了量化依据，而不是靠"感觉这个 prompt 更好"。如果你在做严肃的 Agent 产品，LangSmith（或开源替代如 Langfuse、Arize Phoenix）几乎是必备基建。

Agent 可观测性的三大支柱：trace、metrics、logs

Agent 系统的复杂度远超传统应用——一次用户请求可能触发几十次模型调用、工具调用、子 agent 协作，任何一环出问题都会影响最终结果。没有好的可观测性，agent 系统就是个黑盒，出问题只能瞎猜。可观测性的三大支柱——trace（追踪）、metrics（指标）、logs（日志）——在 agent 场景有特殊的应用方式。

Trace 是 agent 可观测性的核心。它把一次请求的完整执行路径记录下来：从用户输入开始，经过的每一次 LLM 调用（输入/输出/token/延迟）、每一次工具调用（参数/返回/耗时）、每一个子 agent 的边界。trace 要支持"树形结构"——主 agent 调用子 agent 形成嵌套，trace 要能体现这种父子关系。好的 trace 让你能"回放"任意一次请求的完整过程，定位是哪一步出了问题。这是 agent 排障最核心的工具。

Metrics 从个体 trace 聚合成系统级指标。关键指标分几类：性能类（端到端延迟、各环节延迟分布）、成本类（每次请求的 token/费用、模型调用次数）、质量类（用户满意度、任务成功率）、可靠性类（错误率、超时率、重试率）。metrics 用于监控整体健康度和发现系统性问题（如某时段延迟突增）。Logs 记录离散事件，用于补充 trace 的细节——trace 是结构化的主骨架，log 是细节血肉。三者结合，从宏观到微观全覆盖。

Trace完整执行路径单次请求排障回放
Metrics聚合指标系统级监控告警
Logs离散事件事件级细节补充

from opentelemetry import trace, metrics
from opentelemetry.trace import SpanKind
import logging, time, json

logger = logging.getLogger("agent")
tracer = trace.get_tracer("agent")
meter = metrics.get_meter("agent")

# 关键指标
req_counter = meter.create_counter("agent.requests")
latency_hist = meter.create_histogram("agent.latency")
token_counter = meter.create_counter("agent.tokens")
error_counter = meter.create_counter("agent.errors")

class TracedAgent:
    def handle(self, user_input):
        # 顶层 span：整个请求
        with tracer.start_as_current_span("agent.handle", kind=SpanKind.SERVER) as root:
            root.set_attribute("user_input", user_input[:100])
            req_counter.add(1)
            t0 = time.time()
            try:
                result = self._run(user_input, root)
                root.set_attribute("success", True)
                return result
            except Exception as e:
                root.set_attribute("success", False)
                root.record_exception(e)
                error_counter.add(1, {"type":type(e).__name__})
                logger.exception("agent 处理失败")   # log 补充细节
                raise
            finally:
                latency_hist.record(time.time()-t0)

    def _run(self, user_input, parent_span):
        # 嵌套 span：LLM 调用（trace 体现父子关系）
        with tracer.start_as_current_span("llm.call"):
            resp = self.llm.generate(user_input)
            token_counter.add(resp.usage.total_tokens)
            # log 记录细节（trace 不存全部，太占空间）
            logger.debug("LLM 调用", extra={"in":len(user_input),"out":len(resp)})
        # 嵌套 span：工具调用
        with tracer.start_as_current_span("tool.search"):
            results = self.search(resp.tool_input)
        return resp

Agent 评测与回归体系

光观测生产还不够——agent 每次改动（换模型、改 prompt、加工具）都可能引入退化。建立评测与回归体系，能在改动后快速发现质量变化，是 agent 持续迭代的守护网。这比传统软件的测试更难，因为 agent 输出是概率性的，不能简单断言"输入 X 必出 Y"。

评测体系的核心是"黄金集 + 多维评分"。黄金集是一批精心标注的 (输入, 期望特征) 对——期望不是固定输出，而是"答案应该满足的特征"（含某些关键事实、不含某些错误、调用对的工具、格式正确等）。每次改动后跑黄金集，用多维评分（事实正确性、格式合规、工具选择、用户满意度代理指标）评估是否退化。评分可以用规则（确定性高的特征）+ LLM-as-judge（语义特征）。

回归的工程实践是"分级门禁"。核心指标（如事实正确率）退化则阻塞发布；次要指标退化则报警但不阻塞；非核心指标波动则仅记录。这样既守住关键质量，又不被评估噪声卡死迭代。还要做"版本对比"——新版本 vs 旧版本在同一黄金集上的表现对比，差在哪里一目了然。长期积累黄金集和评测结果，形成 agent 质量演进的趋势线。

评测维度评估方法门禁级别
事实正确性规则+核查阻塞（核心）
工具选择正确规则匹配阻塞（核心）
格式合规规则校验阻塞
语义质量LLM judge报警（噪声大）
延迟/成本指标对比报警
用户满意度反馈代理记录
# Agent 评测与回归体系
import json, statistics

class AgentRegression:
    def __init__(self, golden_set, scorer):
        self.golden = golden_set   # [(input, expected_features)]
        self.scorer = scorer       # 多维评分器
        self.baseline = None       # 上线版本的分数

    def evaluate(self, agent_version):
        """跑黄金集，返回多维分数"""
        scores = {dim:[] for dim in ["factual","tool","format","semantic","latency"]}
        for inp, expected in self.golden:
            output = agent_version.run(inp)
            result = self.scorer.score(output, expected)
            for dim in scores: scores[dim].append(result[dim])
        # 聚合：取均值
        summary = {dim:statistics.mean(vals) for dim,vals in scores.items()}
        summary["overall"] = statistics.mean(summary.values())
        return summary

    def gate(self, new_scores, baseline=None):
        """分级门禁：判断能否发布"""
        baseline = baseline or self.baseline
        blocking = ["factual","tool","format"]   # 核心维度
        results = {"block":[], "warn":[], "ok":[]}
        for dim, score in new_scores.items():
            if dim == "overall": continue
            delta = score - baseline.get(dim, score)
            if dim in blocking and delta < -0.02:
                results["block"].append(f"{dim} 退化 {delta:+.2f}")
            elif delta < -0.05:
                results["warn"].append(f"{dim} 下降 {delta:+.2f}")
            else:
                results["ok"].append(f"{dim} {delta:+.2f}")
        can_release = len(results["block"]) == 0
        return can_release, results

# 使用：每次改 prompt/换模型，跑回归
reg = AgentRegression(golden_set, scorer)

★ 本章小结

可观测性是 Agent 系统从 demo 走向生产的分水岭，本章搭建了完整的可观测性体系。首先我们解释了 Agent 因"非确定性 + 多步组合"而比普通服务更需要可观测性，重新定义了三大支柱：调用追踪、成本监控、质量监控。其次，LLM 调用追踪基于 OpenTelemetry 标准，通过 OpenLLMetry 自动 instrumentation 给所有主流 SDK 装上符合 GenAI 语义约定的 span，实现跨语言、跨后端的全链路追踪。第三，Token 成本监控分调用级记账、聚合统计、预算控制三层，价目表应从配置中心动态加载，并在单次调用成本突增时告警以抓住隐性浪费。第四，错误率告警遵循"信噪比优先"原则，用多级阈值覆盖调用异常、模型异常、业务异常、成本异常四类，部署后必须用混沌测试验证告警链路。最后，LangSmith 在通用追踪之上补齐了 LLM 专项的调试与评测能力，能把线上 trace 一键转为评测样本，形成"调试-评测-监控"闭环。把这些组件组合起来，你的 Agent 系统就有了"行车记录仪 + 仪表盘 + 自动刹车"，足以支撑 7x24 小时稳定运行与持续迭代。
