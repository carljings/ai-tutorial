Cline 的 Agent 循环
XML 格式工具调用与 plan/act 双模式

Cline（原名 Claude Dev）是一个 VSCode 扩展形态的 Agent，它在 Agent 循环的设计上有两个鲜明特色：一是用 XML 标签 来结构化工具调用（而非 function calling 的 JSON），二是显式区分 plan 模式和 act 模式——先规划再执行。这两个设计都源于 Cline 对「让用户理解和掌控 Agent 行为」的重视。本章我们拆解它的循环实现。

阅读约 19 分钟 难度 ★★★★

★ 一、为什么用 XML 而非 function calling

大多数现代 Agent（OpenCode、Codex）用 LLM 的 function calling API——模型返回结构化的 JSON，包含工具名和参数。但 Cline 选择让模型在文本输出里用 XML 标签描述工具调用。这是一个有争议但有趣的选择。

选择 XML 的第一个理由是 模型兼容性。和 Aider 的 SEARCH/REPLACE 一样（参见 164 章），纯文本协议不依赖 function calling API，任何模型都能用。Cline 早期专注 Claude，但后来要支持 GPT、DeepSeek、本地模型，XML 让这种扩展零成本。

第二个理由是 可读性。XML 标签在文本流里是可见的，用户在 chat 界面能看到 Agent「正在调用什么工具」——这是透明的。而 function calling 的 JSON 通常被 UI 隐藏，用户不知道 Agent 在干什么。Cline 把工具调用作为文本流的一部分展示，增强了用户对 Agent 行为的感知。

第三个理由和 Claude 模型本身有关。Claude 在训练时被大量「教会」了用 XML 标签结构化输出（Anthropic 的 prompt engineering 指南里大量用 XML）。所以让 Claude 用 XML 调工具，是顺着它的强项走，正确率高。

Cline 的工具调用格式（在 assistant 文本输出里）：

我现在需要读取 auth.py 文件来了解现有的登录逻辑。

<read_file>
<path>src/auth.py</path>
</read_file>

读取后我会分析其中的 authenticate 函数。

注意 XML 工具调用是 嵌在自然语言里的——Agent 先说「我需要读取 auth.py」，然后给出 read_file 的 XML 块，再说「读取后我会分析」。这种「边想边说边调工具」的输出方式，让用户能跟随 Agent 的推理过程，而不是只看到一堆工具调用。

★ 二、XML 解析与流式提取

Cline 要从模型的流式文本输出里，实时提取出 XML 工具调用块。这比解析一次性返回的 JSON 难——你要边收 token 边判断「是不是开始了一个工具调用」「工具调用块结束了没有」。

// Cline 的流式 XML 解析（简化）
export class ToolCallParser {
  private buffer = "";
  private currentTool: { name: string; xml: string } | null = null;

  /** 每收到一段 delta 文本调用 */
  feed(chunk: string): { text?: string; toolCall?: ParsedToolCall } {
    this.buffer += chunk;
    const events: any = {};

    // 如果正在解析工具调用，继续累积直到闭合标签
    if (this.currentTool) {
      const closeTag = `</${this.currentTool.name}>`;
      const closeIdx = this.buffer.indexOf(closeTag);
      if (closeIdx >= 0) {
        // 工具调用块结束
        this.currentTool.xml += this.buffer.slice(0, closeIdx + closeTag.length);
        this.buffer = this.buffer.slice(closeIdx + closeTag.length);
        const parsed = parseToolXML(this.currentTool.xml);
        events.toolCall = parsed;
        this.currentTool = null;
      } else {
        // 还没收完，继续累积
        this.currentTool.xml += this.buffer;
        this.buffer = "";
      }
      return events;
    }

    // 检查是否开始了一个工具调用（即遇到了已知的工具标签）
    for (const toolName of KNOWN_TOOLS) {
      const openTag = `<${toolName}>`;
      const idx = this.buffer.indexOf(openTag);
      if (idx >= 0) {
        // 把标签前的文本作为普通输出
        if (idx > 0) {
          events.text = this.buffer.slice(0, idx);
        }
        // 开始累积工具调用
        this.currentTool = {
          name: toolName,
          xml: this.buffer.slice(idx),
        };
        this.buffer = "";
        return events;
      }
    }

    // 没有工具调用，全部是普通文本
    events.text = this.buffer;
    this.buffer = "";
    return events;
  }
}

// 解析完整的工具 XML 块
function parseToolXML(xml: string): ParsedToolCall {
  // <read_file><path>src/auth.py</path></read_file>
  const match = xml.match(/^<(\w+)>([\s\S]*)<\/\1>$/);
  if (!match) throw new Error("无效的工具 XML");
  const [, name, body] = match;
  // 解析 body 里的子标签作为参数
  const params: Record<string, string> = {};
  const paramRe = /<(\w+)>([\s\S]*?)<\/\1>/g;
  let pm;
  while ((pm = paramRe.exec(body))) {
    params[pm[1]] = pm[2];
  }
  return { name, params };
}

这段解析器的关键挑战是 流式边界的处理。XML 标签可能被 token 流切断——比如 <read_fi 后面接 le>。解析器必须缓冲未完成的片段，直到能判断「这是一个工具标签的开始」或「这是普通文本」。上面的实现通过 currentTool 状态机和 buffer 缓冲实现了这一点。

另一个细节：只有匹配到已知工具名（KNOWN_TOOLS）的标签才被当工具调用。如果模型输出了 <some_random_tag>，它会被当普通文本处理，不会触发解析。这防止了模型无意中输出的 XML 干扰工具系统。

★ 三、plan / act 双模式：先规划后执行

Cline 最有特色的设计是显式区分两种模式。在 plan 模式，Agent 只做规划和信息收集（读文件、搜索），不修改任何东西；在 act 模式，Agent 才真正执行修改（写文件、跑命令）。用户可以在这两种模式间切换。

这个设计的动机是 分离「想」和「做」。很多 Agent 的问题是边想边做——模型一开始就动手改代码，改到一半发现方向错了，已经产生了混乱的中间状态。plan 模式强制 Agent 先完整理解问题、列出步骤，用户确认计划合理后，再进入 act 模式执行。这大大降低了「方向性错误」的代价。

// Cline 的模式管理（简化）
export type Mode = "plan" | "act";

export class ClineProvider {
  private mode: Mode = "plan";

  async handleUserMessage(text: string) {
    if (this.mode === "plan") {
      await this.runPlanLoop(text);
    } else {
      await this.runActLoop(text);
    }
  }

  // plan 模式：只允许只读工具
  private async runPlanLoop(task: string) {
    const allowedTools = ["read_file", "list_dir", "search_codebase"];
    const systemPrompt = `你当前处于 PLAN 模式。
任务：${task}

在 plan 模式下，你只能使用只读工具（read_file、list_dir、search_codebase）来理解代码库。
你的目标是：
1. 充分理解用户的任务和现有代码结构
2. 制定一个清晰的实施计划，列出要修改的文件和步骤
3. 不要修改任何文件

完成后，用 <plan> 标签输出你的计划，用户会决定是否切换到 ACT 模式执行。`;

    await this.runAgentLoop(task, systemPrompt, allowedTools);
  }

  // act 模式：允许所有工具
  private async runActLoop(task: string) {
    const allowedTools = ALL_TOOLS;  // 包括 write_file, execute 等
    const systemPrompt = `你当前处于 ACT 模式。
任务：${task}

在 act 模式下，你可以使用所有工具，包括修改文件和执行命令。
请按照之前制定的计划执行，每一步都要谨慎。`;

    await this.runAgentLoop(task, systemPrompt, allowedTools);
  }

  // 用户切换模式
  setMode(mode: Mode) {
    this.mode = mode;
    this.postMessage({ type: "modeChanged", mode });
  }
}

这段代码揭示了 plan/act 的实现本质——同一套 Agent 循环，通过 system prompt 和允许的工具列表来区分行为。plan 模式不给 write_file、execute 这些有副作用的工具，即使模型想用也用不了。act 模式才放开全部工具。这种「用工具可用性来约束行为」的策略，比「在 system prompt 里说不要改」更可靠——前者是硬约束，模型无法绕过；后者是软约束，模型可能不听话。

★ 四、Agent 循环与工具执行

Cline 的 Agent 循环和前面几章讲的结构类似（准备 → 调模型 → 处理工具调用 → 继续），但因为用 XML 协议，有些细节不同。我们看核心循环：

// Cline 的 Agent 循环（简化）
export class ClineProvider {
  private async runAgentLoop(task: string, systemPrompt: string, allowedTools: string[]) {
    const parser = new ToolCallParser();
    let assistantText = "";

    while (true) {
      // 1. 调模型（流式）
      const stream = await this.model.streamComplete({
        system: systemPrompt,
        messages: this.messages,
      });

      // 2. 边收边解析
      for await (const chunk of stream) {
        const { text, toolCall } = parser.feed(chunk);
        if (text) {
          assistantText += text;
          this.postMessage({ type: "text", content: text });  // 实时显示
        }
        if (toolCall) {
          // 3. 校验工具是否被允许（plan 模式下拦截写工具）
          if (!allowedTools.includes(toolCall.name)) {
            // 把拒绝信息作为 tool_result 喂回去
            this.messages.push({
              role: "assistant",
              content: assistantText,
            });
            this.messages.push({
              role: "user",
              content: `错误：在当前模式下不能使用 ${toolCall.name} 工具。`,
            });
            continue;
          }

          // 4. 执行工具
          const result = await this.executeTool(toolCall);
          this.postMessage({ type: "toolResult", tool: toolCall.name, result });

          // 5. 把工具结果加入消息，继续下一轮
          this.messages.push({ role: "assistant", content: assistantText });
          this.messages.push({
            role: "user",
            content: `[工具 ${toolCall.name} 的结果]\n${result}`,
          });
          assistantText = "";
          break;  // 跳出 for await，回到 while 开始新一轮调模型
        }
      }

      // 6. 如果整轮没有工具调用，循环结束
      if (!this.lastRoundHadToolCall()) break;
    }
  }
}

注意第 3 步的 工具允许性校验——这是 plan/act 双模式的执行点。即使模型在 plan 模式下输出了 <write_file>，循环也会拦截它，返回错误信息。模型看到错误后，通常会切换到只读工具继续规划。这种「失败反馈引导模型行为」的模式，我们在 149、154 章也见过。

★ 五、工具结果的表达：XML 化的反馈

工具执行完，结果要以什么格式喂回给模型？Cline 用的是 XML 化的文本反馈，而不是结构化 JSON：

[工具 read_file 的结果]
<file_content path="src/auth.py" lines="45">
def authenticate(username, password):
    user = db.find_user(username)
    if user and verify_password(password, user.password_hash):
        return create_token(user)
    return None
</file_content>

[工具 execute 的结果]
<command>npm test</command>
<exit_code>1</exit_code>
<output>
FAIL  src/auth.test.js
  ● authenticate › should reject wrong password

    expect(received).toBe(expected)
    Expected: "invalid"
    Received: "error"
</output>

用 XML 包装结果有两个好处。一是 结构清晰：模型能区分 file_content 的 path 和实际内容、command 的 exit_code 和 output。二是 和工具调用格式对称：调用用 XML，结果也用 XML，整个对话的「机器可读部分」风格统一，模型更容易处理。

★ 六、与 function calling 方案的对比

把 Cline 的 XML 方案和 OpenCode/Codex 的 function calling 放在一起对比，能看出各自的取舍：

  • 正确率：function calling 由 API 层保证结构化输出的正确性，接近 100%；XML 要靠模型的自觉 + 解析器的容错，错误率略高。
  • 模型兼容性：XML 支持任何模型，function calling 要求模型支持特定 API。
  • 可读性：XML 嵌在文本里，用户能看到工具调用的上下文；function calling 通常被 UI 隐藏。
  • 流式处理：XML 的流式解析更复杂（要处理标签切断）；function calling 的流式有标准化的 delta 格式。
  • 调试：XML 出错时容易看到原始文本定位问题；function calling 的 JSON 错误需要看 API 响应。

总体来说，function calling 是「更工程化、更可靠」的选择，XML 是「更兼容、更透明」的选择。Cline 选 XML 是因为它早期专注 Claude（Claude 的 XML 训练强）且重视用户感知，这是符合它定位的合理决策。

★ 本章小结

Cline 的 Agent 循环在「透明性」和「可控性」上做了独特探索。要点回顾：

  • XML 工具调用：嵌在自然语言文本里，模型兼容性好、可读性强、契合 Claude 的训练偏好。
  • 流式解析：状态机 + buffer 处理标签被 token 切断的情况，只匹配已知工具名避免误触发。
  • plan/act 双模式：plan 只给只读工具（硬约束），act 才放开写工具，分离「想」和「做」降低方向性错误代价。
  • 工具允许性校验：循环里拦截不允许的工具，失败反馈引导模型行为。
  • XML 化结果：工具结果也用 XML 包装，和调用格式对称，结构清晰风格统一。
  • 与 function calling 对比：XML 更兼容更透明，function calling 更可靠更工程化，各有取舍。
