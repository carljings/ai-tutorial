KV Cache
自回归推理为什么离不开它

生成式大模型（GPT、GLM、Llama）是「自回归」的——一个字一个字往外蹦。如果每生成一个字，都把前面所有字重新跑一遍注意力，1000 字的文章要算上百万次冗余矩阵乘，慢到没法用。KV Cache 的思路简单到爆：把算过的 K、V 存下来，下次直接复用。这一招把推理复杂度从 O(n²) 降到 O(n)，是大模型能「实时对话」的基石。本章还讲清显存计算和 vLLM 的 PagedAttention。

阅读约 28 分钟 难度 ★★★★ 前置：第 98、99 章

★ 自回归生成：一次一个 token 的代价

生成式 Transformer 的推理方式叫自回归（autoregressive）：根据已有的 token 序列，预测下一个 token；把这个新 token 接到序列末尾，再预测下一个……如此循环。

假设要生成 1000 个 token。第 1 步输入 1 个 token 预测第 2 个；第 2 步输入 2 个 token 预测第 3 个……第 999 步输入 999 个 token 预测第 1000 个。如果不做任何优化，总计算量是：

总 FLOPs ∝ 1 + 2 + 3 + ... + 999 = 999×1000/2 ≈ 50 万（按序列长度的平方级增长）

而且每一步都要重新计算前面所有 token 的 K、V，但它们其实根本没变——前 998 个 token 的 K、V，在第 999 步和第 998 步是完全一样的！这是巨大的浪费。

第 1 步：输入 [t1] → 预测 t2
第 2 步：输入 [t1,t2] → 预测 t3（t1 的 K/V 重算了！）
第 3 步：输入 [t1,t2,t3] → 预测 t4（t1,t2 的 K/V 又重算了！）
▼ 没有 cache：每步都重复算所有历史
总计算量 O(n²)，无法用于实时推理
图 100-1 · 没有 KV cache 时，自回归生成的冗余计算

这种 O(n²) 的冗余，让生成 1000 字可能要几十秒。怎么破？把算过的 K、V 存起来，别重算。

01 KV Cache 的核心思想

KV Cache 的逻辑极其朴素。在注意力里，K = X·Wk，V = X·Wv。对于已经处理过的 token，它们的 K、V 是固定的（参数 Wk、Wv 不变，输入也不变）。所以：

把每个 token 的 K、V 算一次后存进缓存（KV cache），后续步骤直接读缓存，不再重算。

具体流程变成：

  • 第 1 步（prefill）：输入 [t1]，算 K1、V1，存进 cache。用 Q1 attend [K1] 得到预测。
  • 第 2 步：只算新 token t2 的 Q2、K2、V2。把 K2、V2 追加到 cache。用 Q2 attend [K1, K2]（K1 来自 cache，不重算）。
  • 第 k 步：只算新 token 的 Q、K、V，把 K、V 追加进 cache，用 Q attend 所有历史 K、V。

关键点：每步只算一个新 token 的 Q、K、V，注意力是 1×n（一个 Q 查询所有历史 K），而不是 n×n。总计算量从 O(n²) 降到 O(n)。

💡 类比：做笔记

没有 cache 像每次开会都从头翻所有历史会议纪要。有 cache 像你有个笔记本，每次只记新内容，需要时翻笔记本就行。笔记（K、V）记一次，永久复用。

02 为什么只 cache K 和 V，不 cache Q

这是理解 KV cache 的关键。回顾注意力：softmax(Q·K^T)·V。在第 k 步：

  • Q：只有当前 token t_k 的 Q 需要参与计算（要预测下一个 token）。历史 token 的 Q 在这一步用不到——因为分数矩阵只需要「当前 Q × 所有历史 K」这一行。
  • K、V：所有历史 token 的 K、V 都要用（当前 token 要 attend 它们）。

所以 Q 是「一次性的」，用完就丢；K、V 是「累积的」，要反复用。缓存只对反复用的东西有意义，于是只 cache K、V。这就是「KV Cache」名字的由来。

自回归推理的每一步：当前 Q · [所有历史 K（来自 cache）]^T → softmax → · [所有历史 V（来自 cache）] → 输出

03 两个阶段：Prefill 和 Decode

实际推理分成两个阶段，它们的计算模式完全不同：

阶段输入计算模式特点
Prefill（预填）整个 prompt（如 500 token）n × n 注意力计算密集，并行度高，一次算完填充 cache
Decode（解码）每次 1 个新 token1 × n 注意力访存密集，串行，读 cache 比算还慢

Prefill 阶段一次性处理 prompt，算量大但可并行，GPU 利用率高。Decode 阶段每步只算 1 个 token，算量极小但要从显存读整个 KV cache——访存带宽成为瓶颈。这就是为什么 decode 阶段往往比 prefill 慢得多（按每 token 计），也是推理优化的重点。

📌 Decode 是访存瓶颈

Decode 时每生成 1 个 token，要从显存读取整个 KV cache（可能几 GB），只做一点点矩阵乘。算力（FLOPS）远没用满，时间全花在搬数据上。这叫 memory-bound。优化方向：①减小 cache（GQA、量化）；②增大 batch（多个请求共享模型权重读取）；③连续批处理（Continuous Batching）。

04 代码：手写一个 KV Cache 注意力

用一个最小实现看清楚 cache 怎么工作：

kv_cache_attention.py —— 带 KV cache 的注意力层
import torch
import torch.nn as nn
import torch.nn.functional as F
import math

class CachedAttention(nn.Module):
    def __init__(self, d_model, n_heads):
        super().__init__()
        self.n_heads = n_heads
        self.d_k = d_model // n_heads
        self.W_qkv = nn.Linear(d_model, 3 * d_model, bias=False)
        self.W_o = nn.Linear(d_model, d_model, bias=False)
        # KV cache：每层一对
        self.k_cache = None  # (B, n_heads, S, d_k)
        self.v_cache = None

    def forward(self, x, use_cache=True):
        """
        x: (B, S_new, d_model) —— 本次新增的 token
        use_cache=True 时复用历史 K/V
        """
        B, S_new, _ = x.shape
        qkv = self.W_qkv(x)
        q, k, v = qkv.chunk(3, dim=-1)

        def reshape(t):
            return t.view(B, S_new, self.n_heads, self.d_k).transpose(1, 2)

        q, k, v = reshape(q), reshape(k), reshape(v)

        if use_cache and self.k_cache is not None:
            # 拼接历史 K/V
            k = torch.cat([self.k_cache, k], dim=2)
            v = torch.cat([self.v_cache, v], dim=2)

        # 更新 cache
        if use_cache:
            self.k_cache = k
            self.v_cache = v

        # 注意力：q 是 S_new 个查询，k/v 是全部历史
        scores = q @ k.transpose(-2, -1) / math.sqrt(self.d_k)
        weights = F.softmax(scores, dim=-1)
        out = weights @ v
        out = out.transpose(1, 2).contiguous().view(B, S_new, -1)
        return self.W_o(out)

    def reset_cache(self):
        self.k_cache = None
        self.v_cache = None

# 模拟自回归生成
torch.manual_seed(0)
attn = CachedAttention(d_model=64, n_heads=8)
attn.reset_cache()

# Prefill：处理 prompt（5 个 token）
prompt = torch.randn(1, 5, 64)
_ = attn(prompt, use_cache=True)
print(f"Prefill 后 cache 长度: {attn.k_cache.shape[2]}")  # 5

# Decode：逐个生成 3 个新 token
for step in range(3):
    new_token = torch.randn(1, 1, 64)  # 每次只输入 1 个 token
    _ = attn(new_token, use_cache=True)
    print(f"Decode 第 {step+1} 步后 cache 长度: {attn.k_cache.shape[2]}")
# 输出：6, 7, 8 —— cache 线性增长，历史 K/V 不重算

核心就是 torch.cat([self.k_cache, k], dim=2) 这一步：把历史的 K 和新 token 的 K 拼起来。Q 只用新的，K/V 用全部。这就是 KV cache 的全部魔法。

05 KV Cache 的显存计算

cache 省了计算，但吃显存。精确公式：

KV cache 显存 = 2 × n_layers × n_kv_heads × seq_len × head_dim × dtype_bytes
kv_memory.py —— 主流模型的 KV cache 显存对比
def kv_cache_gb(n_layers, n_kv_heads, head_dim, seq_len, dtype="fp16"):
    bytes_per = {"fp16": 2, "int8": 1, "int4": 0.5}[dtype]
    mem = 2 * n_layers * n_kv_heads * seq_len * head_dim * bytes_per
    return mem / 1024**3

models = {
    "Llama-2-7B (MHA, 32KV头)":  dict(n_layers=32, n_kv_heads=32, head_dim=128),
    "Llama-2-70B (MHA, 64KV头)": dict(n_layers=80, n_kv_heads=64, head_dim=128),
    "Llama-3-70B (GQA, 8KV头)":  dict(n_layers=80, n_kv_heads=8,  head_dim=128),
    "GLM-4-9B (GQA, 4KV头)":     dict(n_layers=40, n_kv_heads=4,  head_dim=128),
}

for name, cfg in models.items():
    for seq in [4096, 32768]:
        mem = kv_cache_gb(seq_len=seq, **cfg)
        print(f"{name:32s} @{seq:>6} tokens: {mem:6.2f} GB (FP16)")
    print()

你会看到触目惊心的数字：Llama-2-70B 在 32K 上下文时 KV cache 高达 80 GB，比 70B 模型参数本身（FP16 约 140 GB）的一半还多。而 GQA 的 Llama-3-70B 只有 10 GB，省了 87%。这就是 GQA 的实战价值。

06 PagedAttention：像操作系统管内存一样管 cache

即便用了 GQA，KV cache 的管理仍是个难题。传统做法是为每个请求预分配连续的显存（按最大可能长度预留）。问题：

  • 内部碎片：预分配 4K，实际只生成 500，3.5K 显存浪费。
  • 无法动态扩容：生成超长就要重新分配、拷贝，昂贵。
  • 并发难：不同请求长度不一，显存池难以高效复用。

vLLM 提出的 PagedAttention 借鉴了操作系统的虚拟内存分页：把 KV cache 切成固定大小的「页（block）」（比如每页存 16 个 token 的 K/V），用页表管理逻辑位置到物理位置的映射。请求需要更多空间时，按页分配，不需要连续。

逻辑视图：请求 A 序列 [t1...t500]，连续
▼ PagedAttention 分页
物理页 3
t1~t16
物理页 7
t17~t32
物理页 12
t33~t48
...
页表：逻辑→物理映射，按需分配，无碎片
图 100-2 · PagedAttention：逻辑连续，物理分页，按需分配

PagedAttention 的好处：

  • 几乎零碎片：只按实际使用的页分配，内部碎片最多一页（16 token）。
  • 动态扩容：序列变长就多分配几页，无需拷贝。
  • 高并发：不同请求的页可以散布在显存各处，显存利用率接近 100%。
  • 支持 Continuous Batching：请求可以随时加入/退出 batch，每个用各自的页。

vLLM 凭借 PagedAttention，把推理吞吐量提升了 2~4 倍。这是 2023 年推理框架的革命性进步，现在几乎所有高性能推理框架（vLLM、TGI、TensorRT-LLM）都采用了类似思想。

💡 类比：虚拟内存

操作系统给每个进程一个「看起来连续」的虚拟地址空间，实际映射到物理上分散的页帧。进程不用关心物理布局，按需分页。PagedAttention 对 KV cache 做了完全一样的事——这就是它名字里「Paged」的由来。

07 KV Cache 的其他优化手段

除了 GQA 和 PagedAttention，还有一系列压缩 cache 的技巧：

技术原理省显存
KV 量化FP16 → INT8/INT42~4 倍
KV 淘汰（Eviction）丢弃注意力权重低的 token 的 K/V取决于淘汰率
共享前缀 cache多请求共用系统提示的 K/V系统提示越长省越多
Cache 压缩用更小的向量表示 K/V有损

其中共享前缀 cache 在实际产品里特别有用：很多请求有相同的系统提示（几百到几千 token），这部分 K/V 只算一次，所有请求共享。GLM、Claude 的 API 都在底层做了这个优化，长系统提示的批量请求成本大幅下降。

★ KV Cache 显存计算器

KV cache 在长上下文批量推理时常常比权重还占显存。手动算容易漏掉 GQA 和量化因子，写个计算器一劳永逸：

from dataclasses import dataclass

@dataclass
class ModelCfg:
    n_layers: int
    n_kv_heads: int       # GQA 后的 KV head 数
    head_dim: int
    hidden: int
    vocab: int
    dtype_bytes: int = 2  # FP16=2, INT8=1, INT4=0.5

def kv_cache_bytes(cfg: ModelCfg, batch: int, seq_len: int,
                   kv_quant_bytes: int | None = None) -> int:
    """KV cache 显存 = 2(K和V) * L * n_kv * S * head_dim * batch * dtype。"""
    dt = kv_quant_bytes if kv_quant_bytes else cfg.dtype_bytes
    return 2 * cfg.n_layers * cfg.n_kv_heads * seq_len * cfg.head_dim * batch * dt

def model_weights_gb(cfg: ModelCfg) -> float:
    """粗估权重显存（参数量 * dtype 字节）。"""
    # 近似：权重 ≈ 12 * hidden^2 * n_layers + vocab*hidden
    params = 12 * cfg.hidden**2 * cfg.n_layers + cfg.vocab * cfg.hidden
    return params * cfg.dtype_bytes / 1024**3

# GLM-4-9B 量级
glm = ModelCfg(n_layers=40, n_kv_heads=2, head_dim=128,
               hidden=4096, vocab=151552, dtype_bytes=2)

for batch, seq in [(1, 4096), (1, 32768), (32, 4096), (32, 32768)]:
    kv_fp16 = kv_cache_bytes(glm, batch, seq) / 1024**3
    kv_int8 = kv_cache_bytes(glm, batch, seq, kv_quant_bytes=1) / 1024**3
    print(f"batch={batch:2d} seq={seq:5d} | KV(FP16)={kv_fp16:6.2f}GB "
          f"KV(INT8)={kv_int8:6.2f}GB")

print(f"\n权重: {model_weights_gb(glm):.1f} GB")
# 结论：batch=32 + seq=32768 时 KV cache(18GB) 已超过权重，
# 这就是长上下文批量服务必须上 GQA + KV 量化的原因

08 本章小结

KV Cache 核心原理
  • 把历史 token 的 K、V 存下来，避免重算
  • 每步只算新 token 的 Q/K/V，注意力 1×n
  • 复杂度从 O(n²) 降到 O(n)
  • 只 cache K、V（Q 一次性，不用缓存）
工程实战要点
  • Prefill 计算密集，Decode 访存密集
  • 显存 = 2·L·n_kv·S·d·dtype，长上下文易爆
  • GQA / 量化 / PagedAttention 三板斧
  • vLLM 的 PagedAttention 提速 2~4 倍

KV Cache 解决了「推理重算」的问题，但注意力的另一个噩梦还没解决——训练和 prefill 阶段的 O(n²) 计算与显存。这个问题在长上下文（32K、128K）时尤其致命：分数矩阵 n×n 大到显存放不下。下一章的 Flash Attention，用 tiling 分块和 online softmax，巧妙绕开了「把整个 n×n 矩阵实例化」的步骤，让注意力既快又省显存。
