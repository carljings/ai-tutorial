◆ 第 220 章 · Agent 设计模式

自主循环模式

持续向目标前进的 agent

深度进阶★★★

从"回答一次"到"持续工作"

前面的模式大多是"用户问一次，agent 答一次"。自主循环（Autonomous Loop）模式不同：给 agent 一个目标，它持续工作直到目标达成或无法继续。比如"监控这个网站，价格降到 500 就通知我"——agent 会周期性地检查、判断、必要时通知。这是 agent 最接近"自主智能"的形态。

自主循环的关键设计

自主循环要解决几个独特问题，这些是单轮 agent 不需要考虑的：

问题挑战解决方式
何时停止怎么判断"目标达成"可验证的成功条件 + 最大轮次兜底
资源控制无限循环烧钱预算上限（token/时间/调用次数）
状态持久化中途断了能恢复状态存盘（checkpoint）
错误恢复某步失败怎么办重试 + 降级 + 人工介入
触发时机什么时候跑下一轮定时/事件驱动/条件触发

自主循环的骨架

async def autonomous_loop(goal, agent, budget, stop_condition):
    """自主循环：持续向目标前进。"""
    state = {"goal": goal, "step": 0, "spent": 0}
    while True:
        state["step"] += 1
        # 1. 预算检查（防止无限烧钱）
        if state["spent"] > budget:
            return {"status": "budget_exceeded", "state": state}
        # 2. 让 agent 决定下一步
        action = agent.decide_next(state)
        # 3. 执行
        try:
            result = await action.execute()
            state["spent"] += action.cost
            state["last_result"] = result
        except Exception as e:
            state["errors"].append(e)
            # 错误恢复：重试或降级
            continue
        # 4. 检查终止条件
        if stop_condition(state):
            return {"status": "goal_achieved", "state": state}
        # 5. 持久化状态（断点恢复）
        await checkpoint(state)
        # 6. 等待下一轮（定时/事件）
        await wait_next_trigger()
💡 要点
自主循环最关键的不是算法，而是"安全护栏"：预算上限防烧钱、最大轮次防死循环、状态持久化防中断丢失、人工介入通道防失控。没有这些护栏的自主 agent 是危险的。

自主循环的典型应用

自主循环适合"需要持续监控/探索"的任务：价格监控 agent（周期检查）、深度研究 agent（持续搜索直到信息充分）、自动化运维 agent（监控异常并修复）、个人助理 agent（主动提醒和代办）。这些任务的共同点是：目标明确但需要持续努力，不能一次完成。AutoGPT 是早期自主循环的代表（虽然效果有限），现代的 Deep Research agent 是更成熟的实现。

目标分解：从模糊目标到可执行子任务

第一节给了自主循环的骨架，但骨架里的"让 agent 决定下一步"是个黑盒。自主 agent 最难的一步其实是"目标分解"——把"写一份市场调研报告"这种模糊目标，拆成可执行、可验证的子任务。下面给出一个生产级的目标分解器实现。

# 目标分解：把模糊目标拆成可验证的子任务
from dataclasses import dataclass
from typing import Literal

@dataclass
class SubGoal:
    description: str
    success_criteria: str       # 可验证的成功标准
    tool_hint: str | None       # 建议用什么工具
    priority: Literal["high", "medium", "low"]
    estimated_steps: int        # 预估需要几轮

class GoalDecomposer:
    """把高层目标分解成可执行的子目标列表。"""
    def __init__(self, llm):
        self.llm = llm

    async def decompose(self, goal: str) -> list[SubGoal]:
        prompt = f"""你是目标分解专家。把以下目标分解为 3-7 个子目标。
        每个子目标必须满足：
        1. 有明确的、可验证的成功标准（不是'充分调研'而是'找到3个竞品的定价数据'）
        2. 可独立执行（不依赖未完成的兄弟目标，除非声明依赖）
        3. 粒度适中（一个子目标 1-3 步可完成）

        目标: {goal}

        输出 JSON 列表，每项:
        {{description, success_criteria, tool_hint, priority, estimated_steps}}"""
        resp = await self.llm.invoke(prompt)
        return [SubGoal(**s) for s in resp.subgoals]

    async def refine(self, subgoal: SubGoal, feedback: str) -> SubGoal:
        """根据执行反馈细化子目标（太粗就拆细，太细就合并）。"""
        prompt = f"""子目标 '{subgoal.description}' 执行困难：
        {feedback}
        请调整：要么拆成更小的步骤，要么放宽成功标准。输出调整后的子目标。"""
        resp = await self.llm.invoke(prompt)
        return SubGoal(**resp)

# 使用示例
decomposer = GoalDecomposer(llm)
subgoals = await decomposer.decompose("为我们的 AI 编辑器做一份竞品分析报告")
# 可能输出：
# SubGoal("调研主流 AI 编辑器", "列出 5 款及其核心功能", "search", "high", 3)
# SubGoal("收集定价数据", "每款至少一个定价方案", "search", "high", 2)
# SubGoal("分析差异化", "对比至少 3 个维度", "analyze", "medium", 2)
# SubGoal("撰写报告", "1500字结构化报告", "write", "high", 1)

这个分解器的核心设计是"成功标准必须可验证"。这是自主 agent 不跑偏的关键——如果子目标是"充分调研"，agent 永远不知道什么叫"充分"，可能无限调研下去；改成"找到 3 个竞品的定价数据"，agent 检查"是否找到 3 个"就知道这个子目标完成了。`estimated_steps` 字段用于预算控制——如果所有子目标的预估步数加起来超过预算，分解器应该提前预警或简化计划。`refine` 方法实现了"执行中发现子目标太难就重新拆分"的自适应能力，这是自主 agent 在长任务中不卡死的关键。

💡 要点
目标分解是自主 agent 成败的分水岭。AutoGPT 早期效果差，很大程度就是因为分解质量低——它把目标拆得太泛或太死板。现代 Deep Research agent 效果好，核心改进就是分解更精细、成功标准更可验证。投入精力优化分解 prompt，比调任何模型参数都有效。

自我评估：让 agent 判断自己的进展

自主循环里，"检查终止条件"这一步如果只靠硬编码规则，会很脆弱。更好的做法是让 agent 自己评估"我离目标还有多远"——这就是自我评估。但自我评估有风险（模型可能自我欺骗），需要设计约束机制。

# 自我评估：agent 判断自己离目标的距离
from dataclasses import dataclass

@dataclass
class SelfAssessment:
    progress: float             # 0.0~1.0 完成度
    confidence: float           # 评估置信度
    remaining_subgoals: list[str]   # 还没完成的子目标
    should_continue: bool       # 是否应该继续
    reason: str                 # 判断理由

class SelfAssessor:
    def __init__(self, llm):
        self.llm = llm

    async def assess(self, goal: str, subgoals: list, state: dict) -> SelfAssessment:
        # 关键：用"已完成 vs 未完成"的客观证据，而非主观感觉
        prompt = f"""客观评估任务完成度。
        原始目标: {goal}
        子目标及完成状态:
        {self._format_subgoals(subgoals, state)}

        评估要求：
        1. 只基于客观证据判断（子目标的 success_criteria 是否满足）
        2. progress 要保守，宁可低估也不要高估
        3. 如果有任何关键子目标未完成，should_continue=true
        4. confidence 反映你对评估本身的把握（证据不足时降低）

        输出 JSON:
        {{progress, confidence, remaining_subgoals, should_continue, reason}}"""
        resp = await self.llm.invoke(prompt)
        return SelfAssessment(**resp)

    def _format_subgoals(self, subgoals, state):
        lines = []
        for sg in subgoals:
            evidence = state.get("completed", {}).get(sg.description, "未完成")
            lines.append(f"- [{evidence}] {sg.description} (标准: {sg.success_criteria})")
        return "\n".join(lines)

# 防自我欺骗的约束机制：
async def guarded_assess(assessor, goal, subgoals, state, external_checker=None):
    """带外部校验的自我评估，防止 agent 自欺欺人。"""
    self_eval = await assessor.assess(goal, subgoals, state)

    # 约束 1：如果 agent 说完成了，用独立检查验证
    if not self_eval.should_continue and external_checker:
        external_result = await external_checker(goal, state)
        if not external_result.passed:
            # 外部检查不通过 → 覆盖 agent 的自我评估
            return SelfAssessment(
                progress=self_eval.progress * 0.7,   # 打折
                confidence=0.3,   # 降低置信度
                remaining_subgoals=external_result.gaps,
                should_continue=True,
                reason=f"外部检查发现未满足: {external_result.gaps}",
            )

    # 约束 2：progress 连续 3 轮不变 → 可能卡住了
    if state.get("stall_counter", 0) >= 3:
        return SelfAssessment(0.5, 0.2, ["检测到停滞"],
                              should_continue=False, reason="连续停滞，建议人工介入")

    return self_eval

自我评估的最大风险是"模型自我欺骗"——agent 倾向于高估自己的完成度（尤其是用了它自己生成的"证据"时）。`guarded_assess` 的两个约束就是针对这个：① 当 agent 说"完成了"时，用一个独立的检查器（可以是另一个 LLM 或确定性函数）验证，不通过就覆盖评估；② 追踪 progress 是否连续多轮停滞——如果 agent 连续 3 轮都说"80% 完成"但没进展，很可能卡住了，应该停止并请求人工介入。这种"不信任自我评估"的防御性设计，是自主 agent 安全的关键。生产实现里，external_checker 最好用确定性代码（如"检查文件是否生成""检查数据条数"），而非再调一个 LLM——因为两个 LLM 可能犯同样的错。

终止条件判断：多重护栏

第一节提到"可验证的成功条件 + 最大轮次兜底"，这里展开成完整的多重终止条件系统。自主 agent 的终止绝不能只靠单一条件——那样要么太容易误终止（任务没完成就停了），要么容易不终止（死循环）。正确做法是组合多个条件，任一触发就评估是否真的该停。

# 终止条件：多重护栏
from dataclasses import dataclass
from enum import Enum

class TerminationReason(Enum):
    GOAL_ACHIEVED = "所有子目标完成"
    BUDGET_EXHAUSTED = "预算（token/时间/步数）耗尽"
    MAX_ITERATIONS = "达到最大轮次"
    STUCK = "连续停滞"
    ERROR_RATE = "错误率过高"
    HUMAN_STOP = "人工停止"
    NO_PROGRESS = "多轮无进展"

@dataclass
class TerminationCheck:
    """组合多个终止条件，返回是否终止及原因。"""
    max_iterations: int = 30
    max_tokens: int = 100_000
    max_time_seconds: int = 600
    max_consecutive_errors: int = 5
    max_stall_rounds: int = 3        # progress 不变的最大轮数

    def should_terminate(self, state: dict, assessment: SelfAssessment) -> tuple[bool, TerminationReason]:
        # 条件 1：目标达成（最理想的终止）
        if assessment.progress >= 0.95 and not assessment.remaining_subgoals:
            return True, TerminationReason.GOAL_ACHIEVED

        # 条件 2：迭代上限
        if state["iteration"] >= self.max_iterations:
            return True, TerminationReason.MAX_ITERATIONS

        # 条件 3：预算耗尽
        if state["tokens_spent"] > self.max_tokens:
            return True, TerminationReason.BUDGET_EXHAUSTED
        if state["elapsed_seconds"] > self.max_time_seconds:
            return True, TerminationReason.BUDGET_EXHAUSTED

        # 条件 4：连续错误
        recent = state["history"][-self.max_consecutive_errors:]
        if len(recent) == self.max_consecutive_errors and all(r.failed for r in recent):
            return True, TerminationReason.ERROR_RATE

        # 条件 5：停滞检测
        recent_progress = [h.progress for h in state["history"][-self.max_stall_rounds:]]
        if len(recent_progress) >= self.max_stall_rounds:
            if max(recent_progress) - min(recent_progress) < 0.05:
                return True, TerminationReason.STUCK

        return False, None

# 集成到自主循环
async def safe_autonomous_loop(goal, agent, budget):
    decomposer = GoalDecomposer(agent.llm)
    assessor = SelfAssessor(agent.llm)
    terminator = TerminationCheck(max_iterations=30, max_tokens=budget)

    subgoals = await decomposer.decompose(goal)
    state = {"iteration": 0, "tokens_spent": 0, "history": [], "elapsed_seconds": 0}

    while True:
        state["iteration"] += 1
        # 执行下一个子目标
        result = await agent.execute_next(subgoals, state)
        state["history"].append(result)
        state["tokens_spent"] += result.tokens_used

        # 自我评估
        assessment = await assessor.assess(goal, subgoals, state)
        result.progress = assessment.progress

        # 终止检查（多重条件）
        should_stop, reason = terminator.should_terminate(state, assessment)
        if should_stop:
            return {
                "status": "completed" if reason == TerminationReason.GOAL_ACHIEVED else "terminated",
                "reason": reason.value,
                "progress": assessment.progress,
                "iterations": state["iteration"],
                "tokens": state["tokens_spent"],
            }

        await checkpoint(state)   # 持久化

这个多重护栏系统的设计原则是"宁可保守终止，不要无限运行"。六个终止条件覆盖了所有常见失败模式：目标达成（正常结束）、预算耗尽（防烧钱）、迭代上限（防死循环）、错误率（防故障扩散）、停滞（防空转）。注意 `GOAL_ACHIEVED` 的判断阈值是 progress >= 0.95 而非 1.0——这承认了"完美完成"往往不现实，95% 已是高质量交付，追求最后 5% 可能要花 50% 的预算。停滞检测（条件 5）是最有价值的护栏之一：它通过"最近 N 轮 progress 变化幅度"判断 agent 是否在空转——如果连续 3 轮 progress 变化不到 5%，说明 agent 卡住了，继续跑只是浪费钱，不如停下来让人工介入。这种"能识别自己卡住并主动求助"的能力，是自主 agent 从"玩具"走向"工具"的标志。

终止条件触发阈值含义后续动作
目标达成progress>=0.95任务完成返回结果
预算耗尽token/时间超限资源用尽返回部分结果
迭代上限iteration>=30可能死循环返回部分结果+告警
连续错误5轮全失败系统性故障返回错误+请求人工
停滞3轮无进展卡住了返回当前状态+建议
⚠️ 落地提醒
自主 agent 在生产环境部署时，"终止条件"比"执行能力"更重要。一个能力中等但能可靠终止的 agent，远比一个能力很强但可能无限运行的 agent 安全。建议把终止逻辑做成独立模块，充分测试每个条件——因为这是防止 agent 失控的最后防线。另外，任何非 GOAL_ACHIEVED 的终止都应通知人类，让人类决定是重新规划、放宽目标、还是放弃任务。

自主循环的安全边界与终止条件

自主循环模式（autonomous loop）让 agent 自己决定下一步做什么、持续运行直到目标达成。这是 agent 自主性的极致，也是风险最高的模式——没有人工介入，agent 一旦"跑偏"可能持续做无用功甚至造成破坏。安全边界和终止条件是自主循环的生命线，缺一不可。

安全边界是"agent 不能逾越的红线"。它分几层。动作白名单：明确 agent 能自主执行哪些动作（读文件、查信息），哪些必须人工确认（发邮件、删数据、花钱）。资源上限：单次自主循环的总开销封顶（最多花多少钱、调多少次工具、跑多久），到了上限强制停止。影响范围隔离：agent 的操作限制在沙箱或特定范围内，不能影响外部系统（生产环境、真实用户）。这些边界是硬性的，agent 自己不能解除。

终止条件解决"循环何时停"。光靠 agent 自己判断"目标达成了"不可靠——它可能误判（认为完成其实没完成）或死循环（永远觉得"再做一点会更好"）。所以要有外部强制的终止条件：目标达成校验（用独立的验收器检查产出是否真满足目标，而非 agent 自评）、停滞检测（连续 N 步没有实质进展则停，可能卡住了）、预算耗尽（时间/金钱/步数用完）、异常检测（产出质量持续下降、或进入危险区域）。多个条件任一触发即终止。

终止条件触发信号解决的问题处理
目标达成验收器通过正常完成返回结果
停滞检测N 步无进展卡死循环停止 + 报告现状
预算耗尽时间/金钱/步数到顶成本失控返回最优中间结果
质量下降产出评分连降越做越差回滚 + 停止
危险区域触及安全红线破坏风险立即终止 + 告警
外部中断用户/监控停止人为干预保存现场退出
# 自主循环 + 多重终止条件 + 安全边界
import time
class AutonomousLoop:
    def __init__(self, agent, verifier, policy):
        self.agent = agent
        self.verifier = verifier      # 独立验收器（非 agent 自评）
        self.policy = policy          # 安全边界（预算/白名单等）

    def run(self, goal):
        budget = {"steps":0, "tokens":0, "cost":0.0, "start":time.time()}
        history = []
        last_progress_step = 0
        best_result = None
        while True:
            # === 检查所有终止条件（每步先查）===
            # 1. 预算耗尽
            if budget["steps"] >= self.policy.max_steps:
                return self._finish("budget_steps", best_result, history)
            if budget["cost"] >= self.policy.max_cost:
                return self._finish("budget_cost", best_result, history)
            if time.time()-budget["start"] > self.policy.max_time:
                return self._finish("budget_time", best_result, history)
            # 2. 停滞检测：连续 N 步无进展
            if budget["steps"] - last_progress_step > self.policy.stall_limit:
                return self._finish("stalled", best_result, history)
            # 3. 危险区域：触及安全红线
            if self._in_danger_zone(history):
                return self._finish("danger", best_result, history)

            # === 执行一步（受安全边界约束）===
            action = self.agent.decide_next(goal, history)
            if not self.policy.is_allowed(action):   # 白名单检查
                # 越界动作：拒绝 + 让 agent 换方案
                history.append(("rejected", action))
                continue
            result = self.agent.execute(action)
            budget["steps"] += 1
            budget["tokens"] += result.tokens_used
            budget["cost"] += result.cost
            history.append((action, result))

            # 4. 目标达成（独立验收，非 agent 自评）
            if self.verifier.check(goal, history):
                return self._finish("achieved", result, history)
            # 5. 质量下降检测
            if self._is_regression(history):
                return self._finish("regression", best_result, history)
            # 更新最优 + 进展标记
            if self._is_progress(result, best_result):
                best_result = result; last_progress_step = budget["steps"]

    def _finish(self, reason, result, history):
        return {"status":reason, "result":result,
                "steps":len(history), "history":history}

自主性的分级与人机协作设计

自主循环不是"全有或全无"——自主性是个连续光谱，从完全人工到完全自主，中间有多个级别。理解自主性分级，才能根据任务的风险和复杂度选择合适的自主级别，并在关键节点设计人机协作，避免"要么全交（风险高）要么全管（累死人）的两极化。

业界常用 SAE 自动驾驶分级的思想类比 agent 自主性，大致分五级。L1 辅助：agent 只提供建议，所有执行由人决定（如"建议这样改代码"，人确认才执行）。L2 部分自主：agent 自主执行低风险操作，高风险需人确认（如自动跑测试、但部署要人批）。L3 条件自主：agent 自主完成大部分工作，遇到不确定时请求人介入（如自动开发，遇到歧义问人）。L4 高度自主：agent 在限定领域完全自主，仅极端情况需人（如自动运维，故障时才报警）。L5 完全自主：无需人工，自己搞定一切（目前基本不存在，且多数场景不需要）。

人机协作的设计要点是"按风险和置信度动态调整自主级别"。不是固定一个级别，而是根据当前操作的"风险×置信度"实时调整——风险低且置信度高就自主执行，风险高或置信度低就请求人确认。这种动态自主既保证了效率（能自主的自主），又控制了风险（该问的必问）。关键还要设计好"请求人介入"的体验——不能频繁打扰（烦死），也不能沉默太久（失控），要在关键决策点恰当地拉人进来。

自主级别agent 角色人角色适合任务
L1 辅助建议/分析所有执行高风险探索
L2 部分自主低风险自主确认高风险开发辅助
L3 条件自主大部分自主不确定时介入常规开发
L4 高度自主限定域全自主极端情况运维/监控
L5 完全自主无需人工几乎不介入极少（理想）
# 动态自主级别：按风险×置信度实时调整
class DynamicAutonomy:
    def __init__(self, agent, policy):
        self.agent = agent
        self.policy = policy   # 定义各级别的允许动作

    def execute(self, action):
        risk = self._assess_risk(action)         # low/medium/high/critical
        confidence = self.agent.confidence(action)  # 0-1
        level = self._decide_level(risk, confidence)

        if level == "L1":
            # 仅建议，等人执行
            return {"proposal": action, "need_human": True}
        elif level == "L2":
            # 低风险自主，高风险确认
            if risk in ("low",):
                return self.agent.run(action)
            return {"proposal": action, "need_confirm": True}
        elif level == "L3":
            # 自主，低置信度才问
            if confidence < 0.6:
                return {"action": action, "need_review": True, "auto_if_no_resp": True}
            return self.agent.run(action)
        elif level == "L4":
            # 高度自主，仅 critical 才问
            if risk == "critical":
                return {"proposal": action, "urgent": True}
            return self.agent.run(action)

    def _decide_level(self, risk, confidence):
        """风险×置信度 → 自主级别"""
        matrix = {
            ("low",):     "L4",   # 低风险：高度自主
            ("medium",):  "L3" if confidence>0.7 else "L2",
            ("high",):    "L2" if confidence>0.8 else "L1",
            ("critical",):"L1",   # 关键：仅建议
        }
        for keys, lvl in matrix.items():
            if risk in keys: return lvl
        return "L1"   # 默认保守

    def _assess_risk(self, action):
        if action.type in ("read","query","test"): return "low"
        if action.type in ("write","deploy"):     return "high"
        if action.type in ("delete","pay","send"):return "critical"
        return "medium"

本章小结

自主循环让 agent 从"一问一答"进化为"持续工作"。它的核心挑战不是推理能力，而是工程上的安全护栏：预算控制、状态持久化、错误恢复、终止条件。这是 agent 走向真正实用的关键模式——大多数生产环境的 agent 都需要某种程度的自主循环。
