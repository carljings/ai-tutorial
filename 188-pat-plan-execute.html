◆ 第 212 章 · Agent 设计模式

Plan-and-Execute

把规划和执行解耦

深度进阶★★★

ReAct 的问题：缺乏全局视野

ReAct 是"走一步看一步"——每步只根据当前观察决定下一步。这在简单任务上够用，但复杂任务需要全局规划。比如"做一份市场调研报告"，ReAct 可能东查一个数据西查一个，没有结构。Plan-and-Execute 模式解决这个问题：先用一个 planner 生成完整的步骤计划，再用 executor 逐步执行。

两段式架构

Plan-and-Execute 把 agent 分成两个角色：

角色职责类比
Planner拿到任务后生成多步骤计划项目经理拆解任务
Executor按计划逐步执行，每步可用工具工程师执行每个子任务

关键设计：planner 和 executor 可以是同一个 LLM（不同 prompt），也可以是不同模型（planner 用强的、executor 用便宜的）。分离的好处是 planner 能"全局思考"不受工具调用细节干扰，executor 专注"做好每一步"。

Plan-and-Execute 的循环

def plan_and_execute(task, llm, tools, max_replans=3):
    """两段式：先规划，再执行，必要时重规划。"""
    for iteration in range(max_replans):
        # 阶段一：规划
        plan = planner(task, llm)  # 返回步骤列表
        # 例: ["查市场规模", "查竞品", "查趋势", "综合写报告"]

        results = []
        # 阶段二：逐步执行
        for step in plan:
            result = executor(step, tools, llm)  # 每步可能是个 ReAct 循环
            results.append(result)

        # 阶段三：检查是否需要重规划
        if needs_replan(task, results, llm):
            task = replan(task, results, llm)  # 调整任务描述
            continue
        else:
            return synthesize(task, results, llm)  # 综合所有结果
💡 要点
Plan-Execute 比 ReAct 强在：planner 有全局视野（先想清楚再动手）、executor 可以并行（多个独立步骤同时跑）、可以重规划（执行中发现计划不对就调整）。代价是多了一次"规划"的 LLM 调用。

何时用 Plan-Execute 而非 ReAct

简单任务（1-3 步能完成）用 ReAct 更快——规划反而增加开销。复杂任务（5 步以上、有依赖关系、需要综合）用 Plan-Execute 更可靠——先规划能避免"走偏"。一个经验法则：如果你自己都需要先列个清单才能完成的任务，agent 就该用 Plan-Execute。

规划器与执行器的架构细节

第一节给了 Plan-Execute 的骨架循环，这里深入两个角色的内部架构。生产级的 planner 和 executor 都不是单次 LLM 调用，而是各自有完整的子流程。理解这些子流程，才能在 planner 给出的计划"不靠谱"或 executor 执行"卡壳"时定位问题。

# 规划器与执行器的生产级架构
from dataclasses import dataclass, field
from typing import Literal

@dataclass
class Step:
    id: str
    description: str
    tool: str | None              # 要用的工具（None=纯推理步）
    depends_on: list[str] = field(default_factory=list)   # 依赖的前置步骤
    status: Literal["pending", "running", "done", "failed"] = "pending"
    result: dict | None = None

class ProductionPlanner:
    """规划器：不只是生成步骤，还要做依赖分析和可行性检查。"""
    def __init__(self, llm_strong, llm_cheap):
        self.llm_strong = llm_strong   # 规划用强模型（要全局思考）
        self.llm_cheap = llm_cheap     # 校验用便宜模型

    async def plan(self, task: str, available_tools: list) -> list[Step]:
        # 1) 任务分解：把大任务拆成可执行步骤
        steps = await self._decompose(task, available_tools)
        # 2) 依赖分析：标注哪些步骤互相依赖
        steps = self._analyze_dependencies(steps)
        # 3) 可行性检查：确认每步都有对应工具或可纯推理完成
        self._validate(steps, available_tools)
        # 4) 拓扑排序：确保执行顺序正确（被依赖的先执行）
        ordered = self._topo_sort(steps)
        return ordered

    async def _decompose(self, task, tools):
        prompt = f"""把任务分解为 3-8 个可执行步骤。
        任务: {task}
        可用工具: {[t.name for t in tools]}
        每步输出 JSON: {{id, description, tool}}"""
        resp = await self.llm_strong.invoke(prompt)
        return [Step(**s) for s in resp.steps]

    def _analyze_dependencies(self, steps):
        """用 LLM 或规则判断步骤间依赖，构建 DAG。"""
        for step in steps:
            # 简化：如果步骤描述里提到"上一步结果"，就依赖前一步
            for other in steps:
                if other.id != step.id and f"{other.id}结果" in step.description:
                    step.depends_on.append(other.id)
        return steps

    def _topo_sort(self, steps):
        """拓扑排序：被依赖的步骤排在前面。"""
        ordered, visited = [], set()
        def visit(s):
            if s.id in visited: return
            for dep in s.depends_on:
                dep_step = next(x for x in steps if x.id == dep)
                visit(dep_step)
            visited.add(s.id)
            ordered.append(s)
        for s in steps: visit(s)
        return ordered

class ProductionExecutor:
    """执行器：每步是个 mini-ReAct，支持重试和并行。"""
    def __init__(self, llm, tools):
        self.llm = llm
        self.tools = {t.name: t for t in tools}

    async def execute_step(self, step: Step, context: dict) -> Step:
        if step.status == "done":
            return step   # 已完成（并行去重）
        step.status = "running"
        try:
            if step.tool:
                tool = self.tools[step.tool]
                result = await self._run_with_react(step, tool, context)
            else:
                # 纯推理步：直接调 LLM
                result = await self.llm.invoke(step.description, context=context)
            step.result = result
            step.status = "done"
        except Exception as e:
            step.status = "failed"
            step.result = {"error": str(e)}
        return step

    async def _run_with_react(self, step, tool, context):
        """单步内的 ReAct 循环（观察-思考-行动）。"""
        for _ in range(5):   # 单步最多 5 轮 ReAct
            action = await self.llm.decide_action(step.description, context)
            obs = await tool.run(action.params)
            context["observation"] = obs
            if action.done:
                return obs
        return context.get("observation")

这段架构代码体现了 Plan-Execute 模式的两个关键进阶能力：① 依赖分析和拓扑排序让 executor 能自动判断"哪些步骤可以并行"——如果两个步骤互不依赖，就能同时执行，大幅降低总延迟；② 每个执行步骤内部是一个 mini-ReAct 循环——这意味着 Plan-Execute 不是"一次规划永远线性执行"，而是"宏观有计划、微观有灵活"，兼顾了全局视野和局部适应性。生产实现还要在 planner 里加"步骤数控制"（任务太大时分层规划）和"预算估算"（预估总 token 消耗防止超支）。

ReWOO 模式：推理不执行的解耦

ReWOO（Reasoning WithOut Observation）是 Plan-Execute 的一个优雅变体。传统 ReAct 每步都要把观察结果塞回 prompt，导致 token 膨胀。ReWOO 的思路是：规划阶段一次性生成所有步骤（用占位符引用未来结果），执行阶段批量填充，最后一次性综合——中间步骤不需要 LLM 介入。这大幅减少了 LLM 调用次数和 token 消耗。

# ReWOO：推理与执行彻底解耦
class ReWOOAgent:
    """规划时用占位符，执行时批量填充，综合时一次完成。"""
    def __init__(self, llm, tools):
        self.llm = llm
        self.tools = tools

    async def run(self, task: str):
        # 阶段一：一次性生成完整计划（用 #E1, #E2 引用未来结果）
        plan = await self._plan(task)
        # plan 示例:
        # #E1 = search_web("公司A 财报")
        # #E2 = search_web("公司A 新闻")
        # #E3 = analyze(#E1 + #E2)

        # 阶段二：批量执行所有步骤，收集证据
        evidence = {}
        for step in plan.steps:
            # 把引用替换成实际结果
            resolved = self._substitute(step.call, evidence)
            result = await self.tools[step.tool].run(resolved)
            evidence[step.id] = result   # evidence["#E1"] = ...

        # 阶段三：一次性综合（只有这一步需要 LLM 深度推理）
        answer = await self._solve(task, plan, evidence)
        return answer

    async def _plan(self, task):
        prompt = f"""为任务生成步骤计划。用 #E1, #E2... 标记每步，
        后续步骤可引用前面的标记。不要执行，只规划。
        任务: {task}
        可用工具: {list(self.tools.keys())}
        输出格式:
        #E1 = tool("args")
        #E2 = tool("#E1 的相关参数")"""
        resp = await self.llm.invoke(prompt)
        return self._parse_plan(resp)

    def _substitute(self, call_text, evidence):
        """把 #E1 替换成 evidence 里的实际结果。"""
        for eid, result in evidence.items():
            call_text = call_text.replace(eid, str(result))
        return call_text

    async def _solve(self, task, plan, evidence):
        prompt = f"""根据证据回答任务。
        任务: {task}
        计划: {plan}
        证据: {evidence}
        请综合所有证据给出最终答案。"""
        return await self.llm.invoke(prompt)

# 对比 ReAct vs ReWOO 的 LLM 调用次数（5步任务）：
# ReAct:  每步都调 LLM = 5 次决策调用 + 1 次综合 = 6 次
# ReWOO:  1 次规划 + 1 次综合 = 2 次（省 67%）
# token:  ReAct 把全部历史塞 prompt，ReWOO 各步独立（省更多）

ReWOO 的精妙在于它识别出了 ReAct 的一个本质浪费：中间步骤的"思考"其实大部分可以预判——规划阶段就能知道"E3 要用 E1 和 E2 的结果"，没必要等 E1、E2 执行完再调 LLM 决定 E3 怎么做。通过用占位符（#E1）把这种依赖关系在规划时就编码好，执行阶段变成了纯粹的"工具批量调用"，把昂贵的 LLM 推理压缩到首尾两次。代价是：如果执行中发现某步结果和预期差很远（比如 E1 搜索没返回有用信息），ReWOO 无法像 ReAct 那样灵活调整——它只能用差的结果硬着头皮综合。所以 ReWOO 适合"步骤可预测、结果相对确定"的任务，不适合"高度探索性"的任务。

💡 要点
ReWOO 的占位符思想（#E1 引用未来结果）影响了后续很多框架。LangChain 的 Plan-and-Execute、甚至 OpenAI 的 function calling 里的参数互引用，都有这个影子。理解"用符号引用解耦规划和执行"这个思想，比记住 ReWOO 这个名字更重要。

任务分解策略：如何把大任务拆好

Plan-Execute 的成败，70% 取决于 planner 分解任务的质量。分解太粗（2 步）则 executor 压力大、易失败；分解太细（20 步）则规划本身耗时长、易出错。下面总结几种经过验证的分解策略，以及它们的适用边界。

分解策略思路适用任务风险
按阶段分解调研→分析→写作→审核流程明确的任务阶段内仍复杂
按维度分解查财务/查新闻/查竞品多源信息聚合维度遗漏
按子问题分解把大问题拆成子问题研究类任务子问题有依赖
分层分解先拆大块，大块再拆小块超大型任务规划成本高
按工具分解每步对应一个工具调用工具链明确的任务工具间逻辑断裂

实践中最常用的是"按阶段"和"按维度"的混合：先用阶段分大框架（调研→分析→输出），每个阶段内部再按维度拆（调研阶段并行查多个数据源）。关键是控制单步复杂度——一个步骤应该"一个工具调用 + 一次简单推理"就能完成，如果某步需要多次工具调用，说明它该被进一步拆分。给 planner 的 prompt 里加一个约束"每步应在 30 秒内可完成"，能有效避免规划出过大的步骤。最后，所有分解策略都受益于"分解后让 LLM 自检一遍"——把分解结果回灌给 LLM 问"这个计划有什么遗漏"，往往能补上漏掉的步骤。

Plan 的质量评估与自适应重规划

Plan-and-Execute 的成败 80% 取决于 Plan 的质量。一个差的 plan（步骤遗漏、顺序错误、依赖搞反）会让 Execute 阶段越执行越偏，最终结果南辕北辙。所以工程上必须能评估 plan 质量，并在发现 plan 不靠谱时及时重规划，而不是硬着头皮执行完。

Plan 质量的评估维度有四个。完备性：是否覆盖了达成目标的所有必要步骤（有没有漏掉关键环节）。正确性：步骤之间的依赖顺序是否正确（不能先部署再测试）。可执行性：每步是否明确到能被 Execute 执行（不能是"处理一下数据"这种模糊描述）。充分性：步骤是否冗余或不足（太多浪费、太少不够）。评估可以用 LLM-as-judge 自动打分，也可以维护"plan 模板库"做相似度匹配——类似任务的成熟 plan 作为参照。

自适应重规划是 Plan-Execute 区别于纯 ReAct 的核心优势。Execute 过程中发现某步失败、或得到的新信息表明原 plan 有误时，不应硬撑，而是回到 Planner 重新规划剩余部分。关键是"重规划是增量的而非全推倒"——已完成的有效步骤保留，只重新规划失败点之后的步骤，避免重复劳动和上下文爆炸。判断何时触发重规划的信号：某步连续失败 N 次、Observation 与 plan 预期严重不符、用户中途修改目标。

评估维度检查问题不达标的后果评估方式
完备性必要步骤是否齐全结果不完整LLM judge / 模板对照
正确性依赖顺序对不对执行报错或返工依赖图校验
可执行性每步是否够明确Executor 卡住Executor 能否解析
充分性步骤不多不少冗余或不足对标历史 plan
鲁棒性有无失败兜底一步错全盘崩检查异常分支
# 带自适应重规划的 Plan-Execute
class PlanExecuteAgent:
    def __init__(self, planner, executor, max_replans=3):
        self.planner = planner
        self.executor = executor
        self.max_replans = max_replans

    def run(self, goal):
        plan = self.planner.plan(goal)        # 初始 plan
        results = {}
        replan_count = 0
        i = 0
        while i < len(plan):
            step = plan[i]
            try:
                result = self.executor.execute(step, results)
                results[step["id"]] = result
                # 检查结果是否符合 plan 预期
                if not self._matches_expectation(result, step):
                    raise PlanMismatch(step, result)
                i += 1
            except (StepFailed, PlanMismatch) as e:
                replan_count += 1
                if replan_count > self.max_replans:
                    return {"status":"failed", "reason":"重规划次数耗尽", "partial":results}
                # 增量重规划：保留已完成，只重规划剩余
                remaining_goal = self._derive_remaining(goal, results, plan[i:])
                new_tail = self.planner.replan(remaining_goal, failed_step=step, error=e)
                plan = plan[:i] + new_tail    # 替换剩余部分
                print(f"重规划（第{replan_count}次）：从步骤 {step['id']} 起重新规划 {len(new_tail)} 步")
        return {"status":"ok", "results":results}

    def _matches_expectation(self, result, step):
        """结果是否符合 plan 的预期（如某步应产出非空文件）"""
        check = step.get("expect")
        if check == "non_empty": return bool(result)
        if check == "valid_json":
            import json
            try: json.loads(result); return True
            except: return False
        return True   # 无明确预期则默认通过

Planner 与 Executor 的解耦设计

Plan-Execute 模式的一个设计要点是 Planner 和 Executor 的解耦。耦合在一起（同一个 prompt 既规划又执行）会丧失这种模式的核心优势——规划的稳定性和执行的灵活性。解耦后，Planner 可以用更强的模型（规划需要全局视野，值得用贵模型），Executor 可以用更快更便宜的模型（单步执行是局部任务），还能并行化。

解耦的关键是清晰的步骤契约。Planner 产出的每一步必须是一个自包含的、可独立执行的任务描述：明确的输入（依赖哪些前序步骤的产出）、明确的动作（调什么工具/做什么操作）、明确的输出（产出什么、格式如何）。Executor 拿到一个步骤后，不需要知道全局 plan，只看这一步的契约就能执行。这种"步骤即微任务"的设计让 Executor 可以并行执行无依赖的步骤、可以分布式调度、可以重试单步而不影响全局。

解耦还带来可替换性：同一个 plan 可以用不同的 Executor 执行（人工执行、自动化执行、混合执行）。这在"人在回路"场景特别有用——高风险步骤路由给人工 Executor，低风险步骤自动执行，路由策略不侵入 plan 本身。

设计维度耦合（一体式）解耦（分离式）解耦的收益
模型选择同一模型做规划+执行Planner 用强模型，Executor 用快模型成本与质量平衡
并行性串行（规划完才执行）无依赖步骤可并行延迟降低
可重试失败要整体重来单步重试韧性提升
可替换执行方式写死Executor 可换（人/自动）支持人在回路
可测试难单独测plan 和执行各自测质量易保障
# 解耦的步骤契约 + 可替换 Executor
from dataclasses import dataclass, field
from typing import Any

@dataclass
class Step:
    id: str
    action: str                  # 工具名 / 操作类型
    inputs: dict                 # 依赖前序步骤的产出：{"data": "@step3.output"}
    expect: str = "any"          # 输出预期：non_empty / valid_json / any
    human: bool = False          # 是否需人工执行
    output: Any = field(default=None)

class DecoupledExecutor:
    def __init__(self, auto_executor, human_executor):
        self.auto = auto_executor
        self.human = human_executor

    def execute(self, step, results):
        # 解析输入：把 "@step3.output" 替换为实际前序产出
        resolved = self._resolve_inputs(step.inputs, results)
        # 按步骤属性路由：人工步骤走人工，其余自动
        if step.human:
            return self.human.execute(step, resolved)
        return self.auto.execute(step.action, resolved)

    def _resolve_inputs(self, inputs, results):
        resolved = {}
        for k, v in inputs.items():
            if isinstance(v, str) and v.startswith("@"):
                # @step3.output → results["step3"]["output"]
                ref = v[1:].split(".", 1)
                resolved[k] = results[ref[0]][ref[1]] if len(ref)>1 else results[ref[0]]
            else:
                resolved[k] = v
        return resolved

# 同一个 plan，不同执行模式：
# 自动模式：所有 step.human=False，全自动跑完
# 人在回路：高风险步骤 human=True，路由给人工确认后继续
# 关键：plan 不变，只换 Executor，实现"自动/人工"灵活切换

本章小结

Plan-and-Execute 把 agent 的"想"和"做"分离：planner 全局规划、executor 逐步执行、必要时重规划。它适合复杂、多步骤、需要综合的任务。BabyAGI、Plan-and-Solve 都是这个模式的实例。
