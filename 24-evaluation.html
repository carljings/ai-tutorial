AI 评估方法论

怎么知道你的模型真的变好了——评估比训练更难

深度进阶★★★

为什么评估是 LLM 最难的部分

传统机器学习的评估很简单：你有一个测试集，算准确率/F1，分数高就是好。LLM 评估完全不同——输出是开放式文本，"好"没有标准答案，同一个回答有人觉得好有人觉得差。这就导致了一个尴尬局面：**你的模型可能确实变好了，但你没有可靠的指标来证明它**。没有好的评估，就没有可靠的迭代——你不知道改动是帮助还是伤害。

💡 要点
评估的核心困境：**LLM 的输出是开放的、主观的、多维的，没有单一指标能衡量"好不好"**。整个 LLM 评估领域就是在解决"如何用工程手段量化一个本质上主观的东西"。

评估的三大范式

范式原理优点缺点
标准基准（benchmark）固定测试集+自动评分可复现、快、便宜过拟合/数据泄漏
LLM-as-judge用 GPT-4 给回答打分灵活、接近人类 judge 自身有偏
人工评估人类标注员打分最接近真实贵、慢、不可复现

标准基准：MMLU / HumanEval / MT-Bench

最常用的 LLM 基准：MMLU（57 学科多选题，测知识广度）、HumanEval（164 编程题，测代码能力）、GSM8K（小学数学应用题，测推理）。这些基准能快速横向对比模型，但有两个致命问题：

① **数据泄漏**——测试题可能混进了训练数据（尤其 GPT-4/Claude 的训练集不公开），分数虚高。解决办法是用动态生成的题（如 LiveCodeArena 实时出题）或私有测试集。② **过拟合**——模型专门针对基准优化，基准分数涨了但真实能力没涨。这就是"Goodhart 定律"：当一个指标成为目标，它就不再是好指标。

LLM-as-judge：用 AI 评估 AI

当前最流行的评估方式：用 GPT-4 当"裁判"，给两个模型的回答打分。优点是灵活（可以评估任何维度）、便宜（比人工快 100 倍）、接近人类判断（GPT-4 和人类的相关性约 80%+）。但有几个已知偏差要注意：

LLM-as-judge 的偏差表现缓解
位置偏差倾向选第一个/第二个随机化顺序，取两次平均
长度偏差倾向选更长的回答在 prompt 里强调"长度不是质量"
自我偏好GPT-4 偏好 GPT-4 风格用多个不同模型交叉评估
能力上限judge 比被评估的弱，评不准judge 必须是当前最强模型
# LLM-as-judge 的典型 prompt（简化版）
JUDGE_PROMPT = """你是一个严格的评估员。
请比较以下两个对同一问题的回答，按质量打分。

问题: {question}
回答 A: {response_a}
回答 B: {response_b}

评分维度: 准确性、完整性、清晰度、安全性
请输出 JSON: {{"winner": "A"|"B"|"tie", "reason": "..."}}
"""

# 关键控制：
# 1. 随机化 A/B 顺序，消除位置偏差
# 2. 用 GPT-4 级 judge（弱模型评判强模型会失准）
# 3. 多次评估取多数（单次评估方差大）

人工评估：最后的真相

当自动评估给出矛盾结论时，必须回到人工评估。人工评估的关键是**标注规范**——"好"必须被精确定义（如"准确性：回答的事实是否正确？1-5 分"），且要有多个标注员交叉检验（计算 IAA）。RLHF 训练时用的人工偏好数据，本身就是一种高质量的人工评估。

RAG 专项评估：RAGAS

RAG 系统的评估比通用 LLM 更复杂，因为它有两个环节（检索+生成）都可能出错。RAGAS 是当前最流行的 RAG 评估框架，它拆成三个维度：

维度衡量什么怎么算
Faithfulness（忠实度）回答是否基于检索到的内容（不编造）检查回答的每个claim能否在检索内容里找到支撑
Answer Relevancy（回答相关性）回答是否切题用 LLM 判断回答和问题的相关性
Context Precision（检索精度）检索到的内容是否真的相关检查检索的chunk是否和问题相关

RAGAS 的精髓是把"RAG 好不好"拆成了可定位的三个数字——如果忠实度低，是生成环节在编造；如果检索精度低，是检索环节召回差；如果回答相关性低，是理解环节有问题。这种"分环节归因"让 RAG 调优有了明确方向。

评估的工程化：持续评估流水线

评估不是"训完评一次"，而是要嵌入开发流程的持续环节。一个成熟的评估流水线：

持续评估流水线

代码提交 → 自动触发 → 跑基准(MMLU/HumanEval 子集)
                      → 跑 LLM-as-judge(100 条 golden set)
                      → 跑 RAGAS(如果改了 RAG)
                      → 对比上次结果（回归检测）
                      → 任何指标下降 >2% → 阻断合并
                      → 全部通过 → 人工抽检 50 条
                      → 通过 → 上线（金丝雀发布）

关键：golden set 是你的"回归测试集"，
      每次改动都跑，确保不退步。
💡 要点
评估的终极建议：**建立一个固定的 golden set（100-500 条精选问题+期望特征），每次模型/prompt 改动都跑一遍**。这不是为了得到一个绝对分数，而是为了做"回归检测"——确保你的改动没有引入退步。没有 golden set 回归检测的迭代，就是盲人摸象。

RAGAS 完整评估代码实战

前面讲了 RAGAS 的三个维度，但没有给可运行代码。这里给出一份能直接接到现有 RAG 管线上的完整实现，并且补上第四个常被忽略的维度 Context Recall（召回率）——衡量「该召回的相关内容是否都召回了」。四个维度齐全后，RAG 调优才真正做到可定位。

ragas_eval.py —— 完整的 RAG 四维评估
"""
ragas_eval.py —— 用 RAGAS 框架评估 RAG 系统
依赖：pip install ragas datasets
四个维度：faithfulness / answer_relevancy / context_precision / context_recall
"""
from datasets import Dataset
from ragas import evaluate
from ragas.metrics import (
    faithfulness, answer_relevancy, context_precision, context_recall
)

def build_eval_dataset(rag_pipeline, golden_qa):
    """golden_qa: [(question, reference_answer), ...]
    用 RAG 跑出答案和检索上下文，组装成 RAGAS 需要的格式"""
    rows = []
    for q, ref in golden_qa:
        answer, contexts = rag_pipeline.query(q)   # contexts 是检索到的 chunk 列表
        rows.append({
            "question": q,
            "answer": answer,
            "contexts": contexts,
            "ground_truth": ref,            # reference_answer 用于 context_recall
        })
    return Dataset.from_list(rows)

def run_ragas(dataset, llm=None, embeddings=None):
    """跑四个指标，返回 dict。judge 模型建议用最强模型"""
    return evaluate(
        dataset,
        metrics=[faithfulness, answer_relevancy,
                 context_precision, context_recall],
        llm=llm, embeddings=embeddings,
    )

# 把分数映射到「该改哪一环」，方便定位
def diagnose(scores):
    issues = []
    if scores["faithfulness"] < 0.8:
        issues.append(("生成环节在编造", "加强 prompt 里的『只基于上下文回答』约束，或加 citation"))
    if scores["context_precision"] < 0.7:
        issues.append(("检索噪音多", "调 top_k、加 rerank、缩小 chunk"))
    if scores["context_recall"] < 0.7:
        issues.append(("该召回的没召回", "换更大 embedding 模型、改善分块、补语料"))
    if scores["answer_relevancy"] < 0.8:
        issues.append(("答非所问", "加 query 改写、检查系统 prompt 是否偏题"))
    return issues or [("各维度均达标", "可继续扩 golden set 找坏 case")]

注意 RAGAS 的指标本身要调 LLM，成本不低——评估 100 条通常要花几十次 LLM 调用。生产里一般把 golden set 控制在 100-200 条，并在 CI 里只在 RAG 相关改动时才跑全量。

A/B 测试框架：线上才是终极裁判

离线评估再准，也无法覆盖线上真实用户问法。当一个版本离线指标持平甚至略低，但产品直觉觉得更好时，只有 A/B 测试能给出答案。下面是一个最小可用的 A/B 测试框架，包含流量分配、指标收集和统计显著性判断。

ab_test.py —— 线上 A/B 测试骨架
import hashlib, time
from dataclasses import dataclass, field
from typing import Callable

@dataclass
class Variant:
    name: str
    run: Callable                 # 实际推理函数
    impressions: int = 0
    thumbs_up: int = 0            # 用户点赞/复制/继续追问等正向信号
    thumbs_down: int = 0

class ABTest:
    def __init__(self, control: Variant, treatment: Variant,
                 traffic=0.5, min_sample=500):
        self.control = control
        self.treatment = treatment
        self.traffic = traffic              # treatment 流量占比
        self.min_sample = min_sample

    def assign(self, user_id):
        """用 user_id 哈希做稳定分流，同一用户始终进同一组"""
        h = int(hashlib.md5(user_id.encode()).hexdigest(), 16) % 1000 / 1000
        return self.treatment if h < self.traffic else self.control

    def record(self, variant, positive: bool):
        variant.impressions += 1
        if positive:
            variant.thumbs_up += 1
        else:
            variant.thumbs_down += 1

    def significant(self):
        """简化版 Z 检验：两组好评率差异是否显著。样本不足返回 None"""
        if self.control.impressions < self.min_sample or \
           self.treatment.impressions < self.min_sample:
            return None
        def rate(v): return v.thumbs_up / max(v.impressions, 1)
        p1, p2 = rate(self.control), rate(self.treatment)
        n1, n2 = self.control.impressions, self.treatment.impressions
        pooled = (self.control.thumbs_up + self.treatment.thumbs_up) / (n1 + n2)
        se = (pooled * (1 - pooled) * (1/n1 + 1/n2)) ** 0.5
        z = (p2 - p1) / se if se > 0 else 0
        return {"control_rate": p1, "treatment_rate": p2,
                "z": z, "significant": abs(z) > 1.96}

关键设计点：① 用 user_id 哈希做稳定分流，避免同一用户在两组间反复横跳污染数据；② 设 min_sample 下限，样本不足时不算显著性，避免小样本噪声误导决策；③ 正向信号优先用「用户复制回答」「继续追问」这类隐式反馈，比显式点赞覆盖率高得多。

Golden Set 构建：从 0 到 500 条

golden set 是评估的基石，但「怎么挑这 500 条」决定了回归检测的有效性。下面是经验证有效的构成配比，以及一条不可省的「毒丸」策略。

类别占比来源作用
核心功能 case40%产品 PRD / 主流程保证主链路不退化
历史 bad case30%线上工单 / 用户反馈防止已修问题复发
边界与异常20%红队 + 长尾输入覆盖空输入、超长、注入
对抗毒丸10%故意构造的陷阱题检测模型是否「死记」答案

「毒丸」是常被忽略的一环：在 golden set 里放几条表述相近但答案相反的题，能发现模型是不是在背答案而不是真懂。如果版本升级后毒丸答对率反而下降，说明模型在过拟合 golden set，必须立刻扩集。

golden_set.py —— golden set 的版本化与回归
import json, hashlib
from pathlib import Path

class GoldenSet:
    def __init__(self, path="golden_set.jsonl"):
        self.path = Path(path)
        self.cases = [json.loads(l) for l in self.path.read_text().splitlines()]
        # 记录内容指纹，CI 里用来检测 golden set 是否被悄悄改过
        self.fingerprint = hashlib.md5(
            self.path.read_bytes()).hexdigest()[:8]

    def regress_against(self, runner, last_scores, tolerance=0.02):
        """跑全量 case，与上次分数对比，任一指标降幅 > tolerance 即报回归"""
        cur = {}
        for c in self.cases:
            for metric, value in runner(c).items():
                cur.setdefault(metric, []).append(value)
        cur_avg = {m: sum(v)/len(v) for m, v in cur.items()}
        regressions = []
        for m, score in cur_avg.items():
            if m in last_scores and last_scores[m] - score > tolerance:
                regressions.append((m, last_scores[m], score))
        return {"scores": cur_avg, "regressions": regressions,
                "fingerprint": self.fingerprint}
⚠ golden set 的两个纪律

第一，golden set 必须版本化——内容变了指纹就变，CI 要拒绝指纹与代码 commit 不匹配的运行。第二，禁止为了「让分涨」而删 bad case，这是最常见的自欺欺人，会让回归检测彻底失效。正确做法是 bad case 永久保留，新增 case 才允许扩。

本章小结

LLM 评估是最难的工程问题，因为输出是开放的、主观的。三大范式：标准基准（快但易过拟合）、LLM-as-judge（灵活但有偏差，注意位置/长度/自我偏好偏差）、人工评估（最准但贵）。RAG 专项用 RAGAS 拆成忠实度/相关性/检索精度（再加召回率）四个可定位维度，本章给出了完整可运行的 RAGAS 评估代码。线上评估用 A/B 测试框架，靠稳定分流与隐式反馈拿真实裁判。工程化核心是建立版本化的 golden set 做持续回归检测——没有评估就没有可靠迭代。
