Codex 整体架构
Rust 内核 + TypeScript 前端的 queue-pair 协议

OpenAI 的 Codex CLI 是一个有意思的工程样本：它的核心引擎 codex-rs 用 Rust 编写，负责所有「重活」——LLM 调用、沙箱执行、工具编排；而前端（终端 UI 或 IDE 集成）可以用 TypeScript 或其他语言编写，通过一个基于 queue-pair 的 Op/Event 协议和内核通信。这种「Rust 内核 + 语言无关前端」的架构，和 OpenCode 的「全 TypeScript」形成了鲜明对比。本章我们拆解 Codex 的进程边界和协议设计。

阅读约 20 分钟 难度 ★★★★★

★ 一、为什么内核要写 Rust

Codex 的核心是 codex-rs，一个 Rust crate。选择 Rust 而不是 Node/Python，背后是几项硬需求。

第一，沙箱执行需要系统调用。Codex 在 macOS 上用 seatbelt（sandbox-exec）、在 Linux 上用 Landlock 来隔离命令执行，这些都涉及底层 syscall 和进程管理。Rust 对 FFI 和系统调用有原生支持，而 Node.js 调用底层 syscall 要走 native addon，体验割裂。

第二，并发安全。Agent 内核要同时处理 LLM 流式响应、多个工具执行、用户输入、UI 更新。Rust 的所有权和 Send/Sync 机制在编译期就排除了数据竞争，这对一个高并发的内核是巨大保障。用 Go 也能做并发，但 GC 停顿在实时流式场景下可能造成卡顿。

第三，单文件分发。Rust 编译出的是单个原生二进制，没有运行时依赖。用户安装 Codex 不需要先装 Node 或 Python，下载一个可执行文件就能跑。这对 CLI 工具的传播极其重要。

第四，性能确定性。无 GC 意味着延迟可预测，token 计数、消息序列化这些高频操作的耗时稳定，不会出现偶尔的百毫秒级停顿。

★ 二、内核与前端：一条进程边界

Codex 的运行模型是 一个 codex-rs 内核进程 + 一个或多个前端。前端通过子进程的 stdin/stdout（或一个本地 socket）和内核通信。前端的实现语言不重要，只要它会说 Op/Event 协议。

官方默认的前端是终端 TUI（用 Rust 的 ratatui 库写，和内核编译在一起或作为子命令）。但也支持外部前端模式：VS Code 扩展、Web 界面都可以作为前端连上来。这种架构让 Codex 内核成了一个「无头 Agent 引擎」，可以被任何界面驱动。

★ 三、Op/Event 协议：queue-pair 的精髓

内核和前端的通信是一对队列：前端往内核发 Op（操作请求），内核往前端发 Event（事件通知）。两者都是 JSON 序列化的消息，通过 stdin/stdout 传输。关键设计是：Op 和 Event 的收发是完全异步的——前端可以连续发多个 Op，内核也可以连续吐多个 Event，彼此不阻塞。

// codex-rs/src/protocol.rs（概念示意）
use serde::{Serialize, Deserialize};

/// 前端 -> 内核 的操作请求
#[derive(Serialize, Deserialize)]
#[serde(tag = "type")]
pub enum Op {
    /// 用户输入了一条消息
    UserInput { text: String },
    /// 中断当前执行
    Interrupt,
    /// 审批一个待确认的工具调用
    Approve { call_id: String, decision: ApprovalDecision },
    /// 请求当前会话状态
    GetSession,
}

/// 内核 -> 前端 的事件通知
#[derive(Serialize, Deserialize)]
#[serde(tag = "type")]
pub enum Event {
    /// Agent 开始思考
    AgentStarted,
    /// LLM 流式输出的一个 token 片段
    OutputDelta { text: String },
    /// Agent 要执行一个命令，需要审批
    ExecApprovalRequest { call_id: String, command: String, cwd: String },
    /// 命令执行完成
    ExecComplete { call_id: String, exit_code: i32, output: String },
    /// 一轮 Agent 循环结束
    TurnComplete,
    /// 发生错误
    Error { message: String },
}

这种基于 enum（tagged union）的消息定义是 Rust serde 的典型用法。每条消息用一个 type 字段区分种类，前端反序列化时根据 type 走不同分支。这比定义一堆独立的 message 类型更紧凑，也更容易演进——加一种新 Event 只需在 enum 里加一个 variant，旧前端忽略不认识的 type 即可向后兼容。

★ 四、内核里的两条异步循环

codex-rs 内部用 tokio（Rust 的 async 运行时）驱动两条核心循环：Op 接收循环 和 Agent 主循环。它们通过 channel（mpsc）解耦。

// codex-rs/src/main.rs（概念示意）
use tokio::sync::mpsc;

#[tokio::main]
async fn main() {
    let (op_tx, op_rx) = mpsc::channel::<Op>(64);
    let (event_tx, event_rx) = mpsc::channel::<Event>(64);

    // 任务 1：从 stdin 读 Op，塞进 channel
    let op_reader = tokio::spawn(async move {
        let stdin = tokio::io::stdin();
        let mut reader = BufReader::new(stdin);
        loop {
            let line = read_line(&mut reader).await;
            match line {
                Some(json) => {
                    if let Ok(op) = serde_json::from_str::<Op>(&json) {
                        op_tx.send(op).await.unwrap();
                    }
                }
                None => break,  // stdin 关闭
            }
        }
    });

    // 任务 2：从 event channel 读 Event，写到 stdout
    let event_writer = tokio::spawn(async move {
        let stdout = tokio::io::stdout();
        let mut writer = BufWriter::new(stdout);
        while let Some(event) = event_rx.recv().await {
            let json = serde_json::to_string(&event).unwrap();
            writeln!(writer, "{}", json).await.unwrap();
            writer.flush().await.unwrap();
        }
    });

    // 主任务：跑 Agent 循环，消费 Op，产生 Event
    run_agent_loop(op_rx, event_tx).await;
}

这段代码展示了 Codex 内核的骨架：stdin 和 stdout 是和前端通信的边界，内部用 mpsc channel 把「IO」和「业务」彻底解耦。Agent 主循环只需要从 op_rx 拿 Op、往 event_tx 发 Event，完全不关心消息是怎么从网络或管道来的。这种解耦让内核可以同时对接多个前端（每个前端一组 channel），也为后面的并行 Agent 执行打下了基础。

★ 五、Agent 主循环：Op 驱动 Event 产生

Agent 主循环是 codex-rs 的心脏。它不是简单的「收一条 Op 跑一次 LLM」，而是一个状态机：等待用户输入 → 调 LLM → 处理 tool_call（可能要审批）→ 把结果喂回 LLM → 直到 LLM 不再请求工具，一轮结束。

// codex-rs/src/agent_loop.rs（概念示意）
async fn run_agent_loop(mut op_rx: Receiver<Op>, event_tx: Sender<Event>) {
    let mut session = Session::new();

    while let Some(op) = op_rx.recv().await {
        match op {
            Op::UserInput { text } => {
                session.add_message(Message::user(text));
                // 进入一轮 Agent 循环
                run_turn(&mut session, &event_tx).await;
            }
            Op::Interrupt => {
                session.abort_current();
                event_tx.send(Event::Error { message: "已中断".into() }).await.ok();
            }
            Op::Approve { call_id, decision } => {
                session.resolve_approval(&call_id, decision).await;
            }
            _ => {}
        }
    }
}

async fn run_turn(session: &mut Session, event_tx: &Sender<Event>) {
    event_tx.send(Event::AgentStarted).await.ok();
    loop {
        // 1. 调 LLM，流式接收
        let response = session.call_model_streaming(|delta| {
            event_tx.blocking_send(Event::OutputDelta { text: delta }).ok();
        }).await;

        // 2. 如果 LLM 请求了工具调用
        if let Some(tool_calls) = response.tool_calls {
            for call in tool_calls {
                // 3. 需要审批的先请求
                let approved = request_approval(&event_tx, &call).await;
                if !approved { continue; }
                // 4. 在沙箱里执行
                let result = session.execute_sandboxed(&call).await;
                event_tx.send(Event::ExecComplete { .. }).await.ok();
                session.add_message(Message::tool_result(call.id, result));
            }
            // 继续循环，把工具结果喂回 LLM
        } else {
            // 5. LLM 没请求工具，本轮结束
            event_tx.send(Event::TurnComplete).await.ok();
            break;
        }
    }
}

这段伪代码揭示了几个关键点。第一，审批是异步的：request_approval 会发一个 ExecApprovalRequest 事件，然后等待前端通过 Op::Approve 回复——这期间 Agent 循环是挂起的，但不阻塞 op_rx（因为审批是通过 session 内部的 oneshot channel 回调的）。第二，沙箱执行是内核的职责，前端完全不参与命令执行，只看到结果事件。第三，中断是协作式的：Op::Interrupt 设置一个标志位，LLM 调用和工具执行会检查它，不是硬 kill。

★ 六、配置层：profiles 与分层覆盖

Codex 的配置用 TOML 写，支持 profiles（命名配置集）和分层覆盖（全局 → 项目 → 命令行参数）。这让一套二进制能适配多种使用场景——比如「工作」profile 用强模型 + 严格沙箱，「实验」profile 用便宜模型 + 宽松沙箱。我们会在第 161 章专门讲配置系统，这里只点出它的存在。

★ 七、与 OpenCode 架构的对比

把 Codex 和 OpenCode 放在一起看，能看出两种截然不同的工程哲学。

OpenCode 选择「全栈 TypeScript + Bun」，优势是开发速度快、contributor 门槛低（前端开发者也能改内核）、代码可读性高。代价是性能受限于 V8/JSCore 的 GC，且沙箱等底层能力要依赖外部工具或 native addon。

Codex 选择「Rust 内核 + 协议化前端」，优势是性能强、沙箱原生、单二进制分发、内核可被任意语言的前端复用。代价是 contributor 门槛高（要会 Rust）、开发迭代慢（编译时间长）、调试 Rust 的异步代码比调试 TypeScript 难。

两种选择没有绝对优劣，取决于项目目标。OpenCode 像一个「快速演进的开源产品」，Codex 像一个「工业级的引擎」。理解这种差异，有助于你在选型或自研 Agent 时做出合适的架构决策。

★ 本章小结

Codex 的架构是「内核-前端分离」的典范。要点回顾：

  • Rust 内核：codex-rs 用 Rust 写，为了沙箱 syscall、并发安全、单二进制分发、无 GC 的延迟确定性。
  • 进程边界：内核是一个进程，前端通过 stdin/stdout 或 socket 连接，语言无关。
  • Op/Event queue-pair：前端发 Op（请求），内核发 Event（通知），用 serde tagged enum 定义，异步收发不阻塞。
  • 两条 tokio 任务：Op 接收循环和 Event 发送循环通过 mpsc channel 和 Agent 主循环解耦。
  • Agent 主循环：状态机式驱动，审批异步等待、沙箱内核执行、中断协作式。
  • 与 OpenCode 对比：全 TypeScript vs Rust 内核，快速迭代 vs 工业级，是两种合理的工程选择。
