重排序 Rerank
cross-encoder vs bi-encoder 与 top-N 策略

检索召回的 top-50，质量往往参差不齐——有的语义相关，有的只是关键词巧合。重排序（Rerank）用一个更精确但更慢的模型，对这 50 个重新打分，把真正相关的推到前面。这一步通常能让最终答案准确率提升 10-20 个百分点，是 RAG 里 ROI 最高的环节之一。本文拆解 cross-encoder 和 bi-encoder 的本质区别、bge-reranker 的实操、以及 top-N 怎么选。

阅读约 16 分钟 难度 ★★★

★ cross-encoder vs bi-encoder：精度与速度的根本权衡

embedding 模型（用于检索）是 bi-encoder（双塔）：query 和 document 分别独立编码成向量，再用余弦相似度比较。因为 document 可以离线编码好存进向量库，查询时只需要编码 query + 一次向量检索，速度极快（毫秒级，可扛上万 QPS）。代价是 query 和 document 的交互发生在"最后的相似度计算"——只有点积这一步，没有更深的信息流动，精度有天花板。

rerank 模型是 cross-encoder（交叉编码）：把 query 和 document 拼成一个序列 [CLS] query [SEP] document [SEP]，喂给一个 transformer，让它们在 self-attention 里充分交互，最后输出一个相关性分数。这种深度交互让 cross-encoder 的精度远高于 bi-encoder（在 MS MARCO 上，cross-encoder 的 MRR 通常比 bi-encoder 高 10-20 个点）。

代价是速度：cross-encoder 不能离线编码 document，每次都要把 (query, document) 一起喂进模型算一次 forward。所以它不能用于全库检索（1 亿次 forward 要几小时），只能用于对少量候选重排序。这就是经典的"先 bi-encoder 粗召 → 再 cross-encoder 精排"两阶段架构。

两阶段架构
┌─────────────────┐    top-50    ┌─────────────────┐   top-5    ┌─────────┐
│ bi-encoder 检索  │ ───────────▶ │ cross-encoder   │ ─────────▶ │ 喂给 LLM │
│ (毫秒, 全库)     │              │ 重排序 (几十ms) │            │         │
└─────────────────┘              └─────────────────┘            └─────────┘
快但不准                            慢但准                          最终答案

为什么 cross-encoder 更准：一个直觉

bi-encoder 把 document 压成一个 768 维向量，信息损失巨大——一个 500 字的文档被压成 768 个数字，查询里某个关键词和文档里某个具体段落的细粒度关系早就丢了。cross-encoder 不压缩，query 的每个 token 直接和 document 的每个 token 做 attention，能捕捉"这个词和那段话在语义上对应"这种细粒度关系。用类比：bi-encoder 像看摘要判断相关性，cross-encoder 像读全文判断——前者快但粗，后者慢但准。

★ bge-reranker 实操

BAAI 的 bge-reranker 系列是中文场景最受欢迎的开源 rerank 模型，有 base/large/重排多种规格。我们用 FlagEmbedding 走一遍。

# pip install FlagEmbedding
from FlagEmbedding import FlagReranker
import time

# 加载 reranker（中文场景用 bge-reranker-large，英文用 bge-reranker-base）
reranker = FlagReranker(
    "BAAI/bge-reranker-large",
    use_fp16=True                  # 半精度，速度快一倍，精度损失可忽略
)

# 1. 对一批 (query, document) 对打分
query = "怎么排查 Java 内存泄漏"
documents = [
    "Java 内存泄漏通常用 jmap 查看堆内存占用，用 MAT 分析对象引用链。",
    "Python 的内存管理是自动的，一般不需要手动释放。",
    "OutOfMemoryError 是 Java 常见错误，可以通过调整 JVM 堆大小 -Xmx 解决。",
    "JavaScript 的垃圾回收机制和 Java 类似，都是标记清除。",
    "使用 VisualVM 可以实时监控 JVM 的内存和线程状态。",
]

pairs = [[query, doc] for doc in documents]
t0 = time.time()
scores = reranker.compute_score(pairs, normalize=True)   # normalize 输出 0-1
print(f"重排序耗时: {(time.time()-t0)*1000:.0f} ms")

# 按分数排序
ranked = sorted(zip(scores, documents), key=lambda x: -x[0])
for score, doc in ranked:
    print(f"{score:.4f}  {doc}")
# 输出：Java 内存泄漏相关文档（1,3,5）会排到前面，Python/JS 的会沉下去

FP16 与 batch 调优

  • use_fp16=True：GPU 上速度翻倍，精度损失 <0.5%，生产环境默认开。
  • compute_score 支持 batch：候选多时一次传入，比循环单条快几倍。
  • 文档超长要截断：reranker 的 max_length 通常 512，长文档要截前 512 token 或分块打分取 max。

★ top-N 选择策略：召回多少、重排多少、喂 LLM 多少

三个数字的权衡是 RAG 调优的核心：bi-encoder 召回 K1（送进 reranker 的数量）、rerank 输出 K2（送进 LLM 的数量）、rerank 内部计算 K3（reranker 实际打分的数量，可等于 K1）。常见配置和取舍：

典型配置（按场景）
┌──────────────────┬────────┬────────┬─────────┬──────────────────────────┐
│ 场景              │ 召回K1 │ 重排K2 │ 重排K3  │ 说明                     │
├──────────────────┼────────┼────────┼─────────┼──────────────────────────┤
│ 高质量问答（QA）   │ 50     │ 5      │ 50      │ 重排全部，喂 5 个        │
│ 长文档摘要         │ 30     │ 3      │ 30      │ 文档长，喂太多撑爆上下文 │
│ 低延迟实时搜索     │ 20     │ 10     │ 20      │ 牺牲精度换速度           │
│ 多轮对话           │ 100    │ 5      │ 50      │ 大召回小重排，避免遗漏   │
└──────────────────┴────────┼────────┼─────────┼──────────────────────────┘

几个经验法则：

  • K1（召回）宁多勿少：reranker 能纠正排序但救不回没召回的。K1 通常 30-100，看 latency 预算。
  • K2（喂 LLM）看上下文窗口：4K 窗口喂 3-5 个 500 字的片段；128K 窗口可喂 10-20 个。但注意"Lost in the Middle"问题（139 篇），喂太多中间的会被忽略。
  • threshold 过滤：rerank 分数低于阈值（比如 0.3）的直接丢弃，避免喂噪声给 LLM。bge-reranker normalize 后 0.3 是个常见阈值。
  • 不要过度重排：K3=K1 全重排是默认，但如果 K1 很大（如 500），可以先做一遍轻量 rerank（小模型）取 top-50，再用大 reranker 精排，省时间。
# 完整的两阶段检索 + 重排 + 阈值过滤
def retrieve_and_rerank(query, vector_index, reranker,
                       k1=50, k2=5, threshold=0.3):
    # 1. bi-encoder 粗召
    q_vec = embed(query)
    candidates = vector_index.search(q_vec, k=k1)     # [(id, score, doc), ...]

    # 2. cross-encoder 重排
    pairs = [[query, c.doc_text] for c in candidates]
    scores = reranker.compute_score(pairs, normalize=True)

    # 3. 阈值过滤 + 取 top-k2
    ranked = sorted(zip(scores, candidates), key=lambda x: -x[0])
    final = [(s, c) for s, c in ranked if s >= threshold][:k2]

    if not final:
        # 兜底：阈值太严一个都没过，退回 top-k2
        final = ranked[:k2]
    return final

# A/B 测试：有重排 vs 无重排的答案质量
# 通常 RAGAS 的 answer_relevancy 提升 10-20 个点（见 140 篇）

★ 主流 rerank 模型对比

  • bge-reranker（BAAI）：中文首选，base/large/重排都有，开源免费，FlagEmbedding 库调用方便。
  • Cohere rerank：商用 API，英文效果顶级，中文也不错，按调用收费。
  • jina-reranker：多语言，长文档支持好（支持 8K token），开源。
  • Voyage rerank：商用，和自家 embedding 配合最佳。
  • LLM as reranker：直接用 GPT-4/Claude 给候选打分或做 pairwise 比较，精度最高但成本极高（每候选一次 LLM 调用），只适合候选很少的精排。

选型建议：中文开源首选 bge-reranker-large；英文预算够选 Cohere；要长文档选 jina；追求极致精度且不在乎成本可上 LLM as reranker。

★ 重排序的进阶：级联、长文档与成本控制

基础重排序用单个 cross-encoder 对 top-K 打分，进阶系统会做更精细的优化。

级联重排。候选数很大（如 500）时，用大 cross-encoder 全打分太贵。级联做法：先用轻量 reranker（小模型或快速启发式）把 500 缩到 50，再用大 reranker 精排 50。两段都能用 cross-encoder，只是模型大小不同。这能在大候选量下保持精度同时控成本。

长文档处理。cross-encoder 的 max_length 通常 512 token，长文档（几千 token）超出要截断。粗暴截断前 512 会丢失后半部分信息。更好的做法：① 把长文档分块，每块单独打分取 max（或加权平均）；② 用支持长上下文的 reranker（jina-reranker 支持 8K）；③ 先用向量检索定位相关段，只对相关段 rerank。

批量与缓存。reranker 支持批量打分，一次传 50 对比循环单条快几倍。另外，同一文档对不同查询的 rerank 分数不缓存（依赖查询），但文档的预处理（tokenize、encode）可以缓存。对热点文档做 embedding 缓存能省一部分开销。

# 级联重排：轻量 → 重量
def cascade_rerank(query, candidates, light_model, heavy_model,
                   light_k=50, final_k=10):
    # 第一级：轻量 reranker 缩到 50
    pairs = [[query, c.text] for c in candidates[:500]]
    light_scores = light_model.compute_score(pairs, normalize=True)
    top50 = [c for _, c in sorted(zip(light_scores, candidates[:500]),
                                  key=lambda x:-x[0])[:light_k]]
    # 第二级：重量 reranker 精排 50
    pairs2 = [[query, c.text] for c in top50]
    heavy_scores = heavy_model.compute_score(pairs2, normalize=True)
    final = [c for _, c in sorted(zip(heavy_scores, top50),
                                  key=lambda x:-x[0])[:final_k]]
    return final

长文档分块打分：

def rerank_long_doc(query, doc, reranker, chunk_size=400):
    chunks = [doc[i:i+chunk_size] for i in range(0, len(doc), chunk_size)]
    pairs = [[query, c] for c in chunks]
    scores = reranker.compute_score(pairs, normalize=True)
    return max(scores)   # 取最相关段的分数代表整文档

★ 重排序的成本与延迟优化实战

rerank 是 RAG 流水线里延迟和成本的大头（cross-encoder forward 几十毫秒、GPU 占用高），优化它对整体体验影响大。

延迟拆解。一次 rerank 50 个候选的延迟构成：tokenize ~2ms，模型 forward ~30-50ms（GPU），后处理 ~1ms。forward 是大头，且和候选数近似线性。所以延迟优化主要在减少 forward 次数和加速 forward。

GPU 利用。cross-encoder 在 GPU 上比 CPU 快 10-20 倍，生产必上 GPU。但 GPU 贵，要榨干利用率：批处理（攒多个查询的候选一起 forward）、FP16/INT8 量化（速度翻倍精度损失小）、TensorRT 加速。一个 GPU 卡能扛几百 QPS 的 rerank。

按需 rerank。不是所有查询都需要 rerank。简单事实查询（向量召回 top-3 分数远高于第 4）直接用召回结果即可；复杂或模糊查询才触发 rerank。用一个轻量分类器判断"要不要 rerank"，能省 30-50% 的 rerank 调用。

异步流水线。rerank 和后续的 LLM 生成可以部分重叠：rerank 出 top-3 就先开始 LLM 生成（用前 3 个），同时 rerank 继续算剩下的作为补充。这种流水线能把端到端延迟降 30%。

# 按需 rerank：判断是否需要
def maybe_rerank(query, candidates, reranker, threshold=0.1):
    # 如果 top-1 和 top-4 的向量分数差距大，说明召回质量高，跳过 rerank
    if candidates[0].score - candidates[3].score > threshold:
        return candidates[:10]
    # 否则触发 rerank
    return reranker.rerank(query, candidates, top_k=10)
# 实测：30-50% 的查询能跳过 rerank，整体延迟和成本显著下降

这些优化让 rerank 在生产里既精准又可控成本。RAG 系统的"快、准、省"三角，rerank 是关键调节点。

★ 重排序在不同业务场景的差异化应用

重排序不是一刀切，不同业务场景对它的需求和配置不同。

场景一：客服问答 RAG。查询短、答案精确性要求高、延迟敏感（用户等着）。配置：召回 top-30，rerank 取 top-3，重排模型用 base 级（快），延迟预算 50ms 内。重点是精度而非覆盖。

场景二：法律/医疗文档检索。精度要求极高（错一个答案代价大）、延迟不敏感（专业用户愿意等）。配置：召回 top-100，级联 rerank（轻量缩到 30 + 重量精排 10），用 large 级重排模型，可接受 200ms 延迟。重点是极致精度。

场景三：电商商品搜索。召回量大、要兼顾个性化（用户偏好）、延迟极敏感（<30ms）。配置：召回 top-50，rerank 用轻量模型 + 业务特征（点击率、转化率、库存），取 top-20。这里 rerank 模型常和业务排序模型融合。

场景四：长文档摘要 RAG。文档长（每块几百字）、要覆盖多个相关块。配置：召回 top-30，长文档 rerank 用支持长上下文的模型，取 top-5。重点是块的相关性而非关键词。

# 不同场景的 rerank 配置模板
RERANK_CONFIGS = {
    "客服问答": {"recall_k": 30, "rerank_k": 3, "model": "base", "latency_budget_ms": 50},
    "法律医疗": {"recall_k": 100, "rerank_k": 10, "model": "large", "latency_budget_ms": 200,
                  "cascade": True, "light_k": 30},
    "电商搜索": {"recall_k": 50, "rerank_k": 20, "model": "base",
                  "extra_features": ["ctr", "conversion", "stock"], "latency_budget_ms": 30},
    "长文档":   {"recall_k": 30, "rerank_k": 5, "model": "long_context", "latency_budget_ms": 100},
}
def rerank_for_scenario(query, candidates, scenario, configs):
    cfg = configs[scenario]
    model = load_reranker(cfg["model"])
    if cfg.get("cascade"):
        # 级联
        light = load_reranker("base")
        top = light.rerank(query, candidates[:cfg["recall_k"]], cfg["light_k"])
        return model.rerank(query, top, cfg["rerank_k"])
    pairs = [[query, c.text] for c in candidates[:cfg["recall_k"]]]
    scores = model.compute_score(pairs, normalize=True)
    return [c for _, c in sorted(zip(scores, candidates), reverse=True)[:cfg["rerank_k"]]]

业务特征融合是电商/推荐场景的重点。纯相关性 rerank 模型不知道"这个商品卖得好不好"，要把业务特征（点击率、销量、新鲜度、库存）融合进排序。常见做法：rerank 分数 × 业务加权，或用 LambdaMART 学习融合权重。这让 rerank 既懂语义又懂业务。

个性化 rerank。同一查询不同用户应该看到不同结果。把用户特征（历史点击、偏好）作为 rerank 的额外输入，或用用户 embedding 和文档 embedding 交互。这是从"通用 rerank"到"个性化 rerank"的升级。

这些差异化配置说明 rerank 没有"通用最优"，要根据业务的精度/延迟/个性化需求定制。盲目套用别人的配置，往往效果不好。

★ 重排序模型的微调与业务适配

开源 rerank 模型（bge-reranker 等）是通用训练的，在特定业务上未必最优。微调能显著提升。

微调数据构建。需要 (query, doc, relevance) 三元组。来源：① 用户点击日志（点击=相关，展示未点击=负例）；② 人工标注（贵但准）；③ LLM 合成（用 GPT-4 生成训练对，量大但有噪声）。点击日志是最实用的来源——几乎免费的监督信号。

微调方法。rerank 模型本质是 cross-encoder，微调用交叉熵损失（相关/不相关的二分类）或 margin ranking loss（正例分数 > 负例分数一定 margin）。后者更适合检索场景。学习率要小（1e-5 到 5e-5），epochs 少（1-3），避免过拟合。

# 用点击日志微调 rerank 模型（概念流程）
def finetune_reranker(base_model, click_logs):
    """click_logs: [(query, clicked_doc, shown_not_clicked_docs), ...]"""
    optimizer = AdamW(base_model.parameters(), lr=2e-5)
    for epoch in range(2):
        for query, pos, negs in click_logs:
            pos_score = base_model.predict(query, pos)
            for neg in negs:
                neg_score = base_model.predict(query, neg)
                # margin ranking loss：正例分数要比负例高 margin
                loss = max(0, 0.3 - (pos_score - neg_score))
                loss.backward(); optimizer.step()
    return base_model

微调的收益与风险。收益：在特定业务上，微调后的 rerank 比通用模型提升 5-15% 的 NDCG。风险：过拟合——训练数据少时，微调后在训练集好但泛化差。对策：留出验证集，监控验证集指标；用早停。

持续微调。业务变化（新商品、新术语）会让 rerank 模型过时。定期用最新点击数据增量微调，保持模型新鲜。但要防止灾难性遗忘——增量微调可能让模型忘了通用能力。对策：增量数据里混入部分原始训练数据。

★ 重排序的延迟优化与异步流水线

rerank 是延迟敏感环节，几个优化能让它在延迟预算内运行。

减少 rerank 候选数。最直接的优化。rerank top-50 比 top-200 快 4 倍。前提是召回质量够（top-50 里包含最终 top-10）。先用便宜的方法（如向量检索）粗排到 top-50，再 rerank。不要无脑 rerank top-200。

模型蒸馏。把大 rerank 模型（large 级）蒸馏成小的（base 级），延迟降一半、精度掉 2-5%。如果延迟预算紧，蒸馏是高 ROI 优化。用大模型当 teacher 蒸馏 student，student 上线。

异步流水线。rerank 不必阻塞主流程：先返回粗排结果给用户（快），后台 rerank 完后再更新（流式或推送）。这让用户感知延迟大降，代价是结果会"变"（从粗排换成精排），要 UI 提示。

# 异步 rerank 流水线
async def search_with_async_rerank(query):
    # 1. 立即返回粗排结果（快）
    coarse = await vector_search(query, k=20)
    yield {"stage": "coarse", "results": coarse[:5]}   # 用户先看到
    # 2. 后台 rerank
    reranked = await rerank(query, coarse)
    yield {"stage": "reranked", "results": reranked[:5]}  # 精排后更新
# 用户 50ms 看到粗排结果，150ms 看到精排结果，感知延迟大降

GPU 批处理。rerank 模型在 GPU 上批量跑，攒多个查询的候选一起 rerank，吞吐提升 5-10 倍。代价是攒批的延迟（10-20ms）。适合高并发场景，单查询低并发场景 CPU ONNX 更省。

缓存 rerank 结果。相同 (query, doc) 对的 rerank 分数不变，可缓存。命中率取决于查询和文档的重复率，热门文档场景能省 20-40%。用 LRU 缓存 + 限制大小（rerank 分数缓存比向量占内存少）。

这些优化组合，能让 rerank 从"延迟瓶颈"变成"可承受的精排环节"。关键是按延迟预算倒推配置：先定延迟上限（如 100ms），再选模型大小、候选数、是否异步。

★ 本章小结

  • bi-encoder 快但粗（独立编码 + 点积），用于全库检索；cross-encoder 慢但准（拼接 + 深度交互），用于少量候选重排。两阶段架构是 RAG 标准。
  • cross-encoder 更准的本质是不压缩、query 和 document 的 token 级深度交互。
  • bge-reranker 是中文首选，FP16 + batch 调优，长文档要截断或分块。
  • 三个数字：召回 K1（30-100，宁多勿少）、重排输出 K2（看上下文窗口）、重排计算 K3（默认=K1）。
  • 阈值过滤避免喂噪声给 LLM；模型选型看语言/预算/长文档需求。
