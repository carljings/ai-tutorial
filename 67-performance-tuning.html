LLM 性能调优
延迟、吞吐与成本的三维优化

LLM 推理优化的目标从来不是单一的「快」，而是在延迟（用户体验）、吞吐（系统效率）、成本（商业可行）这三个互相拉扯的维度之间找到最优平衡点。这一章用 vLLM benchmark 代码和 INT4 量化对比数据，把这三个维度的优化方法讲透——让同样一张 GPU，扛住 3 倍的流量、成本砍到三分之一。

阅读约 29 分钟 难度 ★★★★

★ 一、理解三个维度的权衡

LLM 推理优化有三个核心维度，它们之间是互相制约的：

  • 延迟（Latency）：首 token 延迟（TTFT）和每 token 延迟（TPOT）。决定用户体验，越低越好。
  • 吞吐（Throughput）：单位时间处理的 token 总量或请求数。决定 GPU 利用率和单位成本，越高越好。
  • 成本（Cost）：每千 token 或每请求的 GPU 费用。决定商业模型的健康度，越低越好。

这三个维度的核心矛盾是：降低延迟通常要牺牲吞吐（比如减小 batch size），提升吞吐通常会增加延迟（比如增大 batch size）。优化的本质，是在这个三角里找到你的业务场景对应的「甜点」。

# 延迟-吞吐权衡的直观演示
def latency_vs_throughput(batch_size, model_size_gb, bandwidth_gbps):
    """
    batch_size: 并发请求数
    返回：每个请求的延迟，总吞吐
    """
    # 每 token 需要读一遍模型权重
    # batch 越大，GPU 利用率越高，但每个请求要排队
    per_token_compute = model_size_gb / bandwidth_gbps  # 秒/token（单请求）
    # batch 模式下，多个请求分摊了权重读取
    effective_per_token = per_token_compute / (batch_size ** 0.5)  # 简化
    # 但 batch 越大，调度延迟越高
    scheduling_delay = 0.01 * batch_size  # 简化
    latency = effective_per_token + scheduling_delay
    throughput = batch_size / latency  # 请求/秒
    return latency, throughput

# 对比不同 batch size
for bs in [1, 4, 16, 64]:
    lat, thr = latency_vs_throughput(bs, model_size_gb=16, bandwidth_gbps=2000)
    print(f"batch={bs:>3}: 单请求延迟={lat*1000:.0f}ms, 吞吐={thr:.1f} req/s")
# batch=1: 延迟最低，但吞吐最差（GPU 没吃满）
# batch=64: 吞吐最高，但单请求延迟上升（排队）

★ 二、延迟优化：让用户尽快看到输出

对 C 端产品（聊天、写作），延迟是核心体验。首 token 延迟（TTFT）超过 2 秒，用户感知明显变差。优化 TTFT 的几个手段：

  • 用更小的模型：7B 比 70B 快 10 倍，如果质量可接受，这是最直接的优化。
  • 量化压缩：FP16 → INT8 → INT4，每级提速 1.5-2 倍。
  • PagedAttention：vLLM 的核心技术，优化 KV Cache 内存管理，减少碎片。
  • Speculative Decoding：用小模型预测，大模型验证，成功能一次生成多个 token。
  • 缩短 prompt：TTFT 和 prompt 长度正相关，精简 prompt 直接降延迟。

★ 三、吞吐优化：榨干 GPU 的每一滴算力

对 B 端批量处理（文档摘要、数据标注），吞吐是核心指标。吞吐优化的关键在于让 GPU 始终满载工作。

★ 四、成本优化：让每一分算力都花在刀刃上

成本优化是商业可持续的关键。LLM 服务的成本结构里，GPU 折旧/租金占 70%+，所以优化方向就是「用更少的 GPU 跑同样的流量」。

  • 选对模型档位：简单任务用小模型（GPT-4o-mini 比 GPT-4 便宜 20 倍），难任务才用大模型。
  • 量化：INT4 量化让 70B 模型能在单卡 A100 跑，省下一半硬件成本。
  • Prompt 缓存：重复的 system prompt 可以缓存 KV，省掉重复计算。
  • 请求路由：简单请求路由到小模型，复杂请求才上大模型。

★ 五、量化对比：INT4 到底香不香

量化是 LLM 推理优化最有效的单一手段。但很多人担心量化损失精度。我们用真实数据对比 FP16、INT8、INT4 的效果和速度差异。

# 量化方案对比（Llama-3-8B 为例）

★ 六、Continuous Batching：吞吐优化的核心

传统的 batching（静态批处理）有个大问题：一个 batch 里所有请求必须等最长的那个完成才能开始下一批，短的请求白白等待。Continuous Batching（连续批处理）解决了这个问题——它可以在每个 token 生成步完成后，动态加入新请求、踢出已完成的请求。

# Continuous Batching 的概念演示
class ContinuousBatcher:
    """简化版的连续批处理调度器。"""
    def __init__(self, max_batch_size=128):
        self.max_batch_size = max_batch_size
        self.active_requests = []

    def step(self, new_requests):
        # 1. 移除已完成的请求，释放槽位
        self.active_requests = [r for r in self.active_requests if not r.done]
        # 2. 用空出的槽位接收新请求
        while new_requests and len(self.active_requests) < self.max_batch_size:
            self.active_requests.append(new_requests.pop(0))
        # 3. 一次前向推理，所有活跃请求一起算
        # （vLLM 内部用 PagedAttention 高效管理变长 KV Cache）
        self.forward_step()
        return len(self.active_requests)

# 对比静态 batching vs 连续 batching
# 静态：batch=4，必须等最慢的请求完成（比如 100 token）
#   吞吐 = 4 请求 / 100 步
# 连续：快请求（20 token）完成后立即换新请求进来
#   GPU 始终满载，吞吐提升 2-4 倍

vLLM、TGI、TensorRT-LLM 这些推理引擎的核心卖点就是 Continuous Batching。如果你还在用「一个请求一次推理」的朴素方式，切到 vLLM 能直接获得 3-5 倍的吞吐提升——这是 2024 年后 LLM 推理优化的最大 free lunch。

★ 七、Speculative Decoding：延迟优化的新武器

这是 2024 年开始流行的延迟优化技术。核心思想：用一个小的「草稿模型」快速预测多个 token，然后用大模型一次性验证这些预测。如果预测正确，就一次跳过多个 token；如果错误，就用大模型的结果纠正。数学上保证输出和纯大模型完全一致（无损）。

# Speculative Decoding 的概念流程
def speculative_decoding(prompt, draft_model, target_model, num_speculative=4):
    """推测解码：小模型预测，大模型验证。"""
    current_text = prompt
    while not is_finished(current_text):
        # 1. 小模型快速生成 num_speculative 个候选 token
        draft_tokens = draft_model.generate(current_text, num_speculative)
        # 2. 大模型一次性验证这些 token（并行计算）
        # 大模型对每个位置输出概率分布
        target_logits = target_model.forward(current_text + draft_tokens)

        # 3. 接受小模型预测正确的部分
        accepted = 0
        for i, token in enumerate(draft_tokens):
            target_token = target_logits[i].argmax()
            if token == target_token:
                accepted += 1   # 预测正确，接受
            else:
                break           # 预测错误，用大模型的结果替换

        # 4. 更新文本：接受的草稿 token + 大模型纠正的 1 个 token
        current_text += draft_tokens[:accepted]
        current_text += target_token  # 大模型的纠正

    return current_text

# 效果：如果小模型预测准确率 70%，每次平均接受 3 个 token
# 大模型一次前向能产出 4 个 token，延迟降低约 40-60%
# 关键：小模型必须和大模型足够「同步」（同架构同训练数据最好）

★ 八、智能路由：分级处理

不是所有请求都需要最强模型。一个聪明的做法是做请求路由：简单请求走小模型，复杂请求才上大模型。

# 智能请求路由
class ModelRouter:
    """根据请求复杂度路由到不同模型。"""
    def __init__(self):
        self.small_model = load_model("llama-3-8b")      # 便宜快
        self.large_model = load_model("gpt-4o")           # 贵但强
        self.complexity_classifier = load_classifier()

    def route(self, request):
        complexity = self.assess_complexity(request)
        if complexity == "simple":
            return self.small_model.generate(request)
        else:
            return self.large_model.generate(request)

    def assess_complexity(self, request):
        """评估请求复杂度。"""
        # 启发式规则
        if len(request) < 50 and "?" in request:
            return "simple"   # 短问答
        if any(k in request for k in ["证明", "推导", "重构", "分析"]):
            return "complex"  # 复杂任务
        # 用分类模型判断
        return self.complexity_classifier.predict(request)

# 效果：70% 的请求走小模型，成本降低 60%，复杂任务质量不打折

深 延迟、吞吐、成本的三维优化

LLM 性能优化不是单看一个指标，而是延迟、吞吐、成本三维联动的权衡——优化一个往往牺牲另一个。下面给出三维优化的决策框架和量化方法。

三维指标的计算与权衡

先算清三个指标的当前值和理论极限，才能知道优化空间在哪、瓶颈是什么。

def perf_model(gpu, model_b, qps, avg_tokens, price_per_gpu_h):
    """计算延迟/吞吐/成本三维指标。"""
    # 延迟：单请求 = prefill + decode 时间
    ttft = 0.3                         # 首字延迟(秒)，取决于 prefill
    decode_tps = 50                    # 生成吞吐(字符/s)，取决于带宽
    latency = ttft + avg_tokens / decode_tps
    # 吞吐：单位时间总 token
    throughput = qps * avg_tokens
    # 成本：每百万 token 美元
    gpu_needed = -(-throughput // 2000)   # 每卡2000 tok/s
    cost_per_mtok = gpu_needed * price_per_gpu_h * 3600 / (throughput * 3600) * 1e6
    return {"延迟(s)":round(latency,2), "吞吐(tok/s)":int(throughput),
            "成本($/百万tok)":round(cost_per_mtok,2), "GPU":int(gpu_needed)}

print(perf_model("A100", 7, qps=10, avg_tokens=200, price_per_gpu_h=2))
# 例：延迟 4.3s、吞吐 2000 tok/s、成本 ~$2/百万tok
# 优化目标决定方向：降延迟→小模型/Speculative；提吞吐→批处理；降成本→量化/路由

这个三维模型揭示了一个核心矛盾：降延迟往往要降批大小（少等请求凑批），但降批大小就降吞吐、升成本。所以优化前必须先问：业务最在意哪个维度？

优化目标主要手段对其他维度影响适合场景
降延迟小模型/投机解码/Prefix Cache吞吐略降、成本升对话、实时交互
提吞吐大批处理/连续批处理延迟升批处理、离线
降成本量化/小模型路由/缓存精度可能降成本敏感、大规模
提精度大模型/RAG/少采样延迟升、成本升高质量要求

分场景的最优策略组合

不同业务场景，三维优化的侧重完全不同。实时对话场景（客服、助手）最重延迟，策略是：流式输出降 TTFT 体感 + 小模型做简单路由 + Prefix Cache 复用系统提示，宁可单请求成本略高也要保证秒回。批处理场景（文档处理、数据标注）最重吞吐和成本，延迟无所谓，策略是：攒大批 + 连续批处理 + 量化，把 GPU 利用率榨干。高质量场景（专业写作、代码生成）最重精度，策略是：用最强模型 + RAG 接事实库 + 低温度，成本和延迟都让位给质量。一个常被忽略的高价值优化是请求路由：把简单请求（占 70% 流量）路由到便宜小模型，复杂请求才用大模型，整体成本能砍 60% 以上而不损失关键场景质量。另一个是语义缓存：对相同/相似查询缓存结果，命中率高的场景（如客服重复问题）能省下大量重复推理。永远是先度量再优化：用真实流量建三维基线，找到当前最受限的维度，针对性投入，而不是盲目堆所有优化手段——有些优化互相冲突，全开反而更差。

深 性能调优 FAQ 与权衡

性能优化最怕「单看一指标」，这些问题帮建立全局观。

Q1：老板要「又快又便宜又准」，怎么平衡？

明确优先级——三维（延迟/吞吐/成本/精度）常互相冲突，不能全要。按业务定：实时对话重延迟（牺牲点成本）、批处理重吞吐成本（牺牲延迟）、高质量场景重精度（牺牲成本延迟）。然后对症组合：降延迟→流式+小模型路由+Prefix Cache；提吞吐→大批+连续批处理；降成本→量化+路由+缓存；提精度→大模型+RAG。请求路由是降本利器（简单走小模型，复杂走大模型，砍 60%）。永远是先度量定位瓶颈再优化，盲目堆手段反而互相冲突更差。

Q2：优化到什么程度算够？

设定明确目标，达标就停。如「P99 < 2s、单请求成本 < $X、准确率 > Y%」，达标后转向更高价值的事（质量/功能），而非无止境优化。过度优化是浪费——把延迟从 2s 优化到 1.5s 用户无感，但投入巨大。判断标准：优化收益是否用户可感知或商业可量化，不能就停。记录基线（优化前后对比）证明价值，也防后续倒退。性能优化要有 ROI 思维，不是越极致越好，而是「够用且性价比高」最好。

★ 本章小结

这一章我们覆盖了 LLM 性能调优的三个维度。核心要点：

  • 三维权衡：延迟、吞吐、成本互相制约，优化的本质是找到业务场景的甜点。C 端重延迟，B 端重吞吐，商业看成本。
  • 延迟优化：更小模型、量化、PagedAttention、Speculative Decoding、精简 prompt。
  • 吞吐优化：Continuous Batching 是核心，让 GPU 利用率从 30% 拉到 80%+，vLLM 是首选引擎。
  • 量化对比：INT4 让 7B-13B 模型显存砍到 1/3、速度提升 50%，常规任务质量无损，是生产默认选择。70B+ 用 INT8 更稳。
  • Speculative Decoding：小模型预测大模型验证，数学无损，延迟降 40-60%。
  • 智能路由：简单请求走小模型，复杂请求上大模型，成本降 60%。
