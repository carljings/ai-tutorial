文本分块策略
固定 / 递归 / 语义分块与 LangChain 实操

Embedding 模型再好，喂进去的文本是一坨乱七八糟的片段，出来的向量也是垃圾。"怎么把长文档切成适合 embedding 的块"——这个看似琐碎的问题，实际是 RAG 系统里对最终效果影响最大的工程决策之一。本文拆解固定分块、递归分块、语义分块三种策略，给出 chunk_size 和 overlap 的经验值，并用 LangChain 走一遍实操。

三个原因决定了 RAG 必须分块：

  1. embedding 模型的输入长度限制：大多数 embedding 模型 max_length=512 token，超长会被截断，截断的部分信息全丢。
  2. embedding 质量随长度下降：即便模型支持长输入（如 bge-m3 支持 8K），把一个 8000 token 的文档压成一个 1024 维向量，信息密度太低，向量分不清"这个文档主要讲什么"，检索精度塌方。
  3. LLM 上下文窗口的有效利用：检索回来要喂给 LLM，一个 5000 字的文档占满半个上下文，LLM 只能看一两个文档；切成 500 字一块，能喂 10 个，召回的覆盖面大得多。

分块的核心目标：让每个块语义自洽（一块讲清楚一件事），同时块与块之间有适当重叠（不切断上下文）。

★ 固定分块：最简单的基线

按固定字符数（或 token 数）切，每块之间留一段重叠（overlap）。这是最朴素也最常用的方案，适合大多数场景。

它的关键参数有两个：

  • chunk_size：每块多大。太小（如 100 字）则语义不完整，一个论点被切成两半；太大（如 2000 字）则信息密度低、embedding 质量差。
  • overlap：相邻块的重叠部分。作用是防止句子被硬切断——如果块边界正好切在一个关键句中间，重叠能让这个句在两块都完整出现，检索时无论命中哪块都能拿到完整语义。

经验值

  • chunk_size：中文 200-500 字 / 英文 200-500 token 是甜区。问答类场景偏小（300 字），长文档摘要类偏大（800 字）。
  • overlap：通常 chunk_size 的 10-20%（如 chunk_size=500，overlap=50-100）。太小切句风险大，太大则冗余、浪费存储。

★ 递归分块：尊重文档结构

固定分块的痛点是"硬切"——它不管段落边界，可能在句中或词中切断。递归分块改进了这一点：先用最强的分隔符（如段落 \n\n）切，如果切出来的块还是太大，再用次强的分隔符（如句号）切，依此类推，直到块小于 chunk_size。

递归分块保留语义边界的概率高得多——它优先在段落、句子边界切，只在万不得已时才硬切。这是 LangChain 的 RecursiveCharacterTextSplitter 默认策略，也是生产环境最常用的。

# LangChain 三种分块器实操对比
from langchain_text_splitters import (
    CharacterTextSplitter,
    RecursiveCharacterTextSplitter,
)
# pip install langchain-text-splitters

sample = """
## 第一章：向量检索基础

向量检索是把文本转成向量后，用距离度量找最相似的过程。它的核心是 embedding 模型。

### 1.1 暴力检索

暴力检索是计算查询向量到所有库向量的距离，复杂度 O(N*d)。它准确但慢。

### 1.2 近似最近邻

ANN 用聚类、图、量化等手段，牺牲少量精度换大幅提速。HNSW 是当前主流。
"""

# 1. 固定分块（按字符，硬切）
fixed = CharacterTextSplitter(
    separator="\n\n", chunk_size=80, chunk_overlap=20,
    length_function=len
)
# 2. 递归分块（按分隔符优先级，尽量不硬切）
recursive = RecursiveCharacterTextSplitter(
    separators=["\n\n", "\n", "。", "，", " ", ""],   # 优先级从高到低
    chunk_size=80, chunk_overlap=20,
    length_function=len
)

print("=== 固定分块 ===")
for i, c in enumerate(fixed.split_text(sample)):
    print(f"[{i}] {repr(c[:50])}...")
print("\n=== 递归分块 ===")
for i, c in enumerate(recursive.split_text(sample)):
    print(f"[{i}] {repr(c[:50])}...")
# 递归分块的块边界更自然，多在段落/句子处，语义更完整

★ 语义分块：按内容语义切

固定和递归分块都是基于"字符位置"的，不理解语义。语义分块更进一步：用 embedding 模型给每个句子编码，计算相邻句子的语义相似度，在相似度突然下降的地方切——这通常意味着话题转换。

语义分块的效果最好（切出来的块语义最聚合），但代价是要对每个句子算 embedding，慢且贵。适合高质量场景（如法律、医疗文档），或离线一次性处理。LangChain 的 SemanticChunker 实现了这个策略。

# 语义分块 —— 按 embedding 相似度切
from langchain_experimental.text_splitter import SemanticChunker
from langchain_community.embeddings import HuggingFaceEmbeddings
# pip install langchain-experimental

embeddings = HuggingFaceEmbeddings(model_name="BAAI/bge-small-zh-v1.5")

# 三种断点策略
# "percentile": 相似度低于百分位（默认 95）时切
# "standard_deviation": 相似度低于均值-1*std 时切
# "interquartile": 用四分位距判定异常点
splitter = SemanticChunker(
    embeddings,
    breakpoint_threshold_type="percentile",
    breakpoint_threshold_amount=95,        # 数值越小块越多
)

chunks = splitter.split_text(sample)
for i, c in enumerate(chunks):
    print(f"[{i}] {repr(c[:60])}...")
# 块边界会落在语义转折处，而非固定字符处

★ 针对不同文档类型的进阶策略

  • Markdown / HTML：用 MarkdownHeaderTextSplitter / HTMLHeaderTextSplitter，按标题层级切，每块带上 H1/H2 元数据。这样检索时可以按章节过滤。
  • 代码：用 RecursiveCharacterTextSplitter 配合语言特定的分隔符（如 Python 的 def/class/import），保证函数不被切断。LangChain 提供 language 参数自动选分隔符。
  • PDF：先解析（unstructured/pdfplumber），按页/段落切，注意表格和图注的特殊处理。
  • 对话记录：按轮次切，每块包含完整 Q&A 对。
# Markdown 按标题切 + 递归细切（生产推荐组合）
from langchain_text_splitters import MarkdownHeaderTextSplitter, RecursiveCharacterTextSplitter

headers = [("#", "h1"), ("##", "h2"), ("###", "h3")]
md_splitter = MarkdownHeaderTextSplitter(headers_to_split_on=headers)
md_chunks = md_splitter.split_text(sample)   # 第一层：按标题切

# 第二层：对每个 md 块再递归细切到 embedding 友好的大小
rec = RecursiveCharacterTextSplitter(chunk_size=300, chunk_overlap=50)
final_chunks = []
for chunk in md_chunks:
    for piece in rec.split_text(chunk.page_content):
        final_chunks.append({
            "text": piece,
            "metadata": chunk.metadata,      # 保留 h1/h2/h3 元数据
        })
print(f"最终 {len(final_chunks)} 块")
for c in final_chunks:
    print(c["metadata"], "→", c["text"][:40])

★ 常见坑

  • chunk_size 过大：以为"块越大信息越多越好"，结果 embedding 分不清主题、检索精度下降。500 字通常是比 2000 字更好的起点。
  • overlap 为 0：省了存储，但切句风险大，关键信息被切两半时两块都检索不到。至少留 10% overlap。
  • 按字符数而非 token 数：中文一字约 1-2 token，英文一词约 1-1.5 token，按字符切容易超出 embedding 的 512 token 限制。重要场景用 tiktoken 按 token 切。
  • 忽略元数据：分块时不带文档标题、章节、页码，检索回来喂 LLM 时 LLM 不知道上下文，答非所问。务必保留元数据。
  • 所有文档一种策略：FAQ 短文本、法律长文档、代码混用一套分块参数，必然有场景效果差。按文档类型分流。

★ 分块质量的量化评估

分块策略选哪个、参数怎么调，不能靠感觉，要量化评估。评估分块质量的两个维度：

块内聚度（intra-chunk coherence）：一个块内部的语义是否聚焦。可以用块内句子的平均 embedding 相似度衡量——相似度高说明块语义聚合，低说明块语义杂乱（被硬切了）。理想分块的块内聚度高。

块间区分度（inter-chunk distinctness）：不同块的语义是否可区分。用块向量之间的平均相似度衡量——相似度低说明块各自独立、不重复，高说明块之间冗余（overlap 太大或切分不合理）。理想分块的块间区分度高。

这两个指标配合 recall@k（端到端检索效果）一起看，能诊断分块问题：块内聚度低 → 切得太碎或切在了语义转折处；块间区分度低 → overlap 太大或文档本身重复多；recall 低 → 综合问题，要具体分析。

# 分块质量评估
def evaluate_chunking(chunks, embed_model):
    chunk_vecs = embed_model.encode(chunks, normalize_embeddings=True)
    # 块内聚度：每个块内部句子的平均相似度
    intra_coherences = []
    for chunk in chunks:
        sents = split_sentences(chunk)
        if len(sents) < 2: continue
        sent_vecs = embed_model.encode(sents, normalize_embeddings=True)
        sim = sent_vecs @ sent_vecs.T
        intra_coherences.append(sim[np.triu_indices(len(sents), 1)].mean())
    # 块间区分度：1 - 块向量间平均相似度
    inter_sim = (chunk_vecs @ chunk_vecs.T)
    inter_distinct = 1 - inter_sim[np.triu_indices(len(chunks), 1)].mean()
    return {
        "块内聚度": np.mean(intra_coherences),
        "块间区分度": inter_distinct,
    }
# 对比不同 chunk_size 的分块质量
for cs in [200, 400, 800]:
    chunks = recursive_split(doc, chunk_size=cs)
    print(f"chunk_size={cs}:", evaluate_chunking(chunks, model))

这种量化评估让分块调优从"试一试"变成"测一测"。结合端到端 recall，能找到最优分块配置。

★ 分块的元数据与上下文保留

分块不只是切文本，还要保留让它能被正确检索和使用的元数据。这是常被忽视但极关键的一步。

每个块应该带的元数据：

  • 来源文档信息：文档 ID、标题、作者、发布时间。检索回来喂 LLM 时，这些信息让 LLM 能说"根据 2024 年 3 月的 Acme 财报..."而非泛泛回答。
  • 结构位置：章节路径（H1/H2/H3）、页码、段号。让 LLM 知道这块内容在原文档的什么位置，理解上下文层级。
  • 块前后文摘要：前一块和后一块的简短摘要。136 篇的 Contextual Retrieval 就是这个思路的极致版。即便不做到那个程度，简单的前后块标题也能帮 LLM 理解连贯性。
  • chunk_id 和 parent_id：parent-child 分块（136 篇）需要这两个 ID 建立映射。
# 带完整元数据的分块
def chunk_with_metadata(doc_text, doc_meta, chunk_size=400):
    chunks = recursive_split(doc_text, chunk_size=chunk_size)
    enriched = []
    for i, chunk_text in enumerate(chunks):
        enriched.append({
            "text": chunk_text,
            "metadata": {
                **doc_meta,                      # 文档级元数据
                "chunk_index": i,                # 块在文档里的序号
                "total_chunks": len(chunks),
                "prev_chunk_summary": summarize(chunks[i-1]) if i>0 else None,
                "next_chunk_summary": summarize(chunks[i+1]) if i<len(chunks)-1 else None,
            }
        })
    return enriched
# 检索回来时把元数据一起喂 LLM，答案质量和可信度显著提升

元数据保留得好，能让"检索回来的块"变成"有上下文的、可追溯的信息"，这对 RAG 答案的准确性和可解释性提升巨大。很多 RAG 效果差，根源不是模型不行，是元数据丢了导致 LLM 拿到的是"孤立碎片"。

★ 高级分块：基于文档结构的智能分块

固定/递归/语义分块是基础，进阶系统会基于文档的语义结构做更智能的分块。

基于布局的分块。PDF/Word/HTML 文档有天然的布局结构（标题、段落、表格、图注）。用 layout 解析器（unstructured、LayoutLM）先识别这些结构单元，按结构单元分块。比如一个表格作为一个块（不切碎），一个图注和它的图作为一块。这种分块保留了文档的语义完整性，对含表格图的文档（财报、论文）效果远好于纯文本分块。

基于语义角色的分块。用 NLP 识别文本的语义角色（定义、例子、论证、结论），按角色边界分块。比如把"定义 + 它的例子"作为一块，"论证 + 它的论据"作为一块。这种分块让每块语义自洽，但需要语义角色标注模型，复杂度高，适合高价值场景。

滑动窗口 + 结构感知。在递归分块基础上，加入"结构感知"——遇到表格/代码块/列表时，不切分它们，整体作为一个块或跳过。LangChain 的某些 splitter 支持。

# 基于 layout 的分块（用 unstructured）
from unstructured.partition.pdf import partition_pdf
from langchain_text_splitters import RecursiveCharacterTextSplitter
# 1. 解析 PDF 识别结构单元
elements = partition_pdf("report.pdf")
# elements 已经按标题、段落、表格分类
structured_chunks = []
for el in elements:
    if el.category == "Table":
        structured_chunks.append({"text": str(el), "type": "table"})  # 表格不切
    elif el.category == "Title":
        structured_chunks.append({"text": str(el), "type": "title"})
    else:
        # 正文用递归分块
        sub = RecursiveCharacterTextSplitter(chunk_size=400, chunk_overlap=50).split_text(str(el))
        for s in sub:
            structured_chunks.append({"text": s, "type": "text"})
# 这种分块保留了表格/标题的完整性，检索和回答质量显著提升

分块后处理。分块后可以做几个增强：① 块标题前置——把所属章节的标题加到块前面，让块脱离上下文也能被理解（类似 Contextual Retrieval 的轻量版）；② 块摘要生成——用小模型给每个块生成一句话摘要，存摘要的 embedding（检索用摘要 embedding，返回用原文）；③ 关键词提取——给每块提取关键词作为元数据，支持关键词过滤/混合检索。

# 块标题前置 + 摘要 embedding
def enrich_chunk(chunk, section_path):
    """section_path: ['财报', '2024 Q3', '云服务']"""
    title_prefix = " > ".join(section_path)
    enriched = f"[{title_prefix}]
{chunk}"
    return enriched
def chunk_with_summary(chunk, summary_model):
    summary = summary_model.summarize(chunk)   # 一句话摘要
    return {"text": chunk, "summary": summary,
            "embed_text": summary,   # 用摘要 embed（更聚焦）
            "retrieve_text": chunk}  # 检索命中后返回原文

这些智能分块让分块质量从"机械切分"上升到"语义感知"。投入产出比：layout 分块对含结构化元素的文档提升巨大（recall 涨 10-20%），块标题前置几乎零成本但提升明显（涨 5-8%）。建议按文档类型选合适的智能分块策略。

★ 分块与 Token 限制：理解 LLM 的真实输入约束

分块的"chunk_size"常按字符设，但 LLM 的输入约束是 token。理解两者的换算和 token 预算分配很重要。

字符 vs Token。中文一个字≈1-1.5 token，英文一个单词≈1-1.5 token，代码 token 化不规则（一个运算符可能一个 token）。所以"chunk_size=500 字符"在中文≈500 token，在英文≈350 token，在代码可能更多。设 chunk_size 时要按目标语言的 token 比例算。

Token 预算分配。LLM 上下文窗口（如 128K）要分给：系统 prompt、用户查询、检索到的上下文、生成预留。分配示例（128K 窗口）：系统 prompt 2K、查询 1K、上下文 100K、生成预留 25K。所以上下文实际能用 100K token。如果每块 500 token，能塞 200 块——但塞这么多会触发 Lost in the Middle，实际塞 10-30 块最优。

# Token 预算分配与块数计算
def plan_context_budget(model_window=128000, chunk_tokens=500):
    budget = {
        "system_prompt": 2000,
        "user_query": 1000,
        "generation_reserve": 25000,
    }
    context_budget = model_window - sum(budget.values())
    max_chunks = context_budget // chunk_tokens
    # 实际推荐块数（考虑 Lost in the Middle）
    practical_chunks = min(max_chunks, 20)
    print(f"窗口 {model_window}: 上下文预算 {context_budget} token")
    print(f"理论最多 {max_chunks} 块，实际推荐 {practical_chunks} 块")
    return practical_chunks
plan_context_budget()
plan_context_budget(model_window=8192)   # 小窗口模型对比

chunk_size 与检索粒度的关系。块小（200 token）→ 检索精准但信息碎片，LLM 要拼多块才能答；块大（1000 token）→ 信息完整但检索可能命中无关部分。折中是 300-500 token，配合 parent-child（小块检索、大块返回）兼顾两者。

不同内容类型的块大小。代码块要大（一个完整函数/类，避免切断逻辑）；对话/短文本块要小（按发言/段落）；长文档说明块中等。一刀切的 chunk_size 对混合内容不好——用结构感知分块（135 篇深入版讲过）按内容类型调整。

★ 分块质量的量化评估方法

分块好不好不能只凭感觉，要有量化评估方法。

评估维度一：检索召回率。固定 embedding 模型和查询，只变分块策略，看哪种分块的 recall@k 高。这是最直接的评估——分块是为了检索，检索好就是好。要在反映真实查询的评估集上测。

评估维度二：答案完整性。检索到块后，看 LLM 能否基于块完整回答问题。有些分块检索 recall 高但答案碎（块太小，信息不完整），有些 recall 中等但答案好（块大小合适，信息完整）。用 RAGAS 的 faithfulness 和 answer_relevancy 评估。

评估维度三：块的内聚度。每块是否语义自洽（不跨主题）。用 LLM 给每块打"内聚度分"（1-5），或用 embedding 算块内句子的平均相似度（高则内聚）。内聚度低的分块（如切断了定义和例子）检索和回答都差。

# 分块质量的多维度评估
def evaluate_chunking(chunks, eval_set, embed_model, llm):
    results = {}
    # 1. 检索召回
    index = build_index([embed_model(c) for c in chunks])
    recall = eval_recall(index, eval_set)
    # 2. 答案完整性
    faith_scores = []
    for q, a_expected in eval_set[:50]:
        retrieved = index.search(embed_model(q), k=5)
        answer = llm.generate(q, retrieved)
        faith_scores.append(ragas_faithfulness(q, answer, retrieved))
    # 3. 块内聚度（块内句子平均相似度）
    cohesions = []
    for c in chunks[:200]:
        sents = split_sentences(c)
        if len(sents) > 1:
            embs = [embed_model(s) for s in sents]
            sim = avg_pairwise_similarity(embs)
            cohesions.append(sim)
    results["recall"] = recall
    results["faithfulness"] = mean(faith_scores)
    results["cohesion"] = mean(cohesions)
    return results
# 对比不同分块策略的三维度，选综合最优

评估的坑。① 只看 recall 忽略完整性——小块 recall 高但答案碎，要综合看；② 评估集太小——几十条查询不足以区分分块策略差异，要几百条；③ 忽略块大小对延迟的影响——大块检索慢、上下文长，要纳入延迟成本。

分块策略的对比实验。固定其它环节，只变分块（固定 400 vs 递归 vs 语义 vs parent-child），跑评估。通常结论：parent-child 综合最优（小块检索准、大块返回完整），递归次之，固定最差。但具体数据集可能不同，必须自测。这个量化评估让分块从"凭经验"变成"数据驱动"。

★ 本章小结

  • 分块必要：embedding 有长度限制、长文本信息密度低、LLM 上下文要高效利用。目标是块语义自洽 + 适当重叠。
  • 固定分块：最简单，chunk_size 200-500 字（中文）/ token（英文），overlap 10-20%。
  • 递归分块：按分隔符优先级切，尊重段落/句子边界，是生产首选（LangChain RecursiveCharacterTextSplitter）。
  • 语义分块：按 embedding 相似度切，质量最高但慢且贵，适合高质量离线场景。
  • 按文档类型选策略：Markdown 按标题切、代码按语言切、PDF 先解析；务必保留元数据；别用 token 数算的字符切法。
