Agent 框架横评
LangChain、LangGraph、AutoGen、CrewAI、OpenAI Agents

"到底该用哪个 Agent 框架？"这是 2024-2025 年开发者问得最多的问题。答案不是"哪个最好"，而是"哪个最适合你的场景"。本章用每个框架约 20 行的最小可运行示例，让你直观感受它们的 API 风格、抽象层级和设计哲学。从 LangChain 的全能工具箱，到 LangGraph 的状态图，到 CrewAI 的角色扮演，到 OpenAI Agents 的官方简洁，一次看个明白。

阅读约 26 分钟 难度 ★★★★

★ Agent 框架的三个抽象层级

所有 Agent 框架本质上都在解决同一个问题：如何把"LLM 调用 + 工具使用 + 状态管理"封装成可复用的抽象。但不同框架选择了不同的抽象层级，这决定了它们的易用性和灵活性。低层框架（如 LangChain 的核心）提供细粒度的组件（Prompt、Memory、Tool、Chain），需要你自己组装。中层框架（如 LangGraph）提供"状态图"抽象，让你用节点和边描述 Agent 的工作流。高层框架（如 CrewAI）提供"角色"和"任务"抽象，用最自然的语言描述多 Agent 协作。

选型的核心原则是：抽象层级越高，上手越快但灵活性越低。如果你做的是标准的多步推理 Agent，高层框架能帮你省大量代码；如果你需要精细控制每一步的行为（如自定义路由、人在环路、复杂的错误恢复），中层或低层框架更合适。下面逐一拆解五大框架。

★ LangChain：大而全的工具箱

LangChain 是最早也是最知名的 LLM 应用框架。它的定位是"LLM 应用的瑞士军刀"——从 prompt 管理、记忆系统、工具调用、文档加载到向量库集成，几乎覆盖了 LLM 应用的每一个环节。LangChain 的优势是生态庞大、文档丰富、集成众多（支持上百种工具和数据源）。但它的缺点也很明显：抽象层级多、API 变动频繁（v0.1 到 v0.3 大改多次）、过度封装导致调试困难。

LangChain 最适合"快速搭建原型"和"需要大量第三方集成的场景"。但如果你只需要简单的 Agent 功能，LangChain 可能过于笨重。它更适合作为"组件库"按需使用，而不是全盘采用。

# LangChain 最小 Agent 示例（ReAct 模式）
from langchain.agents import create_tool_calling_agent, AgentExecutor
from langchain_openai import ChatOpenAI
from langchain.tools import tool
from langchain_core.prompts import ChatPromptTemplate

# 定义工具（用装饰器自动生成 schema）
@tool
def search_web(query: str) -> str:
    """搜索网络获取最新信息"""
    return f"搜索结果: 关于'{query}'的最新信息..."

@tool
def calculate(expression: str) -> str:
    """计算数学表达式"""
    return str(eval(expression))  # 生产环境请用安全的计算库

# 创建 Agent
llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
tools = [search_web, calculate]
prompt = ChatPromptTemplate.from_messages([
    ("system", "你是助手，可以用工具回答问题。"),
    ("human", "{input}"),
    ("placeholder", "{agent_scratchpad}"),
])

agent = create_tool_calling_agent(llm, tools, prompt)
executor = AgentExecutor(agent=agent, tools=tools, verbose=True)

# 运行
result = executor.invoke({"input": "3 的 5 次方是多少？再搜一下今天的新闻。"})
print(result["output"])
# LangChain 会自动: 理解问题 → 调 calculate → 调 search_web → 综合回答

★ LangGraph：状态图驱动的精细控制

LangGraph 是 LangChain 团队推出的"Agent 编排框架"，弥补了 LangChain Agent 在复杂工作流上的不足。它的核心抽象是有状态的循环图（Stateful Cyclic Graph）——你定义一个状态对象、若干节点函数和条件边，LangGraph 负责按图执行、维护状态、支持循环和分支。这让你可以实现任意复杂的 Agent 工作流：多步推理、人在环路审批、错误重试、多 Agent 协作。

LangGraph 的优势是"可控性"——你可以精确定义每一步做什么、什么条件下跳转、状态如何传递。它还内置了 checkpoint 持久化（支持中断恢复）、流式输出、并行节点等生产级特性。对于需要精细控制 Agent 行为的场景（如合规审批、复杂决策树），LangGraph 是当前最佳选择。

# LangGraph 最小示例：带反思的写作 Agent
from typing import TypedDict, Literal
from langgraph.graph import StateGraph, END
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o-mini")

class State(TypedDict):
    topic: str
    draft: str
    critique: str
    approved: bool

def write_node(state: State) -> State:
    """写作节点：生成初稿"""
    state["draft"] = llm.invoke(f"写一段关于{state['topic']}的短文").content
    return state

def review_node(state: State) -> State:
    """审查节点：评估并决定是否通过"""
    feedback = llm.invoke(f"评估这段文字的质量，回答 PASS 或 FAIL：\n{state['draft']}").content
    state["approved"] = "PASS" in feedback
    state["critique"] = feedback
    return state

def route(state: State) -> Literal["write", "__end__"]:
    """路由：通过则结束，否则重写"""
    return "__end__" if state["approved"] else "write"

# 构建状态图
graph = StateGraph(State)
graph.add_node("write", write_node)
graph.add_node("review", review_node)
graph.add_edge("__start__", "write")
graph.add_edge("write", "review")
graph.add_conditional_edges("review", route)  # review → write 或 END

app = graph.compile()
result = app.invoke({"topic": "人工智能的未来"})
print(result["draft"])  # 通过审查的最终版本

★ AutoGen：多 Agent 对话的先驱

AutoGen 由微软研究院开发，是最早专注于"多 Agent 对话"的框架。它的核心思路是：定义多个具有不同角色和能力（如 AssistantAgent、UserProxyAgent）的 Agent，让它们通过对话协作完成任务。一个典型的设置是：一个"程序员 Agent"写代码，一个"执行 Agent"（UserProxy）运行代码并反馈结果，两者循环对话直到任务完成。

AutoGen 的优势在于"对话式协作"的自然性——多个 Agent 像人类团队一样讨论、分工、迭代。特别适合代码生成、数据分析等需要"生成-执行-反馈"循环的任务。AutoGen 0.4 版本（原 AG2）进行了大规模重构，引入了更灵活的事件驱动架构。它的局限是配置较复杂，多 Agent 对话的"收敛性"（防止无限循环）需要仔细调参。

# AutoGen 最小示例：代码编写 + 执行协作
from autogen_agentchat.agents import AssistantAgent, CodeExecutorAgent
from autogen_agentchat.teams import RoundRobinGroupChat
from autogen_ext.models.openai import OpenAIChatCompletionClient
from autogen_ext.code_executors.local import LocalCommandLineCodeExecutor

model_client = OpenAIChatCompletionClient(model="gpt-4o-mini")

# Agent 1：程序员（写代码）
coder = AssistantAgent(
    name="coder",
    model_client=model_client,
    system_message="你是 Python 程序员，用代码解决问题。代码用 ```python 块包裹。",
)

# Agent 2：执行者（运行代码并反馈）
executor = CodeExecutorAgent(
    name="executor",
    code_executor=LocalCommandLineCodeExecutor(work_dir="code_output"),
)

# 组成团队：轮流发言
team = RoundRobinGroupChat([coder, executor], max_turns=6)

# 运行任务
import asyncio
result = asyncio.run(team.run(task="计算斐波那契数列前 20 项并打印"))
print(result)
# AutoGen 流程: coder 写代码 → executor 执行 → 如果报错 coder 修复 → 循环

★ CrewAI：角色扮演的多 Agent 协作

CrewAI 是最"高层"的 Agent 框架——它的抽象是"船员"（Agent/角色）和"任务"（Task），你像组建一个团队一样描述每个成员的角色、目标、背景，然后分配任务。CrewAI 自动协调各角色按顺序或并行执行任务，共享上下文。这种"角色驱动"的设计让代码读起来像剧本，非常适合非技术背景的人理解 Agent 工作流。

CrewAI 的优势是"开发体验极佳"——几行代码就能定义一个多角色团队。适合内容创作、市场调研、文档撰写等"创意型"任务。它的局限是控制粒度较粗——如果你想精细控制某一步的行为，会发现 CrewAI 的抽象不够灵活。CrewAI 也是纯 Python 框架，与 LangChain 的组件生态部分兼容。

# CrewAI 最小示例：内容创作团队
from crewai import Agent, Task, Crew, Process

# 定义角色（像招人一样描述每个成员）
researcher = Agent(
    role="研究员",
    goal="收集关于给定主题的全面信息",
    backstory="你是有 10 年经验的研究员，擅长快速搜集和整理信息。",
    llm="gpt-4o-mini",
    verbose=True,
)

writer = Agent(
    role="撰稿人",
    goal="把研究结果写成一篇引人入胜的文章",
    backstory="你是资深科技撰稿人，擅长把复杂概念讲得通俗有趣。",
    llm="gpt-4o-mini",
)

# 分配任务（每个任务指定给一个角色）
research_task = Task(
    description="研究 {topic} 的核心概念、最新进展和争议点。",
    agent=researcher,
    expected_output="一份结构化的研究笔记",
)

write_task = Task(
    description="基于研究结果写一篇 800 字的科普文章。",
    agent=writer,
    expected_output="一篇完整的科普文章",
    context=[research_task],  # 依赖研究任务的输出
)

# 组建团队，按顺序执行
crew = Crew(
    agents=[researcher, writer],
    tasks=[research_task, write_task],
    process=Process.sequential,  # 顺序执行：先研究后写作
)

result = crew.kickoff(inputs={"topic": "扩散模型"})
print(result.raw)  # 最终的文章

★ OpenAI Agents SDK：官方极简主义

2025 年 3 月，OpenAI 发布了 Agents SDK（取代了之前的 Swarm 实验框架），它的设计哲学是"极简主义"——用最少的抽象，提供最核心的 Agent 能力。核心概念只有三个：Agent（定义角色和工具）、Handoff（Agent 之间的转交）、Guardrails（输入输出安全检查）。没有复杂的图、没有事件系统，就是纯粹的函数调用循环。

OpenAI Agents 的优势是与 OpenAI 生态深度集成（模型、工具、追踪开箱即用），学习成本极低（核心 API 不超过 10 个函数）。特别适合"以 OpenAI 模型为主、需要快速上线"的场景。它的局限是绑定 OpenAI 生态较深，虽然支持其他模型但体验不如原生。

# OpenAI Agents SDK 最小示例
from agents import Agent, Runner, function_tool

# 定义工具
@function_tool
def get_weather(city: str) -> str:
    """获取指定城市的天气"""
    return f"{city}今天晴，25度"

# 定义 Agent（角色 + 工具 + 指令）
travel_agent = Agent(
    name="旅行助手",
    instructions="你是一个旅行规划助手，帮用户查询天气和规划行程。",
    tools=[get_weather],
)

# 运行（Runner 自动处理工具调用循环）
result = Runner.run_sync(travel_agent, "帮我查查北京的天气")
print(result.final_output)

# 多 Agent 转交（Handoff）
billing_agent = Agent(
    name="账单助手",
    instructions="你负责处理账单和退款问题。",
)

support_agent = Agent(
    name="客服",
    instructions="你是通用客服，账单问题转给 billing_agent。",
    handoffs=[billing_agent],  # 遇到账单问题自动转交
)

result = Runner.run_sync(support_agent, "我要退款")
# support_agent 会把请求转交给 billing_agent 处理

选型建议：
- 快速原型 + 大量集成 → LangChain
- 需要精细控制工作流 → LangGraph（推荐生产环境）
- 多 Agent 代码协作 → AutoGen
- 非技术人员描述工作流 → CrewAI
- 纯 OpenAI 生态 + 快速上线 → OpenAI Agents SDK
- 不确定？先从 LangGraph 或 OpenAI Agents 入手，它们最通用

五大 Agent 框架各有定位：LangChain 是组件齐全的工具箱，适合需要大量集成的原型开发；LangGraph 用状态图提供精细的工作流控制，是复杂生产级 Agent 的最佳选择；AutoGen 专注多 Agent 对话协作，适合代码生成-执行循环；CrewAI 用角色和任务的高层抽象让多 Agent 协作像写剧本一样简单；OpenAI Agents SDK 以极简主义提供官方级体验，适合 OpenAI 生态快速上线。所有框架的核心模式都是"LLM + 工具调用 + 状态循环"，差异只在抽象层级和 API 风格。选型原则：抽象越高上手越快但越不灵活，根据场景的复杂度和控制需求对号入座。不必纠结选错——先动手做，理解核心模式后切换框架成本低。未来框架会继续收敛，但"理解 Agent 的核心原理"这项能力永远保值。

import numpy as np
from dataclasses import dataclass
from typing import List, Optional

@dataclass
class PerformanceMetrics:
    latency_ms: float
    throughput_ops: float
    memory_mb: float
    accuracy: float

def benchmark(fn, iterations=100):
    """通用性能基准测试"""
    import time
    t0 = time.time()
    for _ in range(iterations):
        result = fn()
    elapsed = time.time() - t0
    return PerformanceMetrics(
        latency_ms=elapsed * 1000 / iterations,
        throughput_ops=iterations / elapsed,
        memory_mb=0,  # 实际中用 psutil 获取
        accuracy=0.0  # 根据具体任务计算
    )

# 示例：对比两种方案的性能
def approach_a(data):
    """方案A：直接计算"""
    return np.softmax(data @ data.T / np.sqrt(data.shape[-1]), axis=-1)

def approach_b(data):
    """方案B：分块计算（省内存）"""
    chunk_size = 64
    results = []
    for i in range(0, data.shape[0], chunk_size):
        chunk = data[i:i+chunk_size]
        results.append(np.softmax(chunk @ data.T / np.sqrt(data.shape[-1]), axis=-1))
    return np.vstack(results)

data = np.random.randn(512, 768)
metrics_a = benchmark(lambda: approach_a(data))
metrics_b = benchmark(lambda: approach_b(data))
print(f"方案A: {metrics_a.latency_ms:.1f}ms, {metrics_a.throughput_ops:.0f} ops/s")
print(f"方案B: {metrics_b.latency_ms:.1f}ms, {metrics_b.throughput_ops:.0f} ops/s")

常见问题FAQ

Q: 如何选择合适的方案？ 先用方案A跑通验证正确性，再根据实际瓶颈（内存/延迟/精度）切换到方案B或C。永远不要过早优化。

Q: 生产环境最大的坑是什么？ 过度工程化——在需求不明确时投入大量精力做复杂架构。建议MVP先行，收集反馈后迭代。

Q: 如何评估效果？ 定义明确的成功指标（准确率/延迟/成本），用golden set做回归测试，A/B测试验证改进效果。至少100个样本才有统计显著性。
