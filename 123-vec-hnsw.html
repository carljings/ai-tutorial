HNSW 原理
多层跳表如何实现亚毫秒级向量检索

当 IVF 还在纠结"训练多久、重建多频繁"时，HNSW（Hierarchical Navigable Small World）凭借图结构的天然增量性，成了当今向量数据库的事实标准。Milvus、Qdrant、Weaviate、pgvector 全部内置 HNSW 作为默认索引。本章拆解它的多层跳表结构，讲清 M、efConstruction、efSearch 三个参数如何决定召回率与延迟，并用 hnswlib 库跑通从建库到查询的完整流程。

调 HNSW 不能只看查询性能，还要预估建索引的耗时和内存占用——这两者在亿级数据上可能决定方案可行性。下面给出基于参数的解析估算，帮你在建库前做容量规划。

# HNSW 构建时间与内存解析估算
def hnsw_estimate(N, dim, M, efC):
    """N: 向量数; dim: 维度; M: 图度数; efC: 建图候选数。"""
    # 1) 内存: 向量本体 + 图邻居指针
    vec_mem = N * dim * 4               # float32
    # 每层每节点最多 M 个邻居(双向), 平均层数 ~ log(N)
    import math
    avg_layers = math.log(N) / math.log(2) if N > 1 else 1
    graph_mem = N * M * avg_layers * 4 * 2   # 邻居int32 + 双向
    total_mem = (vec_mem + graph_mem) / 1e9  # GB
    # 2) 构建时间: 每个节点插入需 efC 次距离计算 + 图维护
    dist_per_insert = efC * math.log(N)      # 对数级搜索
    # 假设单次距离计算 1us(粗略, 含内存访问)
    build_s = N * dist_per_insert * 1e-6
    return {"内存_GB": round(total_mem, 2),
            "构建耗时_分钟": round(build_s/60, 1)}
for N in [1_000_000, 100_000_000, 1_000_000_000]:
    est = hnsw_estimate(N, 768, M=16, efC=200)
    print(f"N={N:>13,} -> {est}")

8 HNSW 参数调优实战实验

前面讲了 HNSW 的多层跳表结构和 M、efConstruction、efSearch 三个参数的含义，但"含义"和"会调"之间隔着一堆真实数据上的实验。我们把这三个参数用 hnswlib 在 10 万条 128 维向量上做一组网格实验，直接看它们如何影响召回率和延迟。这种"用数据说话"的调参方式，比死记"M 取 16"靠谱得多——因为最优值高度依赖你的数据分布、召回目标和延迟预算。

实验设计的关键是先有一份可信的 ground truth。我们用暴力计算得到每条查询的真实 top-10 近邻，再用 HNSW 的结果和它比对，交集占比就是召回率。先准备数据和 ground truth：

# -*- coding: utf-8 -*-
"""HNSW 三个核心参数调优：M、efConstruction、efSearch。
用 hnswlib 在同一份数据上跑网格实验，观察召回率与延迟的权衡。
前置: pip install hnswlib numpy
"""
import numpy as np, hnswlib, time

dim = 128
N = 100_000
data = np.random.randn(N, dim).astype("float32")
queries = np.random.randn(200, dim).astype("float32")

# 用暴力计算作为 ground truth
def brute_topk(q, k=10):
    sim = data @ q.T            # [N, Q]
    return [np.argpartition(sim[:, i], -k)[-k:] for i in range(q.shape[1])]

gt = brute_topk(data, queries, k=10)   # 真实近邻

核心函数 build_and_eval 把"建索引 + 查询 + 算召回"封装成一次调用，方便做网格扫描。注意一个工程细节：efSearch 是查询时设置的（用 set_ef 动态调整），而 M 和 efConstruction 是建索引时固定的——这意味着你可以在不重建索引的前提下，用 efSearch 在线权衡召回和延迟，这是 HNSW 极实用的特性。跑一组 M × efSearch 的网格，你会看到一个清晰的权衡曲面：

def build_and_eval(M, efC, efS, k=10):
    """建索引 + 查询，返回 (召回率, 平均延迟ms)。"""
    idx = hnswlib.Index(space="l2", dim=dim)
    idx.init_index(max_elements=N, ef_construction=efC, M=M)
    t0 = time.time()
    idx.add_items(data, np.arange(N))
    build_s = time.time() - t0
    idx.set_ef(efS)                       # 查询时动态调 efSearch
    t0 = time.time()
    labels, _ = idx.knn_query(queries, k=k)
    q_ms = (time.time() - t0) / len(queries) * 1000
    # 召回率 = 预测近邻与真实近邻的交集占比
    recall = np.mean([
        len(set(labels[i]) & set(gt[i])) / k for i in range(len(queries))
    ])
    return recall, q_ms, build_s

# 网格实验：M 和 efSearch 的权衡
print(f"{'M':>4} {'efC':>5} {'efS':>5} {'recall':>8} {'latency_ms':>11} {'build_s':>8}")
for M in [8, 16, 32]:
    for efS in [20, 50, 100]:
        r, lat, bs = build_and_eval(M, 200, efS)
        print(f"{M:>4} {200:>5} {efS:>5} {r:>8.3f} {lat:>11.2f} {bs:>8.1f}")
# 经验结论: M=16 是甜点; efSearch 调到 50 召回>95% 且延迟可控。

这组实验能提炼出三条可迁移的调参经验。第一，M 是建索引时就要定死的基础参数，它决定图的度数（每个节点的邻居数），M 越大召回越高但内存和建图时间也线性涨——对大多数场景 M=16 是甜点，追求极致召回用 32，内存极度紧张用 8。第二，efConstruction 建图越精细，后续查询越省力，建议固定在 200~400，它只在建索引时花一次成本，换来的是更高质量的图结构。第三，efSearch 是你日常调参的主旋钮，从 50 起步，召回不够就往上加（通常加到 100~200 能突破 99%），延迟超标就往下调。最后给一个生产建议：把 efSearch 做成可在线热调的配置项，配合"召回率监控 + 延迟监控"做自动调参——流量低时调高 efSearch 保召回，流量高峰自动调低保延迟，这是大厂向量服务的标准做法。

1 为什么图结构比聚类更适合增量场景

IVF 的痛点是"聚类中心固定后，新数据如果落在新区域，召回率会持续下降"。图结构绕开了这个问题：它不预设任何中心，而是让每个向量成为图中的一个节点，与它最近的若干邻居建立边。查询时从入口点出发，沿着边不断"贪心"跳向更近的节点，直到无法再改进为止。这种"导航式"搜索天然支持增量插入——新节点只需连到几个近邻，图的连通性就自然维持。

HNSW 的核心创新是借鉴跳表（Skip List）的分层思想，把单层图变成多层图。最底层包含所有节点，上层逐层稀疏采样（每个节点以指数衰减的概率出现在更高层）。查询时先在最顶层做"粗导航"快速定位到大致区域，再逐层下降做"细导航"精确逼近目标。这就像用地图找路：先用全国地图定位到城市，再用城市地图定位到街道，最后用街景找到具体门牌。

2 多层跳表与 NSW 的关键区别

要理解 HNSW，得先理解它的前身 NSW（Navigable Small World）。NSW 是单层图，查询时从随机入口点出发贪心搜索，问题是早期建的边（long-range edge）负责远距离跳跃，后期建的边（short-range edge）负责精确逼近，但两者混在一起，搜索路径又长又容易陷入局部最优。

HNSW 用分层解决了这个混乱：上层只保留 long-range 边，负责快速跨区域跳跃；下层保留所有边，负责精确局部搜索。节点 i 的最高层级由一个指数分布随机决定：level = floor(-ln(uniform(0,1)) * mL)，其中 mL = 1/ln(M)。这意味着绝大多数节点只在第 0 层（最底层），只有少数"幸运儿"会出现在高层，形成天然的稀疏骨架。这种结构让搜索复杂度从 NSW 的 O(log N * log N) 降到 O(log N)。

一个常被忽略的细节：HNSW 每层的邻居数不是固定的。第 0 层（底层）每个节点最多有 2 * M 个邻居（因为要连更多近距离点保证精度），而第 1 层及以上每层最多 M 个。这个设计兼顾了底层的精度需求和上层的导航效率。

3 三个参数：M、efConstruction、efSearch

HNSW 的行为完全由三个参数控制，理解它们就能调出适合自己数据的索引。

M（图的度数）：每个节点在第 1 层及以上保留的最大邻居数（底层是 2M）。M 越大，图越稠密，召回率越高，但内存占用和构建时间线性增长。推荐范围 8 到 48，常见默认值 16。对于高维向量（768 维以上）或对召回率要求高的场景，M 取 32；对于低维（128 维）或追求低延迟，M 取 8 到 12。

efConstruction（构建时搜索宽度）：建索引时，为每个新节点寻找邻居的候选列表大小。这个值只影响构建速度和索引质量，不影响查询。值越大，建的图质量越高（召回率上限越高），但构建越慢。推荐范围 100 到 500，默认 200。如果你的数据是一次性导入的，把 efConstruction 调到 400 能显著提升最终召回率。

efSearch（查询时搜索宽度）：这是运行时可调的参数，相当于 IVF 的 nprobe。efSearch 越大，搜索越深，召回率越高但延迟越高。关键是它必须大于等于你想要的 top-k。下表是 M=16、efConstruction=200 时典型的权衡曲线（100 万 768 维向量）：

efSearch=16   recall@10≈0.82  延迟 0.4ms
efSearch=50   recall@10≈0.95  延迟 0.8ms
efSearch=100  recall@10≈0.98  延迟 1.5ms
efSearch=200  recall@10≈0.995 延迟 3.2ms

注意 HNSW 在 recall 0.95 时的延迟（0.8ms）比 IVF（2.6ms）低 3 倍，这就是它成为默认索引的原因。但代价是内存：HNSW 需要存储图结构，每个节点的邻居列表用 int32 存储要占 2M * 4 字节，加上原始向量，总内存是原始数据的 1.3 到 1.8 倍。

4 hnswlib 实操：从建库到查询

hnswlib 是 HNSW 作者 Malkov 开源的 C++ 库（带 Python 绑定），是 FAISS 之外最流行的 ANN 库，也是 Milvus、Qdrant 内部 HNSW 实现的基础。下面用它在 100 万向量上构建索引并调参。

import numpy as np
import hnswlib
import time

# 1. 准备数据
np.random.seed(42)
N, D = 1_000_000, 768
data = np.random.randn(N, D).astype('float32')
labels = np.arange(N)

# 2. 初始化索引
index = hnswlib.Index(space='cosine', dim=D)

# 3. 构建：M 控制度数，ef_construction 控制建图质量
index.init_index(max_elements=N, ef_construction=400, M=32)

# 4. 添加数据（支持分批 add，天然增量）
print("构建 HNSW 索引...")
t0 = time.time()
index.add_items(data, labels)
print(f"构建耗时 {time.time() - t0:.1f}s")

# 5. 设置查询参数并搜索
index.set_ef(50)  # ef_search，运行时可调，必须 >= k
queries = np.random.randn(100, D).astype('float32')

t0 = time.time()
labels_found, distances = index.knn_query(queries, k=10)
latency = (time.time() - t0) / 100 * 1000
print(f"ef=50: 延迟 {latency:.2f}ms")

# 6. 评估召回率（对比暴力搜索）
gt = np.argsort(-(data @ queries.T), axis=0)[:10].T
recall = sum(len(set(labels_found[i]) & set(gt[i])) for i in range(100)) / 1000
print(f"recall@10 = {recall:.4f}")

# 7. 持久化
index.save_index("index.hnsw")
index2 = hnswlib.Index(space='cosine', dim=D)
index2.load_index("index.hnsw", max_elements=N)

# 8. 增量插入新数据（HNSW 的杀手锏）
new_data = np.random.randn(10000, D).astype('float32')
index2.resize_index(N + 10000)
index2.add_items(new_data, np.arange(N, N + 10000))

这段代码展示了 HNSW 相对 IVF 的三大优势：第一，add_items 天然支持增量，不需要重建；第二，set_ef 运行时切换精度，无需重新建索引；第三，save/load 持久化后可以跨进程加载。注意 resize_index 需要预留空间，生产环境建议初始化时把 max_elements 设为预期的 1.5 倍，避免频繁 resize。

5 插入算法的内部逻辑

理解插入算法能帮你诊断"为什么我的召回率上不去"。HNSW 插入一个新节点 q 的过程如下：先随机生成 q 的最高层级 l；从最顶层入口点出发，用 ef=1 的贪心搜索下降到第 l+1 层，找到该层的最近节点作为入口；然后从第 l 层开始往下，每一层用 ef_construction 大小的候选列表搜索邻居，从中选出 M 个（底层 2M）建立双向边，如果被连的节点邻居数超限就触发"剪枝"（保留最分散的 M 个）。

剪枝策略是 HNSW 召回率的关键。hnswlib 用的是"启发式选邻居"：不只选最近的 M 个，而是选在空间上最分散的 M 个，避免所有邻居挤在一个方向导致搜索路径单一。这也是为什么 ef_construction 要足够大——候选池太小，剪枝没有选择空间，图的质量就上不去。

6 生产调参的实战经验

调 HNSW 参数时，建议遵循"先保召回，再压延迟"的原则。第一步固定 ef_construction=400、M=32，扫描 ef_search 找到能达到目标 recall（比如 0.95）的最小值。第二步如果延迟仍超标，尝试降低 M（比如到 16），但要重新建索引验证召回率没掉。第三步，如果内存紧张，考虑用 HNSW+PQ 混合索引（图结构存全精度向量坐标用 PQ 压缩）。

一个容易踩的坑：不同 distance metric 下 HNSW 表现差异大。cosine 和 L2 通常表现最好，内积（IP）在向量未归一化时召回率会明显下降，因为 IP 不满足三角不等式，贪心搜索的理论保证失效。解决方案是统一做 L2 归一化后用 cosine 或 L2，等价于 IP 且数值稳定。

7 本章小结

HNSW 用多层跳表结构把 NSW 的单层图升级为"粗导航 + 细导航"的分层导航，实现了 O(log N) 的搜索复杂度和亚毫秒级延迟，同时天然支持增量插入，这使它成为当今向量数据库的事实标准。三个参数中，M 决定图的稠密度（默认 16，高维用 32），ef_construction 决定建图质量（默认 200，一次性导入用 400），ef_search 决定查询精度与延迟（运行时可调，需 >= k）。相比 IVF，HNSW 在召回率和延迟上都更优，代价是更高的内存占用。掌握 hnswlib 的 API 后，迁移到 Milvus、Qdrant 等生产系统只是配置层面的差异，底层逻辑完全一致。
