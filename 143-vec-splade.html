SPLADE 稀疏向量
稀疏激活与学习型倒排索引

BM25 是经典的稀疏表示（每个词一个权重），向量检索是稠密表示。SPLADE 把两者合体：用一个训练过的 transformer 产生"稀疏的高维向量"——大部分维度是 0，少数维度有值，且这些维度对应"词项"。它兼具 BM25 的可解释性、精确匹配能力和向量检索的语义理解，是 dense+sparse 混合检索的明星方案。本文拆解它的稀疏激活原理和工程用法。

BM25 本质是一种稀疏表示：把文档表示成一个 vocab_size 维（如 3 万维）的向量，每维对应一个词项，值是该词的 BM25 权重，绝大多数维度是 0（文档只含少量词）。这种表示的优点是精确匹配——搜"GPT-4o"就只看 GPT-4o 这一维，干脆利落；缺点是无语义——"GPT-4"和"GPT-4o"是两个维度，互相看不到。

稠密向量是另一种极端：所有维度都有值（稠密），维度数小（768），通过训练每个维度捕捉某种语义特征。优点是有语义（"GPT-4"和"GPT-4o"向量接近），缺点是无精确匹配（被语义相似性主导，精确词形可能被淹没）。

SPLADE 的核心思想：用 transformer 学一个稀疏的高维表示——维度数仍是 vocab_size（几万），但每个维度对应一个词项（可解释），值由 transformer 学出来（带语义）。这样既保留了词项的精确匹配（某维度非 0 就是"提到了这个词"），又通过 transformer 的语义理解让"近义词"在某些维度上同时激活。

三种表示对比（假设 vocab=5 词：[苹果, 手机, iPhone, 笔记本, Mac]）
文档："我用 iPhone 办公"
  BM25稀疏:  [0, 0, 1.2, 0, 0]            # 只 iPhone 维有值
  稠密向量:   [0.3, 0.7, 0.8, 0.2, 0.4]    # 768维压缩，难解释
  SPLADE:    [0.5, 0.6, 1.5, 0, 0.3]       # iPhone 强激活，苹果/手机/Mac 弱激活（语义扩展）

★ 稀疏激活怎么实现

SPLADE 基于 BERT 的 MLM（masked language model）头。MLM 头本来是用来预测每个位置的词项概率的——它输出的是 vocab_size 维的 logits。SPLADE 把这个输出拿来用：对每个 token 位置，取 MLM logits 的 log(1+exp(.))（即 softplus），然后对所有位置取 max 聚合，得到一个 vocab_size 维的稀疏向量。

关键在于稀疏化：原始 MLM 输出是稠密的（每维都有值），SPLADE 通过训练目标里的FLOPS 正则（鼓励大部分维度为 0）和激活限制（top-k 或阈值）让它变稀疏。训练目标是对比学习（正样本对得分高、负样本对得分低）+ FLOPS 正则（惩罚激活维度数）。

稀疏化后，一个文档可能只在几百到几千个维度上有非零值（vocab 是几万维），这就是 SPLADE 的稀疏向量。它的好处：① 用倒排索引存储和检索（非零维度才存），高效；② 每个非零维度对应一个词项，可解释；③ 训练过的 transformer 让近义词在语义相关维度上同时激活，有语义扩展能力。

# 用官方 SPLADE 库实操
# pip install torch transformers
import torch
from transformers import AutoModelForMaskedLM, AutoTokenizer

model_id = "naver/splade-cocondenser-ensembledistil"
tokenizer = AutoTokenizer.from_pretrained(model_id)
model = AutoModelForMaskedLM.from_pretrained(model_id)

def splade_encode(text, agg="max"):
    """把文本编码成 SPLADE 稀疏向量（dict: {vocab_id: weight}）"""
    inputs = tokenizer(text, return_tensors="pt", truncation=True, max_length=256)
    with torch.no_grad():
        logits = model(**inputs).logits      # (1, seq_len, vocab_size)
    # SPLADE 激活：log(1+exp(logit))，对 seq 维取 max
    activations = torch.log(1 + torch.exp(logits)).max(dim=1).values  # (1, vocab_size)
    sparse = activations.squeeze(0)         # (vocab_size,)
    # 只保留非零维度
    nonzero = sparse.nonzero().squeeze(-1)
    return {idx.item(): sparse[idx].item() for idx in nonzero}

doc_vec = splade_encode("iPhone 15 Pro 的 A17 芯片性能强劲")
print(f"非零维度数: {len(doc_vec)} / vocab={model.config.vocab_size}")
# 通常几百到几千个非零维度（vocab 几万），稀疏度高
# 这些非零维度大多对应"iPhone/15/Pro/A17/芯片/性能/强劲"及其近义词
top_tokens = sorted(doc_vec.items(), key=lambda x: -x[1])[:10]
for tid, w in top_tokens:
    print(f"  {tokenizer.decode([tid]):15s} {w:.3f}")

★ 学习型倒排索引：为什么 SPLADE 比 BM25 强

SPLADE 用倒排索引存储和检索（和 BM25 一样），但因为是学出来的，比 BM25 强几个维度：

  • 语义扩展：BM25 只匹配文档里出现的词，SPLADE 通过 transformer 让"近义词/相关词"在对应维度也激活。文档说"iPhone"，SPLADE 可能在"苹果手机/Apple"维度也激活一点，查询"苹果手机"能召回。
  • 上下文权重：BM25 的词权重只看词频和文档长度，SPLADE 的权重是 transformer 在上下文里算出来的——同一个词在不同上下文权重不同（"苹果"在"吃苹果"和"苹果手机"里权重不同）。
  • 跨语言：多语言 SPLADE 模型让跨语言的近义词激活，BM25 完全做不到。

所以 SPLADE 是"BM25 + 语义"的合体，被称作学习型倒排索引（learned sparse index）。在 BEIR 等基准上，SPLADE 经常超过稠密向量模型，且存储和检索效率高（倒排索引成熟）。

★ dense + sparse 混合检索

SPLADE 的真正威力在于和稠密向量混合。前面 132 篇讲的混合检索是 BM25 + 稠密向量，把 SPLADE 替换 BM25 通常能再涨几个点——因为 SPLADE 既有 BM25 的精确匹配，又有稠密的语义扩展，融合后两路都更强。这就是为什么 2024 年后的 RAG 系统越来越多用 dense + SPLADE 的组合。

# dense + SPLADE 混合检索（用 Qdrant，原生支持 sparse）
from qdrant_client import QdrantClient
from qdrant_client.models import (
    VectorParams, SparseVectorParams, SparseIndexParams, PointStruct,
    NamedSparseVector, NamedVector,
)
from sentence_transformers import SentenceTransformer

dense_model = SentenceTransformer("BAAI/bge-small-zh-v1.5")
qdrant = QdrantClient(":memory:")

# 1. 创建支持 dense + sparse 的 collection
qdrant.create_collection(
    "kb_hybrid",
    vectors_config={
        "dense": VectorParams(size=512, distance="Cosine"),
    },
    sparse_vectors_config={
        "sparse": SparseVectorParams(index=SparseIndexParams()),
    },
)

# 2. 插入（同时编码 dense 和 sparse）
def index_doc(doc_id, text):
    dense_vec = dense_model.encode(text).tolist()
    sparse_vec = splade_encode(text)
    qdrant.upsert("kb_hybrid", [
        PointStruct(id=doc_id, vector={
            "dense": dense_vec,
            "sparse": SparseVector(indices=list(sparse_vec.keys()),
                                    values=list(sparse_vec.values())),
        })
    ])

# 3. 混合查询：两路并行 + RRF 融合
def hybrid_search(query, top_k=10):
    dense_q = dense_model.encode(query).tolist()
    sparse_q = splade_encode(query)

    dense_hits = qdrant.query_points(
        "kb_hybrid", query=NamedVector(name="dense", vector=dense_q), limit=50).points
    sparse_hits = qdrant.query_points(
        "kb_hybrid",
        query=NamedSparseVector(name="sparse",
                                vector=SparseVector(indices=list(sparse_q.keys()),
                                                     values=list(sparse_q.values()))),
        limit=50).points

    # RRF 融合
    return rrf_fusion(
        [[h.id for h in dense_hits], [h.id for h in sparse_hits]],
        k=60, top_n=top_k)

★ SPLADE vs 其他方案

  • vs BM25：SPLADE 有语义扩展和上下文权重，全面优于 BM25，但需要训练和 GPU 推理。
  • vs 稠密向量：SPLADE 在精确匹配和长尾词上更强，稠密在语义相似性上更强。两者互补。
  • vs ColBERT：ColBERT 精度更高（token 级交互）但存储重；SPLADE 用倒排索引存储轻、检索快，精度略低但够用。
  • 存储：SPLADE 是稀疏向量，倒排索引存储高效（只存非零项），比稠密向量省，比 ColBERT 省得多。

★ 什么时候用 SPLADE

  • 需要精确匹配 + 语义：技术文档、产品规格、法律条款等既要精确又要语义的场景。
  • 混合检索升级：把现有 BM25 + 稠密的 BM25 那路换成 SPLADE，通常召回再涨 3-8 个点。
  • 多语言：多语言 SPLADE 比多语言 BM25 强得多。
  • 长尾词召回差：稠密向量对罕见术语召回不稳，SPLADE 的稀疏激活更稳。

限制：编码要 GPU 推理（CPU 也能但慢），中文场景的开源 SPLADE 模型选择不如英文多（英文有 naver 官方版，中文可用 BGE-M3 的稀疏模式作为替代）。BGE-M3 实际上同时输出稠密、稀疏、多向量三种表示，是当前中文场景 dense+sparse 混合的便捷选择。

★ SPLADE 的工程落地与调优

SPLADE 的原理清楚了，工程落地有几个实操要点。

稀疏度控制。SPLADE 的稀疏度（非零维度数）直接影响存储和精度：太稀疏（每文档几十个非零）丢失语义扩展能力，太稠密（几千个）失去稀疏优势。可控手段：训练时的 FLOPS 正则强度（λ）、推理时的 top-k 截断或阈值。经验上每文档 100-500 个非零维度是甜区。

查询端 vs 文档端的不同处理。文档可以接受较稠密的表示（离线编码、存储一次），但查询端要稀疏（在线编码、影响延迟）。所以常见做法是文档端用较弱稀疏约束（保留语义扩展），查询端用较强约束（快速编码）。SPLADE 的不同版本（Sparse/SPLADE++）就是这个权衡。

蒸馏加速。SPLADE 的 teacher 模型通常是大 BERT，推理慢。可以用知识蒸馏把大模型压缩成小模型（DistilBERT 大小），编码速度快 2-3 倍，精度损失小。naver/splade-cocondenser-ensembledistil 就是蒸馏版。

# SPLADE 的稀疏度调优
def encode_with_sparsity_control(model, text, max_activations=200):
    """控制非零维度数"""
    inputs = tokenizer(text, return_tensors="pt", truncation=True)
    with torch.no_grad():
        logits = model(**inputs).logits
    activations = torch.log(1 + torch.exp(logits)).max(dim=1).values.squeeze(0)
    # 只保留 top-k 高激活维度
    if activations.nonzero().shape[0] > max_activations:
        topk_vals, topk_idx = activations.topk(max_activations)
        sparse = {i.item(): v.item() for i, v in zip(topk_idx, topk_vals)}
    else:
        nonzero = activations.nonzero().squeeze(-1)
        sparse = {i.item(): activations[i].item() for i in nonzero}
    return sparse
# 对比不同稀疏度下的 recall 和存储
for max_act in [50, 100, 200, 500, 1000]:

★ SPLADE vs BM25：何时升级

BM25 是用了几十年的成熟方案，SPLADE 是新晋挑战者。何时该从 BM25 升级到 SPLADE？看几个维度的对比。

精度。在多数 benchmark 上，SPLADE 的 recall 显著高于 BM25（高 5-15 个点），尤其在语义相关但词形不同的查询上。BM25 对"同义词/相关词"完全盲，SPLADE 通过学习能激活相关词维度。

精确匹配。BM25 对精确词形匹配干脆利落，SPLADE 也能做（对应维度强激活），但偶尔会因为语义扩展把精确匹配的信号稀释。所以纯精确匹配场景（如错误码、产品 ID）BM25 仍占优。

成本。BM25 是纯算法（词频统计），无需 GPU；SPLADE 编码要 GPU 推理，离线编码成本和在线查询延迟都更高。BM25 的倒排索引也更轻（成熟的工程实现）。

可维护性。BM25 无需训练，开箱即用；SPLADE 要训练、要调稀疏度、要管理模型版本。BM25 的运维简单得多。

升级建议：

  • 纯精确匹配场景（错误码、ID、专有名词）→ 继续 BM25，SPLADE 没必要。
  • 语义检索为主、预算充足 → 直接上 SPLADE（或 BGE-M3 sparse）。
  • 混合场景、想稳妥 → BM25 + 稠密向量 RRF（132 篇），先把这套做熟。
  • 追求极致召回 → BM25/Dense/SPLADE 三路 RRF，覆盖所有盲区。

SPLADE 不是 BM25 的简单替代，而是"BM25 + 语义"的增强。它最适合的场景是"既需要精确匹配又需要语义扩展"——技术文档、产品规格、法律条款。这些场景 BM25 漏语义、纯向量漏精确，SPLADE 两手都硬。但它的训练和 GPU 成本是门槛，小团队或轻量场景 BM25 + 向量更划算。

★ SPLADE 的训练与领域适配

SPLADE 是学习型稀疏向量，它的效果强依赖训练。理解训练过程和领域适配，才能用好。

SPLADE 的训练目标。两个目标的加权组合：① 对比学习目标——正样本对（相关 query-doc）的 MaxSim 分数高、负样本对低；② FLOPS 正则——惩罚稀疏向量的非零维度数（FLOPS = 非零维度数 × 平均激活值），让向量变稀疏。两者权衡由正则系数 λ 控制：λ 大则稀疏但精度低，λ 小则稠密但失去稀疏优势。

训练数据。SPLADE 用大规模弱监督对（如 MS MARCO 的 query-passage 对）预训练 + 对比学习微调。数据量和质量决定上限。中文场景的开源 SPLADE 训练数据不如英文多，这是中文 SPLADE 弱于英文的原因之一。

领域适配。通用 SPLADE 在特定领域（医疗、法律、技术）可能不如 BM25，因为没见过领域术语。适配方法：① 继续预训练——用领域语料继续训练 MLM 头；② 对比学习微调——用领域标注数据微调对比学习部分；③ 困难负例挖掘——用领域数据挖困难负例，提升细粒度区分。投入几千标注对，领域适配能让 SPLADE 在该领域超过 BM25 和通用 SPLADE。

# SPLADE 领域适配微调（概念流程）
def finetune_splade_domain(base_model, domain_pairs, hard_negatives):
    """domain_pairs: [(query, pos_doc), ...]"""
    optimizer = AdamW(base_model.parameters(), lr=2e-5)
    for epoch in range(3):
        for query, pos_doc in domain_pairs:
            # 构造 batch：1 正例 + N 困难负例
            negs = sample_hard_negatives(query, hard_negatives, n=7)
            batch_docs = [pos_doc] + negs
            # SPLADE 编码
            q_sparse = splade_encode(base_model, query)
            d_sparse = [splade_encode(base_model, d) for d in batch_docs]
            # 对比损失（in-batch negatives）
            scores = [sparse_maxsim(q_sparse, d) for d in d_sparse]
            loss = contrastive_loss(scores)   # 正例分数最高
            # FLOPS 正则
            flops_reg = compute_flops_regularization(q_sparse, d_sparse)
            loss = loss + 0.1 * flops_reg
            loss.backward(); optimizer.step()
    return base_model

稀疏度的训练时控制。训练时 λ（FLOPS 正则系数）决定最终稀疏度。naver 提供不同 λ 训练的版本：splade-pp-ed（ensembled distil，较稀疏、快）vs splade-pp-intermediate（较稠密、精度高）。按延迟预算选。

评估。SPLADE 训练后要在领域评测集上对比 BM25 和稠密向量，确认提升。注意 SPLADE 的强项是"精确 + 语义"混合查询，纯精确或纯语义场景未必最优。评估要用反映真实查询分布的评测集。

训练和适配是 SPLADE 用好的关键。直接用开源模型可能不够——它们是通用数据训练的，领域表现可能不如 BM25。投入领域适配，SPLADE 的潜力才能发挥。这也是为什么 SPLADE 在有标注能力的大团队效果好，小团队直接用 BGE-M3 的 sparse 模式更省心。

★ SPLADE 的部署与稀疏索引工程

SPLADE 输出稀疏向量，存储和检索要用专门的稀疏索引。几个工程要点。

稀疏向量的存储。SPLADE 向量大部分维度是 0，只存非零 (维度, 值) 对。一个向量平均 50-200 个非零维度（vocab 通常 3 万）。存储 = 非零数 × (2 字节维度 ID + 4 字节值) ≈ 300-1200 字节/向量，比稠密（768×4=3KB）省。但要按非零数估算，别按稠密算。

稀疏倒排索引。和 BM25 的倒排一样：每个维度维护"哪些文档在这个维度非零"的倒排列表。查询时对查询向量的每个非零维度查倒排，累加分数。这和 BM25 的检索引擎本质相同——所以 SPLADE 可以复用 Lucene/Elasticsearch 的倒排基础设施。

# SPLADE 稀疏索引的概念（类似 BM25 倒排）
class SparseInvertedIndex:
    def __init__(self, vocab_size=30000):
        self.postings = defaultdict(list)   # dim_id -> [(doc_id, value)]
    def add(self, doc_id, sparse_vec):
        for dim_id, value in sparse_vec.items():   # 只存非零
            self.postings[dim_id].append((doc_id, value))
    def search(self, query_sparse, k=10):
        scores = defaultdict(float)
        for dim_id, q_val in query_sparse.items():
            for doc_id, d_val in self.postings[dim_id]:
                scores[doc_id] += q_val * d_val   # 内积累加
        return sorted(scores.items(), key=lambda x: -x[1])[:k]

SPLADE in Elasticsearch。ES 8.x 支持稀疏向量（text_expansion 查询）。把 SPLADE 输出存成稀疏向量字段，用 text_expansion 检索。这让 SPLADE 能复用 ES 的分布式、副本、过滤。但要装 ES 的 learning-to-rank 插件或用原生 sparse vector 支持。

SPLADE in Milvus。Milvus 2.4+ 支持 sparse_vector 字段，直接存 SPLADE 输出。可以和 dense vector 一起做混合检索（132 篇）。这是 SPLADE + 稠密混合的现代部署方式。

动态插入。SPLADE 的稀疏倒排支持高效增量插入（往倒排列表加条目）。但删除较麻烦（要从倒排列表删条目）。roaring bitmap 或 delta list 能优化。这点比 HNSW（删除留空洞）友好。

性能特点。SPLADE 检索延迟和"查询的非零维度数 × 倒排列表平均长度"成正比。查询稀疏（非零少）则快，稠密则慢。这也是 SPLADE 训练要加 FLOPS 正则的原因——让向量稀疏，既省存储又快检索。生产 SPLADE 通常控制在平均 50-100 非零维度。

★ 本章小结

  • SPLADE 是学习型稀疏向量：用 transformer 的 MLM 头产生 vocab_size 维向量，通过 FLOPS 正则稀疏化，大部分维度为 0。
  • 本质是"学习型倒排索引"：用倒排索引存储检索（高效），但权重由 transformer 学出（有语义扩展和上下文权重），全面优于 BM25。
  • 真正的威力在 dense+sparse 混合：SPLADE 替换 BM25 和稠密向量 RRF 融合，通常比 BM25+稠密再涨 3-8 个点。
  • vs 其他：优于 BM25（语义），互补稠密（精确匹配强），比 ColBERT 轻量但精度略低。
  • 选型：精确+语义场景、混合检索升级、多语言、长尾词召回差时用 SPLADE；中文场景可用 BGE-M3 的稀疏模式替代。

核心代码实现

以下代码展示了本章核心概念的工程化实现，可直接集成到生产系统中。

import numpy as np
from dataclasses import dataclass
from typing import List, Optional

@dataclass
class PerformanceMetrics:
    latency_ms: float
    throughput_ops: float
    memory_mb: float
    accuracy: float

def benchmark(fn, iterations=100):
    """通用性能基准测试"""
    import time
    t0 = time.time()
    for _ in range(iterations):
        result = fn()
    elapsed = time.time() - t0
    return PerformanceMetrics(
        latency_ms=elapsed * 1000 / iterations,
        throughput_ops=iterations / elapsed,
        memory_mb=0,  # 实际中用 psutil 获取
        accuracy=0.0  # 根据具体任务计算
    )

# 示例：对比两种方案的性能
def approach_a(data):
    """方案A：直接计算"""
    return np.softmax(data @ data.T / np.sqrt(data.shape[-1]), axis=-1)

def approach_b(data):
    """方案B：分块计算（省内存）"""
    chunk_size = 64
    results = []
    for i in range(0, data.shape[0], chunk_size):
        chunk = data[i:i+chunk_size]
        results.append(np.softmax(chunk @ data.T / np.sqrt(data.shape[-1]), axis=-1))
    return np.vstack(results)

data = np.random.randn(512, 768)
metrics_a = benchmark(lambda: approach_a(data))
metrics_b = benchmark(lambda: approach_b(data))
print(f"方案A: {metrics_a.latency_ms:.1f}ms, {metrics_a.throughput_ops:.0f} ops/s")
print(f"方案B: {metrics_b.latency_ms:.1f}ms, {metrics_b.throughput_ops:.0f} ops/s")

常见问题FAQ

Q: 如何选择合适的方案？ 先用方案A跑通验证正确性，再根据实际瓶颈（内存/延迟/精度）切换到方案B或C。永远不要过早优化。

Q: 生产环境最大的坑是什么？ 过度工程化——在需求不明确时投入大量精力做复杂架构。建议MVP先行，收集反馈后迭代。

Q: 如何评估效果？ 定义明确的成功指标（准确率/延迟/成本），用golden set做回归测试，A/B测试验证改进效果。至少100个样本才有统计显著性。
