知识库搭建全景

一份覆盖 RAG / Agentic RAG / GraphRAG / 本体的选型与实战地图

深度进阶★★★

为什么"知识库"是大模型落地的第一战场

大模型有两个先天缺陷：① 知识截止（训练数据有日期，不知道今天的事）；② 幻觉（不知道就编）。这两个缺陷在企业场景里是致命的——客服不能编造退换货政策，医生不能编造用药剂量，律师不能编造法条。解决办法不是"训个更大的模型"，而是"给模型外挂一个知识库"——这就是 RAG（检索增强生成）的本质：让模型在回答前先去查资料。

但"搭一个知识库"远比想象中复杂。 naive RAG（切块→嵌入→向量检索→拼接 prompt）只是最原始的形态，召回率往往只有 50-60%。真正生产可用的知识库，要解决文档解析、智能切块、混合检索、重排序、多跳推理、知识图谱融合、自校正等一系列问题。本章把这些技术演进梳理成一条清晰的路线，并对接 GitHub 上当前最主流的开源项目，让你能按需选型。

💡 要点
本章定位：知识库技术的"地图+选型指南"。从朴素 RAG 的局限出发，逐级上升到 Advanced RAG（混合检索/重排序）、Agentic RAG（自规划/自校正检索）、GraphRAG（知识图谱+社区层级）、本体驱动知识库（语义结构化），最后给出 5 大开源项目的横向对比与选型决策树。

第一代：朴素 RAG（Naive RAG）及其四大痛点

最基础的 RAG 流水线是：文档 → 固定大小切块 → embedding → 存向量库 → 查询时取 top-k → 拼进 prompt → LLM 生成。这套流程在 demo 里看着很美，但上生产会暴露四个经典痛点：

痛点表现根本原因
切块太粗关键信息被切断或淹没在噪声里固定大小切块不懂语义边界
检索不准召回的 chunk 和问题无关纯向量相似度≠语义相关，尤其在多跳问题上
无法推理"A 的父亲的儿子"查不到单次检索只能找表面相似，不会组合多步
不知道自己不知道检索到垃圾也照答（幻觉）没有质量评估/回退机制

这四个痛点催生了 RAG 的三代演进：Advanced RAG 解决前两个（更聪明的切块+混合检索+重排序），Agentic RAG 解决后两个（让 agent 自主规划检索+自校正），GraphRAG 和本体则从"结构化知识"角度补足向量检索的盲区。

第二代：Advanced RAG——混合检索与重排序

Advanced RAG 的核心改进是把"单一向量检索"升级为"混合检索 + 重排序"：

Advanced RAG 检索流水线

查询 → ┌─ 稠密向量检索（语义相似，top 20）
       ├─ 稀疏检索 BM25（关键词精确匹配，top 20）
       └─ (可选) 元数据过滤（按来源/日期/权限筛选）
         ↓
       候选合并（去重，~40 条）
         ↓
       重排序 Reranker（cross-encoder 精排，如 bge-reranker）
         ↓
       取 top 5 喂给 LLM

关键：向量检索负责"召回广度"（语义相关都捞回来），
      reranker 负责"排序精度"（把真正相关的排到前面）。
      这套组合能把召回率从 60% 提到 85%+。

为什么需要 reranker？因为向量检索用的 bi-encoder（查询和文档独立编码）速度快但精度有限；reranker 用的 cross-encoder（查询和文档拼接后一起编码）精度高但慢。所以标准做法是：bi-encoder 粗召回大量候选，cross-encoder 精排取 top-k。这是用"两阶段"换"速度+精度"的权衡，和搜索引擎的"召回+排序"是同构的。

智能切块：比固定大小重要得多

Advanced RAG 的另一个关键改进是"智能切块"。固定大小（如每 512 token 切一刀）会粗暴切断语义单元（把一个完整的论证切成两半）。生产级做法是：① 按 Markdown 标题/HTML 结构切（保留文档的天然层级）；② 语义切块（用 embedding 相似度判断边界）；③ 父子切块（parent-child，检索小块、返回大块上下文）。LlamaIndex 和 RAGFlow 在这块做得最细。

第三代：Agentic RAG——让检索"会思考"

Agentic RAG 是 2024-2025 年最重要的 RAG 进化。核心思想：**把"检索"从一个固定的流水线，升级成一个能自主决策的 agent**。它不再是无脑"查一次向量库"，而是会：判断要不要查、查哪里、查得对不对、要不要重查。Agentic RAG 有三个标志性子模式：

模式核心机制代表论文/实现
Self-RAG模型生成"反思 token"自评：要不要检索？结果相关吗？回答有依据吗？Self-RAG (arXiv:2310.11511)
Corrective RAG (CRAG)检索评估器打分；检索质量差时回退到 web 搜索CRAG (arXiv:2401.15884)
Agentic RAG (Router)agent 自主规划：拆解多跳问题、路由到不同数据源、迭代检索LangGraph / LlamaIndex Agents

Self-RAG：让模型学会"自我怀疑"

Self-RAG 训练模型在生成时额外输出"反思 token"：[Retrieve]（需要检索吗）、[IsRel]（检索结果相关吗）、[IsSup]（回答有依据吗）。这让模型能在"知道就答、不知道就查、查到垃圾就不答"之间自主切换，而不是无脑检索+无脑生成。本质是给 RAG 加了一个"元认知"层。

CRAG：检索质量差就换数据源

Corrective RAG（CRAG）的关键创新是一个"检索评估器"：它给每条检索结果打分（相关/模糊/无关）。如果本地知识库检索质量差（大量"模糊/无关"），CRAG 不硬答，而是**回退到 web 搜索**（如 Tavili/Google）补充信息。这解决了"知识库没覆盖的问题硬编造"的痛点——查不到就去网上查，而不是瞎编。

Agentic RAG：多跳推理的 agent 循环

完整的 Agentic RAG 是一个 agent 循环：agent 收到复杂问题（如"对比 A 公司和 B 公司 2024 年的营收增长"），会自主拆解成子问题（查 A 营收、查 B 营收、计算增长、对比），每个子问题路由到合适的数据源（财务库/web/向量库），检索后评估质量，不够就重查或改写 query，最后综合多步结果作答。这是用"agent 的规划+反思能力"彻底替代"固定流水线"。LangGraph 和 LlamaIndex 的 Workflows 是当前实现 Agentic RAG 的主流框架。

💡 要点
Agentic RAG 的本质转变：从"检索是流水线的一环"变成"检索是 agent 的工具"。agent 像一个会查资料的研究员——自主决定查什么、查几次、查得对不对。代价是延迟和成本上升（多轮 LLM 调用），换来的是多跳推理能力和"知之为知之"的可靠性。

GraphRAG：知识图谱让 RAG 看见"关系"

向量检索有一个根本盲区：它只懂"语义相似"，不懂"关系结构"。问"A 的父亲的儿子是谁"，向量检索会找含"A""父亲""儿子"的 chunk，但无法做"A→父亲→儿子"的关系推理。GraphRAG（微软开源，[microsoft/graphrag](https://github.com/microsoft/graphrag)，~34K stars）用知识图谱补足这个盲区。

GraphRAG 的工作流程（官方文档 [microsoft.github.io/graphrag](https://microsoft.github.io/graphrag/)）：① 用 LLM 从文档里抽取实体和关系，构建知识图谱；② 对图做**社区检测**（community detection，如 Leiden 算法），把节点聚成层级社区；③ 为每个社区生成摘要；④ 查询时，既可在社区摘要层做"全局理解"（global search，适合"总结性"问题），也可在图上做"局部推理"（local search，适合"具体实体关系"问题）。

GraphRAG vs 向量 RAG 的互补

向量 RAG 擅长：  "找和这个问题语义相似的段落"
                → 适合事实查询、定义查询

GraphRAG 擅长： "跨文档的关系推理 + 全局总结"
                → "这部小说里所有角色的关系网"
                → "这个行业上下游的主要玩家"
                → "A 公司通过哪些子公司持有 B"

两者互补：向量负责"找段落"，图负责"推理关系"。
生产级知识库常把两者并联（hybrid retrieval）。

GraphRAG 的一个关键能力是"社区层级摘要"——它能把海量文档压缩成多层级的话题地图，回答"这个数据集整体在讲什么"这类全局性问题，这是纯向量 RAG 完全做不到的（向量检索只能找局部相似，无法做全局总结）。代价是构建成本高（要跑大量 LLM 抽取实体/关系/摘要），适合"数据相对静态、值得一次性深加工"的场景。

本体驱动知识库：给知识加上"语义骨架"

本体（ontology）是知识库的"语义骨架"——它定义了一个领域里有哪些实体类型、它们有哪些属性、之间有什么关系。比如医疗本体定义了"疾病—症状—药物—禁忌"的关系结构；金融本体定义了"公司—子公司—产品—财报"的结构。有了本体，知识库就不是一堆散乱的 chunk，而是一个**结构化、可推理**的知识网络。

本体和 GraphRAG 的关系：GraphRAG 默认是"自由抽取"（LLM 自己决定抽什么实体什么关系），而本体驱动是"按预定义 schema 抽取"（你先定义好本体，LLM 按本体填充）。后者的好处是抽取结果规范、可做严格的逻辑推理（如"所有对青霉素过敏的患者，不能用含青霉素的药"），代价是要先设计本体。GraphRAG 社区有大量关于[预定义本体的讨论](https://github.com/microsoft/graphrag/discussions/2196)，RDF/OWL 是传统语义网的本体标准，Neo4j 是图数据库的载体。

知识表示结构推理能力典型场景
纯向量 chunk无结构文本块语义相似（弱）通用问答
GraphRAG（自由图）自由知识图谱关系遍历（中）关系发现、全局总结
本体驱动（schema 图）规范化的类型/属性/关系逻辑推理（强）医疗/法律/金融的严格规则

LLM Wiki：让大模型自己维护知识库

一个更新的趋势是"LLM Wiki"——不是人手动维护知识库，而是让 LLM 自主地阅读、整理、更新一个结构化的知识库（类似维基百科）。它结合了 RAG（检索）+ agent（自主整理）+ 知识图谱（结构化存储）。代表思路如 Stanford 的 STORM（多视角生成维基条目）：agent 模拟多个"编辑"，从不同角度调研一个主题，各自写草稿，再综合成一篇结构化的 wiki 文章。这种"自维护知识库"是知识库技术的未来方向——知识不再静态，而是 agent 持续学习和整理的活体。

GitHub 五大开源项目横向对比（2026 实测 stars）

下面是当前最主流的 5 个知识库/RAG 开源项目，stars 为 2026 年实测（变动频繁，以仓库实时为准）：

项目Stars定位最适合
RAGFlow (infiniflow)~83K深度文档理解 + RAG 引擎复杂文档（表格/版式/扫描件）的企业知识库

怎么选

① **要快速搭一个能用的私有问答（不想写代码）** → AnythingLLM。开箱即用，拖文档进去就能问，适合个人/小团队。② **文档很复杂（PDF 表格、扫描件、混乱版式）** → RAGFlow。它的深度文档理解（Deep Document Understanding）是强项，能把表格、版式正确解析成可检索的结构。③ **要深度定制 RAG/Agentic RAG 流水线** → LlamaIndex。它的索引类型、检索器、agent workflow 最丰富，适合有开发能力的团队做精细控制。④ **需要关系推理/全局总结** → GraphRAG（搭配向量 RAG 并联）。⑤ **要生产级可控、模块化、好测试** → Haystack 2.x。

💡 要点
选型决策树：不想写代码→AnythingLLM；文档复杂→RAGFlow；要定制→LlamaIndex；要关系推理→GraphRAG；要生产可控→Haystack。现实里常见组合：用 RAGFlow/LlamaIndex 做检索引擎，用 GraphRAG 补关系推理，用 AnythingLLM 做前端——各取所长。

知识库的工程化：不止于检索

一个生产级知识库还有一堆检索之外的工程问题：① **权限控制**（不同用户只能查自己有权限的文档，要在检索层做过滤）；② **增量更新**（文档改了，要重建对应 chunk 的索引，不能全量重建）；③ **评估闭环**（用 RAGAS 等工具量化召回率/忠实度/相关性，持续优化）；④ **缓存**（高频问题的答案缓存，避免重复检索+生成）；⑤ **多租户**（一套索引服务多个团队/客户，隔离数据）。这些往往比"选哪个 RAG 框架"更决定项目成败。

本章小结

知识库技术经历了三代演进：朴素 RAG（固定切块+向量检索，召回率低）→ Advanced RAG（混合检索+重排序+智能切块，召回率 85%+）→ Agentic RAG（Self-RAG/CRAG/Router agent，会思考的检索，支持多跳推理）。GraphRAG 用知识图谱补足"关系推理"盲区，本体驱动给知识加"语义骨架"，LLM Wiki 让知识库"自维护"。开源选型：AnythingLLM（开箱即用）、RAGFlow（复杂文档）、LlamaIndex（定制）、GraphRAG（关系推理）、Haystack（生产可控）。知识库是大模型落地的第一战场，也是工程师价值最高的领域。下一章讲智能体落地的两大核心痛点：安全与交互。
