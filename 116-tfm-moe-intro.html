MoE 混合专家入门
用稀疏激活实现「大而不贵」

一个 671B 参数的 DeepSeek-V3，推理时每个 token 只激活 37B——这就是 MoE 的魔法。它把一个巨大的 FFN 拆成几十个小 FFN（专家），每个 token 只调用其中两个。结果是：模型容量像 671B 一样大，计算成本却像 37B 一样便宜。本章从直觉到机制，把稀疏激活、容量因子、负载均衡这三大核心概念彻底讲透。

★ 一、稠密模型的困境

在讲 MoE 之前，先看清传统「稠密」模型面临什么瓶颈。Transformer 的每一层有两个主要计算：attention 和 FFN。其中 FFN 通常占整个模型参数的 60% 到 70%，是计算和显存的大头。稠密模型的特点是：每个 token 都要经过 FFN 的全部参数。

这意味着模型参数量和计算量是绑定的——参数翻倍，计算也翻倍。所以训练一个 700B 的稠密模型，每个 token 的计算量是 70B 模型的 10 倍，训练成本和推理延迟都爆炸。scaling law 告诉我们参数越多越好，但计算预算是有限的，怎么破这个「参数越多计算越贵」的死结？

1.1 MoE 的核心洞察

MoE（Mixture of Experts，混合专家）的答案是：解耦参数量和计算量。把一个巨大的 FFN 拆成 N 个小 FFN（叫专家，expert），再用一个路由器（router/gate）决定每个 token 该去哪个专家。每个 token 只激活 top-k 个专家（比如 N=8 里选 2 个），所以虽然总参数是 N 倍，但计算量只有 k/N。

用一个生活类比：稠密模型像是一个全能选手，每个问题都要动用全部脑力，累且慢；MoE 像是一个专家团队，有 8 个不同领域的专家，每次根据问题派给最合适的 2 个去处理。团队整体能力远超单个全能选手（因为可以容纳更多专长），但每次处理的成本却更低（只用 2 个人而非 8 个）。

这正是 DeepSeek-V3、Mixtral、GPT-4 采用 MoE 的根本原因：在不增加单 token 计算量的前提下，大幅增加模型的总参数容量。

★ 二、稀疏激活：MoE 的核心机制

稀疏激活（sparse activation）是 MoE 区别于稠密模型的根本特征。理解它，就理解了 MoE 为什么能「大而不贵」。

2.1 从稠密 FFN 到稀疏 MoE 层

回顾一下稠密 FFN 的结构：输入 x 经过两个线性层和一个激活函数，输出 y = W2 * activation(W1 * x)。这里所有参数都被用上。

MoE 层把这个 FFN 替换成 N 个并行的 FFN（专家 E1...EN），外加一个路由器 G。对每个 token，路由器根据 token 表示算出 N 个分数，选分数最高的 k 个专家，把 token 分别送进去，再把这 k 个专家的输出按路由分数加权求和。数学上：y = sum_{i in top-k} G(x)_i * E_i(x)。其中 G(x) 是路由器对 token x 输出的 softmax 分布。

2.2 稀疏在哪里

关键在于「top-k 选择」。如果 N=8、k=2，那么每个 token 只会进入 2 个专家，另外 6 个专家的参数完全不参与计算。这就是稀疏——大部分参数在每个 token 上是「睡着」的。从计算图看，等效计算量只有稠密版的 2/8 = 25%。但模型的总参数量是 8 倍，所以「参数容量」是稠密版的 8 倍。

这种「容量大、计算省」的特性，让 MoE 在固定计算预算下能用更多参数，从而获得更好的 scaling law。研究显示，在同等训练 FLOPs 下，MoE 模型的效果显著优于稠密模型。

★ 三、容量因子：吞吐与丢弃的平衡

稀疏激活带来了一个新问题：负载不均。如果所有 token 都想挤进同一个专家（比如代码 token 都想进「代码专家」），那个专家就会过载。为了在 GPU 上高效并行，MoE 引入了「容量因子」（capacity factor）来限制每个专家能处理的 token 数。

3.1 什么是容量因子

容量因子 C 定义了每个专家在一个 batch 内最多处理的 token 数：容量 = (batch_size * seq_len / N) * C。也就是说，每个专家的「容量桶」大小是「平均分配数 × C」。如果 C=1，每个专家只能处理平均份额；C=1.25 允许 25% 的超额。超过容量的 token 会被丢弃（drop），直接透传到下一层。

3.2 容量因子的取舍

C 越大，丢弃越少，模型质量越好，但显存和计算开销越大（桶要开得更大）；C 越小，效率越高但丢弃越多，质量下降。训练时常用 C=1.25 到 1.5，推理时如果用动态 batching 可以把 C 设得很大甚至不限（因为推理一个序列内 token 不会挤同一专家到过载）。

token 被丢弃是个损失——它相当于在这一层「跳过」了 FFN 处理。所以负载均衡（让 token 均匀分布到各专家）是 MoE 训练的头等大事，我们下一节专门讲。

★ 四、负载均衡：MoE 训练的头号难题

MoE 训练最容易出问题的就是负载不均：路由器学会了把所有 token 都送到少数几个专家，导致大部分专家闲置（参数浪费），少数专家过载（token 被丢弃）。这会让模型效果大幅下降。

4.1 为什么会负载不均

路由器是端到端训练的，一开始随机初始化时如果某几个专家偶然表现好一点，路由器就会更多地往它们派 token，它们得到更多训练数据变得更好，路由器更偏好它们——这是个正反馈循环，最终坍缩到「赢者通吃」的局面。这和上一章讲的重复问题类似，都是缺乏负反馈导致的。

解决思路也是引入负反馈：通过一个额外的损失函数（auxiliary loss，辅助损失）惩罚负载不均，强制路由器把 token 均匀分配。

4.2 Switch Transformer 的辅助损失

Switch Transformer 论文提出了最经典的辅助损失。设 batch 内 token 总数为 T，专家数为 N，路由器对 token t 选中的专家为 e(t)（只选 1 个的情况）。定义两个量：每个专家 i 收到的 token 比例 f_i = (1/T) * sum_t [e(t)==i]（实际负载），以及每个专家 i 的平均路由概率 P_i = (1/T) * sum_t G(x_t)_i（路由器分配的概率）。

辅助损失定义为：L_aux = alpha * N * sum_i (f_i * P_i)。这是一个 N 个专家的加权和，当所有专家负载和概率都均匀（都是 1/N）时取最小值 alpha。alpha 是一个很小的系数（如 0.01），确保辅助损失只是「轻微」地推动均衡，不喧宾夺主。训练时把它加到主语言建模损失上。

直觉上，f_i 衡量「实际有多少 token 进了专家 i」，P_i 衡量「路由器有多想往专家 i 派」。两者乘积越大说明专家 i 越过载。最小化这个乘积之和，就逼迫路由器往冷门专家派 token，实现均衡。

4.3 用代码理解辅助损失

把上面的公式直接翻译成代码，你会看到这个损失是如何惩罚不均衡的。下面这段代码构造一个极端不均衡的路由分布，观察 auxiliary loss 的大小变化。

import torch
import torch.nn.functional as F

def switch_aux_loss(router_probs, expert_assignment, num_experts, alpha=0.01):
    """
    Switch Transformer 的辅助损失。
    router_probs: (num_tokens, num_experts) 路由器输出的概率（带梯度）
    expert_assignment: (num_tokens,) 每个 token 实际选中的专家 id
    """
    num_tokens = router_probs.shape[0]
    # P_i：每个专家收到的平均路由概率（可导，带梯度）
    P = router_probs.mean(dim=0)                              # (num_experts,)
    # f_i：每个专家实际收到的 token 比例（硬分配，停梯度）
    one_hot = F.one_hot(expert_assignment, num_experts).float()  # (num_tokens, num_experts)
    f = one_hot.mean(dim=0).detach()                          # (num_experts,)
    # 辅助损失：N * sum_i (f_i * P_i)
    loss = num_experts * (f * P).sum()
    return alpha * loss

# 模拟两种路由情况
num_experts = 4
num_tokens = 100
torch.manual_seed(0)

# 情况 1：严重不均衡——几乎所有 token 都去专家 0
probs_unbalanced = torch.tensor([[0.9, 0.04, 0.03, 0.03]] * num_tokens)
assign_unbalanced = torch.zeros(num_tokens, dtype=torch.long)  # 全选专家 0
loss_unbal = switch_aux_loss(probs_unbalanced, assign_unbalanced, num_experts, alpha=1.0)

# 情况 2：完全均衡——各专家平均分配
probs_balanced = torch.tensor([[0.25, 0.25, 0.25, 0.25]] * num_tokens)
assign_balanced = torch.tensor([i % num_experts for i in range(num_tokens)])
loss_bal = switch_aux_loss(probs_balanced, assign_balanced, num_experts, alpha=1.0)

print(f"不均衡情况的辅助损失: {loss_unbal.item():.4f}")
print(f"均衡情况的辅助损失:   {loss_bal.item():.4f}")
print(f"理想最小值（完全均匀时）: {1.0:.4f}")
# 不均衡时 loss 远大于 1，均衡时接近 1，这就是辅助损失的惩罚机制

运行后你会清楚地看到：不均衡情况的损失远大于均衡情况。这就是 auxiliary loss 逼迫路由器均衡分配的数学力量。实际训练时把主损失（语言建模的交叉熵）和这个小系数的 aux_loss 加起来一起优化即可。

★ 五、从零实现一个简单 MoE 层

把概念落到代码。下面我们用 PyTorch 实现一个最小可运行的 MoE 层，包含路由、top-k 选择、专家计算、加权输出。这是理解 Mixtral、DeepSeek 架构的基础。

import torch
import torch.nn as nn
import torch.nn.functional as F

class Expert(nn.Module):
    """一个专家就是一个普通的 FFN。"""
    def __init__(self, d_model, d_ff):
        super().__init__()
        self.w1 = nn.Linear(d_model, d_ff)
        self.w2 = nn.Linear(d_ff, d_model)

    def forward(self, x):
        return self.w2(F.gelu(self.w1(x)))

class MoELayer(nn.Module):
    def __init__(self, d_model, d_ff, num_experts=8, top_k=2):
        super().__init__()
        self.num_experts = num_experts
        self.top_k = top_k
        self.router = nn.Linear(d_model, num_experts)
        self.experts = nn.ModuleList(
            [Expert(d_model, d_ff) for _ in range(num_experts)]
        )

    def forward(self, x):
        """
        x: (batch, seq_len, d_model)
        返回: (batch, seq_len, d_model)
        """
        B, S, D = x.shape
        x_flat = x.view(-1, D)                       # (B*S, D)
        gate_logits = self.router(x_flat)            # (B*S, num_experts)
        gate_probs = F.softmax(gate_logits, dim=-1)

        # 每个 token 选 top_k 个专家
        topk_probs, topk_idx = gate_probs.topk(self.top_k, dim=-1)
        # 归一化 top-k 概率（让它们加起来等于 1）
        topk_probs = topk_probs / topk_probs.sum(dim=-1, keepdim=True)

        out = torch.zeros_like(x_flat)
        for i in range(self.top_k):
            expert_idx = topk_idx[:, i]              # 每个 token 的第 i 个选择
            weight = topk_probs[:, i].unsqueeze(-1)  # 该选择的权重
            # 把每个 token 送进它选中的专家
            for e in range(self.num_experts):
                mask = (expert_idx == e)
                if mask.any():
                    expert_out = self.experts[e](x_flat[mask])
                    out[mask] += weight[mask] * expert_out
        return out.view(B, S, D)

# 测试
torch.manual_seed(0)
moe = MoELayer(d_model=64, d_ff=256, num_experts=8, top_k=2)
x = torch.randn(2, 10, 64)
y = moe(x)
print(f"输入形状: {x.shape}, 输出形状: {y.shape}")
print(f"参数量: {sum(p.numel() for p in moe.parameters())}")
# 注意：实际计算只激活了 top_k=2 个专家，等效计算量是稠密 FFN 的 2/8

这段代码用了一个简单粗暴的双重 for 循环把 token 分发到专家，便于理解但效率低（实际框架如 Megatron、vLLM 用 grouped GEMM 或 scatter/gather 优化）。但它完整展示了 MoE 的核心机制：路由、top-k 选择、归一化、加权聚合。Mixtral、DeepSeek 的 MoE 层在数学上和这个完全一致。

★ 六、MoE 与稠密模型的对比

把两者摆在一起对比，能更清楚地看到 MoE 的优势和代价。

  • 参数效率：MoE 用更少的计算达到更好的效果。Mixtral 8x7B 总参数 47B，但激活参数只有 13B，效果对标 70B 级稠密模型。
  • 训练成本：同样效果，MoE 训练 FLOPs 约为稠密模型的三分之一到一半。这是为什么开源社区能负担起 Mixtral、DeepSeek 的训练。
  • 推理显存：MoE 的劣势——所有专家都要常驻显存，即使每次只用 2 个。47B 的 Mixtral 推理时要加载 47B 参数，比 13B 稠密模型吃显存。
  • 推理速度：单 token 的计算量小（只 2 个专家），但访存大（要读 2 个专家的权重）。在 memory-bound 场景，MoE 的速度优势会被访存抵消，需要 PagedAttention、专家并行等优化。
  • 训练稳定性：MoE 训练更难调，路由坍缩、负载不均、专家利用率低都是常见坑，需要辅助损失和精细的超参调整。

★ 七、MoE 的工程变体

MoE 在大模型里已经演化出多种变体，了解它们的差异有助于读论文和选型。

  • Switch Transformer：Google 2021 的工作，把 top-k 简化到 top-1（每 token 只去 1 个专家），极致简化路由，适合超大规模训练。
  • GShard：最早把 MoE 用到千亿级规模的方案，提出了 expert parallelism（专家并行）和辅助损失的工程实现。
  • Mixtral：Mistral 的开源 MoE，top-2 路由，8 个专家，证明了稀疏 MoE 在开源社区也能达到顶级效果。
  • DeepSeekMoE：采用「细粒度专家」（更多但更小的专家）+「共享专家」（永远激活的专家处理通用知识），进一步降低冗余，是当前效率最高的 MoE 设计之一。
  • 专家并行：跨 GPU 分布专家（每个 GPU 放几个专家），通过 all-to-all 通信交换 token，是训练超大 MoE 的关键工程。

★ MoE Top-K Gating 实现

MoE（Mixture of Experts）的核心是 Top-K 门控路由：每个 token 只激活少数专家，总参数大但单次计算量小。下面实现这个路由机制并观察负载分布：

import torch
import torch.nn as nn
import torch.nn.functional as F

class TopKRouter(nn.Module):
    """MoE 的 Top-K 路由：每个 token 选 K 个专家，其余专家不激活。
    这是 MoE 稀疏激活的核心机制(Mixtral 用 K=2, 8 个专家)。
    """
    def __init__(self, hidden: int, n_experts: int, top_k: int = 2,
                 noise_std: float = 1.0):
        super().__init__()
        self.top_k = top_k
        self.n_experts = n_experts
        self.noise_std = noise_std
        self.gate = nn.Linear(hidden, n_experts, bias=False)

    def forward(self, x: torch.Tensor):
        # x: (batch, seq, hidden)
        logits = self.gate(x)                       # (B, S, n_experts)
        if self.training and self.noise_std > 0:
            # 训练时加噪声，促进负载均衡(避免所有 token 挤同一个专家)
            noise = torch.randn_like(logits) * self.noise_std
            logits = logits + noise
        # Top-K 选择：每个 token 选分数最高的 K 个专家
        topk_logits, topk_idx = logits.topk(self.top_k, dim=-1)
        topk_weights = F.softmax(topk_logits, dim=-1)  # 只对选中的 K 个 softmax
        return topk_idx, topk_weights                 # 用于后续加权汇聚专家输出

# 模拟一个 token 序列的路由决策
torch.manual_seed(0)
router = TopKRouter(hidden=4096, n_experts=8, top_k=2)
tokens = torch.randn(2, 10, 4096)   # 2 条句子，各 10 token
idx, w = router(tokens)

print("路由结果 (每 token 选 2 个专家):")
for t in range(5):
    print(f"  token {t}: 专家{idx[0, t].tolist()} 权重{[round(v,3) for v in w[0,t].tolist()]}")

# 负载均衡统计：看专家被选中的次数是否均匀
counts = torch.bincount(idx.flatten(), minlength=8)
print(f"\n专家命中次数: {counts.tolist()}")
# 理想情况各专家命中接近 token 总数*K/n_experts
# 若某专家长期 0 命中 → 需加 auxiliary loss 强制负载均衡
# MoE 的本质：用路由换稀疏 → 总参数多但单次激活参数少 → 大模型可训得起

★ 本章小结

这一章我们把 MoE 的基本概念和核心机制讲透了。要点回顾：

  • 核心动机：稠密模型参数和计算绑定，MoE 通过稀疏激活解耦两者，实现「参数多、计算少」。
  • 稀疏激活：把大 FFN 拆成 N 个专家，路由器给每个 token 选 top-k 个，只激活 k/N 的参数，但享受 N 倍的参数容量。
  • 容量因子：限制每个专家的 token 桶大小，超容量 token 被丢弃，是吞吐和质量的平衡旋钮，训练常用 1.25 到 1.5。
  • 负载均衡：MoE 训练的头号难题，路由容易坍缩到少数专家。用辅助损失（L_aux = alpha * N * sum(f_i * P_i)）惩罚不均。
  • 实现要点：路由、top-k 选择、概率归一化、加权聚合是核心，工程上用 grouped GEMM 或 scatter/gather 加速。
  • 权衡：训练和单 token 计算更省，但推理显存更大（所有专家常驻），训练稳定性更难调。

理解了 MoE 的宏观架构，下一章我们会钻进路由器这个最关键的部件——top-k gating 怎么实现、noisy top-k 解决什么问题、auxiliary loss 具体怎么写代码。这是让 MoE 真正跑起来、训得稳的关键。
