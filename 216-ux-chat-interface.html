对话界面设计

Agent 的"门面"决定体验

深度进阶★★★

Agent UI 比传统 UI 更难

传统软件的 UI 是"按钮+表单"——功能固定、操作明确。Agent 的 UI 是"对话+流式+工具可视化"——输出不确定、响应是渐进的、中间有工具调用。这带来全新的设计挑战：怎么让用户理解"Agent 正在做什么"？怎么展示工具调用的过程？怎么处理长任务？这些都是传统 UI 没遇到过的问题。

Agent 对话界面的核心组件

组件作用设计要点
消息气泡展示对话内容区分用户/Agent/系统/工具结果
流式渲染逐字显示回答打字机效果，Markdown 实时渲染
工具调用卡片展示 Agent 在调什么工具可折叠，显示输入/输出/耗时
思考过程展示 Agent 的推理可选展示（折叠/展开）
操作按钮停止/重新生成/复制触手可及
状态指示正在输入/思考/执行让用户知道没卡死

流式输出的交互设计

流式输出是 Agent 体验的关键。用户发出请求后，如果等 10 秒才一次性返回全部文字，体验极差。流式让用户在第一个 token（<1秒）就看到响应开始，心理等待感大幅降低。设计上要注意：逐字渲染时 Markdown 要正确处理（不能半截表格）、代码块要有语法高亮、长回答要有"继续生成"的暂停点。

💡 要点
流式输出的心理学：人类对"开始有反应"的容忍度远高于"等待中"。1 秒出第一个字 + 9 秒完成 >> 10 秒一次出完。这是为什么所有现代 AI 产品都用 SSE 流式。

工具调用的可视化

Agent 在执行任务时会调用工具（搜索、读文件、执行代码）。这些过程对用户是"黑盒"——如果不展示，用户不知道 Agent 在干什么。好的设计是展示工具调用卡片：工具名+参数+结果摘要+耗时。这些卡片默认折叠（不干扰），用户感兴趣可展开看细节。这增加了透明度和信任感。

人工审批界面

当 Agent 要执行危险操作（删文件、执行命令、发消息），应该弹出审批界面让用户确认。设计要点：清晰展示"要做什么"（操作描述+影响范围）、"为什么"（Agent 的理由）、"选项"（允许/拒绝/修改）。审批界面是 Agent 安全的最后一道用户侧防线。

React 流式渲染：从 SSE 到逐字显示

上一节讲了流式输出的设计原则，这一节给出可运行的前端代码。流式渲染的核心挑战不是"接收数据"（fetch + ReadableStream 就行），而是"边接收边渲染 Markdown"——当半截代码块还没闭合时，语法高亮不能崩；当表格只收到一半时，不能渲染成乱码。下面这段 React 代码用 `useChat` 风格封装了完整的流式接收 + 增量 Markdown 渲染。

// React 流式对话渲染：SSE 接收 + 增量 Markdown
import { useState, useCallback, useRef } from 'react';

function useStreamingChat(apiUrl) {
  const [messages, setMessages] = useState([]);
  const [isStreaming, setIsStreaming] = useState(false);
  const abortRef = useRef(null);

  const sendMessage = useCallback(async (text) => {
    // 1) 先把用户消息塞进列表
    setMessages(prev => [...prev, { role: 'user', text }]);
    // 2) 预占一个 assistant 气泡，content 边收边填
    setMessages(prev => [...prev, { role: 'assistant', text: '', streaming: true }]);
    setIsStreaming(true);

    const controller = new AbortController();
    abortRef.current = controller;

    try {
      const res = await fetch(apiUrl, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ messages: [...messages, { role: 'user', text }] }),
        signal: controller.signal,   // 支持中途取消
      });
      const reader = res.body.getReader();
      const decoder = new TextDecoder();
      let buffer = '';

      while (true) {
        const { done, value } = await reader.read();
        if (done) break;
        buffer += decoder.decode(value, { stream: true });
        // SSE 格式：以 "data: " 开头，按双换行分块
        const lines = buffer.split('\n\n');
        buffer = lines.pop();         // 留下最后不完整的块
        for (const line of lines) {
          const json = line.replace(/^data: /, '').trim();
          if (json === '[DONE]') break;
          const { delta } = JSON.parse(json);
          // 3) 把增量 token 追加到 assistant 气泡
          setMessages(prev => {
            const next = [...prev];
            const last = next[next.length - 1];
            next[next.length - 1] = { ...last, text: last.text + delta };
            return next;
          });
        }
      }
    } finally {
      setMessages(prev => {
        const next = [...prev];
        const last = next[next.length - 1];
        next[next.length - 1] = { ...last, streaming: false };
        return next;
      });
      setIsStreaming(false);
    }
  }, [messages]);

  const stop = useCallback(() => abortRef.current?.abort(), []);
  return { messages, sendMessage, stop, isStreaming };
}

这段代码有三个生产级细节：① 用 AbortController 支持"停止生成"——用户点停止时调用 `controller.abort()`，fetch 会抛异常中断读取；② 用 `buffer.split('\n\n')` + `buffer = lines.pop()` 处理 SSE 分包——网络层不保证一个完整 SSE 事件刚好在一次 read 里，可能出现半截事件，必须缓存拼回；③ 预占一个 streaming 状态的 assistant 气泡，然后不断替换它的 text 字段——这比"每次追加一条新消息"性能好得多，避免 React 重渲染整个列表。

消息气泡组件：区分角色与状态

消息气泡是对话界面的最小完整单元。一个工业级气泡要处理：四种角色（用户/助手/系统/工具）、流式中的光标、Markdown 渲染、代码块复制按钮、错误状态。下面这个组件用 `react-markdown` + `remark-gfm` 处理增量 Markdown，并实现流式光标。

// 消息气泡组件：支持流式光标 + 增量 Markdown
import ReactMarkdown from 'react-markdown';
import remarkGfm from 'remark-gfm';

function MessageBubble({ message }) {
  const { role, text, streaming, error } = message;
  const isUser = role === 'user';

  return (
    <div className={`bubble ${isUser ? 'bubble-user' : 'bubble-assistant'}`}>
      <div className="bubble-avatar">{isUser ? '🧑' : '🤖'}</div>
      <div className="bubble-body">
        {error ? (
          <span className="error">⚠️ 生成失败：{error}</span>
        ) : (
          <ReactMarkdown
            remarkPlugins={[remarkGfm]}
            components={{
              // 代码块：加复制按钮
              code({ inline, className, children }) {
                const lang = /language-(\w+)/.exec(className || '')?.[1];
                return inline ? (
                  <code className="inline-code">{children}</code>
                ) : (
                  <div className="code-block">
                    <button className="copy-btn"
                      onClick={() => navigator.clipboard.writeText(String(children))}>
                      复制
                    </button>
                    <pre><code className={className}>{children}</code></pre>
                  </div>
                );
              },
            }}
          >
            {text}
          </ReactMarkdown>
        )}
        {/* 流式中显示打字光标 */}
        {streaming && <span className="cursor">▋</span>}
      </div>
    </div>
  );
}

这里的关键工程取舍是"用成熟的 Markdown 库 vs 自己写增量解析器"。推荐前者：自己写增量 Markdown 解析（处理半截代码块、未闭合表格）极易出 bug，而 react-markdown 内部每次都对完整 text 重渲染，性能在千字以内完全够用。只有当单条消息超过几万字（如生成长文档）时，才需要考虑虚拟化或分段渲染。

💡 要点
流式光标（▋）看似细节，实则是体验关键。它给用户两个心理暗示：① 系统还在工作，没卡死；② 光标位置就是"下一个字会出现的地方"。没有光标的流式渲染，用户经常会怀疑"是不是断网了"。

打字机效果实现：从粗糙到精细

"打字机效果"有两种实现层次。粗糙版：直接把每个 SSE delta 原样 append——问题是模型可能一次返回一大段（比如一整段代码），瞬间糊上去，没有"逐字"感。精细版：把每次收到的 delta 拆成单字符，用 requestAnimationFrame 控制每帧只渲染 N 个字符，制造恒定的打字节奏。下面是精细版的实现。

// 精细打字机：恒定速率渲染，即使后端一次返回一大段
import { useRef } from 'react';

function useTypewriter(speed = 30) {   // speed: 每帧渲染字符数
  const [displayed, setDisplayed] = useState('');
  const queueRef = useRef('');          // 待渲染的字符队列
  const rafRef = useRef(null);
  const targetRef = useRef('');         // 当前目标全文

  const push = useCallback((delta) => {
    targetRef.current += delta;
    queueRef.current += delta;
    if (!rafRef.current) tick();
  }, []);

  const tick = useCallback(() => {
    setDisplayed(prev => {
      const take = Math.min(speed, queueRef.current.length);
      const next = prev + queueRef.current.slice(0, take);
      queueRef.current = queueRef.current.slice(take);
      if (queueRef.current.length > 0) {
        rafRef.current = requestAnimationFrame(tick);
      } else {
        rafRef.current = null;
      }
      return next;
    });
  }, [speed]);

  const reset = useCallback(() => {
    setDisplayed(''); queueRef.current = '';
    targetRef.current = ''; cancelAnimationFrame(rafRef.current);
    rafRef.current = null;
  }, []);

  return { displayed, push, reset };
}

// 使用：在收到 SSE delta 时调用 push(delta)
// rendered 文本传给 MessageBubble 的 text 字段

这种"用队列 + RAF 控速"的打字机有一个隐藏好处：当后端因为网络抖动延迟到达时，前端不会卡顿——它会用队列里缓存的字符继续以恒定速率"打字"，给用户稳定的节奏感。这比"delta 到一次就渲染一次"体验更可控。不过要权衡：如果用户想尽快看到完整答案，恒定速率反而是拖累。生产做法是给一个"加速"按钮（按住时 speed 调大），兼顾仪式感与效率。

方案实现复杂度体验适用
直接 append delta低时快时慢，像网络卡顿内部工具、MVP
RAF 恒定打字机中节奏稳定，有仪式感C 端产品、演示
分段降速（代码块慢、正文快）高最自然，引导阅读教育、长文产品

对话界面的认知负荷管理

对话界面（chat interface）看似简单（一个输入框+消息流），但设计不好会让用户不知所措。LLM 应用的对话界面尤其面临"认知负荷"挑战——agent 的回复可能很长、含多个选项、需要用户决策，用户容易信息过载。管理认知负荷是对话界面设计的核心。

认知负荷的来源有几个。信息密度过高：一次回复塞太多内容（长篇大论、多个建议），用户读不完也抓不住重点。决策点过多：频繁问用户"你要 A 还是 B"，每个决策都消耗注意力。状态不透明：用户不知道 agent 在干嘛、进展如何，焦虑等待。历史混乱：多轮对话后消息堆积，找之前的结论困难。每个来源都有对应的设计对策。

信息密度对策是"渐进披露 + 结构化"。长回复拆成"摘要 + 详情可展开"——先给一句话结论，细节折叠让用户按需展开。多选项用"推荐 + 备选"而非平行罗列——明确推荐哪个（"我建议 A，因为..."），其他作为备选，减少用户选择困难。决策点对策是"批量决策 + 默认值"——能合并的决策一次性问，能猜默认值的别问（用合理默认 + 允许纠正）。状态对策是"过程可见"——流式输出 + 进度提示，让用户感知 agent 在工作。

负荷来源表现设计对策
信息密度高回复太长摘要+可展开详情
决策点多频繁问选择推荐+默认值
状态不透明不知在干嘛流式+进度提示
历史混乱找不到旧信息结构化+可搜索
术语难懂技术黑话通俗化+术语解释
# 对话界面的认知负荷管理（前端组件示例）
class ManagedChatResponse:
    """把 agent 长回复组织成低认知负荷的结构"""
    def render(self, agent_output):
        # 1. 摘要先行：一句话结论
        summary = agent_output.get("summary", agent_output["text"][:80])
        html = [f'
{summary}
']
        # 2. 推荐方案（而非平行罗列多选项）
        if agent_output.get("options"):
            recommended = agent_output["options"][0]   # 第一个是推荐
            html.append(f'
建议：{recommended["label"]}'
                        f'{recommended["reason"]}
')
            # 备选折叠，不分散注意力
            if len(agent_output["options"]) > 1:
                html.append('
其他选项
    ')
                for opt in agent_output["options"][1:]:
                    html.append(f'
  • {opt["label"]}
    • ')
                html.append('
')
        # 3. 详情按需展开
        if len(agent_output["text"]) > 200:
            html.append(f'
查看完整分析'
                        f'
{agent_output["text"]}
')
        return "".join(html)

# 默认值策略：能猜的别问
class SmartDefaults:
    def ask(self, question, context):
        # 如果能从历史/偏好猜出答案，直接用默认 + 允许纠正
        guessed = self._guess_from_context(question, context)
        if guessed and guessed.confidence > 0.8:
            return (f"我猜你想问 {guessed.value}（{question}），"
                    f"如果不是请点这里纠正")
        return question   # 猜不出才问

多模态对话的交互模式

对话界面不再只是"文字来回"，越来越多 LLM 应用支持多模态——图片输入、语音、文件、富媒体输出（图表、卡片、可交互组件）。多模态让交互更自然强大，但也带来新的设计挑战：什么时候用什么模态、模态间如何切换、如何保持一致性。

模态选择的原则是"用最适合任务的模态"。文字适合抽象描述和推理；图片适合视觉信息（截图、图表、设计稿）；语音适合移动场景和无障碍；交互组件适合结构化输入（表单比让用户用文字描述结构化数据好）。不要为了炫技用复杂模态——简单的文字问答就够时，别强求用户传图或语音。好的设计是让用户"自然地用最省力的模态"。

模态切换的体验要流畅。用户传一张截图问问题，agent 应能直接基于图回答，而非要求用户"用文字描述图片"。agent 的输出也可以混用模态——文字解释 + 配图表 + 可点击的交互卡片。关键挑战是"跨模态一致性"——文字说的和图展示的要一致，不能矛盾。还要处理"模态降级"——某些环境不支持某模态（如纯文字终端），要能优雅降级到文字。

模态适合不适合设计要点
文字抽象/推理视觉信息结构化排版
图片视觉/截图抽象概念支持标注
语音移动/无障碍复杂结构转文字备份
交互组件结构化输入开放探索降低输入成本
图表数据展示定性信息可交互探索
// 多模态对话的前端处理
class MultimodalChat {
  async sendMessage(content) {
    // content 可能是文字/图片/文件/语音 的混合
    const message = await this.normalize(content);
    // 根据内容类型选择最佳交互
    const response = await this.agent.respond(message);
    // 渲染：agent 输出可能是多模态混合
    return this.renderMultimodal(response);
  }

  async normalize(content) {
    // 语音先转文字（保留原音频备份）
    if (content.type === 'audio') {
      const text = await this.stt(content.audio);
      return {...content, text, originalAudio: content.audio};
    }
    // 图片：提取基本信息，不用让用户描述
    if (content.type === 'image') {
      return {...content, metadata: await this.getImageInfo(content.image)};
    }
    return content;
  }

  renderMultimodal(response) {
    // agent 输出混用模态：文字 + 图表 + 交互卡片
    let html = '';
    if (response.text) html += `
${response.text}
`;
    if (response.chart) html += this.renderChart(response.chart);  // 可交互图表
    if (response.card) html += this.renderCard(response.card);     // 结构化卡片
    if (response.actions) html += this.renderActions(response.actions); // 可点按钮
    return html;
  }

  // 模态降级：环境不支持时优雅降级
  renderWithFallback(response, capabilities) {
    if (response.chart && !capabilities.svg) {
      // 不支持 SVG：降级为文字表格
      return this.chartToTable(response.chart);
    }
    return this.renderMultimodal(response);
  }
}

本章小结

Agent 对话界面的设计核心是"透明度"——让用户知道 Agent 在想什么、做什么、结果是什么。流式渲染降低等待感、工具卡片增加透明度、审批界面保障安全。好的 Agent UI 让用户感到"可控"而非"黑盒"。
