DiskANN
Vamana 图、SSD Beam Search 与内外存混合架构

当向量库涨到几十亿条，HNSW 的内存（几百 GB 到 TB）成了无法承受之重。DiskANN 是微软研究院提出的解决方案：把图放 SSD，用 Vamana 图的高质量 + beam search 的批量访存，让一台普通服务器（64GB 内存 + 一块 NVMe SSD）就能扛下百亿级向量。本文拆解 Vamana 图构建、SSD beam search 查询、以及内外存混合的精妙设计。

阅读约 20 分钟 难度 ★★★★★

★ HNSW 的内存天花板与 DiskANN 的破局

HNSW 的性能优势建立在一个前提上：整张图常驻内存。一张 1 亿节点的 HNSW 图，光存原始向量就要 300GB（768 维 float32），加上图的边（每节点 M=16 个邻居，约 64 字节），再算上优先队列等运行时结构，轻松突破 400GB。这在生产环境意味着一台机器装不下，要上集群——成本、运维复杂度都陡增。

DiskANN 的洞察是：SSD 的随机读取带宽（NVMe 可达 10-50 万 IOPS）足以支撑图遍历，前提是把随机小读转成批量读。普通 HNSW 查询是"取一个邻居 → 算距离 → 取下一个邻居"的串行随机访问，每次只读几十字节，对 SSD 极不友好（IOPS 浪费）。DiskANN 的核心改造就是用 beam search 把这种串行随机读变成批量并行读——一次取一批邻居，把 SSD 的吞吐榨干。

这样做的结果是：图可以放 SSD（百亿向量只要几 TB SSD，内存只放压缩后的向量和图的入口点），单机就能搞定。代价是延迟比纯内存 HNSW 高（毫秒到几十毫秒），但对于"内存装不下"的超大规模场景，这是唯一可行解。

★ Vamana 图：比 HNSW 更激进的剪枝

DiskANN 用一种叫 Vamana 的图构建算法（论文里叫它 NSG 的近亲）。Vamana 和 HNSW 的最大区别是：它是单层的，不分层。单层图的好处是结构简单、SSD 上的访问模式更规律；坏处是没有"上层快速接近"的机制，所以它通过两个手段弥补：

  1. 更激进的 alpha 参数：Vamana 的邻居选择有个参数 α（类似 HNSW 的启发式但带权重），构建时分两阶段——先用 α=1 构建一遍（等价于 HNSW 启发式），再用 α>1（如 1.2）重新修剪一遍。大 α 允许删除一些短边、保留长边，提升图的全局连通性，减少查询跳数。
  2. 精心选入口点：Vamana 选一个"近似质心"（medoid，到所有点距离之和最小的点）作为固定入口，而不是 HNSW 那种"最高层节点"。从质心出发能更快到达任意区域。

Vamana 构建流程（离线，一次性）：

  1. 计算 medoid 作为入口点 s。
  2. 对每个点 p，从 s 出发做一次贪心搜索得到候选邻居集 L。
  3. 用 α-修剪从 L 里选邻居连边。
  4. 所有点插完后，把 α 调大再修剪一遍全局图。

这套构建是离线的、一次性的，所以耗时较长（1 亿点要几小时），但构建质量极高——Vamana 图在内存里的 recall-QPS 甚至略优于 HNSW。DiskANN 的工程贡献是把这张高质量图挪到 SSD 上还能跑得动。

# Vamana 风格的简化构建（教学用，真实 DiskANN 用 C++ + SSD 优化）
import numpy as np
import heapq

def greedy_search(graph, q, entry, max_visits, L):
    """从 entry 出发贪心搜索，返回候选集（容量 L）和访问序列"""
    visited = {entry}
    cand = [(np.linalg.norm(q - graph[entry]['vec']), entry)]
    heapq.heapify(cand)
    result = list(cand)
    while cand and len(visited) < max_visits:
        _, c = heapq.heappop(cand)
        for nb in graph[c]['neighbors']:
            if nb not in visited:
                visited.add(nb)
                d = np.linalg.norm(q - graph[nb]['vec'])
                if len(result) < L or d < max(result)[0]:
                    heapq.heappush(cand, (d, nb))
                    result.append((d, nb))
                    if len(result) > L:
                        result.remove(max(result))
    return result

def prune_with_alpha(q_vec, cand_list, R, alpha):
    """α-修剪：保留方向多样且满足 alpha 约束的邻居"""
    cand_list.sort(key=lambda x: x[0])      # 按距离升序
    selected = []
    for d, n in cand_list:
        if len(selected) >= R:
            break
        # alpha 约束：n 到 q 的距离 × alpha 必须 < n 到任一已选邻居的距离
        keep = all(alpha * d < np.linalg.norm(
                    graph[n]['vec'] - graph[s]['vec']) for _, s in selected)
        if keep:
            selected.append((d, n))
    return selected

# 注意：真实 DiskANN 的构建在 C++ 实现（DiskANN repo），Python 版仅用于理解

★ SSD Beam Search：把随机读变批量读

这是 DiskANN 最有工程价值的部分。在 SSD 上跑 Vamana 图查询，朴素做法是：从入口 s 出发，取 s 的邻居列表（一次 SSD 读），算距离，选最近的下一步，再读它的邻居……每次只读几十字节，SSD 吞吐利用率不到 1%。

Beam search 的改造：维护一个宽度为 W 的候选队列，每轮从队列里取 W 个节点的邻居列表，合并成一次批量 SSD 读（一次 I/O 把 W 个节点的邻居都取回），算完距离后把新发现的近邻加入队列，淘汰远的。这样把"W 次串行小读"变成"1 次批量读"，SSD 吞吐利用率拉满。

具体到 SSD，DiskANN 还做了几个关键优化：

  • 扇区对齐存储：把图的节点按 SSD 扇区（4KB）对齐打包，一次扇区读能取回多个节点，进一步减少 I/O 次数。
  • SSD 预读（read-ahead）：访问一个扇区时顺便预取相邻扇区，因为图的访问有局部性。
  • 压缩向量缓存：内存里放一份 PQ 压缩的向量（每向量几十字节），先用压缩向量粗排，只对真正候选去 SSD 读原始向量精排。
# Beam search 的伪代码 —— 直观感受"批量读"
def diskann_beam_search(query, entry, k, beam_width, L):
    """
    query:      查询向量
    entry:      固定入口点（medoid）
    beam_width: W，每轮批量读多少个节点的邻居
    L:          候选队列容量
    """
    visited = {entry}
    # 候选队列（最小堆，按到 query 的距离）
    pq = [(dist_memcache(query, entry), entry)]    # 用内存里的压缩向量算
    result = []

    while pq:
        # 1. 取出 beam_width 个最近候选
        batch = []
        while pq and len(batch) < beam_width:
            d, n = heapq.heappop(pq)
            if n not in visited:
                batch.append(n); visited.add(n)

        # 2. 关键：一次批量 SSD 读，取回所有 batch 节点的邻居列表
        neighbor_lists = ssd_batch_read(batch)     # 1 次大 I/O 取多个扇区

        # 3. 对每个新邻居，用内存压缩向量算近似距离
        for nbs in neighbor_lists:
            for nb in nbs:
                if nb not in visited:
                    d_approx = dist_compressed(query, nb)   # 内存 PQ 距离
                    if len(result) < L or d_approx < result[-1][0]:
                        heapq.heappush(pq, (d_approx, nb))

        # 4. 更新结果集
        result.extend([(d, n) for d, n in batch])
        result = sorted(result)[:L]

    # 5. 对 top-N 用 SSD 上的原始向量精确重排
    top = [n for _, n in result[:200]]
    raw_vecs = ssd_batch_read_raw(top)
    final = sorted(top, key=lambda n: dist_exact(query, raw_vecs[n]))[:k]
    return final

★ 内外存混合：三层数据布局

DiskANN 把数据按访问频率分三层存储，是它能单机扛百亿的关键：

  • 内存层（小）：入口点 medoid、PQ 压缩向量（全量，每向量几十字节，百亿也才几百 GB）、导航图骨架。这部分要快，常驻内存。
  • SSD 层（大）：完整 Vamana 图（节点 + 边）和原始 float 向量，按扇区打包。查询时用 beam search 批量读。
  • 可选 SSD 缓存：高频访问的图区域缓存到内存，进一步降延迟（论文叫"cached beams"）。

这个分层让 DiskANN 在"内存 64GB + SSD 2TB"的廉价配置上就能装下 5-10 亿向量，单查询延迟 5-30ms，QPS 几百到几千。对比纯内存 HNSW 同样规模要 4-8 台高内存机器，TCO 降一个数量级。

★ 实战：用 DiskANN 的官方实现

# DiskANN 官方提供 Python binding (diskannpy)
# pip install diskannpy   (Linux/Mac，需要支持 AVX 的 CPU)
import diskannpy as dap
import numpy as np

d = 768
N = 10_000_000   # 一千万（教学规模；真实可上百亿）
# 注意：真实场景从磁盘流式加载，这里用随机数据演示 API
xb = np.random.randn(N, d).astype(np.float32)

# 1. 构建 Vamana 图并写入 SSD
builder = dap.DiskANNBuilder(
    distance="l2",
    vector_dtype=np.float32,
    dimensions=d,
)
graph = builder.build(
    data=xb,
    index_path="/tmp/diskann_index",       # 输出到 SSD
    l_build=200,         # 构建时的候选集宽度（类似 efConstruction）
    r=64,                # 每节点最大邻居数（比 HNSW 的 M 大）
    max_degree=64,
)
# 构建产物：.data（原始向量，扇区对齐）/ .index（图）/ .pq（PQ 压缩向量）

# 2. 加载查询器（内存只放 PQ 压缩向量 + 入口点）
searcher = dap.DiskANNSearcher(
    disk_index="/tmp/diskann_index.index",
    data_path="/tmp/diskann_index.data",
    vector_dtype=np.float32,
    dimensions=d,
)

# 3. 查询
xq = np.random.randn(5, d).astype(np.float32)
import time
t0 = time.time()
ids, dists = searcher.search(
    query=xq, k=10,
    l_search=200,        # 查询候选集宽度（类似 efSearch）
    beam_width=4,        # beam search 宽度，越大 SSD 吞吐越高
)
print(f"DiskANN 单次查询: {(time.time()-t0)/5*1000:.1f} ms")
print("top-10 ids:", ids)

调参要点

  • r（邻居度数）：建议 64-128，比 HNSW 大，因为单层图要靠更多边保证连通。
  • l_build：构建质量旋钮，200-400 常见，构建慢但图质量高。
  • l_search：查询召回旋钮，100-300，和 HNSW 的 efSearch 同理。
  • beam_width：SSD 吞吐旋钮，4-8 常见，取决于 SSD 的 IOPS。

★ 什么时候选 DiskANN

  • 向量规模超过单机内存（>5 亿，或内存预算 <100GB 但数据 >50GB）。
  • 延迟要求相对宽松（10-50ms 可接受）。
  • 数据相对静态（DiskANN 不擅长高频更新，重建成本高）。
  • 成本敏感（不想上集群）。

反过来，规模小、延迟要求极低、数据频繁变动，还是 HNSW 内存方案更合适。DiskANN 和 HNSW 不是替代关系，而是规模分层上的互补。

★ DiskANN 的扩展：FreshDiskANN 与混合索引

原始 DiskANN 是静态的（构建一次后只读），这限制了它的应用范围。学术界和工业界后续做了几个扩展。

FreshDiskANN（也叫 Vamana 的动态版）支持增量插入：新向量来了，在内存里维护一个小的"动态图"缓冲区，定期合并到 SSD 上的主图。合并时重新跑一遍局部 Vamana 修剪，保证增量后的图质量。这让 DiskANN 能用于流式数据场景，但实现复杂、合并有延迟。

SPANN（Microsoft）是另一个扩展：把数据按空间划分成多个"posting list"（类似 IVF 的簇），每个 list 内部用 DiskANN 风格的图。查询时只加载相关 list 的图到内存，实现"按需加载"的内存效率。SPANN 在 Azure 搜索里大规模部署，单集群扛千亿向量。

混合内存-SSD 索引：把高频访问的"热区"图缓存到内存，冷区留 SSD。这结合了 HNSW 的低延迟和 DiskANN 的大容量。Milvus、Qdrant 的新版本都有类似设计。

# FreshDiskANN 的概念流程
class FreshDiskANN:
    def __init__(self, ssd_index, mem_buffer_size=10000):
        self.ssd = ssd_index                  # SSD 上的主图
        self.buffer = []                      # 内存缓冲新向量
        self.buffer_size = mem_buffer_size

    def insert(self, vec):
        self.buffer.append(vec)
        if len(self.buffer) >= self.buffer_size:
            self._merge_to_ssd()              # 攒够一批合并

    def _merge_to_ssd(self):
        # 1. 把 buffer 里的向量加到 SSD 主图
        # 2. 对新加入的节点做局部 Vamana 修剪（重新选邻居）
        # 3. 清空 buffer
        new_ids = self.ssd.add_and_prune(self.buffer)
        self.buffer = []

    def query(self, q, k):
        # 同时查 SSD 主图和内存 buffer
        ssd_hits = self.ssd.beam_search(q, k*2)
        buf_hits = brute_force(self.buffer, q, k*2)
        return merge_and_rerank(ssd_hits, buf_hits, k)

这些扩展说明 DiskANN 不是终点，而是一个可演进的架构基座。理解它的核心思想（Vamana 图 + beam search + 内外存分层），就能理解所有变体。

★ DiskANN 生产部署的实操经验

把 DiskANN 部署到生产，几个实操经验值得分享。

SSD 选型很关键。DiskANN 的查询瓶颈是 SSD 随机读，所以要选高 IOPS 的 NVMe SSD（随机读 4K IOPS 至少 50 万）。便宜的 SATA SSD（几万 IOPS）会让 DiskANN 慢得不如纯内存方案。企业级 NVMe（如 Intel Optane）是 DiskANN 的最佳搭档。

beam_width 调参。beam_width 决定每次批量读多少节点，直接影响 SSD 利用率和延迟。典型值 4-8：太小（1-2）SSD 利用率低、延迟高；太大（>16）单次查询读太多、QPS 掉。要根据你的 SSD IOPS 和查询 QPS 目标调。

PQ 压缩向量的精度。内存里放的 PQ 压缩向量用于粗排，它的精度决定 SSD 精排的命中率。PQ 太糙（m 小）会让粗排丢真近邻，SSD 精排也救不回。建议 m=64-128，宁可内存多占点。

监控 SSD 健康。DiskANN 对 SSD 写入不重（构建一次后主要读），但要监控 SSD 的磨损度和坏块。SSD 故障会让整个索引不可用，要有快照备份和快速重建流程。

延迟的 P99 抖动。SSD 偶尔的 GC 或坏块重映射会让单次查询延迟飙到几百毫秒。应对是设置超时 + 重试 + 多副本。对延迟敏感的核心查询，可以加一层内存缓存热区。

这些经验说明 DiskANN 不是"装上就用"的方案，它对硬件和运维有特定要求。但它换来的"单机扛百亿、成本降一个数量级"的收益，对大规模场景是值得的。

★ Vamana 图构建的深度优化

Vamana 图构建是 DiskANN 质量的根本，它的两阶段 α 修剪有几个深度细节值得展开。

第一阶段 α=1 的意义。等价于 HNSW 的启发式选择——保留方向多样的邻居。这一阶段建立的图局部连通性好，但全局可能有大距离的"孤岛"（某些区域之间没有长边连接）。

第二阶段 α>1 的作用。用更大的 α（如 1.2）重新修剪，允许删除一些短边、保留长边。直觉：α>1 时，"e 和已选邻居 r 的距离 < α × e 到 q 的距离"才跳过 e，这意味着只有当 e 明显比已选邻居近（在 q 的方向上）才保留。结果是一些短边（连到很近但同向的邻居）被删，长边（连到稍远但跨区域的邻居）被保留。这提升了图的全局连通性——查询时能跨大距离跳。

为什么是两阶段而非直接用 α>1。直接用大 α 构建会让图早期就丢失短边，局部连通性差，搜索时找不到候选。先 α=1 建立良好的局部图，再 α>1 加长边，两阶段结合兼顾局部和全局。

# Vamana 两阶段构建（教学版）
def build_vamana(data, R=64, L_build=200, alpha1=1.0, alpha2=1.2):
    N = len(data)
    # 选 medoid 作为入口
    medoid = find_medoid(data)
    graph = {i: [] for i in range(N)}
    # 第一阶段：α=1 构建所有边
    for p in range(N):
        candidates = greedy_search(graph, data[p], medoid, L_build, R)
        neighbors = prune(candidates, data[p], R, alpha1)
        for n in neighbors:
            graph[p].append(n); graph[n].append(p)
    # 第二阶段：用 α=1.2 重新修剪所有边
    for p in range(N):
        neighbors = prune(graph[p], data[p], R, alpha2)
        # 重新连边（删旧的、加新的）
        graph[p] = neighbors
    return graph
# 第一阶段约占总构建时间的 60%，第二阶段 40%
# 最终图的搜索跳数比单阶段少 20-30%，这是 DiskANN 延迟优势的来源

构建复杂度。Vamana 构建要对每个节点做一次搜索（找候选），复杂度 O(N · 搜索成本)。搜索成本是 O(log N · 跳数)，所以总构建是 O(N · log²N) 量级。1 亿节点要几小时，这是 DiskANN 构建慢的原因。优化手段：① 并行构建（多线程同时处理不同节点）；② 增量构建（新数据只做局部）；③ 用 GPU 加速搜索部分。

构建质量评估。怎么知道构建的图好不好？看"搜索跳数分布"：好图的平均跳数少且稳定，差图跳数方差大（有的查询几跳，有的几十跳）。还可以看图的度数分布——好图度数均匀（接近 R），差图有热门点（度数远超 R）或孤立点（度数极低）。

这些深度细节说明 Vamana 不是"随便连个图"，它的每个设计选择都有明确目的。理解了这些，调 DiskANN 的 L_build、R 参数时就知道它们影响什么、怎么权衡。

★ 本章小结

  • DiskANN 解决 HNSW 内存装不下的问题：图放 SSD，单机扛百亿，靠 beam search 把 SSD 随机读榨干。
  • Vamana 图是单层高质量近邻图，用 α 修剪（两阶段，第二阶段 α>1）保证全局连通，从 medoid 入口出发。
  • SSD beam search 把"W 次串行小读"变"1 次批量读"，配合扇区对齐、预读、PQ 内存粗排，让 SSD 吞吐利用率拉满。
  • 三层数据布局：内存放 PQ 压缩向量 + 入口点，SSD 放完整图 + 原始向量，可选缓存热区。
  • 选型：规模超内存、延迟宽松、数据静态、成本敏感时选 DiskANN；反之选 HNSW。

import numpy as np
from dataclasses import dataclass
from typing import List, Optional

@dataclass
class PerformanceMetrics:
    latency_ms: float
    throughput_ops: float
    memory_mb: float
    accuracy: float

def benchmark(fn, iterations=100):
    """通用性能基准测试"""
    import time
    t0 = time.time()
    for _ in range(iterations):
        result = fn()
    elapsed = time.time() - t0
    return PerformanceMetrics(
        latency_ms=elapsed * 1000 / iterations,
        throughput_ops=iterations / elapsed,
        memory_mb=0,  # 实际中用 psutil 获取
        accuracy=0.0  # 根据具体任务计算
    )

# 示例：对比两种方案的性能
def approach_a(data):
    """方案A：直接计算"""
    return np.softmax(data @ data.T / np.sqrt(data.shape[-1]), axis=-1)

def approach_b(data):
    """方案B：分块计算（省内存）"""
    chunk_size = 64
    results = []
    for i in range(0, data.shape[0], chunk_size):
        chunk = data[i:i+chunk_size]
        results.append(np.softmax(chunk @ data.T / np.sqrt(data.shape[-1]), axis=-1))
    return np.vstack(results)

data = np.random.randn(512, 768)
metrics_a = benchmark(lambda: approach_a(data))
metrics_b = benchmark(lambda: approach_b(data))
print(f"方案A: {metrics_a.latency_ms:.1f}ms, {metrics_a.throughput_ops:.0f} ops/s")
print(f"方案B: {metrics_b.latency_ms:.1f}ms, {metrics_b.throughput_ops:.0f} ops/s")

常见问题FAQ

Q: 如何选择合适的方案？ 先用方案A跑通验证正确性，再根据实际瓶颈（内存/延迟/精度）切换到方案B或C。永远不要过早优化。

Q: 生产环境最大的坑是什么？ 过度工程化——在需求不明确时投入大量精力做复杂架构。建议MVP先行，收集反馈后迭代。

Q: 如何评估效果？ 定义明确的成功指标（准确率/延迟/成本），用golden set做回归测试，A/B测试验证改进效果。至少100个样本才有统计显著性。
