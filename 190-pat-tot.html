Tree of Thoughts

树状探索 + 评估回溯

深度进阶★★★

CoT 是一条路，ToT 是一棵树

Chain of Thought（CoT）让 LLM"一步步想"，但它只走一条路——如果中间某步想错了，后面全错。Tree of Thoughts（ToT）解决这个问题：在关键决策点生成多个候选思路，评估哪个更有希望，深入有希望的、放弃没希望的，必要时回溯。这就像下棋时的"博弈树搜索"——不是只想一步，而是探索多条可能，择优而行。

ToT 的四步循环

步骤做什么类比
Decompose把问题分解成可独立思考的子问题棋局的几个走法
Generate为每个子问题生成多个候选思路每种走法的多个分支
Evaluate评估每个候选的"前景"（打分）评估每步棋的好坏
Search用 BFS/DFS 搜索，深入高分的、回溯低分的博弈树搜索
def tree_of_thoughts(problem, llm, max_depth=5, branching=3):
    """ToT: 树状搜索推理路径。"""
    # 每个节点 = 一个部分推理状态 + 评分
    best = None
    frontier = [{"state": problem, "score": 1.0, "path": []}]

    for depth in range(max_depth):
        next_frontier = []
        for node in frontier:
            # 1. Generate: 生成 branching 个候选下一步
            candidates = llm.generate_thoughts(node["state"], n=branching)
            for cand in candidates:
                new_state = node["state"] + "\n" + cand
                # 2. Evaluate: 评估这个候选的前景
                score = llm.evaluate_state(new_state, problem)
                new_node = {"state": new_state, "score": score,
                           "path": node["path"] + [cand]}
                next_frontier.append(new_node)
                # 找到答案了？
                if llm.is_solution(new_state):
                    return new_state

        # 3. Prune: 只保留评分最高的几个（beam search）
        next_frontier.sort(key=lambda x: x["score"], reverse=True)
        frontier = next_frontier[:branching]  # beam width

    return frontier[0]["state"]  # 最佳路径
💡 要点
ToT 的代价是 LLM 调用次数爆炸（每个节点要 generate + evaluate，分支越多调用越多）。但它带来的提升也是数量级的：在经典"24 点"游戏上，Tree of Thoughts 达到 74% 成功率，而 Chain of Thought 仅 4%——因为这类问题需要试错和回溯，线性推理无能为力。ToT 适合"解谜题、24点、创意写作"等需要深度探索的任务，不适合简单的问答——那种用 CoT 就够。

CoT vs ToT vs ReAct 怎么选

模式探索方式适合任务成本
CoT单条线性推理简单推理、数学低（1 次调用）
ReAct线性 + 工具反馈需要外部信息的任务中（每步 1 次）
ToT树状搜索 + 回溯需要试错的难题高（指数级调用）

BFS 与 DFS：两种树搜索策略的实现

ToT 的搜索算法借鉴自经典 AI 搜索理论，主要有两种策略：BFS（广度优先，逐层展开，适合解在浅层）和 DFS（深度优先，一条路走到底，适合解在深层）。前面示例里的 beam search 是 BFS 的一种变体——每层只保留评分最高的几个节点。下面分别实现完整的 BFS 和 DFS 版本，并说明各自的适用场景。

# ToT 的两种搜索策略：BFS（广度）和 DFS（深度）
from dataclasses import dataclass
from typing import Callable

@dataclass
class ThoughtNode:
    state: str           # 当前推理状态（部分解）
    score: float         # 前景评分 0-1
    path: list           # 到达此节点的思考路径
    depth: int

class TreeOfThoughts:
    def __init__(self, llm, max_depth=5, branching=3, beam_width=3):
        self.llm = llm
        self.max_depth = max_depth
        self.branching = branching      # 每个节点生成几个候选
        self.beam_width = beam_width    # BFS 每层保留几个

    def search_bfs(self, problem: str) -> str:
        """广度优先 + beam search：逐层展开，每层只留最优的几个。"""
        frontier = [ThoughtNode(problem, 1.0, [], 0)]
        for depth in range(self.max_depth):
            next_layer = []
            for node in frontier:
                # 1. 生成 branching 个候选思路
                thoughts = self.llm.generate_thoughts(node.state, self.branching)
                for t in thoughts:
                    new_state = node.state + "\n→ " + t
                    # 2. 评估每个候选的前景
                    score = self.llm.evaluate(new_state, problem)
                    new_node = ThoughtNode(new_state, score,
                                          node.path + [t], depth + 1)
                    # 找到解就立刻返回
                    if self.llm.is_solution(new_state):
                        return new_state
                    next_layer.append(new_node)
            # 3. 剪枝：只保留评分最高的 beam_width 个（核心！）
            next_layer.sort(key=lambda x: x.score, reverse=True)
            frontier = next_layer[:self.beam_width]
            print(f"Depth {depth+1}: 保留 {len(frontier)} 个节点，"
                  f"最高分 {frontier[0].score:.2f}")
        return frontier[0].state

    def search_dfs(self, problem: str) -> str:
        """深度优先：一条路走到底，不行就回溯。"""
        def _dfs(node: ThoughtNode, visited_threshold=0.3) -> str:
            if node.depth >= self.max_depth:
                return node.state if self.llm.is_solution(node.state) else None
            thoughts = self.llm.generate_thoughts(node.state, self.branching)
            # 按评分排序，优先探索高分分支
            scored = [(self.llm.evaluate(node.state + "\n" + t, problem), t)
                      for t in thoughts]
            scored.sort(reverse=True)
            for score, t in scored:
                if score < visited_threshold:
                    continue  # 太低分的直接跳过（剪枝）
                new_state = node.state + "\n→ " + t
                if self.llm.is_solution(new_state):
                    return new_state
                # 递归深入
                result = _dfs(ThoughtNode(new_state, score,
                             node.path + [t], node.depth + 1))
                if result:  # 这条路找到解了
                    return result
                # 否则回溯，尝试下一个候选
            return None  # 所有候选都失败
        return _dfs(ThoughtNode(problem, 1.0, [], 0)) or "未找到解"

BFS 和 DFS 的选择取决于问题特征：BFS 适合解在浅层、且每步选择影响大的问题（如规划类任务，前几步走错全盘皆输，所以要每层都仔细评估择优）；DFS 适合解在深层、且能快速判断死路的问题（如解谜，需要深入探索某条路，走到死胡同快速回溯）。实践中 BFS（beam search）更常用，因为它通过剪枝控制了宽度，成本可预测；DFS 在最坏情况可能指数爆炸。

状态评估函数：ToT 的"裁判"

ToT 的效果好坏，70% 取决于 Evaluate 阶段的状态评估函数——它判断"当前这个推理状态有没有希望"。评估函数太松（什么都说有希望），退化成盲目搜索，浪费调用；太严（动不动判死刑），可能剪掉真正有希望的分支。设计一个好的评估函数是 ToT 的核心难点。

# 状态评估函数：判断推理状态的前景
class StateEvaluator:
    """评估当前推理状态离目标有多近。"""

    def evaluate(self, state: str, problem: str) -> float:
        """返回 0-1 的分数，越高越有希望。"""
        score = self.llm.evaluate(
            f"""问题：{problem}
            当前进度：{state}

            评估当前推理状态离正确答案有多近。
            打分（0.0-1.0）并说明理由：
            - 1.0: 已经接近或得到正确答案
            - 0.7: 方向正确，继续推进很可能成功
            - 0.3: 方向可能有问题，但有希望
            - 0.0: 走入死胡同，应该放弃

            只输出 JSON：{{{"score": 0.X, "reason": "..."}}}
            """,
            response_format="json"
        )
        return score["score"]

    def evaluate_relative(self, candidates: list[str], problem: str) -> list[float]:
        """相对评估：多个候选一起比较（比绝对评估更准）。"""
        # 让 LLM 同时看多个候选，排出优劣
        ranking = self.llm.evaluate(
            f"""问题：{problem}
            有以下几个候选推理路径，请评估它们的前景并排序：
            {chr(10).join(f"{i}. {c}" for i, c in enumerate(candidates))}

            给每个打 0-1 分，分数要拉开差距（最好的 0.9，最差的 0.3）。
            """,
            response_format="json"
        )
        return ranking["scores"]  # [0.9, 0.4, 0.6, ...]

# 关键技巧：相对评估 > 绝对评估
# 绝对评估（单独打分）：LLM 倾向给所有候选高分（"看起来都不错"）
# 相对评估（一起比较）：LLM 被迫拉开差距，择优更准

# 另一种评估：用外部信号（不靠 LLM 自评）
class ExternalEvaluator:
    """对有客观标准的问题，用外部信号评估更可靠。"""
    def evaluate_math(self, state, problem):
        # 数学题：把中间结果代入验证
        try:
            return 1.0 if verify_equation(state, problem) else 0.2
        except:
            return 0.1  # 推导格式错误

    def evaluate_code(self, state, problem):
        # 编程题：跑测试用例
        result = run_tests(extract_code(state), problem)
        return result.pass_rate  # 通过率作为分数

评估函数有两个层次：LLM 自评（通用，适合没有客观标准的任务如创意写作）和外部信号评估（精确，适合数学/编程等有验证方法的任务）。一个重要技巧是"相对评估优于绝对评估"——让 LLM 同时看多个候选并排序，比逐个打分更准，因为比较能迫使它拉开差距。对数学题，直接代入验算；对编程题，跑测试用例。外部信号永远是比 LLM 自评更可靠的评估器。

ToT vs CoT 对比实验：何时值得用 ToT

ToT 比 CoT 强，但贵得多。到底什么场景值得为 ToT 付出指数级的调用成本？下面用一个对比实验说明：在"需要试错"的任务上 ToT 完胜，在"线性推理"的任务上 CoT 更划算（用 ToT 是浪费）。

CoT 尝试：

💡 实用建议
不要把 ToT 当万能锤。它的成本是 CoT 的 10-30 倍，只有当 CoT 在你的任务上成功率明显不够（比如低于 50%）时才考虑。而且 ToT 可以按需升级：先用 CoT 试，失败了再上 ToT——这样大部分简单问题用便宜的 CoT 解决，只有难题才付出 ToT 的成本。这种"渐进式投入"是生产环境控制成本的常用策略。

Tree of Thoughts 的剪枝与资源分配

Tree of Thoughts（ToT）通过显式探索多条思维路径来提升复杂推理质量，但它的致命弱点是计算成本爆炸——每个节点分支 b 个子节点、探索深度 d，总节点数是 O(b^d)。不加控制的话，哪怕 b=3、d=5 也有 243 个节点，每个节点都要调一次模型，成本和延迟都难以承受。生产里用 ToT，剪枝和资源分配是命门。

剪枝的核心是"早砍差枝，把资源留给好枝"。在每个节点展开子节点后，立刻评估每个子节点的"前景分"（value），只保留 top-K 个继续深入，其余砍掉。前景分的评估可以用单独的 value model（轻量、快），也可以让生成模型自评。关键trade-off是评估的准确度与成本——评估太糙会砍错好枝，评估太细又增加成本。实践中 value model 用小模型（便宜快），生成用大模型（质量高），是常见的分工。

资源分配的进阶是动态预算。不均等地给每个分支分配探索深度——前景好的分支给更多深度继续挖，前景差的早停。这比固定深度更高效。更激进的是迭代深化：先广度优先浅探索所有分支一轮，识别最有希望的几个，再对它们做深度优先深挖。这避免了在无望分支上浪费深度。

策略机制成本适用场景
固定剪枝 top-K每层保留 K 个最优O(K^d)通用，实现简单
阈值剪枝低于前景分的全砍取决于分布分支质量差异大
动态深度好枝深挖、差枝早停自适应分支前景分明
迭代深化先广探后深挖中等不确定哪枝好
预算上限总节点数封顶硬性可控成本敏感场景
# 带剪枝和动态预算的 ToT
import heapq
class ToTSearcher:
    def __init__(self, generator, value_model, beam_width=3, max_nodes=50):
        self.gen = generator        # 生成思维（大模型）
        self.value = value_model    # 评估前景（小模型，便宜）
        self.beam = beam_width      # 每层保留几个
        self.max_nodes = max_nodes  # 总节点预算

    def search(self, problem, max_depth=5):
        # 起始：从问题生成 b 个初始思路
        frontier = [{"thought": t, "path": [t], "score": 0}
                    for t in self.gen.expand(problem, n=self.beam)]
        nodes_used = len(frontier)
        best = None
        for depth in range(max_depth):
            if not frontier or nodes_used >= self.max_nodes: break
            scored = []
            for node in frontier:
                # 评估当前节点前景
                node["score"] = self.value.evaluate(node["thought"], problem)
                scored.append(node)
            # 剪枝：只保留 top-K
            scored.sort(key=lambda n: -n["score"])
            frontier = scored[:self.beam]
            best = frontier[0]   # 记录最优
            # 动态深度：最优分支继续深挖，其余浅探
            if best["score"] > 0.8:
                # 高前景：集中资源深挖这一枝
                children = self.gen.expand_from(best, n=self.beam)
            else:
                # 中低前景：正常广度展开
                children = []
                for node in frontier:
                    children.extend(self.gen.expand_from(node, n=2))
            for c in children: c["path"] = best["path"] + [c["thought"]]
            nodes_used += len(children)
            frontier = children
        return best
# 关键：value model 用小模型（评估便宜），gen 用大模型（生成质量高），
# max_nodes 硬性封顶防止成本失控。b=3,d=5 的朴素 ToT 243 节点，
# 加 beam=3 + max_nodes=50 后实际只展开 ~50 节点，成本降 80%。

ToT 与其他模式的融合：何时该用树搜索

ToT 不是万能的，它的成本决定了只在"值得"的场景用。理解 ToT 与 ReAct、CoT、Reflection 的关系，才能在合适的场景选对工具，而不是"手里拿锤子看什么都是钉子"。

ToT 适合的场景有三个特征：解空间大且需要回溯（一条路走不通要换另一条）、单步质量难判断（要走了几步才知道好不好）、错误代价高（值得花更多计算避免错误）。典型如数学证明、复杂规划、创意生成。反之，如果解空间小或线性（一条路就够）、单步即可判断好坏、错误代价低（错了重做也便宜），用 CoT 甚至直接生成就够，ToT 是过度设计。

ToT 与其他模式的融合能兼顾质量和成本。常见组合：ToT 做宏观探索（找出有希望的大方向）→ 选定方向后用 CoT 深入（在这个方向上细致推理）→ Reflection 复核结果。这样把昂贵的树搜索用在最关键的"方向选择"上，而非全程都搜索。另一种组合是 ToT + ReAct：把"调工具"作为 ToT 的一个节点动作，遇到需要外部信息时分支展开不同查询，选最优结果继续。

场景特征推荐模式理由
解空间大、需回溯、错误代价高ToT探索多路径值得
线性推理、单步可判CoTToT 是过度设计
需调工具、动态决策ReAct树搜索+工具调用
需复核关键输出Reflection局部反思即可
复杂规划+执行Plan-Execute先规划再执行
方向难定 + 细节难做ToT 选向 + CoT 深入分工降本
# 模式选择器：根据任务特征选推理模式
def select_pattern(task):
    score = {"tot":0, "cot":0, "react":0, "reflect":0, "plan_exec":0}
    # 解空间大小
    if task.get("solution_space") == "large":   score["tot"] += 2
    else:                                        score["cot"] += 2
    # 是否需回溯
    if task.get("needs_backtrack"):              score["tot"] += 2
    # 是否需调工具
    if task.get("needs_tools"):                  score["react"] += 2
    # 错误代价
    if task.get("error_cost") == "high":
        score["tot"] += 1; score["reflect"] += 2
    # 是否复杂多步
    if task.get("complexity") == "high":         score["plan_exec"] += 2
    # 时间/成本约束
    if task.get("budget") == "tight":
        score["cot"] += 2; score["tot"] -= 2    # 紧预算时降权 ToT

    best = max(score, key=score.get)
    # 组合建议
    combo = []
    if score["tot"] >= 3 and score["cot"] >= 2: combo = ["tot_direct", "cot_deepen"]
    elif best == "react" and score["tot"] >= 2: combo = ["react_with_tot_branching"]
    return best, combo

# 示例：数学证明（解空间大、需回溯、错误代价高、预算宽松）
# → ('tot', ['tot_direct', 'cot_deepen'])
# 即：用 ToT 选大方向，每枝内用 CoT 深入，兼顾探索与成本

本章小结

Tree of Thoughts 把推理从"一条路"升级为"一棵树"：生成多个候选、评估择优、搜索回溯。它适合需要深度探索的难题（解谜、规划、创意），代价是高昂的 LLM 调用成本。理解了 CoT→ReAct→ToT 的演进，你就掌握了 agent 推理能力的三个层次。
