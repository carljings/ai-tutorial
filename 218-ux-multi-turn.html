多轮对话状态管理

让 AI 记住你 10 轮前说的话——对话状态工程

深度进阶★★★

多轮对话的核心挑战：记忆

单轮对话（一问一答）很简单，但真正的 AI 助手是多轮的——用户会追问、纠正、引用之前的话题。这让"对话状态管理"成了核心工程问题：模型怎么记住前几轮说了什么？什么时候该记住、什么时候该遗忘？记忆太长导致 token 爆炸怎么办？

人类对话的"记忆"是自然且高效的——你能记得 10 分钟前聊的核心内容，但自动遗忘无关细节。AI 要模拟这种"选择性记忆"，需要在工程上显式管理对话状态。这是 AI 助手体验好坏的分水岭——记忆管理差的助手会让用户不断重复自己说过的话，体验极差。

💡 要点
多轮对话的本质：**不是"记住所有历史"，而是"记住重要的、遗忘琐碎的"**。好的对话状态管理像人类记忆——有短期记忆（当前对话上下文）、长期记忆（跨会话的用户画像）、工作记忆（当前任务的关键信息）。

记忆的三个层级

层级内容实现生命周期
短期记忆当前对话的最近几轮滑动窗口/token 截断对话结束即清
工作记忆当前任务的关键信息（如用户姓名/偏好）显式提取+存储任务期间
长期记忆跨会话的用户画像/历史向量库/数据库永久

短期记忆：滑动窗口 + 摘要

最基础的短期记忆是滑动窗口——只保留最近 N 轮对话。但简单截断会丢失早期重要信息。更好的做法是"摘要压缩"：当对话超过窗口时，用 LLM 把旧对话压缩成一段摘要，放回窗口开头。这样既控制了 token 数，又保留了关键信息。

对话状态管理策略（从简到复杂）

L1: 滑动窗口（保留最近 N 轮）
    优点：简单    缺点：早期信息全丢

L2: 滑动窗口 + 摘要压缩
    超过窗口时，旧对话 → LLM 摘要 → 放回窗口头
    优点：保留关键信息    缺点：摘要可能丢细节

L3: 显式状态追踪（提取关键变量）
    每轮提取 "用户姓名/偏好/任务目标" 存到 state
    优点：精准    缺点：要设计 state schema

L4: 检索增强记忆（向量化历史）
    所有历史存向量库，每轮检索相关的
    优点：无限记忆    缺点：延迟+成本

工作记忆：显式状态追踪

很多对话任务有"关键状态"——如订机票助手要记住"出发地/目的地/日期/人数"。这些信息不能靠滑动窗口碰运气记住，必须显式提取和存储。这就是"对话状态追踪（DST）"——每轮对话后，更新一个结构化的 state 对象。

# 对话状态追踪示例（订机票助手）
class FlightBookingState:
    def __init__(self):
        self.departure = None    # 出发地
        self.destination = None  # 目的地
        self.date = None         # 日期
        self.passengers = 1      # 人数
        self.confirmed = False

    def update_from_utterance(self, user_input):
        """每轮对话后，用 LLM 提取并更新状态"""
        extraction = llm.extract(
            user_input,
            schema={"departure": "str?", "destination": "str?",
                    "date": "date?", "passengers": "int?"}
        )
        for k, v in extraction.items():
            if v is not None:
                setattr(self, k, v)

    def missing_fields(self):
        """检查还缺哪些必填字段"""
        required = ["departure", "destination", "date"]
        return [f for f in required if getattr(self, f) is None]

# 用法：每轮检查缺什么，针对性追问
state = FlightBookingState()
# 用户: "我想从北京飞上海，下周二"
state.update_from_utterance("我想从北京飞上海，下周二")
# state: departure=北京, destination=上海, date=下周二
missing = state.missing_fields()  # [] — 信息齐全
if not missing:
    print("确认订单")

长期记忆：跨会话的用户画像

最高级的记忆是跨会话的——助手记住"你是谁、你喜欢什么、你之前问过什么"。这需要把历史对话存成向量库或知识库，新对话开始时检索相关信息。代表产品：ChatGPT 的 Memory 功能（记住你的偏好和事实）。实现上，每轮对话后提取"值得记住的事实"（如"用户是 Python 开发者""用户偏好简洁回答"），存到长期记忆库。

💡 要点
记忆管理的艺术：**记太多 = token 爆炸 + 隐私风险；记太少 = 用户重复自己**。好的平衡是：短期记忆用摘要压缩（省 token），工作记忆显式提取（精准），长期记忆只存"高价值事实"（如用户画像/偏好）。

对话状态机：把闲聊和任务流分开

前面给的订机票状态只解决了"提取字段"，但真实对话里还有一个更难的问题：用户会在任务中途切换话题。比如订机票到一半，用户突然问"上海明天天气怎样？"——这是闲聊，不应该污染订单状态。生产级助手需要一个明确的对话状态机：每个意图有自己的状态，切换意图时自动挂起前一个任务。

dialog_state_machine.py —— 多意图状态机
from enum import Enum
from dataclasses import dataclass, field
from typing import Optional

class Intent(Enum):
    BOOK_FLIGHT = "book_flight"
    CHITCHAT = "chitchat"
    QA = "qa"

@dataclass
class TaskFrame:
    """单个任务上下文，挂起时可被恢复"""
    intent: Intent
    slots: dict = field(default_factory=dict)
    step: str = "init"

@dataclass
class DialogState:
    active: TaskFrame                  # 当前活跃任务
    parked: list = field(default_factory=list)  # 被挂起的任务
    history: list = field(default_factory=list) # 最近 N 轮原文

    def classify(self, user_input, classifier_llm):
        """用小模型快速判意图，避免大模型成本"""
        return Intent(classifier_llm(user_input))

    def switch(self, new_intent):
        """切到新意图：旧的挂起，旧的若已结清则丢弃"""
        if self.active.intent != new_intent:
            if self.active.step != "done":
                self.parked.append(self.active)
            self.active = TaskFrame(intent=new_intent)

    def resume(self):
        """用户回到旧话题时，从 parked 恢复"""
        if self.parked:
            self.active = self.parked.pop()

# 用法：每轮先分类，再决定挂起/恢复/继续
def on_turn(state, user_input, llm):
    intent = state.classify(user_input, llm)
    if intent == Intent.CHITCHAT:
        return llm.chitchat(user_input)   # 不动订单状态
    if intent != state.active.intent:
        state.switch(intent)
    # ... 继续推进当前任务

这种设计的关键 trade-off：意图分类必须快。如果每次都调大模型分类，延迟会显著拖累体验。实际生产里通常用一个小模型（或一个微调过的分类器）专门做意图判断，主模型只负责生成回复。

上下文窗口压缩策略：四种压缩的取舍

当对话很长，token 预算吃紧时，怎么压缩历史是工程上的高频决策点。四种主流策略各有明显的代价，下表给出取舍清单：

策略做法token 节省信息损失额外延迟
朴素截断只留最近 N 轮高极高（早期全丢）零
整段摘要旧对话一次性摘要成一段高中（细节模糊）1 次 LLM 调用
滚动摘要每 N 轮摘要一次，摘要再递归摘要中低（结构保留）分摊，低
选择性保留只留含「事实/决策/待办」的轮次中最低每轮 1 次轻量判断

下面给出滚动摘要的实现。它最大的好处是边界清晰：每次只在超出预算时压缩最旧的一个「块」，避免一次性摘要整段历史时的精度下降。

rolling_summary.py —— 滚动摘要 + token 预算控制
import tiktoken

def num_tokens(text, model="gpt-4o-mini"):
    return len(tiktoken.encoding_for_model(model).encode(text))

class RollingMemory:
    def __init__(self, llm, token_budget=6000, chunk_size=8):
        self.llm = llm
        self.budget = token_budget
        self.chunk_size = chunk_size         # 每 chunk_size 轮摘要一次
        self.summary = ""                   # 已压缩的旧摘要
        self.recent = []                     # 未压缩的近期待处理轮次

    def add_turn(self, role, text):
        self.recent.append(f"{role}: {text}")
        # 攒够一个 chunk 或超出预算时触发压缩
        used = num_tokens(self.summary) + sum(num_tokens(t) for t in self.recent)
        if len(self.recent) >= self.chunk_size or used > self.budget:
            self._compress()

    def _compress(self):
        """把 recent 摘要并并入 summary"""
        chunk = "\n".join(self.recent)
        prompt = ("把以下对话合并进既有摘要，"
                  "保留所有事实、决策、待办，删除客套。既有摘要：\n"
                  + self.summary + "\n新对话：\n" + chunk)
        self.summary = self.llm(prompt)
        self.recent = []

    def context(self):
        """组装最终发给 LLM 的上下文"""
        return ("[历史摘要]\n" + self.summary +
                "\n[最近对话]\n" + "\n".join(self.recent))

# 关键参数调优：
# - token_budget 设为模型上限的 60%，留 40% 给回复
# - chunk_size 过小 → 摘要频繁、细节丢得多
# - chunk_size 过大 → 单次摘要 token 多、延迟尖刺
⚠ 压缩的隐性陷阱

摘要本身是 LLM 生成的，它会继承 LLM 的幻觉。常见 bug 是摘要把用户没说过的事实「补全」进去了。防御办法：在 prompt 里强制要求「只复述出现过的内容，禁止推断」，并在摘要里保留关键数字/实体的原文（用引号包裹），方便后续核对。

多轮对话的状态追踪机制

前面讲了多轮对话的 UX 设计原则，但要把"对话不跑题、Agent 不失忆"这些原则落地，需要一个具体的工程机制——对话状态追踪（Dialog State Tracking, DST）。DST 的核心是维护一个"槽位（slot）"状态机：把完成用户任务所需的信息拆成若干个槽位（如订机票需要出发地、目的地、日期），每一轮对话后更新已填的槽位，并据此决定下一句该说什么（追问缺失槽位、还是确认执行）。这是任务型多轮对话能"记住上下文、主动引导"的底层机制，从 Siri 到智能客服都在用。

# -*- coding: utf-8 -*-
"""对话状态追踪(DST): 在多轮对话里维护"槽位"状态,
让 Agent 记住用户已经提供了哪些信息、还缺哪些。
这是多轮对话不"失忆"的核心机制。
"""
from dataclasses import dataclass, field
from typing import Optional

@dataclass
class Slot:
    name: str
    value: Optional[str] = None
    required: bool = True           # 是否必填才能执行任务

@dataclass
class DialogState:
    intent: Optional[str] = None    # 用户意图
    slots: dict = field(default_factory=dict)   # slot_name -> Slot
    history: list = field(default_factory=list) # 轮次历史

    def update_slot(self, name, value):
        if name not in self.slots:
            self.slots[name] = Slot(name=name)
        self.slots[name].value = value

    def missing_required(self):
        """返回还缺哪些必填槽位。"""
        return [s.name for s in self.slots.values()
                if s.required and s.value is None]

    def is_complete(self):
        return self.intent is not None and not self.missing_required()

DialogState 类把状态机抽象成三个核心要素：intent（用户意图）、slots（槽位字典）、history（轮次历史）。最关键的两个方法是 missing_required（返回还缺哪些必填槽位）和 is_complete（判断信息是否齐全可执行）。这两个方法让 Agent 能在每一轮后精确知道"还差什么"，从而生成有针对性的追问，而不是机械地复述"请提供完整信息"。下面用一个订机票任务演示完整的多轮槽位填充过程：

# 用一个"订机票"任务演示 DST 的多轮填充
def init_flight_booking():
    return DialogState(
        intent="book_flight",
        slots={
            "departure":  Slot("departure", required=True),
            "destination":Slot("destination", required=True),
            "date":       Slot("date", required=True),
        },
    )

state = init_flight_booking()

def user_says(text, state):
    """极简 NLU: 用关键词抽取槽位(真实场景用 LLM/intent分类)。"""
    import re
    if "从" in text:                     # "从北京出发"
        m = re.search(r"从(\S+?)(出发|飞|到)", text)
        if m: state.update_slot("departure", m.group(1))
    if "到" in text or "飞往" in text:    # "到上海"
        m = re.search(r"(?:到|飞往)(\S+)", text)
        if m: state.update_slot("destination", m.group(1))
    if re.search(r"\d{1,2}月\d{1,2}日", text):
        m = re.search(r"(\d{1,2}月\d{1,2}日)", text)
        if m: state.update_slot("date", m.group(1))
    return state

# 模拟多轮对话
dialogue = [
    "我想订机票",
    "从北京出发",
    "到上海",
    "7月20日的",
]
for turn in dialogue:
    user_says(turn, state)
    missing = state.missing_required()
    if missing:
        print(f"用户: {turn}\n  -> 还需要: {missing}")
    else:
        print(f"用户: {turn}\n  -> 信息齐全, 可执行订票: "
              f"{state.slots}")
        break
# DST 的价值: 让 Agent 主动追问缺失信息, 而不是
# 因为少了一个槽位就报错或乱猜。

把 DST 机制用到真实产品里，有几个进阶要点。第一，槽位抽取要用 LLM 而非规则：上面的关键词正则只是示意，真实场景用户表述千变万化（"下周去上海""飞魔都"），用一个小 LLM 做槽位抽取的鲁棒性远高于规则，成本也不高。第二，要支持槽位改写：用户说"改成 7 月 21 日"时要能识别这是对已有 date 槽位的更新而非新增，这需要一个槽位消歧逻辑。第三，要处理意图切换：用户聊到一半突然说"算了，我想查天气"，DST 要能识别意图从 book_flight 切到 query_weather，并保留或清理旧状态——这是多轮对话最难的边界情况。第四，槽位填满后要确认再执行：因为 LLM 抽取可能出错，执行高代价动作（扣款、下单）前要复述确认。把这些细节做扎实，你的多轮对话 Agent 才能像专业客服一样"听懂、记住、引导、确认"，这是多轮对话 UX 从"能聊"到"好用"的分水岭。

本章小结

多轮对话状态管理分三层：短期记忆（滑动窗口+摘要压缩）、工作记忆（显式状态追踪，提取关键变量）、长期记忆（向量化历史，跨会话用户画像）。核心原则：不是"记住所有"，而是"记住重要的、遗忘琐碎的"。对话状态追踪（DST）是任务型助手的基础——每轮更新结构化 state，针对性追问缺失字段。生产级助手还需要对话状态机，把闲聊与任务流隔离，避免话题切换污染任务状态。上下文压缩四种策略中，滚动摘要兼顾精度与延迟，是默认首选；但要注意摘要会继承 LLM 幻觉，必须强制「只复述、不推断」。记忆管理是 AI 助手体验的分水岭。
