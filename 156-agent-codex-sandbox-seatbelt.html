macOS Seatbelt 沙箱
sandbox-exec profile 与进程隔离

让 Agent 执行任意 shell 命令，等同于给它一把能删库的钥匙。Codex 在 macOS 上用的是一个叫 seatbelt 的内核级沙箱——通过 sandbox-exec 命令加载一份 profile 文件，就能把进程的文件系统访问、网络能力、进程创建全部关进笼子。本章我们从一份真实的 profile 文件读起，看懂 Seatbelt 的规则语法，以及 Codex 是怎么动态生成 profile 来约束每次工具执行的。

★ 五个深度问答

Q1: 这个工具的核心设计决策是什么？为什么这么选？

每个Agent工具都有一个"灵魂设计决策"。OpenCode的灵魂是"配置即代码"——所有行为（权限、工具、技能）都通过声明式配置定义。Codex的灵魂是"安全第一"——Rust内核+双层沙箱+Guardian审批。Aider的灵魂是"Git原生"——每次修改自动commit，diff格式编辑。Continue的灵魂是"双模式"——autocomplete和chat共享同一套模型配置但走不同的推理路径。理解灵魂决策，就理解了整个工具的架构方向。

Q2: 生产环境部署时需要注意什么？

三个关键注意事项：(1)API密钥安全——永远不要把密钥写在代码里，用环境变量或密钥管理服务（如HashiCorp Vault）。(2)速率限制——实现指数退避重试，避免429错误。建议设置每分钟最大请求数（如60/min）。(3)成本控制——设置每日token预算上限，超限自动降级到更便宜的模型或拒绝请求。

# 生产级成本控制示例
import time
from collections import defaultdict

class BudgetGuard:
    def __init__(self, daily_limit=1_000_000):
        self.daily_limit = daily_limit
        self.usage = defaultdict(int)
        self.day = time.strftime("%Y-%m-%d")
    
    def check_and_deduct(self, tokens, model="default"):
        today = time.strftime("%Y-%m-%d")
        if today != self.day:  # 新的一天，重置
            self.usage = defaultdict(int)
            self.day = today
        
        if self.usage[today] + tokens > self.daily_limit:
            # 降级到更便宜的模型
            if model == "glm-5":
                return "glm-5-flash", tokens // 2  # flash模型更便宜
            raise BudgetExceeded(f"日预算{self.daily_limit}已用尽")
        
        self.usage[today] += tokens
        return model, tokens

guard = BudgetGuard(daily_limit=500_000)
model, tokens = guard.check_and_deduct(8000, "glm-5")

Q4: 有哪些常见的错误用法和踩坑经验？

五个常见坑：(1)过度信任Agent输出——Agent会"自信地犯错"，生产中必须有人审核。(2)忽略token成本——一个复杂任务可能消耗50K+token，不控制成本很快烧光预算。(3)上下文管理不当——把所有历史都塞进context window，导致重要信息被"稀释"。应该用摘要+检索策略。(4)错误处理缺失——工具调用失败后Agent可能进入异常状态，必须有try-catch+重试+降级。(5)安全边界模糊——给Agent过大的权限（如root执行），一次prompt injection就可能导致数据泄露。

Q5: 未来的演进方向是什么？

Agent工具的三个演进趋势：(1)多模态Agent——不只处理文本，还能理解截图、UI布局、视频流（如Computer Use）。(2)长时自治Agent——从"完成单轮任务"到"持续运行、自我管理、定期报告"（如 Devin 的自主开发模式）。(3)Agent协作网络——多个专业化Agent通过A2A协议协作，每个Agent负责自己擅长的领域（前端/后端/测试/部署），像人类团队一样分工。

关键洞察：Agent工具的核心竞争力正在从"模型能力"转移到"工程能力"。同一个模型，套上不同的Harness/工具链/权限模型，效果可以差出10倍。这跟01-03章讲的Harness理论完全一致——"模型是引擎，工程是底盘"。
核心概念与实现★★★★

Seatbelt 是 macOS（确切说是 XNU 内核）里的 mandatory access control（MAC）框架，它的用户态接口是 sandbox-exec 命令。和 Linux 的 AppArmor 或 SELinux 类似，Seatbelt 允许你定义一套规则——某个进程能访问哪些路径、能连哪些网络、能 spawn 什么子进程——然后强制执行。

Seatbelt 的规则是用一种 Scheme 方言写的（没错，是 Lisp 风格的语法）。一条规则大致长这样：(allow file-write* (subpath "/Users/me/project/node_modules"))，意思是「允许在 node_modules 目录下写文件」。所有未明确 allow 的操作，默认被 deny。

关键特性是：规则只对 sandbox-exec 拉起的进程及其所有子进程生效，对系统其他进程无影响。这意味着你可以把 Agent 的命令执行隔离在一个沙箱里，而你的终端、IDE、其他应用完全不受影响。这种「进程级隔离」比修改系统全局配置安全得多。

★ 二、读一份真实的 profile 文件

我们先看一份 Codex 用来约束命令执行的 profile 文件（简化版，保留核心结构）：

;; codex-mac-sandbox.sb（简化）
;; 版本声明
(version 1)

;; 默认拒绝所有操作
(deny default)

;; === 文件系统访问 ===
;; 允许读取系统库（否则任何程序都跑不起来）
(allow file-read*
  (subpath "/usr/lib")
  (subpath "/usr/share")
  (subpath "/System/Library"))

;; 允许读写项目目录
(allow file-read* file-write*
  (subpath "/Users/me/project"))

;; 显式拒绝项目内的敏感文件
(deny file-read*
  (subpath "/Users/me/project/.env")
  (subpath "/Users/me/project/.env.local"))

;; 允许临时目录（很多程序需要写 /tmp）
(allow file-read* file-write*
  (subpath "/private/tmp")
  (subpath "/private/var/folders"))   ;; macOS 用户级临时目录

;; === 网络访问 ===
(allow network-outbound
  (remote tcp "github.com:443")
  (remote tcp "registry.npmjs.org:443"))
;; 其他网络出站一律拒绝
(deny network-outbound)

;; === 进程控制 ===
;; 允许 fork 和 exec（否则没法跑子命令）
(allow process-fork)
(allow process-exec
  (subpath "/usr/bin")
  (subpath "/bin")
  (subpath "/usr/local/bin"))

;; 禁止发信号给其他进程（防止 kill 干扰外部进程）
(deny signal process-info-target)

这份 profile 展示了 Seatbelt 的核心设计理念：白名单 + 默认拒绝。先 (deny default) 把所有操作设为拒绝，然后逐条 (allow ...) 放行需要的。这种「默认关闭」的策略保证了即使 profile 有疏漏，最坏情况也是「某些功能用不了」，而不是「Agent 能干坏事」。

★ 三、Codex 如何动态生成 profile

上面的 profile 里项目路径是写死的 /Users/me/project。但 Codex 每次会话的项目路径不同、网络需求也不同（有的会话要装 npm 包，有的不需要）。所以 Codex 不是用一份静态 profile，而是 在每次工具执行前动态生成 profile。

// codex-rs/src/sandbox/seatbelt.rs（概念示意）
use std::fs;
use std::path::Path;

pub fn generate_profile(config: &SandboxConfig) -> String {
    let cwd = config.cwd.to_string_lossy();
    let mut allowed_networks = String::new();
    for host in &config.allowed_hosts {
        allowed_networks += &format!(r#"(remote tcp "{}:443")"#, host);
    }

    // 读模板，填充占位符
    format!(r#"(version 1)
(deny default)

;; 系统库读取
(allow file-read* (subpath "/usr/lib") (subpath "/System/Library"))

;; 项目目录读写
(allow file-read* file-write* (subpath "{cwd}"))

;; 敏感文件保护
(deny file-read* (subpath "{cwd}/.env") (subpath "{cwd}/.env.local"))

;; 临时目录
(allow file-read* file-write*
  (subpath "/private/tmp")
  (subpath "/private/var/folders"))

;; 网络（按配置放行）
(allow network-outbound {allowed_networks})
(deny network-outbound)

;; 进程执行
(allow process-exec (subpath "/usr/bin") (subpath "/bin") (subpath "/usr/local/bin"))
(allow process-fork)
"#, cwd = cwd, allowed_networks = allowed_networks)
}

pub fn run_sandboxed(command: &str, config: &SandboxConfig) -> io::Result<Output> {
    // 1. 生成 profile
    let profile = generate_profile(config);
    let profile_path = format!("/tmp/codex-sandbox-{}.sb", uuid::Uuid::new_v4());
    fs::write(&profile_path, &profile)?;

    // 2. 用 sandbox-exec 拉起命令
    let output = Command::new("sandbox-exec")
        .arg("-p").arg(&profile_path)
        .args(["bash", "-c", command])
        .output()?;

    // 3. 清理 profile 文件
    let _ = fs::remove_file(&profile_path);
    Ok(output)
}

这段代码揭示了 Codex 沙箱执行的三步流程：生成 profile → sandbox-exec 拉起 → 清理。注意几个细节。第一，profile 文件用 UUID 命名避免冲突，执行完立即删除（不留痕迹，也防止 profile 被其他进程读到分析你的沙箱策略）。第二，bash -c command 是被 sandbox-exec 拉起的，意味着 bash 进程及其所有子进程都受沙箱约束——无论命令多复杂（管道、子 shell、后台进程），全部在笼子里。第三，网络白名单是从 config 来的，这意味着 Agent 执行 npm install 时只能访问 registry.npmjs.org，没法偷偷把代码外传到其他服务器。

★ 四、调试 profile：为什么我的命令被拒了

用沙箱最头疼的是：命令突然失败了，报错往往是「Operation not permitted」，但不知道是哪条规则挡的。Codex 提供了一个调试技巧——用 sandbox-exec 的 trace 模式。

# 用 -D 让 sandbox-exec 把所有被拒绝的操作打印到 stderr
sandbox-exec -D 1 -p /tmp/codex-sandbox-xxx.sb bash -c "npm install"
# 输出会包含类似：
# sandbox-exec: deny file-write* /etc/test
# sandbox-exec: deny network-outbound evil.example.com:443

Codex 在 dev 模式下会自动启用这个 trace，把被拒绝的操作收集起来，附在工具执行失败的返回里。这样 LLM 看到「npm install 失败」时，能同时看到「deny network-outbound registry.npmjs.org:443」——它就知道是网络权限问题，可以提示用户调整沙箱配置，而不是反复重试。这种「把沙箱审计信息反馈给 LLM」的设计，我们在第 154 章提到过，是 Codex 让 Agent 能理解「为什么失败」的关键。

★ 五、Seatbelt 的局限

Seatbelt 很强大，但不是万能的。Codex 在设计时清楚它的几个局限。

第一，仅限 macOS。Seatbelt 是 XNU 内核的特性，Linux 上没有。所以 Codex 在 Linux 上要用 Landlock（我们在下一章讲）。这种「平台相关的沙箱」增加了维护成本——两套实现、两套规则语法。

第二，规则语法不直观。Scheme 方言对大多数开发者是陌生的，调试 profile 需要专门的学习成本。Codex 团队内部维护了一套「常用 profile 片段」的库，避免每次手写。

第三，Apple 的文档有限。Seatbelt 的官方文档稀少，很多规则的实际行为要靠实验验证。社区维护的 sandbox profile 集合（如 google 的 sandbox-macos）是重要参考。

第四，sandbox-exec 可能在未来版本被限制。Apple 近年来收紧了 sandbox API，未签名的应用使用 sandbox-exec 已经受到限制。Codex 作为开源工具，依赖系统自带的 sandbox-exec，存在被未来 macOS 版本影响的政策风险。

★ 六、网络白名单的实战考量

网络访问是沙箱里最敏感的部分。完全禁止网络，Agent 连 npm install 都做不了；完全放开，又等于没沙箱。Codex 的策略是 按场景配置白名单：

  • 默认配置：只允许常见包管理器的 registry（npm、pip、cargo 等）。
  • 宽松配置（用户主动开启）：允许所有 HTTPS 出站，但仍禁止非标准端口。
  • 严格配置：完全禁止网络，适合纯静态分析场景。
;; 默认配置下的网络白名单片段
(allow network-outbound
  (remote tcp "registry.npmjs.org:443")
  (remote tcp "pypi.org:443")
  (remote tcp "files.pythonhosted.org:443")
  (remote tcp "crates.io:443")
  (remote tcp "static.crates.io:443")
  (remote tcp "github.com:443")
  (remote tcp "raw.githubusercontent.com:443"))
(deny network-outbound)

这种「按域名白名单」的策略比「按 IP」更实用——域名稳定、可读、可维护。但要注意 DNS 解析可能被劫持，高安全场景下要配合 DNS-over-HTTPS 或固定 IP。对大多数开发场景，域名白名单已经足够。

D1 深度剖析：macOS Seatbelt 的规则系统

本章介绍了 Codex 在 macOS 上用 Seatbelt 做沙箱。这一节我们深入 Seatbelt 的规则语法，看如何用 profile 文件精确控制进程的网络和文件访问。

;; Seatbelt profile：限制 Agent 子进程的权限
;; (deny default) 表示默认拒绝，再用 allow 显式放行
(deny default)

;; 允许读取系统库（进程启动必需）
(allow file-read*
  (subpath "/usr/lib")
  (subpath "/System/Library"))

;; 仅允许工作目录的读写，禁止触碰家目录其它部分
(allow file-write*
  (subpath "/Users/dev/project"))   ; 只能写项目目录
(deny file-write*
  (subpath "/Users/dev"))            ; 家目录其余禁止写

;; 完全禁止网络（防止数据外泄或拉取恶意载荷）
(deny network*)

;; 允许进程创建（执行命令需要 fork/exec）
(allow process-fork)
(allow process-exec
  (subpath "/usr/bin")
  (subpath "/bin"))

;; 关键：deny default + 显式 allow 的白名单模式
;; 比 "allow default + deny 危险项" 的黑名单模式安全得多

Seatbelt 的设计哲学是"默认拒绝 + 显式允许"的白名单模式。这与黑名单（默认允许 + 禁止危险项）截然不同——白名单下，任何未预料到的操作（如新出现的系统调用）默认被拦，安全性远高于黑名单。Codex 选择白名单正是因为 Agent 执行的命令不可预测，黑名单永远列不全。

规则类型作用典型用法坑
deny default拒绝所有未明确允许的白名单基础可能误伤必要系统调用
allow file-*文件读写控制限制工作目录符号链接绕过
deny network*禁止网络防数据外泄某些包管理需联网
allow process-exec限制可执行命令仅允许 /usr/bin解释器（python）可绕过

D2 生产避坑：Seatbelt 沙箱的绕过与加固

绕过 1：解释器逃逸

即使限制了可执行文件为 /usr/bin，若允许了 python 或 node，攻击者可用解释器执行任意代码（python -c "import os; os.system('...')"），完全绕过 exec 限制。要么禁止解释器，要么对解释器再加内层沙箱。Codex 对此采用命令分类 + 二次确认。

绕过 2：符号链接与硬链接

工作目录内若有符号链接指向外部（如 node_modules 链到家目录），file-write 规则可能被绕过。Seatbelt 较新版本支持 resolve 选项解析符号链接后再匹配，务必启用。

加固：分层沙箱

单层 Seatbelt 不够。Codex 在 Seatbelt 之外还加了应用层检查（命令分类、路径白名单）。纵深防御是关键——任何单层沙箱都可能被绕过，多层叠加才能提升攻击成本。

★ 本章小结

Seatbelt 让 Codex 在 macOS 上实现了进程级的强隔离。要点回顾：

  • Seatbelt 是 XNU 内核的 MAC 框架，用户态接口是 sandbox-exec，规则用 Scheme 方言写。
  • 白名单 + 默认拒绝：先 deny default，再逐条 allow，最坏情况是功能不可用而非安全事故。
  • 动态生成 profile：Codex 每次执行前按 cwd 和网络配置生成 profile，用 UUID 命名、执行完删除。
  • bash 及所有子进程受约束：无论命令多复杂都在笼子里，网络白名单按域名配置。
  • 调试用 -D 1 trace：把被拒绝的操作打印出来，Codex 在 dev 模式收集这些信息反馈给 LLM。
  • 局限：仅 macOS、规则语法陌生、官方文档少、有政策风险。
