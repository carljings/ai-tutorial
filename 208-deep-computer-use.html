Computer Use
让 AI 像人一样操作电脑

过去的 AI 只能在对话框里回答问题。Computer Use 把这道墙打破了:AI 能看到屏幕、移动鼠标、敲击键盘、点击按钮——像真人一样操作任意软件。Anthropic 在 2024 年推出的 Computer Use API,让 Agent 第一次具备通用的"操作图形界面"能力。本章讲透它的原理、实操代码,以及这项能力背后的机遇与风险。

在 Computer Use 出现之前,Agent 与外部世界的交互主要靠 API。要让 Agent 发邮件,你得提供邮件 API;要让它操作 Excel,你得接入 Office SDK。每接一个新软件都要写新集成,这是 Agent 能力扩展的根本瓶颈。

Computer Use 提供了一条截然不同的路径:既然人类是通过"看屏幕 + 操作鼠标键盘"来使用所有软件的,那让 AI 也学会这套通用接口,就能操作任意软件,无需专门集成。这是一个极具想象力的思路——GUI 是人类为人类设计的最通用的人机接口,让 AI 掌握它,等于打开了"整个数字世界"的入口。

Anthropic 的 Computer Use 能力让 Claude 能够:截取屏幕截图理解当前界面状态、识别界面上的元素(按钮、输入框、菜单)、计算需要移动到的坐标、执行点击/输入/滚动等操作、然后再次截图确认结果。这个"感知-决策-行动"的循环,正是人类操作电脑的方式。这种通用性让一个 Agent 就能处理浏览网页、填表单、操作 ERP、甚至写代码(在 IDE 里操作)等各种各样的任务。

★ 为什么 Computer Use 极其困难

让 AI 操作 GUI 听起来简单,实际上极其困难,这也是为什么它直到 2024 年才真正可用。理解这些难点,能帮你认识到这项技术的分量,也能解释它的局限。

难点一:像素级的视觉理解。GUI 是像素构成的,没有语义标签。AI 要从一张截图里识别出"这是按钮"、"那是输入框"、"这个图标代表保存"——这种视觉语义理解需要极强的多模态能力。早期方案用 OCR 提取文字 + 规则匹配,效果差;现代方案用视觉大模型直接"看懂"界面,才有了质的飞跃。

难点二:坐标定位的精度。决定点击哪里,需要把"我想点那个按钮"转化为具体的像素坐标。不同分辨率、不同布局、动态变化的界面,都让坐标计算充满挑战。Claude 通过在截图上叠加坐标网格,辅助自己定位,这是一种巧妙的工程化手段。

难点三:状态的动态性。网页会加载、弹窗会跳出、动画在播放、元素在变化。AI 每次操作后都要重新评估状态,不能假设界面静止。这种"实时感知"的要求,让 Agent 必须走"操作-观察-再决策"的闭环,而非一次规划到底。

难点四:错误恢复。点击没反应、加载超时、意外弹窗——真实软件使用充满意外。好的 Computer Use Agent 要能识别异常、调整策略、重试或换路径。这种鲁棒性是目前最不成熟的环节,也是 Computer Use 还不能完全可靠的原因。

★ Anthropic Computer Use API 的工作原理

Anthropic 的 Computer Use 能力是通过特殊的工具(tool use)实现的。Claude 被赋予了几个"computer tool",它们让模型可以执行屏幕截图、鼠标键盘操作。整个工作循环是:模型请求截图 → 看到屏幕 → 决定下一步操作(如"点击坐标(120,340)")→ 执行 → 再次截图 → 继续。这个循环一直持续,直到任务完成。

关键的几个工具:computer 工具负责屏幕级别的操作(截图、点击、输入、滚动、按键);text_editor 工具负责文件级别的文本编辑;bash 工具负责命令行操作。这三个工具组合起来,让 Claude 能在完整的桌面环境里完成任务。

这种设计的一个重要特点是模型驱动而非脚本驱动。传统自动化(RPA、Selenium)是写死步骤的脚本,环境一变就失效;Computer Use 是模型实时看屏幕决策,对界面变化有适应性。代价是每次决策都要调用模型,延迟和成本较高,且决策可能出错。

★ 实操:用 Computer Use API 完成一个任务

来看实际怎么用。Anthropic 提供了官方的 reference 实现(一个 Docker 镜像,包含虚拟桌面环境),我们在代码层面通过 tool use 调用。下面演示一个完整的 Computer Use 循环:让 Claude 打开浏览器、搜索某个词、记录结果。

import anthropic
import json, time, base64

client = anthropic.Anthropic()

# Computer Use 需要的三个工具定义
tools = [{
    "type": "computer_20241022",
    "name": "computer",
    "display_width_px": 1024,
    "display_height_px": 768,
    "display_number": 1,
}]

def run_computer_use(task: str, max_steps: int = 15):
    """Computer Use 主循环:模型决策-执行-观察"""
    messages = [{"role": "user", "content": task}]

    for step in range(max_steps):
        # 1. 让模型决策下一步(传入当前屏幕截图)
        screenshot_b64 = capture_screen_b64()
        response = client.beta.messages.create(
            model="claude-3-5-sonnet-20241022",
            max_tokens=4096,
            tools=tools,
            messages=messages,
            system="你是一个能操作电脑的助手。一步步完成任务,每步后观察结果。",
            betas=["computer-use-2024-10-22"],
        )

        # 2. 处理模型返回的工具调用
        if response.stop_reason == "end_turn":
            print("任务完成")
            break

        # 把模型回复加入对话
        messages.append({"role": "assistant", "content": response.content})

        # 执行模型请求的工具调用
        tool_results = []
        for block in response.content:
            if block.type == "tool_use":
                result = execute_tool(block.name, block.input)
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": result,
                })
        messages.append({"role": "user", "content": tool_results})

def execute_tool(name: str, action_input: dict):
    """执行 computer 工具的各种 action"""
    action = action_input.get("action")
    if action == "screenshot":
        return [{"type": "image", "source": {
            "type": "base64",
            "media_type": "image/png",
            "data": capture_screen_b64(),
        }}]
    elif action == "mouse_move":
        move_mouse(action_input["coordinate"])
        return "moved"
    elif action == "left_click":
        click_mouse()
        time.sleep(0.5)   # 等待界面响应
        return "clicked"
    elif action == "type":
        type_text(action_input["text"])
        return "typed"
    elif action == "key":
        press_key(action_input["key"])
        return "key pressed"
    elif action == "scroll":
        scroll(action_input.get("scroll_direction", "down"))
        return "scrolled"

run_computer_use("打开浏览器,搜索'今天上海的天气',告诉我结果")

这段代码揭示了 Computer Use 的核心循环:模型每一步都基于当前屏幕决策,执行一个原子操作,然后重新观察。任务分解、错误处理、路径调整,都由模型自主完成。这种"感知-决策-行动"的闭环,是 Computer Use 的本质。

★ 安全:Computer Use 最大的红线

Computer Use 的能力越强,安全风险越大。一个能操作电脑的 AI,如果被恶意引导或决策失误,可能造成严重后果——删除文件、泄露隐私、执行危险操作。所以安全是 Computer Use 第一优先级,远在能力之上。

几个必须做好的安全措施。沙箱环境:永远不要让 Computer Use 直接操作你的真实电脑。Anthropic 的官方实现就是一个隔离的 Docker 容器,里面是虚拟桌面,即使 AI 失控也不会影响宿主机。生产部署更要严格隔离网络和文件访问。权限最小化:只给 Agent 完成任务所需的最小权限,不要给管理员权限、不要给敏感目录写权限、不要给关键账号的操作权。操作审计:记录 AI 的每一次点击、每一次输入,便于事后追溯和 debug。对关键操作(删除、发送、支付)要做二次确认或人工审核。动作白名单:对高危动作(关机、格式化、发邮件给外部)做白名单拦截,模型即使想也执行不了。

除了这些技术措施,还有内容安全:防止 AI 被诱导做坏事(比如"帮我点击这个可疑链接")、防止 AI 接触到不该看的内容(屏幕上可能有敏感信息)。这些都是 Computer Use 走向生产前必须解决的课题。

★ 典型应用场景

Computer Use 适合什么场景?它的核心价值在"通用"——能操作那些没有 API 的传统软件,或 API 集成成本过高的场景。几个典型方向。

遗留系统集成。许多企业用的老系统(传统 ERP、内部工具、主机系统)没有现代 API,但能通过 GUI 操作。Computer Use 让 Agent 能操作这些系统,打通数字化转型的最后一公里。比如让 Agent 在一个 90 年代的库存系统里查询和录入数据。

跨应用工作流。很多真实工作要跨多个软件:从邮件里提取信息 → 在 CRM 里建客户 → 在财务系统里开票 → 发邮件通知。传统方案要分别集成四个软件的 API;Computer Use 让一个 Agent 直接像人一样操作这四个软件,省去集成成本。

数据录入与表单填写。把 PDF/图片里的信息录入到某个只有 GUI 的系统里,这种重复劳动 Computer Use 能高效完成。

软件测试。让 Agent 像真实用户一样操作软件,测试 UI 流程、发现 bug,这是自动化测试的新形态。

无障碍辅助。为视障用户提供"代为操作电脑"的助手,帮他们完成无法独立完成的 GUI 操作。这是 Computer Use 的社会价值所在。

★ Computer Use 与 RPA 的对比

Computer Use 经常被拿来和 RPA(机器人流程自动化)比较,因为它们都是"自动化操作软件"。但两者有本质区别,理解差异能帮你选对工具。

RPA(如 UiPath、Automation Anywhere)是基于规则的:开发者录制或编写固定的操作脚本(点这里 → 输入这个 → 等那个元素出现 → 再点),系统按脚本回放。它的优点是确定、可审计、执行快;缺点是脆弱——界面一变(按钮移位、流程微调),脚本就失效,维护成本高。

Computer Use 是基于模型的:AI 实时看屏幕,理解当前状态,自主决策下一步。它的优点是鲁棒——界面变化时能自适应调整;缺点是不确定(每次路径可能不同)、慢(每步都要推理)、有出错概率。

所以选择逻辑是:流程极其稳定、追求确定性和速度、规则明确 → RPA;流程多变、需要灵活适应、能容忍不确定性 → Computer Use。两者也可以结合:主干用 RPA 保证稳定,异常情况交给 Computer Use 处理。这正是企业级自动化的趋势。

★ 当前局限与未来方向

诚实地讲,Computer Use 还处在早期,有几个明显局限,选型时要有清醒认识。

速度慢。每一步都要截图、推理、执行,完成一个简单任务可能要几十秒到几分钟,远慢于 API 调用。对实时性要求高的场景不适用。

可靠性不足。视觉理解会误判、点击会点偏、复杂界面会迷失。当前的成功率对简单任务还不错,但对复杂、长流程的任务仍不稳定,需要人工兜底。

成本高。多轮推理 + 大量截图传输,token 消耗和 API 费用都不低。规模化部署要算清成本账。

无法访问无障碍信息。理想情况下,如果软件提供 accessibility tree(无障碍语义树),Agent 能更精确地理解界面。但目前多数软件的无障碍支持参差不齐,Computer Use 主要靠视觉,精度受限。

未来方向上,几个趋势值得期待:更快的视觉模型降低延迟;更强的鲁棒性减少错误;专用硬件(如操作系统级的 Agent API)让操作更精确;以及安全框架的成熟让 Computer Use 能更放心地用。Anthropic、OpenAI、Google 都在押注这个方向,它很可能是 Agent 走向"通用智能体"的关键一环。

★ 实战：生产级 Computer Use 调用循环

上面的最小循环只演示了「决策-执行」。真实部署还需要重试、超时、安全护栏、状态快照、人在回路确认。下面给一个更接近生产的完整循环：把每一步的可撤销操作记进历史，高危动作拦截并请求人工确认，出错自动重试并在连续失败后熔断。

# computer_use_loop.py · 生产级 Computer Use：护栏 + 重试 + 审计 + HITL
import anthropic, time, json, logging

client = anthropic.Anthropic()
logging.basicConfig(level=logging.INFO, format="%(asctime)s [%(levelname)s] %(message)s")
log = logging.getLogger("computer_use")

# 安全护栏：高危动作必须人工确认，禁用动作直接拦截
CONFIRM_ACTIONS = {"key", "left_click"}    # 按键/点击可能触发不可逆操作
BLOCKED_KEYS = {"ctrl+alt+del", "win+l", "ctrl+shift+esc"}
MAX_RETRIES = 3                            # 单步重试上限
MAX_STEPS = 20                             # 整个任务步数上限（防死循环烧钱）

tools = [{"type": "computer_20241022", "name": "computer",
          "display_width_px": 1024, "display_height_px": 768, "display_number": 1}]

def human_confirm(action, inp) -> bool:
    """高危动作请求人工确认（生产里接 Slack/审批系统）。"""
    ans = input(f"⚠️ Agent 想执行 {action}({inp})，允许？[y/N] ")
    return ans.strip().lower() == "y"

def safe_execute(name, action_input):
    """带护栏的动作执行：拦截禁用项、高危项走 HITL、全程审计。"""
    action = action_input.get("action", name)
    # 1. 硬拦截：禁用动作
    if action == "key" and action_input.get("key", "").lower() in BLOCKED_KEYS:
        log.warning(f"拦截禁用按键: {action_input['key']}")
        return "BLOCKED: 该按键已被安全策略禁止"
    # 2. 软确认：高危动作请求人工
    if action in CONFIRM_ACTIONS and not human_confirm(action, action_input):
        log.info("用户拒绝动作")
        return "USER DENIED: 用户拒绝了此操作"
    # 3. 执行并审计
    log.info(f"EXEC {action} {action_input}")
    return execute_tool(name, action_input)       # 复用上一节的 execute_tool

def run_task(task: str):
    messages = [{"role": "user", "content": task}]
    fail_streak = 0
    for step in range(MAX_STEPS):
        try:
            resp = client.beta.messages.create(
                model="claude-3-5-sonnet-20241022", max_tokens=4096,
                tools=tools, messages=messages, betas=["computer-use-2024-10-22"],
                system="你是能操作电脑的助手。一步步完成，每步后观察结果。")
            messages.append({"role": "assistant", "content": resp.content})
            if resp.stop_reason == "end_turn":
                log.info(f"任务完成（{step} 步）"); return
            # 执行工具调用，收集结果
            results = []
            for b in resp.content:
                if b.type == "tool_use":
                    results.append({"type": "tool_result",
                                    "tool_use_id": b.id,
                                    "content": safe_execute(b.name, b.input)})
            messages.append({"role": "user", "content": results})
            fail_streak = 0                          # 成功一步，清零失败计数
            time.sleep(0.5)                            # 等界面刷新
        except Exception as e:
            fail_streak += 1
            log.error(f"第 {step} 步出错({fail_streak}/{MAX_RETRIES}): {e}")
            if fail_streak >= MAX_RETRIES:        # 连续失败 → 熔断
                log.critical("连续失败达上限，熔断停止，需人工介入")
                return
            time.sleep(2 ** fail_streak)              # 指数退避重试
    log.warning(f"达到步数上限 {MAX_STEPS}，任务未确认完成")
    # 保存对话历史，便于事后审计/debug
    with open("computer_use_trace.json", "w") as f:
        json.dump([{"role": m["role"]} for m in messages], f, ensure_ascii=False)

run_task("打开浏览器，搜索'今天上海的天气'，告诉我结果")
# 这个循环的核心增益：护栏防失控、重试抗抖动、审计可追溯、HITL 守底线

这个生产级循环把本章的安全要求落到了代码里：禁用动作硬拦截、高危动作人工确认、连续失败指数退避、达到上限自动熔断、全程审计可追溯。注意它和最小循环的区别——不是「能跑」，而是「敢跑」。Computer Use 的可靠性本就不足，没有这层护栏的 Agent 一旦失控，后果远超聊天机器人。能力之上，安全先行，这是 Computer Use 落地的铁律。

★ Computer Use 的核心循环

Computer Use 的本质是一个"看屏幕→决策→操作→再看"的 agent 循环。它和普通工具调用的区别是用视觉感知替代结构化 API。下面是核心循环的骨架代码：

import json, base64
from openai import OpenAI

client = OpenAI()

def computer_use_loop(task: str, max_steps: int = 20) -> str:
    """Computer Use 的核心: 观察(截图) -> 思考 -> 动作(点击/输入) -> 循环。
    直到模型判定任务完成或达到步数上限。
    """
    messages = [
        {"role": "system", "content":
            "你是一个能操作电脑的 agent。每轮你会收到屏幕截图, "
            "需要输出一个 JSON 动作。可用动作: "
            '{"action":"click","x":..,"y":..}, '
            '{"action":"type","text":".."}, '
            '{"action":"scroll","dy":..}, '
            '{"action":"screenshot"}, '
            '{"action":"done","result":".."}.'},
        {"role": "user", "content": f"任务: {task}"},
    ]

    for step in range(max_steps):
        # 1. 观察：截取当前屏幕(这里用占位函数)
        screenshot_b64 = take_screenshot_b64()

        # 2. 把截图作为多模态输入喂给模型
        obs = messages if step > 0 else messages
        messages.append({
            "role": "user",
            "content": [
                {"type": "text", "text": f"第 {step+1} 步, 当前屏幕:"},
                {"type": "image_url",
                 "image_url": {"url": f"data:image/png;base64,{screenshot_b64}"}},
            ],
        })

        # 3. 模型输出动作(思考+决策)
        resp = client.chat.completions.create(
            model="glm-4.6", messages=messages, temperature=0)
        action = json.loads(resp.choices[0].message.content)
        messages.append({"role": "assistant", "content": json.dumps(action, ensure_ascii=False)})

        # 4. 执行动作, 改变屏幕状态
        if action["action"] == "done":
            return action.get("result", "完成")
        execute_action(action)   # 调用 OS 级 API 真正点击/输入
        # 循环回到第 1 步, 重新截图观察

    return "达到最大步数, 未确认完成"

def take_screenshot_b64() -> str:
    # 生产里: pyautogui.screenshot() 或 mss 抓屏 -> base64
    return base64.b64encode(b"fake_png_bytes").decode()

def execute_action(action: dict):
    # 生产里: pyautogui.click(x, y) / typewrite(text) 等
    print(f"  执行: {action['action']}", action.get("x",""), action.get("text",""))

# Computer Use 的本质 = 视觉感知 + 工具执行的 agent loop
#   它不需要应用暴露 API, 像人一样看屏幕、点鼠标, 通用性最强
#   代价: 每步要截图+多模态推理, 成本高、速度慢, 适合 GUI 自动化/辅助操作
result = computer_use_loop("打开浏览器搜索'GLM 模型'并截图第一个结果")
print("最终结果:", result)

★ 本章小结

Computer Use 让 AI 从"对话框"走进"整个数字世界",通过视觉理解屏幕、鼠标键盘操作软件,实现了通用的软件操作能力。它的难点在像素级视觉理解、精确坐标定位、动态状态感知、错误恢复,这也是它直到最近才可用的原因。

Anthropic 的 Computer Use API 通过 computer/text_editor/bash 三个工具,实现了"模型驱动"的操作循环:截图-决策-执行-观察。我们演示了完整代码。安全是它的第一红线,沙箱、最小权限、审计、动作白名单缺一不可。

Computer Use 适合遗留系统集成、跨应用工作流、数据录入、软件测试、无障碍辅助等场景。相比 RPA 它更鲁棒但更慢更不确定,两者适合不同场景,也可结合使用。当前它在速度、可靠性、成本上仍有局限,但方向意义重大——这是 AI 从"专用助手"走向"通用数字劳动力"的桥梁,值得长期关注和提前布局。
