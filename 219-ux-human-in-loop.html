人在回路HITL
审批、反馈、修正与LangGraph中断机制

完全自动化的 Agent 听起来很美，但在涉及资金、数据、外部副作用的高风险场景里，一次错误可能比一千次成功更昂贵。人在回路（Human-in-the-Loop, HITL）就是给自动化系统装上"人工安全阀"：在关键节点暂停、请人确认或修正、再继续执行。本章深入 HITL 的三种核心模式——approval、approval_with_modification、correction，并用 LangGraph 的 interrupt 机制给出可落地的工程实现。

阅读约 30 分钟 难度 ★★★★

★ 为什么需要人在回路

设想一个客户运营 Agent，它每天给 1000 个高价值客户发个性化邮件。如果模型某次幻觉把"王先生"错写成"王女士"，或把"折扣 5%"写成"折扣 50%"，损失可能远超这 1000 封邮件带来的收益。问题不在于模型不够强——再强的模型也无法保证零错误——而在于"错误代价"和"自动化收益"之间的不平衡。这种不平衡在以下场景尤其突出：金融交易、医疗建议、法律文书生成、对外公关、数据删除、付费 API 调用、权限授予。

HITL 的核心思想是"分级信任"：把决策按风险分级，低风险的交给 Agent 全自动，高风险的引入人工把关。这本质上是把"全人工"和"全自动"两极之间的大片灰色地带，用一套机制填补起来。一套好的 HITL 系统不是简单地"什么都让人确认"，那样会让人变成瓶颈；而是只在真正需要人判断的节点暂停，让人发挥不可替代的价值——比如对模糊意图的判断、对伦理风险的权衡、对创意的最终拍板。

HITL 还有一个常被低估的价值：它产生高质量的标注数据。每一次人工审批、修正、反馈，都是一条带上下文的训练样本。把这些样本沉淀下来，可以用来微调模型、优化 prompt、训练分类器来自动判断"哪些决策可以放宽人工"。换句话说，HITL 不是"永久的成本"，而是"通向更高自动化水平的阶梯"。

★ 三种核心模式：approval / approval_with_modification / correction

HITL 在工程上有三种典型实现模式，对应不同的介入深度和信任度。理解它们的差异是设计 HITL 系统的第一步。

第一种是 approval 模式（审批）。Agent 在执行不可逆操作前暂停，把完整计划提交给人审批，人选择"批准"或"拒绝"后系统才继续。这是介入最深的模式，人相当于系统的"决策把关人"。典型场景是发邮件、转账、删除数据。审批可以是同步的（人在界面上立刻看到并决策），也可以是异步的（提交到工单系统，人按节奏处理，Agent 在等待中挂起状态）。优点是延迟可控、实现简单，缺点是结果大量需要修改时，"通过/拒绝"二元判断不够用——人只能全否重做，浪费了模型已完成的 80%。

第二种是 approval_with_modification 模式（批准并修改）。介于 approval 和 correction 之间：人介入的程度比 correction 深（直接改内容），但比 approval 浅（不阻断执行，而是允许人工修改后执行）。人不仅能拒绝，还能在 Agent 结果基础上直接改——改一段文案、调一个参数、删一个工具调用。典型场景是文档生成、代写邮件、营销文案这种"骨架对、细节需打磨"的半结构化输出。它对 Agent 输出质量更宽容，默认"模型可能 80% 对，剩下 20% 让人补"，人工效率远高于纯 approval 的"全否重写"。实现上需要前端有可编辑界面，后端能把修改后的版本回灌到 Agent 状态。

第三种是 correction 模式（修正）

第三种是 correction 模式（修正）。Agent 输出后，人可以直接编辑修改，修改后的版本作为"金标准"被记录下来。它不阻断执行，而是事后让用户直接修改内容，这些修改变成训练信号用于改进模型。典型场景是文档生成、代码生成、翻译——AI 出草稿，人改完直接用，同时把改后版本喂回模型做 DPO/SFT 训练。它是 RLHF 和 DPO 在生产中的主要数据来源，也是把 Agent 从"通用模型"演进到"懂你业务的专家"的核心机制。下面这个表格总结了三者的关键差异。

实际系统往往三种模式并用：低风险步骤直接自动执行，中风险步骤走 approval_with_modification 让人修改后执行，高风险步骤走 approval 严格审批。下一节我们看如何在 LangGraph 里实现这些模式。

★ LangGraph 的 interrupt 机制

LangGraph 是 LangChain 团队推出的 Agent 编排框架，它把 Agent 建模为一张"状态图"（StateGraph）：节点是处理函数，边是状态转移。这种图模型天然适合表达 HITL——你可以在某个节点暂停整张图的执行，把状态持久化下来，等人介入后再恢复。LangGraph 提供的 interrupt 函数就是实现这种"暂停-恢复"的核心原语。

interrupt 的工作机制是：当节点函数调用 interrupt(value) 时，LangGraph 会把 value 作为"待确认内容"抛出，并把当前图的完整状态持久化到 checkpoint（通常是数据库）。Agent 进程此时可以退出，状态不会丢。当人决策后，外部调用 graph.invoke(Command(resume=decision), config) 来恢复执行，interrupt 函数会返回这个 decision，节点函数继续往下跑。这种机制让"等待人"可以跨越进程、跨越时间——哪怕人 8 小时后才回来决策，Agent 也能无缝续上。下面是一个完整的 approval 模式实现。

# pip install langgraph langchain-openai
from typing import TypedDict, Annotated
from langgraph.graph import StateGraph, START, END
from langgraph.types import interrupt, Command
from langgraph.checkpoint.memory import MemorySaver
from langchain_openai import ChatOpenAI

class State(TypedDict):
    user_intent: str
    draft_email: str
    approved: bool

def write_draft(state: State) -> State:
    """Agent 起草邮件。"""
    model = ChatOpenAI(model="gpt-4o-mini")
    draft = model.invoke(
        f"为以下意图写一封简短商务邮件：{state['user_intent']}"
    ).content
    return {"draft_email": draft}

def human_approve(state: State) -> State:
    """暂停等待人工审批。interrupt 把草稿抛给外部，等人 resume。"""
    decision = interrupt({
        "type": "approval",
        "prompt": "请审批以下邮件：",
        "email": state["draft_email"],
    })
    # decision 是外部通过 Command(resume=...) 传入的值
    # 通常是 {"approved": True/False} 或 {"approved": True, "edited": "..."}
    if decision.get("edited"):
        return {"draft_email": decision["edited"], "approved": True}
    return {"approved": decision["approved"]}

def send_email(state: State) -> State:
    if state["approved"]:
        print(f"[发送] {state['draft_email']}")
    else:
        print("[取消] 用户拒绝发送")
    return state

# 构建状态图：写草稿 → 人审 → 发送
g = StateGraph(State)
g.add_node("write_draft", write_draft)
g.add_node("approve", human_approve)
g.add_node("send", send_email)
g.add_edge(START, "write_draft")
g.add_edge("write_draft", "approve")
g.add_edge("approve", "send")
g.add_edge("send", END)

# MemorySaver 把状态存内存；生产环境用 PostgresSaver / RedisSaver
app = g.compile(checkpointer=MemorySaver())

这段代码的关键是 interrupt 的双向通信：它既把信息"抛出去"给人看，又能"接住"人的决策。注意 checkpointer 是必备的——没有持久化，进程一退出状态就没了，HITL 也就无从谈起。生产环境应该用 PostgresSaver 或 RedisSaver，确保状态跨进程、跨重启可恢复。

★ 完整的暂停-恢复调用流程

有了上面的图定义，我们来看外部如何驱动整个 approval 流程。这涉及两次 invoke：第一次触发执行并在 interrupt 处暂停返回，第二次传入人的决策恢复执行。理解这两次调用的衔接，是用好 LangGraph HITL 的关键。下面这段代码完整演示了从启动到恢复的全过程。

# 每个会话用唯一 thread_id 标识，checkpointer 据此隔离不同会话的状态
config = {"configurable": {"thread_id": "user-42-thread-1"}}

# 第一次调用：执行到 approve 节点时，interrupt 会被抛出
result = app.invoke(
    {"user_intent": "通知客户合同下周二签约，时间下午 3 点"},
    config=config,
)
# 此时 result 会包含 __interrupt__ 字段，里面有抛出的草稿内容
print("Agent 暂停，等待审批。草稿：")
print(result["draft_email"])

# ... 这段时间由前端展示草稿、用户点"批准"或"拒绝"或直接编辑 ...

# 第二次调用：用 Command(resume=...) 把人的决策传回去，恢复执行
user_decision = {"approved": True}   # 或 {"approved": False}
final = app.invoke(Command(resume=user_decision), config=config)
print("最终状态：", final)

这里有一个工程要点：thread_id 必须稳定，因为 checkpointer 是按它来定位持久化状态的。如果你的系统是多实例部署，同一个会话的两次 invoke 可能落到不同进程上，只要它们连同一个 checkpointer 后端，状态就能正确恢复。这是 LangGraph 适合生产部署的关键设计。

把这套流程套到三种模式上，差异只在 interrupt 抛出的内容和 resume 传回的内容。approval 模式抛"待审批计划"、收"批准/拒绝"；approval_with_modification 模式抛"AI 草稿"、收"修改后内容并执行"；correction 模式抛"AI 草稿"、收"修改后内容"作为训练金标准（不阻断当次执行）。理解了这一层，你就能灵活组合出任意复杂的 HITL 工作流——比如"双签"（两人审批）、"分级审批"（金额越大审批层级越高）、"自助修正"（AI 自我检查后再交人审）等高级模式。

★ 进阶：动态信任与渐进自动化

HITL 的最高境界不是"什么都让人审"，而是"让该审的审、能自动的自动"。这就需要引入动态信任机制：根据历史数据动态调整哪些决策需要人工介入。核心思路是给每类决策维护一个"信任分数"：分数高时降低审批频率（抽样审），分数低时回到全审批。分数的依据可以是历史批准率、错误代价、用户偏好等。

一个简单而有效的策略是"指数衰减抽样"：某类操作连续被批准 N 次，下一次需要审批的概率降为 1/k；一旦出现一次拒绝或修正，立刻回到 N=0 的全审批状态。这种策略让 HITL 系统随时间"学会放手"——重复出现的低风险操作逐渐自动化，新出现或高风险的操作保持人工把关。它本质上是在线学习的一种朴素形式，不需要训练模型，只用计数器就能实现。下面是这段逻辑的简化实现。

from dataclasses import dataclass

@dataclass
class TrustScore:
    consecutive_approved: int = 0   # 连续被批准的次数
    rejected_once: bool = False     # 是否出现过拒绝/修正

    def needs_review(self, base_rate=1.0):
        """返回本次操作是否需要人工审批。"""
        if self.rejected_once and self.consecutive_approved < 5:
            return True   # 出过问题，恢复期强制全审
        # 连续批准越多，抽样概率越低；最多每 10 次抽 1 次
        import random
        k = min(10, 2 ** self.consecutive_approved // 4)
        return random.random() < (1.0 / max(1, k)) * base_rate

    def update(self, approved: bool):
        if approved:
            self.consecutive_approved += 1
        else:
            self.consecutive_approved = 0
            self.rejected_once = True

# 在 Agent 路由层使用
trust = trust_repo.get(action_type="refund", tenant="acme")
if trust.needs_review():
    decision = interrupt({"action": "refund", "amount": amount})
    trust.update(decision["approved"])
else:
    execute_refund(amount)   # 跳过审批，直接执行
trust_repo.save(action_type="refund", tenant="acme", score=trust)

这种动态信任机制让 HITL 系统从"静态成本"变成"动态投资"：初期人工介入多，随着系统证明自己可靠，介入逐渐减少，最终大部分决策全自动、少数高风险或新模式仍有人把关。这是工业级 Agent 系统走向大规模落地的必经之路。

★ 置信度阈值自动路由：让人审只发生在真正需要时

如果每个请求都走人工，HITL 的成本会吃掉 LLM 带来的所有效率收益。现实中，绝大多数请求其实模型是能搞定的，只有少数低置信度的 case 才需要人介入。所以 HITL 的工程核心不是"怎么暂停"，而是"什么时候该暂停"——这就是置信度阈值自动路由的价值。它的基本逻辑是：高置信度自动执行，低置信度转人工，灰色地带做二次校验。

置信度的来源有好几种。最直接的是模型自己的 logprob 或 token 概率，但现代对话 API 往往不暴露这个，需要用 logprobs 参数显式请求。更稳的做法是让模型自评——加一个"你对这个答案有多确定？给 0-1 的分数"的元问题，虽然模型会过度自信，但校准后排序信号是有效的。还可以用外部判别器：训练一个轻量分类器，输入是 (prompt, response)，输出是"这个回答是否需要人审"的概率，用历史 correction 数据训练，这种学习型路由往往比固定阈值准得多。

路由策略通常是多级漏斗。第一级用 logprob 或自评分做粗筛，置信度 >0.9 直接放行，<0.5 直接拒绝让人兜底；中间的灰色地带进入第二级，调用更贵的判别器或让模型重新生成几次取多数 vote。这种漏斗式路由能把人工审核量压到总请求的 5% 以下，同时保证高风险 case 不被漏掉。阈值不是定死的，要用真实流量做 A/B：把 5% 流量随机全人工审核，作为 ground truth 校准阈值和判别器的召回率。最后强调：HITL 的价值不在暂停本身，而在人类反馈能闭环回到系统——要么即时修改 state 影响当次结果，要么异步进入训练数据改进模型，每一笔人工干预都应让下一次自动通过率提升一点点。

人在回路（HITL）的介入时机设计

人在回路（human-in-the-loop）让 agent 在关键环节拉人进来——但"什么时候拉人"是核心设计问题。拉得太频繁，用户烦死、失去自动化价值；拉得太少，agent 出错没人拦。介入时机的精细设计，决定 HITL 是"恰到好处的协助"还是"烦人的打断"。

介入时机的判断标准是"风险×置信度"。风险高（错了代价大）且 agent 置信度低（不确定）时，必须拉人确认——这是 HITL 的核心场景。风险低或置信度高时，agent 自主执行不打扰。中间地带（中风险或中置信度）可以"异步通知"而非阻塞——告知用户"我准备这样做，有问题请干预"，不强制等待。这种分级介入既保证高风险有人把关，又不让低风险操作被频繁打断。

介入的方式也分级别。同步阻塞：agent 完全停下等用户决定（用于关键决策，如支付前确认）。异步通知：agent 继续，但告知用户可干预（用于中等风险）。事后审核：agent 先执行，事后让人抽检（用于低风险但需监督）。三种方式对应不同介入成本，按场景选。还要设计"介入疲劳"防护——如果频繁触发人工，要么是 agent 能力不足（该改进 agent），要么是阈值设错（该调阈值），不能让用户无止境地被打断。

介入级别触发条件agent 行为适合场景
同步阻塞高风险+低置信停下等决定支付/删除/提交
异步通知中风险/中置信继续+可干预批量操作
事后审核低风险需监督先执行+抽检日常运营
不打扰低风险高置信完全自主查询/读取
escalationagent 卡住转人工接管复杂异常
# HITL 介入时机决策器
class HITLDecider:
    def __init__(self, thresholds=None):
        t = thresholds or {"block_risk":0.7,"block_conf":0.5,
                           "notify_risk":0.4,"notify_conf":0.7}
        self.t = t
        self.intervention_count = {}   # 跟踪介入频率防疲劳

    def decide(self, action, context):
        risk = self._assess_risk(action)         # 0-1
        confidence = self._assess_confidence(action, context)  # 0-1
        # 高风险 + 低置信 → 同步阻塞
        if risk >= self.t["block_risk"] and confidence < self.t["block_conf"]:
            return self._intervene("block", action)
        # 中风险或中置信 → 异步通知
        if risk >= self.t["notify_risk"] or confidence < self.t["notify_conf"]:
            return self._intervene("notify", action)
        # 其余：不打扰，自主执行
        return {"mode":"autonomous"}

    def _intervene(self, mode, action):
        # 介入疲劳防护：频繁触发说明 agent 或阈值有问题
        key = action.type
        self.intervention_count[key] = self.intervention_count.get(key,0)+1
        if self.intervention_count[key] > 10:
            return {"mode":mode, "warning":
                    "此类型频繁需人工，建议提升 agent 能力或调整阈值"}
        if mode == "block":
            return {"mode":"block", "prompt":f"需要您确认：{action.describe()}"}
        return {"mode":"notify", "message":f"将执行：{action.describe()}，可干预"}

    def _assess_risk(self, action):
        if action.type in ("read","query","test"): return 0.1
        if action.type in ("write","deploy"): return 0.6
        if action.type in ("delete","pay","send"): return 0.9
        return 0.3

HITL 的反馈收集与学习闭环

HITL 不只是"让人做决定"，更是"收集人的反馈来改进 agent"的宝贵机会。每次人工介入都产生高质量反馈数据（人对这个 case 的判断、为什么这么决定），这些数据是 agent 学习的黄金素材。把 HITL 从"成本中心"变成"数据金矿"，是成熟 HITL 系统的标志。

反馈收集的关键是"低摩擦 + 结构化"。低摩擦：人工介入时的额外操作要尽量少——一键批准/拒绝比让人写理由好（但可选用）。结构化：不只收集"是/否"决定，要记录决定的"原因"（为什么改、为什么拒），这才是可学习的信号。还要记录"上下文"——当时 agent 想做什么、基于什么信息，便于后续分析 agent 在哪类情况下会犯错。

学习闭环把反馈变成改进。即时学习：当前会话内，把人的决定作为强信号指导 agent 后续行为（人拒绝了 X，本会话别再尝试 X）。离线学习：积累的人工反馈构成训练集，用于微调模型或优化规则。规则提取：从大量人工反馈里发现模式（"涉及金额超 1 万的都需人工"），沉淀成自动规则，减少未来同类情况的人工介入。这个闭环让 agent 越用越聪明，人工介入越来越少（飞轮效应）。

反馈要素收集内容用途
决定批准/拒绝/修改即时指导
原因为什么这么决定可学习信号
上下文agent 当时状态归因分析
结果决定后的效果强化学习
耗时人花多久介入成本评估
# HITL 反馈收集与学习闭环
import json, time
from collections import defaultdict

class HITLFeedbackLoop:
    def __init__(self, store):
        self.store = store   # 反馈数据存储
        self.session_rules = {}   # 即时学习规则

    def collect_intervention(self, action, human_decision, context):
        """收集一次人工介入的完整反馈"""
        record = {
            "ts": time.time(),
            "action": action.to_dict(),
            "decision": human_decision["choice"],   # approve/reject/modify
            "reason": human_decision.get("reason"),  # 结构化原因
            "modification": human_decision.get("modified_to"),
            "context": context,                      # agent 当时状态
            "latency": human_decision["time_spent"], # 人工耗时
        }
        self.store.append(record)
        # 即时学习：把人的决定作为本会话规则
        self._learn_in_session(action, human_decision)
        return record

    def _learn_in_session(self, action, decision):
        """当前会话内应用人的反馈"""
        key = action.type
        if decision["choice"] == "reject":
            # 人拒绝了这类 action，本会话类似action提高警惕
            self.session_rules[key] = {"auto_reject_similar": True,
                                        "learned_from": decision["reason"]}

    def should_intervene_now(self, action):
        """应用即时学习的规则"""
        rule = self.session_rules.get(action.type)
        if rule and self._is_similar(action, rule):
            return True   # 之前人拒绝过类似的，现在直接介入
        return self._default_policy(action)

    def extract_rules_offline(self):
        """从大量人工反馈提取自动规则（减少未来人工）"""
        records = self.store.load_all()
        # 按场景统计：哪类情况经常被人工拒绝
        reject_patterns = defaultdict(int)
        for r in records:
            if r["decision"] == "reject":
                pattern = self._generalize(r["action"])  # 如"金额>1万的删除"
                reject_patterns[pattern] += 1
        # 高频拒绝模式 → 沉淀成自动规则
        new_rules = []
        for pattern, count in reject_patterns.items():
            if count > 5:   # 出现 5+ 次，值得自动化
                new_rules.append({"when":pattern, "action":"auto_reject",
                                  "source":f"从{count}次人工拒绝提取"})
        return new_rules

# 人工介入频率下降，agent 越来越自主

★ 本章小结

人在回路是高风险 AI 系统的安全基石，本章系统梳理了它的理论与实践。首先我们解释了 HITL 存在的必要性——根源在于"错误代价"与"自动化收益"的不平衡，在金融、医疗、法律、对外公关等场景尤为关键；HITL 还附带产生高质量标注数据，是通向更高自动化的阶梯。其次，HITL 有三种核心模式：approval 在执行前阻断等待审批、approval_with_modification 允许人工修改后执行、correction 事后让用户直接修改内容作为训练信号，三者按风险分级组合使用。第三，LangGraph 的 interrupt 机制是实现 HITL 的利器，它通过"抛出待确认内容 + 持久化状态 + Command resume 恢复"的双向通信，让 Agent 能跨进程、跨时间地等待人介入，生产环境务必搭配 PostgresSaver / RedisSaver 等持久化 checkpointer。第四，完整的调用流程是两次 invoke 衔接：第一次触发执行并在 interrupt 暂停，第二次用 Command(resume=...) 传回决策恢复执行，thread_id 是状态定位的锚点。最后，HITL 的高级形态是动态信任机制——用连续批准计数实现指数衰减抽样，让重复的低风险操作逐渐自动化、高风险或新模式保持人工把关，把 HITL 从"静态成本"变为"渐进自动化"。掌握这些，你就能设计出既安全又高效的 Agent 系统，在风险和效率之间找到最优平衡。
