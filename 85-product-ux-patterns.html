AI产品的四种UX模式
对话、流式、错误处理与人在回路

传统软件的交互是"输入-处理-输出"的确定性循环，而 AI 产品的输出本质是概率性的、可变的、有时甚至是错的。这种不确定性倒逼出全新的 UX 模式：对话式让产品有了性格，流式输出缓解了等待焦虑，错误处理教会我们优雅地承认失败，人在回路则把人变成系统的安全阀。本章拆解这四类模式，并给出可落地的设计 checklist 和代码骨架。

阅读约 30 分钟 难度 ★★★

★ 对话式 UX：让产品学会"说话"

对话式 UX 是 ChatGPT 之后被复制最多的形态。它的核心不是"加一个聊天框"，而是把原本分散在表单、按钮、菜单里的功能，统一收拢进一条会话流。用户用自然语言表达意图，系统理解并执行，再把结果以可继续追问的形式返回。这种模式的巨大优势在于降低了学习成本——用户不用记住"哪个按钮在哪"，直接说要什么。

但对话式 UX 也带来了独特的设计挑战。第一，对话有状态，你必须管理上下文窗口、维护多轮记忆、决定何时清空。第二，对话有节奏，模型不能一口气把所有信息倒出来，要分轮次、留追问空间。第三，对话有"性格"，一个生硬的系统指令式回复和一个有人格的助手回复，用户感受天差地别。下面这份 checklist 是上线前必须逐条核对的。

对话式 UX 设计 checklist：

  • 是否维护了会话级上下文（最近 N 轮或 token 上限内）？是否在超出时优雅截断而非报错？
  • 是否提供"重新开始会话"的显式入口，避免历史污染新意图？
  • 系统的每条回复是否给用户明确的下一步（追问建议、快捷操作）？
  • 是否区分"系统消息"和"助手消息"，避免把内部状态泄露给用户？
  • 是否设置了人设（persona），并在所有回复中保持一致语气？
  • 是否支持多模态输入（语音、图片），并在不支持时给出友好提示？
from dataclasses import dataclass, field
from collections import deque
import time, uuid

@dataclass
class Conversation:
    """带滑动窗口的多轮会话管理。"""
    user_id: str
    history: deque = field(default_factory=lambda: deque(maxlen=20))  # 最近 20 条
    system_prompt: str = "你是一名简洁、专业的助手，回答控制在 3 句话内。"

    def add(self, role: str, content: str):
        self.history.append({"role": role, "content": content})

    def to_messages(self):
        # 系统提示始终置于首位，保证人设不被历史稀释
        return [{"role": "system", "content": self.system_prompt},
                *self.history]

# 前端展示时，每条消息都要带唯一 id，便于流式追加与重试
def new_message(role, content=""):
    return {"id": str(uuid.uuid4()), "role": role,
            "content": content, "ts": time.time()}

一个常被忽视的细节：会话窗口的截断策略。简单按条数截断会丢失早期重要上下文，更好的做法是按 token 数截断，并保留系统提示和最近若干轮。否则用户提到第一轮里的某个实体时，模型会因为上下文被切掉而"失忆"。

★ 流式输出 UX：让等待变成参与

大模型生成一段 500 字的回复，端到端可能需要 5 到 15 秒。如果让用户盯着一个转圈等到全部生成完再显示，体验会非常糟糕——心理学称之为"未填充时间感知"。流式输出（streaming）通过逐 token 返回，把用户从"等待者"变成"旁观者"，感知延迟从"15 秒"压缩到"首个 token 到达的 300 毫秒"。这是现代 AI 产品几乎必备的能力。

流式输出的工程实现基于 Server-Sent Events（SSE）或 WebSocket。SSE 更简单，单向服务端推送就够；WebSocket 适合需要双向实时交互的场景（如语音对话）。前端的核心是"逐字渲染 + 打字机效果 + 自动滚动到底部"，并且要让用户能随时点"停止生成"——这一点至关重要，否则用户在模型开始胡说时只能干瞪眼。

流式输出设计 checklist：

  • 是否显示"正在思考"的状态，且首个 token 到达后立刻切换为"正在生成"？
  • 是否提供"停止生成"按钮，且停止后保留已生成内容可编辑？
  • 是否自动滚动到底部，但当用户主动上滑时暂停自动滚动？
  • 是否对 markdown / 代码块做增量渲染，而非等全部生成再解析？
  • 是否在网络中断时自动重连，并恢复之前的生成？
  • 是否对超长输出设置上限（如 4096 token），避免无限生成？
# 后端：FastAPI + SSE 流式返回
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import json

app = FastAPI()

async def stream_chat(messages):
    # 伪代码：实际调用 OpenAI / 自建推理服务
    async for chunk in llm_client.chat_stream(messages):
        delta = chunk.choices[0].delta.content or ""
        # SSE 协议：每条消息以 "data: " 前缀，两个换行结尾
        yield f"data: {json.dumps({'delta': delta})}\n\n"
    yield "data: [DONE]\n\n"

@app.post("/chat")
def chat(payload: dict):
    conv = Conversation(user_id=payload["user_id"])
    for m in payload["messages"]:
        conv.add(m["role"], m["content"])
    return StreamingResponse(
        stream_chat(conv.to_messages()),
        media_type="text/event-stream",
        headers={"Cache-Control": "no-cache", "X-Accel-Buffering": "no"},
    )

关键的 HTTP 头是 X-Accel-Buffering: no，它告诉 Nginx 不要缓冲这个响应，否则你会在生产环境里发现流式效果失效——token 全部攒齐后才一次性返回，前端的逐字渲染就毫无意义了。这个坑踩过的人不在少数。

★ 错误处理 UX：优雅地承认失败

AI 模型会犯错，这是事实。可能输出幻觉、可能拒绝回答、可能触发安全过滤、可能因为输入超长被截断、可能因为限流返回 429。一个好的 AI 产品不是"假装从不犯错"，而是建立一套优雅的错误处理 UX，让用户在遇到问题时知道发生了什么、能做什么。错误的 UX 处理质量，往往直接决定用户留存。

错误处理的核心理念是"可解释 + 可恢复"。可解释意味着错误信息要让用户看懂——不要把 "RateLimitError: 429" 直接抛给用户，而是说"系统繁忙，请稍后再试"。可恢复意味着错误后要给用户一条出路——重试、换个问法、联系人工。下面这份 checklist 把常见错误场景穷举出来。

错误处理设计 checklist：

  • 是否对超时、限流、服务不可用做分类，并展示不同的友好文案？
  • 是否对内容安全过滤做解释，告知用户为何被拒（涉敏、违规）？
  • 是否对幻觉高风险（如医疗、法律、金融）做来源标注或免责声明？
  • 是否提供"重新生成"按钮，且支持调整温度等参数重试？
  • 是否记录错误日志，并对接到监控告警系统？
  • 是否在长上下文截断时提示用户"信息已省略"，避免误导？
class AIError(Exception):
    """统一的 AI 错误基类，前端按 code 映射文案。"""
    def __init__(self, code: str, user_msg: str, retryable: bool = True):
        self.code = code
        self.user_msg = user_msg
        self.retryable = retryable
        super().__init__(user_msg)

# 在调用层做错误归一化
def safe_chat(messages):
    try:
        return llm_client.chat(messages)
    except RateLimitError:
        raise AIError("rate_limit", "系统当前繁忙，请 30 秒后重试。", retryable=True)
    except ContextLengthError:
        raise AIError("too_long", "对话过长，请精简后重试或开启新会话。", retryable=False)
    except ContentFilterError as e:
        raise AIError("filtered", f"内容被安全策略拦截：{e.reason}", retryable=False)
    except TimeoutError:
        raise AIError("timeout", "响应超时，请重试或换个更简短的问题。", retryable=True)
    except Exception:
        # 兜底：永远不要把原始 traceback 暴露给用户
        raise AIError("unknown", "出了点小问题，已记录，请稍后重试。", retryable=True)

# 前端按 code 渲染不同 UI：rate_limit 显示倒计时按钮，filtered 显示警示色等

这里有一个反模式值得警惕：很多团队喜欢把所有错误都包装成 "Something went wrong"。这等于把所有诊断信息都丢掉了，用户既不知道是不是自己的问题，也不知道该不该重试，最后只能放弃。错误文案是 AI 产品里被低估的"微小但关键"的体验设计点。

★ 人在回路 UX：把人变成安全阀

不是所有 AI 决策都该全自动执行。当一个 Agent 要发邮件、转账、删除数据、调用付费 API 时，错误的代价远超重试成本。这时候"人在回路"（Human-in-the-Loop, HITL）就成了必要设计：让 AI 完成大部分工作，但在关键节点暂停，请人确认、修正或反馈，再继续执行。HITL 把"全自动"和"全人工"之间这大片灰色地带填补上，是高价值 AI 产品落地的关键模式。

HITL 在 UX 上有几个变体。最常见的是 approval 模式：AI 生成计划，人点"批准"或"拒绝"后系统才执行。其次是 feedback 模式：AI 输出后请人打分或点赞踩，反馈用于训练或重排。还有 correction 模式：AI 输出可被人直接编辑，编辑后的版本作为"金标准"回流。这三种模式对应不同的信任度和成本权衡，下一章我们会展开 LangGraph 的 interrupt 机制实现，这里先给出 UX 侧的设计要点。

人在回路设计 checklist：

  • 是否清晰标注"待确认"状态，让用户知道系统在等他决策？
  • 是否展示 AI 的完整计划（而非只摘要），便于用户判断风险？
  • 是否提供"批准 / 拒绝 / 修改"三种动作，而不仅是二元选择？
  • 是否在审批超时后设定默认安全行为（默认不执行）？
  • 是否记录每次审批的结果，用于后续自动化阈值的动态调整？
  • 是否对高风险操作（删除、转账）做二次确认或生物校验？
from enum import Enum
from datetime import timedelta

class ApprovalDecision(Enum):
    APPROVED = "approved"
    REJECTED = "rejected"
    REVISED = "revised"

async def run_with_approval(plan, user):
    """Agent 提交计划，等待用户审批后再执行。"""
    # 1) 把 plan 暂存，状态置为 pending_approval
    task = await db.create_task(
        user_id=user.id,
        plan=plan,
        status="pending_approval",
        expires_at=now() + timedelta(minutes=10),  # 10 分钟未审批自动作废
    )
    # 2) 通知用户（站内信 / 邮件 / 推送）
    await notify(user, f"需要你确认一个操作：{plan.summary}")

    # 3) 阻塞等待用户决策（实际用消息队列 / WebSocket 异步唤醒）
    decision = await wait_for_decision(task.id, timeout=600)
    if decision is None:
        await task.update(status="expired")
        return "操作已超时取消"

    if decision == ApprovalDecision.APPROVED:
        result = await execute(plan)            # 真正执行
        await audit_log(user, plan, "executed") # 留痕
        return result
    elif decision == ApprovalDecision.REJECTED:
        await audit_log(user, plan, "rejected")
        return "操作已取消"
    else:  # REVISED：用户修改了计划，重新走一遍
        return await run_with_approval(decision.revised_plan, user)

HITL 的本质是"信任的渐进释放"。一个成熟的做法是动态调整审批阈值：当一个操作类型连续被批准 100 次，可以自动降低审批频率（比如每 10 次抽审一次）；一旦出现一次拒绝或修改，立刻回到全审批状态。这种"信任分数"机制让 HITL 在安全性和效率之间找到自适应平衡，是工业级 Agent 系统的高级模式。

深 AI 产品 UX 模式：信任与控制感

AI 产品的 UX 难点在于如何让用户信任一个会出错的系统。好的 UX 不是隐藏错误，而是给用户控制感和可修正性。下面拆解几个核心 UX 模式及其设计要点。

UX 模式的选择决策

不同场景该用哪种交互模式，取决于任务确定性、错误代价、用户期望。下面是一个模式选择器。

ux_patterns = {
  "对话式": {"适合":"开放式问答、探索", "要点":"支持多轮、流式、可追问", "坑":"用户不知边界"},
  "流式生成": {"适合":"长文本生成", "要点":"首字快、可中途停止", "坑":"出错时已生成大段"},
  "表单/结构化": {"适合":"明确输入输出(抽取、分类)", "要点":"约束输入、校验输出", "坑":"灵活性低"},
  "人在回路": {"适合":"高风险/高代价", "要点":"预览+确认、可回滚", "坑":"打断心流"},
  "渐进披露": {"适合":"复杂结果", "要点":"先概要后细节、可展开", "坑":"用户看不到深层"},
  "错误友好": {"适合":"所有(必备)", "要点":"坦诚错误、给修正路径", "坑":"甩锅、无解法"},
}
def pick_pattern(task_type, risk):
    if risk == "高": return "人在回路"
    if task_type == "开放": return "对话式 + 流式"
    if task_type == "明确": return "表单/结构化"
    return "渐进披露"
print(pick_pattern("开放", "低"))   # 对话式 + 流式

这个选择器的核心原则：风险越高越要人在回路，任务越开放越要对话式。给医疗、金融类高代价动作加确认，是 UX 也是安全的双重保障。

UX 模式解决的问题设计要点反模式
流式生成等待焦虑首字优先、可停止全生成完才显示
人在回路高风险预览、确认、可撤销静默执行不可逆

信任构建与“可修正性”设计

AI 产品 UX 的最高目标是建立并维持用户信任。信任来自三点：可预期（行为一致、不随机崩）、可解释（能给理由、能溯源）、可修正（错了能改、能教）。具体设计上：流式输出不仅降等待焦虑，还让用户“看到 AI 在思考”，增加过程透明度；来源引用让用户能核实关键信息，比“AI 说了算”可信得多；错误友好处理是最被低估的信任建设——坦诚说“我不确定”并提供修正路径（“要不要换个问法/补充信息”），比硬装确定然后出错更能留住信任，用户对“坦诚的局限”容忍度远高于“自信的错误”。可修正性是核心：允许用户编辑 AI 输出、标注错误、提供额外信息，这些不仅是反馈数据，更是给用户控制感——感觉自己能驾驭这个工具，而非被它摆布。人在回路对高风险场景必不可少：让用户预览并确认 AI 的动作（如“即将发送这封邮件，确认？”），既防错又增信任。避免过度承诺：UX 文案不要暗示“永远正确”，而要诚实传达“擅长什么、局限在哪”，管理预期本身就是信任管理。最后，一致性比偶尔惊艳更重要——一个每次都“还不错”的产品，比一个“时好时坏”的产品更让人信任和依赖，所以 UX 设计要追求稳定可预期的体验下限，而非偶尔的高光上限。

深 产品 UX FAQ 与信任

AI 产品 UX 的核心是「让用户信任」，这些问题最关键。

Q1：用户说「AI 总是出错不敢用」，怎么设计 UX 重建信任？

坦诚局限+给控制感。别隐藏错误，而是：显式置信度（不确定就提示「仅供参考」）；来源引用（可点击核实，比硬信强）；优雅降级（低置信引导澄清而非瞎答）；一键反馈和修正（让用户能教它，有控制感）；人在回路（高风险动作确认）。用户对「坦诚不确定+可修正」的容忍度，远高于「自信地出错」。信任来自三点：可预期（一致不崩）、可解释（给理由）、可修正（错了能改）。一次硬装确定然后错，比十次坦诚不确定更伤信任。

Q2：流式输出为什么对 AI 产品这么重要？

降等待焦虑+增加透明度。等 AI 生成完才显示，用户焦虑且不知它在干嘛；流式让用户「看到 AI 在思考/打字」，首字快（TTFT 低）就有「秒回」感，心理等待感大降。还可中途停止（发现方向错立即停，省时间省 token）。实现上用 SSE/WebSocket 流式传输。对长文本生成尤其重要——流式让 10 秒的生成感觉像 2 秒。首字延迟比总延迟更影响体感，所以优化 TTFT（prefix cache、小模型首字）比优化总吞吐对 UX 更直接。AI 产品几乎所有实时交互都该流式，这是基础 UX。

★ 本章小结

AI 产品的 UX 与传统软件截然不同，根源在于输出的概率性和不确定性。本章拆解了四类核心模式。对话式 UX 把功能收拢进会话流，关键在于会话状态管理、节奏控制和一致的人设，必须重视上下文窗口的 token 级截断而非简单按条数截断。流式输出 UX 通过 SSE / WebSocket 逐 token 返回，把感知延迟从"总耗时"压缩到"首 token 延迟"，工程上务必设置 X-Accel-Buffering: no 头避免 Nginx 缓冲，并提供停止生成与自动滚动控制。错误处理 UX 的核心理念是"可解释 + 可恢复"，要按错误类型分类展示友好文案并给出重试路径，绝不能把原始 traceback 暴露给用户。人在回路 UX 在高风险决策节点引入人工审批，包含 approval / feedback / correction 三种变体，本质是"信任的渐进释放"，可通过信任分数动态调整审批频率。四类模式并非孤立，而是层层叠加：一个成熟的 AI 产品往往同时具备对话界面、流式渲染、优雅的错误处理和关键节点的人审机制。掌握这套模式语言，你就能从"能跑的 demo"迈向"用户愿意持续使用的产品"。
