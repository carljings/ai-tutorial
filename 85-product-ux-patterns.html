AI 产品 UX 模式
对话、流式、错误处理与人在回路

技术再强，如果用户不会用、用得不爽，产品就失败了。AI 产品有一套独特的 UX（用户体验）设计模式，和传统软件很不一样——怎么设计对话界面、怎么做流式输出体验、AI 犯错了怎么优雅处理、什么时候该让人介入。这一章拆解四类核心 UX 模式，每个都配设计原则和实现代码，让你的 AI 产品不仅「能用」，还「好用」。

阅读约 27 分钟 难度 ★★★

★ 一、为什么 AI 产品需要专门的 UX 设计

AI 产品的输出是概率性的、可能是错的、需要时间生成的。这三个特性让它的 UX 设计和传统软件完全不同：

  • 延迟更长：传统软件毫秒级响应，AI 要等几秒甚至几十秒。
  • 输出不确定：传统软件给确定结果，AI 可能给错答案。
  • 需要用户引导：用户不知道怎么和 AI 对话，需要界面引导。

好的 AI UX 设计能把这些「劣势」转化为「优势」——让等待变可接受（流式输出），让不确定变透明（置信度展示），让不会用变会引导（示例和模板）。

★ 二、对话模式：不只是聊天框

对话界面（Chat UI）是 AI 产品最常见的交互形式，但很多人只做一个简单的输入框+输出区。优秀的对话 UX 要考虑更多细节。

★ 三、流式模式：让等待变可接受

流式输出（streaming）是 AI 产品最重要的 UX 优化之一。比起「等 10 秒一次性显示」，用户更喜欢「立刻看到第一个字，然后逐字输出」。这把感知延迟从 10 秒降到了 1 秒。

3.1 流式输出的 UX 细节

// 流式输出的前端实现（核心细节）
async function streamResponse(message) {
    // 1. 立即显示用户消息 + AI 的「正在输入」状态
    appendMessage("user", message);
    const aiMessage = appendMessage("ai", "");
    showTypingIndicator(aiMessage);

    // 2. 发起流式请求
    const response = await fetch("/api/chat", {
        method: "POST",
        body: JSON.stringify({message}),
    });
    const reader = response.body.getReader();
    const decoder = new TextDecoder();

    // 3. 移除「正在输入」，开始逐字渲染
    removeTypingIndicator(aiMessage);

    let fullText = "";
    while (true) {
        const {done, value} = await reader.read();
        if (done) break;

        const chunk = decoder.decode(value);
        // 解析 SSE 数据
        const lines = chunk.split("\n");
        for (const line of lines) {
            if (line.startsWith("data: ")) {
                const data = JSON.parse(line.slice(6));
                if (data.content) {
                    fullText += data.content;
                    // 增量渲染（只更新变化的部分，性能好）
                    updateMessage(aiMessage, fullText);
                }
            }
        }
    }

    // 4. 流结束后，启用操作按钮（复制/重新生成/反馈）
    enableActions(aiMessage);
}

4.2 错误处理的前端实现

// 优雅的错误处理实现
async function safeChat(message) {
    try {
        showLoading();
        const response = await callAI(message);

        // 检查是否被安全过滤
        if (response.filtered) {
            showFriendlyMessage(
                "抱歉，我无法回答这个问题。可以换个角度提问吗？",
                type: "warning"
            );
            return;
        }

        // 检查置信度
        if (response.confidence < 0.5) {
            showResponseWithWarning(
                response.text,
                "我对这个回答不太确定，建议核实一下。"
            );
        } else {
            showResponse(response.text);
        }

    } catch (error) {
        if (error.type === "timeout") {
            showErrorWithRetry(
                "响应超时了。可能是问题太复杂，再试一次？",
                () => safeChat(message)  // 重试回调
            );
        } else if (error.type === "rate_limit") {
            showErrorWithCountdown(
                "请求太频繁了，请等待 X 秒后再试",
                cooldownSeconds: error.retry_after
            );
        } else {
            showErrorWithRetry(
                "出了点小问题。你的内容已保存，点击重试。",
                () => safeChat(message)
            );
        }
    } finally {
        // 无论如何，恢复输入框
        enableInput();
        // 保存用户输入，防止丢失
        saveDraft(message);
    }
}

★ 六、建立信任：AI 产品的核心 UX 目标

AI 产品最大的 UX 挑战不是「好不好用」，而是「信不信任」。用户如果不信任 AI，功能再强也不会用。建立信任的几个设计原则：

# 建立 AI 信任的设计原则
trust_building_principles = {
    "透明度": """
    - 明确告诉用户这是 AI 生成的（不要假装是人）
    - 展示 AI 的能力和局限（「我擅长 X，不擅长 Y」）
    - 提供信息来源（RAG 场景标注引用）
    - 让用户看到 AI 的「思考过程」（CoT 或步骤展示）
    """,

    "一致性": """
    - 相似的问题给出相似的回答（降低 temperature）
    - 输出格式稳定（用户能预测 AI 的行为）
    - 风格一致（不要忽正式忽随意）
    """,

    "可纠错": """
    - 用户能轻松纠正 AI 的错误
    - AI 能记住用户的纠正（下次不犯同样错误）
    - 提供反馈通道，让用户觉得「我的意见被重视」
    """,

    "渐进信任": """
    - 新用户先体验低风险功能（问答、建议）
    - 信任建立后再开放高风险功能（自动执行、写操作）
    - 不要一上来就让 AI 做不可逆的事
    """,

    "诚实": """
    - AI 不确定时明确说「我不确定」
    - AI 不会时说「我做不到」，而不是编造
    - 出错时承认错误，不要狡辩
    """,
}

★ 本章小结

这一章我们覆盖了 AI 产品的四类核心 UX 模式。核心要点：

  • 对话模式：不只是聊天框，要设计输入区、消息展示、上下文管理、操作便利。空状态要有示例引导。
  • 流式模式：把感知延迟从 10 秒降到 1 秒，配合「思考中」反馈和可中断设计。
  • 错误处理：分技术错误、内容错误、安全拦截、能力边界四类，每类有对应的友好处理方式。核心是保存用户输入、提供重试、友好文案。
  • 人在回路：确认后执行、选择而非填写、渐进式披露、可撤销。让用户参与关键决策但不烦人。
  • 建立信任：透明度、一致性、可纠错、渐进信任、诚实。信任是 AI 产品最宝贵的资产。

UX 是 AI 产品成功的最后一公里——技术让产品能用，UX 让产品好用。这也是整个实战案例系列的最后一章。从第 51 章的数学基础到这里，你已经完成了从理论到工程、从技术到产品、从个人到团队的完整旅程。接下来，就是把这些知识用起来的实践了。
