AI 法规合规
EU AI Act、中国生成式 AI 办法与美国行政令全景

2024-2026 年是全球 AI 监管密集落地的三年。EU AI Act 正式生效，中国《生成式人工智能服务管理暂行办法》已经让一大批应用下架整改，美国的行政令也在重塑行业格局。对 AI 从业者来说，合规不再是「法务部门的事」，而是直接影响产品设计、技术选型、甚至能不能上线的硬约束。这一章拆解三大法域的核心要点，并给出可执行的合规 checklist。

阅读约 28 分钟 难度 ★★★

★ 一、为什么技术人必须懂合规

很多工程师对法规有天然的抵触，觉得「这是限制创新」。但在 AI 领域，合规无知是致命的。举几个真实案例：

  • 2024 年，意大利数据保护局率先封禁 ChatGPT，原因是隐私合规问题，OpenAI 不得不紧急整改才解封。
  • 中国多款 AI 应用因「未做算法备案」「生成内容违规」被下架，整改周期动辄几个月，错失市场窗口。
  • 欧盟 GDPR 罚款上限是全球营收的 4%，对大公司是几十亿欧元。

合规的本质不是「限制」，而是「让你能长期合法地做生意」。把合规前置到产品设计阶段，比事后整改便宜十倍。这一章帮技术人建立合规认知，不是让你成为律师，而是让你知道红线在哪。

★ 二、EU AI Act：全球第一部综合 AI 法

欧盟的《人工智能法案》（AI Act）2024 年 8 月正式生效，是全球第一部全面规制 AI 的法律。它的核心思想是「基于风险的分层监管」——按 AI 系统的风险等级施加不同程度的义务。

2.1 四级风险分类

  • 不可接受风险（禁止）：全面禁止。比如社会评分系统、实时人脸识别（公共场所）、操纵人类行为的 AI。这些直接违法。
  • 高风险：允许但严格监管。涵盖医疗设备、招聘筛选、信用评分、司法辅助、关键基础设施等。必须做合规评估、数据治理、人类监督、日志留存。
  • 有限风险：透明度义务。比如聊天机器人必须告知用户「你在和 AI 对话」，deepfake 必须标注「AI 生成」。
  • 最小风险：基本不管。比如垃圾邮件过滤、游戏 AI。
# 判断你的 AI 应用属于哪一级风险（决策树示例）
def classify_risk_level(application):
    """根据应用特征判断 EU AI Act 风险等级。"""
    # 不可接受风险清单
    forbidden = ["social_scoring", "realtime_biometric_in_public",
                 "manipulative_behavior", "exploiting_vulnerabilities"]
    if any(feature in application["features"] for feature in forbidden):
        return "不可接受风险（禁止）"

    # 高风险清单
    high_risk_domains = ["medical_device", "recruitment", "credit_scoring",
                         "judicial_system", "critical_infrastructure",
                         "education_admission", "law_enforcement"]
    if application["domain"] in high_risk_domains:
        return "高风险"

    # 有限风险：需要透明度
    if application.get("interacts_with_humans") or application.get("generates_content"):
        return "有限风险（需透明度）"

    return "最小风险"

# 案例：一个 AI 简历筛选系统
resume_screener = {
    "domain": "recruitment",           # 招聘属于高风险
    "features": ["automated_filtering"],
    "interacts_with_humans": True,
}
print(classify_risk_level(resume_screener))  # 高风险

2.2 高风险 AI 的核心义务

如果你的应用被判定为高风险，必须履行以下义务：建立完整的风险管理系统、使用高质量训练数据（避免偏见）、详细的日志和可追溯性、人类监督机制、上市前合规评估、在欧盟数据库注册。这些义务从设计阶段就要考虑，事后补几乎不可能。

★ 三、中国生成式 AI 管理办法

中国在 2023 年 8 月实施了《生成式人工智能服务管理暂行办法》，是全球最早针对生成式 AI 的专门法规。它的特点是与时俱进、执行力度强，对国内 AI 从业者影响直接。

3.1 核心要求

  • 算法备案：提供公众服务的生成式 AI 必须在网信办做算法备案，未备案不得上线。
  • 训练数据合规：训练数据来源必须合法，尊重知识产权，不得含有违法内容。
  • 内容安全：生成内容必须符合社会主义核心价值观，不得生成违法、虚假、有害信息。
  • 标识义务：AI 生成的图片、视频必须显式或隐式标识「AI 生成」。
  • 个人信息保护：涉及个人信息的，必须遵守《个人信息保护法》，取得用户同意。
# 中国生成式 AI 上线的合规 checklist
compliance_checklist_china = {
    "上线前必须完成": [
        "算法备案（网信办）",
        "安全评估（含内容安全、数据安全）",
        "ICP 备案（如果是网站/App）",
        "大模型服务备案（如适用）",
        "用户协议和隐私政策更新",
    ],
    "技术层面": [
        "内容安全过滤（敏感词、违法内容）",
        "训练数据清洗和合规审查",
        "用户输入实时审核",
        "生成内容 AIGC 标识（图片/视频水印）",
        "日志留存 6 个月以上",
    ],
    "运营层面": [
        "建立用户投诉举报机制",
        "未成年人保护机制",
        "模型更新后的重新评估",
        "应急响应预案",
    ],
}

# 每一项都有对应的违规后果：警告、罚款、下架整改
# 实际操作中，算法备案通常需要 2-6 个月，要提前规划

3.2 实操要点

在国内做生成式 AI，最耗时的是算法备案和安全评估。建议产品立项时就启动备案流程，因为审核周期长。内容安全是技术重点——必须在模型输出前加一层安全过滤，拦截违法有害内容。这块有成熟的开源方案（如敏感词库、内容分类模型），但必须结合业务场景定制。

★ 四、美国行政令与行业自律

美国的 AI 监管走的是「行政令 + 行业自律」路线，没有像欧盟那样的统一立法。拜登政府 2023 年 10 月签署的《关于安全、可靠、可信 AI 的行政令》是当前最重要的联邦层面文件。

4.1 行政令的核心内容

  • 红队测试要求：开发超大规模模型（算力超过阈值）的开发者必须向政府提交安全测试结果。
  • 标准制定：NIST 牵头制定 AI 安全评估标准（AI RMF）。
  • 联邦机构使用：各联邦机构使用 AI 必须做风险评估和透明度报告。
  • 内容标识：要求商务部制定 AI 生成内容的标识标准。

4.2 各州法律

美国各州在填补联邦立法空白。加州的 SB-53（关注 AI 安全）、科罗拉多州的 AI 反歧视法（要求算法透明）、伊利诺伊州的 BIPA（生物信息保护）都对实际业务有影响。如果面向美国市场，必须做各州法律的差异分析。

★ 七、Privacy by Design：把合规写进代码

最高级的合规不是事后审查，而是从设计阶段就把合规要求嵌入系统。这叫 Privacy by Design（隐私保护设计）。几个工程实践：

# 隐私保护设计的工程实践示例

class PrivacyAwareLLMService:
    def __init__(self):
        self.log_retention_days = 180    # 日志保留 180 天
        self.pii_filter = PIIDetector()  # 个人信息检测器

    def process_request(self, user_input, user_id):
        # 1. 输入端：检测并脱敏个人信息
        sanitized, detected = self.pii_filter.detect_and_mask(user_input)
        if detected:
            self.log_pii_attempt(user_id)  # 记录（脱敏后）

        # 2. 处理：调用模型，但不传原始 PII
        response = self.call_model(sanitized)

        # 3. 输出端：内容安全过滤
        if not self.content_filter.is_safe(response):
            response = self.get_safe_fallback()

        # 4. 日志：只记必要信息，自动过期
        self.log_interaction(user_id, sanitized, response, ttl=self.log_retention_days)

        # 5. 用户权利：提供查询和删除接口
        return response

    def delete_user_data(self, user_id):
        """响应用户的数据删除请求（GDPR/PIPL 要求）。"""
        self.log_db.delete_by_user(user_id)
        self.cache.invalidate_user(user_id)
        return {"status": "deleted", "user_id": user_id}

深 全球 AI 法规对照与合规落地

AI 法规在全球快速分化：欧盟重风险分级、美国重行政引导、中国重生成式内容。做全球产品的团队必须理解三套体系的差异，并做合规适配。下面把核心差异结构化。

按风险分级的合规自查

多数法规按风险分级监管。先判断你的产品属于哪一级，再对应履行义务。下面是一个风险分级自查脚本。

def classify_risk(features):
    """按 EU AI Act 思路做风险分级自查。"""
    if features.get("管控关键基础设施") or features.get("影响基本权利"):
        return "不可接受风险", "禁止或极严格监管"
    if features.get("用于招聘/信贷/司法/教育") or features.get("生物识别"):
        return "高风险", "需注册、风险评估、人工监督、日志留存"
    if features.get("生成内容") or features.get("与用户交互"):
        return "有限风险", "需透明告知(标注AI生成、可识别合成)"
    return "最小风险", "无特殊义务，鼓励自律"

# 中国生成式AI办法额外关注：训练数据合法性、生成内容真实准确、标注服务规范
features = {"生成内容": True, "用于招聘/信贷/司法/教育": False}
level, duty = classify_risk(features)
print(f"风险等级: {level}\n合规义务: {duty}")
# 输出会提示：作为生成式AI，必须对用户透明告知、标注AI生成内容

合规落地的工程化清单

把法规要求翻译成工程动作，是合规落地的关键。几个必做的工程项：AI 生成内容标注（文本加水印或声明、图片加元数据标识），这是多数法规的透明性要求；训练数据审计台账，记录每批数据来源和授权，应对“数据合法”审查；用户数据删除机制，支持 GDPR 的“被遗忘权”，能按用户要求删除其数据；内容安全过滤，对生成内容做合规检测，拦截违法有害信息（中国法规的重点）；风险评估文档，对高风险系统留存风险评估记录和日志，供监管核查。跨境数据流动也要注意：用户数据出境需符合当地数据出境规则（中国的数据出境安全评估）。合规不是法务一个人的事，而是产品、工程、法务协同的系统工程——越早把合规要求纳入设计，改造成本越低；临上线才发现合规问题，往往要伤筋动骨。把合规当成“护城河”而非“负担”，能让你在监管收紧时从容应对，而临时补课的竞品会很狼狈。

Q1：我的 AI 产品需要做「安全评估」吗？

Q2：AI 生成的内容要标注「AI 生成」吗？

多数法域要求透明告知。中国规定深度合成内容应显著标识；欧盟 AI Act 要求有限风险系统透明告知；AIGC 内容应可被识别（如水印、元数据）。工程实现：文本加声明（「以上由 AI 生成」）、图片加不可见水印（元数据标识）、视频/音频加合成标识。这是透明性义务，多数情况必须做，不做可能被监管处罚。还要保留生成日志便于追溯。合规要趁早纳入产品设计，而非事后补——产品架构层面支持（如内容带 provenance 元数据）比事后加标注更彻底。

★ AI 监管合规的落地清单与节奏

面对日益密集的 AI 监管政策（无论是数据保护、算法备案、生成内容标识还是高风险应用的准入要求），最常见的两个极端反应是：要么完全忽视，等到政策落地才仓促应对；要么过度反应，把所有功能都按最高风险等级处理，导致研发效率被拖垮。正确的做法是基于业务实际影响，做"分级合规"——先识别哪些功能属于高风险（如涉及医疗、金融、就业决策、个人敏感信息），对这些做严格审查；对低风险功能则采用轻量化的自查清单，保持迭代速度。

合规工作的一个关键认知是：它不是一次性验收，而是持续义务。数据来源的合法性、模型训练记录的可追溯、生成内容的可审计，这些都需要在系统设计阶段就嵌入，而非事后补录。例如，训练数据的"数据卡片"（datasheet）、模型决策的日志留痕、用户对生成内容的反馈通道，都应当作为默认功能而非可选附件。一旦系统在设计时没有这些能力，后期补齐的成本会非常高。

跨国业务还需要特别关注"管辖权差异"。不同地区对同一类应用（如人脸识别、自动化决策、生成式内容）的态度和具体要求差别很大。建议为每个主要市场维护一份"合规差异表"，并在架构上预留"按地区开关功能"的能力，避免一个地区的监管变化导致全球产品被迫下线。同时，与法务团队建立定期同步机制，因为监管政策本身也在快速演化。

★ 跨境数据与算法审计的工程要求

当 AI 产品跨境运营时，数据合规的复杂度会呈指数级上升。不同司法辖区对"个人数据"的定义、跨境传输的门槛、数据本地化存储的要求各不相同。例如某些地区要求用户数据必须存储在境内服务器，某些地区对"自动化决策"赋予了用户拒绝权，还有些地区对生成式 AI 内容强制要求显著标识。这些差异不能靠事后补丁解决，而必须在架构设计阶段就引入"数据主权层"——根据用户所属地区路由数据存储、模型推理、日志留存策略。

算法审计是另一个工程密集型要求。越来越多监管要求企业能够证明其模型决策的可解释性与公平性，这意味着系统必须完整记录：模型版本、输入特征、决策结果、以及影响决策的关键因子。这些审计日志的粒度、保留期限、访问权限都需要预先设计。一个常见错误是只在"出事"后才试图从分散的日志中拼凑决策链路——此时往往已经丢失了关键信息，且重建成本极高。正确做法是把审计日志作为系统的一等公民，与业务日志分离存储，并定期做完整性校验。

此外，合规要求的变化频率远高于产品迭代频率，因此合规能力本身需要"可配置化"。例如新增一个地区的备案要求时，不应当需要修改核心代码，而应当通过配置开关启用对应的审计模块、数据路由规则、用户通知模板。这种把合规规则与业务逻辑解耦的设计，是规模化合规的关键工程能力。同时，建议为合规团队提供只读的数据访问面板，使其能独立核验而不依赖工程团队的人工配合。

★ 本章小结

这一章我们梳理了三大法域的 AI 监管框架和实操要点。核心结论：

  • EU AI Act：基于风险的四级分类，高风险应用（医疗、招聘、信用等）义务最重，设计阶段就要考虑合规。
  • 中国生成式 AI 办法：算法备案是上线前提，内容安全是技术重点，备案周期长要提前规划。
  • 美国行政令：联邦以行政令 + NIST 标准为主，各州法律差异大，面向美国市场要做州级分析。
  • 跨境业务：数据本地化、内容标识、用户知情同意是三大共性要求，工程上要分区部署。
  • 合规 Checklist：数据、模型、透明度、备案四大类，立项时跑一遍，上线前再审一遍。
  • Privacy by Design：最高级的合规是写进代码——输入脱敏、输出过滤、日志过期、用户数据删除接口。
