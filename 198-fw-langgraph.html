LangGraph

agent 的图时代：状态 + 节点 + 边

深度进阶★★★

为什么需要 LangGraph

LangChain 的 AgentExecutor 用固定循环跑 agent——对简单场景够用，但复杂场景（条件分支、循环、人工介入、多 agent）难以表达。LangGraph 把 agent 建模成"状态图"（StateGraph）：每个处理步骤是一个节点，步骤间的流转是边，可以条件分支、可以循环、可以并行。这就像用流程图来设计 agent，表达力远超固定循环。

三大核心概念：State / Node / Edge

概念是什么类比
State在节点间流转的共享数据（TypedDict）流水线上的产品
Node一个处理步骤（Python 函数）流水线上的工位
Edge节点间的连接（固定/条件）传送带的方向
from langgraph.graph import StateGraph, END
from typing import TypedDict, Annotated

# 1. 定义状态：节点间流转的数据
class AgentState(TypedDict):
    messages: Annotated[list, "add_messages"]  # 消息历史
    next: str  # 下一步该去哪

# 2. 定义节点（每个是一个函数）
def call_model(state: AgentState):
    response = model.invoke(state["messages"])
    return {"messages": [response]}

def call_tool(state: AgentState):
    # 执行工具调用
    ...
    return {"messages": [tool_result]}

# 3. 构建图
graph = StateGraph(AgentState)
graph.add_node("agent", call_model)
graph.add_node("tools", call_tool)
graph.set_entry_point("agent")

# 4. 条件边：根据状态决定下一步
def should_continue(state):
    last_msg = state["messages"][-1]
    if last_msg.tool_calls:
        return "tools"   # 有工具调用，去 tools 节点
    return END            # 没有了，结束

graph.add_conditional_edges("agent", should_continue)
graph.add_edge("tools", "agent")  # tools 完了回 agent

# 5. 编译运行
app = graph.compile()
result = app.invoke({"messages": [user_message]})
💡 要点
LangGraph 的精髓是"条件边"——它让你用代码精确描述 agent 的控制流。ReAct 循环只是"agent → tools → agent"的循环；更复杂的（多 agent 路由、人工审批分支、错误重试）都能用图清晰表达。这是它取代 AgentExecutor 的根本原因。

LangGraph 的高级能力

除了基本的图结构，LangGraph 还支持：持久化（checkpointer，状态可存盘恢复，支持断点续传和 time-travel）、人工介入（interrupt，在某节点暂停等人确认再继续）、子图（把复杂逻辑封装成子图复用）、流式（逐 token 流式输出）。这些是生产级 agent 必备的能力。

StateGraph 构建实战：一个带工具调用和审批的完整 agent

前面的入门示例展示了 StateGraph 的基本用法，现在构建一个更贴近生产的完整 agent——它具备工具调用、条件路由、人工审批分支。这个例子覆盖了 LangGraph 最常用的模式，是构建生产级 agent 的模板。

# LangGraph 完整 agent：工具调用 + 人工审批 + 错误处理
from langgraph.graph import StateGraph, END
from typing import TypedDict, Annotated, Literal
from langgraph.graph.message import add_messages

# 1. 定义状态（agent 运行中流转的所有数据）
class AgentState(TypedDict):
    messages: Annotated[list, add_messages]  # 消息历史（自动累积）
    needs_approval: bool                     # 是否需要人工审批
    retry_count: int                         # 重试次数（防死循环）
    tool_results: list                       # 工具调用结果

# 2. 定义节点
def call_model(state: AgentState):
    """主节点：调 LLM 决定下一步。"""
    response = model.invoke(state["messages"])
    # 判断是否需要人工审批（高风险操作）
    needs_approval = any(tc["name"] in DANGEROUS_TOOLS
                        for tc in response.tool_calls)
    return {"messages": [response], "needs_approval": needs_approval}

def call_tools(state: AgentState):
    """工具节点：执行 LLM 请求的工具调用。"""
    last_msg = state["messages"][-1]
    results = []
    for tc in last_msg.tool_calls:
        try:
            result = tools[tc["name"]].invoke(tc["args"])
            results.append(ToolMessage(result, tool_call_id=tc["id"]))
        except Exception as e:
            results.append(ToolMessage(f"工具出错：{e}", tool_call_id=tc["id"]))
    return {"messages": results, "retry_count": state.get("retry_count", 0) + 1}

def human_review(state: AgentState):
    """人工审批节点：暂停等用户确认。"""
    # interrupt 让图暂停，等用户输入后才继续
    approval = interrupt({
        "question": "以下操作需要确认：",
        "tool_calls": state["messages"][-1].tool_calls,
    })
    if approval == "approved":
        return {"messages": state["messages"]}  # 继续，去执行工具
    return {"messages": [HumanMessage("用户拒绝了该操作")]}

# 3. 定义条件边（图的核心：控制流逻辑）
def route_after_model(state: AgentState) -> Literal["tools", "human", "__end__"]:
    """根据 LLM 输出决定下一步去哪。"""
    last_msg = state["messages"][-1]
    if not last_msg.tool_calls:
        return END                        # 没有工具调用，任务完成
    if state.get("needs_approval"):
        return "human"                    # 高风险操作，先审批
    return "tools"                        # 普通操作，直接执行

def route_after_tools(state: AgentState) -> Literal["agent", "__end__"]:
    """工具执行完，根据重试次数决定。"""
    if state.get("retry_count", 0) >= 5:
        return END                        # 防止无限循环
    return "agent"                        # 回到 agent 继续推理

# 4. 构建图
graph = StateGraph(AgentState)
graph.add_node("agent", call_model)
graph.add_node("tools", call_tools)
graph.add_node("human", human_review)

graph.set_entry_point("agent")
graph.add_conditional_edges("agent", route_after_model)  # 条件分支
graph.add_edge("tools", "agent")                          # 固定边
graph.add_conditional_edges("human", lambda s: "tools")   # 审批通过去工具

条件边深入：图的"大脑"

条件边是 LangGraph 区别于固定循环的核心。它本质是一个函数：输入当前状态，输出"下一个该去哪个节点"。这个函数可以包含任意复杂的判断逻辑——基于 LLM 的输出、基于状态计数器、基于外部条件。下面展示几种常见的条件边模式。

# 条件边的常见模式

# 模式1：路由分发（一个节点根据内容分发到不同处理器）
def route_by_intent(state) -> str:
    """根据用户意图路由到不同专家节点。"""
    intent = classify(state["messages"][-1].content)
    routing = {
        "coding": "code_expert",
        "writing": "writing_expert",
        "math": "math_expert",
        "chat": END,
    }
    return routing.get(intent, END)

graph.add_conditional_edges("router", route_by_intent)

# 模式2：循环 + 退出条件（ReAct 循环的本质）
def should_continue(state) -> str:
    """ReAct：有工具调用就继续，没有就结束。"""
    last = state["messages"][-1]
    if last.tool_calls and state.get("steps", 0) < 10:  # 步数上限
        return "tools"
    return END

# 模式3：错误处理分支
def handle_error(state) -> str:
    """出错时决定重试还是降级。"""
    if state.get("error_count", 0) < 3:
        return "retry"       # 重试
    return "fallback"        # 重试够了，降级处理

# 模式4：质量门控（生成后评估，不够好就重来）
def quality_gate(state) -> str:
    """评估输出质量，决定接受还是重新生成。"""
    quality = evaluate_quality(state["messages"][-1].content)
    if quality > 0.8:
        return END              # 质量达标，输出
    if state.get("regen_count", 0) < 2:
        return "regenerate"     # 不够好，重新生成
    return END                  # 重生成了还不行，凑合用

这四种模式覆盖了 90% 的 agent 控制流需求：分发（按内容路由到不同处理）、循环（ReAct 的核心）、错误处理（重试与降级）、质量门控（评估后决定是否接受）。用条件边把这套逻辑表达出来，比在 AgentExecutor 里写一堆 if-else 清晰得多——因为图的结构本身就是流程图，一眼能看出控制流。

Checkpoint 持久化：让 agent 能"存档"和"读档"

生产级 agent 有个刚需：状态要能持久化。原因有二：① 断点续传——长任务中途崩了或要暂停，能从上次的状态恢复；② 人工介入——agent 跑到某步暂停等人审批，审批完继续。LangGraph 的 checkpointer 就是为此设计——它把每个节点执行后的状态快照存下来，支持"读档"到任意一步（甚至回退）。

# Checkpoint 持久化：状态存档 + 断点续传 + time-travel
from langgraph.checkpoint.memory import MemorySaver  # 内存版
from langgraph.checkpoint.sqlite import SqliteSaver   # SQLite 持久化
import sqlite3

# 1. 带持久化的编译
conn = sqlite3.connect("agent_checkpoints.db", check_same_thread=False)
checkpointer = SqliteSaver(conn)

app = graph.compile(
    checkpointer=checkpointer,
    interrupt_before=["human"],  # 在 human 节点前暂停（等审批）
)

# 2. 每个会话有唯一 thread_id（状态按会话隔离）
config = {"configurable": {"thread_id": "user-123-session-1"}}

# 第一次运行：跑到 human 节点前暂停
result = app.invoke(
    {"messages": [HumanMessage("删除生产环境的旧数据库")]},
    config=config,
)
# agent 在"删数据库"前暂停了（因为 interrupt_before=["human"]）

# 3. 查看当前状态（暂停在哪、状态是什么）
state = app.get_state(config)
print(f"暂停在节点：{state.next}")  # ('human',)
print(f"当前消息数：{len(state.values['messages'])}")

# 4. 人工审批后继续（提供审批结果，恢复执行）
app.update_state(config, {"approval": "approved"})
result = app.invoke(None, config=config)  # None = 从暂停处继续

# 5. Time-travel：回退到之前某一步（撤销错误决策）
history = list(app.get_state_history(config))
# 找到"调工具前"的那一步
target = [h for h in history if h.next == ("tools",)][0]
# 从那一步重新开始（换条路走）
app.invoke(None, config={"configurable": {
    "thread_id": "user-123-session-1",
    "checkpoint_id": target.config["configurable"]["checkpoint_id"],
}})
💡 生产价值
Checkpoint 持久化是 LangGraph 杀手级能力，它解决了 agent 的两大生产难题：① 人工介入（interrupt）——agent 自动跑到需要人决策的点暂停，人审完再继续，这是"人在环中"(Human-in-the-loop) 系统的基础；② 可恢复性——长任务（如深度研究、批量处理）中途断了能续上，不用从头重来。配合 time-travel，还能回退重跑（比如发现某步决策错了，回退到那步换条路）。这些是 AgentExecutor 完全做不到的，也是 LangGraph 成为生产级首选的原因。

LangGraph 的状态设计与持久化

LangGraph 把 agent 建模成状态机/图，核心抽象是 State（贯穿整个图的状态对象）和 Node（修改状态的函数）。这套模型的能力上限，取决于 State 设计得好不好。State 设计不当，会导致状态膨胀、节点耦合、调试困难。生产化 LangGraph 的第一课就是"把 State 设计对"。

State 设计的第一原则是"显式声明所有流转字段"。LangGraph 用 TypedDict 或 dataclass 定义 State schema，每个字段都是图节点间共享的数据。新手常犯的错是 State 里塞太多临时字段（某节点算的中间结果其实只自己用），导致状态臃肿、每个节点都能看到不相关数据。正确做法是 State 只放"需要跨节点共享"的字段，节点内部的临时变量用普通局部变量，不进 State。

第二原则是"明确字段的合并语义"。LangGraph 支持字段的 reducer（如 messages 字段用 add reducer，新消息追加而非覆盖）。不指定 reducer 默认是覆盖，容易出 bug——比如 messages 如果不指定 add，每步会覆盖前一步的消息。第三原则是"状态可序列化"——为了支持 checkpoint（断点恢复）、time travel（回滚到历史状态）、分布式执行，State 的所有字段必须可序列化（避免放 file handle、数据库连接等不可序列化对象）。

State 设计原则做法违反的后果
显式声明TypedDict 定义所有字段隐式传递难追踪
最小共享只放跨节点数据状态臃肿、节点耦合
合并语义指定 reducer（add/自定义）消息被覆盖丢失
可序列化只用 JSON 兼容类型checkpoint 失败
版本兼容新增字段给默认值旧 checkpoint 无法恢复
# LangGraph State 设计：规范 vs 反模式
from typing import TypedDict, Annotated, Literal
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages

# ✅ 规范：显式 schema + reducer + 最小共享
class ResearchState(TypedDict):
    messages: Annotated[list, add_messages]   # reducer: 追加而非覆盖
    topic: str                                 # 跨节点共享的研究主题
    sources: list                              # 跨节点共享的来源
    draft: str                                 # 当前草稿（覆盖语义）
    # 注意：节点内部临时变量（如循环计数）不进 State，用私有变量

def research_node(state: ResearchState):
    # 临时变量不污染 State
    local_cache = {}
    for query in expand_queries(state["topic"]):
        local_cache[query] = search(query)   # 临时
    sources = flatten(local_cache.values())
    return {"sources": sources, "messages": [("ai", f"找到 {len(sources)} 条来源")]}

def write_node(state: ResearchState):
    # 只读 State 里需要的字段，写时只更新自己负责的字段
    draft = compose(state["sources"], state["topic"])
    return {"draft": draft}   # 只更新 draft，不动其他

graph = StateGraph(ResearchState)
graph.add_node("research", research_node)
graph.add_node("write", write_node)
graph.add_edge(START, "research")
graph.add_edge("research", "write")
graph.add_edge("write", END)

LangGraph 的人机协作与断点恢复

LangGraph 相比纯 LangChain 的两大杀手锏是人机协作（human-in-the-loop）和断点恢复（checkpointing）。这两个能力让 agent 能处理需要人工审批的场景，且长时间运行的复杂任务可以中断后继续。用好它们是 LangGraph 生产化的关键。

人机协作的核心是"interrupt"机制。在图的某个节点前设置 interrupt，执行到这里会暂停，把当前 State 持久化，等待人工输入（批准/修改/拒绝）后继续。典型用法是高风险动作前的人工确认——如发邮件、执行交易、部署代码，agent 准备好动作后暂停，人审核 State 里的计划，批准后才真正执行。这把"自主性"和"安全性"平衡得很好：日常自动跑，关键节点人把关。

断点恢复解决"长任务中途挂了怎么办"。LangGraph 的 checkpointer 把每步执行后的 State 存到外部（内存/SQLite/Postgres），任务任何时候中断（崩溃、人为暂停、超时），都能从最后一个 checkpoint 恢复继续，不用从头跑。这对几分钟到几小时的长任务至关重要。进阶能力是"time travel"——不仅能恢复到最后状态，还能回滚到任意历史状态重新执行（如发现第 3 步决策错了，回到第 3 步重跑）。这是调试和优化的利器。

能力机制典型用途工程要点
人机协作interrupt 暂停高风险动作审批State 必须可序列化
断点恢复checkpointer 持久化长任务容错选对存储后端
time travel历史 State 重放调试/优化checkpoint 保留够多
并发分支fan-out/fan-in并行加速分支 State 隔离
动态路由conditional edges条件分支路由函数要纯
# LangGraph 人机协作 + 断点恢复
from langgraph.checkpoint.sqlite import SqliteSaver
from langgraph.graph import StateGraph, START, END
from langgraph.types import interrupt, Command

class DeployState(TypedDict):
    plan: str
    approved: bool
    result: str

def plan_node(state):
    plan = generate_deploy_plan()
    return {"plan": plan}

def review_node(state):
    # interrupt：暂停等待人工审批
    # 把 plan 展示给人，人决定 approve/reject/modify
    decision = interrupt({"prompt": "审核部署计划", "plan": state["plan"]})
    if decision.get("action") == "approve":
        return {"approved": True, "plan": decision.get("modified_plan", state["plan"])}
    return {"approved": False}

def deploy_node(state):
    if not state["approved"]:
        return {"result": "已取消"}
    return {"result": execute_deploy(state["plan"])}

graph = StateGraph(DeployState)
graph.add_node("plan", plan_node)
graph.add_node("review", review_node)
graph.add_node("deploy", deploy_node)
graph.add_edge(START, "plan")
graph.add_edge("plan", "review")
graph.add_conditional_edges("review", lambda s: "deploy" if s["approved"] else END)
graph.add_edge("deploy", END)

# 关键：加 checkpointer，支持断点恢复
checkpointer = SqliteSaver.from_conn_string("checkpoints.db")
app = graph.compile(
    checkpointer=checkpointer,
    interrupt_before=["review"],   # review 前暂停
)

# 首次运行：到 review 前暂停，返回 thread_id
config = {"configurable": {"thread_id": "deploy-001"}}
result = app.invoke({"plan":""}, config)
# → 暂停在 review，State 已存入 checkpoints.db

# 人工审核后恢复（可能几小时后）
app.invoke(Command(resume={"action":"approve"}), config)
# → 从 checkpoint 恢复，继续执行 deploy

# 即便中途崩溃，下次用同 thread_id 启动也能从最后 checkpoint 恢复

本章小结

LangGraph 把 agent 从"固定循环"升级为"状态图"：State 管数据、Node 管处理、Edge 管流转。条件边让它能表达任意复杂的控制流。它是 LangChain 家族中做生产级 agent 的推荐方案——AgentExecutor 能做的它都能做，还能做 AgentExecutor 做不到的。
