LLM 常见 FAQ 与疑难解答
50 个最常被问到的问题

学了 89 章，你可能还有很多具体疑问。本章收集 50 个最常被问到的问题——从「为什么模型会幻觉」到「7B 模型能跑在什么显卡上」——每个给出清晰、实用的解答。

阅读约 40 分钟难度 ★★☆FAQ 必备

01 基础概念 FAQ（15 问）

Q1: 大模型为什么叫「大」？多大算大？

「大」指参数量大。一般认为 1B（10 亿）以上算大模型，100B+ 算超大。GPT-3 是 175B，GLM-5 系列是千亿级 MoE。参数量决定模型的「容量」——能存储多少知识、学会多复杂的模式。

Q2: Token 到底是什么？一个汉字等于几个 token？

Token 是模型处理文本的最小单位。中文一个字通常 ≈ 1-2 个 token（取决于词表），英文一个单词 ≈ 1-3 个 token。计费、上下文长度、推理成本都按 token 算。用 GLM 的 Tokenizer 可以精确计算。

Q3: 为什么大模型会「幻觉」（编造内容）？

模型的本质是「预测最可能的下一个 token」，而不是「检索事实」。当它不确定时，会生成看起来合理但不真实的内容——因为训练目标是最小化预测误差，不是保证事实正确。缓解：RAG 引用+系统提示「不确定就说」+事实核查。

Q4: 温度（Temperature）到底调多少合适？

取决于任务：事实问答/代码生成/JSON 输出 → 0.1-0.3（确定性高）；对话/写作 → 0.5-0.8（有创意但不离谱）；头脑风暴 → 0.9-1.2（最大创意）。没有万能值，用评估集测。

Q5: 上下文窗口 1M token 意味着什么？真能用满吗？

1M token ≈ 750K 英文单词 ≈ 500K 汉字 ≈ 一本长篇小说。理论上能塞这么多，但实际越长越贵越慢越易出错（Lost in the Middle）。GLM-5.2 强调「真正可用 1M」——意味着在全长度都有稳定高召回率，这是 DSA 等技术打磨的结果。

Q6: 模型能力是「涌现」的还是渐进的？

两者都有。大部分能力（语法、知识）是渐进提升的。但某些能力（如多位数运算、指令遵循）会在某个 loss 阈值后突然出现——这就是「涌现」。这也是为什么训练要坚持下去的原因。

Q7: 开源模型和闭源模型差距多大？

2026 年差距大幅缩小。GLM-5.2、DeepSeek-R1 等开源模型在多数任务上已接近甚至超越闭源。但闭源在前沿能力（超长推理、复杂工具调用）上仍有优势。开源的优势是可控、私有化、免费。

Q8: 什么是 MoE？为什么它更省？

MoE 把 FFN 层替换成多个「专家」网络。每个 token 只激活其中 2 个（如 8 选 2）。总参数很大（容量大），但单次计算只算一小部分（推理快）。这就是「大容量低计算」的秘密。

Q9: Embedding 向量的维度为什么是 768/1024/4096？

维度是设计选择。维度高 → 表达能力强但计算/存储成本高。768 是 BERT 时代的标准，1024 是当代 Embedding 的主流，4096+ 是大模型隐状态的维度。太高没必要（信息量有限），太低丢失语义。

Q10: 为什么用 softmax 而不是直接取最大？

直接取最大（argmax）= 贪心解码 = 每次输出一样 = 单调重复。Softmax 给出概率分布，可以从中采样，引入多样性和创造性。温度控制这个分布的「陡峭」程度。

Q11: 自回归生成为什么是一个 token 一个 token？

因为训练目标就是「给定前文，预测下一个 token」。推理时，模型生成一个 token 后，把它加入上下文，再预测下一个——循环。这是因果性的要求：不能看未来。

Q12: KV Cache 为什么那么占显存？

每个 token 在每层每头都要存 K 和 V 向量。一个 80 层、64 头、head_dim=128 的模型，128K 上下文，KV Cache 就要数百 GB。这就是长上下文的工程瓶颈。

Q13: 为什么训练用 bf16 而不是 fp32？

bf16 用 16 bit 存储但保留 fp32 的数值范围（指数位相同），精度略低但不会溢出。省一半显存+一半带宽，训练快很多。fp32 精度虽高但没必要。

Q14: Scaling Laws 说模型越大越好，那为什么不做 10 万亿参数？

理论上是，但受限于：① 算力成本（10 万亿参数训练费可能是百亿美元级）；② 数据量（没有足够高质量数据喂它）；③ 工程复杂度（分布式训练的通信/稳定性）。MoE 是折中方案。

Q15: 模型的「知识」存在哪里？

主要存在 MLP 层的权重里（第 1 章说过，MLP 占 2/3 参数）。Attention 负责「检索」相关知识，MLP「存储」知识。这就是为什么 RAG（外挂知识库）有效——它把知识检索外部化了。

02 工程实践 FAQ（15 问）

Q16: RAG 和微调怎么选？

加知识/可溯源/实时更新 → RAG。改行为/风格/降成本（小代大）→ 微调。两者常组合。详见第 26/41 章。

Q17: 我的 RAG 检索效果差，怎么排查？

按链路排查：① 分块对不对（语义是否完整）；② Embedding 好不好（换更好的模型）；③ 检索策略（加 BM25 混合）；④ Rerank（加 Cross-Encoder 精排）；⑤ 查询改写（HyDE）。详见第 79 章。

Q18: 7B 模型需要什么显卡？

推理：INT4 量化 → 6GB 显存（GTX 1060 级别）；FP16 → 14GB（RTX 4090）。微调（QLoRA）：~10GB。详见第 59 章。

Q19: Agent 经常死循环怎么办？

① 重复检测（连续 3 步相同动作→跳出）；② 步数上限（硬限制）；③ 系统提示「换思路」；④ 预算上限。详见第 4/79 章。

Q20: 怎么降低 API 调用成本？

① Prompt Caching（固定前缀缓存，省 50-90%）；② 模型分级（简单走小模型）；③ 精简 prompt；④ 结果缓存（Redis）；⑤ 上下文裁剪。详见第 67/83 章。

Q21: vLLM 和 Ollama 有什么区别？用哪个？

vLLM：服务端高并发，PagedAttention+连续批处理，吞吐之王。Ollama：本地体验，基于 llama.cpp，一行命令跑模型。服务器用 vLLM，个人本地用 Ollama。详见第 34 章。

Q22: 模型输出格式不稳定怎么办？

① 低温度（0.1-0.3）；② JSON schema 强制（structured output）；③ few-shot 示范；④ 后处理+重试。详见第 19 章。

Q23: 怎么评估我的 AI 系统好不好？

建评估集（几十条标注样本），每次改动跑回归。指标看：成功率/准确率/满意度/成本。RAG 用 RAGAS，Agent 看完成率。详见第 24 章。

Q24: 私有化部署需要什么条件？

① 硬件（GPU/显存，见 Q18）；② 推理引擎（vLLM/llama.cpp）；③ 模型权重（GLM 开源）；④ 监控运维。详见第 16/59 章。

Q25: LoRA 微调需要多少数据？多少时间？

500-5000 条高质量数据即可见效。单卡（如 RTX 4090）微调 7B 模型，3 epochs 约 1-4 小时。数据质量远比数量重要。详见第 26/46 章。

Q26: Agent 的工具调用不稳定（参数格式错）怎么办？

① 用支持 function calling 的模型（GLM-5.2 原生支持）；② JSON schema 校验+自动修复重试；③ 工具描述写清楚（参数类型/必填/示例）。详见第 4 章。

Q27: 多 Agent 什么时候用？什么时候别用？

用：需要多种专长/需要制衡/需要并行。别用：能用单 Agent 解决的。多 Agent 协调成本高，能用单兵别上军团。详见第 5 章。

Q28: MCP 和 function calling 有什么区别？

Function calling 是模型能力（输出工具调用意图），MCP 是协议标准（定义工具怎么暴露）。MCP 让工具跨框架通用（写一次到处用）。详见第 8 章。

Q29: 怎么处理多语言/中英混合场景？

① Embedding 用 BGE-m3（多语言）；② 系统提示指定语言；③ few-shot 示范；④ 查询改写统一语言。详见第 47 章。

Q30: 模型升级后效果变差了怎么办？

① 跑评估集回归，确认退化范围；② 版本锁定（暂时不升级）；③ 灰度切换（先小流量）；④ 调整 prompt 适配新版。详见第 58 章。

03 进阶与前沿 FAQ（10 问）

Q31: 推理模型（o1/R1）和普通模型怎么选？

推理模型适合：复杂数学/困难代码/深度推理。普通模型适合：简单问答/对话/创意/快响应。推理模型慢且贵，简单问题别用。详见第 17 章。

Q32: 合成数据会害了模型吗（Model Collapse）？

不加控制地用 AI 生成数据训 AI 会退化。但加了质量过滤+混入真实数据+多样性控制，合成数据是安全的且极其有效的。GLM-5 的大规模训练必然包含合成数据。详见第 18 章。

Q33: Transformer 会被 Mamba/SSM 取代吗？

短期不会。Transformer 生态太成熟（FlashAttention/推理引擎/Scaling 验证）。但 Mamba 在超长序列/低延迟推理有优势，混合架构（Mamba+Attention）是务实方向。详见第 32 章。

Q34: 长上下文（1M）会取代 RAG 吗？

不会完全取代。长上下文适合几十页文档，但几万页知识库还得 RAG。且长上下文贵、有 Lost in the Middle。最佳实践是长上下文+RAG 组合。详见第 6 章。

Q35: 什么是「涌现能力」？为什么重要？

某些能力在模型规模/数据达到某个阈值后突然出现（如多位数运算、指令遵循）。这意味着 loss 下降不等于线性提升——坚持训练，临界点可能在几步之后。详见第 2 章。

Q36: RLVR 和 RLHF 有什么区别？

RLHF 用人类偏好（主观、贵）；RLVR 用可验证的正确性（客观、免费，但只适合数学/代码等有标准答案的任务）。RLVR 是推理模型的核心。详见第 17 章。

Q37: 什么是 Agentic Coding？和普通编程辅助有什么区别？

Agentic Coding = AI 自主读代码/改代码/跑测试/修 bug（多步自主），不是一问一答。Claude Code/GLM-5.2 的编程能力就是 Agentic。详见第 28 章。

Q38: Constitutional AI 是什么？为什么重要？

给模型一部「宪法」（原则清单），让它自我批评、自我修正，用修正后的数据训练。减少人工标注，让对齐可规模化。Claude 系列用的就是这套。详见第 15 章。

Q39: 什么是 Any-to-Any 模型？

一个模型，任意输入（文/图/音/视）→ 任意输出。把所有模态统一成 token，一个 Transformer 处理一切。这是通往 AGI 的路线。详见第 37 章。

Q40: AI 离 AGI 还有多远？

争议很大。乐观派认为 3-10 年（推理+Agent+多模态+自学习融合）。谨慎派认为还有根本性突破需要。共识：我们正走在正确的方向上，速度比预期快。详见第 56/78 章。

FAQ 里最常被问的两类问题是「这个报错什么意思」和「我环境配对了吗」。光靠文字解释效率低，下面用两个可直接运行的脚本来兜底：第一个诊断常见 API 调用报错并给出修复建议，第二个一键自检本地 AI 开发环境是否就绪。

diagnose_api_error.py · 常见 LLM API 报错的自动诊断
"""捕获 LLM API 调用异常，按错误类型给出修复建议。"""
import openai

# 常见报错 → (关键词, 原因, 修复建议)
ERROR_TABLE = [
    ("AuthenticationError", "API Key 无效或过期",
     "检查环境变量 API_KEY 是否正确，到控制台重新生成。"),
    ("RateLimitError", "触发速率/额度限制",
     "加退避重试、降低 QPS、或升级套餐额度。"),
    ("context_length_exceeded", "输入超过模型上下文窗口",
     "缩短 prompt、用 RAG 只喂相关片段、换更大窗口的模型。"),
    ("ConnectionError", "网络不通/DNS 解析失败",
     "检查代理设置、base_url 是否可达、ping 测试。"),
    ("BadRequestError", "请求参数非法",
     "检查 model 名、messages 格式、temperature 范围。"),
    ("InternalServerError", "服务端错误",
     "重试 2-3 次；持续报错看服务商状态页。"),
]

def diagnose(err: Exception):
    name = type(err).__name__
    msg = str(err).lower()
    for key, reason, fix in ERROR_TABLE:
        if key.lower() in name.lower() or key.lower() in msg:
            print(f"❌ 报错：{name}")
            print(f"   原因：{reason}")
            print(f"   修复：{fix}")
            return
    print(f"❓ 未知报错：{name}: {err}")
    print("   建议：把完整 traceback 贴出来，或查官方文档。")

# 用法：包一层 try/except，出错自动诊断
def safe_chat(client, **kwargs):
    try:
        return client.chat.completions.create(**kwargs)
    except Exception as e:
        diagnose(e); raise

if __name__ == "__main__":
    client = openai.OpenAI(api_key="sk-invalid")   # 故意用错 key 演示
    try:
        safe_chat(client, model="glm-4-plus", messages=[{"role":"user","content":"hi"}])
    except Exception:
        pass
# 输出会指出 AuthenticationError → 检查 API_KEY，而不是让人面对一堆 traceback 发呆
check_env.py · AI 开发环境一键自检
"""检查本地 AI 开发环境：Python 版本、GPU、关键库、API Key 是否就绪。"""
import sys, os, importlib

def status(ok, msg):
    print(f"{'✅' if ok else '❌'} {msg}")
    return ok

all_ok = True

# 1. Python 版本（≥3.9）
v = sys.version_info
all_ok &= status(v >= (3, 9), f"Python {v.major}.{v.minor}.{v.micro}")

# 2. GPU 可用性（torch.cuda / MPS）
try:
    import torch
    if torch.cuda.is_available():
        all_ok &= status(True, f"CUDA GPU: {torch.cuda.get_device_name(0)}")
    elif torch.backends.mps.is_available():
        all_ok &= status(True, "Apple MPS (Metal) 可用")
    else:
        status(False, "无 GPU，仅 CPU（推理会慢）"); all_ok = False
except ImportError:
    status(False, "未安装 torch"); all_ok = False

# 3. 关键库是否齐全
for lib in ["openai", "numpy", "faiss", "transformers"]:
    try:
        importlib.import_module(lib)
        all_ok &= status(True, f"已安装 {lib}")
    except ImportError:
        status(False, f"缺少 {lib} → pip install {lib}"); all_ok = False

# 4. API Key 是否配置
for key in ["ZHIPUAI_API_KEY", "OPENAI_API_KEY"]:
    has = bool(os.getenv(key))
    all_ok &= status(has, f"环境变量 {key} {'已设置' if has else '未设置'}")

print("
" + ("🎉 环境就绪，可以开始第 11 章项目了！" if all_ok
      else "⚠️ 有项目未通过，按提示修复后再继续。"))
# 把这个脚本放进项目根目录，新人 clone 后跑一遍就能发现环境问题

这两个脚本是把 FAQ 变成工具的思路：与其反复回答「我这个错怎么办」「我环境对吗」，不如给一个能自动诊断、自动检查的脚本。新手跑一遍 check_env.py 就知道自己缺什么，遇到报错套一层 safe_chat 就有修复建议——这比文档里的文字说明高效得多。
