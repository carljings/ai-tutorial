LLM 基础与原理
从 Token 到 Transformer 的底层旅行

要看懂 GLM-5.2 是怎么「炼」出来的，得先理解它在炼什么。本章从最底层的 Token 一路讲到 Transformer 架构、Self-Attention、MoE 稀疏专家，把大模型这头「黑箱」拆成你能看得懂的零件。

阅读约 25 分钟 难度 ★★☆ 前置：无

01 一切始于 Token：模型眼里的世界

人类看文字是「字」，模型看文字是「Token」。这是理解一切大模型的第一道门槛。

为什么不直接用「字」或「词」

早期自然语言处理（NLP）用「词」作为单位，中文做分词、英文按空格切。但这套做法在多语言、新词、拼写错误面前非常脆弱：

  • 「ChatGPT」「元宇宙」「yyds」这些新词根本进不了词典；
  • 不同语言切分粒度差异巨大，词表要么爆炸要么稀疏；
  • 同一个词根的不同形态（run / running / ran）被当成完全无关的东西，浪费了语义。

于是有了 子词分词（Subword Tokenization）：把词切成更小的、可复用的「零件」。当代大模型几乎都用它的某个变体。

🔑 一句话记住

Token 是模型处理文本的最小单位。1 个英文单词 ≈ 1.3 个 token，1 个汉字 ≈ 0.6~1 个 token（取决于词表和编码）。计费、上下文长度、推理成本，全部按 token 算。

BPE：最主流的分词算法

BPE（Byte Pair Encoding，字节对编码）是 GPT 系列、GLM、LLaMA 等绝大多数模型的选择。它的训练思路异常朴素：

初始化

把所有训练语料拆成最细的字符（或字节）序列，每个不同的字符都是一个初始 token。

统计相邻频率

数一数语料里所有相邻字符对出现的次数，找出最高频的那一对。

合并

把最高频的字符对合并成一个新的 token，加入词表。

重复

不断重复「统计-合并」，直到词表大小达到预设目标（比如 5 万、10 万、15 万）。

这样训练出来的词表有个绝妙性质：常见词整体进词表（如 "the"、"learning"），罕见词被拆成可复用的子词（如 "tokenization" → "token" + "ization"），完全没见过的词也能用字符兜底。这就解决了词典爆炸和未登录词（OOV）的难题。

用 Python 直观感受 BPE 的合并过程
# 一个极简的 BPE 训练演示（仅教学用）
from collections import Counter

def get_pairs(word):
    # word 是一个 tuple，如 ('l','o','w','e','r')
    return Counter(zip(word[:-1], word[1:]))

corpus = ['low', 'lower', 'newest', 'widest']
vocab  = {w: tuple(w + '$') for w in corpus}  # 加结束符

for _ in range(10):                      # 合并 10 轮
    pairs = Counter()
    for w in vocab.values():
        pairs.update(zip(w[:-1], w[1:]))
    best = pairs.most_common(1)[0][0]        # 找到最高频相邻对
    if best[1] < 2: break
    new_token = ''.join(best)            # 例如 ('e','s') -> 'es'
    # 把所有词里的这对相邻符合并
    vocab = {w: tuple(' '.join(' '.join(p) for p in ww).replace(
        best[0]+' '+best[1], new_token).split()) for w, ww in vocab.items()}
    print(f"合并: {best} -> {new_token}")

BPE 的兄弟们：WordPiece / Unigram / SentencePiece

算法核心思想代表用户
BPE贪心合并最高频字符对GPT 系列、GLM、LLaMA
WordPiece类似 BPE，但用「似然增益」选合并对象BERT、DistilBERT
Unigram反向：先有大词表，再按概率删到目标大小T5、ALBERT
SentencePiece把语言无关的字节级 BPE/Unigram 封装成工具多语言模型（含中文友好）
💡 为什么中文模型也得用子词

中文没有空格，但按「字」分会让词表僵化。现代中文友好的模型（如 GLM）多采用 字节级 BPE（Byte-level BPE）：把文本先编码成 UTF-8 字节，再在字节层面做 BPE。好处是任何语言的任何字符都不会 OOV——因为它最终都能还原到 256 个字节。GLM、GPT-2/4 都是这条路。

词表大小是个工程权衡

词表越大，每个 token 承载的信息越多（同样的话用更少 token 表达），推理更快、更省钱；但词表越大，Embedding 层参数越多，模型也更难训练。当前主流：

模型词表大小分词器
GPT-4 / GPT-4o~100ktiktoken (BPE)
GLM-4 / GLM-5 系列~150k+字节级 BPE

Claude未公开自研 BPE
「Token 决定了模型的世界边界。模型永远无法直接理解 Token 以外的东西——这也是为什么后来需要 RAG、Tool Use、Skills 等机制来给它「装上手脚」。」

02 Embedding：让 Token 有了「意思」

分完词，每个 token 还只是一个 ID（整数）。模型要让它们有「意义」，靠的是 Embedding（词嵌入）：把每个 token ID 映射成一个高维向量（比如 4096 维或 12288 维的浮点数组）。

向量空间里的「语义距离」

训练好的 Embedding 有个神奇的性质：语义相近的词，向量也相近。

国王 [0.23, -0.41, ...] ─ 男人 ─ +女人 ─▶ 女王 [0.19, -0.38, ...]
向量运算 ≈ 语义运算，这是 Embedding 最迷人的地方

经典例子：国王 - 男人 + 女人 ≈ 女王。这种「语义可加性」让模型能做类比推理。这也是后面 RAG 里向量检索能成立的数学基础——我们会在第 6 章细讲。

位置编码：告诉模型「顺序」

Self-Attention 本身是「无序」的——它把序列当成一个集合。但语言显然有顺序：「狗咬人」和「人咬狗」意思天差地别。所以必须额外告诉模型每个 token 在哪个位置，这就是 位置编码（Positional Encoding）。

📐

绝对位置编码

给每个位置一个固定的向量（正弦/余弦或可学习）。简单直接，但泛化到训练时没见过的长度会变差。

🌀

旋转位置编码 RoPE

当前主流（LLaMA、GLM、Qwen 都用）。通过旋转矩阵把「相对位置」编码进 Attention，天然支持外推到超长上下文，是 GLM-5.2 实现 1M token 的重要基石。

03 Transformer：当代大模型的总骨架

2017 年 Google 的论文《Attention Is All You Need》提出了 Transformer。此后几乎所有主流大模型（GPT、GLM、Claude、LLaMA、Gemini）都建立在它的变体之上。理解 Transformer，就理解了大模型的「脊柱」。

整体结构：Encoder 与 Decoder

原始 Transformer 是 Encoder-Decoder 结构，用于翻译。但当代生成式大模型（LLM）几乎清一色是 Decoder-Only（只有解码器，也叫因果语言模型）。GLM 系列则走出了一条独特的「自回归空白填充」路线，后演进为成熟的 decoder-only。

Self-Attention：让 token 们「互相看见」

Self-Attention 是 Transformer 的灵魂。它的核心思想是：序列中的每个 token，都去「查询」其他所有 token，决定该把多少注意力分给谁。

它通过三个矩阵把每个 token 投影成三个角色：

角色符号比喻
Query（查询）Q「我在找什么？」——我作为搜索者发出的请求
Key（键）K「我有什么？」——我作为被检索者的标签
Value（值）V「我的内容是什么」——如果被选中，我贡献的信息

计算过程（缩放点积注意力）：

Attention(Q, K, V) = softmax( Q·Kᵀ / √dₖ ) · V
  1. 打分：每个 token 的 Query 和所有 token 的 Key 做点积，得到「相关度分数」；
  2. 缩放：除以 √dₖ（Key 的维度开根号），防止分数过大导致 softmax 梯度消失；
  3. 归一化：softmax 把分数变成概率（加起来等于 1）；
  4. 加权：用这些概率对 Value 加权求和，得到该 token 的新表示。
🔑 直觉理解

Self-Attention 让每个词都根据上下文重新定义自己。「苹果」在「我吃了一个苹果」里会因为关注到「吃」而指向水果；在「苹果发布了新手机」里会因为关注到「发布」「手机」而指向公司。这就是大模型能理解上下文的根源。

多头注意力（Multi-Head Attention）

一组 Q/K/V 只能学到一种「关注模式」。把它们拆成多组（比如 32 个「头」），每组独立做注意力，再拼接起来——这就是多头注意力。不同头能学到不同关系：有的关注语法、有的关注指代、有的关注情感。这也是 GLM-5.2 这类模型能捕捉丰富语义的关键。

MLP（前馈网络）：每个 token 的「独立思考」

Attention 负责 token 间的信息交换，MLP（Feed-Forward Network）负责每个 token 内部的信息加工。它的结构很简单：两层线性变换 + 一个非线性激活（如 SwiGLU）。可以把它理解为：每个 token 在吸收了上下文后，自己「消化」一下，把信息压缩/重组进新的表示。

⚠️ 一个常见误解

很多人以为「参数都在 Attention 里」。其实恰恰相反——在大多数 Transformer 模型中，MLP 占了约 2/3 的参数量。Attention 是「路由」，MLP 才是「知识存储」。这也是为什么 MoE 主要改造的就是 MLP 层。

04 MoE：用「稀疏」撬动超大模型

当模型越做越大（几百亿、几千亿参数），一个残酷的现实是：每个 token 都要过完所有参数，算力和显存吃不消。MoE（Mixture-of-Experts，混合专家）给出的答案是：模型可以很大，但每个 token 只激活其中一小部分。

核心思想：路由 + 专家

MoE 把普通的 MLP 层替换成「多个并行的 MLP（专家）+ 一个路由器（Gate）」。每个 token 进来，路由器决定把它送给哪几个专家处理。

MoE 带来的三大收益

💾

更大容量

总参数可以做到很大（如 6710 亿），存储更多知识，但单次推理只动用一小部分。

⚡

更快推理

每个 token 的计算量（FLOPs）只相当于一个更小的稠密模型，速度更快。

🎯

专业分工

不同专家会自发学会处理不同领域（代码、数学、中文、常识），实现隐式专业化。

MoE 也不是免费的午餐

⚠️ MoE 的工程难点
  • 显存占用：虽然计算少，但所有专家都得驻显存，对显存要求极高；
  • 通信开销：专家常分布在不同 GPU 上，token 要在不同卡间「搬运」（All-to-All 通信）；
  • 负载均衡：路由器可能偷懒只用少数专家，需要加辅助损失（load balancing loss）强制均衡；
  • 训练不稳定：稀疏激活让梯度更复杂，调参更难。

代表性 MoE 模型：GLM 系列、DeepSeek-V3（6710 亿总参/370 亿激活）、Mixtral 8×7B、Qwen-MoE。2026 年，MoE 已经成为「千亿级开源模型」的事实标准架构。

05 KV Cache：让生成又快又省的魔法

大模型生成文本是一个 token 一个 token 吐的。每生成一个新 token，都要重新计算前面所有 token 的 Attention。如果每次都重算，长文本生成会慢到不可用。

观察发现：Attention 里 Key 和 Value 只取决于已有的 token，新生成的 token 不改变它们的值。所以可以把已算好的 K、V 缓存下来，下一轮直接复用——这就是 KV Cache。

⚠️ KV Cache 是显存大户

KV Cache 的大小随上下文长度线性增长。一个 70B 模型在 128K 上下文下，单条请求的 KV Cache 就可能高达几十 GB。这也是为什么长上下文（如 GLM-5.2 的 1M token）在工程上极其困难——后续章节会讲 DSA、Paged Attention 等优化。

07 深度推导：Self-Attention 的完整数学

把第 3.2 节的公式拆到最细。设输入序列长度 T、隐状态维度 d。对于头数为 h 的多头注意力，每个头维度 dh = d/h。

单头注意力的逐步计算

  1. 线性投影：输入 X (T×d) 通过三个权重矩阵投影成 Q、K、V：
    Q = X·WQ，K = X·WK，V = X·WV，每个都是 T×dh
  2. 打分矩阵：S = Q·KT，得到 T×T 的分数矩阵，Sij 是位置 i 对位置 j 的关注分数
  3. 缩放：S = S / √dh（防止 softmax 饱和）
  4. 因果掩码（Decoder 才有）：把 Sij（i < j，即未来位置）置为 −∞
  5. 归一化：A = softmax(S, dim=-1)，每行加起来等于 1
  6. 加权求和：O = A·V，输出 T×dh，每个位置是所有 V 的加权和

复杂度分析：计算量 O(T²·d)（Q·KT 和 A·V 各占一半），显存 O(T²)（分数矩阵）。这就是为什么长序列 Attention 又慢又费显存——序列翻倍，计算量变 4 倍。

多头的并行与合并

h 个头各自独立算 attention，得到 h 个 T×dh 的输出，沿最后一维拼接成 T×d，再过一个输出投影 WO。数学上等价于在 d 维空间里学习 h 个不同的「关注模式」。

08 深度推导：旋转位置编码 RoPE

第 2.2 节提到 RoPE 是 GLM 等模型实现长上下文的关键。它的数学很优雅：通过对 Q、K 施加旋转矩阵，把相对位置信息编码进 Attention 的点积。

核心性质：希望 <qm, kn>（位置 m 的 query 与位置 n 的 key 的点积）只依赖于相对位置 m−n。RoPE 的解是：把 qm 旋转一个角度 mθ，kn 旋转 nθ，则：

<Rmq, Rnk> = <q, Rm−nk>    # 只依赖 m−n
Rm = 旋转角度 mθ 的 2×2 旋转矩阵块对角堆叠

其中 θ 是预设的频率（不同维度用不同 θ，覆盖不同尺度）。RoPE 的妙处：不引入额外参数、不改变 Attention 计算、天然支持任意长度、相对位置天然编码。配合「NTK-aware 缩放」等技巧，可以从训练时的 4K 外推到 1M——这正是 GLM-5.2 长上下文的数学基础。

09 代码实战：从零手写 Self-Attention

理解 Attention 最好的方式就是手写一遍。下面用 NumPy 实现，每一步对应上面的数学：

attention_from_scratch.py —— 用 NumPy 手写 Self-Attention
"""
attention_from_scratch.py —— 从零实现 Self-Attention
每一步都对应数学公式，无任何抽象
"""
import numpy as np

def softmax(x, axis=-1):
    e = np.exp(x - x.max(axis=axis, keepdims=True))   # 减最大值防溢出
    return e / e.sum(axis=axis, keepdims=True)

def self_attention(X, d_k, mask=None):
    """
    X: (T, d)  输入序列
    d_k: 每个头的维度
    返回: (T, d_k)  attention 输出
    """
    T, d = X.shape

    # ① 随机初始化权重矩阵（真实模型是学出来的）
    W_q = np.random.randn(d, d_k) * 0.1
    W_k = np.random.randn(d, d_k) * 0.1
    W_v = np.random.randn(d, d_k) * 0.1

    # ② 投影成 Q, K, V
    Q = X @ W_q          # (T, d_k)
    K = X @ W_k          # (T, d_k)
    V = X @ W_v          # (T, d_k)

    # ③ 打分 + 缩放
    scores = Q @ K.T / np.sqrt(d_k)    # (T, T)

    # ④ 因果掩码（Decoder 用：不让看未来）
    if mask == "causal":
        tril = np.tril(np.ones((T, T)))               # 下三角
        scores = np.where(tril == 1, scores, -np.inf)

    # ⑤ softmax 归一化
    attn = softmax(scores, axis=-1)      # (T, T) 每行和为 1

    # ⑥ 加权求和
    output = attn @ V                     # (T, d_k)

    return output, attn

# ---------- 试跑 ----------
np.random.seed(0)
X = np.random.randn(5, 8)    # 5 个 token，每个 8 维
out, attn = self_attention(X, d_k=4, mask="causal")
print("输出 shape:", out.shape)       # (5, 4)
print("注意力矩阵（每行和为 1）:")
print(np.round(attn, 2))
# 上三角是 0（被掩码），对角线和下方有值——这就是「每个 token 只看自己和之前」

这 40 行就是 Attention 的全部真相。跑一遍，打印注意力矩阵，你会亲眼看到「因果掩码」（上三角为 0）和「注意力权重」（每行加起来为 1）。GLM-5.2 里的 Attention，本质就是这个，只是维度大了几个数量级、加了 RoPE、优化了计算（FlashAttention）。

10 深度：KV Cache 的显存计算

第 5 节说 KV Cache 是显存大户，到底多大？精确计算：

KV Cache 显存 = 2 × n_layers × n_heads × head_dim × seq_len × batch × dtype_bytes
2 = K 和 V 各一份；dtype: fp16=2字节, int8=1字节, int4=0.5字节

举例：一个 80 层、64 头、head_dim=128 的模型，序列 128K，batch=1，fp16：

2 × 80 × 64 × 128 × 131072 × 1 × 2 = 274 GB 😱

单条请求的 KV Cache 就要 274GB！这就是为什么长上下文工程上极其困难。GLM-5.2 的 1M 上下文，靠 DSA（稀疏化 KV）、PagedAttention（分页管理）、量化（KV int4）等技术把显存压下来。长上下文不仅是模型能力问题，更是工程优化问题。

★ BPE 分词示例

BPE（Byte Pair Encoding）是大模型最常用的分词算法。理解它的最好方式是手动实现一遍训练过程：从字符开始，反复合并最高频的相邻对。下面是一个完整的最小实现：

import re
from collections import Counter, defaultdict

def get_pairs(word: tuple) -> set:
    """返回一个 word(token 序列)里所有相邻 bigram。"""
    return {(word[i], word[i+1]) for i in range(len(word)-1)}

def train_bpe(corpus: list[str], num_merges: int) -> list[tuple]:

    for text in corpus:
        for w in re.findall(r"\S+", text):
            vocab[tuple(list(w) + [""])] += 1
    word2count = dict(vocab)

                pair_freq[p] += freq
        if not pair_freq:
