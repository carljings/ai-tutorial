Linux Landlock 沙箱
syscall 程序化的文件系统访问控制

Linux 没有 sandbox-exec，但有 Landlock——一个从 5.13 内核开始合入的安全模块。和 Seatbelt 的「profile 文件描述规则」不同，Landlock 是 程序化 的：你的程序在运行时通过 syscall 创建规则、绑定到自己，限制后续所有文件操作的可见性。Codex 在 Linux 上用 Landlock 实现了和 macOS Seatbelt 对等的隔离。本章我们看 Landlock 的 API 设计，并和 Seatbelt 做正面对比。

阅读约 18 分钟 难度 ★★★★

★ 一、Landlock 的设计哲学：程序化而非声明式

Landlock 的设计哲学和 Seatbelt 截然不同。Seatbelt 是声明式的——你写一份 profile 文件，用一种专门的 DSL 描述规则，然后由 sandbox-exec 解析执行。Landlock 是程序化的——你直接在代码里调用 syscall，用 C/Rust 的 API 一条条加规则。

这种差异有几个深层原因。第一，Linux 内核社区更偏爱「程序化、可组合」的接口（看 io_uring、eBPF 的设计），而非引入新的 DSL。第二，Landlock 的规则是 绑定到进程的——你的程序自己创建 ruleset，然后把自己关进去，不需要外部 daemon 或 wrapper 命令。第三，Landlock 是非特权用户也能用的——不需要 root，任何进程都能给自己加限制，这降低了使用门槛。

一个关键特性：Landlock 规则一旦应用（enforce），就不可撤销。进程无法自己解除限制，子进程继承父进程的限制且只能加更多。这种「单向收紧」保证了沙箱的不可逃逸性——即使进程被攻陷，也无法提权解除限制。

★ 二、Landlock 的核心 syscall 流程

使用 Landlock 大致分四步：创建 ruleset → 添加规则 → 应用 ruleset → 此后受约束。我们用 C 风格的伪代码展示这个流程（Codex 在 Rust 里通过 libc crate 调用这些 syscall）：

// landlock 的使用流程（C 伪代码）
#include <linux/landlock.h>
#include <sys/syscall.h>
#include <fcntl.h>

int main() {
    // 【1】创建一个 ruleset，声明要管哪些类型的访问
    struct landlock_ruleset_attr attr = {
        .handled_access_fs = LANDLOCK_ACCESS_FS_WRITE_FILE |
                             LANDLOCK_ACCESS_FS_REMOVE_FILE |
                             LANDLOCK_ACCESS_FS_REMOVE_DIR |
                             LANDLOCK_ACCESS_FS_MAKE_DIR,
        // 注意：没列的类型（如 read）不受约束
    };
    int ruleset_fd = syscall(SYS_landlock_create_ruleset,
                             &attr, sizeof(attr), 0);

    // 【2】添加规则：允许写项目目录
    int proj_fd = open("/home/me/project", O_PATH);
    struct landlock_path_beneath_attr path_attr = {
        .allowed_access = LANDLOCK_ACCESS_FS_WRITE_FILE |
                          LANDLOCK_ACCESS_FS_MAKE_DIR,
        .parent_fd = proj_fd,
    };
    syscall(SYS_landlock_add_rule, ruleset_fd,
            LANDLOCK_RULE_PATH_BENEATH, &path_attr, 0);
    close(proj_fd);

    // 【3】应用 ruleset 到当前进程
    prctl(PR_SET_NO_NEW_PRIVS, 1, 0, 0, 0);  // 必须：禁止获取新特权
    syscall(SYS_landlock_restrict_self, ruleset_fd, 0);
    close(ruleset_fd);

    // 【4】此后，所有文件写操作都受约束
    // 只能在 /home/me/project 下写文件，其他地方写会 EPERM
    int fd = open("/etc/passwd", O_WRONLY);  // 失败，EPERM
    int fd2 = open("/home/me/project/test.txt", O_WRONLY | O_CREAT, 0644);  // 成功
}

这段代码揭示了 Landlock 的几个关键点。第一，handled_access_fs 决定管什么：你在创建 ruleset 时声明要管「文件写、删除、建目录」等类型，没声明的类型（如读、执行）完全不受影响。第二，规则是「允许某路径下某操作」：用 parent_fd 指定一个目录，allowed_access 指定允许的操作。第三，prctl(PR_SET_NO_NEW_PRIVS) 是前置条件：它禁止进程通过 setuid 等机制提权，这是 Landlock 安全性的基石——否则一个 setuid root 的子进程可能绕过限制。第四，restrict_self 之后不可撤销，子进程继承。

★ 三、Codex 在 Rust 里如何调用 Landlock

Codex 用 Rust 的 libc crate 直接调 syscall。为了类型安全，它封装了一个 landlock 模块。我们看核心逻辑：

// codex-rs/src/sandbox/landlock.rs（概念示意）
use std::os::unix::io::RawFd;
use std::path::Path;

// libc 常量（Linux 内核头定义）
const LANDLOCK_CREATE_RULESET_VERSION: u32 = 1 << 0;
const LANDLOCK_ACCESS_FS_WRITE_FILE: u64 = 1 << 5;
const LANDLOCK_ACCESS_FS_REMOVE_FILE: u64 = 1 << 6;
const LANDLOCK_ACCESS_FS_REMOVE_DIR: u64 = 1 << 7;
const LANDLOCK_ACCESS_FS_MAKE_DIR: u64 = 1 << 8;

#[repr(C)]
struct LandlockRulesetAttr {
    handled_access_fs: u64,
}

#[repr(C)]
struct LandlockPathBeneathAttr {
    allowed_access: u64,
    parent_fd: RawFd,
}

pub struct LandlockSandbox {
    ruleset_fd: RawFd,
}

impl LandlockSandbox {
    pub fn new() -> std::io::Result<Self> {
        // 检查内核版本是否支持 Landlock
        let attr = LandlockRulesetAttr {
            handled_access_fs: LANDLOCK_ACCESS_FS_WRITE_FILE
                | LANDLOCK_ACCESS_FS_REMOVE_FILE
                | LANDLOCK_ACCESS_FS_REMOVE_DIR
                | LANDLOCK_ACCESS_FS_MAKE_DIR,
        };
        let fd = unsafe {
            libc::syscall(
                libc::SYS_landlock_create_ruleset,
                &attr as *const _,
                std::mem::size_of::<LandlockRulesetAttr>(),
                0,
            )
        };
        if fd < 0 {
            return Err(std::io::Error::last_os_error());  // 可能是内核太老
        }
        Ok(Self { ruleset_fd: fd as RawFd })
    }

    /// 允许在指定路径下进行指定操作
    pub fn allow_path(&mut self, path: &Path, access: u64) -> std::io::Result<()> {
        let parent_fd = path.file_desc()?;  // 打开目录拿 fd
        let path_attr = LandlockPathBeneathAttr {
            allowed_access: access,
            parent_fd,
        };
        let ret = unsafe {
            libc::syscall(
                libc::SYS_landlock_add_rule,
                self.ruleset_fd,
                1,  // LANDLOCK_RULE_PATH_BENEATH
                &path_attr as *const _,
                0,
            )
        };
        close(parent_fd);
        if ret < 0 { return Err(std::io::Error::last_os_error()); }
        Ok(())
    }

    /// 应用限制到当前线程及所有后续 fork 出的子进程
    pub fn restrict_self(self) -> std::io::Result<()> {
        // 1. 禁止获取新特权（必须）
        let ret = unsafe { libc::prctl(libc::PR_SET_NO_NEW_PRIVS, 1, 0, 0, 0) };
        if ret < 0 { return Err(std::io::Error::last_os_error()); }

        // 2. 应用 ruleset
        let ret = unsafe {
            libc::syscall(libc::SYS_landlock_restrict_self, self.ruleset_fd, 0)
        };
        if ret < 0 { return Err(std::io::Error::last_os_error()); }
        Ok(())
    }
}

这段代码展示了 Codex 在 Linux 上构建沙箱的完整逻辑。注意它封装成了 new → allow_path → restrict_self 三步式 API，使用起来很自然。还有一个重要细节：restrict_self 必须在 fork 子进程之前或之后调用都行，但子进程会继承限制。Codex 的做法是：在主进程里配置好 ruleset，然后 fork 出执行命令的子进程前先 restrict_self，这样子进程天然受限。

★ 四、内核版本检测与降级

Landlock 需要 Linux 5.13+（完整功能要 5.19+ 或 6.2+，不同 access 类型分批合入）。Codex 必须在运行时检测内核支持，并在不支持的系统上降级。这是跨版本 Linux 工具的常见挑战。

// 检测 Landlock 支持并决定策略
pub fn check_landlock_support() -> LandlockSupport {
    // 尝试创建一个空的 ruleset，看是否成功
    let attr = LandlockRulesetAttr { handled_access_fs: 0 };
    let fd = unsafe {
        libc::syscall(
            libc::SYS_landlock_create_ruleset,
            &attr as *const _,
            std::mem::size_of::<LandlockRulesetAttr>(),
            LANDLOCK_CREATE_RULESET_VERSION,  // 只查询版本，不实际创建
        )
    };
    if fd < 0 {
        let err = std::io::Error::last_os_error();
        match err.raw_os_error() {
            libc::ENOSYS => LandlockSupport::None,        // 内核无 Landlock
            libc::EOPNOTSUPP => LandlockSupport::Disabled, // 内核有但未启用
            _ => LandlockSupport::None,
        }
    } else {
        // fd 是支持的版本号
        if fd >= 1 { LandlockSupport::V1 } else { LandlockSupport::None }
    }
}

pub enum LandlockSupport {
    None,       // 完全不支持，Codex 警告或拒绝执行危险命令
    Disabled,   // 内核支持但未启用，提示用户
    V1,         // 支持，正常用
}

这种「探测 → 降级 → 警告」的三段式策略，是处理平台能力差异的通用模式。在不支持 Landlock 的系统上，Codex 不会默默放行，而是明确告诉用户「当前环境无法沙箱隔离，工具执行被限制为只读」或要求用户确认风险后才能执行写操作。

★ 五、与 Seatbelt 的对比

把 Landlock 和 Seatbelt 放在一起，能看出 Linux 和 macOS 在安全机制设计上的文化差异。

接口形态：Seatbelt 是声明式（profile 文件 + DSL），Landlock 是程序化（syscall + 代码）。前者更易写规则、调试时有 trace；后者更灵活、可组合，但规则散落在代码里不易审计。

网络管控：Seatbelt 能限制网络出站的目标地址和端口，Landlock（截至当前版本）只管文件系统，不管网络。这是 Landlock 的重大局限——Codex 在 Linux 上对网络的隔离要靠其他机制（如 unshare network namespace 或 seccomp 过滤 connect syscall）补足。

权限要求：Seatbelt 需要 sandbox-exec 命令（系统自带，但未签名应用可能受限）；Landlock 不需要 root，普通用户可用，门槛更低。

不可逆性：Seatbelt 的限制理论上可以通过结束 sandbox-exec 进程解除（子进程自然死亡）；Landlock 的 restrict_self 不可撤销，但进程退出也就解除了。两者实际效果相当。

生态成熟度：Seatbelt 是 macOS 老牌特性，工具链成熟；Landlock 是 2021 年才合入的新特性，规则细节仍在演进（如 ABI 版本），第三方库支持还在发展。

★ 六、网络隔离的补足：unshare + seccomp

因为 Landlock 不管网络，Codex 在 Linux 上补了一层：用 unshare 创建独立的 network namespace。这会让进程看到一个「空」的网络栈——没有网卡、没有路由，任何网络请求都会失败。如果 Agent 需要访问特定网络（如 npm registry），Codex 在 namespace 内手动配置一个虚拟网卡和代理规则。

// 用 unshare 隔离网络（概念示意）
use libc::{unshare, CLONE_NEWNET};

pub fn isolate_network() -> std::io::Result<()> {
    // 创建新的 network namespace
    let ret = unsafe { unshare(CLONE_NEWNET) };
    if ret < 0 {
        return Err(std::io::Error::last_os_error());
        // 注意：CLONE_NEWNET 需要 CAP_SYS_ADMIN，
        // 普通用户要配 user namespace 才能用
    }
    Ok(())
}

这种「Landlock 管文件 + unshare 管网络」的组合，在 Linux 上近似实现了 Seatbelt 的完整隔离能力。代价是实现复杂度更高——需要处理 namespace 创建、权限、代理配置等细节。这也解释了为什么 Codex 的 Linux 沙箱代码比 macOS 版本更长。

★ 本章小结

Landlock 是 Linux 给 Agent 沙箱的新答案。要点回顾：

  • 程序化而非声明式：Landlock 通过 syscall 创建 ruleset、加规则、restrict_self，规则绑定到进程，不可撤销。
  • 四步流程：create_ruleset → add_rule → restrict_self → 受约束，prctl(PR_SET_NO_NEW_PRIVS) 是安全基石。
  • 白名单语义：声明 handled_access_fs 决定管哪些操作类型，未列的不受影响；规则是「某路径下允许某操作」。
  • 内核版本检测：需要 5.13+，Codex 探测后在不支持时降级为只读或警告，不默默放行。
  • 局限：不管网络：Landlock 只管文件系统，网络隔离要靠 unshare network namespace 或 seccomp 补足。
  • 与 Seatbelt 对比：程序化 vs 声明式、无 root vs 系统 command、不管网络 vs 能限制网络，各有取舍。

两种沙箱（Seatbelt + Landlock）让 Codex 成了一个能在双平台安全执行任意命令的 Agent。下一章（159 章）我们看沙箱之上的那一层——命令执行本身：fork、流式捕获、超时处理。
