错误恢复策略
重试、死循环检测与 fallback chain

Agent 在真实环境里会遇到各种失败：工具执行超时、API 返回错误、文件匹配失败、模型陷入死循环。一个健壮的 Agent 不是不失败，而是失败后能优雅恢复。本章我们梳理三类核心的错误恢复策略——重试、死循环检测、fallback chain——并看它们在不同开源项目里的具体实现。

最朴素的错误恢复是重试——失败了再来一次。但「再来一次」如果完全不改变输入，大概率还是失败（爱因斯坦对疯狂的定义：做同样的事期待不同结果）。有效的重试必须 改变某些条件。

对 Agent 来说，重试通常不是代码层面的自动重试（那是 API client 的活，如 axios-retry），而是 把错误反馈给 LLM，让它在下一轮自己修正。这是 Agent 比传统软件更灵活的地方——传统软件遇到错误只能按预设逻辑处理，Agent 能理解错误并创造性应对。

// Agent 层面的重试（反馈给 LLM）
async function executeToolWithFeedback(
  toolCall: ToolCall,
  session: Session,
): Promise<ToolResult> {
  const result = await executeTool(toolCall);

  if (!result.isError) return result;

  // 失败了，把错误反馈给 LLM，让它在下一轮决定怎么办
  session.addMessage({
    role: "tool",
    content: `[执行失败] ${result.content}\n\n` +
             `请分析失败原因，可能需要：\n` +
             `- 检查命令拼写或参数\n` +
             `- 读取相关文件确认上下文\n` +
             `- 换一种方法尝试`,
    tool_call_id: toolCall.id,
  });

  // 不在这里自动重试，让 Agent 循环的下一轮由 LLM 决定
  return result;
}

这段代码体现了一个重要原则：错误反馈要具体且可操作。不要只说「失败了，请重试」，要给出可能的排查方向。LLM 看到具体提示后，更可能采取正确的修正动作，而不是盲目重试同一个操作。

对于 代码层面的自动重试（针对网络抖动、临时性错误），策略是经典的指数退避：

// API 调用的指数退避重试
async function callWithRetry<T>(
  fn: () => Promise<T>,
  options: { maxRetries?: number; baseDelay?: number } = {}
): Promise<T> {
  const maxRetries = options.maxRetries ?? 3;
  const baseDelay = options.baseDelay ?? 1000;

  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      return await fn();
    } catch (err) {
      // 只对可重试的错误重试（网络错误、5xx、429）
      if (!isRetryable(err) || attempt === maxRetries) throw err;

      const delay = baseDelay * Math.pow(2, attempt) +
                    Math.random() * 1000;  // 加抖动避免雷同
      await sleep(delay);
    }
  }
  throw new Error("unreachable");
}

function isRetryable(err: any): boolean {
  if (err.status >= 500) return true;           // 服务端错误
  if (err.status === 429) return true;          // 限流
  if (err.code === "ECONNRESET") return true;   // 网络断开
  if (err.code === "ETIMEDOUT") return true;    // 超时
  return false;  // 4xx（除 429）不重试，是请求本身的问题
}

注意 isRetryable 的区分：只对临时性错误重试。4xx 错误（除了 429 限流）通常是请求本身的问题（参数错、鉴权失败），重试不会成功，应该立即反馈给 LLM。这种「区分错误类型」的策略避免了无意义的重试浪费。

★ 二、死循环检测：Agent 最怕的故障

Agent 有一种特别危险的失败模式：死循环。模型反复调用同一个工具、同样的参数，永远不结束。这会疯狂消耗 token，且永远完不成任务。Codex 在 Agent 循环里用重复调用检测来防这个（参见 154 章），我们在这里更系统地讨论。

死循环有几种典型模式：

  • 完全重复：连续 N 次调用完全相同的工具 + 参数。最明显，最容易检测。
  • 振荡：在两个状态间来回切换（A→B→A→B→...）。稍微隐蔽。
  • 渐近但不收敛：每次参数略有变化，但目标永远达不成（如反复微调一个永远不过的测试）。最难检测。
// 死循环检测器（简化）
export class LoopDetector {
  private history: ToolCall[] = [];
  private readonly windowSize = 10;

  /** 检查当前调用是否构成循环 */
  detect(call: ToolCall): LoopWarning | null {
    this.history.push(call);
    if (this.history.length > this.windowSize) {
      this.history.shift();
    }

    // 检测 1：连续重复（最近 3 次完全相同）
    const recent = this.history.slice(-3);
    if (recent.length === 3 &&
        recent.every(c => this.sameCall(c, recent[0]))) {
      return {
        type: "repetition",
        message: `连续 3 次相同的工具调用（${call.name}），可能是死循环`,
      };
    }

    // 检测 2：振荡（A-B-A-B 模式）
    if (this.history.length >= 4) {
      const last4 = this.history.slice(-4);
      if (this.sameCall(last4[0], last4[2]) &&
          this.sameCall(last4[1], last4[3]) &&
          !this.sameCall(last4[0], last4[1])) {
        return {
          type: "oscillation",
          message: `检测到振荡模式（${last4[0].name} ↔ ${last4[1].name}）`,
        };
      }
    }

    // 检测 3：同一工具调用次数过多
    const toolCounts: Record<string, number> = {};
    for (const c of this.history) {
      toolCounts[c.name] = (toolCounts[c.name] || 0) + 1;
    }
    for (const [tool, count] of Object.entries(toolCounts)) {
      if (count >= 5) {
        return {
          type: "frequency",
          message: `窗口内 ${tool} 被调用了 ${count} 次，异常频繁`,
        };
      }
    }

    return null;
  }

  private sameCall(a: ToolCall, b: ToolCall): boolean {
    return a.name === b.name &&
           JSON.stringify(a.params) === JSON.stringify(b.params);
  }
}

检测到循环后怎么处理？有几种策略：软警告（把警告反馈给 LLM，让它自己调整）、硬中断（直接停止 Agent 循环，要求用户介入）、强制提示（在下一轮的 system 消息里强调「你似乎在重复，请尝试不同的方法」）。大多数项目用软警告 + 硬中断的组合——先软警告给 LLM 一次自我修正的机会，连续几次警告后升级为硬中断。

★ 三、Fallback Chain：一种不行换另一种

有些任务有多种实现路径，一条走不通就换另一条。这就是 fallback chain——预先定义一组策略，按优先级尝试，第一个成功的为准。

对代码编辑 Agent，典型的 fallback chain 是我们第 169 章讲的 Cline 的策略：SEARCH/REPLACE 补丁 → 全文件重写 → 重新读取后再试。每一步失败触发下一步，而不是卡死在第一步。

// 通用的 fallback chain（简化）
export class FallbackChain<TInput, TResult> {
  constructor(private strategies: Array<{
    name: string;
    attempt: (input: TInput) => Promise<TResult>;
    shouldFallback: (err: any) => boolean;  // 这个错误要不要触发 fallback
  }>) {}

  async run(input: TInput): Promise<TResult> {
    const errors: string[] = [];
    for (const strategy of this.strategies) {
      try {
        return await strategy.attempt(input);
      } catch (err) {
        errors.push(`${strategy.name}: ${err.message}`);
        if (!strategy.shouldFallback(err)) {
          // 不可 fallback 的错误，立即抛出
          throw err;
        }
        // 可 fallback，继续下一个策略
      }
    }
    throw new Error(`所有策略都失败：\n${errors.join("\n")}`);
  }
}

// 文件编辑的 fallback chain
const editChain = new FallbackChain([
  {
    name: "search-replace",
    attempt: async (edit) => applySearchReplace(edit),
    shouldFallback: (err) => err.code === "MATCH_NOT_FOUND",
  },
  {
    name: "full-rewrite",
    attempt: async (edit) => applyFullRewrite(edit),
    shouldFallback: (err) => err.code === "SYNTAX_ERROR",
  },
  {
    name: "read-then-edit",
    attempt: async (edit) => {
      // 先重新读文件，再试
      const content = await readFile(edit.path);
      return applySearchReplace({ ...edit, currentContent: content });
    },
    shouldFallback: () => false,  // 最后一步，不 fallback
  },
]);

fallback chain 的关键是 shouldFallback 函数——它决定 哪些错误值得换策略，哪些应该立即抛出。比如 MATCH_NOT_FOUND 值得换全文件重写（匹配问题是补丁策略的弱点），但 PERMISSION_DENIED 不值得 fallback（换策略也绕不过权限）。这种「区分错误性质」的设计，避免了无意义的 fallback 浪费。

★ 四、超时的恢复策略

超时（工具执行超过时限）是常见的失败。对超时的恢复不能简单重试——如果是命令本身慢（如跑大测试套件），重试只会再超时一次。恢复策略应该是：

  1. 识别超时原因：是命令慢、还是死锁、还是等待用户输入？
  2. 如果是命令慢：询问用户是否加长超时，或建议拆分任务（跑单个测试而非全套）。
  3. 如果是死锁：kill 进程，报告「可能死锁」，让 LLM 换方法。
  4. 如果是等待输入：提示「命令需要交互输入，Agent 无法处理」，建议用户手动跑。
// 超时恢复（简化）
async function handleTimeout(
  command: string,
  session: Session,
): Promise<ToolResult> {
  // 分析命令特征
  if (/test|spec/i.test(command)) {
    return {
      content: `[超时] 测试命令执行超时。可能测试套件太大。
建议：
- 用 -t 参数只跑相关测试
- 或增加 timeout（当前 120s）后重试`,
      isError: true,
    };
  }
  if (/^(npm install|yarn|pnpm install)/.test(command)) {
    return {
      content: `[超时] 安装依赖超时。可能网络慢或依赖树太大。
建议：
- 检查网络（特别是公司代理）
- 尝试 --registry 换源
- 或增加 timeout 到 300s`,
      isError: true,
    };
  }
  // 通用超时反馈
  return {
    content: `[超时] 命令执行超过 120 秒被终止。
如果这是预期的长时间命令，请告诉我加长超时。
如果非预期，命令可能卡住了（死锁或等待输入）。`,
    isError: true,
  };
}

这种 按命令类型给定制建议 的反馈，比泛泛说「超时了」有效得多。LLM 看到具体建议，能更好地决定下一步。

★ 五、熔断：连续失败时停下来

如果 Agent 连续多次失败（不管怎么重试都不行），继续尝试是浪费。这时候应该 熔断——停止尝试，把控制权交还给用户。这是从电路保护借鉴的思想：电流过载时保险丝熔断，防止起火。

// 熔断器（简化）
export class CircuitBreaker {
  private failureCount = 0;
  private readonly threshold = 5;
  private lastFailures: string[] = [];

  recordSuccess() {
    this.failureCount = 0;
    this.lastFailures = [];
  }

  recordFailure(reason: string): boolean {
    this.failureCount++;
    this.lastFailures.push(reason);
    return this.failureCount >= this.threshold;
  }

  getFailures(): string[] {
    return [...this.lastFailures];
  }

  isTripped(): boolean {
    return this.failureCount >= this.threshold;
  }
}

// 在 Agent 循环里
async function runAgentLoop(session: Session) {
  const breaker = new CircuitBreaker();
  for (let turn = 0; turn < MAX_TURNS; turn++) {
    const result = await runTurn(session);
    if (result.success) {
      breaker.recordSuccess();
    } else {
      const tripped = breaker.recordFailure(result.error);
      if (tripped) {
        // 熔断：停止循环，报告给用户
        session.addMessage({
          role: "system",
          content: `[熔断] Agent 连续失败 ${breaker.getFailures().length} 次：
${breaker.getFailures().map((f, i) => `${i+1}. ${f}`).join("\n")}

已停止自动尝试。请用户介入：
- 检查任务是否可行
- 调整环境（权限、依赖等）
- 或换一种方式描述任务`,
        });
        break;
      }
    }
  }
}

熔断的核心价值是 防止 Agent 在死胡同里无限消耗。没有熔断，Agent 可能连续失败 50 次（每次都烧 token），直到达到 MAX_TURNS。有熔断，5 次失败就停，把资源还给用户做更有价值的事。

★ 本章小结

错误恢复是 Agent 健壮性的核心。要点回顾：

  • Agent 层重试：把错误具体反馈给 LLM，让它创造性修正，而非盲目重试；API 层用指数退避 + 抖动，只重试临时性错误。
  • 死循环检测：完全重复（连续相同）、振荡（A-B-A-B）、频率异常（同工具调用过多）三种检测，软警告 + 硬中断组合。
  • Fallback chain：多策略按优先级尝试，shouldFallback 决定哪些错误值得换策略，避免无意义 fallback。
  • 超时恢复：按命令类型给定制建议（测试慢、install 慢、死锁、等输入），优于泛泛报错。
  • 熔断：连续失败达阈值就停，把失败原因汇总报告用户，防止在死胡同无限烧 token。
  • 反馈要具体可操作：所有错误反馈都应给排查方向，不要只说「失败」。
