端侧部署实战
手机、嵌入式与量化+蒸馏+剪枝组合拳

不是所有 AI 都跑在云端 GPU 上。手机拍照里的场景识别、智能手表上的心率异常检测、汽车座舱里的离线语音助手、工厂流水线上的视觉质检——这些场景要么对延迟敏感、要么对隐私敏感、要么网络不可靠，必须把模型塞进手机或嵌入式设备。端侧部署是一场"在毫瓦级功耗和兆字节存储里榨性能"的极限工程。本章带你穿越主流端侧框架，并掌握量化、蒸馏、剪枝的组合优化策略。

阅读约 33 分钟 难度 ★★★★

★ 为什么要把模型放到端侧

端侧部署（On-Device / Edge Deployment）指模型直接在用户的手机、手表、汽车、IoT 设备上推理，不依赖云端服务器。这种部署方式在三类场景下不可替代。第一是延迟敏感：增强现实（AR）滤镜要 30ms 内出结果，往返云端根本来不及；自动驾驶感知要在毫秒级响应，网络抖动就是事故。第二是隐私敏感：医疗影像、人脸识别、个人语音，用户不愿也不能上传到服务器。第三是离线与成本：偏远地区、飞行模式、无网络设备仍要能用，且每次推理都走云端会带来高昂的带宽和算力账单。

但端侧部署的约束极其严苛，与云端部署是两个世界。一张手机 NPU（如苹果 A17、高通骁龙 8 Gen 3）的算力大约 30-50 TOPS，而一张 H100 是 2000 TOPS 级别，差了 40 倍以上。内存方面，旗舰手机能给模型用的 RAM 通常只有几百 MB，且还要和应用本身、操作系统共享。功耗方面，手机电池持续大电流放电会快速发热降频，模型必须在几百毫瓦内完成推理。存储方面，应用商店对单个 App 的体积有上限，模型文件动辄上百 MB 会让用户望而却步。这些约束倒逼出"极小、极快、极省"的模型设计哲学。

端侧模型的典型规模有一个经验区间：视觉模型（MobileNet、EfficientNet-Lite 系列）大约 5-20 MB、几十毫秒推理；语音唤醒模型（KWS）小于 1 MB、几毫秒推理；端侧 NLP 模型（MobileBERT、distilled BERT）大约 30-100 MB、上百毫秒；端侧 LLM（通过 INT4 量化的 1B-3B 模型）大约 1-3 GB、每秒十几到几十 token。理解这些量级，是做端侧项目可行性评估的起点。

★ 三大端侧框架：TFLite / Core ML / MLC-LLM

端侧框架的核心职责是把训练好的模型转换成设备原生格式，并调用硬件加速器（NPU/DSP/GPU）高效推理。三大主流框架各有所长：TensorFlow Lite 覆盖最广，Core ML 是苹果生态的唯一选择，MLC-LLM 则是端侧跑大模型的黑马。

TensorFlow Lite（TFLite）是 Google 推出的跨平台端侧推理框架，支持 Android、iOS、嵌入式 Linux、微控制器（TFLite Micro）。它的生态最完善——从算子覆盖、硬件委托（GPU/NNAPI/Core ML delegate）、量化工具（TFLite Converter）到模型分析器一应俱全。Android 设备尤其推荐 TFLite，因为能通过 NNAPI 直接调用各厂商 NPU。痛点是对动态形状、控制流的支持不如静态模型好，跑大语言模型较吃力。

Core ML是苹果的端侧框架，是 iOS/macOS 应用的唯一原生选择。它的优势是深度集成苹果 Neural Engine（ANE）——苹果自研 NPU，能效比远超 GPU。把模型转成 Core ML 格式（.mlmodelc）后，框架会自动在 CPU/GPU/ANE 间选最优后端。Core ML 还原生支持对模型做 INT8/FP16 量化、层融合、内存优化。下面这段代码演示用 coremltools 把 PyTorch 模型转到 Core ML 并量化。

# pip install coremltools torch torchvision numpy
import coremltools as ct
import torch, torchvision
import numpy as np

# 1) 加载一个训练好的 PyTorch 模型（以 MobileNetV3 为例）
model = torchvision.models.mobilenet_v3_small(weights="DEFAULT").eval()

# 2) 用 example input 做 trace（Core ML 需要静态形状）
example = torch.rand(1, 3, 224, 224)
traced = torch.jit.trace(model, example)

# 3) 转换为 Core ML 中间格式（ML Program）
mlp = ct.convert(
    traced,
    inputs=[ct.TensorType(shape=example.shape, name="image")],
    compute_precision=ct.precision.FLOAT16,   # 全 FP16，省一半内存
    minimum_deployment_target=ct.target.iOS15,
)

# 4) 进一步做权重量化到 INT8（激活仍 FP16，精度损失极小）
mlp_int8 = ct.optimize.coreml.quantize_weights(
    mlp, mode="linear_symmetric", dtype=np.int8
)

# 5) 保存为 .mlpackage，Xcode 直接集成进 App
mlp_int8.save("MobileNetV3.mlpackage")

# 输出模型大小对比：
#   PyTorch FP32:  ~21 MB
#   Core ML FP16:  ~11 MB
#   Core ML INT8:  ~6 MB   （且在 ANE 上推理速度可能反而更快）

MLC-LLM是基于 Apache TVM 编译器栈的通用大模型部署框架，最大特点是能在手机、笔记本、甚至浏览器（WebGPU）上跑大语言模型。它通过 TVM 的统一中间表示，把 LLaMA、Phi、Qwen 等模型编译成针对 iPhone A 系列、安卓骁龙、苹果 Silicon、Intel/AMD 各类后端的原生代码，并支持 INT4 量化。MLC-LLM 让"手机本地跑 LLM"从口号变成现实——一台 iPhone 15 Pro 能以每秒约 20 token 跑一个 3B 的 INT4 模型。

★ 量化、蒸馏、剪枝的组合策略

单靠某一种技术很难把模型压到端侧可承受的体积和速度，工业界通行做法是把量化、蒸馏、剪枝三种技术组合使用，称为"压缩三件套"。理解三者的作用机制和最佳组合顺序，是端侧优化的核心。先看三者的定位：剪枝减少参数数量（让模型变稀疏或变窄）、蒸馏用大模型教小模型（让小模型在更少参数下达到接近的效果）、量化降低每个参数的位宽（让每个参数占的存储和算力更少）。三者作用在模型的不同维度，因此可以叠加。

典型的组合流程是"蒸馏优先，剪枝其次，量化最后"。第一步，用知识蒸馏训练一个结构上就适合端侧的小模型（比如从 BERT-base 蒸馏到 6 层的 TinyBERT，或从 LLaMA-7B 蒸馏到 1.3B）。第二步，对小模型做剪枝进一步压缩（比如 magnitude pruning 剪掉 30% 权重）。第三步，对剪枝后模型做量化（权重 INT4/INT8，激活 INT8 或 FP16）。这个顺序的原因是：蒸馏改变模型结构，应该在最前面；剪枝改变权重拓扑，需要在量化前完成以避免稀疏模式被量化干扰；量化是最后一步，因为它对精度的影响需要直接在目标硬件上验证，且量化后通常不再做重训练。下面这段代码演示蒸馏 + 量化的组合实现。

import torch
import torch.nn as nn
import torch.nn.functional as F

# === 第一步：知识蒸馏（大模型 teacher 教小模型 student）===
class TinyStudent(nn.Module):
    def __init__(self, num_classes=10):
        super().__init__()
        self.encoder = nn.TransformerEncoder(
            nn.TransformerEncoderLayer(d_model=256, nhead=4, dim_feedforward=1024),
            num_layers=4,   # 远小于 BERT 的 12 层
        )
        self.head = nn.Linear(256, num_classes)
    def forward(self, x):
        return self.head(self.encoder(x).mean(dim=1))

def distill(teacher, student, train_loader, epochs=5, T=4.0, alpha=0.5):
    """teacher 输出软标签，student 同时学硬标签和软标签。"""
    opt = torch.optim.AdamW(student.parameters(), lr=5e-5)
    for epoch in range(epochs):
        for x, y in train_loader:
            with torch.no_grad():
                t_logits = teacher(x)
            s_logits = student(x)
            # 蒸馏 loss = alpha * KL(软标签) + (1-alpha) * CE(硬标签)
            kl = F.kl_div(
                F.log_softmax(s_logits / T, dim=-1),
                F.softmax(t_logits / T, dim=-1), reduction="batchmean"
            ) * (T * T)
            ce = F.cross_entropy(s_logits, y)
            loss = alpha * kl + (1 - alpha) * ce
            opt.zero_grad(); loss.backward(); opt.step()

# === 第二步：对蒸馏后的小模型做量化（PyTorch 动态量化，最简单）===
student = TinyStudent().eval()
distill(teacher, student, train_loader)

quantized = torch.quantization.quantize_dynamic(
    student,
    {nn.Linear, nn.TransformerEncoderLayer},   # 对这两类层做权重 INT8 量化
    dtype=torch.qint8,
)
# 体积对比：FP32 ~80MB → INT8 ~25MB → 再叠加 TFLite INT8 部署约 ~12MB
# 精度通常保留 95%+，推理速度在支持 INT8 的 NPU 上提升 2-3 倍。

组合策略的关键经验是"每一步都要验证精度"。不要一口气把蒸馏+剪枝+量化全做完才评测——那样一旦精度掉了，你都不知道是哪一步的锅。正确做法是每做一步就在验证集上评测，记录精度-体积-速度三角的 trade-off，逐步逼近帕累托最优。

★ 端侧部署的工程实战要点

端侧部署的坑远不止模型压缩本身，还涉及硬件适配套、内存管理、功耗控制、模型分发等多个工程问题。这些细节往往决定一个端侧项目能不能真正落地。

第一是硬件适配套（hardware adaptation kit）。同一份模型在不同手机的 NPU 上表现差异巨大——高通骁龙、联发科天玑、华为麒麟、苹果 ANE 的算子支持、精度支持、内存模型都不同。生产环境通常要为不同芯片准备不同优化版本的模型，或在运行时根据设备能力（如 Android 的 DeviceUtils、iOS 的 MLModel 配置）动态选择。第二是内存峰值控制。端侧推理的峰值内存比模型文件大小更重要，因为峰值超了就会被系统杀进程。常见优化手段是分块推理、流式激活、复用中间 buffer。第三是功耗与热设计。持续高频推理会让手机发热降频，体验崩坏。建议把推理任务批量化、避开长时间连续运行，并利用系统的低功耗核（如 Android 的 LITTLE 核）处理轻量任务。

第四是模型分发与更新。几十 MB 的模型塞进 App 包会让下载量暴跌，业界做法是首次启动时按需下载模型、做差分更新、用 Play Asset Delivery / iOS On-Demand Resources 等机制分阶段下发。第五是版本兼容。端侧模型一旦下发到用户设备，就很难强制更新，必须考虑模型版本与 App 版本、OS 版本、硬件代际的兼容矩阵，做好降级策略——NPU 不可用时回退到 CPU 推理，新模型加载失败时回退到上一版。下面这段伪代码展示运行时硬件适配的典型逻辑。

// Android (Kotlin) 运行时硬件适配伪代码
fun loadOptimizedModel(context: Context): Interpreter {
    val options = Interpreter.Options()
    // 1) 检测设备 NPU 能力
    val nnapiAvailable = NnApiDelegate.isNnApiAvailable(context)
    val devicePerf = readDevicePerfClass()   // OEM 通过 PowerStats 暴露

    when {
        nnapiAvailable && devicePerf >= PERF_HIGH -> {
            // 旗舰机：用完整 INT8 模型 + NNAPI delegate 跑 NPU
            val nnApi = NnApiDelegate(
                NnApiDelegate.Options().setAllowFp16(false)
            )
            options.addDelegate(nnApi)
            return Interpreter(loadModelFile("model_v3_int8.tflite"), options)
        }
        devicePerf >= PERF_MEDIUM -> {
            // 中端机：用 INT8 模型但跑 GPU delegate
            val gpu = GpuDelegate()
            options.addDelegate(gpu)
            return Interpreter(loadModelFile("model_v3_int8.tflite"), options)
        }
        else -> {
            // 低端机：用更小的 INT8 模型，CPU 4 线程
            options.setNumThreads(4)
            return Interpreter(loadModelFile("model_v3_small_int8.tflite"), options)
        }
    }
    // 配合 try/catch：任何 delegate 加载失败都回退到纯 CPU + 上一版模型
}

这种"按设备能力分发不同模型"的策略是头部 App（相机、相册、翻译）的标配。它把"一个模型适配所有设备"的难题，转化为"按性能分档准备多个模型"的工程问题，虽然增加了模型维护成本，但能保证从旗舰机到低端机都有可用体验。

import numpy as np
from dataclasses import dataclass
from typing import List, Optional

@dataclass
class PerformanceMetrics:
    latency_ms: float
    throughput_ops: float
    memory_mb: float
    accuracy: float

def benchmark(fn, iterations=100):
    """通用性能基准测试"""
    import time
    t0 = time.time()
    for _ in range(iterations):
        result = fn()
    elapsed = time.time() - t0
    return PerformanceMetrics(
        latency_ms=elapsed * 1000 / iterations,
        throughput_ops=iterations / elapsed,
        memory_mb=0,  # 实际中用 psutil 获取
        accuracy=0.0  # 根据具体任务计算
    )

# 示例：对比两种方案的性能
def approach_a(data):
    """方案A：直接计算"""
    return np.softmax(data @ data.T / np.sqrt(data.shape[-1]), axis=-1)

def approach_b(data):
    """方案B：分块计算（省内存）"""
    chunk_size = 64
    results = []
    for i in range(0, data.shape[0], chunk_size):
        chunk = data[i:i+chunk_size]
        results.append(np.softmax(chunk @ data.T / np.sqrt(data.shape[-1]), axis=-1))
    return np.vstack(results)

data = np.random.randn(512, 768)
metrics_a = benchmark(lambda: approach_a(data))
metrics_b = benchmark(lambda: approach_b(data))
print(f"方案A: {metrics_a.latency_ms:.1f}ms, {metrics_a.throughput_ops:.0f} ops/s")
print(f"方案B: {metrics_b.latency_ms:.1f}ms, {metrics_b.throughput_ops:.0f} ops/s")

常见问题FAQ

Q: 如何选择合适的方案？ 先用方案A跑通验证正确性，再根据实际瓶颈（内存/延迟/精度）切换到方案B或C。永远不要过早优化。

Q: 生产环境最大的坑是什么？ 过度工程化——在需求不明确时投入大量精力做复杂架构。建议MVP先行，收集反馈后迭代。

Q: 如何评估效果？ 定义明确的成功指标（准确率/延迟/成本），用golden set做回归测试，A/B测试验证改进效果。至少100个样本才有统计显著性。
