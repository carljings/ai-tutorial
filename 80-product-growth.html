产品增长
用户留存、A/B 测试与数据驱动决策

做一个 AI 产品容易，让它在市场上增长很难。很多 AI 应用上线后用户量停滞不前，留存率惨不忍睹——用户来试一次就再也不回来。这一章是 AI 产品增长的实战手册：用户留存怎么做、A/B 测试怎么设计、怎么用 SQL 做数据驱动决策。无论你是产品经理还是工程师，这套增长方法论都直接可用。

★ 二、用户留存：AI 产品生死线

留存是 AI 产品最关键的指标。获取一个新用户的成本是留住一个老用户的 5-25 倍。如果你的次日留存只有 10%，说明 90% 的用户试一次就走了——这通常意味着产品没有持续价值。

2.1 留存分析 SQL

-- 计算同期群留存率（Cohort Retention）
-- 这是产品分析最核心的查询之一

WITH user_cohorts AS (
    -- 按用户首次使用日期分组（同期群）
    SELECT
        user_id,
        DATE(first_use_time) AS cohort_date,
        DATE(first_use_time) AS first_day
    FROM user_events
    WHERE event_type = 'first_use'
),
daily_active AS (
    -- 每个用户的活跃日期
    SELECT DISTINCT
        user_id,
        DATE(event_time) AS active_date
    FROM user_events
    WHERE event_type = 'session_start'
)
SELECT
    cohort_date,
    -- 第 N 天的留存率 = （第 N 天还活跃的用户数 / 同期群总人数）
    COUNT(DISTINCT CASE WHEN active_date = first_day THEN user_id END) AS day_0,
    COUNT(DISTINCT CASE WHEN active_date = first_day + INTERVAL '1 day' THEN user_id END) AS day_1,
    COUNT(DISTINCT CASE WHEN active_date = first_day + INTERVAL '7 day' THEN user_id END) AS day_7,
    COUNT(DISTINCT CASE WHEN active_date = first_day + INTERVAL '30 day' THEN user_id END) AS day_30,
    -- 转化成百分比
    ROUND(100.0 * COUNT(DISTINCT CASE WHEN active_date = first_day + INTERVAL '1 day' THEN user_id END)
          / COUNT(DISTINCT CASE WHEN active_date = first_day THEN user_id END), 1) AS day1_retention_pct,
    ROUND(100.0 * COUNT(DISTINCT CASE WHEN active_date = first_day + INTERVAL '7 day' THEN user_id END)
          / COUNT(DISTINCT CASE WHEN active_date = first_day THEN user_id END), 1) AS day7_retention_pct
FROM user_cohorts uc
LEFT JOIN daily_active da ON uc.user_id = da.user_id
GROUP BY cohort_date
ORDER BY cohort_date DESC
LIMIT 30;

-- 解读：
-- Day 1 留存 < 20%：产品激活有问题，用户没感受到价值
-- Day 7 留存 < 10%：产品缺乏持续价值
-- Day 30 留存 < 5%：用户没有形成习惯
-- 优秀 AI 产品：Day 1 > 40%, Day 7 > 20%, Day 30 > 10%

★ 三、A/B 测试：用数据做决策

A/B 测试是数据驱动决策的核心工具。不要靠直觉判断哪个方案好——让真实用户的数据告诉你。但 AI 产品的 A/B 测试比传统产品复杂，因为输出质量本身是模糊的。

3.1 AI 产品的 A/B 测试设计

# AI 产品 A/B 测试的完整流程
class ABTestFramework:
    def __init__(self):
        self.min_sample_size = 1000  # 每组最少样本量
        self.significance_level = 0.05  # 统计显著性

    def design_test(self, hypothesis, metric, variants):
        """设计 A/B 测试。"""
        return {
            "假设": hypothesis,           # "版本 B 的留存率比 A 高 5%"
            "核心指标": metric,           # "day7_retention"
            "对照组": variants["A"],      # 当前版本
            "实验组": variants["B"],      # 新版本
            "样本量": self.calculate_sample_size(metric),
            "持续时间": "至少 14 天（覆盖完整周期）",
            "分流方式": "随机分桶，确保用户特征均衡",
        }

    def calculate_sample_size(self, metric, baseline_rate, mde=0.05):
        """计算所需样本量（最小可检测效应 MDE）。"""
        import math
        # 简化的样本量计算（基于正态近似）
        z_alpha = 1.96   # 95% 置信度
        z_beta = 0.84    # 80% 统计功效
        p = baseline_rate
        n = ((z_alpha + z_beta) ** 2 * 2 * p * (1-p)) / (mde ** 2)
        return int(math.ceil(n))

3.2 A/B 测试结果分析 SQL

-- A/B 测试结果分析
-- 假设：用户被随机分到 control 和 treatment 两组

SELECT
    experiment_group,
    COUNT(DISTINCT user_id) AS total_users,
    -- 留存率
    ROUND(100.0 * COUNT(DISTINCT CASE
        WHEN days_active >= 7 THEN user_id END) / COUNT(DISTINCT user_id), 2
    ) AS day7_retention_pct,
    -- 平均使用次数
    ROUND(AVG(total_sessions), 1) AS avg_sessions,
    -- 平均满意度（1-5 分）
    ROUND(AVG(satisfaction_score), 2) AS avg_satisfaction,
    -- 付费转化率
    ROUND(100.0 * COUNT(DISTINCT CASE
        WHEN is_paid THEN user_id END) / COUNT(DISTINCT user_id), 2
    ) AS conversion_pct
FROM experiment_results
WHERE experiment_id = 'prompt_v2_test'
  AND date_range BETWEEN '2026-06-01' AND '2026-06-14'
GROUP BY experiment_group;

-- 预期输出：
-- experiment_group | total_users | day7_retention | avg_sessions | satisfaction | conversion
-- control          | 5000        | 18.5           | 12.3         | 4.1          | 3.2
-- treatment        | 5000        | 22.1           | 15.7         | 4.3          | 4.1
--
-- 结论：treatment 组在所有指标上都优于 control
-- 还需做统计显著性检验（t-test 或 chi-square），确认差异不是偶然

★ 四、数据驱动决策的完整流程

数据驱动不是「看仪表盘」，而是「用数据回答业务问题」。这里用几个真实的产品分析查询来演示。

-- 分析 1：找出高价值用户的行为特征
-- 什么样的用户最容易付费？
WITH user_stats AS (
    SELECT
        u.user_id,
        u.is_paid,
        COUNT(DISTINCT DATE(e.event_time)) AS active_days,
        COUNT(*) FILTER (WHERE e.event_type = 'chat') AS total_chats,
        AVG(e.session_duration) AS avg_session_duration,
        MAX(DATE(e.event_time) - MIN(DATE(e.event_time))) AS lifespan
    FROM users u
    JOIN events e ON u.user_id = e.user_id
    WHERE e.event_time >= NOW() - INTERVAL '30 days'
    GROUP BY u.user_id, u.is_paid
)
SELECT
    is_paid,
    AVG(active_days) AS avg_active_days,
    AVG(total_chats) AS avg_chats,
    ROUND(AVG(avg_session_duration), 1) AS avg_session_min
FROM user_stats
GROUP BY is_paid;

-- 发现：付费用户平均活跃天数是免费用户的 3 倍
-- 启示：促活是转化的前提，先提升使用频次

-- 分析 2：流失用户预测（谁最可能流失？）
SELECT
    user_id,
    days_since_last_active,
    total_sessions_30d,
    last_satisfaction_score,
    -- 简单的流失风险打分
    CASE
        WHEN days_since_last_active > 7 THEN '高风险'
        WHEN days_since_last_active > 3 AND total_sessions_30d < 5 THEN '中风险'
        ELSE '低风险'
    END AS churn_risk
FROM user_engagement_summary
ORDER BY churn_risk, days_since_last_active DESC;

-- 对高风险用户做定向召回（邮件/Push/优惠）

★ 六、增长飞轮设计

最好的增长不是持续投放获取流量，而是建立自运转的增长飞轮——产品本身就能带来新用户。

★ 本章小结

这一章我们覆盖了 AI 产品增长的核心方法论。核心要点：

  • AARRR 漏斗：AI 产品的特殊挑战是激活难（prompt 门槛）、留存难（没形成习惯）、变现难（免费替代品多）。
  • 留存是生死线：用 Cohort SQL 分析留存率，优秀 AI 产品 Day1>40%、Day7>20%、Day30>10%。
  • 留存策略：缩短 Aha Moment、融入工作流、个性化、留存触发器、习惯养成。
  • A/B 测试：让数据做决策，注意统计显著性，AI 产品要看质量指标不能只看行为指标。
  • 数据驱动：用 SQL 找高价值用户特征、预测流失、做定向召回。
  • 增长飞轮：内容飞轮、模板飞轮、协作飞轮、数据飞轮——让产品自带增长引擎。
