上下文打包
Lost in the Middle 与重排序策略

检索回 10 个文档一股脑塞进 prompt，LLM 真能平等对待吗？2023 年斯坦福的论文《Lost in the Middle》给出了否定答案：LLM 对 prompt 开头和结尾的信息关注多，对中间的信息关注少。这意味着文档的排列顺序、上下文窗口的利用率，会直接影响答案质量。本文拆解这个问题，给出几种实用的上下文打包和重排序策略。

《Lost in the Middle: How Language Models Use Long Contexts》这篇论文做了一个简洁有力的实验：给 LLM 一份长文档，在里面某个位置插入一个关键事实，然后问一个必须依赖这个事实才能回答的问题。结果发现：LLM 的回答准确率随关键事实所在位置呈 U 型曲线——在文档开头和结尾时准确率高（70-80%），在中间时显著下降（30-40%）。

原因在于 transformer 的注意力机制（虽然长上下文模型用了 RoPE、滑动窗口等优化，但本质问题仍在）：每个 token 对其他 token 的注意力在长上下文里被稀释，中间位置的 token 既不被"开头强注意力"也不被"结尾近因效应"覆盖，容易被忽略。

这对 RAG 的直接影响是：你精心检索回来的 top-10 文档，如果按检索分数降序塞进 prompt，排在第 4-7 位的文档（中间位置）很可能被 LLM 忽略。而很多时候最相关的文档恰好在中间——这就解释了为什么有时明明检索到了正确文档，LLM 还是答错。

★ 三种重排序打包策略

1. 把最好的放两头（interleave / 两端优先）

最直接的解法：既然中间被忽略，就把最重要的文档放到开头和结尾，把次要的放中间。具体做法是把 rerank 后的文档按 [1, 3, 5, 7, 9, 10, 8, 6, 4, 2] 这种交错顺序排——奇数位（高质量）往前和往后放，偶数位（次要）挤中间。

论文实验显示这个简单技巧能把答案准确率提升 5-10 个百分点，几乎是零成本的提升。

2. 截断：少即是多

另一个反直觉的发现：在 Lost in the Middle 效应下，塞 10 个文档有时不如塞 5 个。因为塞多了不仅中间文档被忽略，还引入了更多噪声干扰 LLM。所以策略是：rerank 后只取 top-3 到 top-5，宁缺毋滥。

选几个的依据：看 rerank 分数的分布。如果 top-3 分数远高于第 4 个（有明显断层），取 top-3 就够；如果分数平缓下降，可以多取几个但要配合重排序。

3. map-reduce：长文档分块多次问

当文档本身很长（每个都几千字），塞 3 个就占满上下文，这时换 map-reduce 策略：对每个文档单独问 LLM"这个文档能不能回答 query？如果能，答案是什么"，再合并各文档的子答案。这样每个文档独占 LLM 注意力，不受 Lost in the Middle 影响。代价是 LLM 调用次数等于文档数，成本高。

# 策略 1：交错重排，把高质量文档放两端
def interleave_reorder(ranked_docs):
    """
    ranked_docs: 按 rerank 分数降序的文档列表
    return: 优化后顺序（高质量在两端）
    """
    n = len(ranked_docs)
    if n <= 2:
        return ranked_docs
    # 奇数索引（0,2,4...）正向放前面，偶数索引（1,3,5...）反向放后面
    front = ranked_docs[0::2]                    # [d0, d2, d4, ...]
    back  = list(reversed(ranked_docs[1::2]))    # [..., d5, d3, d1]
    return front + back

# 示例：10 个文档
docs = [f"doc_{i}(score={10-i})" for i in range(10)]
reordered = interleave_reorder(docs)
print("重排后:", reordered)
# 输出: doc_0, doc_2, doc_4, doc_6, doc_8, doc_9, doc_7, doc_5, doc_3, doc_1
# 最高分的 doc_0 在最前，次高的 doc_1 在最后，最低分的 doc_9 在中间附近

★ 上下文窗口利用率

另一个常被忽视的问题：塞多少内容进上下文窗口。128K 窗口的模型，不代表塞满 128K 效果最好。实测发现：

  • 塞到 50-70% 容量通常是甜区。留点余量给 LLM 推理，避免被信息淹没。
  • 塞满 100%，Lost in the Middle 效应加剧，且 LLM 容易"放弃思考"直接抄最近的片段。
  • 太少（<10%）又浪费窗口、信息不足。RAG 场景一般 20-50% 是合理的起点。

实操上，按 token 计算而非字数。用 tiktoken 精确计算每个文档的 token 数，按目标利用率（如 50%）预算总 token 数，再决定塞几个文档。

# 精确计算 token 数 + 按预算打包
import tiktoken

enc = tiktoken.encoding_for_model("gpt-4o")
def count_tokens(text):
    return len(enc.encode(text))

def pack_context(docs, max_tokens=6000, target_utilization=0.6):
    """
    docs: rerank 后的文档列表（已降序）
    max_tokens: 模型上下文窗口（如 128000）
    target_utilization: 目标利用率（0.6 = 60%）
    return: 打包后的文档列表 + 实际 token 数
    """
    budget = int(max_tokens * target_utilization)
    packed, total = [], 0
    for doc in docs:
        t = count_tokens(doc)
        if total + t > budget:
            break
        packed.append(doc)
        total += t
    return packed, total

# 完整流程：rerank → 截断 → 交错重排 → 打包
ranked = reranker.rerank(query, candidates, top_k=10)
packed, used = pack_context(ranked, max_tokens=8192, target_utilization=0.6)
final_order = interleave_reorder(packed)
prompt = build_prompt(query, final_order)
print(f"实际塞入 {len(final_order)} 个文档，{used} tokens")

★ 进阶：上下文压缩与引用标注

上下文压缩

检索回的文档常有大量无关内容（页眉页脚、过渡句、不相关段落）。在打包前用一个小模型或 LLM 把每个文档压缩成"和 query 最相关的 1-2 句"，能显著提升信噪比。Anthropic 的 Contextual Retrieval（136 篇）也是这个思路的变体。代价是每文档一次压缩调用，可用 batch + 小模型控成本。

引用标注

让 LLM 在答案里标注每个论断来自哪个文档（"[1]"、"[2]"），有几个好处：① 用户可点击验证，增加信任；② 方便定位幻觉（标了引用但引用里没这内容）；③ 强制 LLM "对号入座"，减少幻觉。实现上在 prompt 里要求"每个事实后加 [文档编号]"，并把文档编号写清楚。

# 引用标注的 prompt 模板
RAG_PROMPT_WITH_CITATION = """你是一个严谨的问答助手。请只基于下面提供的文档回答问题。
要求：
1. 每个事实性论断后必须标注来源，格式 [文档N]，N 是文档编号。
2. 如果文档里没有相关信息，直接说"提供的文档中没有相关信息"，不要编造。
3. 答案要简洁，重点突出。

文档：
{docs_with_numbers}

问题：{question}

回答："""

def build_prompt_with_citation(query, docs):
    docs_text = "\n\n".join([f"[文档{i+1}] {d}" for i, d in enumerate(docs)])
    return RAG_PROMPT_WITH_CITATION.format(
        docs_with_numbers=docs_text, question=query)

# 实测：加引用后，faithfulness（见 140 篇）通常提升 10-15 个点

★ 上下文工程：从打包到提示词设计

上下文打包不只是排序和截断，它和提示词设计紧密结合，统称"上下文工程"。这是 RAG 里最考验工程艺术的部分。

提示词结构。一个优秀的 RAG prompt 不只是"文档 + 问题 + 回答"，而是有清晰结构：

  • 角色定义：让 LLM 扮演专家角色（"你是一个严谨的技术文档助手"），提升答案的专业性。
  • 约束规则：明确告诉 LLM 怎么用文档（"只基于提供的文档"、"不确定就说不知道"、"每个事实标注来源"）。
  • 文档呈现：用清晰分隔符和编号呈现文档，让 LLM 容易引用（"[文档1] ... [文档2] ..."）。
  • 问题重述：把用户问题用清晰的语言重述，避免歧义。
  • 输出格式：要求结构化输出（要点列表、引用标注），便于用户阅读和验证。
# 结构化 RAG prompt 模板
RAG_PROMPT_TEMPLATE = """你是一个严谨的{domain}助手。请基于下面提供的文档回答用户问题。

【回答规则】
1. 只基于【提供的文档】回答，不要编造文档外的信息。
2. 如果文档中没有相关信息，直接说"提供的文档中没有相关信息"。
3. 每个事实性论断后用 [文档N] 标注来源。
4. 答案要点化，重点突出。

【提供的文档】
{numbered_docs}

【用户问题】
{question}

【你的回答】
"""
def build_rag_prompt(query, docs, domain="技术"):
    numbered = "

".join([f"[文档{i+1}] {d}" for i, d in enumerate(docs)])
    return RAG_PROMPT_TEMPLATE.format(
        domain=domain, numbered_docs=numbered, question=query)

动态上下文。不同查询类型的 prompt 应该不同：事实查询用简洁 prompt（要精确）、解释性查询用引导 prompt（要展开）、对比查询用结构 prompt（要分点对比）。Router 在分流时不仅选检索路径，也选 prompt 模板。

Few-shot 示例。在 prompt 里加 1-2 个"问题-文档-答案"的示例，能显著提升答案格式和引用质量。但示例要精挑细选，覆盖典型场景，且不能太多（占上下文）。

上下文工程的核心认知：检索回来的文档质量再好，prompt 写得烂也白搭。很多团队花大力气优化检索，却用最简陋的 prompt，效果出不来。prompt 和检索是 RAG 的两条腿，要协同优化。

★ 上下文不足时的兜底策略

有时检索回来的上下文确实不够回答问题——可能库里就没相关文档，或检索失误。这时硬让 LLM 答只会幻觉。需要兜底策略。

置信度判断。让 LLM（或单独的分类器）判断"基于这些文档能不能回答这个问题"。如果不能，明确告诉用户"我没有足够信息"，而非编造。这比幻觉好得多——用户至少知道要换种问法或找别处。

追问澄清。当查询模糊或上下文不足时，LLM 反问用户澄清："你是指 Acme Corp 的 API 还是 SDK？" 这把"答不好"转化为"获取更多信息后再答"，提升最终解决率。

多轮检索。第一轮检索不足时，让 LLM 基于已有上下文生成"追问查询"，再检索一轮。这是 Self-RAG / Flare 的思路——动态决定要不要再检索。多轮检索能捞回第一轮漏的相关文档。

降级到通用知识。某些场景（如闲聊、常识问题）可以允许 LLM 用通用知识回答，但要明确标注"以下是基于通用知识的回答，未从知识库检索"。这区别于"基于文档的回答"，避免混淆。

# 兜底策略：判断 + 追问 + 多轮检索
def rag_with_fallback(query, retriever, llm, max_rounds=2):
    context = []
    for round_i in range(max_rounds):
        new_docs = retriever.search(query if round_i==0 else followup_query, k=10)
        context.extend(new_docs)
        # 让 LLM 判断能否回答
        can_answer = llm.judge(query, context)
        if can_answer.confident:
            return llm.generate(query, context)
        if round_i == 0:
            # 第一轮不足，生成追问查询
            followup_query = llm.generate_followup(query, context)
    # 多轮还不行，降级
    if query_is_factual(query):
        return "提供的知识库中没有足够信息回答这个问题。"
    else:
        return llm.generate_with_disclaimer(query)  # 通用知识 + 免责声明

这些兜底策略让 RAG 系统"知道自己不知道"，避免硬答导致的幻觉。一个成熟的 RAG 系统，处理"答不了"的能力和处理"答得好"的能力一样重要——用户对"诚实地说不知道"的容忍度，远高于对"自信地胡说"的容忍度。

★ 流式生成与上下文的动态管理

现代 LLM 支持流式输出（token by token），这改变了上下文管理的思路——不必等所有上下文就绪才开始生成，可以动态填充。

流式 RAG 的思路。传统 RAG 是"检索完 → 拼 prompt → 一次性生成"。流式 RAG 可以：先用第一批检索结果（top-3）开始生成，同时后台继续检索更多（top-10）和 rerank。生成过程中如果发现需要更多信息，动态追加。这样用户看到第一个 token 的延迟从"检索+rerank+生成"降到"检索 top-3 + 生成首 token"。

Speculative RAG。更激进的流式：用一个快的小 LLM 基于 top-3 快速生成草稿，同时用慢的大 LLM 基于 rerank 后的 top-10 生成。如果两者一致，用草稿（快）；如果不一致，用大 LLM 的（准）。这是借鉴 speculative decoding 的思路，能显著降延迟。

# 流式 RAG：边检索边生成
async def streaming_rag(query):
    # 1. 快速检索 top-3，立即开始生成
    top3 = await quick_retrieve(query, k=3)
    stream = llm.stream_generate(query, top3)
    first_chunk = await stream.__anext__()
    yield first_chunk   # 用户立刻看到首 token
    # 2. 后台继续 rerank + 补充检索
    full_context = await rerank_and_expand(query, k=10)
    # 3. 如果 top3 不够，用 full_context 重新生成剩余部分
    if needs_more_info(first_chunk, full_context):
        async for chunk in llm.stream_generate(query, full_context):
            yield chunk
    else:
        async for chunk in stream:
            yield chunk

上下文的动态追加。生成过程中 LLM 可能发现"需要某个具体数据"，这时可以中断生成、触发定向检索（用已生成内容作为新查询）、追加到上下文、继续生成。这是 agentic RAG 的雏形——LLM 主动决定何时检索什么。

上下文窗口的智能分配。长上下文模型（128K）不代表要把 128K 填满。智能分配：核心证据（rerank top-3）放前 20%、支持信息（top-10）放中间、背景（top-30 摘要）放后。配合 Lost in the Middle 的重排策略（前面讲过），让最重要的信息在注意力强的位置。

# 上下文窗口的智能分配
def smart_context_packing(ranked_docs, window_tokens=32000):
    # 分层：核心 / 支持 / 背景
    core = ranked_docs[:3]           # top-3，最重要
    support = ranked_docs[3:10]      # top-10，支持
    background = [summarize(d) for d in ranked_docs[10:30]]  # 摘要
    # 分配 token 预算：核心 40%、支持 40%、背景 20%
    core_budget = int(window_tokens * 0.4)
    support_budget = int(window_tokens * 0.4)
    bg_budget = window_tokens - core_budget - support_budget
    # 按预算截断 + 交错重排（核心放两端）
    packed = interleave_reorder(
        truncate(core, core_budget) + truncate(support, support_budget) + truncate(background, bg_budget)
    )
    return packed

这些动态上下文管理技术是 RAG 的前沿。它们让 RAG 从"静态流水线"变成"动态智能体"，能用更低的延迟、更精准的上下文产出更好的答案。但复杂度也高，适合追求极致体验的成熟系统，初学者先用基础的打包策略。

★ 上下文中的引用与可追溯性

除了怎么打包上下文，还要考虑怎么让 LLM 在答案中引用来源——可追溯性是 RAG 可信度的关键。

为什么引用重要。用户（尤其专业场景）要验证答案。没有引用，用户不知道答案哪来的，不敢信。有引用，用户能点开看原文，建立信任。法律/医疗/金融场景，引用几乎是硬性要求。

实现引用的方法。① 编号引用——上下文里的每块标 [1][2][3]，prompt 要求 LLM 答案里用 [1] 这样的标记引用。这是最常用的方法，简单有效。② XML 标签——每块用 <source id="1">...</source> 包裹，要求 LLM 输出 <cite>1</cite>。结构化更强，便于解析。③ 后处理匹配——不要求 LLM 引用，生成后用文本匹配找出答案句子对应哪些块。鲁棒但实现复杂。

# 编号引用的 prompt 构建
def build_prompt_with_citations(query, ranked_docs):
    context = "

".join([f"[{i+1}] {doc.text}" for i, doc in enumerate(ranked_docs)])
    prompt = f"""基于以下资料回答问题。每个论断后用 [编号] 标注来源。

{context}

问题：{query}

要求：1. 只基于上面资料回答 2. 每个事实论断后用 [编号] 引用 3. 资料不足时说明"""
    return prompt
# 解析引用
def parse_citations(answer):
    import re
    return {int(n) for n in re.findall(r'\[(\d+)\]', answer)}
# 引用覆盖率高的答案更可信，可作为质量信号

引用质量。LLM 可能乱引用（标了 [1] 但内容其实来自 [3]）。检查方法：把答案的每个引用句子和被引块做相似度匹配，相似度低说明引用错。这能自动检测幻觉引用。

引用与上下文打包的配合。打包时给每块清晰的编号/标签，LLM 才能正确引用。如果上下文被打乱或截断，引用编号会错。所以打包策略（重排、截断）要保证块的编号在 prompt 里清晰可数。

UI 层的引用展示。答案里的 [1] 要能点击跳转到原文块。这需要前端把引用编号映射回块的元数据（原文、来源文档、页码）。好的引用 UI 让 RAG 从"给答案"升级到"给可信答案"，用户体验质的飞跃。

引用是 RAG 区别于纯 LLM 的核心价值——可追溯、可验证。设计上下文打包时要同步考虑引用机制，让 RAG 的可信度从一开始就内建。

★ 本章小结

  • Lost in the Middle：LLM 对 prompt 中间位置的信息关注度低，呈 U 型曲线，导致中间的文档被忽略。
  • 策略 1 交错重排：把高质量文档放两端、次要的放中间，几乎零成本提升 5-10 个点准确率。
  • 策略 2 截断：rerank 后只取 top-3 到 top-5，少即是多，避免噪声。看分数断层决定取几个。
  • 策略 3 map-reduce：长文档场景下对每个文档单独问再合并，绕开 Lost in the Middle，成本高。
  • 窗口利用率：塞到 50-70% 是甜区，按 token 而非字数算；进阶可加上下文压缩（提信噪比）和引用标注（降幻觉、增信任）。

import numpy as np
from dataclasses import dataclass
from typing import List, Optional

@dataclass
class PerformanceMetrics:
    latency_ms: float
    throughput_ops: float
    memory_mb: float
    accuracy: float

def benchmark(fn, iterations=100):
    """通用性能基准测试"""
    import time
    t0 = time.time()
    for _ in range(iterations):
        result = fn()
    elapsed = time.time() - t0
    return PerformanceMetrics(
        latency_ms=elapsed * 1000 / iterations,
        throughput_ops=iterations / elapsed,
        memory_mb=0,  # 实际中用 psutil 获取
        accuracy=0.0  # 根据具体任务计算
    )

# 示例：对比两种方案的性能
def approach_a(data):
    """方案A：直接计算"""
    return np.softmax(data @ data.T / np.sqrt(data.shape[-1]), axis=-1)

def approach_b(data):
    """方案B：分块计算（省内存）"""
    chunk_size = 64
    results = []
    for i in range(0, data.shape[0], chunk_size):
        chunk = data[i:i+chunk_size]
        results.append(np.softmax(chunk @ data.T / np.sqrt(data.shape[-1]), axis=-1))
    return np.vstack(results)

data = np.random.randn(512, 768)
metrics_a = benchmark(lambda: approach_a(data))
metrics_b = benchmark(lambda: approach_b(data))
print(f"方案A: {metrics_a.latency_ms:.1f}ms, {metrics_a.throughput_ops:.0f} ops/s")
print(f"方案B: {metrics_b.latency_ms:.1f}ms, {metrics_b.throughput_ops:.0f} ops/s")

常见问题FAQ

Q: 如何选择合适的方案？ 先用方案A跑通验证正确性，再根据实际瓶颈（内存/延迟/精度）切换到方案B或C。永远不要过早优化。

Q: 生产环境最大的坑是什么？ 过度工程化——在需求不明确时投入大量精力做复杂架构。建议MVP先行，收集反馈后迭代。

Q: 如何评估效果？ 定义明确的成功指标（准确率/延迟/成本），用golden set做回归测试，A/B测试验证改进效果。至少100个样本才有统计显著性。
