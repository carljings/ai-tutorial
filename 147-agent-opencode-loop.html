源码级视角

循环：LLM 推理 → 解析工具调用 → 执行工具 → 把结果拼回上下文 → 再推理。直到模型不再调工具返回最终答案。会话状态与消息流转。

我们不只是"用"这些开源 Agent，而是拆开源码，看清它们的 agent loop、工具系统、权限模型是如何实现的。这种"源码级理解"让你能批判性地使用它们、为它们扩展工具、甚至构建自己的 agent。

Agent Loop 的结构

OpenCode 的 agent loop 本质是一个"推理-行动-观测"的循环。每一轮：模型基于当前上下文推理，要么返回最终答案（循环结束），要么发起工具调用；工具执行后把结果拼回上下文，进入下一轮推理。

循环伪代码

async def agent_loop(session, user_message):
    messages = session.messages + [{"role": "user", "content": user_message}]
    while True:
        # 1. 调用 LLM
        response = await llm.complete(
            model=session.model,
            messages=messages,
            tools=session.allowed_tools,  # 工具 schema 列表
        )
        messages.append(response.message)
        # 2. 判断是否结束
        if response.finish_reason == "stop":
            return response.message.content  # 最终答案
        # 3. 执行工具调用
        for tool_call in response.tool_calls:
            # 权限检查

                "role": "tool",
                "tool_call_id": tool_call.id,

事件循环的完整伪代码：从用户输入到最终答案

第一节给了 agent loop 的骨架，这里展开成包含所有工程细节的完整版。一个生产级 agent loop 要处理：上下文管理（消息历史增长怎么办）、流式输出（边生成边返回）、并行工具调用、中断恢复、token 预算控制。下面这段伪代码覆盖了这些细节。

async def production_agent_loop(session, user_message, on_token=None):
    """
    完整的 agent 事件循环，支持流式、并行工具、中断、预算控制。
    on_token: 可选的流式回调，每生成一个 token 调用一次。
    """
    # 1. 初始化本轮消息
    messages = session.messages + [{"role": "user", "content": user_message}]
    # 上下文窗口管理：消息太多时压缩历史
    if count_tokens(messages) > session.max_context:
        messages = await compress_history(messages, session.max_context)

    iteration = 0
    while iteration < session.max_iterations:   # 防死循环
        iteration += 1
        # token 预算检查：超预算则强制结束
        if session.token_spent > session.token_budget:
            return {"answer": "已达 token 预算上限，请缩小问题范围。",
                    "reason": "budget_exceeded"}

        # 2. 调用 LLM（支持流式）
        response = await llm.complete(
            model=session.model,
            messages=messages,
            tools=session.allowed_tools,
            stream=on_token is not None,
            on_token=on_token,   # 流式回调
        )
        messages.append(response.message)
        session.token_spent += response.usage.total_tokens

        # 3. 判断终止：模型不再调工具 = 最终答案
        if not response.tool_calls:
            session.messages = messages   # 持久化会话
            return {"answer": response.message.content,
                    "iterations": iteration,
                    "tokens": session.token_spent}

        # 4. 执行工具调用（支持并行）
        tool_results = await asyncio.gather(*[
            execute_single_tool(session, tc) for tc in response.tool_calls
        ], return_exceptions=True)

        # 5. 把所有工具结果拼回上下文
        for tc, result in zip(response.tool_calls, tool_results):
            if isinstance(result, Exception):
                content = f"工具执行错误: {result}"
            else:
                content = result
            messages.append({
                "role": "tool",
                "tool_call_id": tc.id,
                "name": tc.name,
                "content": content,
            })

    # 6. 超过最大迭代仍未完成
    return {"answer": "任务过于复杂，达到最大推理轮数。",
            "reason": "max_iterations", "partial": messages}

这段完整版相比第一节骨架多了五个生产级细节：① `compress_history` 在上下文超窗口时压缩历史——通常用"保留首尾、摘要中间"的策略，否则长对话会撞上下文限制；② `token_budget` 全局预算控制——防止 agent 为了一个任务烧掉过多 token，超预算强制终止；③ `stream + on_token` 流式回调——让前端能实时显示生成内容；④ `asyncio.gather` 并行执行多个工具调用——当模型一次返回多个工具调用时，并行执行能显著降延迟；⑤ `return_exceptions=True` 让单个工具失败不阻塞其他工具——失败的变成错误消息反馈给模型。这些细节是把 agent 从"能跑的 demo"变成"可靠的系统"的分水岭。

💡 要点
agent loop 的复杂度主要在"边界情况处理"而非"正常流程"。正常流程（推理→调工具→再推理）几十行就够；但要处理上下文溢出、预算超限、工具超时、用户中断、并行冲突……这些边界情况占了 80% 的代码。读开源 agent 源码时，重点看这些边界处理，那是工程功力的体现。

tool_call 的完整生命周期

工具调用不是"执行完就完了"——它有完整的生命周期：解析→权限检查→参数校验→执行→超时控制→结果格式化→审计记录。理解这个生命周期，才能在工具出问题时精准定位是哪一环坏了。下面把每一环拆开讲。

# tool_call 的完整生命周期处理
import asyncio
import time
from datetime import datetime

async def execute_single_tool(session, tool_call):
    """单个工具调用的完整生命周期。"""
    start_time = time.time()
    audit = {"tool": tool_call.name, "args": tool_call.args,
             "timestamp": datetime.now().isoformat()}

    # 阶段 1：解析（LLM 输出可能格式错）
    try:
        args = parse_tool_args(tool_call.args)
    except ParseError as e:
        audit["status"] = "parse_failed"
        session.audit_log.append(audit)
        return {"error": f"参数解析失败: {e}"}

    # 阶段 2：权限检查（安全第一）
    if not session.can_run(tool_call.name, args):
        audit["status"] = "denied"
        session.audit_log.append(audit)
        return {"error": f"权限不足：无权执行 {tool_call.name}"}

    # 阶段 3：参数校验（schema 层面）
    tool = session.tools.get(tool_call.name)
    errors = validate_args(args, tool.schema)
    if errors:
        audit["status"] = "invalid_args"
        session.audit_log.append(audit)
        return {"error": f"参数校验失败: {errors}"}

    # 阶段 4：执行（带超时和重试）
    try:
        result = await asyncio.wait_for(
            tool.run(args),
            timeout=session.tool_timeout,   # 单工具超时
        )
    except asyncio.TimeoutError:
        audit["status"] = "timeout"
        session.audit_log.append(audit)
        return {"error": f"工具执行超时（>{session.tool_timeout}s）"}
    except Exception as e:
        audit["status"] = "error"
        audit["error"] = str(e)
        session.audit_log.append(audit)
        return {"error": f"执行错误: {e}"}

    # 阶段 5：结果格式化（截断过长输出）
    formatted = format_result(result, max_chars=2000)

    # 阶段 6：审计记录
    audit["status"] = "success"
    audit["duration_ms"] = int((time.time() - start_time) * 1000)
    audit["result_size"] = len(str(result))
    session.audit_log.append(audit)

    return formatted

# 每个工具调用都留下一串审计足迹：
# {tool: "write_file", status: "success", duration_ms: 12, ...}
# {tool: "run_command", status: "denied", ...}   ← 被权限拦截
# {tool: "search", status: "timeout", ...}        ← 超时

这个生命周期里最容易被忽视的是"审计记录"（阶段 6）。生产 agent 必须记录每次工具调用的工具名、参数、状态、耗时、结果大小——这不是可选的日志，是出问题时的唯一追溯手段。当用户报告"agent 删了我的文件"，没有审计日志你就无法查证是哪个工具调用、传了什么参数、是否被授权。`asyncio.wait_for` 的超时控制也很关键——没有它，一个卡住的工具调用会让整个 agent loop 永久阻塞。设一个合理的 tool_timeout（如 30 秒）能保证单个工具故障不拖垮整个系统。

生命周期阶段失败处理典型问题
解析返回错误给 LLMLLM 输出非法 JSON
权限检查拒绝并记录越权访问
参数校验返回校验错误类型不对/缺必填
执行超时/异常捕获工具卡死或崩溃
结果格式化截断过长输出输出撑爆上下文
审计记录始终记录事后无法追溯

中断与恢复：保存检查点的工程实现

第一节的 loop 没有"中断后能恢复"的能力——一旦进程重启或网络断开，进行到一半的对话就丢了。生产 agent 需要 checkpoint 机制：每隔几轮或每次工具调用后，把当前状态（messages、iteration、token_spent）持久化。这样即使中断，也能从最近的 checkpoint 恢复。下面是简化的实现。

# 中断恢复：checkpoint 机制
import json

class CheckpointManager:
    def __init__(self, storage_path):
        self.path = storage_path

    async def save(self, session_id, state):
        """保存检查点：messages + 进度 + token 统计。"""
        checkpoint = {
            "session_id": session_id,
            "messages": state["messages"],
            "iteration": state["iteration"],
            "token_spent": state["token_spent"],
            "timestamp": time.time(),
        }
        # 原子写入：先写临时文件再 rename，防写到一半崩溃
        tmp = f"{self.path}/{session_id}.tmp"
        final = f"{self.path}/{session_id}.json"
        with open(tmp, 'w') as f:
            json.dump(checkpoint, f)
        os.rename(tmp, final)   # 原子操作

    async def load(self, session_id):
        """加载最近检查点。"""
        try:
            with open(f"{self.path}/{session_id}.json") as f:
                return json.load(f)
        except FileNotFoundError:
            return None

# 集成到 agent loop：每轮工具调用后存检查点
async def resumable_agent_loop(session, user_message, checkpoint_mgr):
    checkpoint = await checkpoint_mgr.load(session.id)
    if checkpoint:
        # 恢复：从检查点继续，而非从头开始
        messages = checkpoint["messages"]
        iteration = checkpoint["iteration"]
        print(f"从检查点恢复（第 {iteration} 轮）")
    else:
        messages = session.messages + [{"role": "user", "content": user_message}]
        iteration = 0

    while iteration < session.max_iterations:
        iteration += 1
        response = await llm.complete(model=session.model, messages=messages,
                                       tools=session.allowed_tools)
        messages.append(response.message)
        if not response.tool_calls:
            return response.message.content

        for tc in response.tool_calls:
            result = await execute_single_tool(session, tc)
            messages.append({"role": "tool", "tool_call_id": tc.id,
                             "content": result})
        # 关键：每轮工具调用后存检查点
        await checkpoint_mgr.save(session.id, {
            "messages": messages, "iteration": iteration,
            "token_spent": session.token_spent,
        })

checkpoint 机制的精髓是"原子写入"（`os.rename`）——先写临时文件，写完再 rename 成正式文件。这样即使写检查点时进程崩溃，也只留下一个不完整的临时文件，正式文件仍是上一个完整版本，不会出现"检查点写了一半"的损坏状态。恢复时从检查点继续，跳过已完成的工具调用，省下重新推理和调工具的成本。代价是每次存检查点有 I/O 开销——所以频率要权衡：存太频繁影响性能，存太少恢复时丢的多。折中是"每轮工具调用后存一次"，这是性价比最高的策略。

⚠️ 落地提醒
checkpoint 里存的 messages 可能包含敏感信息（用户数据、API key 误暴露等）。生产环境存检查点前要做敏感字段脱敏，存储位置要加密访问。另外，检查点要设 TTL（如 24 小时后自动清理），否则会无限堆积占满磁盘。
