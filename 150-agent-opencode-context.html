OpenCode 上下文管理
压缩、淘汰与文件树注入

Agent 跑久了必然遇到一个尴尬：对话历史越来越长，读过的文件越堆越多，很快就把模型的 context window 塞满，要么报错要么费用飙升。OpenCode 把上下文当成一种「需要主动管理的资源」，内置了 token 计数、消息压缩、文件树注入三套机制。本章我们看看一个聪明的 Agent 是怎么在有限的窗口里腾挪转圜的。

现代模型的 context window 已经从 GPT-3 的 2K 涨到了 128K 甚至 1M，但「标称容量」和「实际可用容量」是两回事。原因有三：第一，输入越长，推理延迟越高，token 计费也线性增长；第二，研究表明模型对超长上下文存在「lost in the middle」现象——中间位置的信息容易被忽略；第三，工具调用的输入输出本身就占大量 token，一次 read_file 几百行代码就可能吃掉几千 token。

所以 OpenCode 的策略不是「尽量塞满」，而是 「主动留白」：始终给当前轮的输出预留空间，主动淘汰过时消息，用摘要替代历史细节。它把 context 管理分成了三层：实时 token 计数、消息压缩、文件树注入。我们逐个看。

★ 二、实时 token 计数：知道还剩多少弹药

所有压缩决策的前提是「知道现在用了多少 token」。OpenCode 不是发完请求才知道超了——它在每次拼装请求前先做一次本地 token 计数，超限就触发压缩。计数用的是一个本地 tokenizer（基于 tiktok 的 wasm 版本或类似实现）。

// packages/opencode-core/src/context/counter.ts（简化）
import { encode } from "gpt-tokenizer";  // 纯 JS 的 BPE tokenizer

export function countMessageTokens(msg: Message): number {
  // 每条消息有固定开销（角色标记、分隔符），约 4 token
  const overhead = 4;
  let body = msg.role + ":";
  if (typeof msg.content === "string") {
    body += msg.content;
  } else {
    // 多模态消息：文本累加，图片按占位估算
    for (const part of msg.content) {
      if (part.type === "text") body += part.text;
      else body += " [image:~750 tokens] ";  // 图片粗估
    }
  }
  return overhead + encode(body).length;
}

export function countSessionTokens(messages: Message[]): number {
  return messages.reduce((sum, m) => sum + countMessageTokens(m), 0);
}

注意两个细节。第一是图片的粗估：OpenCode 把每张图按约 750 token 估算（这是 OpenAI vision 模型的典型开销），因为图片的精确 token 数要等服务器算，本地拿不到。第二是每条消息的固定开销：不同模型对消息格式的编码不同，OpenCode 用一个保守的 4 token overhead 作为通用估算，可能在某些模型上略偏多，但宁可高估也别低估——低估会导致压缩不及时而请求失败。

★ 三、消息压缩：当历史变成负担

当 token 计数接近阈值（通常是模型 context 的 70-80%，留出输出空间）时，OpenCode 触发压缩。压缩的核心思想是：把早期的多轮对话 + 工具调用结果，合并成一条摘要消息。

// packages/opencode-core/src/context/compress.ts（简化）
export async function compressIfNeeded(
  messages: Message[],
  maxTokens: number,
  model: ModelConfig
): Promise<Message[]> {
  const RESERVED_FOR_OUTPUT = 4096;
  const budget = maxTokens - RESERVED_FOR_OUTPUT;

  let current = [...messages];
  while (countSessionTokens(current) > budget && current.length > 4) {
    // 保留最近 4 条（最近的上下文最重要），压缩更早的
    const toCompress = current.slice(0, current.length - 4);
    const recent = current.slice(current.length - 4);

    const summary = await summarizeWithLLM(toCompress, model);
    current = [
      { role: "system", content: `[早期对话摘要]\n${summary}` },
      ...recent,
    ];
  }
  return current;
}

async function summarizeWithLLM(oldMessages: Message[], model: ModelConfig): Promise<string> {
  // 用同一个模型（或更便宜的模型）做摘要
  const resp = await callLLM({
    model,
    messages: [
      { role: "system", content: "请把以下对话和工具调用历史压缩成一段简洁摘要，保留：用户目标、已做的关键决定、已修改的文件、尚未解决的问题。丢弃冗长的工具输出原文。" },
      { role: "user", content: JSON.stringify(oldMessages) },
    ],
    // 用低温度确保摘要稳定
    temperature: 0.2,
  });
  return resp;
}

这段代码揭示了一个权衡：压缩本身要花一次 LLM 调用，是有成本的。所以 OpenCode 不会频繁压缩，而是「攒到快超限了再压一次」。同时保留最近 4 条原始消息，因为最新的上下文（用户刚说的话、刚执行的工具）对当前轮的决策影响最大，不能被摘要稀释。

还有一个被忽略的细节：工具的返回值往往是压缩的大头（一次 execute 的 stdout 可能上千行）。OpenCode 在压缩前会先对单条超长的工具结果做「就地截断」——保留头尾各 100 行，中间用 [... 省略 N 行 ...] 替代。这往往能在不丢失关键信息（开头通常是错误摘要，结尾是最终状态）的前提下，把单条消息从 5000 token 砍到 500。

★ 四、文件树注入：给 Agent 一张「地图」

这是 OpenCode 上下文管理里最有意思的设计之一。在每轮对话开始时（或首次引用某个目录时），OpenCode 会自动把当前工作目录的文件树结构作为一条 system 消息注入。这张「地图」让 LLM 不必先调 list_dir 才知道项目长什么样——它能直接基于树结构判断「这个功能应该在 src/auth/ 下」。

// 生成文件树摘要（简化）
export async function buildFileTree(cwd: string, maxDepth = 3): Promise<string> {
  const SKIP = new Set(["node_modules", ".git", "dist", "build", ".next"]);
  const lines: string[] = [];

  async function walk(dir: string, prefix: string, depth: number) {
    if (depth > maxDepth) return;
    const entries = await Array.fromAsync(Bun.Glob("*").scan(dir));
    entries.sort();
    for (const name of entries) {
      if (SKIP.has(name)) {
        lines.push(`${prefix}${name}/  [skipped]`);
        continue;
      }
      const isDir = await isDirectory(join(dir, name));
      lines.push(`${prefix}${name}${isDir ? "/" : ""}`);
      if (isDir) {
        await walk(join(dir, name), prefix + "  ", depth + 1);
      }
    }
  }

  await walk(cwd, "", 0);
  // 限制总行数，避免巨型仓库撑爆
  const capped = lines.slice(0, 500).join("\n");
  return capped;
}

// 注入到 system prompt
const tree = await buildFileTree(session.cwd);
const systemWithTree = `${baseSystemPrompt}

## 当前项目结构
${tree}`;

这里有几个工程细节。第一，跳过 node_modules 等大目录——这些目录文件多但对理解项目结构毫无帮助，注入它们纯粹浪费 token。第二，限制最大深度——3 层通常够看清楚一个中型项目的骨架，再深就是细节，应该按需通过 read_file 获取。第三，总行数封顶——对超大仓库（如 monorepo），即使深度有限，文件数也可能爆炸，500 行的硬上限是最后的保险。

★ 五、消息淘汰的优先级：什么该留，什么该扔

当压缩还不够时（比如一次工具返回了巨量输出），OpenCode 会按优先级主动淘汰消息。优先级从低到高大致是：

  1. 工具调用的完整输出（最容易膨胀，优先截断或替换为摘要）。
  2. 早期的来回对话（已被摘要覆盖的部分）。
  3. read_file 的文件内容（如果文件没改，可以稍后重新读）。
  4. 最近 2-3 轮的用户指令和 assistant 回复（当前任务的核心，几乎不淘汰）。
  5. system prompt + 文件树（每轮都保留，是 Agent 的「世界观」）。

这种优先级不是写死的常量，而是隐含在压缩算法的实现里——它通过「保留最近 N 条 + 摘要早期 + 截断超长工具输出」的组合，自然实现了上述优先级。理解这个优先级有助于你诊断「为什么 Agent 忘了我之前说的话」——通常是因为那段对话被压缩进了摘要，而摘要丢失了细节。

★ 六、压缩的成本与质量权衡

压缩不是免费的。除了那次摘要用的 LLM 调用费用，还有「信息损失」的隐性成本。OpenCode 在源码里暴露了几个可调旋钮，让用户在质量和成本间权衡：

  • 压缩阈值：设低一点（如 50%），更早压缩、单次请求更便宜，但摘要次数多；设高一点（如 85%），单次请求贵但摘要次数少。
  • 摘要用的模型：可以用更便宜的小模型做摘要（如 haiku 级别），主对话用大模型。OpenCode 允许为摘要任务单独配置 model。
  • 保留的最近消息数：保留越多，上下文越完整但压缩空间越小。

实际使用中，一个常见的调优方向是：对长会话把阈值调到 60%，对短会话调到 85%。因为长会话本来就是要持续工作的，频繁压缩可以保持每轮请求的轻量；短会话则希望尽量保留原始信息，避免摘要失真。

D1 深度剖析：上下文收集的优先级策略

本章讲了 OpenCode 的上下文收集。这一节我们深入"有限上下文窗口下该塞什么"的核心难题，并用代码展示一个基于优先级的上下文打包器。

// 基于优先级的上下文打包器
interface ContextItem {
  content: string;
  priority: number;        // 越大越重要
  source: string;          // 来源标记
  truncateable?: boolean;  // 是否可截断
}

function packContext(items: ContextItem[], budget: number): string {
  // 1. 按优先级降序排序
  const sorted = [...items].sort((a, b) => b.priority - a.priority);
  const packed: string[] = [];
  let used = 0;
  for (const item of sorted) {
    const tokens = estimateTokens(item.content);
    if (used + tokens <= budget) {
      packed.push(item.content);
      used += tokens;
    } else if (item.truncateable) {
      // 可截断项：保留头部填满剩余预算
      const remain = budget - used;
      packed.push(item.content.slice(0, remain * 4) + "\n[...已截断]");
      used = budget;
      break;
    }
    // 不可截断且超预算：直接丢弃
  }
  return packed.join("\n\n");
}

// 优先级约定（数字越大越优先保留）：
// 100 - 用户当前指令（最高，绝不丢）
// 90 - 系统提示词
// 80 - 关键文件内容（用户明确 @ 的）
// 60 - 相关代码片段（grep/AST 命中）
// 40 - 对话历史（近期优先）
// 20 - 工具定义
// 10 - 环境信息（cwd/git branch）

这套优先级体系的核心洞察是：并非所有上下文同等重要。用户当前指令和系统提示词是"必须完整保留"的，而历史对话和工具结果可以截断或丢弃。按优先级打包能最大化有限窗口的信息密度——这正是 RAG 和 Agent 系统的通用优化方向。

上下文类型优先级可截断典型大小丢失后果
用户当前指令100否小致命（任务丢失）
系统提示词90否中严重（行为漂移）
关键文件80是（尾部）大中（信息缺失）
对话历史40是随轮次增长低（可重建）
环境信息10是小极低

D2 生产避坑：上下文管理的常见误区

误区 1：把整个代码库塞进上下文

初学者常想"把项目所有文件都给模型看"。但即便窗口够大，大量无关代码会稀释信号，LLM 反而抓不住重点。正确做法是按需检索：先 grep/AST 找相关文件，只把命中的片段给模型。少即是多。

误区 2：历史对话无限累积

多轮对话中，早期消息往往已过时（如已被修改的代码讨论）。无限保留会让窗口被陈旧信息占满。滑动窗口 + 摘要是标准解法：保留最近 N 轮原文，更早的压缩成摘要。

误区 3：token 估算不准导致截断

用字符数 ÷4 估 token 在中文/代码场景误差很大（中文约 1 字 ≈ 1.5 token）。截断点算错会让关键信息被误删。生产环境应用真实 tokenizer 估算，或留 20% 安全余量。

★ 本章小结

上下文管理是 Agent 能长时间运行而不崩溃的关键。OpenCode 的做法可以总结为一套「主动管理」哲学。要点回顾：

  • 标称容量 ≠ 可用容量：要考虑延迟、成本、lost-in-the-middle 效应，主动留白而非尽量塞满。
  • 本地 token 计数：用纯 JS tokenizer 在拼装请求前预估，超限就触发压缩，避免请求失败。
  • 消息压缩：把早期对话 + 工具输出摘要成一条 system 消息，保留最近 4 条原始上下文，单条超长工具结果就地截断头尾。
  • 文件树注入：把项目结构作为 system prompt 的一部分，跳过 node_modules、限制深度和总行数，让 LLM 有「地图」不用每次 list_dir。
  • 淘汰优先级：工具输出 > 早期对话 > 文件内容 > 最近对话 > system prompt，由压缩算法自然实现。
  • 可调旋钮：压缩阈值、摘要模型、保留条数都是可配置的，长会话和短会话各有最优策略。

下一章（151 章）我们转向另一个扩展上下文能力的关键机制——MCP 集成。如果说文件树是给 Agent 一张静态地图，MCP 就是给它一把打开外部世界的万能钥匙。

★ 深度面试与实践指南

Q1: 本地 vs 云端 上下文管理 的实际性能差距有多大？

以 Continue 的 @codebase 检索为基准：本地 all-MiniLM-L6-v2（CPU）约 500 docs/s，OpenAI text-上下文-3-small 约 3000 docs/s。但本地零延迟（无网络往返），云端有 50-200ms 延迟。对于<1000 个文档的小项目，本地更优；对于大型代码库（>10K 文件），云端更快。

Q2: 如何优化 上下文管理 索引的更新策略？

三种策略：(1)全量重建——每次删除旧索引重建，简单但慢，适合小项目。(2)增量更新——只对新修改的文件重新生成向量并更新索引，适合 Git 跟踪的项目。(3)懒加载——首次查询时才为相关文件生成向量并缓存，适合超大代码库。

# Continue 风格的增量索引更新
import hashlib, os, json

class IncrementalIndex:
    def __init__(self, index_path, embed_fn):
        self.index_path = index_path
        self.embed_fn = embed_fn
        self.file_hashes = self._load_hashes()
    
    def update(self, filepaths):
        updated = []
        for fp in filepaths:
            content = open(fp).read()
            h = hashlib.md5(content.encode()).hexdigest()
            if self.file_hashes.get(fp) != h:  # 文件有变更
                vec = self.embed_fn(content)
                self.file_hashes[fp] = h
                updated.append((fp, vec))
        self._save_hashes()
        return updated  # 返回需要更新到向量库的文件
    
    def _load_hashes(self):
        try: return json.load(open(self.index_path))
        except: return {}
    def _save_hashes(self):
        json.dump(self.file_hashes, open(self.index_path, 'w'))

Q4: 如何降低 上下文管理 的存储成本？

三个策略：(1)用 PQ（乘积量化）把 1024 维浮点向量压缩到 32 字节，存储降低 128 倍。(2)只存 Top-K 最相关的向量到内存，其余溢出到磁盘。(3)用 Matryoshka 上下文管理（如 OpenAI-3 支持）按需截断维度——存储时用 256 维，查询时用 1024 维。

实践经验：在 Continue 中配置 上下文sProvider: "transformers.js" 跑本地 MiniLM 是零成本起步方案。代码库超过 5000 文件时再迁移到 BGE + Qdrant 组合，兼顾性能和成本。

补充：性能调优清单

最后附上一份生产环境的性能调优清单，帮助你快速定位和解决常见问题：

  • 延迟过高：检查模型API的P99延迟，考虑启用流式输出降低首token延迟。目标：首token < 500ms。
  • 准确率不够：增加few-shot示例，调整chunk_size（建议512-1024），或换更大的embedding模型。
  • 成本超标：启用语义缓存（对相同或相似query复用结果），设置daily token budget，简单任务路由到flash模型。
  • 索引膨胀：定期清理过期向量，用PQ压缩存储，设置最大索引大小阈值触发自动清理。

记住：先正确，再快速，最后便宜。不要过早优化，先用最简单的方案跑通，再根据实际瓶颈针对性优化。
