Roo Code 多模式
architect / code / debug / ask 的场景化切换

Roo Code 是 Cline 的一个 fork（原叫 Roo Cline），它在 Cline 的基础上发展出了一套独特的 多模式系统：architect（只规划不改）、code（完整开发）、debug（专注排查 bug）、ask（纯问答）。用户可以根据当前任务切换模式，每种模式有定制的 system prompt 和工具集。本章我们看这套模式系统的设计，以及它相比 Cline 的 plan/act 双模式的进化。

阅读约 17 分钟 难度 ★★★★

★ 一、从双模式到四模式：场景化的深化

Cline 的 plan/act 双模式（参见 168 章）解决了「先想后做」的基本分离。但实际使用中，用户会发现「plan」这个模式太宽泛——有时候你只是想让 AI 回答一个关于代码的问题（不需要规划任何改动），有时候你想让它专注排查一个 bug（需要的是诊断而非规划）。把这些都塞进「plan」模式，system prompt 会变得臃肿而泛泛。

Roo Code 的解法是 进一步场景化，把 plan 拆成 architect（架构规划）、debug（调试）、ask（问答）三个独立模式，加上完整的 code 模式，共四种。每种模式有：

  • 独立的 system prompt，定义角色和行为准则。
  • 定制的工具集，只给该模式需要的工具。
  • 专属的 UX 提示，让用户清楚当前在什么模式下。

这种「场景即模式」的设计哲学，让 Agent 在每个场景下都表现得「专精」，而不是「什么都行但什么都不精」。

★ 二、四种模式的角色定义

我们逐一拆解四种模式的定位和 system prompt 设计思路。

// Roo Code 的模式定义（简化）
export type Mode = "architect" | "code" | "debug" | "ask";

export const MODE_CONFIGS: Record<Mode, ModeConfig> = {
  architect: {
    name: "architect",
    label: "架构师",
    description: "分析需求、设计方案、规划技术实现",
    allowedTools: ["read_file", "list_dir", "search_codebase", "list_code_definition_names"],
    systemPrompt: `你是一位资深软件架构师。

你的职责：
- 理解用户的需求和技术约束
- 阅读现有代码，分析架构和依赖关系
- 设计技术方案，考虑可扩展性、可维护性、性能
- 输出清晰的实施计划，列出要创建/修改的文件和步骤

你不写代码。你的输出是方案和计划，由用户切换到 code 模式执行。
即使被催促，也只规划不实现。`,
  },

  code: {
    name: "code",
    label: "开发者",
    description: "完整开发：读写文件、执行命令、迭代修复",
    allowedTools: ALL_TOOLS,  // 包括 write_file, execute 等
    systemPrompt: `你是一位高效的全栈开发者。

你的工作方式：
- 按照（可能来自 architect 模式的）计划执行
- 每次修改后运行相关测试验证
- 遇到失败时分析原因并修复，不要绕过
- 保持改动最小化，不引入不必要的重构
- 完成后总结你做了什么`,
  },

  debug: {
    name: "debug",
    label: "调试专家",
    description: "诊断 bug、分析根因、提出修复",
    allowedTools: ["read_file", "execute", "search_codebase"],  // 能跑命令但谨慎写
    systemPrompt: `你是一位专注的调试专家。

你的诊断流程：
1. 复现：先尝试复现 bug（运行测试或最小复现脚本）
2. 定位：根据错误信息和堆栈，找到相关代码
3. 假设：提出可能的根因假设
4. 验证：通过加日志、读代码、跑实验验证假设
5. 修复：确认根因后再改代码，不要盲目试错

关键原则：
- 不要在没理解根因前就改代码（那是在猜）
- 改完后必须验证 bug 确实修复了`,
  },

  ask: {
    name: "ask",
    label: "问答助手",
    description: "回答关于代码库的问题，不修改任何东西",
    allowedTools: ["read_file", "search_codebase"],  // 纯只读
    systemPrompt: `你是一位熟悉这个代码库的同事。

用户会问你关于代码的问题。你的任务是：
- 准确回答，基于实际代码而非猜测
- 引用具体的文件和行号作为依据
- 如果不确定，明确说「我不确定」而不是编造
- 不修改任何文件，即使被要求（建议用户切换到 code 模式）`,
  },
};

注意 debug 模式的一个微妙设计：它允许 execute 但限制 write_file。这是因为在调试中，运行测试、跑复现脚本、加临时日志都是必要的（要 execute），但在根因确认前不应该改业务代码（不给 write_file）。这种「按诊断流程阶段精细配置工具」的设计，比 Cline 的 plan（完全只读）更贴合调试场景。

★ 三、模式切换的 UX 与状态保持

用户在四种模式间切换是高频操作，UX 必须顺滑。Roo Code 的设计是：

// 模式切换（简化）
export class ModeManager {
  private currentMode: Mode = "code";
  private modeHistory: Mode[] = [];

  switchTo(mode: Mode) {
    if (mode === this.currentMode) return;
    this.modeHistory.push(this.currentMode);
    this.currentMode = mode;

    // 关键：切换模式不重置对话历史
    // 用户可以在 architect 模式规划，切到 code 模式时
    // 之前的对话上下文仍然可见
    this.notifyUI({
      type: "mode_changed",
      mode,
      config: MODE_CONFIGS[mode],
    });

    // 在对话里插入一条系统消息，标记模式切换
    this.messages.push({
      role: "system",
      content: `[模式切换：${this.modeHistory.slice(-1)[0]} → ${mode}]`,
    });
  }

  // 快捷回到上一个模式
  switchToPrevious() {
    if (this.modeHistory.length > 0) {
      const prev = this.modeHistory.pop()!;
      this.currentMode = prev;
      this.notifyUI({ type: "mode_changed", mode: prev });
    }
  }
}

这里最关键的设计是 切换模式不重置对话。典型的工作流是：用户在 ask 模式问「这个 bug 在哪」→ AI 分析后给出假设 → 用户切到 debug 模式 → AI 继续基于刚才的分析深入诊断 → 用户切到 code 模式 → AI 执行修复。整个过程中对话是连续的，AI 在每个模式里都能看到之前的上下文。这种「模式特化 + 上下文连续」的组合，让多模式成为流畅的工作流而非割裂的工具。

★ 四、工具守卫：模式约束的硬执行

和 Cline 的 plan/act 一样（参见 168 章），Roo Code 的模式约束是 硬执行 的——不是靠 system prompt 说「请不要改文件」，而是工具列表里根本没给 write_file，模型想用也用不了。我们看这个守卫怎么在循环里实现：

// 工具守卫（简化）
export class ToolGuard {
  constructor(private mode: Mode) {}

  checkToolAllowed(toolName: string): { allowed: boolean; reason?: string } {
    const config = MODE_CONFIGS[this.mode];
    if (config.allowedTools.includes(toolName)) {
      return { allowed: true };
    }
    return {
      allowed: false,
      reason: this.getRejectionMessage(toolName),
    };
  }

  private getRejectionMessage(toolName: string): string {
    const messages: Record<string, string> = {
      write_file: `当前是 ${this.mode} 模式，不允许修改文件。` +
                  `如需修改，请切换到 code 模式。`,
      execute: `当前是 ${this.mode} 模式，不允许执行命令。` +
               `如需执行，请切换到 code 或 debug 模式。`,
    };
    return messages[toolName] || `工具 ${toolName} 在当前模式下不可用。`;
  }
}

// 在 Agent 循环里
async function executeToolCall(call: ToolCall, guard: ToolGuard) {
  const check = guard.checkToolAllowed(call.name);
  if (!check.allowed) {
    // 把拒绝原因作为 tool_result 返回，让 LLM 知道为什么不行
    return {
      content: `[权限拒绝] ${check.reason}`,
      isError: true,
    };
  }
  return await executeActualTool(call);
}

注意拒绝时的 引导性消息——它不只是说「不行」，还告诉 LLM「想干这个请切换到哪个模式」。这让 LLM 在被拒后能给出有用的建议（「我需要切到 code 模式才能帮你改这个文件，要切换吗？」），而不是干巴巴地报错。这种「拒绝即引导」的设计，提升了用户体验。

★ 五、自动模式建议

用户有时候不知道该用哪个模式。Roo Code 有一个智能建议功能——根据用户的输入，推荐合适的模式：

// 自动模式建议（简化，实际可用规则或小模型）
function suggestMode(userInput: string, currentMode: Mode): Mode | null {
  const input = userInput.toLowerCase();

  // 明确的模式信号
  if (/^(设计|方案|架构|plan|design)/.test(input)) return "architect";
  if (/^(修复|bug|报错|debug|为什么.*错)/.test(input)) return "debug";
  if (/^(这是什么|解释|为什么|how|what|why)/.test(input)) return "ask";
  if (/^(实现|添加|创建|写一个|refactor)/.test(input)) return "code";

  // 当前模式不适配的信号
  if (currentMode === "ask" && /改|修改|实现/.test(input)) {
    return "code";  // 想改文件但还在 ask，建议 code
  }
  if (currentMode === "code" && /这是什么|解释一下/.test(input)) {
    return "ask";   // 想问问题但还在 code，建议 ask
  }

  return null;  // 不建议切换
}

这种基于关键词的简单规则，就能覆盖大部分场景。更复杂的实现可以用一个小模型分类，但规则的简单和可预测性对 UX 更重要——用户能理解「我说『修复』它就建议 debug」的逻辑，而模型分类是个黑盒。

★ 六、用户自定义模式

Roo Code 还支持 用户自定义模式——用户可以创建自己的模式，定义 system prompt 和工具集。这让团队可以为特定工作流定制模式，比如：

{
  "customModes": [
    {
      "name": "reviewer",
      "label": "代码审查员",
      "allowedTools": ["read_file", "search_codebase"],
      "systemPrompt": "你是严格的代码审查员。检查代码的风格、潜在 bug、安全漏洞、性能问题。给出具体的改进建议，引用行号。不改代码。"
    },
    {
      "name": "doc-writer",
      "label": "文档撰写",
      "allowedTools": ["read_file", "write_file"],
      "systemPrompt": "你专门为代码写文档。读取代码后生成清晰的注释、README、API 文档。只改文档文件（*.md），不改业务代码。",
      "filePatternRestriction": "*.md"
    }
  ]
}

注意 doc-writer 模式的 filePatternRestriction——它限制了 write_file 只能写 .md 文件。这是比纯工具列表更细粒度的控制。这种「自定义模式 + 文件模式限制」的组合，让团队能为各种专项工作流建立安全护栏。

★ 七、与 Cline 双模式的对比

把 Roo Code 的四模式和 Cline 的双模式放在一起，能看出场景化程度的差异：

  • 精细度：Cline 的 plan 涵盖了 architect + debug + ask 三个场景，system prompt 是折中；Roo Code 每个场景独立 prompt，更专精。
  • 工具配置：Cline 的 plan 完全只读；Roo Code 的 debug 允许 execute（运行测试）但不允许 write，更贴合调试需求。
  • UX 复杂度：四模式比双模式选择成本高，但自动建议功能缓解了这个问题。
  • 可扩展性：Roo Code 支持自定义模式，Cline 是固定的 plan/act。

总体来说，Roo Code 的多模式是对 Cline 双模式的「垂直深化」，适合对场景精细化有要求的团队。Cline 的双模式更简洁，适合不想做太多选择的用户。两种设计没有绝对优劣，是不同的产品哲学。

★ 本章小结

Roo Code 的多模式系统是「场景化 Agent」的典范。要点回顾：

  • 四模式：architect（规划不改）、code（完整开发）、debug（诊断谨慎写）、ask（纯问答），每种有独立 prompt 和工具集。
  • debug 的精细工具：允许 execute 跑测试但不给 write_file，贴合「先诊断后修复」的流程。
  • 模式切换不重置对话：上下文连续，多模式是流畅工作流而非割裂工具。
  • 工具守卫硬执行：不在 prompt 里说「别改」，而是工具列表里没给，拒绝时给引导性消息。
  • 自动模式建议：基于关键词规则推荐模式，简单可预测优于黑盒模型。
  • 自定义模式：用户创建专属模式，支持 filePatternRestriction 细粒度文件控制。
  • 对比 Cline：四模式更精细更可扩展，双模式更简洁，是垂直深化 vs 轻量通用两种哲学。
