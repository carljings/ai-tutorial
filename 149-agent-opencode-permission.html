OpenCode 权限模型
allow / ask / deny 三态下的工具级管控

给 Agent 一个 execute 工具，它就有能力删掉你整个仓库。OpenCode 深知这种风险，所以构建了一套细粒度的权限系统：不是简单的「允许/禁止」二选一，而是 allow（直接放行）、ask（每次问用户）、deny（一律拒绝）三态，且可以按工具 + 参数粒度配置。本章我们拆解这套权限状态机的实现，看清一个 tool_call 在执行前要过几道关卡。

设想一个朴素的权限模型：每个工具有一个 enabled 开关，true 就允许调用，false 就禁止。这听起来够用，但实际用起来会面临两难。比如 execute 工具——你既想让 Agent 自动跑 npm test（否则每次都要手动确认，体验极差），又绝对不想让它自动执行 rm -rf /。在二态模型下，你要么把 execute 全局打开（危险），要么全局关掉（Agent 就残废了）。

OpenCode 的解法是引入第三个状态 ask：对于无法预判安全性的工具调用，先暂停，把「要执行什么」展示给用户，由用户现场决定。这样三类工具就有了清晰的归属：

  • allow：只读、无害的工具，如 read_file、list_dir。一旦配置为 allow，后续不再打断用户。
  • ask：有副作用但可控的，如 write_file、execute。每次调用前弹确认。
  • deny：明确禁止的，如读取 .env、执行 git push --force。直接拒绝，连问都不问。

这三态还可以组合规则细化，比如「execute 工具默认 ask，但如果命令以 npm test 开头则 allow」。这就是 OpenCode 权限模型的精髓：不是对工具一刀切，而是对「工具 × 参数」的组合做决策。

★ 二、权限状态机的数据结构

OpenCode 的权限决策不是一堆 if/else，而是一个可配置的规则列表 + 一个决策函数。每条规则是一个 matcher，描述「在什么条件下应该 allow / ask / deny」。决策时按顺序匹配，第一个命中的规则决定结果。

// packages/opencode-core/src/permission/types.ts（简化）
export type PermissionDecision = "allow" | "ask" | "deny";

export interface PermissionRule {
  tool: string;                       // 工具名，支持通配符 "*" 
  decision: PermissionDecision;
  // 可选的参数匹配器（不写则匹配该工具的所有调用）
  match?: {
    // 对 read_file/write_file：匹配文件路径（glob）
    path?: string;                    // 如 "**/*.env" 或 "src/**"
    // 对 execute：匹配命令（正则或前缀）
    command?: string;                 // 如 "npm test*" 或 "git *"
  };
}

export interface PermissionConfig {
  rules: PermissionRule[];            // 按顺序匹配
  defaultDecision: PermissionDecision; // 兜底，默认 "ask"
}

这套数据结构最精妙的地方在于 match 字段的多态：对于文件类工具，match.path 用 glob 匹配路径；对于命令类工具，match.command 用模式匹配命令字符串。同一个规则结构适配了不同工具的语义，让配置保持简洁。

★ 三、决策引擎：从规则到结论

有了规则列表，决策就是一个遍历匹配的过程。我们看核心的 decide 函数：

// packages/opencode-core/src/permission/decide.ts（简化）
import { minimatch } from "minimatch";

export function decide(
  config: PermissionConfig,
  toolCall: { tool: string; params: any }
): PermissionDecision {
  for (const rule of config.rules) {
    // 1. 工具名匹配（支持 "*" 通配）
    if (rule.tool !== "*" && rule.tool !== toolCall.tool) continue;

    // 2. 参数匹配
    if (rule.match) {
      if (rule.match.path) {
        const target = toolCall.params.filePath ?? toolCall.params.path ?? "";
        if (!minimatch(target, rule.match.path)) continue;
      }
      if (rule.match.command) {
        const cmd = toolCall.params.command ?? "";
        // 把 "npm test*" 转成正则
        const re = new RegExp("^" + rule.match.command.replace(/\*/g, ".*"));
        if (!re.test(cmd)) continue;
      }
    }

    // 3. 第一个命中的规则决定结果
    return rule.decision;
  }

  // 4. 没有命中任何规则，用兜底策略
  return config.defaultDecision;
}

这里有三个工程考量。第一，顺序优先：规则按声明顺序匹配，所以更具体的规则要写在前面（比如「execute 命令 rm 开头 → deny」要写在「execute → ask」之前）。第二，不命中走 default：OpenCode 把 default 设为 ask，遵循「不确定就问」的安全默认。第三，匹配器可扩展：除了 path 和 command，未来可以加更多维度，比如匹配工具调用的「目标进程」或「网络端点」。

★ 四、ask 流程：阻塞 + 用户响应

当决策结果是 ask 时，OpenCode 不能直接执行也不能直接拒绝——它要暂停 Agent 循环，把请求展示给用户，等用户做出选择。这用到了我们在第 146 章提到的 emitAndWait 机制。

// 在工具执行前的拦截器（简化）
async function withPermission(
  toolCall: ToolCall,
  ctx: ToolContext
): Promise<ToolResult> {
  const decision = decide(ctx.session.permission, toolCall);

  if (decision === "deny") {
    return {
      content: [{ type: "text",
        text: `权限被拒绝：工具 ${toolCall.tool} 不被允许执行此操作。` }],
      isError: true,
    };
  }

  if (decision === "ask") {
    // 发出权限请求事件，阻塞等待用户响应
    const userChoice = await ctx.bus.emitAndWait("permission.request", {
      tool: toolCall.tool,
      params: toolCall.params,
      // 提供给用户的选项
    });
    if (userChoice === "deny") {
      return { content: [{ type: "text", text: "用户拒绝了此次调用。" }], isError: true };
    }
    // 用户可能选择 "allow always"，把这条规则固化下来
    if (userChoice === "allow-always") {
      ctx.session.permission.rules.unshift({
        tool: toolCall.tool,
        decision: "allow",
        match: extractMatcher(toolCall),
      });
    }
  }

  // allow 或用户同意，真正执行
  return await toolCall.execute(toolCall.params, ctx);
}

这里最值得品的是 allow-always 的学习机制：用户每次手动确认都很烦，所以 OpenCode 允许用户在确认时勾选「以后都允许」——这会动态生成一条精确匹配当前调用的 allow 规则，unshift 到规则列表最前面，下次同样调用就直接放行。这让权限系统从「静态配置」进化成了「交互式学习」——用得越久越顺手，但危险操作仍然每次问。

★ 五、路径白名单：文件访问的纵深防御

除了工具级的 allow/ask/deny，OpenCode 还有一层独立的文件路径白名单。即使 read_file 工具被 allow，如果目标路径不在白名单内，仍然会被拒。这是纵深防御（defense in depth）思想——不依赖单一关卡。

默认的白名单规则包括：

  • 当前会话的工作目录（cwd）及其子目录：默认允许。
  • 用户主目录下的 .ssh、.gnupg、.env 等敏感文件：默认 deny，即使在 cwd 内也要额外确认。
  • cwd 之外的系统路径（如 /etc、/usr）：默认 ask，需用户确认。
// 路径安全检查（简化）
const SENSITIVE_PATTERNS = [
  "**/.env",
  "**/.env.*",
  "**/.ssh/**",
  "**/.gnupg/**",
  "**/id_rsa*",
  "**/*.pem",
  "**/*.key",
];

export function checkPathSafety(absPath: string, cwd: string): PermissionDecision {
  // 敏感文件一律 deny
  if (SENSITIVE_PATTERNS.some(p => minimatch(absPath, p))) {
    return "deny";
  }
  // cwd 内允许
  if (absPath.startsWith(cwd)) {
    return "allow";
  }
  // cwd 外需确认
  return "ask";
}

这种设计的好处是即使用户不小心把 execute 配成了 allow，Agent 也无法通过 cat ~/.ssh/id_rsa 这种命令泄露密钥——因为路径检查会拦截。两层防护交叉生效，大大降低了误操作的风险。

★ 六、配置文件：把权限规则写下来

权限规则不只是运行时动态学习的，OpenCode 还支持通过配置文件预定义。这让团队可以把一套权限策略提交到仓库，所有成员共享。配置文件一般长这样：

{
  "permission": {
    "default": "ask",
    "rules": [
      { "tool": "read_file", "decision": "allow" },
      { "tool": "list_dir",  "decision": "allow" },
      { "tool": "execute", "decision": "allow", "match": { "command": "npm test*" } },
      { "tool": "execute", "decision": "allow", "match": { "command": "git status" } },
      { "tool": "execute", "decision": "deny",  "match": { "command": "rm -rf *" } },
      { "tool": "execute", "decision": "deny",  "match": { "command": "git push --force*" } },
      { "tool": "write_file", "decision": "deny", "match": { "path": "**/.env*" } },
      { "tool": "write_file", "decision": "ask" }
    ]
  }
}

注意规则的排列顺序：deny 类规则放在 allow 类之前，确保危险操作先被拦截。这种「先 deny 后 allow」的排列原则是配置权限系统时的通用最佳实践——你可以类比 iptables 防火墙规则的匹配逻辑。

★ 本章小结

OpenCode 的权限模型是它在「能力」和「安全」之间找平衡的核心机制。要点回顾：

  • 三态模型：allow / ask / deny 比二态更精细，ask 是关键——让无法预判的操作走人工确认。
  • 规则即数据：权限决策不是硬编码 if/else，而是可配置的规则列表按顺序匹配，第一个命中者决定结果。
  • 工具 × 参数粒度：match 字段让规则能精确到「execute 跑 npm test 才 allow」，而非对 execute 一刀切。
  • 交互式学习：allow-always 把用户每次确认固化为新规则，权限系统越用越顺手。
  • 路径白名单纵深防御：工具级权限之外，独立的敏感文件检查作为第二道关卡。
  • 配置可版本化：权限规则写成 JSON，团队共享，deny 规则排在前面是通用最佳实践。

权限解决了「能不能调用」的问题。下一章（150 章）我们转向另一个核心挑战——上下文管理：当对话越来越长、读过的文件越来越多，OpenCode 如何在有限的 context window 里塞下最有价值的信息。
