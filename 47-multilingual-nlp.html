多语言NLP
从中文分词到跨语言迁移的工程全景

当你的模型必须同时理解"我爱北京天安门"和"I love Paris"，工程问题就远不止换个词表那么简单。本文从中文分词的特殊性出发，一路谈到跨语言预训练、低资源语言救场方案，以及工业界最常用的多语言评测基准，带你建立一套真正能跑在 100+ 语言上的工程心智模型。

阅读约 18 分钟 难度 ★★★★

★ 中文分词：NLP里最特殊的"第一公里"

英文NLP在很长一段时间里，分词几乎等于按空格切。直到 BPE、WordPiece 出现，大家才把 subword 当成默认选项。但中文、日文这类 CJK 语言天然没有空格，分词从一开始就是一道独立的工程难题。"南京市长江大桥"到底是"南京市/长江大桥"还是"南京市长/江大桥"？这种歧义在英文里几乎不存在，在中文里却几乎每一句话都会遇到。

传统中文分词的两大流派是基于词典的最大匹配和基于统计模型的序列标注（CRF/HMM）。Jieba 是词典派最经典的实现，它的核心思路是：先构建一个大规模的前缀词典，用 DAG（有向无环图）枚举所有可能的切分路径，再用基于词频的动态规划选出概率最大的路径，最后对未登录词用 HMM 做一次识别。这套机制在 2014 年开源时几乎是中文分词的事实标准，时至今日它在轻量、确定性强、部署成本低的场景里依然是首选。

HanLP 则代表了统计/深度学习派的集大成者。它不仅提供感知机、CRF 等多种分词模型，还内置了词性标注、命名实体识别、依存句法分析、语义依存等一整套管道。对于需要"不只是分个词"的复杂业务，HanLP 几乎是开源方案里最完整的选择。它还提供了基于预训练词向量的神经网络分词模型，准确率在标准数据集上能压过很多商业服务。

需要注意的是，现代大语言模型其实已经绕开了显式分词——它们用 SentencePiece 学到的 subword 词表天然支持中文，每个汉字或高频汉字组合就是一个 token。但这并不意味着分词研究没有意义：在搜索引擎、关键词提取、知识图谱构建、规则匹配这些经典场景里，分词依然是基础设施级的能力。理解 Jieba 和 HanLP 的原理，对调试 LLM 的 tokenization、估算中文 token 成本也非常有帮助。

# Jieba 三种分词模式对比 + 自定义词典
import jieba
import jieba.posseg as pseg

text = "南京市长江大桥欢迎您，今天我们讨论Transformer的多语言能力"

# 1. 精确模式：默认，适合绝大多数场景
print("精确模式:", "/".join(jieba.cut(text, cut_all=False)))
# 南京市/长江大桥/欢迎/您/，/今天/我们/讨论/Transformer/的/多/语言/能力

# 2. 全模式：把所有可能的词都扫出来，适合搜索引擎建索引
print("全模式:  ", "/".join(jieba.cut(text, cut_all=True)))
# 南京/南京市/京市/市长/长江/长江大桥/江大/大桥/欢迎/您/...

# 3. 搜索引擎模式：在精确模式基础上对长词再切一刀
print("搜索引擎:", "/".join(jieba.cut_for_search(text)))

# 4. 加载自定义词典，解决新词（领域术语、产品名）
jieba.load_userdict("my_dict.txt")  # 格式：词语 词频 词性
# 也可以动态调整：jieba.add_word("大语言模型", freq=10000, tag='n')
jieba.add_word("大语言模型", freq=10000, tag='n')

# 5. 词性标注（pseg 接口）
for word, flag in pseg.cut("南京市长江大桥欢迎您"):
    print(f"{word}({flag})", end=" ")
# 南京市(ns) 长江大桥(n) 欢迎(v) 您(r)

★ 跨语言迁移：XLM-R 与 mBERT 为什么能"举一反百"

跨语言迁移的核心梦想是：在一种语言（通常是资源丰富的英语）上训练，然后在零样本设定下直接迁移到上百种语言上，还能保持不错的性能。mBERT（Multilingual BERT）是第一个让人眼前一亮的实践——Google 在训练时把 104 种语言的 Wikipedia 数据混在一起，共享同一个词表和同一套 Transformer 参数，结果发现模型在跨语言任务上表现出惊人的"对齐"能力：英语上学到的"动物"概念，居然能被法语、印地语、甚至斯瓦希里语共享。

XLM-R（基于 XLM 的 RoBERTa 版本）把这条路推到了新的高度。Meta 用 CommonCrawl 上 100 种语言、2.5TB 的数据重新训练，并在训练策略上做了关键改进：更大的词表（25 万 token，专门平衡低资源语言）、更长训练步数、去掉了语言嵌入信号（mBERT 需要显式告诉模型当前是哪种语言，XLM-R 不需要，让模型自己学对齐）。在 XNLI 这个跨语言推理基准上，XLM-R 平均比 mBERT 高出十几个百分点，尤其是在低资源语言上优势更明显。

但跨语言迁移并不是"自动生效"的魔法。它对以下三个因素非常敏感：第一，语言之间的类型学距离——英语到德语（同属日耳曼语族）迁移效果远好于英语到日语（完全不同的语系）；第二，预训练数据中该语言的占比——斯瓦希里语在语料里只有几 GB，自然不如英语的几百 GB；第三，是否做了继续预训练（continued pretraining）。一个常见的工程做法是：在目标语言的领域语料上对 XLM-R 再做一轮 MLM 训练，然后再在少量标注数据上微调，效果往往能再涨一截。

# 加载多语言模型做零样本跨语言推理
from transformers import AutoTokenizer, AutoModelForSequenceClassification
import torch

MODEL = "xlm-roberta-large"   # 也可换 "bert-base-multilingual-cased"
tok = AutoTokenizer.from_pretrained(MODEL)
model = AutoModelForSequenceClassification.from_pretrained(MODEL, num_labels=3)

# 关键技巧：用少量英文样本微调后，直接预测多语言句子
samples = [
    ("I love this movie, it's fantastic!", 2),   # 2=positive
    ("The plot was boring and slow.",      0),   # 0=negative
    ("It was okay, not great not bad.",    1),   # 1=neutral
]
# ... 在英文上 fine-tune（省略训练循环）

# 零样本跨语言预测：英文训出来的模型直接推断中文、法语、阿拉伯语
test = [
    "这部电影太棒了，我非常喜欢！",     # 中文
    "Ce film est incroyable!",          # 法语
    "هذا الفيلم رائع جداً",             # 阿拉伯语
]
model.eval()
with torch.no_grad():
    for s in test:
        ids = tok(s, return_tensors="pt", truncation=True, max_length=128)
        logits = model(**ids).logits
        print(s, "→", model.config.id2label[logits.argmax().item()])
# 三句都应预测为 positive，这就是跨语言迁移的力量

★ 低资源语言：当训练数据只有几 MB 时怎么办

世界上大约有 7000 种语言，其中有像样语料库的可能不到 200 种。对于剩下的绝大多数——非洲的豪萨语、东南亚的爪哇语、南美的克丘亚语——我们既没有几 GB 的预训练数据，也没有几千条标注样本。低资源语言处理是多语言 NLP 真正的前沿，也是商业价值最被低估的方向之一。

工程上常用的几条策略可以叠加使用。第一是回译（back-translation）数据增强：用机器翻译把目标语言翻成英语，再用英语样本扩充训练集。第二是跨语言词嵌入对齐：用 MUSE 或 Vecmap 这种无监督方法，把低资源语言的词向量映射到英语空间，从而复用英语学到的语义结构。第三是提示学习（prompt-based learning）：与其微调整个模型，不如固定模型只学一个轻量提示向量，这样几百条样本就足够。第四是利用多语言预训练模型作为起点，再配合 LoRA 这类参数高效微调，把"需要的数据量"从几万降到几百。

还有一个经常被忽视的杠杆：语料清洗。低资源语言的原始数据噪声极大——很多所谓"斯瓦希里语"网页其实混了大量英文、错别字、机器翻译垃圾。花一周时间写个严格的语言识别 + 质量过滤 pipeline，往往比再训十轮模型涨点还明显。这就是为什么 Meta 在训练 NLLB-200（支持 200 种语言互译的模型）时，把大量工程投入放在了数据挖掘和清洗上，而不是单纯堆模型规模。

★ 多语言评测基准：怎么客观衡量"懂多种语言"

没有基准就没有比较。多语言 NLP 有几个绕不开的评测集。XNLI（Cross-lingual Natural Language Inference）是最经典的——它把英文的 MultiNLI 翻译成 14 种语言，要求模型判断前提和假设之间的蕴含/矛盾/中立关系。因为标注是共享的，所以可以直接看出模型在不同语言上的"理解"差距。一个典型现象是：英文准确率 80% 的模型，到斯瓦希里语可能掉到 60%，这种"跨语言 gap"正是衡量迁移质量的核心指标。

MULTI-FC（Multilingual Fact-Checking）则聚焦事实核查任务，覆盖几十种语言，每条样本带多个声明和证据，要求模型判断真假。它的难点在于：事实核查强依赖外部知识，低资源语言能拿到的证据本身就稀少，所以这个基准能很好地暴露"知识迁移"而非仅仅是"语言理解"的问题。其他常用基准还包括 XTREME（Google 出的多任务跨语言基准，涵盖分类、问答、检索、序列标注四类任务）、PAWS-X（跨语言 paraphrase 识别）、TyDi QA（类型学多样的问答数据集）。

选基准时有几个工程考量。首先，要看你的业务语言是不是被覆盖——XNLI 没有 Vietnamese，做越南语的项目就用不上。其次，要注意"翻译噪声"——很多基准是把英文人工/机器翻译成其他语言，翻译质量本身可能就把模型卡住了，导致你测的不是模型能力而是翻译质量。这一点在 MULTI-FC 上尤其明显：低资源语言的事实核查证据往往是机器翻译的，翻译错误会直接导致"假新闻被判为真"。最后，强烈建议建立内部评测集，至少覆盖你业务最关心的 3-5 种语言，用真实用户数据而非翻译数据。开源基准是地板，业务指标才是天花板。

还有一类常被忽视的评测维度是"跨语言一致性"（cross-lingual consistency）。同一个问题用英语问和用斯瓦希里语问，模型应该给出语义一致的答案，否则就说明模型对不同语言的理解有偏差。检测一致性的方法是把同一个测试集翻成多语言版本，对比模型在各语言上的预测分布差异。差异大的样本往往是模型在低资源语言上的盲区，可以反过来指导数据采集和继续预训练的优先级。这种"以评测驱动数据"的闭环，是现代多语言 NLP 团队的标准工作流。

把上面四块拼起来——可靠的分词、强有力的多语言预训练底座、针对低资源语言的 augmentation 策略、客观的跨语言评测——你就拥有了一个真正可工程化的多语言 NLP 系统。它不会让所有语言都达到英语的性能，但能让"80 分语言、50 分语言、20 分语言"形成一个可管理的梯度，而不是断崖式崩塌。这才是多语言 NLP 在现实世界里能做到，也值得做的目标。
