局部敏感哈希 LSH
p-stable 分布、欧氏距离哈希族与碰撞概率

局部敏感哈希（Locality Sensitive Hashing, LSH）是向量检索里理论最优雅、却也是工程上最容易跑偏的方法。它用一组特殊的哈希函数让"近的向量大概率哈希到同一个桶"，从而把最近邻搜索转化为桶内搜索。本文从 p-stable 分布讲起，推导碰撞概率公式，并用代码实现一个可用的 LSH 索引，最后讲清它为什么在主流向量库里逐渐退场。

阅读约 18 分钟 难度 ★★★★★

★ LSH 的核心定义：近的更易碰撞

普通哈希函数（如 MD5）的设计目标是"输入差一点，输出完全不同"——用于去重和查找。LSH 反其道而行：它要的是"输入越近，哈希碰撞概率越高"。形式化定义是：一个哈希函数族 H 称为 (r, cr, p1, p2)-敏感的，如果对任意两点 p, q：

  • 当 dist(p,q) ≤ r 时，Pr[h(p)=h(q)] ≥ p1
  • 当 dist(p,q) ≥ cr 时，Pr[h(p)=h(q)] ≤ p2
  • 其中 c > 1 是近似比，且 p1 > p2（近的点碰撞概率严格高于远的点）

有了这样的函数族，检索流程是：把所有数据库向量哈希到桶里，查询时算 q 的哈希，只在同桶（或相邻桶）里做精确搜索。近的点大概率落在同桶被找到，远的点大概率分到不同桶被剪掉。这就是 LSH 用"碰撞概率差异"实现剪枝的本质。

LSH 的理论保证很强：经过恰当的 AND/OR 构造（后面讲），可以在次线性时间 O(N^ρ) 内以高概率找到 (1+ε)-近似最近邻，其中 ρ = ln(p1)/ln(p2) < 1。这是 LSH 论文 1998 年发表时的里程碑结果。但工程实践里，这个理论保证的代价是参数难调、内存占用不省、长尾召回不稳。

★ p-stable 分布：欧氏距离 LSH 的数学基础

对于欧氏距离（L2），最经典的 LSH 函数族基于 p-stable 分布。一个分布称为 p-stable，如果对任意实数 v1,...,vn 和独立同分布的 a1,...,an，线性组合 Σ vi·ai 的分布与 (Σ|vi|^p)^(1/p) · a 同分布，其中 a 服从原分布。

两个重要的 p-stable 分布：p=1 是柯西分布，p=2 是高斯分布。后者直接对应欧氏距离——这正是我们需要的。基于高斯分布（p=2-stable）的 LSH 哈希函数定义为：

h_{a,b}(v) = floor( (a·v + b) / w )

其中 a 是从 d 维标准高斯分布采样的随机向量，b 是从 [0, w] 均匀采样的偏移，w 是桶宽（关键参数）。直觉是：a·v 把 d 维向量投影到一维（保留距离信息，因为高斯是 2-stable），再除以 w 分桶。两个向量 v, u 的投影差 a·(v-u) 服从 N(0, ||v-u||²)（2-stable 性质），所以它们的距离 ||v-u|| 越大，投影差越可能超过 w 落到不同桶。

碰撞概率公式

对一对距离 c=||v-u|| 的向量，单次哈希碰撞概率为：

p(c) = 1 - 2·Φ(-w/c) - (2/(√(2π)·w/c)) · (1 - exp(-w²/(2c²)))

其中 Φ 是标准正态分布的累积分布函数。这个公式看上去吓人，但关键结论很简单：p(c) 是 c 的单调递减函数，c 越大碰撞概率越低。这就是 LSH 工作的基础——距离和碰撞概率有单调关系。w 控制曲线的陡峭程度：w 小则区分度高但召回低，w 大则召回高但区分度低。

# 画出 p(c) 曲线，直观感受 w 的影响
import numpy as np
from scipy.stats import norm
import matplotlib.pyplot as plt

def collision_prob(c, w):
    if c < 1e-9:
        return 1.0
    x = w / c
    return 1 - 2*norm.cdf(-x) - (2/(np.sqrt(2*np.pi)*x)) * (1 - np.exp(-x**2/2))

c_range = np.linspace(0.1, 5, 200)
for w in [1.0, 2.0, 4.0]:
    plt.plot(c_range, [collision_prob(c, w) for c in c_range], label=f"w={w}")
plt.xlabel("distance c"); plt.ylabel("collision prob p(c)")
plt.legend(); plt.grid(True); plt.title("LSH p-stable 碰撞概率曲线")
plt.savefig("lsh_curve.png", dpi=100)
# 结论：w 越大曲线越平（区分度低），w 越小越陡（区分度高但整体召回低）

★ AND/OR 构造：调出 p1 和 p2 的差距

单个 LSH 函数的 p1 和 p2 差距通常不够大（比如 p1=0.8, p2=0.5），剪枝效果弱。LSH 用两个组合操作放大这个差距：

  • AND 构造（串联）：用 K 个独立哈希函数，只有全部碰撞才算同桶。新碰撞概率 p^K，把高概率和低概率都压低，但低概率压得更狠，差距拉大。代价是召回下降。
  • OR 构造（并联）：用 L 组 AND 哈希表，任一组碰撞就算候选。新碰撞概率 1-(1-p^K)^L，召回回升。

组合后，最终碰撞概率 P(p) = 1 - (1 - p^K)^L。它是个 S 型曲线：对中间的 p 几乎是线性，但对很小的 p 几乎压到 0、对接近 1 的 p 几乎抬到 1。这正是我们要的——把"近（p1）"和"远（p2）"的碰撞概率分别推到接近 1 和 0。

参数 K 和 L 的选择是个权衡：K 大则假阳性少（远的不易混入）但召回低，L 大则召回高但内存和查询时间增加（要查 L 个哈希表）。经验上 K ≈ log(N)/log(1/p2)、L ≈ N^ρ，其中 ρ=ln(1/p1)/ln(1/p2)。

# 看清 K 和 L 如何塑造最终的碰撞概率曲线
def combined_prob(p, K, L):
    return 1 - (1 - p**K)**L

p_range = np.linspace(0, 1, 200)
for K, L in [(1,1), (5,10), (10,20), (15,30)]:
    plt.plot(p_range, [combined_prob(p, K, L) for p in p_range],
             label=f"K={K}, L={L}")
plt.xlabel("单函数碰撞概率 p"); plt.ylabel("组合后碰撞概率")
plt.legend(); plt.grid(True); plt.title("AND/OR 构造的 S 型曲线")
plt.savefig("lsh_andor.png", dpi=100)
# K,L 越大，曲线越接近阶跃 —— 越能精准区分近/远

★ 从零实现一个可用的 LSH 索引

理论清楚了，我们写一个能跑的 LSH 索引，看清工程上每个组件。

import numpy as np
from collections import defaultdict

class LSHIndex:
    def __init__(self, d, n_tables=10, hash_size=8, w=4.0, seed=42):
        """
        d:         向量维度
        n_tables:  OR 构造的哈希表数 L
        hash_size: AND 构造的串联哈希数 K
        w:         桶宽（关键参数）
        """
        self.d = d; self.L = n_tables; self.K = hash_size; self.w = w
        rng = np.random.default_rng(seed)
        # 为每个表预生成 K 个投影向量 a 和偏移 b
        self.A = rng.standard_normal((self.L, self.K, d))   # (L, K, d)
        self.B = rng.uniform(0, w, (self.L, self.K))        # (L, K)
        self.tables = [defaultdict(list) for _ in range(self.L)]

    def _hash_one_table(self, v, t):
        proj = (self.A[t] @ v + self.B[t]) / self.w          # (K,)
        return tuple(np.floor(proj).astype(int))             # 一个 K 维 tuple 作为桶键

    def insert(self, vecs):
        for i, v in enumerate(vecs):
            for t in range(self.L):
                key = self._hash_one_table(v, t)
                self.tables[t][key].append(i)

    def query(self, q, k=10):
        cand = set()
        for t in range(self.L):
            key = self._hash_one_table(q, t)
            cand.update(self.tables[t].get(key, []))
        return cand     # 候选集，再交给精确距离排序

# 实测
d = 128; N = 50_000
data = np.random.randn(N, d).astype(np.float32)
lsh = LSHIndex(d, n_tables=15, hash_size=8, w=4.0)
lsh.insert(data)
q = np.random.randn(d).astype(np.float32)
cand = lsh.query(q)
print(f"候选集大小: {len(cand)} / {N}  (剪枝率 {1-len(cand)/N:.2%})")
# 在候选集上精确排序
if cand:
    cand = np.array(list(cand))
    dists = np.linalg.norm(data[cand] - q, axis=1)
    top = cand[np.argsort(dists)[:10]]
    print("LSH top-10:", top)

跑这段代码你会观察到一个尴尬现象：候选集大小很不稳定，有时几十个、有时几千个。这正是 LSH 的工程痛点——召回方差大。w 调小，多数查询候选为空（召回低）；w 调大，候选爆炸（接近暴力）。要在所有查询上稳定达到 95% 召回，L 和 K 要调得很保守，结果速度优势就被吃掉了。

★ 为什么 LSH 在主流向量库里退场

  • 参数难调且数据敏感：w/K/L 强依赖数据分布，换一份数据要重调，不像 HNSW 那套参数几乎通用。
  • 召回方差大：不同查询的候选集大小差异巨大，P99 延迟难控制，对在线服务不友好。
  • 内存不省：需要 L 个哈希表，每张表存全量 ID，内存和 HNSW 相当甚至更高。
  • 不擅长高召回：要 recall@10 ≥ 0.95 通常需要很大的 L，速度优势被抹平，跑不赢 HNSW。
  • 动态更新麻烦：插入会改变桶分布，删改要标记，重建成本高。

但 LSH 不是没价值。它在理论分析（次线性时间保证）、位图指纹去重（MinHash/_simhash 之于文档）、海量粗筛（先 LSH 桶筛再精确）等场景仍有用武之地。理解 LSH 也能帮你理解后续 ScaNN、PQ 这些更现代方法的设计动机。

★ 本章小结

  • LSH 的定义是"近的点碰撞概率高、远的低"，形式化为 (r, cr, p1, p2)-敏感，p1>p2。
  • 欧氏距离 LSH 用 p=2-stable 的高斯分布：h(v)=floor((a·v+b)/w)，a~N(0,I)，碰撞概率 p(c) 随距离 c 单调递减。
  • AND 构造（K 个串联）压低假阳性，OR 构造（L 张表并联）召回回升，组合概率 1-(1-p^K)^L 呈 S 型，放大近/远差距。
  • K 和 L 的选择是召回-速度-内存的三方权衡，强依赖数据分布。
  • 工程上 LSH 因参数难调、召回方差大、内存不省、不擅长高召回，在主流向量库里被 HNSW/IVFPQ 取代；但在理论、指纹去重、粗筛等场景仍有价值。
