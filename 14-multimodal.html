多模态大模型深度
从「只读文字」到「看懂世界」

真实世界不止文字。图像、音频、视频承载了远比文本更丰富的信息。多模态大模型让 AI 能看、能听、能说，走向通用感知。本章拆解多模态的架构原理（视觉编码器、跨模态连接器）、训练方法、典型应用，以及 GLM-4V、GPT-4V 背后的技术。

★ 为什么要多模态

纯文本模型有三个根本局限：

🖼️

看不懂图

图表、截图、公式、手写——文字描述又慢又失真。

🔇

听不见音

语音的情感、语调、环境音，文字无法传达。

🎥

不懂视频

动作、空间关系、时间演变，纯文本束手无策。

多模态模型把这些原生的感知能力接进大模型，让它像人一样「眼观六路、耳听八方」。

视觉编码器：把图像变成「视觉 token」

图像不能直接进 LLM，要先编码成向量序列。最主流的是 ViT（Vision Transformer）：

  • 把图像切成固定大小的「patch」（如 16×16 像素）；
  • 每个 patch 展平成一个向量，加位置编码；
  • 过 Transformer 编码器，输出每个 patch 的语义向量；
  • 一张图就变成了一串「视觉 token」，和文本 token 本质一样。

常用预训练视觉编码器：CLIP（OpenAI，图文对齐）、SigLIP（Google，CLIP 改进）、InternViT（高分辨率）。GLM-4V 用的是自研视觉编码器，针对中文场景优化。

跨模态连接器：让视觉和语言「说同一种话」

视觉编码器输出的向量和 LLM 的嵌入空间维度不一定一样，需要连接器对齐。三种主流方案：

方案原理代表
线性/MLP 投影一个矩阵把视觉向量投影到 LLM 空间LLaVA（最简单有效）
Q-Former用一组可学习 query 压缩视觉信息BLIP-2、InstructBLIP
交叉注意力LLM 的某些层插入跨模态注意力Flamingo

文本侧不变

文本依然走第 1 章的 Tokenizer + Embedding。关键创新只在「把视觉/音频 token 和文本 token 拼在一起喂给 LLM」——对 LLM 来说，它们都是序列里的 token，一视同仁处理。这就是多模态「统一」的本质。

02 多模态训练：从对齐到指令

多模态模型的训练通常分三阶段：

阶段 1：模态对齐预训练

用大量「图像-描述」对，训练连接器，让视觉特征能映射到语言空间。常用 LAION 等大规模图文数据集。

阶段 2：多模态指令微调

用「图像+指令-回答」数据做 SFT，让模型学会「看图回答问题」。数据如 LLaVA-Instruct。

阶段 3：多模态对齐

RLHF/DPO，对齐人类偏好，减少幻觉（比如不瞎编图里没有的东西）。

⚠️ 多模态幻觉更严重

多模态模型有个特有问题：看了图却瞎编。比如图里没有红色汽车，它却坚持说有。原因是视觉理解不精确 + 语言模型的「补全」倾向。缓解：用更细粒度的视觉编码（高分辨率）、训练时强调「基于图像事实」、对幻觉专门做 RLHF。

05 走向统一：Any-to-Any 模型

2026 年的前沿趋势是统一多模态模型：一个模型，任意输入（文/图/音/视）→ 任意输出（文/图/音/视）。代表：GPT-4o、Gemini、GLM-4V+。

关键技术挑战：

  • 统一 token 化：把所有模态都表示成 token（图像用 patch token、音频用声学 token）；
  • 统一架构：一个 Transformer 处理所有模态，而非多个模型拼接；
  • 生成能力：不仅要理解，还要能生成各模态内容（融合扩散模型或自回归生成）；
  • 训练数据：需要海量多模态对齐数据。
统一多模态是通往 AGI（通用人工智能） 的必经之路——一个真正智能的系统，必须能像人一样用多种感官感知世界。

★ ViT patch 化的最小实现

前面讲了 ViT 把图像切成 patch。但「切 patch」到底在代码里是什么操作？其实就三步：把图像切成网格 → 每个小块展平成向量 → 加位置编码。下面这段 PyTorch 代码用十几行实现了 ViT 的 PatchEmbed 模块——它是一切视觉 Transformer 的入口。跑一遍你就会发现，「图像变成 token 序列」没有任何黑魔法，本质就是一个 reshape + 线性投影。

import torch
import torch.nn as nn

class PatchEmbed(nn.Module):
    """把图像切成 patch 并嵌入成 token 序列——ViT 的入口"""
    def __init__(self, img_size=224, patch_size=16, in_chans=3, embed_dim=768):
        super().__init__()
        self.num_patches = (img_size // patch_size) ** 2   # 224/16=14 → 196 个 patch
        # 一个卷积等价于“切 patch + 线性投影”两步合一
        self.proj = nn.Conv2d(in_chans, embed_dim,
                              kernel_size=patch_size, stride=patch_size)
        # 可学习的位置编码：告诉模型每个 patch 在图里的位置
        self.pos_embed = nn.Parameter(torch.randn(1, self.num_patches + 1, embed_dim) * 0.02)

    def forward(self, x):                  # x: (B, 3, 224, 224)
        x = self.proj(x)                     # → (B, 768, 14, 14)
        x = x.flatten(2).transpose(1, 2)  # → (B, 196, 768) 就是 196 个视觉 token
        cls = torch.zeros(x.size(0), 1, x.size(2))  # [CLS] token，用于汇总全图
        x = torch.cat([cls, x], dim=1)      # → (B, 197, 768)
        return x + self.pos_embed            # 加位置编码后送进 Transformer

# 验证：一张 224×224 的图，变成 197 个 768 维 token
patch_embed = PatchEmbed()
dummy_img = torch.randn(1, 3, 224, 224)
tokens = patch_embed(dummy_img)
print(tokens.shape)   # torch.Size([1, 197, 768])
# 至此图像已经和一段 197 词的文本在数据结构上没有区别——都可以喂进 Transformer

这里有个极其精妙的点：Conv2d(kernel=16, stride=16) 这一个卷积层，数学上完全等价于「把图切成 16×16 的小块、每块展平成 768 维向量、再做线性变换」三件事。深度学习框架用卷积来实现只是因为更快。patch_size 是个关键超参——越小（如 8×8）token 越多、细节越丰富但计算量平方级增长；越大（如 32×32）越省算力但丢细节。GLM-4V 等高分辨率模型会用「动态 patch」策略：对高清图先切大块，对感兴趣区域再切小块，兼顾效率和精度。

★ LLaVA 式跨模态连接器实现

有了视觉 token，还差最后一步——把它们「翻译」成 LLM 能懂的语言。LLaVA 用了一个极其简单却出奇有效的方案：一个单层线性投影（MLP），把 ViT 输出的视觉向量映射到 LLM 的嵌入空间，然后直接当作「特殊的文本 token」拼到 prompt 前面。就这么朴素，LLaVA 却成了开源多模态的事实基线。下面是它的核心实现。

import torch.nn as nn

class LLaVAConnector(nn.Module):
    """LLaVA 式连接器：把视觉编码器输出投影到 LLM 词嵌入空间"""
    def __init__(self, vision_dim=1024, llm_dim=4096):
        super().__init__()
        # 核心就这一个 MLP——LLaVA-1.5 升级成两层，效果更好
        self.mlp = nn.Sequential(
            nn.Linear(vision_dim, llm_dim),
            nn.GELU(),
            nn.Linear(llm_dim, llm_dim),
        )

    def forward(self, image_features):       # (B, num_patch, vision_dim)
        return self.mlp(image_features)         # (B, num_patch, llm_dim)

def build_llava_prompt(text, image_tokens, llm_embed, connector):
    # 1. 文本照常分词、嵌入
    text_emb = llm_embed.tokenize(text)         # (1, T_text, llm_dim)
    # 2. 图像经连接器投影成“视觉词”
    vis_emb = connector(image_tokens)           # (1, T_img, llm_dim)
    # 3. 直接拼接——视觉词像普通词一样参与注意力
    seq = torch.cat([vis_emb, text_emb], dim=1)  # (1, T_img+T_text, llm_dim)
    return seq
    # 之后这个序列送进 LLM，LLM 的自注意力天然能让文本 token “看到”图像 token

这段代码揭示了多模态对齐的本质：所谓「让模型看懂图」，就是把图像变成一串和文本「同维度、同空间」的向量，然后让 Transformer 的注意力机制自己去学怎么关联。LLaVA 的训练也极简——第一阶段用「图像-描述」对只冻住其他参数、单独训练这个 MLP 做对齐；第二阶段用指令微调数据训练整体遵循指令。这种「轻连接器 + 重利用已有 LLM」的思路，是为什么开源社区能用很小的算力训出不错多模态模型的原因。相比之下，Q-Former（BLIP-2 用的）更复杂——它用一组可学习的 query 主动从图像特征里「抽取」固定数量的 token，能压缩冗余视觉信息，但训练更难收敛。工程上，LLaVA 的简单方案往往是最先该试的基线。

深 多模态落地：常见故障与避坑指南

多模态模型从 demo 走向生产，最大的障碍不是模型能力，而是工程上的“细节刺客”。视觉 token 的计费、图像预处理的疏忽、模态未对齐的幻觉，每一项都能让一个看起来很炫的 demo 在线上惨败。下面把真实项目里高频踩坑点逐一拆解，并给出可直接复用的检测代码。

坑一：图像分辨率与 token 数量失控

很多团队直接把用户上传的 4K 图片喂给模型，结果一张图被切成上千个 patch，瞬间撑爆上下文窗口，首字延迟从 1 秒飙到 30 秒，账单也按视觉 token 计费翻了几十倍。正确做法是在 API 入口做统一预处理：长边缩放到模型设计的输入尺寸（如 448 或 336），并对超大图分块（tile）后逐块识别再合并。下面这段预处理代码是 GLM-4V / LLaVA 类模型的通用前置逻辑。

import base64, io
from PIL import Image
from PIL.Image import Exif

def encode_image_for_vlm(path, max_side=448, quality=85):
    """把任意尺寸/方向的图片规范化为 VLM 期望的输入，返回 base64 与 patch 数估算。"""
    img = Image.open(path).convert("RGB")
    # 1) 修正手机竖拍的方向（EXIF Orientation 丢失是最隐蔽的 bug）
    try:
        exif = img.getexif()
        if exif.get(0x0112, 1) != 1:           # Orientation tag
            from PIL import ImageOps
            img = ImageOps.exif_transpose(img)
    except Exception:
        pass
    w, h = img.size
    scale = max_side / max(w, h)
    if scale < 1.0:                            # 只缩不放，避免插值模糊
        img = img.resize((max(1, int(w*scale)), max(1, int(h*scale))), Image.LANCZOS)
    patch = 14
    n_patch = (img.width // patch) * (img.height // patch)   # 视觉 token 估算
    buf = io.BytesIO(); img.save(buf, format="JPEG", quality=quality)
    return base64.b64encode(buf.getvalue()).decode(), n_patch

b64, n = encode_image_for_vlm("huge_4k_photo.jpg")
print(f"视觉 token 约 {n}；超过 2000 则 TTFT 显著劣化，应分块或裁剪")

关键经验：视觉 token 也是 token，在计费和延迟上和文本 token 一视同仁。给多模态接口设 token 预算上限，比事后看账单要靠谱得多。

坑二：模态未对齐导致“看图说胡话”

当视觉特征和语言空间没对齐好（比如用了未经过指令微调的连接器权重），模型会幻觉出图片里根本没有的内容——把白色汽车说成红色，把空桌子说成摆满文件。生产中必须加入“拒答与幻觉校验”：用一组“图中是否有 X”的探测问题，统计不存在物体的误报率（幻觉率）。GLM-4V 在中文物体识别上的幻觉率官方公布低于同级开源模型，但零幻觉仍不可能，所以关键决策（医疗影像、工业质检、自动驾驶）必须有人工复核或第二模型交叉验证，绝不能让单一多模态模型一锤定音。上线前建议跑一遍 POPE 或同类幻觉基准，建立幻觉率的基线，后续每次模型升级都回归对比。

Q1：多模态模型为什么对图表、公式、手写识别特别差？

这些内容包含精细的空间结构和密集符号，对视觉编码器分辨率要求极高。低分辨率下，一个公式被压成几十个 patch，细节全丢，模型只能猜。解法是超采样——对图表做高分辨率裁剪单独识别，或采用「版面分析+OCR+多模态理解」的多阶段管线，而非端到端单模型硬啃。这也是专业文档解析系统的通用架构。

Q2：多模态 RAG 和纯文本 RAG 的关键区别是什么？

检索端要做跨模态匹配——用文本 query 检索图片，或用图片检索文档，要求 embedding 模型支持跨模态（如 CLIP 把图文映射到同一空间）。工程坑是多模态索引构建成本高，且图文混合检索相关性更难判。生产中常用混合检索：文本走文本索引、图片走视觉索引，再融合重排，而非强行塞进单一索引。结果要引用溯源——告诉用户结论来自哪张图哪个区域，提升可信度。
