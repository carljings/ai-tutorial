从 MHA 到 MQA、GQA
推理加速的关键一跃

标准多头注意力（MHA）每个头都有独立的 K、V，推理时要为每个头都存一份 KV cache，显存压力巨大。MQA 和 GQA 的思路极其朴素：让多个头共享同一份（或几份）K、V，显存立省一大半，速度也跟着涨。Llama 2/3、Mistral、GLM 都用了 GQA。本章讲清 KV 共享的原理、显存到底省了多少、为什么效果几乎不掉，以及 MHA/MQA/GQA 的选择权衡。

阅读约 24 分钟 难度 ★★★★ 前置：第 98 章

★ MHA 的推理痛点：KV cache 随头数爆炸

先回顾标准多头注意力（MHA）。h 个头，每个头有独立的 Wk_i、Wv_i 投影，产生独立的 K_i、V_i。推理时，每生成一个 token，就要把它的 K、V 存进 KV cache，供后续 token 注意力使用。

问题在于：每个头都要存一份。KV cache 的显存占用是：

KV cache 显存 = 2 × n_layers × n_heads × seq_len × head_dim × dtype_size
（2 是 K 和 V 两份）

举个真实数字。Llama 2-70B：80 层、64 头、head_dim=128、FP16（2 字节）、上下文 4K：

2 × 80 × 64 × 4096 × 128 × 2 = 约 10.7 GB，光 KV cache 就要 10G+ 显存

如果上下文拉到 32K，KV cache 直接涨到 80 GB+——比模型参数本身还大！这就是为什么大模型推理时显存往往被 KV cache 吃掉一大半，严重限制 batch size 和并发数。

MHA：每头独立 K、V
KV cache = n_heads × (K + V) × seq × head_dim
▼ 头数多 / 上下文长 → 显存爆炸
MQA：所有头共享 1 组 K、V
KV cache = 1 × (K + V) × seq × head_dim → 省 h 倍
▼ 折中
GQA：分 g 组，组内共享 K、V
KV cache = g × (K + V) × seq × head_dim → 省 h/g 倍
图 99-1 · 三种注意力的 KV 共享策略，显存占用递减

01 MQA：所有头共享一组 K/V

Multi-Query Attention（MQA）是 2019 年提出的激进方案。它的改动极其简单：Q 仍然是多头的（每个头有自己的 Wq_i），但 K 和 V 全头共享一份（只有一组 Wk、Wv）。

MQA: Q 是 h 头，K 和 V 只有 1 头（所有 Q 头都去 attend 同一份 K、V）

从公式看，注意力计算变成：

  • 每个头 i：Q_i = X · Wq_i（和 MHA 一样，每头不同）
  • 共享：K = X · Wk，V = X · Wv（所有头用同一份）
  • head_i = softmax(Q_i · K^T / √d_k) · V

KV cache 直接从 h 份降到 1 份——省了 h 倍显存。对于 32 头的模型，KV cache 缩到原来的 1/32。同时，推理时从显存读 K、V 的带宽也降了 h 倍，速度更快。

为什么效果几乎不掉

直觉上，所有头看同一份 K、V，表达力应该大降。但实验发现：MQA 的效果只比 MHA 略差一点点（质量损失通常在 1% 以内）。原因有两个：

  • Q 仍然多头：每个头用不同的 Query 去「查询」同一份 K、V，仍能学到不同的关注模式。多样性主要来自 Q。
  • 很多头本就冗余：研究发现 MHA 里多个头的 K、V 学到的内容高度相似（都在提取类似的「被查询特征」），共享一份损失不大。
💡 类比：图书馆共用书架

MHA 像 h 个读者每人有自己的书架（K/V），各自独立但占地大。MQA 像所有读者共用一个大书架——书（K/V）是一样的，但每个读者用不同的检索方式（Q）去找自己感兴趣的部分。占地省了 h 倍，而只要书足够全，检索效果差不了多少。

02 GQA：MHA 和 MQA 的折中

MQA 虽然省显存，但毕竟激进，长上下文和复杂任务上质量损失有时明显。Grouped-Query Attention（GQA）是 Google 2023 年提出的折中：把 h 个头分成 g 组，组内共享 K、V，组间不同。

GQA: h 个 Q 头分成 g 组（每组 h/g 个头），每组共享一份 K、V
g = 1 时退化为 MQA，g = h 时退化为 MHA

这样 KV cache 是 MHA 的 g/h 倍。比如 h=32、g=8（每 4 个头共享一组），KV cache 省 4 倍，而效果比 MQA（g=1）好得多。Llama 2-70B 用 g=8（即 8 个 KV 头服务 64 个 Q 头），是 GQA 最著名的应用。

方法Q 头数KV 头数KV cache质量代表模型
MHAhh1×最好原始 Transformer, GPT-3
MQAh11/h略差PaLM, Falcon
GQAhgg/h接近 MHALlama 2/3, Mistral, GLM

2024 年后的开源大模型，几乎清一色用 GQA。它被认为是「质量-推理成本」的最佳平衡点：比 MQA 质量高，比 MHA 推理快得多。

03 显存到底省多少：算一笔账

用具体数字感受一下 GQA 的威力。对比 Llama 2-70B（MHA）和 Llama 2-70B-GQA（假设配置）在 32K 上下文下的 KV cache：

kv_cache_calc.py —— KV cache 显存计算器
def kv_cache_bytes(n_layers, n_kv_heads, seq_len, head_dim, dtype_bytes=2):
    """计算 KV cache 显存。
    n_kv_heads: 实际的 KV 头数（MHA=n_heads, MQA=1, GQA=g）
    """
    # 2 份（K 和 V），每个位置每头存 head_dim 个元素
    return 2 * n_layers * n_kv_heads * seq_len * head_dim * dtype_bytes

# Llama 2-70B 配置
cfg = dict(n_layers=80, head_dim=128, dtype_bytes=2)  # FP16
seq_len = 32 * 1024  # 32K 上下文

# MHA: 64 个 KV 头
mha = kv_cache_bytes(n_kv_heads=64, seq_len=seq_len, **cfg)
# MQA: 1 个 KV 头
mqa = kv_cache_bytes(n_kv_heads=1, seq_len=seq_len, **cfg)
# GQA: 8 个 KV 头（每 8 个 Q 头共享一组）
gqa = kv_cache_bytes(n_kv_heads=8, seq_len=seq_len, **cfg)

def gb(b): return b / 1024**3

print(f"MHA KV cache: {gb(mha):.1f} GB")
print(f"MQA KV cache: {gb(mqa):.1f} GB")
print(f"GQA KV cache: {gb(gqa):.1f} GB")
print(f"\nGQA 比 MHA 省 {(1-gqa/mha)*100:.0f}%，比 MQA 多用 {(gqa/mqa-1)*100:.0f}%")

结果（32K 上下文）：MHA 约 80 GB，MQA 约 1.25 GB，GQA 约 10 GB。GQA 比 MHA 省了 87%，而比 MQA 质量高得多。这就是为什么 GQA 成了主流——在显存压力和质量之间找到了甜点。

📌 注意 dtype 的影响

上面用 FP16（2 字节）算。如果用 INT8 量化 KV cache，显存再减半；INT4 再减 4 倍。结合 GQA + KV 量化，70B 模型的 KV cache 可以压到几 GB，让单卡推理成为可能。这是推理服务优化的核心手段。

04 代码：实现 GQA

GQA 的实现和 MHA 几乎一样，只是 K、V 的投影维度从 d_model 变成 n_kv_heads × head_dim，然后广播到所有 Q 头。

    def __init__(self, d_model, n_heads, n_kv_heads):
        super().__init__()
        assert d_model % n_heads == 0
        self.n_heads = n_heads
        self.n_kv_heads = n_kv_heads
        self.n_rep = n_heads // n_kv_heads  # 每个 KV 头服务几个 Q 头
        self.head_dim = d_model // n_heads

        # Q 投影：n_heads 个头
        self.W_q = nn.Linear(d_model, n_heads * self.head_dim, bias=False)
        # K、V 投影：只有 n_kv_heads 个头（更少！）
        self.W_k = nn.Linear(d_model, n_kv_heads * self.head_dim, bias=False)
        self.W_v = nn.Linear(d_model, n_kv_heads * self.head_dim, bias=False)
        self.W_o = nn.Linear(n_heads * self.head_dim, d_model, bias=False)

    def forward(self, x):
        B, S, _ = x.shape
        q = self.W_q(x).view(B, S, self.n_heads, self.head_dim).transpose(1, 2)
        k = self.W_k(x).view(B, S, self.n_kv_heads, self.head_dim).transpose(1, 2)
        v = self.W_v(x).view(B, S, self.n_kv_heads, self.head_dim).transpose(1, 2)

        # 关键：把 n_kv_heads 个 K/V 复制成 n_heads 个（每 n_rep 个 Q 头共享一份）
        k = k.repeat_interleave(self.n_rep, dim=1)  # (B, n_heads, S, head_dim)
        v = v.repeat_interleave(self.n_rep, dim=1)

        scores = q @ k.transpose(-2, -1) / math.sqrt(self.head_dim)
        weights = F.softmax(scores, dim=-1)
        out = weights @ v
        out = out.transpose(1, 2).contiguous().view(B, S, -1)
        return self.W_o(out)

# 对比参数量：GQA 的 W_k/W_v 比 MHA 小
d_model = 512
mha_k = nn.Linear(d_model, d_model)         # MHA: 512×512
gqa_k = nn.Linear(d_model, 8 * 64)          # GQA: 8 KV 头 × 64 = 512→512，但逻辑上只有 8 组
print(f"MHA W_k 参数: {mha_k.weight.numel()}")    # 262144
print(f"GQA W_k 参数: {gqa_k.weight.numel()}")    # 262144? 看配置，这里 dim 一样
# 真正的差异在推理时的 KV cache 头数，不是投影参数

关键看 repeat_interleave 这一步：它把少的 KV 头复制成和 Q 头一样多，之后就和普通 MHA 一样算了。推理时不用真的复制——可以优化成「多个 Q 头直接读同一份 KV」，但训练时为了和已有 MHA kernel 兼容，通常用 repeat。

05 从 MHA「升级」到 GQA 的技巧

一个实用技巧：已经训练好的 MHA 模型，不用从头重训就能转成 GQA。方法是 「mean conversion」——把相邻的 KV 头的权重做平均，合并成一个。

比如 h=32 头想转成 g=8 组（每 4 头合并），就把 Wk 的第 0、1、2、3 头平均成新的第 0 组，依此类推。研究表明这种简单的平均转换，配合少量微调（几亿 token），就能达到接近原生 GQA 的效果。这就是 Uptraining 方法，Llama 2 论文里用过。

Uptraining 流程：①加载 MHA 预训练权重；②把 KV 头按组平均合并；③在原数据上继续预训练少量步骤（约原预训练量的 5%）；④得到质量接近原生 GQA 的模型。

这让已经投入巨资训练的 MHA 模型（如早期的 GPT、Llama 1）能低成本迁移到 GQA，享受推理加速。是工业界非常实用的技术。

06 MHA / MQA / GQA 怎么选

总结一下三种方案的选择逻辑：

🏆

选 MHA 当

追求最高质量，推理成本不是瓶颈（如研究、评测刷榜）。或模型规模小（KV cache 本就不大）。

⚡

选 MQA 当

推理延迟/显存是绝对优先级（如边缘设备、超长上下文）。愿意接受 1~3% 的质量损失。

✅

选 GQA 当（默认推荐）

绝大多数生产场景。质量接近 MHA，推理快很多。2024+ 的开源模型几乎全用它。

📊

组数 g 的选择

经验：g 设为让每头服务 4~8 个 Q 头（n_rep = 4~8）。如 h=32 选 g=4 或 8。

⚠️ GQA 不是万能解

①训练时 GQA 比 MHA 略慢（多一次 repeat 操作）；②g 太小接近 MQA，质量下降；③对小模型（参数 < 1B），KV cache 本就不是瓶颈，GQA 收益有限，反而可能因表达力下降而 hurting。小模型仍常用 MHA。

07 为什么共享 K/V 而不共享 Q

一个常被问的问题：为什么是共享 K、V，而不是共享 Q？答案在于推理时的计算结构。

推理（生成）时，每一步只处理一个新 token：①算这个 token 的 Q、K、V；②把 K、V 存进 cache；③用这个 token 的 Q 去查询 cache 里所有历史 K、V。Q 是当前 token 的、临时的，用完就丢；K、V 是要长期存进 cache 的。

所以：共享 Q 没用（Q 不存 cache，省不了显存）；共享 K、V 才省 cache。这就是为什么优化目标永远指向 K、V。这也是 KV cache 这个名字里只有 K、V 没有 Q 的原因。

★ GQA 的 KV 共享维度对比

MQA 和 GQA 的本质是"多个 query head 共享同一组 K/V"，从而缩小 KV cache。但共享到什么程度，对显存和精度影响差异巨大。下面用代码量化三种方案的 KV cache 体积：

import torch

def kv_cache_size(n_layers, n_kv_heads, seq_len, head_dim, dtype_bytes, name):
    """计算 KV cache 字节数。"""
    # 每 layer: K 和 V 各 (n_kv_heads, seq_len, head_dim)
    bytes_ = 2 * n_layers * n_kv_heads * seq_len * head_dim * dtype_bytes
    return bytes_

# 7B 级模型典型配置
L, H_full, H_gqa, H_mqa, D = 32, 32, 8, 1, 128
SEQ = 32768
DT = 2  # FP16

mha = kv_cache_size(L, H_full, SEQ, D, DT, "MHA")
gqa = kv_cache_size(L, H_gqa,  SEQ, D, DT, "GQA")
mqa = kv_cache_size(L, H_mqa,  SEQ, D, DT, "MQA")

def gb(b): return b / 1024**3

print(f"序列长度 {SEQ}, head_dim={D}, {L} 层, FP16")
print(f"MHA (kv={H_full}): {gb(mha):.2f} GB")
print(f"GQA (kv={H_gqa}):  {gb(gqa):.2f} GB   省 {(1-gqa/mha)*100:.0f}%")
print(f"MQA (kv={H_mqa}):  {gb(mqa):.2f} GB   省 {(1-mqa/mha)*100:.0f}%")

# 模拟 GQA 的 key/value 广播：每 4 个 query head 共享 1 个 kv head
group = H_full // H_gqa
q = torch.randn(H_full, 64, D)      # 32 个 query head
kv = torch.randn(H_gqa, 64, D)      # 8 个 kv head
kv_shared = kv.repeat_interleave(group, dim=0)  # 广播到 32
print(f"\nGQA group={group}: kv heads {H_gqa} -> 共享广播后 {kv_shared.shape[0]} 与 query 对齐")
# GQA 是精度和显存的甜点：MQA 太极端损失精度，MHA 太贵，GQA 省 75% KV 且几乎不掉点

08 本章小结

三种方案对比
  • MHA：h 头全独立，质量最佳，KV cache 最大
  • MQA：KV 共享 1 份，省 h 倍，质量略降
  • GQA：KV 分 g 组，省 h/g 倍，质量接近 MHA
关键洞察
  • 推理显存瓶颈在 KV cache，不在模型参数
  • Q 多样性 > K/V 多样性，故共享 K/V 损失小
  • GQA 是 2024+ 开源大模型的事实标准
  • MHA 可通过 Uptraining 低成本转 GQA
