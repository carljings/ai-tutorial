01 项目一：数据分析 Agent

目标：用户上传 CSV/数据库，用自然语言做数据分析（统计/画图/洞察）。

用户：「分析销售趋势，按地区画图」
Agent 理解意图 → 生成 pandas/SQL 代码
沙箱执行代码 → 得到结果/图表
Agent 解读结果 → 输出洞察
图 60-1 · 数据分析 Agent 架构
data_analyst.py —— 核心 Loop
"""
data_analyst.py —— 自然语言数据分析 Agent
用 Agent Loop + 代码执行（PEV）做数据分析
"""
import pandas as pd
from zhipuai import ZhipuAI
client = ZhipuAI()

SYSTEM = """你是数据分析师。用户会给你数据分析需求。
你用 pandas 写代码分析，每次只写一段可执行代码。
我会执行并把结果返回，你再解读。
可用：df（已加载的 DataFrame）、matplotlib。
"""

def analyze(df: pd.DataFrame, question: str):
    # 把 df 的 schema 告诉模型
    schema = df.dtypes.to_string()
    messages = [
        {"role": "system", "content": SYSTEM + f"\n数据列：\n{schema}"},
        {"role": "user", "content": question},
    ]
    for _ in range(10):
        resp = client.chat.completions.create(model="glm-5.2", messages=messages, tools=TOOLS)
        msg = resp.choices[0].message
        messages.append(msg)
        if not msg.tool_calls:
            return msg.content
        for tc in msg.tool_calls:
            # 在沙箱执行模型写的代码（关键：安全沙箱！）
            code = json.loads(tc.function.arguments)["code"]
            result = safe_exec(code, {"df": df})  # safe_exec 用 exec 但限制权限
            messages.append({"role": "tool", "tool_call_id": tc.id, "content": str(result)[:2000]})
⚠️ 代码执行安全

模型生成的代码要执行，必须在沙箱里（Docker 容器/受限 exec），禁止文件/网络/系统操作。否则用户可能被恶意代码攻击。

02 项目二：AI 写作助手

目标：帮用户写文章/邮件/报告。支持续写、改写、润色、格式化。

✍️

核心功能

续写/改写/润色/摘要/翻译/格式化/大纲生成。

🏗️

架构

直连调用（第 41 章模式①）+ 微调品牌风格 + 流式输出。

writing_assistant.py —— 写作助手核心
MODES = {
    "续写": "请续写以下内容，保持风格一致：\n{text}",
    "润色": "请润色以下文字，使其更专业流畅，但不改变原意：\n{text}",
    "改写": "请用{style}风格改写：\n{text}",
    "摘要": "请用 3 句话总结：\n{text}",
}
def assist(mode, text, style="正式"):
    prompt = MODES[mode].format(text=text, style=style)
    resp = client.chat.completions.create(model="glm-5.2",
        messages=[{"role":"user","content":prompt}], stream=True)
    return resp  # 流式返回

03 项目三：智能客服系统

目标：自动回答客户咨询，基于产品文档 + 工单历史，复杂问题转人工。

客户问题
意图分类（简单/复杂/情绪化）
简单→RAG 直答
复杂→Agent 深查
情绪→安抚+转人工
带引用的回答 / 转人工
图 60-2 · 智能客服分流架构

关键设计：

  • 意图路由：先用快模型分类，简单问题不调大模型（省钱）；
  • RAG + 引用：回答必须带出处（第 6/13 章）；
  • 情绪检测：检测到愤怒/焦虑，立即转人工；
  • 人工兜底：无法回答/低置信度时转人工，不能硬编；
  • 工单积累：把人工解决的案例存进 RAG，持续学习。
customer_service.py —— 客服分流核心
def handle_query(query, user_mood):
    # ① 情绪检测：愤怒立即转人工
    if user_mood == "angry":
        return {"action": "transfer_human", "reason": "情绪安抚"}

    # ② 意图分类（用快模型省成本）
    intent = classify_intent(query)   # simple / complex / faq

    if intent == "faq":
        # ③ RAG 直答
        docs = retriever.search(query, top_k=3)
        confidence = docs[0]["score"] if docs else 0
        if confidence < 0.5:
            return {"action": "transfer_human", "reason": "低置信度"}
        answer = generate(query, docs)
        return {"action": "answer", "answer": answer, "refs": docs}

    elif intent == "complex":
        # ④ Agent 深查（多轮检索）
        return agent_handle(query)

    else:
        # ⑤ 兜底
        return {"action": "transfer_human"}

★ 项目四：深度研究 Agent

前三个项目偏「单轮任务」。第四个项目更进一步——做一个能自主多轮检索、阅读、综合的深度研究 Agent：给它一个开放问题（如「2026 年小模型端侧推理有哪些新进展」），它自己拆解子问题、联网检索、读论文、交叉验证、最后产出带引用的结构化报告。这是把本书前 59 章的能力（RAG、Agent Loop、工具调用、CoT）集大成的项目。

"""research_agent.py —— 自主深度研究 Agent（ReAct + 多工具）"""
import json, re
from zhipuai import ZhipuAI

client = ZhipuAI()
TOOLS = {
    "web_search":  lambda q: client.web_search.search(q).search_result,
    "read_url":    lambda u: fetch_and_clean(u),       # 抓网页正文
    "note":        notes.append,                      # 写入研究笔记
}

SYS = """你是深度研究 Agent。对每个问题：
1. Thought: 分析还需要查什么
2. Action: 从 web_search/read_url/note 选一个，给出参数
3. 等待 Observation
最多循环 8 轮，最后用 note 提交带引用的 markdown 报告。
每次 Action 严格用 JSON: {"tool": "...", "args": {...}}"""

def research(question, max_steps=8):
    messages = [{"role":"system", "content":SYS},
                {"role":"user", "content":question}]
    for step in range(max_steps):
        resp = client.chat.completions.create(model="glm-5.2", messages=messages)
        text = resp.choices[0].message.content
        messages.append({"role":"assistant", "content":text})
        m = re.search(r'\{[^{}]*"tool"[^{}]*\}', text, re.S)
        if not m:                       # 没有动作 = 研究结束
            return text
        call = json.loads(m.group(0))
        obs = TOOLS[call["tool"]](*call["args"].values())
        messages.append({"role":"user", "content":f"Observation: {obs[:2000]}"})
    return "已达最大步数，请查笔记。"

report = research("2026 年小模型端侧推理有哪些新进展？给出 3 个代表性技术及来源。")
print(report)

这个 Agent 的精髓在于把「研究」这个模糊任务分解成可执行的工具循环。几个让它真正好用的细节：read_url 要做正文抽取（去掉导航/广告），否则噪声会污染 LLM 判断；note 工具让 Agent 主动积累中间发现，避免在长循环里「失忆」；每条 Observation 要截断（这里取前 2000 字），否则上下文会爆炸。它还能自然地做交叉验证——同一个问题搜多个来源，发现矛盾就再查，这正是「研究」区别于「单次检索」的本质。

★ 项目选型决策矩阵

本书讲了六个项目，但你真正要做产品时该从哪个入手？答案是看你的约束：预算、数据、延迟、合规要求不同，最优起点完全不同。下面这张决策矩阵把六个项目按「上手难度、所需数据、典型 ROI、合规风险」打分（★越多越强），帮你快速对号入座。一个通用经验：从写作助手或客服系统起步——它们数据门槛低、ROI 可量化、风险可控，是验证「AI 是否真能帮业务」的最小可行项目。

★ 这些项目最容易踩的坑

决策矩阵告诉你选什么，但没告诉你选了之后会怎么死。下面把六个项目最常见的失败模式列出来——这些都是真实项目里反复出现的「翻车现场」，提前知道能省几个月弯路。一个共性规律：大多数 AI 项目失败不是模型不行，而是没想清楚验收标准——「更智能」无法度量，「客服首问解决率从 40% 提到 60%」才能度量。

⚠️ 写作助手

坑：追求「万能」导致每个文体都做不精。正解：先只做一种文体（如营销文案），把质量打磨到能替人，再扩展。

⚠️ 客服系统

坑：知识库不更新→模型答旧政策；或检索不到就硬编。正解：建立知识库同步机制 + 明确「无答案时转人工」的兜底。

⚠️ 数据分析 Agent

坑：让模型直接执行生成的代码，等于给了「数据库 root」。正解：代码必须在沙箱跑，只读副本 + 资源限额。

⚠️ 深度研究 Agent

坑：检索源全是 SEO 垃圾，结论看似有理实则站不住脚。正解：限定可信源（官方/学术），强制标注引用并做交叉验证。

⚠️ 编程助手

坑：没有代码库上下文，建议全凭臆想。正解：必须接 RAG 检索项目代码，否则就是「通用教程生成器」。

⚠️ 知识库 RAG

坑：分块太大（答案被稀释）或太小（丢失上下文）。正解：用语义分块 + 重叠窗口，并在评估集上调参。

这六个坑的根因可以归纳成三个词：边界、数据、度量。定义清楚能力边界（不做什么），保证数据质量（喂进去的是什么决定出来的是什么），确立可度量的验收标准（什么叫成功）。这三件事做扎实，AI 项目成功的概率就从「碰运气」变成「可预期」。
