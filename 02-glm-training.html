GLM-5.2 从 0 到 1
一个 2026 年大模型是如何诞生的

这是全书的重头戏之一。我们将以智谱 GLM-5.2（2026 年 6 月发布）为锚点，完整复盘一个当代大模型从「一堆原始网页」到「能写代码、能推理、能跑智能体」的全过程：预训练 → 中期训练 → 对齐 → 智能体后训练。

阅读约 40 分钟 难度 ★★★★ 前置：第 01 章

★ 先认识 GLM 家族

在讲训练之前，先用一条时间线把 GLM 家族对齐。理解它的演进，就理解了过去几年大模型技术的主流路线。

2020

GLM 论文发表

清华团队提出 General Language Model (GLM)，用「自回归空白填充」统一了 BERT（理解）和 GPT（生成）两种范式。这是 GLM 名字的由来，也是它区别于纯 GPT 路线的基因。

2023.03

ChatGLM-6B 开源

首个能在消费级显卡（6GB 显存）上跑的中文对话模型，引爆国内开源生态。INT4 量化让大模型「飞入寻常百姓家」。

2023.10

ChatGLM3 / 工具调用

引入函数调用（Function Calling）、代码解释器，向智能体迈出第一步。

2024

GLM-4 系列

对标 GPT-4 的多模态能力，发布 GLM-4V（视觉）、All Tools、长上下文。开源 GLM-4-9B，性能领先同尺寸。

2025

GLM-4.5 / GLM-4.6 / GLM-5

MoE 架构全面落地，聚焦「智能体工程（Agentic Engineering）」。GLM-5 技术报告《from Vibe Coding to Agentic Engineering》公开了 27 万亿 token 预训练、Muon Split 等关键细节。

2026.06.13 🆕

GLM-5.2 发布

智谱迄今能力最强的开源模型。核心升级：真正可用的 1M（百万）token 上下文、更强的 Agentic Coding（智能体编程）、MIT 协议开源、使用华为昇腾训练。4 个月内的第三次重大迭代。

📌 本章说明

GLM-5.2 是 GLM-5 的快速迭代版本，其独立完整技术报告在发布时尚未公开。本章的训练流程以 GLM-5 技术报告（arXiv 2602.15763）为核心依据，并结合智谱官方文档、媒体报道与公开访谈，合理推断 GLM-5.2 的训练要点。这是理解「一个 2026 年大模型怎么训」最权威的一手资料。

02 第〇步：数据——一切的基础

业内有句话：「数据是新的代码」。模型的差距，80% 来自数据。GLM-5 用了 27 万亿 token 的语料——这是一个天文数字，背后是极其复杂的工程。

数据从哪来

🌐

网页爬取

CommonCrawl、自建爬虫。覆盖广但有大量噪声，需深度清洗。占比最大。

💻

代码

GitHub、GitLab 等代码仓库。对训练推理能力至关重要——GLM-5 在训练初期就重点引入代码与推理数据。

📚

书籍与论文

高质量长文本，是知识密集型和学术能力的主要来源。

🔢

数学与科学

数学题、科学问答、定理证明。训练逻辑推理与符号操作。

🗣️

对话与问答

论坛、问答社区、合成对话。提供对话格式与常识。

🧪

合成数据

用更强的模型生成的高质量数据。2026 年最重要的趋势之一，后续会详述。

数据清洗：决定上限的脏活累活

原始数据里混着广告、乱码、重复、低质、有害、机密内容。清洗流水线通常包括：

去重（Deduplication）

用 MinHash、SimHash 等算法去除完全重复和近似重复的内容。重复数据会让模型「死记硬背」而非泛化。

质量过滤（Quality Filter）

用小模型或规则打分，剔除乱码、过短、广告、模板化内容。常用启发式：困惑度过滤、分类器过滤。

安全与合规

去除有害、涉密、个人信息（PII）。法律红线，必须做。

语言识别与配比

识别语言，按预设比例混合。GLM 是中英双语模型，中文配比显著高于海外模型。

格式归一化

统一编码、清理 HTML、提取正文、规范标点。

⚠️ 数据配比是玄学也是科学

「代码占多少？数学占多少？中文占多少？」这些配比没有公式，只能靠大量实验。GLM-5 的关键经验是：训练初期就大幅增加代码与推理数据的比重，而不是后期才补——这从根本上塑造了模型的「思维骨架」，也是它能成为顶级 Agentic Coding 模型的基因。

数据质量的三层模型

可以用一个金字塔来理解预训练数据：

▲ 高质量合成数据（顶）
▲ 精选书籍 / 论文 / 代码
▲ 清洗后的网页 / 对话
▲ 原始互联网语料（底）

03 第一阶段：预训练（Pretraining）

预训练是整个流程中最贵、最久、最关键的一步。GLM-5 的预训练用了 27 万亿 token，数千张 GPU 跑数月。它的目标只有一个：

「让模型学会预测下一个 token。」

听起来简单到荒谬，但正是这个目标，配合海量数据，让模型涌现出了语言、知识、推理、甚至一定的规划能力。

训练目标：下一个 token 预测

给定一段文本 [t₁, t₂, ..., tₙ]，模型要学习的是：在看到 t₁..tᵢ 的情况下，tᵢ₊₁ 是什么。形式化地，最大化对数似然：

L = − Σᵢ log P(tᵢ₊₁ | t₁, ..., tᵢ ; θ)
🔑 为什么「预测下一个 token」就够了？

要精确预测下一个 token，模型必须理解语法、掌握事实、学会推理。比如要预测「法国的首都是___」的下一个词，它必须「知道」巴黎是法国首都。这个看似简单的任务，逼着模型把整个世界的统计规律压缩进参数里。Ilya Sutskever 曾说：「预测下一个 token，本质上就是在对整个世界建模。」

Muon Split：让训练稳定的黑科技

GLM-5 技术报告里提到一个关键技术：Muon Split。这是一个优化器/数值稳定性相关的技术。

大模型训练最怕的就是不稳定——Loss 突然爆炸、梯度消失或溢出、Attention 分数过大导致数值溢出。一旦发生，整轮训练可能作废，损失以百万美元计。

GLM-5 报告指出：借助 Muon Split，训练过程中 attention logits 的规模始终保持稳定，无需任何裁剪（clipping）。这是个了不起的工程成就，因为：

  • 传统做法靠裁剪 logits 来防爆炸，但裁剪会损失信息、损害模型能力；
  • 能在不裁剪的情况下保持稳定，意味着优化器对训练动力学有更好的控制；
  • 这让模型可以训得更深、更大、更久，而不必担心数值崩溃。
💡 给工程师的直觉

你可以把 Muon Split 类比为「自适应的数值减震器」：它不限制模型的表达空间（不裁剪），而是在优化器层面让每一步的更新幅度恰到好处，避免训练过程「失控」。这类底层优化是顶级实验室的核心 know-how，往往不公开细节，但对最终模型质量影响巨大。

分布式训练：数千张 GPU 如何协同

一个千亿参数模型，单张 GPU 装不下也训不动。必须把模型和数据切分到多张卡上，这是分布式训练的核心难题。常见并行策略：

并行维度切的是什么典型场景
数据并行（DP）每张卡拿不同数据，模型各有一份模型能装下单卡时
张量并行（TP）把单个矩阵乘法切到多卡层内切分，需高速互联
流水线并行（PP）把不同层放到不同卡层数多时
专家并行（EP）不同专家放不同卡MoE 模型专用
序列并行（SP）切分超长序列长上下文训练

实际训练是多种并行组合（3D / 4D / 5D 并行），需要精心设计以平衡计算、通信和显存。GLM-5.2 使用华为昇腾训练，意味着这套分布式栈在国产硬件上跑通了——这本身就是个重大工程突破。

训练损失曲线：长什么样

预训练的 loss 曲线通常是一个平滑下降的幂律曲线，但有几个特征值得记住：

  • 前期快速下降：模型在学最基础的词频和语法；
  • 中期缓慢下降：在学更复杂的知识和推理；
  • 能力涌现（Emergence）：某些能力（如多位数加法、指令遵循）会在某个 loss 阈值后突然出现，而非渐进提升——这是大模型最迷人的现象之一；
  • Loss spike（尖刺）：偶尔会突然跳上去再回落，往往意味着数据里有异常或优化器状态波动，是工程师最紧张的瞬间。

04 第二阶段：中期训练（Mid-training）

预训练结束后，得到的是一个「知识渊博但不会对话」的基座模型。你给它「法国首都是」，它能补「巴黎」；但你问它「请告诉我法国的首都」，它可能会继续编「请告诉我法国的首都和主要城市……」。中期训练就是让它开始「像样」。

GLM-5 技术报告明确提到中期训练阶段进行了上下文扩展等能力增强。主要工作包括：

📏

上下文长度扩展

预训练通常在 4K~8K 上下文做。中期训练逐步扩展到 32K → 128K → 1M。技术手段：继续预训练（继续用长文本 + RoPE 频率调整）、DSA（见下节）。

🧩

能力专项强化

针对推理、代码、数学等高价值能力，做持续预训练（Continual Pretraining），注入高质量专项数据。

🌐

多语言扩展

增加多语言数据配比，提升跨语言能力。

🔗

格式注入

引入对话、指令、Markdown、JSON 等结构化格式数据，让模型「认识」这些格式。

DSA：长上下文的关键技术

GLM-5 引入了 DSA 技术（技术报告中明确提及），目标是：在确保长文本处理高度可靠的同时，大幅降低训练与推理开销。这是 GLM-5.2 能做到「真正可用 1M token」的基石。

为什么长上下文这么难？回顾第 1 章的 KV Cache：

⚠️ 长上下文的三重地狱
  • 显存爆炸：KV Cache 随长度线性增长，1M token 的 KV Cache 对大模型可能是几百 GB；
  • 计算爆炸：Attention 是 O(n²) 复杂度，1M² = 10¹² 次运算；
  • 可靠性下降：上下文越长，模型越容易「忘」中间内容（Lost in the Middle 现象），在中间位置的信息召回率显著下降。

行业里解决这三重地狱的典型思路（DSA 类技术通常综合了其中若干）：

  • 稀疏注意力 / 稀疏 KV：不要求每个 token 关注所有历史，只关注「重要的」一部分；
  • 滑动窗口 + 全局锚点：局部密集注意 + 少量全局 token 做信息枢纽；
  • KV 压缩 / 量化：把 KV Cache 压缩或低比特量化存储；
  • 分块 / 分层注意力：层次化地组织注意力。
💡 GLM-5.2 的「1M 真正可用」意味着什么

很多模型宣称「支持 128K/1M 上下文」，但实际测试中，把关键信息放在中间位置时召回率会暴跌。GLM-5.2 强调「真正可用」，意味着它在 1M 全长度、信息任意位置都有稳定的高召回率——这是 DSA 等技术打磨的结果，也是它能用于真实代码库级编程（整仓库上下文）的前提。

05 第三阶段：对齐（SFT + RLHF）

经过中期训练，模型已经有能力了，但它不知道你想让它怎么用这份能力。它可能啰嗦、可能答非所问、可能有偏见、可能拒绝你的合理请求。对齐（Alignment）就是让模型的行为符合人类意图与价值观。

SFT：监督微调

SFT（Supervised Fine-Tuning）是最直接的对齐手段。做法是构造高质量的「指令-回答」对，让模型学着模仿：

SFT 训练样本示例（JSONL 格式）
{
  "messages": [
    {"role": "system",    "content": "你是一个简洁的编程助手。"},
    {"role": "user",      "content": "Python 里怎么反转字符串？"},
    {"role": "assistant", "content": "s[::-1] 即可。例如 s[::-1] 对 'abc' 返回 'cba'。"}
  ]
}

SFT 数据的质量远比数量重要。业界共识：几千条精标数据可能胜过几十万条低质数据。LIMA 论文甚至证明 1000 条高质量样本就能让模型具备良好的指令遵循能力。

RLHF：基于人类反馈的强化学习

SFT 之后，模型会「按格式回答」了，但回答的好坏怎么衡量？不同回答哪个更好？这就需要 RLHF。它的三段式：

训练奖励模型（Reward Model）

收集大量「同一个问题的两个回答 + 人类标注哪个更好」的数据，训练一个能给任意回答打分的模型（RM）。RM 学会了「人类的偏好」。

强化学习优化（通常是 PPO）

让 SFT 模型生成回答，用 RM 打分，再用强化学习算法（如 PPO）更新模型参数，让它倾向于生成高分回答。

防止「奖励黑客」（Reward Hacking）

模型可能找到 RM 的漏洞刷分（比如生成长但空洞的内容）。用 KL 散度约束，限制模型不能偏离 SFT 版本太远。

DPO：RLHF 的简化替代

PPO 训练复杂、不稳定、显存占用大。DPO（Direct Preference Optimization）直接用偏好数据优化模型，省去了显式的奖励模型和强化学习循环，更稳定、更简单，成为 2024-2026 年的主流选择之一。GLM 系列也大量采用这类直接偏好优化方法。

⚠️ 对齐税（Alignment Tax）

对齐过程可能让模型在某些能力上变弱——比如 RLHF 后代码能力下降。这是著名的「对齐税」。顶级团队会通过精心设计的数据配比、能力保留训练来对冲。GLM-5 强调 ARC（Agentic/Reasoning/Coding）能力，意味着对齐时特别小心不要损害这三项。

06 第四阶段：智能体后训练（Agentic Post-training）

这是 GLM-5/5.2 区别于普通对话模型的核心差异化。技术报告标题就叫《from Vibe Coding to Agentic Engineering》——「从随性编程到智能体工程」。

普通对话模型只能「一问一答」，而智能体模型要能：

  • 理解一个复杂的多步任务，并自己拆解；
  • 调用工具（搜索、代码执行、文件操作）；
  • 观察工具返回的结果，决定下一步；
  • 在出错时自我纠正；
  • 持续工作直到任务完成。

从对话数据到轨迹数据

对齐阶段用的是「单轮/多轮对话」，而智能体后训练用的是轨迹数据（Trajectories）——一条轨迹是一串「思考-行动-观察」的完整执行过程：

一条智能体轨迹的简化示意
# 任务：找出当前目录下最大的文件

[Thought]  我需要先列出所有文件及其大小，可以用 ls -la 或 du。
[Action]   run_shell(command="du -ah . | sort -rh | head -5")
[Observation]  128M  ./data/large.parquet
                64M  ./models/baseline.pt
                ...

[Thought]  最大的是 data/large.parquet，128M。任务完成。
[Action]   finish(answer="data/large.parquet (128M)")
[Observation]  任务完成。

用大量这样的高质量轨迹做 SFT 和偏好优化，模型就学会了「智能体行为模式」。这也是为什么后面的 Harness、Agent Loop 如此重要——模型的智能体能力需要匹配的运行时来发挥。

在 Harness 里训练

LangChain 的《The Anatomy of an Agent Harness》指出一个关键趋势：Claude Code、Codex 这类产品会「在 Harness 里」做后训练。也就是说，不只是用静态轨迹数据，而是让模型在真实的 Harness 环境（带工具、带反馈）里反复执行任务、收集反馈、再优化。

这就形成了一个飞轮：Harness 越好 → 收集到的训练数据越好 → 模型在 Harness 里表现越好 → Harness 又能做得更复杂。GLM-5.2 强化 Agentic Coding，正是踩在这个飞轮上。

08 合成数据：2026 的最大变量

2026 年大模型训练最显著的变量，是合成数据（Synthetic Data）的全面崛起。原因很简单：

「高质量人类数据快被用完了。」

于是大家开始用更强的模型生成数据来训更强的模型。这条「自我提升」路线的关键是质量控制：

  • 生成：用顶级模型按模板生成大量任务和解法；
  • 验证：对代码/数学这类可验证的任务，用执行器/形式化验证自动过滤错误的；
  • 去重与多样化：避免合成数据高度同质化导致模型过拟合；
  • 难度过滤：只保留「略难于当前模型能力」的数据，最大化学习效率。
💡 可验证奖励的强化学习（RLVR）

对于代码、数学这类有标准答案的任务，可以跳过人类标注：让模型生成多个解法，用验证器判断对错，直接用对错做奖励信号训模型。这是 2025-2026 推理模型（如 o1/R1 类）的核心技术之一，也是合成数据飞轮的关键一环。

09 昇腾训练：国产算力的里程碑

GLM-5.2 一个特别值得记录的细节：使用华为昇腾（Ascend）训练。这不仅是技术选型，更是一个产业信号。

过去几年，顶级大模型几乎清一色在 NVIDIA GPU（A100/H100/Blackwell）上训练。昇腾能跑通千亿 MoE、27T token 级别的训练，意味着：

  • 国产高端 AI 芯片的硬件能力已达到训练顶级模型的门槛；
  • 围绕昇腾的软件栈（CANN、MindSpore、各类并行框架适配）已足够成熟；
  • 中国大模型团队有了在制裁下持续迭代的算力保障。

当然，挑战仍在：生态成熟度、工具链丰富度、与 CUDA 生态的差距，都需要时间补齐。但 GLM-5.2 的发布证明这条路走得通。

11 数学基石：训练背后的公式

前面讲了「做什么」，这一节补上「为什么」——把训练过程中最核心的几个公式摆出来。理解这些数学，你才能真正读懂 GLM-5 技术报告里的每一句话。

下一个 token 预测的完整数学

给定语料 x₁, x₂, ..., x_T，模型参数为 θ，因果语言模型的训练目标是最大化对数似然（等价于最小化负对数似然损失）：

L(θ) = − (1/T) · Σt=1..T log P(xt | x<t; θ)

其中 P(xt | x<t) 由模型最后一层 softmax 给出：

P(xt = w) = exp(zw) / Σw' exp(zw')

这里 z = ht · ET 是隐状态与词嵌入矩阵的内积（LM Head），ht 是第 t 个位置经过所有 Transformer 层后的输出向量。整个梯度通过 softmax 回传到模型的每一个参数。

Scaled Dot-Product Attention 的推导

Attention 为什么要除以 √dk？这是初学者最常问的问题。推导如下：

  • 设 Q、K 各分量独立、均值为 0、方差为 1；
  • 点积 Q·K = Σi=1..dk QiKi，方差为 dk（每个乘积方差为 1，共 dk 项）；
  • 当 dk 很大（如 64、128）时，点积的绝对值会很大；
  • softmax 对大值极其敏感：最大值会几乎独占概率，梯度趋近 0（梯度消失）；
  • 除以 √dk 把方差拉回 1，让 softmax 工作在合理区间。
Attention(Q,K,V) = softmax( QKT / √dk ) V

MoE 路由的数学

Top-K MoE 路由器是一个线性层加 softmax。给定隐状态 h、N 个专家，路由器输出选择 Top-K 个专家：

g(x) = softmax( Wgate · h )    # 路由权重
G(x) = KeepTopK(g(x), k)    # 只保留 Top-K，其余置 0
y(x) = Σi∈TopK G(x)i · Ei(x)    # 加权求和

为防止路由器偷懒（只用少数专家），加一个负载均衡损失：

Lbalance = α · N · Σi fi · Pi
fi = 该专家被选中的 token 比例  |  Pi = 该专家的平均路由概率

当所有专家均匀分担时，fi = Pi = 1/N，乘积最小。任何不均衡都会增大这个损失，从而驱动路由器均衡分配。

DPO 的推导（直接偏好优化）

RLHF 的 PPO 需要训练奖励模型 + 强化学习，复杂。DPO 的洞察是：可以跳过显式奖励模型，直接从偏好数据推导出最优策略。核心等式：

LDPO(θ) = − E(x,yw,yl) [ log σ( β log( πθ(yw|x) / πref(yw|x) ) − β log( πθ(yl|x) / πref(yl|x) ) ) ]

其中 yw 是偏好（赢）的回答，yl 是不偏好（输）的回答，πθ 是当前策略，πref 是参考策略（通常是 SFT 后的模型），β 是 KL 约束强度，σ 是 sigmoid。直观上：这个损失让模型提高赢的回答相对参考的概率，降低输的回答相对参考的概率，而 β 防止偏离太远。

12 Scaling Laws：规模化是科学，不是堆料

2020 年 OpenAI 的 Scaling Laws 论文揭示了一个深刻规律：模型的损失随参数量 N、数据量 D、算力 C 呈幂律下降，且可以精确预测。这是「大力出奇迹」的理论基础，也是 GLM-5 敢于用 27 万亿 token 训练的依据。

L(N) = (Nc/N)αN    L(D) = (Dc/D)αD    L(C) = (Cc/C)αC
典型值：αN≈0.076, αD≈0.095, αC≈0.05

Chinchilla：最优算力分配

DeepMind 的 Chinchilla 论文修正了 Scaling Laws 的一个关键结论：给定固定算力预算，参数量和数据量应该按大致 1:20 的比例同时增长（token 数约为参数数的 20 倍）。很多早期大模型「参数大、数据少」是次优的。GLM-5 用 27 万亿 token 训练千亿级模型，正是遵循了这个原则——数据量要配得上参数量。

模型参数 Ntoken DD/N评价
GPT-3175B300B1.7严重欠训练
Chinchilla70B1.4T20最优
Llama 270B2T28过训练（换推理效率）
GLM-5千亿级27T高充分训练 + 推理优化

过训练（Over-training）：用数据换推理速度

Llama 2/3 故意把 D/N 提到 20 以上（「过训练」），用更多训练算力换更小的推理模型：同样能力，参数更少，推理更快更便宜。这是 2024-2026 年的主流策略——训练成本一次性，推理成本持续发生，所以宁可训练时多花，也要推理时省。GLM-5.2 的 MoE 设计也是同一逻辑：总参数大（容量大），激活参数小（推理快）。

13 亲手写：一个最小可训练的语言模型

讲了这么多原理，不如亲手训一个。下面是一个真正能在 CPU 上跑起来的最小 GPT——它有 Tokenizer、Embedding、Self-Attention、训练循环。虽然只有几万参数，但骨架和 GLM-5.2 完全一致。理解它，就理解了大模型训练的「最小内核」。

mini_gpt.py —— 一个能在 CPU 上训练的最小 GPT（PyTorch）
"""
mini_gpt.py —— 最小可训练 GPT
理解这个文件 = 理解大模型训练的最小内核
依赖：pip install torch
"""
import torch
import torch.nn as nn
import torch.nn.functional as F

# ---------- 超参数 ----------
VOCAB   = 100      # 词表大小（演示用，真实是 10w+）
D_MODEL = 64       # 隐状态维度
N_HEAD  = 4        # 注意力头数
N_LAYER = 2        # Transformer 层数
CTX     = 32       # 上下文长度
HEAD    = D_MODEL // N_HEAD   # 每个头的维度 = 16

# ---------- 单个注意力头 ----------
class Head(nn.Module):
    def __init__(self):
        super().__init__()
        self.q = nn.Linear(D_MODEL, HEAD, bias=False)
        self.k = nn.Linear(D_MODEL, HEAD, bias=False)
        self.v = nn.Linear(D_MODEL, HEAD, bias=False)
        self.register_buffer("tril", torch.tril(torch.ones(CTX, CTX)))

    def forward(self, x):
        B, T, C = x.shape
        q = self.q(x); k = self.k(x); v = self.v(x)        # (B,T,HEAD)
        wei = q @ k.transpose(-2, -1) * (HEAD ** -0.5)   # 缩放点积
        wei = wei.masked_fill(self.tril[:T,:T]==0, float("-inf"))  # 因果掩码
        wei = F.softmax(wei, dim=-1)
        return wei @ v                                          # (B,T,HEAD)

# ---------- 多头注意力 ----------
class MultiHead(nn.Module):
    def __init__(self):
        super().__init__()
        self.heads = nn.ModuleList([Head() for _ in range(N_HEAD)])
        self.proj  = nn.Linear(D_MODEL, D_MODEL)
    def forward(self, x):
        out = torch.cat([h(x) for h in self.heads], dim=-1)
        return self.proj(out)

# ---------- 一个 Transformer Block = Attention + MLP ----------
class Block(nn.Module):
    def __init__(self):
        super().__init__()
        self.attn = MultiHead()
        self.ln1  = nn.LayerNorm(D_MODEL)
        self.mlp  = nn.Sequential(nn.Linear(D_MODEL, 4*D_MODEL), nn.GELU(), nn.Linear(4*D_MODEL, D_MODEL))
        self.ln2  = nn.LayerNorm(D_MODEL)
    def forward(self, x):
        x = x + self.attn(self.ln1(x))   # 残差 + LayerNorm（Pre-Norm）
        x = x + self.mlp(self.ln2(x))
        return x

# ---------- 完整的 GPT ----------
class MiniGPT(nn.Module):
    def __init__(self):
        super().__init__()
        self.tok_emb = nn.Embedding(VOCAB, D_MODEL)
        self.pos_emb = nn.Embedding(CTX, D_MODEL)        # 简单的可学习位置编码
        self.blocks  = nn.Sequential(*[Block() for _ in range(N_LAYER)])
        self.ln_f    = nn.LayerNorm(D_MODEL)
        self.head    = nn.Linear(D_MODEL, VOCAB)         # LM Head

    def forward(self, idx, targets=None):
        B, T = idx.shape
        pos  = torch.arange(T, device=idx.device)
        x    = self.tok_emb(idx) + self.pos_emb(pos)     # token + 位置
        x    = self.blocks(x)
        x    = self.ln_f(x)
        logits = self.head(x)                            # (B,T,VOCAB)
        if targets is None:
            return logits, None
        loss = F.cross_entropy(logits.view(-1, VOCAB), targets.view(-1))
        return logits, loss

# ---------- 训练循环（这就是预训练的最小形态） ----------
torch.manual_seed(42)
model = MiniGPT()
opt   = torch.optim.AdamW(model.parameters(), lr=1e-3)

# 假数据：一段重复的模式，让模型学会预测
data = torch.randint(0, VOCAB, (1000, CTX + 1))   # 1000 条样本

for step in range(500):
    ix  = torch.randint(0, len(data), (32,))        # batch=32
    batch = data[ix]
    x, y  = batch[:, :-1], batch[:, 1:]              # 输入 / 目标错开一位
    logits, loss = model(x, y)
    opt.zero_grad()
    loss.backward()                                     # 反向传播
    opt.step()                                          # 更新参数
    if step % 50 == 0:
        print(f"step {step:3d}  loss {loss.item():.3f}")
💡 强烈建议跑一遍

把这个文件存下来，pip install torch 后运行。你会看到 loss 从 ~4.6 一路下降——这就是预训练最核心的画面。然后试着改：把 N_LAYER 调大、CTX 调大，看 loss 降得更低；把数据换成真实文本，它就开始学语言了。这个 80 行的程序，和 GLM-5.2 的训练循环在结构上没有本质区别——只是规模差了千万倍，加上了 MoE、分布式、对齐等工程。

14 分布式训练深度：ZeRO 与混合并行

千亿模型一张卡装不下，必须切分。除了第 3 节提到的并行维度，最关键的优化是 ZeRO（Zero Redundancy Optimizer），它解决了数据并行里「每张卡都存完整优化器状态/梯度/参数」的冗余。

ZeRO 的三个阶段

阶段切分什么显存节省通信代价
ZeRO-1优化器状态（如 Adam 的 m、v）4x低
ZeRO-2+ 梯度8x中
ZeRO-3+ 参数N 倍（N=卡数）高

ZeRO-3 几乎能把一个超大模型均匀摊到所有卡上，代价是每次前向/反向都要跨卡实时收集（gather）参数，通信量大。实际中常把 ZeRO + 张量并行 + 流水线并行 组合成 3D 并行，在通信和显存间找平衡。GLM-5.2 在昇腾上的训练栈，本质就是这套并行策略的国产硬件适配版。

通信原语：AllReduce / AllGather / ReduceScatter

分布式训练的核心是集合通信：

  • AllReduce：所有卡把各自的梯度加起来，结果每张卡都有一份（数据并行的核心）；
  • AllGather：每张卡把自己的分片收集成完整张量（ZeRO-3 前向时用）；
  • ReduceScatter：先求和再分片（ZeRO-2 反向时用）。

这些操作跑在高速互联上（NVLink、InfiniBand、昇腾的 HCCS），带宽直接决定训练效率。通信与计算的重叠（一边算一边传）是优化的关键技巧。

15 合成数据：代码实现一个自我提升飞轮

第 8 节讲了合成数据的理念，这里给一个可运行的合成数据生成 + 质量过滤流水线，用于训练一个「会解数学题」的模型：

synthetic_data.py —— 合成数据生成与过滤流水线
"""
synthetic_data.py —— 用强模型生成数学题解法，再用验证器过滤
这是 RLVR / 自我提升飞轮的最小实现
"""
import re, random
from zhipuai import ZhipuAI
client = ZhipuAI()

# ---------- ① 生成：用强模型给题目写多种解法 ----------
def generate_solutions(problem, n=5):
    solutions = []
    for _ in range(n):
        resp = client.chat.completions.create(model="glm-5.2", messages=[
            {"role": "system", "content": "你是数学老师。逐步解题，最后用 '答案：X' 给出数值答案。"},
            {"role": "user", "content": problem},
        ], temperature=0.9)   # 高 temperature 保证多样性
        solutions.append(resp.choices[0].message.content)
    return solutions

# ---------- ② 验证：抽答案 + 判对错（可验证任务的核心） ----------
def extract_answer(text):
    m = re.search(r"答案[：:]\s*([-\d./]+)", text)
    return float(m.group(1)) if m else None

def verify(solution, ground_truth):
    pred = extract_answer(solution)
    return pred is not None and abs(pred - ground_truth) < 1e-6

# ---------- ③ 过滤：只保留正确的、多样的、略难的 ----------
def filter_solutions(problem, solutions, gt):
    kept = [s for s in solutions if verify(s, gt)]     # 正确性过滤
    # 去重：相似度高的去掉（这里用简单哈希演示）
    seen, unique = set(), []
    for s in kept:
        key = hash(s[:50])
        if key not in seen:
            seen.add(key); unique.append(s)
    return unique

# ---------- ④ 完整飞轮：生成 → 过滤 → 沉淀为训练数据 ----------
def self_improve_loop(problems_with_answers, rounds=3):
    dataset = []
    for r in range(rounds):
        print(f"\n🔄 第 {r+1} 轮自我提升")
        for problem, gt in problems_with_answers:
            sols  = generate_solutions(problem, n=5)
            good  = filter_solutions(problem, sols, gt)
            dataset.extend([("user", problem), ("assistant", g) for g in good])
        print(f"   已积累 {len(dataset)//2} 条高质量样本")
    return dataset

# 跑起来：给一批题，自动生成训练数据
problems = [("3x + 5 = 20，求 x", 5.0), ("12 的因数有几个？", 6.0)]
data = self_improve_loop(problems, rounds=2)
# data 现在可以直接拿去做 SFT

这就是 GLM-5 这类模型强化推理能力的底层引擎：用模型自己生成解法，用可验证的正确性做过滤，把过滤后的高质量数据拿去训练——模型变强后又能生成更好的数据，形成飞轮。关键在于「可验证」：数学有标准答案、代码能跑测试，所以可以完全自动化；开放式任务则更难。

16 训练稳定性：为什么训练会爆炸

千亿模型训练最大的工程噩梦是不稳定。一个 Loss spike 可能让几周训练白费。常见原因与对策：

问题原因对策
Loss spike（尖刺）异常数据、梯度爆炸、学习率过大梯度裁剪、异常数据检测、回滚 checkpoint
Attention logits 爆炸logits 过大导致 softmax 饱和Muon Split（GLM-5 的做法，无需裁剪即可稳定）、logit 软裁剪
梯度消失/爆炸深网络、初始化不当残差连接、LayerNorm/Pre-Norm、合理初始化（Xavier/He）
发散（Loss → NaN）数值溢出、学习率过高混合精度损失缩放、降低学习率、bf16 代替 fp16
MoE 路由坍缩路由器只用少数专家负载均衡损失、专家容量限制
🔑 为什么 GLM-5 的「无需裁剪」很重要

裁剪（clipping）是一种「牺牲信息换稳定」的手段——它硬把超出范围的值截断，但截断意味着信息丢失，长期会损害模型能力。GLM-5 的 Muon Split 能在不裁剪的前提下保持 attention logits 稳定，意味着它既稳又不损失信息——这是底层优化器设计的胜利，也是模型质量能领先的关键之一。

17 推理优化：训练完之后的另一半战场

训练只是开始，推理才是成本大头。一个模型部署后，每次调用都要算钱。2026 年的推理优化技术栈：

⚡

FlashAttention

重排 Attention 计算，减少 HBM 读写，2-4x 加速。已是标配。

📦

PagedAttention / vLLM

像 OS 虚拟内存一样管理 KV Cache，显存利用率大幅提升，并发请求翻倍。

🔢

量化（Quantization）

fp16 → int8 → int4，显存减半再减半。GLM 系列有成熟的 INT4 量化版。

✂️

推测解码（Speculative Decoding）

用小模型猜几个 token，大模型批量验证，无损加速 2-3x。

🎯

MoE 天然加速

GLM-5.2 的 MoE：总参数大但激活参数小，单次推理只算一小部分。

🌬️

蒸馏与剪枝

把大模型的能力压缩进小模型，端侧部署。ChatGLM-6B 的 INT4 版就是经典案例。

18 训练一个模型的完整检查清单

把整章串起来，给你一份「从头训一个模型」的检查清单（生产级）：

数据层

中期训练

智能体后训练

评估

部署

量化 + KV Cache 优化 + 推理引擎（vLLM）+ 监控

19 Benchmark 复盘：那些跑分到底意味着什么

2026 年看一个模型强不强，绕不开一堆 benchmark 缩写：MMLU、GSM8K、HumanEval、SWE-bench、Terminal Bench……它们各自测什么？分数怎么解读？这一节给你一份「跑分翻译手册」。

综合知识类

Benchmark测什么解读要点
MMLU57 个学科的多选题（文理科常识）分数高 = 知识广，但不代表会推理。已被新题污染，看 MMLU-Pro 更可靠。
MMLU-ProMMLU 升级版，更难、10 选项更抗刷题，区分度更高。
C-Eval / CMMLU中文综合知识中文模型必看。GLM 系列强项。
BBHBig-Bench Hard，复杂推理任务合集测「硬核推理」，区分度大。

数学与代码类（区分度最大）

Benchmark测什么要点
GSM8K小学应用题太简单，顶级模型接近满分，区分度低。看 MATH。
MATH竞赛数学真正考验推理。推理模型优势明显。
HumanEval164 道 Python 编程题单函数级，偏简单。看 SWE-bench。
LiveCodeBench实时更新的编程题防数据污染，更公平。

智能体类（2026 最重要）

Benchmark测什么要点
SWE-bench Verified修真实 GitHub issue最接近「AI 工程师」能力的测试。GLM-5 在此开源 SOTA。
Terminal Bench终端任务（运维、命令行）测真实工程操作能力。
WebArena网页操作（购物、查信息）测 Computer-Using Agent。
GAIA通用 AI 助手（多步推理+工具）综合智能体能力。
⚠️ 看 benchmark 的三个陷阱
  • 数据污染：题目可能进了训练集，分数虚高。看「新题」benchmark 更可靠；
  • 刷题效应：模型可能针对 benchmark 优化，实际体验不一定好；
  • 单点不代表全部：MMLU 高不等于会写代码。要看与你的场景相关的 benchmark。

这也是 GLM-5.2 发布时不急于发跑分表的原因——真实智能体任务的表现才是硬通货。

怎么读一张模型对比表

看到一张 benchmark 对比表，问自己四个问题：

这些 benchmark 与我的场景相关吗？

做客服的看 MMLU 没用，该看 RAG/对话类。

有没有数据污染嫌疑？

分数突然飙升、远超同级别模型，要警惕。

对比公平吗？

同参数量级？同 prompt？同评测方式？

有没有真实用户反馈佐证？

Arena 盲测、社区口碑比跑分更可信。

20 训练日志解读：那些数字在告诉你什么

训练大模型时，工程师盯着一系列指标看。读懂它们，就读懂了训练的「健康状态」：

一份典型的预训练日志（简化）
step  1000  loss 3.214  grad_norm 0.82  lr 3.0e-4  tokens/sec 125k  mem 78%
step  2000  loss 2.856  grad_norm 0.91  lr 3.0e-4  tokens/sec 124k  mem 78%
step  3000  loss 2.634  grad_norm 1.12  lr 2.9e-4  tokens/sec 126k  mem 78%
step  3100  loss 4.201  grad_norm 8.74  lr 2.9e-4  tokens/sec  88k  mem 82%  ⚠️ spike!
step  3200  loss 2.701  grad_norm 0.95  lr 2.9e-4  tokens/sec 125k  mem 78%  ✅ 恢复
step  5000  loss 2.412  grad_norm 0.78  lr 2.7e-4  tokens/sec 125k  mem 78%
...
step 50000  loss 1.892  grad_norm 0.65  lr 1.0e-5  tokens/sec 125k  mem 78%
指标含义健康表现异常信号
loss模型预测的负对数似然平滑下降突然飙升（spike）= 异常数据或梯度爆炸
grad_norm梯度范数（梯度大小）稳定在 0.5-2突然变大 = 训练不稳定，可能要回滚
lr当前学习率按 schedule 变化（warmup→cosine 衰减）不按计划 = 配置错
tokens/sec训练吞吐稳定（受硬件上限约束）下降 = 有瓶颈（通信/IO）
mem显存占用稳定且有余量飙升 = 内存泄漏或 batch 过大
⚠️ Loss Spike 处置流程
  1. 不要慌，先观察：很多 spike 是单次异常数据，下一步会恢复；
  2. 持续 spike：检查最近喂的数据有没有异常（乱码、超长序列）；
  3. grad_norm 爆炸：降低学习率、加强梯度裁剪；
  4. loss 不回来：回滚到上一个 healthy checkpoint，从那里重训；
  5. 根因分析：事后分析 spike 时的数据和参数，防止复发。

能力涌现：loss 与能力的非线性关系

训练日志里最迷人的现象是能力涌现：某些能力（如指令遵循、多位数运算）在 loss 降到某个阈值后突然出现，而非渐进提升。

loss 3.0 → 不会做多位数加法
loss 2.5 → 还是不会
loss 2.3 → 突然会了！（涌现）
loss 2.1 → 越做越好
图 2-5 · 能力涌现：某些能力在阈值后突然出现

这意味着：loss 下降不等于能力线性增长。一个模型可能在 loss 2.5 时看起来啥都不会，到 2.3 时突然「开窍」。这也是为什么训练大模型要坚持训下去——临界点往往就在坚持的几步之后。

21 训练成本拆解：钱花在哪了

训一个千亿模型要花几千万到上亿美元。钱花在哪？大致拆解：

环节占比说明
GPU/昇腾算力~60%几千张卡跑几个月，最大头
数据工程~15%采集、清洗、标注、合成
人力（研究员/工程师）~15%顶级人才昂贵
评估与迭代~5%benchmark、红队、用户测试
基础设施~5%存储、网络、监控

一个粗略估算公式：

训练成本 ≈ 6 × 参数量 × 训练 token 数 × (每 FLOP 价格)
例：175B 参数 × 300B token ≈ 3.5×10²³ FLOPs ≈ 数百万美元（A100 价位）

GLM-5 用 27 万亿 token 训千亿模型，算力消耗是天文数字——这也是为什么能做这件事的团队全球屈指可数，以及为什么昇腾能用、能省钱对国产大模型意义重大。

「下一代模型的差距，不在参数大小，而在数据工程、训练稳定性、对齐艺术、智能体后训练这四件事的精细度上。模型本身正在变便宜，围绕模型的工程正在变贵、变关键。」——这，正是我们下一章要讲的 Harness。
