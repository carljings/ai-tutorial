Softmax 变体
线性、稀疏、PAL 与 Latent Attention

标准 softmax 注意力的 O(n²) 复杂度，是长上下文的硬天花板。Flash Attention 用工程手段省了显存，但计算量仍是平方级。要从根上解决，必须改 softmax 本身。本章梳理四条路线：稀疏注意力（只算重要的，跳过无关的）、线性注意力（用核函数把 softmax 换成可交换的线性运算，复杂度降到 O(n)）、Perceiver 的 PAL（把序列压缩成少量 latent token）、以及 DeepSeek 的 MLA（压缩 KV 维度）。每条路都有取舍。

阅读约 28 分钟 难度 ★★★★★ 前置：第 94、101 章

★ O(n²) 的诅咒

把前面章节的结论再强调一次：标准注意力 softmax(QK^T/√d_k)V 的计算量和显存都是 O(n²)（n 是序列长度）。序列翻倍，开销变 4 倍。

n=4K → 分数矩阵 16M 元素；n=32K → 1B 元素；n=128K → 16B 元素。光存这个矩阵就要几十 GB。

Flash Attention 解决了「存不下」（显存 O(n)），但计算量还是 O(n²·d)。序列足够长时，算力也吃不消。要从根上突破，必须减少需要计算的注意力对数，或者用数学技巧绕开 softmax 的不可交换性。这就是本章几条路线的出发点。

▼ 四条突破路线
稀疏：只算「重要」的对，跳过无关的
线性：核函数替代 softmax，O(n)
PAL：序列压缩成少量 latent token

01 稀疏注意力：只算重要的

观察真实注意力的权重矩阵，会发现大部分权重集中在少数几个对上，其余接近 0。也就是说，n×n 个对里，真正「有信息」的可能只有一小部分。稀疏注意力的思路：只算这些重要的对，跳过无关的。

怎么决定哪些对重要？几种策略：

  • 局部窗口（Local/Window）：每个 token 只关注前后 w 个 token（如 w=512）。语言里大部分依赖是局部的，这个简单策略就能砍掉大部分计算。Longformer、BigBird 用它。
  • 分块（Blockwise）：把序列切成块，只在块内做完整注意力，块间用少量「全局 token」连接。
  • 可学习稀疏（如 Routing/Reformer）：用哈希或聚类，让「可能相关」的 token 分到一组，只在组内算。
  • 动态稀疏：先粗算一遍分数，只对 top-k 高分的对做精确注意力。

GPT-3 训练时用了交替的稀疏 + 稠密注意力：部分层稠密（保证表达力），部分层稀疏（省计算）。Longformer 则是局部窗口 + 少量全局 token 的组合，能处理 32K+ 文档。

⚠️ 稀疏的代价：信息丢失

稀疏注意力假设「无关的对可以忽略」，但这不总是对的。有些长距离依赖（比如文档开头的主语和结尾的代词）权重虽小但语义关键，稀疏掉就丢了。而且稀疏模式不规则时，GPU 难以高效并行（不规则访存），实际加速往往打折。这是稀疏路线没有全面取代标准注意力的原因。

02 线性注意力：用核函数绕开 softmax

这是数学上最优雅的路线。标准注意力的瓶颈在 softmax——它让 QK^T 必须先算完（不可分解）。线性注意力的洞察：把 softmax 替换成一个可分解的核函数 φ，就能利用矩阵乘的结合律换序，把 O(n²) 降到 O(n)。

推导如下。标准注意力：

Attn(Q,K,V) = softmax(QK^T)V ≈ [ Σ_j exp(q_i·k_j) · v_j ] / [ Σ_j exp(q_i·k_j) ]

把 exp(q·k) 替换成核函数 φ(q)·φ(k)（exp 是 Mercer 核的一种，但计算上不可分解；换成 ELU+1、ReLU 等就成了可分解的线性核）：

线性注意力 = [ Σ_j φ(q_i)·φ(k_j)·v_j ] / [ Σ_j φ(q_i)·φ(k_j) ]
  = φ(q_i)·[ Σ_j φ(k_j)⊗v_j ] / [ φ(q_i)·Σ_j φ(k_j) ]

关键：方括号里的 Σ_j φ(k_j)⊗v_j 和 Σ_j φ(k_j) 只依赖 K、V，不依赖 Q。可以预先算好（一个 d×d 矩阵）。于是计算变成：对每个 q_i，做几次矩阵乘，不需要 Q×K^T 这个 n×n 矩阵。复杂度从 O(n²d) 降到 O(nd²)。

方案复杂度质量代表
标准 softmaxO(n²d)最好所有主流 LLM
线性注意力（ELU/ReLU 核）O(nd²)明显下降Linformer, Performer
Performer（FAVOR+）O(nd²) 用随机特征近似 softmax下降Google

问题：质量下降明显。softmax 的指数放大效应是注意力表达力的关键，用线性核替代后，「软选择」退化成「平均」，区分度大减。在短序列上甚至比标准注意力还慢（d² 项开销）。所以线性注意力在 LLM 主流里没普及，更多用在超长序列、对质量要求不极端的场景（如长视频、生物序列）。

linear_attention.py —— 线性注意力实现
import torch
import torch.nn.functional as F

def linear_attention(Q, K, V):
    """线性注意力：用 ELU+1 作为核函数，复杂度 O(nd^2)。
    利用矩阵乘结合律避免 n×n 矩阵。
    """
    # φ(x) = ELU(x) + 1，保证非负（核函数要求）
    phi_Q = F.elu(Q) + 1   # (B, n, d)
    phi_K = F.elu(K) + 1   # (B, n, d)

    # 关键：先算 K^T V (d×d)，再乘 Q
    # 标准是 Q (K^T V)，线性是 (Q K^T) V —— 换序省掉 n×n
    KV = phi_K.transpose(-2, -1) @ V   # (B, d, d)
    Z = phi_Q @ (phi_K.sum(dim=1, keepdim=True).transpose(-2, -1))  # 归一化项
    output = (phi_Q @ KV) / (Z + 1e-6)
    return output

# 对比复杂度
n, d = 8192, 64
Q = torch.randn(1, n, d); K = torch.randn(1, n, d); V = torch.randn(1, n, d)
# 标准：n×n = 8192² ≈ 67M 元素的中间矩阵
# 线性：只有 d×d = 4096 元素，省了 1.6 万倍中间内存
out = linear_attention(Q, K, V)
print("线性注意力输出形状:", out.shape)

03 PAL：把序列压成 latent token

Perceiver（DeepMind 2021）走的是另一条路：不去优化注意力的实现，而是减少要 attend 的 token 数。它引入一组固定的「latent token」（比如 256 或 512 个），让原始输入序列（可能几万个 token）和这组 latent 做 cross-attention，把信息压缩进 latent。后续层只在 latent 之间做注意力。

Perceiver: 输入 (n 个 token) × latent (m 个) → cross-attention → 压缩成 m 个 latent
后续层：m 个 latent 之间做 self-attention，复杂度 O(m²)，与原始 n 无关

m 通常远小于 n（比如 256 vs 10000），所以后续计算极便宜。代价是：信息被「有损压缩」进 m 个 latent，细节会丢失。Perceiver 在多模态（图像、音频、点云这种超长输入）里很有用——把几十万像素压成几百 latent，让 Transformer 能处理。

类似的思路在 Set Transformer（ISAB）、Funnel-Transformer 里也有。本质都是「用一组小规模的锚点 token 去汇总大规模输入」。

04 MLA：压缩 KV 维度（DeepSeek）

DeepSeek-V2 引入了 MLA（Multi-head Latent Attention），是近年来最有创意的注意力变体之一。它的思路：不压缩 token 数，而是压缩 KV 的特征维度。

具体做法：①把每个 token 的 K、V 向量压缩成一个低维的 latent 向量 c（比如从 d 维压到 d_c 维，d_c 远小于 d）；②存进 KV cache 的只是这个低维 latent；③需要用时，从 latent 解压（上投影）回完整的 K、V。

MLA: K = W_K^U · (W_K^D · x)，V = W_V^U · (W_V^D · x)
cache 只存 W_K^D · x 和 W_V^D · x（低维 latent），不存完整 K、V

效果：KV cache 显存大幅下降（DeepSeek-V2 比 MHA 省 ~93%，比 GQA 还省），同时效果接近 MHA。DeepSeek-V2 / V3 凭借 MLA 实现了极高的推理效率，是国产大模型里工程创新的代表。

📌 MLA 和 GQA 的区别

GQA 是「减少 KV 的头数」（横向压缩）；MLA 是「减少 KV 的维度」（纵向压缩）。两者正交，理论上可以叠加。MLA 的精妙在于压缩后的 latent 能在需要时无损地（通过学习的上投影）恢复出 K、V，比简单量化更优雅。代价是推理时多一次上投影矩阵乘，但相对省下的 cache 显存，这点开销微不足道。

06 为什么 softmax 还没被取代

看上面表格会发现一个反直觉的事实：2026 年主流大模型仍然用标准 softmax + Flash Attention，几乎没有用线性、稀疏、PAL 这些「看起来更先进」的方案。为什么？

  • softmax 的表达力难以替代：指数放大让注意力能做「明确的软选择」，这是语言理解的关键。所有替代方案都有质量损失，而在追求极致质量的 LLM 竞赛里，哪怕 1% 的损失都难以接受。
  • Flash Attention 已经足够好：它让标准注意力在 32K~128K 上下文下可行（显存 O(n)、计算虽 O(n²) 但常数小），削弱了换方案的动机。
  • GPU 硬件偏爱稠密矩阵乘：标准注意力是规整的 GEMM，Tensor Core 极度优化。稀疏/线性方案的不规则访存反而跑不满算力。
  • 工程成熟度：标准注意力 + Flash Attention 的工具链（CUDA kernel、量化、分布式）极其成熟，新方案的生态远跟不上。
💡 趋势：混合而非取代

真正在生产里落地的，往往是混合方案：大部分层用标准 softmax + Flash Attention（保证质量），少数层用稀疏或线性（拉长上下文）。比如 GPT-3 训练时稠密稀疏交替，GLM-4-Long 在超长上下文用滑动窗口 + 全局 token。MLA 这种「保持 softmax 但压缩 KV」的思路，因为不牺牲质量，反而比纯线性/稀疏更有前景。

07 代码：实现一个滑动窗口稀疏注意力

sliding_window.py —— 局部窗口稀疏注意力
import torch
import torch.nn.functional as F
import math

def sliding_window_attention(Q, K, V, window_size=512):
    """滑动窗口稀疏注意力：每个 token 只 attend 前后 window 个 token。
    用 mask 实现稀疏（实际生产用 block-sparse kernel 更高效）。
    """
    B, H, N, D = Q.shape
    scores = Q @ K.transpose(-2, -1) / math.sqrt(D)  # (B, H, N, N)

    # 构造窗口 mask：位置 (i,j) 在 |i-j| <= window 内为 1
    positions = torch.arange(N, device=Q.device)
    rel = (positions.unsqueeze(0) - positions.unsqueeze(1)).abs()  # (N, N)
    mask = (rel <= window_size).float().view(1, 1, N, N)

    scores = scores.masked_fill(mask == 0, float('-inf'))
    weights = F.softmax(scores, dim=-1)
    return weights @ V

# 对比：标准 vs 稀疏在长序列上的中间矩阵规模
N = 8192
# 标准：完整 N×N mask，8192² = 67M 元素
# 稀疏（window=512）：每行只有 ~1024 个非零，但用 dense mask 实现仍占 N×N
# 真正省内存要用 block-sparse kernel，这里只是演示逻辑
Q = torch.randn(1, 8, N, 64); K = torch.randn(1, 8, N, 64); V = torch.randn(1, 8, N, 64)
out = sliding_window_attention(Q, K, V, window_size=256)
print("稀疏注意力输出:", out.shape)

注意：上面用 dense mask 实现只是演示逻辑，实际中间矩阵还是 N×N。真正的稀疏注意力要用 block-sparse kernel（如 Triton 写的）跳过 0 块的计算。这就是为什么稀疏注意力工程实现复杂——数学简单，但要榨出性能需要大量 kernel 工程。

★ 稀疏 Softmax 的 PyTorch 实现

稀疏 Softmax 变体（Sparsemax、entmax15）的核心思想是允许输出概率严格为 0，而不是像标准 softmax 那样所有项都 >0。下面是 Sparsemax 的 PyTorch 实现，带投影和梯度：

import torch
import torch.nn.functional as F

def sparsemax(logits: torch.Tensor, dim: int = -1) -> torch.Tensor:
    """Sparsemax：把 softmax 输出变成稀疏的（部分概率严格为 0）。
    通过把低于阈值的 logit 投影到 0 实现，梯度只流过非零项。
    参考 Martins & Astudillo (2016)。
    """
    logits = logits.transpose(dim, -1)
    original_shape = logits.shape
    logits = logits.reshape(-1, logits.size(-1))
    n = logits.size(-1)

    # 1. 排序降序
    sorted_logits, _ = torch.sort(logits, descending=True, dim=-1)
    # 2. 累加求阈值 z：k(1+sorted_k) - cumsum > 0 的最大 k
    cumsum = torch.cumsum(sorted_logits, dim=-1)
    k = torch.arange(1, n + 1, device=logits.device, dtype=logits.dtype)
    cond = (k * sorted_logits - cumsum) > 0
    k_max = cond.sum(dim=-1, keepdim=True).clamp(min=1)
    # 3. 取前 k 个的均值作为阈值
    tau = (cumsum.gather(-1, k_max - 1) - 1) / k_max
    # 4. 投影：logit - tau，负的截断为 0
    out = torch.clamp(logits - tau, min=0)

    out = out.reshape(original_shape).transpose(dim, -1)
    return out

# 对比 softmax vs sparsemax
logits = torch.tensor([[2.0, 1.0, 0.5, -1.0, -3.0]])
sm = F.softmax(logits, dim=-1)
sp = sparsemax(logits, dim=-1)

print("logits   :", logits.tolist()[0])
print("softmax  :", [round(x, 3) for x in sm.tolist()[0]])
print("sparsemax:", [round(x, 3) for x in sp.tolist()[0]])
print("softmax 非零项:", (sm > 0).sum().item(),
      " sparsemax 非零项:", (sp > 0).sum().item())
# sparsemax 把概率集中在少数项上（很多严格为 0），
# 用在多标签/选择任务上比 softmax 更"果断"，减少标签间的平均化

import numpy as np
from dataclasses import dataclass
from typing import List, Optional

@dataclass
class PerformanceMetrics:
    latency_ms: float
    throughput_ops: float
    memory_mb: float
    accuracy: float

def benchmark(fn, iterations=100):
    """通用性能基准测试"""
    import time
    t0 = time.time()
    for _ in range(iterations):
        result = fn()
    elapsed = time.time() - t0
    return PerformanceMetrics(
        latency_ms=elapsed * 1000 / iterations,
        throughput_ops=iterations / elapsed,
        memory_mb=0,  # 实际中用 psutil 获取
        accuracy=0.0  # 根据具体任务计算
    )

# 示例：对比两种方案的性能
def approach_a(data):
    """方案A：直接计算"""
    return np.softmax(data @ data.T / np.sqrt(data.shape[-1]), axis=-1)

def approach_b(data):
    """方案B：分块计算（省内存）"""
    chunk_size = 64
    results = []
    for i in range(0, data.shape[0], chunk_size):
        chunk = data[i:i+chunk_size]
        results.append(np.softmax(chunk @ data.T / np.sqrt(data.shape[-1]), axis=-1))
    return np.vstack(results)

data = np.random.randn(512, 768)
metrics_a = benchmark(lambda: approach_a(data))
metrics_b = benchmark(lambda: approach_b(data))
print(f"方案A: {metrics_a.latency_ms:.1f}ms, {metrics_a.throughput_ops:.0f} ops/s")
print(f"方案B: {metrics_b.latency_ms:.1f}ms, {metrics_b.throughput_ops:.0f} ops/s")

常见问题FAQ

Q: 如何选择合适的方案？ 先用方案A跑通验证正确性，再根据实际瓶颈（内存/延迟/精度）切换到方案B或C。永远不要过早优化。

Q: 生产环境最大的坑是什么？ 过度工程化——在需求不明确时投入大量精力做复杂架构。建议MVP先行，收集反馈后迭代。

Q: 如何评估效果？ 定义明确的成功指标（准确率/延迟/成本），用golden set做回归测试，A/B测试验证改进效果。至少100个样本才有统计显著性。
