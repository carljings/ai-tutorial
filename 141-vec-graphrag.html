GraphRAG
知识图谱构建、社区摘要与微软架构

向量 RAG 擅长"找相关片段"，但遇到"总结整篇文档的主题"、"对比 A 和 B 在多个维度的差异"这类需要跨文档、全局性推理的问题就力不从心——它本质是局部检索。GraphRAG（微软 2024 年开源）用知识图谱 + 社区摘要补足这个短板，是 RAG 在"全局理解"方向上的重要突破。本文拆解它的知识图谱构建、社区检测、分层摘要，以及微软的完整架构。

考虑一个查询："总结这份 500 页财报里公司的主要战略主题。" 向量 RAG 的做法是 embed 这个查询，召回最相似的若干段落，拼给 LLM。问题在于：

  • "主要主题"是全局性信息，分散在整份文档的几百个段落里，召回 top-10 远不能覆盖。
  • 查询的 embedding 和"主题总结"类段落不一定相似（查询是任务，段落是事实），召回会偏。
  • 没有跨段落的结构化关系，LLM 拿到的是孤立片段，难以归纳全局。

这类"query-focused summarization over whole corpus"（对整个语料库做聚焦总结）的任务，是向量 RAG 的结构性盲区。GraphRAG 就是为这类场景设计的。

★ 第一步：用 LLM 从文本抽取知识图谱

GraphRAG 的第一阶段是离线构建知识图谱。对每个文本块，用一个 prompt 让 LLM 抽取其中的实体（人、组织、地点、概念）和关系（实体之间的动作/属性）。比如对一段财报文本：

"Acme Corp 2024 年云服务营收 30 亿美元，同比增长 60%，主要受益于企业客户增长。"

LLM 抽取出：

  • 实体：Acme Corp（组织）、云服务（业务）、企业客户（客户群）。
  • 关系：(Acme Corp, 提供, 云服务)、(云服务, 2024 营收, 30 亿)、(云服务, 同比增长, 60%)、(云服务, 增长驱动, 企业客户)。

对所有文本块重复这个过程，把所有实体和关系汇总成一张图。注意这里要实体消歧——不同块里的"Acme"、"Acme Corp"、"Acme Corporation"要合并成同一个节点（用 LLM 或字符串相似度）。

# GraphRAG 知识图谱抽取的 prompt 骨架（简化版）
ENTITY_EXTRACTION_PROMPT = """你是一个信息抽取助手。从下面的文本中抽取实体和关系。

输出严格的 JSON 格式：
{{
  "entities": [
    {{"name": "实体名", "type": "组织/人物/地点/概念/业务", "description": "简短描述"}}
  ],
  "relationships": [
    {{"source": "实体A", "target": "实体B", "relation": "关系描述", "strength": 1-10}}
  ]
}}

文本：{text}

抽取结果："""

import json, anthropic
client = anthropic.Anthropic()

def extract_entities_relations(text, model="claude-3-5-sonnet-latest"):
    resp = client.messages.create(
        model=model, max_tokens=1000,
        messages=[{"role": "user",
                   "content": ENTITY_EXTRACTION_PROMPT.format(text=text)}],
    )
    return json.loads(resp.content[0].text)

# 对所有文本块抽取，构建图
import networkx as nx
graph = nx.Graph()
for chunk in corpus_chunks:
    result = extract_entities_relations(chunk)
    for e in result["entities"]:
        graph.add_node(e["name"], **e)
    for r in result["relationships"]:
        graph.add_edge(r["source"], r["target"], **r)
print(f"图: {graph.number_of_nodes()} 节点, {graph.number_of_edges()} 边")

★ 第二步：社区检测 + 分层摘要

有了知识图谱，GraphRAG 用图聚类算法（Leiden 算法）把图分成若干社区（community）——社区内节点连接密集、社区间连接稀疏。直观上，每个社区代表一个"主题簇"，比如"财务表现"主题、"产品线"主题、"高管团队"主题。

Leiden 算法是递归的，可以分多层：顶层社区大而粗（如 3-5 个大主题），下层社区小而细（每个大主题下若干子主题）。GraphRAG 默认构建 3 层左右的社区层级。

对每个社区，GraphRAG 用 LLM 生成一份社区摘要（community report）——把这个社区所有实体和关系汇总给 LLM，让它写一段主题概述。顶层社区的摘要覆盖面广，底层社区的摘要细节多。这一步是 GraphRAG 的精髓：把零散的实体和关系压缩成层次化的语义摘要，查询时直接用这些摘要回答全局性问题。

# 社区检测 + 摘要生成（用 networkx + leidenalg）
import community as community_louvain   # pip install python-louvain
# 或用 leidenalg

def detect_communities(graph, resolution=1.0):
    """Louvain/Leiden 社区检测，返回 {node: community_id}"""
    partition = community_louvain.best_partition(graph, resolution=resolution)
    return partition

def generate_community_report(graph, community_nodes, level, model="claude-3-5-sonnet-latest"):
    """为每个社区生成摘要报告"""
    # 收集社区内所有实体和关系
    entities_info = []
    for node in community_nodes:
        attrs = graph.nodes[node]
        entities_info.append(f"- {node}({attrs.get('type','')}): {attrs.get('description','')}")
    rels_info = []
    for u, v, d in graph.subgraph(community_nodes).edges(data=True):
        rels_info.append(f"- {u} --{d.get('relation','')}--> {v}")

    prompt = f"""你在为一个知识图谱社区生成摘要报告（第 {level} 层）。
这个社区包含以下实体：
{chr(10).join(entities_info[:50])}

主要关系：
{chr(10).join(rels_info[:80])}

请写一份结构化的社区摘要（200-500字），涵盖：
1. 这个社区的核心主题
2. 关键实体及其角色
3. 主要关系和趋势
"""
    resp = client.messages.create(
        model=model, max_tokens=800,
        messages=[{"role": "user", "content": prompt}])
    return resp.content[0].text

# 构建多层社区 + 摘要
partition = detect_communities(graph)
communities = {}
for node, cid in partition.items():
    communities.setdefault(cid, []).append(node)

reports = {}
for cid, nodes in communities.items():
    reports[cid] = generate_community_report(graph, nodes, level=0)
print(f"生成 {len(reports)} 份社区摘要")

★ 第三步：两种查询模式

构建好图谱和社区摘要后，GraphRAG 支持两种查询模式，按 query 类型选择：

Global Search（全局搜索）—— 回答全局/总结类问题

对于"总结主题"、"对比趋势"这类全局问题，GraphRAG 把所有顶层社区摘要分批发给 LLM，让每个 batch 的 LLM 基于"自己看到的社区摘要"对 query 给出部分答案，再用 map-reduce 把所有部分答案合并成最终答案。

这个流程绕过了向量检索的局部性——它不召回个别段落，而是基于预先做好的全语料摘要回答。这正是它擅长全局问题的原因。

Local Search（局部搜索）—— 回答具体实体问题

对于"Acme Corp 的 CEO 是谁"这类具体实体问题，GraphRAG 先定位相关实体节点，取出该节点的邻居、所在社区的摘要、相关文本块，拼成上下文喂 LLM。这类似向量 RAG，但用了图谱的结构信息（实体的直接关系）补足语义检索的盲区。

# Global Search 的简化流程
def global_search(query, community_reports, model="claude-3-5-sonnet-latest"):
    """map-reduce: 每个 batch 处理部分社区摘要，再合并"""
    # 1. Map: 把社区摘要分 batch，每个 batch 让 LLM 给部分答案
    batch_size = 5
    partial_answers = []
    for i in range(0, len(community_reports), batch_size):
        batch = community_reports[i:i+batch_size]
        summaries = "\n\n".join([f"社区摘要{j}:\n{r}" for j, r in enumerate(batch)])
        prompt = f"""基于这些社区摘要回答问题。如果你觉得某些摘要有用就用，没用就忽略。
        只输出对你回答问题有帮助的要点，格式为要点列表。如果没有相关信息输出"无"。
        
        问题：{query}
        社区摘要：{summaries}
        """
        resp = client.messages.create(
            model=model, max_tokens=500,
            messages=[{"role": "user", "content": prompt}])
        if "无" not in resp.content[0].text:
            partial_answers.append(resp.content[0].text)

    # 2. Reduce: 合并所有部分答案
    final_prompt = f"""根据这些来自不同社区的部分答案，综合回答问题。
    问题：{query}
    部分答案：{chr(10).join(partial_answers)}
    """
    resp = client.messages.create(
        model=model, max_tokens=1000,
        messages=[{"role": "user", "content": final_prompt}])
    return resp.content[0].text

answer = global_search("总结这份财报的主要战略主题", list(reports.values()))

★ 微软 GraphRAG 的工程架构

微软开源的 GraphRAG（microsoft/graphrag）把上述流程工程化，主要组件：

  • Indexing pipeline：文本 → 切块 → LLM 抽取实体/关系 → 实体消歧 → 图构建 → Leiden 社区检测 → 多层社区摘要生成 → 写入 LanceDB（向量）+ 图存储。这是离线、一次性的（更新时增量）。
  • Query engine：支持 global/local 两种模式，按 query 路由。
  • 配置驱动：YAML 配置 chunks / 抽取 prompt / 社区层级 / LLM 选择，便于调优。

成本是 GraphRAG 的主要顾虑：构建索引要给每个文本块调一次 LLM（抽取）+ 给每个社区调一次 LLM（摘要），一份 1000 块的文档可能要几百到几千次 LLM 调用。微软用 prompt caching、小模型抽取、增量更新来控成本。生产前务必先在小样本上估算成本。

# 微软 GraphRAG 的使用流程（命令行）
pip install graphrag

# 1. 初始化项目
graphrag init --root ./my-graphrag

# 2. 配置 settings.yaml（LLM、数据源、chunks 参数）
# 编辑 ./my-graphrag/settings.yaml

# 3. 构建索引（离线，会调用大量 LLM）
graphrag index --root ./my-graphrag
# 产物：output/ 目录下有 entity/relationship/community 报告 + 向量库

# 4. 全局查询
graphrag query --root ./my-graphrag \
    --method global \
    --query "总结文档的主要主题"

# 5. 局部查询
graphrag query --root ./my-graphrag \
    --method local \
    --query "Acme Corp 的 CEO 是谁"

★ 什么时候用 GraphRAG

  • 全局总结类查询多：主题归纳、趋势分析、对比类问题占比高。
  • 多跳推理：答案需要跨多个文档、沿关系链推理（如"A 的供应商的总部在哪"）。
  • 语料相对静态：GraphRAG 索引构建贵，频繁更新不划算。
  • 预算充足：能承担构建索引的 LLM 成本。

反之，事实问答为主、语料频繁变动、预算敏感，传统向量 RAG 更合适。两者也可以混合：向量 RAG 处理具体事实查询，GraphRAG 处理全局/多跳查询，按 Router 路由。

★ GraphRAG 的成本控制与增量更新

GraphRAG 的最大障碍是成本——构建索引要给每个文本块调 LLM 抽取实体、给每个社区调 LLM 生成摘要，一份 10 万块的语料可能要几十万次 LLM 调用，成本几千到几万美元。这让 GraphRAG 在很多场景"理论好用实际用不起"。下面是几个控成本的实战手段。

分层抽取。不是所有文本块都值得用最强模型抽取实体。用一个轻量分类器先判断"这块是否信息密集"，密集的（如财报正文）用 Sonnet 抽取，稀疏的（如页眉页脚、过渡句）用 Haiku 或直接跳过。这能砍掉 40-60% 的抽取成本。

实体缓存。同一实体在不同块里反复出现（如"Acme Corp"出现几百次），它的描述抽取一次后缓存复用。第二次遇到只验证不重新抽取。配合 prompt caching，相关实体上下文也缓存。

社区摘要的批量生成。把多个小社区的实体关系合并成一个大 batch，一次 LLM 调用生成多份摘要（让 LLM 输出 JSON 列表）。这比逐个社区调用省 N 倍调用次数。

增量更新是长期控本的关键。GraphRAG 不必每次数据变动都全量重建——新文档来了，只抽取新文档的实体关系，合并进已有图，对受影响的社区重新生成摘要（而非全部社区）。这把更新成本从 O(全库) 降到 O(增量)。

# GraphRAG 增量更新
def incremental_update(new_docs, existing_graph, existing_reports):
    # 1. 只抽取新文档的实体关系
    new_entities, new_rels = extract_from_docs(new_docs)
    # 2. 合并进已有图（含消歧）
    affected_nodes = merge_into_graph(existing_graph, new_entities, new_rels)
    # 3. 找到受影响的社区（新节点加入或边变化）
    affected_communities = find_affected_communities(existing_graph, affected_nodes)
    # 4. 只重新生成受影响社区的摘要
    for cid in affected_communities:
        existing_reports[cid] = regenerate_report(existing_graph, cid)
    # 大部分社区不受影响，摘要复用，成本远低于全量重建

组合这些手段，GraphRAG 的成本能降一个数量级，进入可承受范围。但即便如此，GraphRAG 仍比传统向量 RAG 贵得多，所以它只适合"全局理解价值高、值得这个投入"的场景。

★ GraphRAG vs 向量 RAG vs 混合架构的决策

理解了 GraphRAG 的能力和成本，最后的工程问题是：我的场景到底该用哪种？这不是非此即彼，而是按查询类型路由的混合架构。

查询类型与最佳架构的对应：

  • 具体事实查询（"Acme Corp CEO 是谁"）→ 向量 RAG 或 GraphRAG Local Search 都行，向量 RAG 更便宜，优先选。
  • 多跳推理（"Acme 的供应商的总部在哪"）→ GraphRAG Local Search 强于向量 RAG，因为图能沿关系链跳。
  • 全局总结（"总结这份财报的主要战略主题"）→ GraphRAG Global Search 完胜，向量 RAG 几乎做不了。
  • 对比分析（"对比 A 和 B 在多个维度的差异"）→ GraphRAG Global + 向量 RAG 混合，前者给全局后者补细节。
  • 精确关键词（"GPT-4o 错误码 E001"）→ 向量 RAG 或 BM25，GraphRAG 不擅长精确匹配。

混合架构是大型系统的常态。一个 Router 判断查询类型，分流到不同引擎：简单事实走向量 RAG（快、便宜），多跳走 GraphRAG Local（中等成本），全局总结走 GraphRAG Global（贵但能做），关键词走 BM25。每个查询用最合适的工具，整体性价比最高。

混合架构
                    ┌─ 简单事实 ──▶ 向量 RAG (快、便宜)
                    │
   query ──▶ Router ┼─ 多跳推理 ──▶ GraphRAG Local (中等)
                    │
                    ├─ 全局总结 ──▶ GraphRAG Global (贵)
                    │
                    └─ 精确关键词 ─▶ BM25 (最快)

落地节奏。不要一上来就上混合架构——复杂度太高。建议：第一阶段只上向量 RAG，覆盖 80% 查询；第二阶段发现"全局总结类查询效果差"，再加 GraphRAG Global 作为补充；第三阶段按 Router 路由。增量演进比一步到位更务实。

GraphRAG 的适用门槛。GraphRAG 不是所有团队都该用。它的前提是：① 查询里有相当比例的全局/多跳类（>20%）；② 语料相对静态（频繁更新成本高）；③ 预算充足（能承担索引构建）；④ 有工程能力维护图数据库和社区检测。不满足这些，老老实实用向量 RAG + rerank 效果更好、成本更低。

这个决策框架能帮你避免"为了用 GraphRAG 而用 GraphRAG"的陷阱。技术选型的核心是匹配场景，而非追逐新潮。

★ GraphRAG 构建的工程细节：实体消歧与质量保障

GraphRAG 构建里最棘手的工程问题是实体消歧和质量保障，处理不好图谱就是垃圾。

实体消歧的挑战。LLM 从不同块抽取的实体，同一个现实实体可能有不同名字："Acme Corp"、"Acme Corporation"、"Acme"、"ACME"。如果都当作不同节点，图会碎片化，社区检测失败。消歧要做：

  • 别名归并：用字符串相似度 + LLM 判断"这两个名字是不是同一实体"。高相似度的合并。
  • 上下文消歧：同名不同实体（如两个叫"John Smith"的人）要用上下文（所在公司、时间）区分。
  • 实体链接：链接到知识库（如 Wikidata）的标准实体，用标准 ID 消歧。
# 实体消歧：字符串相似 + LLM 验证
def disambiguate_entities(entities):
    """合并指向同一现实实体的别名"""
    merged = {}
    for ent in entities:
        name = ent["name"]
        # 1. 找已有实体里相似的
        candidates = [n for n in merged if string_similarity(n, name) > 0.85]
        if candidates:
            # 2. LLM 判断是否同一实体
            best = max(candidates, key=lambda n: string_similarity(n, name))
            if llm_confirm_same_entity(name, best, ent["description"], merged[best]["description"]):
                merged[best]["aliases"].append(name)
                continue
        merged[name] = {**ent, "aliases": [name]}
    return list(merged.values())

抽取质量的保障。LLM 抽取实体关系会出错（幻觉出不存在的关系、漏掉重要实体）。几个保障手段：

  • 多轮抽取 + 投票：对同一块用 LLM 抽取多次，只保留多次都抽到的实体/关系（高置信）。
  • schema 约束：给 LLM 明确的实体类型 schema（组织/人物/地点/概念），减少类型混乱。
  • 交叉验证：A 块说"X 是 Y 的子公司"，B 块说"Y 收购了 X"——这两个关系应该指向同一事实，交叉验证能发现矛盾。

社区检测的调参。Leiden 算法的 resolution 参数控制社区粒度：高 resolution → 小社区（细粒度），低 resolution → 大社区（粗粒度）。GraphRAG 通常建多层：顶层用低 resolution（3-5 个大主题），下层用高 resolution（每主题若干子主题）。层数和 resolution 要按数据规模调，没有通用最优值。

图谱质量评估。构建完怎么知道图谱好不好？几个指标：① 连通性——最大连通分量占比，低说明有孤岛；② 度数分布——好的图谱度数相对均匀，差的有明星节点（度数极高）；③ 社区模块度（modularity）——Leiden 的优化目标，高说明社区划分清晰；④ 实体覆盖率——重要实体是否都被抽取（人工抽查）。

# 图谱质量评估
import networkx as nx
def evaluate_graph_quality(graph):
    # 连通性
    largest_cc = max(nx.connected_components(graph), key=len)
    connectivity = len(largest_cc) / graph.number_of_nodes()
    # 度数分布
    degrees = [d for _, d in graph.degree()]
    degree_gini = gini(degrees)   # 基尼系数，高说明不均
    # 模块度（用 community 库算）
    import community
    partition = community.best_partition(graph)
    modularity = community.modularity(partition, graph)
    print(f"连通性: {connectivity:.3f} (越接近1越好)")
    print(f"度数基尼: {degree_gini:.3f} (越低越均匀)")
    print(f"模块度: {modularity:.3f} (越高社区越清晰)")
    return {"connectivity": connectivity, "modularity": modularity}

这些工程细节决定了 GraphRAG 的成败。理论优雅，但"实体消歧错了一半、抽取漏了 30%"的图谱，用起来比纯向量 RAG 还差。所以 GraphRAG 不是"调个库就能用"，它需要认真的数据工程和质量把控。这也是为什么 GraphRAG 适合有工程能力的大团队。

★ 本章小结

  • 向量 RAG 的盲区是全局理解：总结主题、跨文档对比、多跳推理这类任务召回局部片段解决不了。
  • GraphRAG 三步走：① LLM 抽取实体/关系构建知识图谱（带消歧）；② Leiden 社区检测把图分成主题簇，多层；③ LLM 为每个社区生成层次化摘要。
  • 两种查询模式：Global Search（map-reduce 所有社区摘要，回答全局问题）；Local Search（图谱定位实体+邻居+社区，回答具体问题）。
  • 微软 GraphRAG 是工程化实现，配置驱动，主要顾虑是索引构建的 LLM 成本（控本用 prompt caching/小模型/增量）。
  • 选型：全局/多跳/静态/预算足用 GraphRAG；事实/动态/敏感预算用向量 RAG；两者可 Router 混合。
