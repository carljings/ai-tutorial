智能体深度实战
从 ReAct 到多智能体协作

前两章我们搭好了 Harness 与 Loop。本章在这之上，讲清智能体的架构范式——ReAct、Plan-and-Execute、Reflection、Multi-Agent；以及让它「记得住事、学得会乖」的记忆系统。看完这章，你就能设计自己的智能体了。

阅读约 35 分钟 难度 ★★★★★ 前置：第 04 章

★ 智能体范式全景

「智能体」不是一种东西，而是一类东西。根据决策结构的不同，主流范式有四种。它们不是互斥的，工业级系统往往是多种范式的组合。

🔄

① ReAct

边想边做，单步推进。最基础，适合大多数任务。

🗺️

② Plan-and-Execute

先规划全局，再逐步执行。适合复杂长任务。

🪞

③ Reflection（反思）

做完后自我审视，发现错误就重来。提升质量。

👥

④ Multi-Agent

多个智能体分工协作，各司其职。处理最复杂任务。

01 Plan-and-Execute：先谋后动

ReAct 是「走一步看一步」，简单任务没问题。但遇到「帮我做一个完整的电商网站」这种大任务，走一步看一步会迷失方向、效率低下。Plan-and-Execute 的思路是：先花一轮专门做全局规划，再按计划逐步执行。

用户：「帮我做一个待办事项 Web 应用」
▼ 规划阶段（Planner）
计划：
1. 初始化项目（Vite + React）
2. 设计数据模型（Todo: id/title/done）
3. 实现列表组件 + 增删改
4. 加 localStorage 持久化
5. 写样式 + 测试
▼ 执行阶段（Executor，逐步）
逐项执行，每步可用 ReAct
▼ 必要时重新规划（Replan）
完成
图 5-1 · Plan-and-Execute：Planner 出计划，Executor 照着做

Planner 与 Executor 分工

关键洞察：规划和执行需要不同的能力，往往由不同（或不同配置的）模型承担。

角色核心能力模型选择
Planner（规划者）任务分解、全局视野、依赖分析强模型（GLM-5.2、Claude Opus 级）
Executor（执行者）单步专注、工具调用、错误处理可快模型（节省成本）
Replanner（重规划）根据执行结果调整计划强模型

动态重规划（Replan）

计划赶不上变化。Executor 执行到第 3 步发现「依赖的库不支持某个功能」，原计划就失效了。聪明的 Plan-and-Execute 会在执行出现偏差时触发重规划，让 Planner 基于新信息重新出计划，而不是死磕原计划。

⚠️ 过度规划的陷阱

有些系统喜欢一开始就规划到极其细致（「第 1 步第 1.1 子步...」），但计划越细越脆弱——前面任何偏差都会让后面所有细节作废。更好的做法是「粗规划 + 滚动细化」：只规划接下来的几步，执行后再规划下几步（类似 Model Predictive Control）。

02 Reflection：让智能体自我进化

人做完事会复盘，智能体也可以。Reflection（反思）范式让智能体在产出后，用一个「自我审视」步骤检查质量，发现问题就修正。这是提升输出质量最简单有效的方法之一。

Reflection 的三种触发时机

🔬

事后反思

完成任务后整体复盘：哪里做得好、哪里可以改进、是否需要返工。

🚨

异常反思

工具失败、测试不过、用户否定时，立刻反思原因并调整。

⏰

定期反思

每 N 步强制反思一次，检查是否偏离目标、是否有更优路径。

Reflection 的极简实现：生成 → 批评 → 修正
def reflect_and_improve(task):
    # ① 初版产出
    draft = llm(f"完成这个任务：{task}")

    # ② 自我批评（换个视角，可以是另一个模型/提示）
    critique = llm(f"审视这个回答，指出 3 个可改进之处：{draft}")

    # ③ 基于批评修正
    final = llm(f"原回答：{draft}\n批评：{critique}\n请给出改进版。")
    return final

就这么几行，质量往往就有可观提升。Reflection 的本质是给模型一个「跳出当前视角」的机会——它生成时是「作者」，批评时是「审稿人」，修正时又是「作者」，角色切换带来更全面的思考。

💡 Reflection 的进阶：自我训练

把「初版 → 反思 → 改进版」的过程作为训练数据，能让模型学会「天生就反思」——这其实就是推理模型（如 o1、R1 类）训练思路的雏形。GLM-5.2 的强推理能力，背后也有这类「隐式反思」的训练。

03 记忆系统：让智能体「记得住事」

没有记忆的智能体，每次对话都从零开始，无法积累经验、无法记住用户偏好。记忆系统是智能体从「一次性工具」变成「长期伙伴」的关键。

记忆的四种类型

类型存什么怎么用
工作记忆当前任务的对话、中间结果直接放上下文窗口
情景记忆过去某次具体交互（何时、何地、做了什么）RAG 检索，回忆「上次类似情况」
语义记忆抽象知识和事实（用户偏好、项目规范）结构化存储，作为系统提示的一部分
程序记忆「怎么做某类事」的技能（Skills）按需加载对应 Skill（见第 8 章）

怎么把记忆接进 Loop

用户发新消息
▼
检索相关记忆（向量搜索 + 关键词）
▼ 注入上下文
模型生成回复
▼
判断「这次交互有什么值得记的？」
▼ 写入记忆库（去重、压缩、打标签）
完成，记忆持续增长
图 5-2 · 记忆的读写循环：检索 → 注入 → 生成 → 沉淀

关键技术点是「值得记的」判断。不能把每句话都存（记忆爆炸、噪声多），要让模型自己判断「这条信息未来有用吗？」——只存有用的、抽象过的、去重后的。MemGPT、Letta、智谱的记忆方案都在探索这块。

记忆的遗忘与冲突

⚠️ 记忆不是越多越好
  • 过时记忆：用户三个月前的偏好可能已变，需要时效性管理；
  • 矛盾记忆：「用户喜欢简洁」和「用户要详细解释」可能都存在，需要冲突解决；
  • 隐私边界：有些信息不该长期记忆（如一次性敏感数据），需要遗忘机制。

04 多智能体协作：从单兵到军团

单个智能体再强，也有认知带宽限制。复杂任务往往需要多个专长不同的智能体协作。这就是多智能体（Multi-Agent）系统。

为什么要多个 Agent

🎭

角色专业化

一个 Agent 当「产品经理」写需求，一个当「程序员」写代码，一个当「测试」找 bug。每个角色专注自己的事，质量更高。

🧠

认知减负

每个 Agent 只需要装自己角色相关的上下文，不会被无关信息干扰（上下文隔离）。

⚖️

相互制衡

「执行者」和「审查者」分开，能避免单 Agent 的盲点。代码 Agent 写完，审查 Agent 复核。

⚡

并行加速

多个 Agent 同时干不同的子任务，整体更快。

多智能体的三种协作模式

模式 A：层级（Hierarchical）

一个「主 Agent」拆解任务、派发给「子 Agent」，子 Agent 完成后汇报，主 Agent 汇总。像公司组织架构。

主 Agent（Orchestrator）
▼ 派发
研究 Agent
编码 Agent
测试 Agent
▼ 汇报
主 Agent 汇总 → 输出
图 5-3 · 层级模式：Claude Code、Cursor 都用这个

模式 B：流水线（Pipeline）

多个 Agent 串成一条流水线，前一个的输出是后一个的输入。像工厂车间。

需求 Agent
▶
设计 Agent
▶
编码 Agent
▶
测试 Agent
▶
部署 Agent
图 5-4 · 流水线模式：MetaGPT 的核心思路

模式 C：辩论/讨论（Debate）

多个 Agent 围绕同一问题各抒己见、相互质疑，最后由一个「仲裁 Agent」综合。适合需要多角度思考的任务。

Agent A：激进方案
Agent B：保守方案
Agent C：折中方案
▼ 相互辩论 / 质疑 / 补充（多轮）
仲裁 Agent：综合得出最优解

多智能体的通信协议

多个 Agent 要协作，必须有通信机制。2026 年这块正在标准化：

  • A2A（Agent-to-Agent）：Google 主推的 Agent 间通信协议，让不同框架的 Agent 能互相对话；
  • ACP（Agent Communication Protocol）：另一套候选标准；
  • 共享黑板（Blackboard）：所有 Agent 读写同一个共享状态，简单但耦合度高；
  • 消息队列：异步通信，适合分布式部署的 Agent。
⚠️ 多 Agent 不是银弹
  • 协调成本：Agent 越多，通信开销、出错点越多；
  • 失控风险：多 Agent 可能陷入「互相甩锅」或「死循环讨论」；
  • 调试困难：出问题时，定位是哪个 Agent 的锅很费劲；
  • 成本翻倍：每个 Agent 都要调模型，N 个 Agent 成本近似 N 倍。

原则：能用单 Agent 解决的，别上多 Agent。只有当任务确实需要多种专长、或需要制衡时，才值得引入多 Agent。

把所学综合起来。假设我们要做一个智能体：给它一个 GitHub 仓库地址，让它产出一份「架构分析报告」。设计如下：

▼ 并发派发

▼

07 怎么评估一个智能体

智能体比模型更难评估，因为它的行为是动态的。常用方法：

  • 任务完成率：在一系列真实任务上，多少比例真正完成了；
  • 效率指标：平均步数、平均 token、平均耗时、平均成本；
  • 稳定性：同一任务跑多次，结果一致性如何；
  • 鲁棒性：在工具失败、输入异常时的恢复能力；
  • 人工评审：对输出质量的主观打分（必要时）；
  • LLM-as-Judge：用强模型当裁判自动打分（便宜但需校准）。

评估智能体时，务必建一个「任务集」持续回归测试。每次改动 Harness/Skill/Prompt，都跑一遍任务集，确保没退化。这是工程化的基本功。

08 本章小结

📌 核心要点
  • 四大范式：ReAct（边想边做）、Plan-Execute（先谋后动）、Reflection（自我反思）、Multi-Agent（军团协作）；
  • 记忆分四类：工作/情景/语义/程序，关键是「值得记」的判断；
  • 多 Agent 三模式：层级、流水线、辩论；能用单 Agent 别上多 Agent；
  • 工业级智能体 = 范式组合 + Harness + 记忆 + 评估闭环；
  • A2A 等协议正在标准化多 Agent 通信。

memory.py —— 智能体的长期记忆系统
"""
memory.py —— 智能体长期记忆：写入、检索、重要性判断
结合向量检索 + 元数据，让智能体跨会话记住事情
"""
import time
from dataclasses import dataclass, field
from sentence_transformers import SentenceTransformer

embedder = SentenceTransformer("BAAI/bge-m3")

@dataclass
class Memory:
    content: str
    category: str        # "preference" / "fact" / "event" / "skill"
    importance: float     # 0-1，越高越重要
    timestamp: float = field(default_factory=time.time)
    embedding: list = field(default_factory=list)

class MemoryStore:
    def __init__(self):
        self.memories = []

    def add(self, content, category, importance):
        # 去重：相似度高的旧记忆更新而非新增
        emb = embedder.encode([content])[0]
        for m in self.memories:
            if cosine(m.embedding, emb) > 0.92:    # 已有极相似记忆
                m.content = content; m.importance = max(m.importance, importance)
                return
        self.memories.append(Memory(content, category, importance, embedding=emb))

    def retrieve(self, query, top_k=5):
        if not self.memories: return []
        q_emb = embedder.encode([query])[0]
        scored = [(cosine(m.embedding, q_emb) * m.importance, m) for m in self.memories]
        # 综合相关性与重要性，取 top_k
        scored.sort(key=lambda x: -x[0])
        return [m for _, m in scored[:top_k]]

    def maybe_remember(self, interaction, llm):
        # 让模型判断：这次交互有什么值得长期记的？
        extraction = llm(f"从这次交互中抽出值得长期记住的事实/偏好（没有就说 NONE）：{interaction}")
        if extraction != "NONE":
            self.add(extraction, "fact", importance=0.7)

# ---------- 接入智能体 ----------
def chat_with_memory(user_msg, store, llm):
    relevant = store.retrieve(user_msg)              # 检索相关记忆
    memory_ctx = "\n".join(m.content for m in relevant)
    reply = llm(f"相关记忆：{memory_ctx}\n用户：{user_msg}")
    store.maybe_remember(user_msg + reply, llm)      # 沉淀新记忆
    return reply

10 代码实战：多智能体层级编排

第 4 节讲了多智能体三种模式。这里实现层级模式——主 Agent 派子 Agent，子 Agent 汇报，主 Agent 综合：

multi_agent.py —— 层级式多智能体编排
"""
multi_agent.py —— 主 Agent + 多个专业子 Agent 的层级编排
每个子 Agent 有自己的角色、工具、独立上下文
"""

# 子 Agent 定义：角色 + 专长 + 工具
SUBAGENTS = {
    "researcher": {
        "role": "你负责调研：用搜索工具收集信息，给出事实摘要。",
        "tools": ["web_search", "read_url"],
    },
    "coder": {
        "role": "你负责编码：写代码、跑测试、修 bug。",
        "tools": ["write_file", "run_code", "read_file"],
    },
    "reviewer": {
        "role": "你负责审查：找出问题、给出改进建议。",
        "tools": ["read_file", "run_code"],
    },
}

def orchestrator(task):
    # 主 Agent：决定派谁、做什么、何时综合
    plan = llm(f"""你是主 Agent。任务：{task}
可用子 Agent：{list(SUBAGENTS.keys())}
制定派发计划：每步派给哪个子 Agent 做什么（JSON）。""")
    steps = json.loads(plan)

    workspace = {}    # 共享工作区（子 Agent 间传递产物）
    for step in steps:
        agent_name = step["agent"]
        subtask    = step["task"]
        # 派给子 Agent（独立上下文窗口）
        result = run_subagent(agent_name, subtask, workspace)
        workspace[step.get("output_key", agent_name)] = result
        print(f"\n📤 {agent_name} 完成：{result[:80]}...")

    # 主 Agent 综合
    final = llm(f"任务：{task}\n各子 Agent 产出：{workspace}\n给出最终成果。")
    return final

def run_subagent(name, subtask, workspace):
    cfg = SUBAGENTS[name]
    # 子 Agent 独立 messages + 角色 + 专属工具
    messages = [{"role": "system", "content": cfg["role"]},
                {"role": "user", "content": f"{subtask}\n已知信息：{workspace}"}]
    # ... ReAct loop（见第 4 章），用 cfg["tools"]
    return react_loop_with_tools(messages, cfg["tools"])

11 高级推理范式：Self-Consistency 与 ToT

除了前面四大范式，还有两个提升推理质量的经典技巧：

Self-Consistency（自洽性采样）

对同一个问题，用高温度采样生成多个不同的推理路径，然后取多数答案。原理：正确答案往往被多条路径支持，错误答案路径各异。

self_consistency.py —— 多数投票提升准确率
def self_consistency(question, llm, n=5):
    # 生成 n 条不同的推理路径
    answers = []
    for _ in range(n):
        ans = llm(question, temperature=0.8)  # 高温度=多样性
        answers.append(extract_final_answer(ans))
    # 投票：取出现次数最多的答案
    from collections import Counter
    most_common = Counter(answers).most_common(1)[0]
    print(f"投票结果: {most_common[0]}（{most_common[1]}/{n} 票）")
    return most_common[0]
# 对数学/逻辑题，self-consistency 通常比单次采样准确率高 5-15%

Tree of Thoughts（思维树）

ReAct 是「一条路走到底」，ToT 是「走迷宫」：把推理组织成一棵树，每个节点是一个中间状态，可以分支、评估、回溯。适合需要探索多种可能性的复杂问题（如数学证明、策略规划）。

初始问题
▼ 生成多个「下一步想法」
想法 A（评估：8/10）
想法 B（评估：3/10）✂️ 剪枝
想法 C（评估：7/10）
▼ 保留高分分支，继续展开
...直到找到解
图 5-7 · Tree of Thoughts：分支+评估+剪枝+回溯

ToT 的代价是计算量大（要评估和展开很多节点），适合难题，简单问题用 ReAct 更划算。2026 年的趋势是用推理模型原生支持这种探索，而非外挂 ToT 框架。

12 智能体设计的七条铁律

综合前几章，给你七条经过实战检验的设计原则：

简单优先

能用单步解决别用多步，能用单 Agent 别上多 Agent。复杂度是万恶之源。

可验证优于可信任

别相信模型的「我觉得对」，要让它「证明给你看」（跑测试、给引用）。PEV 思维。

上下文是稀缺资源

每一段进上下文的内容都要「值」。用子 Agent 隔离、用 RAG 按需取。

失败是常态，要设计恢复

假设每一步都可能失败。重试、回退、换策略。

边界清晰

明确告诉智能体能做什么、不能做什么。权限模型 + 系统提示双管齐下。

可观测才能调试

记录每一步的思考、动作、结果。出问题时能回放定位。

评估闭环驱动迭代

没有评估集就没有改进。建立回归集，每次改动都测。
