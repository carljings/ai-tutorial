RAG 知识库搭建
从分块到检索的完整工程

大模型再强，也有「不知道」的事——你的私有数据、最新资讯、专业领域知识。RAG（检索增强生成）就是给模型外挂一个「知识库」，让它先查资料再回答。本章从分块策略、Embedding、向量检索，一路讲到 Late Chunking、多模态 RAG、GraphRAG。

阅读约 40 分钟 难度 ★★★★ 前置：第 01 章

★ 为什么需要 RAG

大模型有三个天生短板：知识有截止日期（不知道训练后的事）、不知道你的私有数据（公司文档、个人笔记）、会一本正经地胡说（幻觉）。RAG 同时缓解这三个问题。

RAG 的核心思想：不把所有知识塞进模型参数，而是把知识存在外部库，回答前先检索相关片段，塞进上下文，让模型基于检索到的内容作答。

② 拼接：把片段 + 问题组成 prompt
▼
③ 生成：模型基于资料作答

💡 一句话决策

知识是「事实」且会变 → RAG；要改变的是「怎么说话/怎么推理」→ 微调。绝大多数企业场景（客服、文档问答、知识管理）首选 RAG。

02 分块（Chunking）：最被低估的关键步

业内有句话：「RAG 效果 70% 取决于分块。」分块的好坏直接决定检索质量，却最容易被新手忽略。

为什么要分块

  • Embedding 有长度限制：大多数 Embedding 模型只吃几百 token，长文档塞不进去；
  • 检索精度：整篇文档当一个向量太粗，检索时「相关但不精确」；切成小块，每个小块语义集中，检索更准；
  • 上下文预算：检索回来要塞进 prompt，块太大会挤占宝贵的上下文窗口。

分块的核心目的，用一句话说：把语义相同的内容聚到一起，把语义不同的内容切开。

七种主流分块策略

📏

① 固定长度切分

按固定字符/token 数切。最简单，但会粗暴切断句子。实践中常配合重叠（overlap）缓解边界问题。

🔁

② 递归切分（Recursive）

按「段落 → 句子 → 词」层级递归切，尽量在自然边界断开。LangChain 的默认策略，最常用。

📄

③ 文档结构切分

按 Markdown 标题、HTML 标签、PDF 章节切。保留结构信息，质量高，但要解析文档格式。

🧠

④ 语义切分（Semantic）

用 Embedding 计算相邻句子的语义相似度，在「语义跳变」处切。自适应，但慢。

🤖

⑤ LLM 切分

让大模型自己决定在哪切。质量最高但最贵，适合小量高价值文档。

⚙️

⑥ 代码感知切分

按函数/类切代码，保留语法完整性。对代码 RAG 必备。

🆕

⑦ Late Chunking（新范式）

先对整文档 embedding 再切，保留全局上下文。2025-2026 的重要突破，下节详述。

🧩

⑧ 父子分块（Small-to-Big）

用小块做检索（精确），但取回时连带父块（完整上下文）。鱼与熊掌兼得。

分块大小的权衡

块大小优点缺点
小（100-300 token）检索精确，上下文省可能丢失上下文，答案碎片化
大（800-2000 token）上下文完整检索粗，噪音多，贵
中（400-600 token）平衡——常用起点
⚠️ 分块三大坑
  • 切断语义：在句子/段落中间硬切，破坏完整意思。解法：用递归/结构切分 + overlap；
  • 块内主题混杂：一个块里塞了两个不相关话题，检索时无法精确命中。解法：语义切分；
  • 元数据丢失：切完块忘了它来自哪个文档/章节，溯源困难。解法：每个块都带上 source/section 元数据。
用 LangChain 做递归分块（最常用）
from langchain_text_splitters import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,            # 每块约 500 字符
    chunk_overlap=50,         # 相邻块重叠 50 字符，防边界切断
    separators=["\n\n", "\n", "。", ".", " ", ""],  # 优先在段落/句子断
)

chunks = splitter.split_text(long_document)
# 每个 chunk 都是一段相对完整的语义单元

03 Embedding：把文字变成向量

分完块，要把每个块变成一个向量（通常 768~3072 维），这个过程叫 Embedding。回顾第 1 章：Embedding 让「语义相近的文本，向量也相近」，这是向量检索能成立的数学基础。

Embedding 模型怎么选

模型维度特点
BGE-m3（智源）1024多语言/多功能/长文本，中文极强，开源
text-embedding-3（OpenAI）可变闭源 API，效果好但需联网
智谱 embedding-32048/3072中文友好，与 GLM 生态协同
GTE（阿里）768/1024开源，多语言
Cohere embed1024多语言强，闭源

选型要考虑：语言（中文优先 BGE/智谱）、维度（影响存储和速度）、部署方式（开源本地 vs 闭源 API）、MTEB 榜单表现。

💡 稠密向量 vs 稀疏向量

稠密向量（Dense）：每个维度都有值，捕捉语义，擅长「意思相近」。稀疏向量（Sparse）：类似 BM25，关键词命中，擅长「精确匹配」。BGE-m3 同时输出三种向量（dense/sparse/multivec），可以混合检索，效果最好——既懂语义又抓关键词。

04 向量数据库：存与查

Embedding 出来的几百万、几千万个向量，要存在专门的向量数据库里，并能快速做「找最相似的 K 个」检索。

主流向量库

向量库定位适合
Milvus / Zilliz专业分布式向量库大规模、生产级
QdrantRust 写的高性能向量库性能敏感场景
Weaviate带混合检索的向量库需要 dense+sparse 混合
Chroma轻量嵌入式原型、小项目
pgvectorPostgreSQL 扩展已有 PG、不想加组件
Elasticsearch搜索老牌 + 向量已有 ES、要全文+向量

近似最近邻（ANN）索引

暴力找最相似的 K 个要遍历所有向量，太慢。实际用近似最近邻算法加速，牺牲一点点精度换巨大速度提升：

  • HNSW（Hierarchical Navigable Small World）：基于图，查询快、精度高，最主流；
  • IVF：先聚类再在类内搜，适合超大规模；
  • PQ（Product Quantization）：压缩向量省内存，精度有损。

05 检索：不只是向量搜索

很多人以为 RAG 的检索就是「算 query 向量，找最相似的 K 个块」。这只是朴素向量检索，效果往往一般。生产级检索要做更多。

查询改写（Query Rewriting）

用户的原始问题往往不适合直接检索。比如「它怎么用？」里的「它」指代不清。检索前要先改写查询：

  • 指代消解：结合对话历史，把「它」替换成具体实体；
  • 问题扩展：一个问题扩成多个相关问题，分别检索再合并（Multi-Query）；
  • 假设性文档（HyDE）：先让模型生成一个「假想答案」，用假想答案去检索（往往比用问题检索更准）；
  • 子问题分解：复杂问题拆成多个子问题分别检索。

混合检索（Hybrid Search）

向量检索擅长语义，关键词检索（BM25）擅长精确匹配。两者结合效果最好：

融合常用 RRF（Reciprocal Rank Fusion）：不关心分数只看排名，把两路的排名倒数求和，简单且有效。

重排（Rerank）：精筛的关键

向量检索召回的 Top-K 里，常混入「相关但不精确」的块。再上一步Rerank：用一个专门的 rerank 模型（如 BGE-reranker、Cohere Rerank）对候选集精细打分重排。

🔑 为什么 Rerank 这么有效

Embedding 模型为了快，把整段压缩成一个向量，信息损失大；Rerank 模型是交叉编码器（Cross-Encoder），把 query 和每个候选放一起精细计算相关性，精度高得多。典型做法：向量检索召回 50 条 → Rerank 精选 Top 5。这一步往往能让准确率提升 10-30%。

06 进阶：让 RAG 更聪明的招式

Late Chunking：颠覆传统流程

传统流程是「先切块，再 embedding」。但切块后每个块失去了全文上下文，导致「它」「该公司」这类指代无法正确编码。2025 年兴起的 Late Chunking 反过来：先对整文档做 embedding（得到 token 级向量序列），再切块（对每块的 token 向量做池化）。

传统：文档 → 切块 → 逐块 embedding
Late：文档 → 整体 embedding → 切块池化
图 6-4 · Late Chunking：保留全局上下文的新范式

好处：每个块的向量都「知道」全文语境，指代消解自然发生，长文档检索质量显著提升。要求 Embedding 模型支持长输入（如 BGE-m3 支持 8K token）。

父子分块（Small-to-Big / Parent-Child）

矛盾：小块检索精确但上下文不足，大块上下文足但检索粗。父子分块的解法：用小块检索，命中后取回它所属的大块（父块）。

小块（200 token）→ embedding → 检索精确
▼ 命中小块
映射到父块（1500 token）→ 取回完整上下文
▼
送进 prompt：精确检索 + 完整上下文
图 6-5 · 父子分块：检索用小块，返回用大块

多模态 RAG

现实文档里有图片、表格、公式。多模态 RAG 把它们也纳入检索：

  • 图片：用多模态 Embedding（如 CLIP）或先用 VLM 生成描述再 embedding；
  • 表格：保留结构，或转成 Markdown/自然语言描述；
  • 公式：用 LaTeX 保留，或转文本。

GLM-4V、GPT-4V 等多模态模型让多模态 RAG 越来越实用——检索回来的图片可以直接喂给多模态 LLM 理解。

GraphRAG：知识图谱加持

向量检索有个根本短板：不擅长「跨文档的全局推理」。比如「这个领域的主要流派有哪些？」需要综合多篇文档抽象，单纯向量召回做不好。GraphRAG（微软提出）的思路：先用 LLM 从文档里抽取实体和关系，构建知识图谱，再基于图谱做检索和推理。

文档 → LLM 抽取 → 实体 + 关系
▼
知识图谱（节点 = 实体，边 = 关系）
▼ 社区检测 / 图遍历
回答「全局性」「关联性」问题

Agentic RAG：让 RAG 自己决定怎么查

把 RAG 放进 Agent Loop，让智能体自主决定：要不要查、查几次、查哪个库、要不要改写问题、要不要再查一次。Agentic RAG 是 2026 年的主流方向——简单的「一问一查」变成「多轮探索式检索」，质量上限大幅提升。

07 RAG 的评估：怎么知道好不好

RAG 最难的不是搭，是评估。常用的 RAG 评估框架（如 RAGAS、TruLens）从几个维度打分：

维度含义
上下文相关性检索回来的内容，和问题相关吗？（检索质量）
事实一致性回答是否忠于检索到的内容？（有没有幻觉）
回答相关性回答是否切题？（有没有答非所问）
召回率该查到的重要信息，查到了吗？

评估需要一批「问题-标准答案-相关文档」的测试集。可以先人工标几十条，再用 LLM 自动扩展。每次改动分块/检索/Embedding，都跑评估集回归，确保没退化。

09 RAG vs 长上下文：会取代吗

GLM-5.2 有 1M 上下文。一个常见疑问：「上下文这么长，RAG 是不是没必要了？」答案是：不会取代，而是融合。

场景更优选择原因
几十页文档问答长上下文直接塞简单、无检索损失
几万页知识库必须 RAG再长也塞不下
实时更新数据RAG改库即生效
多用户高并发RAG每用户塞 1M 太贵
需要精确溯源RAG能给出处
💡 2026 的最佳实践：Agentic RAG

让智能体动态决策：简单问题直接答，需要资料时先 RAG 检索，复杂问题多轮检索 + 反思。长上下文 + RAG 不是二选一，而是组合拳。GLM-5.2 的长上下文让 RAG 的「塞多少」更从容，但不改变「海量知识必须靠检索」的本质。

11 代码实战：八种分块策略的实现

第 2 节讲了分块的理念，这里把最常用的几种分块策略都用代码实现出来，让你能直观对比它们的差异。

chunking_strategies.py —— 八种分块策略对比
"""
chunking_strategies.py —— 八种分块策略的实现与对比
"""
import re
from typing import List

# ========== ① 固定长度切分 ==========
def fixed_size(text, size=200, overlap=30):
    chunks = []
    i = 0
    while i < len(text):
        chunks.append(text[i:i+size])
        i += size - overlap          # 滑动窗口，带重叠
    return chunks

# ========== ② 递归切分（LangChain 同款思路） ==========
def recursive_split(text, separators=["\n\n", "\n", "。", " ", ""], chunk_size=300):
    # 用最优先的分隔符切，切出的块若仍太大，用下一级分隔符再切
    if len(text) <= chunk_size or not separators:
        return [text]
    sep = separators[0]
    parts = text.split(sep)
    result = []
    for p in parts:
        if len(p) > chunk_size:
            result.extend(recursive_split(p, separators[1:], chunk_size))
        else:
            result.append(p + sep)
    # 合并过小的块
    merged, buf = [], ""
    for p in result:
        if len(buf) + len(p) <= chunk_size:
            buf += p
        else:
            if buf: merged.append(buf)
            buf = p
    if buf: merged.append(buf)
    return merged

# ========== ③ Markdown 结构切分 ==========
def markdown_split(text):
    # 按 # 标题切，保留标题层级
    pattern = r'(?=^#{1,6}\s)'
    sections = re.split(pattern, text, flags=re.MULTILINE)
    return [s.strip() for s in sections if s.strip()]

# ========== ④ 语义切分（基于 Embedding 相似度） ==========
def semantic_split(text, embed_fn, threshold=0.5):
    # 1. 先按句子切
    sentences = re.split(r'(?<=[。.!?])\s*', text)
    if len(sentences) <= 1: return [text]
    # 2. 算相邻句子的 embedding 相似度
    embs = [embed_fn(s) for s in sentences]
    # 3. 相似度低于阈值处 → 语义跳变 → 在这里切
    chunks, buf = [], sentences[0]
    for i in range(1, len(sentences)):
        sim = cosine(embs[i-1], embs[i])
        if sim < threshold:           # 语义跳变
            chunks.append(buf); buf = sentences[i]
        else:
            buf += sentences[i]
    if buf: chunks.append(buf)
    return chunks

# ========== ⑤ 代码感知切分（按函数/类） ==========
def code_split(code, lang="python"):
    # 按 def / class 切，保留完整语法单元
    pattern = r'(?=^(?:async\s+)?def\s+|class\s+)'
    blocks = re.split(pattern, code, flags=re.MULTILINE)
    return [b for b in blocks if b.strip()]

# ========== ⑥ 父子分块（Small-to-Big） ==========
def parent_child_split(text, child_size=200, parent_size=1000):
    # 先切大块（父），再把每个父切成小块（子）
    parents = fixed_size(text, parent_size, overlap=0)
    mapping = []
    for pi, parent in enumerate(parents):
        children = fixed_size(parent, child_size, overlap=30)
        for child in children:
            mapping.append({"child": child, "parent_id": pi, "parent": parent})
    return mapping
    # 检索时用 child 向量命中，返回时给 parent（完整上下文）

# cosine 相似度辅助
def cosine(a, b):
    import math
    dot = sum(x*y for x,y in zip(a,b))
    na = math.sqrt(sum(x*x for x in a))
    nb = math.sqrt(sum(y*y for y in b))
    return dot/(na*nb) if na*nb else 0

实战中选择建议：通用文档用递归切分（②），Markdown/技术文档用结构切分（③），代码用代码切分（⑤），追求精度用语义切分（④）或父子分块（⑥）。没有银弹，最好的做法是用评估集（第 7 节）实测对比。

12 代码实战：混合检索 + Rerank 完整实现

把第 5 节的「向量 + BM25 混合检索 + Rerank」用代码完整实现。这是生产级 RAG 检索的核心：

hybrid_retrieval.py —— 混合检索 + RRF 融合 + Rerank
"""
hybrid_retrieval.py —— 生产级混合检索
向量检索（语义）+ BM25（关键词）+ RRF 融合 + Cross-Encoder Rerank
"""
import math
from collections import Counter
from rank_bm25 import BM25Okapi
from sentence_transformers import SentenceTransformer

# 嵌入模型（检索用）和 Rerank 模型（精排用）
embedder = SentenceTransformer("BAAI/bge-m3")
reranker = SentenceTransformer("BAAI/bge-reranker-v2-m3", backend="onnx")

class HybridRetriever:
    def __init__(self, docs: list[str]):
        self.docs = docs
        # 建向量索引（演示用暴力搜索，生产用 HNSW）
        self.doc_embs = embedder.encode(docs, normalize_embeddings=True)
        # 建 BM25 索引
        self.bm25 = BM25Okapi([d.split() for d in docs])

    def vector_search(self, query, k=20):
        q_emb = embedder.encode([query], normalize_embeddings=True)[0]
        scores = self.doc_embs @ q_emb            # 余弦相似度（已归一化）
        idx = scores.argsort()[::-1][:k]
        return [(i, scores[i]) for i in idx]

    def bm25_search(self, query, k=20):
        scores = self.bm25.get_scores(query.split())
        idx = scores.argsort()[::-1][:k]
        return [(i, scores[i]) for i in idx]

    def rrf_fuse(self, results_list, k_const=60):
        # Reciprocal Rank Fusion：不看分数只看排名
        rrf = Counter()
        for results in results_list:
            for rank, (idx, _) in enumerate(results):
                rrf[idx] += 1 / (k_const + rank + 1)
        return [idx for idx, _ in rrf.most_common()]

    def rerank(self, query, candidate_idxs, top_n=5):
        # Cross-Encoder 精排：把 query 和每个候选拼一起打分
        pairs = [[query, self.docs[i]] for i in candidate_idxs]
        scores = reranker.predict(pairs)
        ranked = sorted(zip(candidate_idxs, scores), key=lambda x: -x[1])
        return ranked[:top_n]

    def search(self, query, final_k=5):
        # 三步走：双路召回 → RRF 融合 → Rerank 精排
        vec_hits = self.vector_search(query, k=20)
        bm25_hits = self.bm25_search(query, k=20)
        fused = self.rrf_fuse([vec_hits, bm25_hits])
        # 取融合后前 50 个，Rerank 精选 final_k 个
        final = self.rerank(query, fused[:50], top_n=final_k)
        return [(self.docs[i], float(s)) for i, s in final]

# 使用
docs = ["GLM-5.2 支持 1M 上下文", "Transformer 用 Self-Attention", "RAG 检索增强生成", ...]
retriever = HybridRetriever(docs)
results = retriever.search("GLM 上下文多长")
# results = [("GLM-5.2 支持 1M 上下文", 0.95), ...]

这套「双路召回 50 → RRF 融合 → Rerank 精选 5」是 2026 年 RAG 检索的黄金组合。向量检索召回语义相关的，BM25 兜底精确关键词，RRF 融合两者优势，Rerank 用更强的 Cross-Encoder 精排。实测比单纯向量检索准确率高 20-40%。

13 HNSW 算法：向量检索为什么这么快

第 4.2 节提到 HNSW 是最主流的 ANN 索引。它为什么快？核心思想是分层跳表：

  • 把向量构建成多层图，顶层稀疏（节点少，跨度大）用于快速定位区域，底层密集（所有节点）用于精确查找；
  • 查询时从顶层开始，大步跳到目标附近，再逐层下沉，精细搜索；
  • 类比：先坐高铁跨省（顶层），再坐地铁跨城（中层），最后步行找门牌（底层）。
顶层（稀疏）：快速跨域定位 ●───●───●
中层：区域细化 ●──●──●──●──●
底层（全量）：精确查找 ●·●·●·●·●·●·●·●
图 6-7 · HNSW 分层图：从粗到细逐层逼近

HNSW 把暴力 O(N) 搜索降到约 O(log N)，精度损失极小（recall > 95%），是 Milvus、Qdrant、pgvector 的默认索引。理解它有助于你调参（M 控制图的连接度，efConstruction/efSearch 控制建图/查询的精度-速度权衡）。

14 代码实战：Late Chunking 实现

第 6.1 节讲的 Late Chunking，核心是「先 embedding 整文档，再切」。实现：

late_chunking.py —— 先 embedding 后切分
"""
late_chunking.py —— Late Chunking 实现
需要支持长输入的 Embedding 模型（如 BGE-m3，支持 8K token）
"""
import numpy as np
from sentence_transformers import SentenceTransformer

model = SentenceTransformer("BAAI/bge-m3")

def late_chunk(doc, chunk_size=256):
    # ① 先按 token 切分文档（记录每个 token 的位置）
    tokens = model.tokenizer(doc, return_offsets_mapping=True, add_special_tokens=False)

    # ② 对整文档做 embedding，得到 token 级向量序列（带全文上下文！）
    # model_kwargs={'output_hidden_states':True} 拿到每层的隐状态
    outputs = model.encode([doc], output_value="token_embeddings", convert_to_numpy=True)
    token_embs = outputs[0]          # (num_tokens, dim)，每个向量都「看过」全文

    # ③ 按 chunk_size 分组，对每组做均值池化 → 得到 chunk 向量
    chunk_embs = []
    for i in range(0, len(token_embs), chunk_size):
        chunk_vec = token_embs[i:i+chunk_size].mean(axis=0)   # 池化
        chunk_embs.append(chunk_vec / np.linalg.norm(chunk_vec))  # 归一化
    return chunk_embs

# 传统 vs Late Chunking 的区别：
# 传统：chunk_text → embed(chunk) → 每个 chunk 向量只有局部信息
# Late：embed(全文档) → 切 token 向量 → 池化 → 每个 chunk 向量带全文上下文

Late Chunking 的精妙之处：每个 chunk 的向量都隐含了全文的上下文（因为是在整文档 embedding 之后切的），所以「它」「该公司」这类指代能被正确编码。对于长文档检索，质量提升显著。代价是要求 Embedding 模型支持长输入，且 embedding 计算量更大。

15 代码实战：Agentic RAG——让检索自主化

第 6.5 节的 Agentic RAG 是 2026 的主流。把 RAG 放进 Agent Loop，让智能体自主决定怎么查：

agentic_rag.py —— 智能体驱动的多轮探索式 RAG
"""
agentic_rag.py —— Agentic RAG：智能体自主决定检索策略
能：改写查询、多次检索、跨库检索、判断是否查够了
"""
def agentic_rag(question, retriever, llm, max_rounds=3):
    context = []
    for round in range(max_rounds):
        # ① 让模型决定下一步：直接答 / 再查 / 改写查询
        decision = llm(f"""已有资料：{context}
问题：{question}
你还需要更多信息吗？
- 如果资料够了，直接回答
- 如果需要，给出一个更精准的检索查询（一次只给一个）""")

        if "查询：" in decision:
            query = decision.split("查询：")[1].strip()
            hits = retriever.search(query, final_k=3)
            context.extend(hits)             # 累积上下文
            print(f"\n🔄 第 {round+1} 轮检索：'{query}' → {len(hits)} 条")
        else:
            # 模型认为资料够了，给出最终答案
            return decision
    return "检索轮数用尽"

# 对比朴素 RAG（一问一查）vs Agentic RAG（多轮探索）：
# 朴素：query → 检索 → 生成（一次性的）
# Agentic：query → 检索 → 判断 → 改写 → 再检索 → ... → 生成（探索式）
# 复杂问题（如「对比 A 和 B 的优劣」）Agentic RAG 准确率显著更高
RAG 给智能体装上了「外置大脑」。但知识库如果只是一堆切片，还不够聪明。下一章，我们讲怎么把知识沉淀成结构化的 LLM Wiki——一个能自我增长、带链接、带版本的知识中枢。
