Agent Loop 详解
一个 while 循环的进化史

剥开所有花哨的概念，一个智能体（Agent）本质上就是一个 while 循环：「感知 → 思考 → 行动 → 观察」，循环到任务完成。本章带你从最朴素的循环，一步步进化到带工具调用、错误恢复、子任务派发、上下文窗口管理的工业级 Loop。

阅读约 30 分钟 难度 ★★★★ 前置：第 03 章

★ 智能体的本质：一个循环

很多人被「智能体」这个词唬住了，觉得是什么高深的东西。其实把所有包装撕掉，一个智能体的核心代码就是：

agent_loop_v0.py —— 最朴素的智能体
while not task_done:
    observation = perceive()        # 感知：当前状态是什么
    thought, action = think(observation)   # 思考：该做什么
    result = execute(action)        # 行动：去做
    observe(result)                 # 观察：结果如何

就这么简单。所有复杂的智能体框架（LangGraph、AutoGPT、Claude Code、GLM 智能体），底层都是这个循环的某种增强版。理解这个循环，就理解了智能体。

① 感知
Perceive
▶
② 思考
Think
▶
③ 行动

▶
④ 观察

↺ 循环，直到完成或停止
图 4-1 · Agent Loop 的四阶段：感知-思考-行动-观察

四阶段分别是什么

阶段大模型语境下的含义例子
① 感知读取当前上下文：用户消息、历史、工具结果「用户说要修 bug，我已经看到报错日志」
② 思考模型推理：下一步该做什么、调用什么工具「我需要先打开报错的文件看看」
③ 行动执行模型决定的动作（工具调用）调用 read_file('app.py')
④ 观察把行动结果加入上下文，作为下一轮的输入文件内容进入对话历史

01 Loop 的四级进化

从玩具到工业级，Agent Loop 经历了四个阶段。我们逐级来看，每一级解决了上一级的什么痛点。

Lv1 · 朴素循环

while not done: call_model()

能跑，但容易死循环、爆上下文、不处理错误。玩具级。

Lv2 · 工具循环

while not done: think() → tool_call()

原生工具调用 + ReAct 推理。能完成多步任务。原型级。

Lv3 · 鲁棒循环

+ 错误恢复 + 上下文管理 + 步数/预算限制

处理异常、压缩上下文、防止失控。准生产级。

Lv4 · 编排循环

+ 子 Agent + 并发 + 反思 + 记忆

多智能体协作、长期记忆、自我反思。生产级。

02 Lv2：带工具调用的 ReAct 循环

现代大模型（GLM-5.2、Claude、GPT）都原生支持工具调用（Tool Use / Function Calling）。这让 Loop 不再需要脆弱的「从文本里解析动作」，而是模型直接输出结构化的工具调用请求。

ReAct：思考与行动交织

ReAct（Reasoning + Acting）是 2022 年提出、至今仍是主流的智能体范式。核心思想：让模型把推理过程和工具调用交织在一起，每一步都先「想」再「做」。

Thought: 我需要查一下今天的天气
▼
Action: search_weather(city="北京")
▼
Observation: 北京今天晴，25°C
▼
Thought: 我已经有信息了，可以回答用户
▼
Answer: 北京今天晴，25°C，适合出行
图 4-2 · ReAct 的 Thought-Action-Observation 循环

这种「先想后做、做了再想」的交织，让模型的行为可解释、可调试。你能看到它每一步的推理，而不是一个黑盒突然给出答案。

用原生工具调用实现 ReAct

agent_loop_v2.py —— 带工具调用的 ReAct 循环
from openai import OpenAI
import json

client = OpenAI(base_url="https://open.bigmodel.cn/api/paas/v4", api_key="YOUR_KEY")
MODEL = "glm-5.2"

# 工具注册表：name → 可调用函数
REGISTRY = {
    "search": lambda q: f"搜索结果：关于 '{q}' 的摘要...",
    "calculator": lambda expr: str(eval(expr)),   # 仅示例，生产别用 eval
}

TOOLS = [
    {"type": "function", "function": {"name": "search", "description": "搜索网络",
        "parameters": {"type": "object", "properties": {"q": {"type": "string"}}, "required": ["q"]}},
    {"type": "function", "function": {"name": "calculator", "description": "数学计算",
        "parameters": {"type": "object", "properties": {"expr": {"type": "string"}}, "required": ["expr"]}},
]

def agent_loop(task, max_steps=10):
    messages = [
        {"role": "system", "content": "你是 ReAct 智能体。逐步思考，需要信息时调用工具。"},
        {"role": "user", "content": task},
    ]
    for step in range(max_steps):
        resp = client.chat.completions.create(model=MODEL, messages=messages, tools=TOOLS)
        msg = resp.choices[0].message
        messages.append(msg)
        if not msg.tool_calls:          # 模型给出最终答案，循环结束
            return msg.content
        # 执行所有工具调用，把结果作为 observation 喂回
        for tc in msg.tool_calls:
            args = json.loads(tc.function.arguments)
            result = REGISTRY[tc.function.name](**args)
            messages.append({"role": "tool", "tool_call_id": tc.id, "content": str(result)})
    return "达到最大步数"

print(agent_loop("123 × 456 等于多少？再搜一下这个数字有没有特殊含义。"))

这就是一个能用的 ReAct 智能体。它会先算乘法（calculator），拿到 56088，再搜索这个数字，最后综合回答。整个过程自主、多步、可观察。

03 Lv3：让 Loop 鲁棒起来

Lv2 的循环在「玩具」环境里很美，但一到真实场景就脆得像玻璃。Lv3 要解决它在生产中会遇到的一系列「死法」。

智能体的七种死法

♾️

① 死循环

模型反复调用同一个工具、反复犯同一个错。消耗大量 token 却原地踏步。

💥

② 工具异常

工具超时、报错、返回乱码。模型不知道怎么处理，要么卡住要么瞎编。

📦

③ 上下文爆炸

对话历史太长，撑爆窗口，或超过预算，或质量下降。

🌊

④ 任务漂移

模型越做越偏，忘了原始目标，跑去做无关的事。

🔐

⑤ 权限越界

模型想做不允许的操作，被拒后不知道退路。

💸

⑥ 预算失控

没有成本上限，一个任务烧掉几十美元。

🧩

⑦ 格式错误

模型输出不合规的 JSON 或参数，导致工具调用失败。

对症下药的防护

针对每种「死法」，工业级 Loop 都有对应防护：

死法防护手段
死循环重复检测（连续 N 步相同动作则强制跳出）+ 步数上限
工具异常try/except 包裹 + 重试 + 退避 + 把错误信息喂回模型让它换思路
上下文爆炸滚动摘要 + 选择性保留 + 子 Agent 隔离
任务漂移每 K 步重申目标 + 完成度自检
权限越界明确告知边界 + 被拒后提供替代方案
预算失控token/成本预算上限，超了就停
格式错误结构化输出约束 + JSON schema 校验 + 自动修复重试
agent_loop_v3.py —— 加了防护的鲁棒循环
def robust_agent_loop(task, max_steps=15, max_tokens=50000):
    messages = build_initial_messages(task)
    used_tokens = 0
    last_actions = []                         # 用于死循环检测

    for step in range(max_steps):
        # 预算检查
        if used_tokens > max_tokens:
            return "⚠️ 预算耗尽"

        # 任务漂移检查：每 5 步重申目标
        if step % 5 == 0 and step > 0:
            messages.append({"role": "system",
                "content": f"提醒：原始任务是「{task}」。请确认是否在正轨上。"})

        try:
            resp = call_model(messages)
            used_tokens += resp.usage.total_tokens
            msg = resp.choices[0].message
            messages.append(msg)

            if not msg.tool_calls:
                return msg.content

            for tc in msg.tool_calls:
                # 死循环检测
                key = (tc.function.name, str(tc.function.arguments))
                if last_actions[-3:].count(key) >= 2:
                    messages.append({"role": "system",
                        "content": "你似乎在重复同一动作，请换一种方法。"})
                    continue
                last_actions.append(key)

                # 工具执行，带异常处理与重试
                result = execute_with_retry(tc, retries=2)
                messages.append({"role": "tool", "tool_call_id": tc.id, "content": result})

        except Exception as e:
            # 模型调用失败：喂回错误，让它自己想办法
            messages.append({"role": "system", "content": f"系统错误：{e}。请调整策略。"})

    return "达到最大步数"
⚠️ 别让模型「假装」成功

一个隐蔽的坑：工具报错时，有些模型会编造一个看似合理的结果继续往下走，而不是承认失败。防护的关键是把原始错误信息原样喂回，并在系统提示里明确：「工具失败时，必须如实报告，不得臆测结果。」这是智能体可靠性的关键纪律。

04 上下文窗口管理：Loop 的生命线

Loop 每转一圈，对话历史就长一截。跑久了必然撑爆窗口。即便 GLM-5.2 有 1M 上下文，也不是无限的，而且越长越贵越慢越容易出错。所以主动管理上下文是工业级 Loop 的必备能力。

窗口管理的四种策略

✂️

滑动窗口

只保留最近 N 轮对话，老的直接丢弃。简单粗暴，但会丢失早期关键信息。

📝

滚动摘要

定期把老对话压缩成摘要，保留信息密度。LangChain 的 ConversationSummaryMemory 就是这个思路。

🎯

选择性保留

按重要性打分，保留关键决策和工具结果，丢弃寒暄和冗余。

🗄️

外部化 + 按需取回

把历史写到外部存储（文件/DB/向量库），需要时用 RAG 取回。这是长任务的最佳方案。

子 Agent：上下文隔离的杀手锏

当一个子任务会产生大量中间过程（比如「研究这个开源项目的架构」要读几十个文件），让主 Agent 直接做会瞬间爆窗。派一个子 Agent 用独立窗口去做，完成后只把结论返回，主 Agent 的窗口几乎不受影响。

主 Agent（窗口：用户对话 + 关键决策）
▼ 派发子任务（独立窗口）
子 Agent A
读 30 个文件
（独立 100K 窗口）
子 Agent B
跑 10 次测试
（独立窗口）
▼ 只返回结论
主 Agent 收到：「架构是 MVC，核心模块是 X、Y」
▼ 主窗口几乎没增长
继续下一步

Claude Code 的子智能体就是这个机制。GLM 智能体、Cursor 的背景 Agent 也都用类似设计。这是处理复杂长任务的标准模式。

05 什么时候停：终止条件设计

一个 Loop 最难的设计之一是「什么时候该停」。停早了任务没完成，停晚了浪费资源。常见终止条件（可组合）：

  • 模型主动停止：模型不再调用工具，给出最终答案（最理想）；
  • 步数上限：硬性安全网，防止无限循环；
  • 预算上限：token/成本/时间超限即停；
  • 目标达成判定：用一个独立的「裁判」模型判断任务是否完成；
  • 用户中断：用户主动喊停；
  • 重复检测：连续多步无进展即停。

06 并发与流式：让 Loop 更快

顺序的 Loop 一步接一步，慢。很多时候多步之间没有依赖，可以并发：

需要读 3 个文件
▶
读 A
读 B
读 C
▶
同时进行，3× 提速
图 4-4 · 并发工具调用：无依赖的步骤并行执行

GLM-5.2、Claude 都支持一轮内返回多个工具调用，Harness 并发执行后一次性把所有结果喂回。这能显著缩短任务总时长。

流式输出则是另一面的优化：模型边生成边返回，用户/系统边接收边处理，不必等整段生成完。对于交互式场景，流式能让体感延迟从「几十秒」降到「几百毫秒」。

09 代码实战：子 Agent 派发系统

第 4.2 节讲了子 Agent 实现上下文隔离。这里给一个可运行的子 Agent 派发实现：

subagent.py —— 主 Agent 派发子任务、隔离上下文
"""
subagent.py —— 子 Agent 派发：主 Agent 用子 Agent 的独立窗口处理子任务
关键：子任务的完整过程不进主 Agent 上下文，只返回结论
"""
def dispatch_subagent(subtask, tools, max_steps=8):
    # 子 Agent 有自己全新的 messages（独立上下文窗口）
    messages = [
        {"role": "system", "content": "你是专注的子智能体。完成任务后给出简洁结论。"},
        {"role": "user", "content": subtask},
    ]
    for _ in range(max_steps):
        resp = client.chat.completions.create(model="glm-5.2", messages=messages, tools=tools)
        msg = resp.choices[0].message
        messages.append(msg)
        if not msg.tool_calls:
            return msg.content       # 只把结论返回给主 Agent
        for tc in msg.tool_calls:
            messages.append({"role": "tool", "tool_call_id": tc.id,
                            "content": execute(tc)})
    return "[子任务超时]"

# 主 Agent 把「研究某库」派给子 Agent，自己上下文不爆
subtask = "研究 langchain 这个库的核心理念，读它的 README 和关键模块，给出 200 字总结。"
conclusion = dispatch_subagent(subtask, tools=RESEARCH_TOOLS)
# 主 Agent 只收到 conclusion（200 字），而不是子 Agent 读了几十 KB 的过程

这就是 Claude Code 子智能体的核心机制。主 Agent 的上下文窗口是宝贵资源，用子 Agent 去做「会产生大量中间过程」的子任务，只回收结论，是处理复杂长任务的标准模式。

10 代码实战：Plan-and-Execute 循环

plan_execute.py —— 先规划后执行的完整实现
"""
plan_execute.py —— Plan-and-Execute 范式实现
Planner 出计划，Executor 逐步执行，必要时 Replan
"""
def plan_and_execute(task):
    # ① 规划
    plan = llm(f"把任务拆成 3-7 个可执行步骤（JSON 数组）：{task}")
    steps = json.loads(plan)

    results = []
    for i, step in enumerate(steps):
        # ② 执行（每个步骤用 ReAct Loop）
        result = react_loop(step, tools=TOOLS)
        results.append(result)

        # ③ 每 3 步检查是否需要重新规划
        if (i+1) % 3 == 0 and i+1 < len(steps):
            check = llm(f"原计划：{steps}\n已完成：{results}\n剩余步骤还合理吗？需要调整就说 NEWPLAN:")
            if "NEWPLAN" in check:           # 触发重规划
                new_plan = json.loads(check.split("NEWPLAN:")[1])
                steps = steps[:i+1] + new_plan    # 替换剩余步骤
    return results

08 本章小结

📌 核心要点
  • 智能体的本质是感知-思考-行动-观察的循环；
  • Loop 有四级进化：朴素 → 工具 → 鲁棒 → 编排；
  • ReAct（思考+行动交织）是主流范式；
  • 工业级 Loop 必须处理七种死法：死循环、异常、爆窗、漂移、越权、超预算、格式错；
  • 子 Agent 实现上下文隔离，是长任务标配；
  • 终止条件、并发、流式是性能与可靠性的关键。
