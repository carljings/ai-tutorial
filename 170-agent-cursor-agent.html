Cursor 的 Agent 模式
从 Tab 到 Composer 到 Agent 的演进

Cursor 是这波 AI 编程工具里最具商业代表性的产品。它的能力经历了三个阶段的演进：Cursor Tab（行内补全）→ Composer（多文件编辑）→ Agent（自主任务执行）。这三个阶段反映了「AI 在编程中角色」的深层变化——从打字助手到编码伙伴到自主工程师。虽然 Cursor 是闭源产品，但它公开的技术特征和 UX 设计深刻影响了整个开源 Agent 生态。本章我们基于公开信息，梳理这条演进路线背后的技术决策。

阅读约 20 分钟 难度 ★★★★

★ 一、Cursor Tab：行内补全的技术突破

Cursor 最早出圈的能力是 Cursor Tab——一个比 GitHub Copilot 更聪明的行内补全。它的技术突破不在模型大小（用的是专门微调的中等模型），而在 上下文构建和延迟优化。

普通的补全（包括早期 Copilot）只看当前文件的光标前后文。Cursor Tab 的特别之处是它能跨文件补全——当你在 A 文件里写 import { User } from "./，它能基于你最近编辑过的 B 文件里的 export，补全出 models"。这要求它维护一个「最近编辑历史」和「跨文件引用图」的轻量索引，在补全时快速检索相关上下文。

// Cursor Tab 的上下文构建（基于公开信息推断）
function buildTabContext(editor: Editor): TabContext {
  const cursorPrefix = editor.getTextBeforeCursor(2000);   // 当前文件前文
  const cursorSuffix = editor.getTextAfterCursor(1000);    // 当前文件后文

  // 关键：检索最近编辑过的相关文件
  const recentFiles = editor.getRecentlyEditedFiles(5);
  const relevantSnippets = [];
  for (const file of recentFiles) {
    // 找当前文件 prefix 里引用到的标识符
    const identifiers = extractIdentifiers(cursorPrefix);
    for (const ident of identifiers) {
      const definition = index.findDefinition(ident, file.path);
      if (definition) {
        relevantSnippets.push({
          file: file.path,
          content: definition.snippet,
          score: computeRelevance(ident, cursorPrefix),
        });
      }
    }
  }

  return {
    prefix: cursorPrefix,
    suffix: cursorSuffix,
    // 用特殊标记把跨文件上下文注入
    crossFileContext: relevantSnippets.slice(0, 3),
  };
}

另一个关键优化是 投机补全（speculative completion）。Cursor 不是等用户停下来才补全，而是预测用户接下来可能打的几个方向，提前发起多个补全请求，用户真停下来了时已经有结果可显示。这种「预打」策略让感知延迟低于实际请求延迟，是 Tab 体感快的重要原因。

★ 二、Composer：多文件编辑的交互革命

Cursor Tab 是单文件、行内的。Composer 是 多文件、块级的编辑模式。用户在 Composer 面板描述「我要做什么」，Cursor 生成一个跨多个文件的 diff，用户审查后一键应用。

Composer 的核心交互创新是 diff-first UX。传统 Agent（包括 Aider、Cline）是边生成边应用——LLM 输出 SEARCH/REPLACE，工具立即改文件。Composer 不这样——它让 LLM 先生成完整的修改方案（所有文件的 diff），用户在 UI 里看到「要改这些文件、每个文件改这些地方」的预览，确认后才统一应用。这种「先预览后应用」的模式，让用户对 AI 的改动有完整掌控。

// Composer 的工作流（概念示意）
interface ComposerProposal {
  files: Array<{
    path: string;
    changeType: "create" | "modify" | "delete";
    diff: UnifiedDiff;          // 标准的 unified diff 格式
    rationale: string;          // 为什么改这个文件
  }>;
  summary: string;              // 整体修改说明
}

// 用户发起 Composer 任务
async function runComposer(task: string, context: ContextItem[]) {
  // 1. Agent 循环：理解任务、读相关文件、生成方案
  const proposal = await agentLoop(task, context);

  // 2. 展示 diff 预览给用户（不立即应用）
  ui.showDiffPreview(proposal);

  // 3. 等待用户决定
  const decision = await ui.waitForUserDecision();
  if (decision === "accept") {
    await applyAllDiffs(proposal.files);
  } else if (decision === "accept-partial") {
    // 用户可以只接受部分文件的改动
    await applyAllDiffs(decision.selectedFiles);
  } else {
    // 拒绝，什么都不改
  }
}

Composer 还引入了一个重要概念：multi-file 编辑的原子性。如果一个修改方案涉及 5 个文件，其中第 3 个应用失败，前两个已经改的怎么办？Composer 的做法是「全有或全无」——先在临时空间准备所有改动，全部成功才提交到实际文件系统，任意一个失败就回滚。这种事务性保证了不会留下「改了一半」的不一致状态。

★ 三、Agent 模式：自主任务执行

Composer 的局限是它只做「代码修改」——读文件、生成 diff、应用。它不会跑测试、不会装依赖、不会迭代修复。Agent 模式补上了这一层——它不仅能改代码，还能 执行命令、观察结果、迭代修正，直到任务完成。

Agent 模式的循环和前面章节讲的通用 Agent 循环（OpenCode、Codex）本质上一样：调模型 → 模型请求工具（读文件/写文件/执行命令）→ 执行 → 反馈 → 继续。但 Cursor 的 Agent 在 UX 上有几个独到设计：

  • 可见的思考过程：Agent 模式会把模型的「思考」展示在 UI 里，让用户看到它当前在干什么、计划下一步做什么。这借鉴了 Cline 的透明性设计（参见 168 章）。
  • 检查点回滚：每完成一个子步骤（如「装好了依赖」「跑通了测试」），Agent 创建一个检查点。用户可以随时回滚到任何检查点，不会因为 Agent 后面走错方向而前功尽弃。
  • 权限分级：和 Codex 类似（参见 149 章），Agent 模式支持 allow/ask/deny，用户可以配置哪些操作自动批准。

★ 四、Cursor 的上下文工程

Cursor 能在 Agent 模式表现出色，很大程度归功于它的 上下文工程——决定哪些信息进 context、以什么顺序、用什么格式。这是闭源产品里最难被复制的部分。

公开信息显示 Cursor 的上下文工程包含几个层次：

  1. 代码库索引：类似 Continue（参见 166 章），用 embedding + 关键词混合检索。但 Cursor 的索引规模和质量可能更高（有专门团队优化）。
  2. 文档检索：@docs 能检索第三方库的官方文档。Cursor 预索引了大量流行库的文档，让 Agent 能引用准确的 API。
  3. Git 历史感知：@git 能让 Agent 看到 commit 历史、PR 讨论，理解代码的演进脉络。
  4. Web 检索：@web 让 Agent 联网搜索，解决「知识截止日期之后的信息」（如新版本的 API 变化）。
// 上下文工程的拼装（概念示意）
function buildAgentContext(task: string, mentions: Mention[]): Message[] {
  const contextBlocks: ContextBlock[] = [];

  for (const mention of mentions) {
    switch (mention.type) {
      case "@file":
        contextBlocks.push({
          type: "file",
          content: readFile(mention.path),
          priority: 10,   // 用户显式引用的优先级最高
        });
        break;
      case "@codebase":
        // 自动检索相关代码
        const results = hybridSearch(task, topK=5);
        contextBlocks.push({
          type: "retrieved_code",
          content: results.map(r => r.content).join("\n---\n"),
          priority: 5,
        });
        break;
      case "@docs":
        const docs = searchDocs(mention.query);
        contextBlocks.push({
          type: "docs",
          content: docs,
          priority: 4,
        });
        break;
      case "@web":
        const webResults = webSearch(mention.query);
        contextBlocks.push({
          type: "web",
          content: webResults,
          priority: 3,
        });
        break;
    }
  }

  // 按 priority 排序，高优先级放前面（lost-in-the-middle 效应让开头结尾更重要）
  contextBlocks.sort((a, b) => b.priority - a.priority);

  // 组装成消息
  const contextText = contextBlocks
    .map(b => formatBlock(b))
    .join("\n\n");

  return [
    { role: "system", content: systemPrompt },
    { role: "user", content: `${contextText}\n\n任务：${task}` },
  ];
}

注意 priority 排序这一步——它不是随意堆砌上下文，而是按重要性排序。这个排序考虑了 lost-in-the-middle 效应（模型对 context 开头和结尾的注意力高于中间），所以最相关的信息放前面或后面，次要的放中间。这种细节是上下文工程的精髓。

★ 五、Cursor 的模型策略

Cursor 不是简单调 OpenAI/Anthropic 的 API，它有复杂的模型组合策略。公开信息显示：

  • 补全用自研小模型：Tab 补全用专门微调的中等模型，兼顾速度和质量。这是延迟和成本的关键。
  • Chat/Agent 用前沿大模型：GPT-4、Claude Sonnet/Opus、Gemini 等，按用户订阅档位提供。
  • 路由策略：简单任务路由到便宜模型，复杂任务路由到强模型。这种「模型路由」是降低成本的常见手段。
  • prompt caching：对不变的 system prompt 和长上下文用 caching，减少重复计算的开销。

★ 六、对开源生态的影响

虽然 Cursor 闭源，它的 UX 设计深刻影响了开源 Agent。我们能在多个开源项目里看到 Cursor 的影子：

  • diff-first UX：Continue、Cline 后来都加入了「先预览 diff 再应用」的模式。
  • 跨文件补全：Continue 的 autocomplete 也加入了最近编辑文件的上下文。
  • @引用语法：@file、@codebase、@docs 成了 Agent 的通用交互语法。
  • 检查点回滚：几乎所有 Agent 都有了某种形式的「撤销 AI 改动」能力。
  • Agent 模式：从 Composer 到 Agent 的演进，定义了「AI 编程助手」的新形态。

理解 Cursor 的演进，有助于你看懂整个 AI 编程工具赛道的趋势——从补全到编辑到自主，AI 的角色在不断加深。开源项目在追赶这条路线，而理解路线图能帮你判断一个开源项目的成熟度和方向。

★ 本章小结

Cursor 的三阶段演进定义了 AI 编程工具的成熟度阶梯。要点回顾：

  • Tab：行内补全，突破在跨文件上下文和投机补全（预打）降低感知延迟。
  • Composer：多文件块级编辑，diff-first UX 先预览后应用，事务性保证不半改。
  • Agent：自主任务执行，可见思考过程、检查点回滚、权限分级。
  • 上下文工程：codebase/docs/git/web 多源检索，按 priority 排序对抗 lost-in-the-middle。
  • 模型策略：补全用自研小模型，Chat/Agent 用前沿大模型，路由降成本，caching 省计算。
  • 生态影响：diff-first UX、@语法、检查点、Agent 模式成了开源 Agent 的标配。
