端侧 RAG 实战

从零搭一个完全跑在本地的私有知识库

深度进阶★★★

为什么"端侧 RAG"是一个独立的工程命题

RAG（检索增强生成）在云端是标配了，但"端侧 RAG"——整个 RAG 流水线（嵌入、检索、生成）都跑在用户设备上——有它独特的一组约束，不能用云端 RAG 的方案平移过来：

维度云端 RAG端侧 RAG
嵌入模型API（OpenAI/Cohere）或大模型本地小模型（bge-small/gte-small）
向量库Milvus/Qdrant 服务zvec 嵌入式
LLMGPT-4 / Claude API本地小模型（Qwen/Llama 7B 量化）
延迟网络主导（100-500ms）本地计算（10-100ms）
隐私数据要出设备完全不出设备
成本按 token 付费零边际成本（电费）
离线不可用完全可用

这一章我们用 zvec 把端侧 RAG 的"检索"那一环搭起来——这是端侧 RAG 最容易出问题的一环（嵌入模型小、向量库要轻、内存有限），也是最契合 zvec 的一环。

💡 要点
端侧 RAG 的核心权衡：用更小的嵌入模型（牺牲一点召回）、更轻的向量库（zvec）、更小的 LLM（牺牲一点生成质量），换来"零延迟、零成本、完全隐私、可离线"。这是一组完全不同于云端 RAG 的优化目标。

完整流水线：文档 → 切块 → 嵌入 → 入库 → 检索 → 生成

下面是一个完整的端侧 RAG 流水线，使用的是 zvec 真实 API（CollectionSchema / Field / create_and_open / insert / VectorQuery / query），不是伪代码：

# === 端侧 RAG 流水线（基于 zvec 真实 API）===
import zvec
from zvec import CollectionSchema, Field, FieldType, MetricType, IndexType, VectorQuery
from embed import LocalEmbedder       # 本地嵌入（如 bge-small-zh，512 维）
from llm import LocalLLM              # 本地 LLM（如 Qwen2.5-7B Q4）
from chunker import MarkdownChunker

# 0. 进程级初始化（只调一次）——给 zvec 一个内存预算
zvec.init(memory_limit_mb=512, query_threads=4)

# 1. 打开（或创建）库 + collection
schema = CollectionSchema(
    name="docs",
    fields=[
        Field(name="id",   dtype=FieldType.INT64, is_primary=True),
        Field(name="vec",  dtype=FieldType.FLOAT_VECTOR, dim=512,
              index=IndexType.HNSW, metric=MetricType.COSINE),
        Field(name="src",  dtype=FieldType.STRING),   # 来源文件
        Field(name="text", dtype=FieldType.STRING),   # 原文 chunk
        Field(name="off",  dtype=FieldType.INT32),    # 偏移
    ],
)
db = zvec.create_and_open("kb.zv", schema=schema)
col = db.collection("docs")
embedder = LocalEmbedder(model="bge-small-zh", dim=512)
llm = LocalLLM(model="qwen2.5-7b-q4")

# 2. 摄入流水线
def ingest(markdown_path):
    text = open(markdown_path).read()
    chunks = MarkdownChunker(chunk_size=256, overlap=32).split(text)
    vectors = embedder.encode([c.text for c in chunks])   # 批量嵌入
    col.insert([
        {"id": i, "vec": vectors[i],
         "src": markdown_path, "text": c.text, "off": c.offset}
        for i, c in enumerate(chunks)
    ])
    db.flush()   # WAL 落盘

# 3. 检索 + 生成
def ask(question, k=4, ef_search=80):
    qvec = embedder.encode([question])[0]
    q = VectorQuery(field="vec", vector=qvec, topk=k, metric=MetricType.COSINE,
                    params={"ef_search": ef_search})
    hits = col.query(q)
    context = "\n\n".join(h["text"] for h in hits)
    prompt = f"参考资料:\n{context}\n\n问题: {question}\n回答:"
    return llm.generate(prompt, max_tokens=512)

逐环节拆解：每个选择的理由

① 嵌入模型：为什么选 bge-small 而不是 OpenAI

端侧嵌入模型有三个硬约束：① 模型要小（< 200MB，能塞进 App）；② CPU/低显存能跑（不能依赖 GPU）；③ 中英文都要好（很多场景是中文）。BAAI 的 bge-small-zh（约 95MB）、阿里的 gte-small（约 90MB）是当前主流选择。它们在 MTEB 上的召回比 text-embedding-3-large 低 5-10 个百分点，但小 30 倍、快 20 倍，端侧值得这个交换。

一个关键优化：批量编码。嵌入模型的 forward 单条和批量几乎一样慢（瓶颈在 Python/调度开销，不在算力），所以 ingest 时一定要批量 encode（如一次 32 条），不要一条条 encode。上面 ingest() 里就是批量嵌入。注意 zvec 的 col.insert() 也接受列表，可以一次插多条，省去反复调用的开销。

② 切块策略：端侧要更"碎"

端侧 LLM 上下文窗口小（7B 模型常用 4K-8K），所以每个 chunk 不能太大，否则塞不下几个。常用 chunk_size=256-512 tokens、overlap=32-64。overlap 是为了"重要内容被切在边界时不丢"。更高级的切块（按标题/段落语义切）在端侧也值得做，因为坏切块的代价在端侧更高（上下文本来就紧）。

③ 入库：HNSW 的 M 选择

端侧知识库通常是"中小规模 + 一次性建好"——几十万到几百万个 chunk，建好后查询为主。这种场景 HNSW 最合适。参考官方 1M benchmark（M=15），端侧几十万 chunk 可以用 M=12-16 够用，省内存。

④ 检索：efSearch 的动态调整

一个端侧特有的小技巧：efSearch（通过 VectorQuery 的 params 传）可以按"用户是否在看实时结果"动态调。用户打字时（流式预览），用低 ef_search（32）快速出结果；用户按回车（正式查询），用高 ef_search（80）出高质量结果。这种"两段式"在端侧能显著改善"响应感"，因为端侧用户对延迟比云端敏感。zvec 的 VectorQuery 支持每次查询带不同 params，正适合这种动态切换。

💡 要点
端侧 RAG 的三大隐藏陷阱：① 嵌入不批量（慢 10 倍）；② 切块太大（上下文塞不下）；③ efConstruction 调太小（索引质量差，召回永远上不去）。这三点比换更强的模型更重要。

增量更新：用户加了新文档怎么办

知识库不是一次性的，用户会持续加文档。zvec 的增量更新很直接：col.insert() 新记录即可，HNSW 支持在线插入。背后 WAL 保证这些插入即使没 flush 也不会丢——进程崩了，下次打开 replay WAL 就回来了。需要注意：频繁小批量插入时，记得定期 flush 让 WAL 落盘 + 触发后台 checkpoint，避免 WAL 文件无限增长。

    # 后台 checkpoint 会异步把内存索引快照写到数据文件

持久化与冷启动：App 重启后索引还在

用户关掉 App 再打开，索引不应该重建。zvec 的做法：每次 flush 把 WAL 落盘，后台 checkpoint 周期性把完整索引快照写到 kb.zv 数据文件，App 启动时 zvec.open("kb.zv") 直接加载最近的 checkpoint + replay WAL，毫秒级就绪。崩溃了也不会丢——WAL 里 checkpoint 之后的写入会被 replay。这是嵌入式库相比"每次启动重建索引"的决定性优势。

一个优化：App 启动后可以后台预热（warm-up）——跑几个常见查询，把热数据对应的页提前加载进物理内存（如果 zvec 用了 mmap），避免用户第一次查询时遇到冷启动延迟。这种"后台预热 + 前台查询"是端侧性能优化的常见模式。

端侧 vs 云端 RAG：什么时候该选哪个

场景推荐理由
个人笔记/日记/私人文档端侧隐私 + 离线 + 零成本
企业内部知识库（涉密）端侧 + 私有云 LLM数据不出设备
客服 FAQ（百万级）云端规模 + 多租户
代码助手（本地代码库语义搜索）端侧低延迟 + 代码不外传
公开知识问答（维基/新闻）云端规模 + 不涉及隐私
车载/机器人本地记忆端侧弱网/离线 + 低延迟
💡 要点
决策口诀：涉隐私→端侧，要离线→端侧，规模大→云端，多用户共享→云端。zvec 不是来取代云端 RAG 的，它让"本来该端侧的"那些场景终于有了顺手的工具。

边缘 RAG 部署实战：端侧检索与生成

前面讲了 zvec 在云端的部署，但有一类场景越来越重要——边缘 RAG：在手机、车载设备、离线嵌入式设备上跑完整的"检索 + 生成"。它的驱动力是隐私（数据不出端）、延迟（无网络往返）、离线可用（无网环境也能问答）。但边缘设备的算力、内存、电量都极度受限，这迫使 RAG 的每个环节都要做激进压缩：embedding 模型要从 768 维降到 128 维、向量要从 FP32 量化到 INT8、LLM 要从 70 亿参数量化到 4 比特甚至更小。我们实现一个最小可用的 EdgeRAG，把"量化索引 + 端侧生成"串起来，帮你建立边缘部署的资源敏感度。

# -*- coding: utf-8 -*-
"""边缘 RAG 部署：在手机/嵌入式设备上跑向量检索 + 小模型生成。
核心是"小 embedding 模型 + 量化索引 + 端侧 LLM"三件套。
"""
import numpy as np

class EdgeRAG:
    def __init__(self, embed_dim=128, topk=3):
        # 边缘端用低维 embedding(省内存) + INT8 量化向量
        self.dim = embed_dim
        self.topk = topk
        self.docs = []                # 文本片段
        self.vecs = None              # INT8 量化向量 [N, dim]

    def build(self, texts, embed_fn):
        """离线建库: embed + INT8 量化压缩。"""
        self.docs = texts
        fvecs = np.array([embed_fn(t) for t in texts], dtype="float32")
        scale = fvecs.abs().max() / 127.0
        self.vecs = np.round(fvecs / scale).astype(np.int8)
        self.scale = scale
        print(f"建库: {len(texts)} 条, 量化后内存="
              f"{self.vecs.nbytes/1024:.1f} KB")

建库阶段的关键决策是embedding 维度和量化精度。维度越低内存越省，但区分度下降、召回变差；128 维是边缘端的甜点，能在召回率和内存间取得平衡。量化用最简单的对称 INT8：把每个维度缩放到 [-127,127] 整数，内存从 4 字节降到 1 字节，对余弦相似度的影响通常小于 2%。查询阶段全在端侧完成，没有网络往返，延迟完全取决于本地算力。下面是在线检索 + 生成的实现：

    def query(self, q_vec, llm_fn):
        """在线检索 + 生成, 全在端侧完成。"""
        q = np.array(q_vec, dtype="float32")
        # INT8 近似距离(查表, 极快)
        db = self.vecs.astype(np.float32) * self.scale
        sim = db @ q / (np.linalg.norm(db, axis=1) *
                        np.linalg.norm(q) + 1e-9)
        idx = np.argpartition(-sim, self.topk)[:self.topk]
        ctx = "\n".join(self.docs[i] for i in idx[np.argsort(-sim[idx])])
        return llm_fn(ctx)

# 演示: 假装 embedding 和 llm
def fake_embed(t):
    rng = np.random.default_rng(hash(t) % (2**32))
    return rng.standard_normal(128).astype("float32")

def fake_llm(ctx):
    return f"(端侧生成) 基于{len(ctx)}字上下文的回答"

rag = EdgeRAG(embed_dim=128, topk=3)
rag.build(["GLM-5.2 支持百万上下文",
           "HNSW 是向量库默认索引",
           "边缘部署要量化压缩"], fake_embed)
print(rag.query(fake_embed("GLM 特性"), fake_llm))

把这套边缘 RAG 落到真实设备，有三个工程权衡要注意。第一，embedding 模型要专门蒸馏到小尺寸：通用的 768 维模型在手机上太重，社区有专门蒸馏的 128/256 维小模型（如 all-MiniLM-L6-v2 的量化版），推理快 5-10 倍。第二，端侧 LLM 选 1B~3B 的量化模型：4 比特量化的 1B 模型只需约 700MB 内存，在现代手机上可跑，虽然能力弱于云端大模型，但配合 RAG 检索的上下文，能在垂直任务上够用。第三，索引要支持懒加载：把向量库按主题分片，只把当前会话相关的分片加载进内存，其余留在闪存按需换入，这样能在内存受限的设备上支撑较大的知识库。边缘 RAG 不是云端 RAG 的"阉割版"，而是面向隐私、延迟、离线场景的独立设计——理解了它的约束，你就能在手机、车机、IoT 设备上做出真正好用的智能问答。

本章小结

端侧 RAG 是一组独特约束下的工程：小嵌入模型（bge-small）、轻量向量库（zvec HNSW）、小 LLM（7B 本地量化）、紧上下文（chunk 要小）。我们用真实 API（CollectionSchema / Field / create_and_open / insert / VectorQuery / query）搭了一个完整流水线。三大隐藏陷阱（嵌入要批量、切块要小、efConstruction 别太小）比换模型更影响效果。WAL + checkpoint 让"装上 App 就有本地知识库、断电不丢"成为现实。下一章我们看 zvec 怎么在手机/IoT 这种极端受限设备上跑得动。
