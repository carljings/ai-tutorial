Codex Guardian 守护进程
网络审计与 auto-approve 规则引擎

沙箱能挡住越权的文件和网络访问，但挡不住「合法但可疑」的行为——比如 Agent 在被允许的网络白名单内，把源码上传到一个看起来合法但其实陌生的域名。Codex 的 Guardian 是一个独立运行的守护进程，它不阻止操作，而是 审计 每一次网络访问、维护一套 auto-approve 规则引擎，让 Agent 的行为可追溯、可治理。本章我们拆解 Guardian 的架构和规则引擎的设计。

第 156、157 章讲的沙箱（Seatbelt / Landlock）解决的是「能不能访问」的问题——某个路径、某个域名在不在白名单里。但它们解决不了「这个在白名单里的访问，到底应不应该发生」的问题。

举个具体场景：Agent 为了装依赖，被允许访问 registry.npmjs.org（这是合理的）。但如果 Agent 发起的请求里，除了访问 npm registry，还顺便把项目的源码作为 POST body 发到了同一个域名下的某个 path——沙箱是放行的（因为域名在白名单），但这显然是可疑行为。

Guardian 就是为了捕获这类「合法但可疑」的行为。它作为一个旁路（sidecar）守护进程，记录 Agent 发起的所有网络请求，对请求的模式做规则匹配，发现异常就告警（或阻断）。这和企业的 DLP（Data Loss Prevention）系统思路类似——不是阻止你用网络，而是监控你在网络上做什么。

★ 二、Guardian 的架构：旁路而非内联

Guardian 的设计选择是 旁路审计，而不是内联拦截。这意味着 Agent 的命令执行流程不经过 Guardian——Guardian 只通过其他机制（如代理、日志订阅）观察行为，不成为执行链路的一环。这样做的好处是不会增加执行延迟、不会因为 Guardian 故障导致 Agent 卡死。

实际实现里，Guardian 作为本地 HTTP/HTTPS 代理运行。沙箱内的进程被配置成 HTTP_PROXY=http://localhost:<guardian-port>，所有出站请求先到 Guardian，由它审计后再转发。这种「代理模式」是透明的——Agent 的代码不需要改，网络库（curl、node fetch）自动尊重环境变量。

★ 三、审计日志：记录每一次网络访问

Guardian 的核心输出是一份审计日志，记录 Agent 会话期间所有出站请求。我们看它的数据结构：

// codex-rs/src/guardian/audit.rs（概念示意）
use serde::{Serialize, Deserialize};
use std::time::SystemTime;

#[derive(Serialize, Deserialize)]
pub struct NetworkAuditEntry {
    pub timestamp: u64,           // Unix 秒
    pub session_id: String,       // 哪个会话发的
    pub method: String,           // GET / POST / ...
    pub url: String,              // 完整 URL
    pub host: String,             // 提取出的 host
    pub request_size: usize,      // 请求体大小（字节）
    pub response_status: Option<u16>,
    pub response_size: Option<usize>,
    pub triggered_by: String,     // 触发它的命令或工具调用 ID
    pub classification: RequestClass,
}

#[derive(Serialize, Deserialize)]
pub enum RequestClass {
    /// 已知安全的包管理器请求
    PackageRegistry,
    /// 已知的代码托管平台
    CodeHost (String),            // 如 "github.com"
    /// 文档/参考查询
    Documentation,
    /// 陌生或可疑的端点
    Unknown,
    /// 命中阻断规则
    Blocked (String),             // 触发的规则 ID
}

impl Guardian {
    pub async fn audit_request(&self, req: &Request) -> RequestClass {
        let host = req.url.host_str().unwrap_or("");
        let class = self.classify(host, req.method.as_str(), req.body_size);
        // 写入审计日志（追加模式，类似 OpenCode 的 JSONL）
        self.log.write(NetworkAuditEntry {
            timestamp: now(),
            host: host.into(),
            url: req.url.to_string(),
            method: req.method.clone(),
            request_size: req.body_size,
            // ... 其他字段
            classification: class.clone(),
            // ...
        }).await;
        class
    }

    fn classify(&self, host: &str, method: &str, body_size: usize) -> RequestClass {
        // 已知包管理器
        if NPM_HOSTS.contains(host) || PYPI_HOSTS.contains(host) {
            // 但如果是 POST 且 body 很大，可能是 publish（可疑）
            if method == "POST" && body_size > 100_000 {
                return RequestClass::Unknown;  // 大 POST 到 registry，标记审查
            }
            return RequestClass::PackageRegistry;
        }
        if host.ends_with("github.com") || host.ends_with("githubusercontent.com") {
            return RequestClass::CodeHost(host.into());
        }
        // 走规则引擎做进一步匹配
        if let Some(rule_id) = self.rules.check_blocked(host, method) {
            return RequestClass::Blocked(rule_id);
        }
        RequestClass::Unknown
    }
}

注意 classify 函数里的一个细节：即使是包管理器域名，如果是大 POST 请求也会被标记为 Unknown。这是因为正常的 install 是 GET（下载包），而 POST 且 body 大往往是 publish（上传包）——Agent 不应该自己往 npm 上发包，这种模式值得审查。这种「基于行为模式而非单纯域名」的分类，是 Guardian 比单纯白名单更聪明的体现。

★ 四、auto-approve 规则引擎：减少确认疲劳

第 149 章我们讲了权限的三态模型，以及「allow-always 学习」如何减少用户确认。Guardian 的 auto-approve 引擎是这个思路的延伸——它维护一套 命令模式到安全等级的映射，让已知安全的模式自动放行。

// codex-rs/src/guardian/rules.rs（概念示意）
use regex::Regex;

pub struct RuleEngine {
    rules: Vec<Rule>,
}

pub struct Rule {
    pub id: String,
    pub pattern: RulePattern,
    pub action: RuleAction,
    pub description: String,
}

pub enum RulePattern {
    /// 命令前缀匹配
    CommandPrefix(String),
    /// 命令正则匹配
    CommandRegex(Regex),
    /// 网络目标匹配
    NetworkHost(String),
}

pub enum RuleAction {
    /// 自动批准，不问用户
    AutoApprove,
    /// 必须问用户
    RequireApproval,
    /// 一律拒绝
    Deny,
    /// 记录但不干预
    AuditOnly,
}

impl RuleEngine {
    pub fn new() -> Self {
        // 内置规则集
        let rules = vec![
            Rule {
                id: "auto-test".into(),
                pattern: RulePattern::CommandPrefix("npm test".into()),
                action: RuleAction::AutoApprove,
                description: "运行测试是无副作用的安全操作".into(),
            },
            Rule {
                id: "auto-git-status".into(),
                pattern: RulePattern::CommandRegex(
                    Regex::new(r"^git (status|diff|log)").unwrap()
                ),
                action: RuleAction::AutoApprove,
                description: "只读 git 操作".into(),
            },
            Rule {
                id: "deny-force-push".into(),
                pattern: RulePattern::CommandRegex(
                    Regex::new(r"git push.*--force").unwrap()
                ),
                action: RuleAction::Deny,
                description: "禁止强制推送".into(),
            },
            Rule {
                id: "audit-curl".into(),
                pattern: RulePattern::CommandRegex(
                    Regex::new(r"(curl|wget)\s+").unwrap()
                ),
                action: RuleAction::AuditOnly,
                description: "网络下载操作，记录但不阻止".into(),
            },
        ];
        Self { rules }
    }

    pub fn evaluate(&self, command: &str) -> &RuleAction {
        for rule in &self.rules {
            if rule.pattern.matches(command) {
                return &rule.action;
            }
        }
        // 默认需要审批
        &RuleAction::RequireApproval
    }
}

这套规则引擎的精妙之处在于 四档 action 而非简单的 allow/deny。AutoApprove 是「已知安全直接放」、RequireApproval 是「不确定问用户」、Deny 是「明确禁止」、AuditOnly 是「放行但记录」。最后一档特别有价值——有些操作（如 curl 下载）不一定危险，但值得留痕，AuditOnly 让你不打断流程的同时保留追溯能力。

★ 五、规则的学习与演进

内置规则集是静态的，但 Guardian 还支持 从用户行为中学习。当用户多次手动批准同一类命令（比如 cargo build），Guardian 会建议把这条规则加入 auto-approve 列表。这种「观察 → 建议 → 用户确认」的模式，比全自动学习更安全——规则变更始终在用户的知情同意下。

// 学习逻辑（概念示意）
impl Guardian {
    pub fn maybe_suggest_rule(&mut self, session_id: &str) {
        // 统计最近 N 次会话里被手动批准的命令
        let approvals = self.log.recent_approvals(session_id, 10);
        let mut counts: HashMap<String, usize> = HashMap::new();
        for cmd in &approvals {
            // 把命令归一化（去掉参数，只看主命令）
            let normalized = normalize_command(cmd);
            *counts.entry(normalized).or_insert(0) += 1;
        }
        // 同一模式被批准超过 5 次，建议加入 auto-approve
        for (cmd, count) in counts {
            if count >= 5 && !self.is_already_auto(cmd) {
                self.suggestions.push(Suggestion {
                    pattern: cmd,
                    reason: format!("最近被手动批准 {} 次", count),
                });
            }
        }
    }
}

注意「归一化」这一步——它把 npm test --verbose、npm test --coverage 归并成 npm test* 来统计，否则每个变体都算独立命令，永远达不到阈值。这种归一化策略在行为分析里很常见。

D1 深度剖析：Guardian AI 审批的决策依据

本章介绍了 Codex 的 Guardian AI 审批机制。这一节我们深入 Guardian 如何综合多维信号做审批决策，并对比它与规则审批的优劣。

# Guardian AI 审批的决策特征（示意）
class GuardianDecision:
    def __init__(self):
        self.features = [
            "command_category",      # read/write/network/dangerous
            "target_paths",          # 影响的文件路径
            "path_sensitivity",      # 路径敏感度（系统/用户/项目）
            "command_history",       # 本会话历史命令
            "deviation_score",       # 与常规模式的偏离度
            "sandbox_mode",          # 当前沙箱强度
            "user_trust_level",      # 用户信任等级
        ]

    def decide(self, cmd, ctx) -> dict:
        score = self._score(cmd, ctx)   # 综合打分 0~1
        if score < 0.3:
            return {"action": "allow", "reason": "低风险常规操作"}
        elif score < 0.7:
            return {"action": "ask", "reason": self._explain(cmd, ctx)}
        else:
            return {"action": "deny", "reason": self._explain(cmd, ctx)}

    def _explain(self, cmd, ctx):
        # 关键：必须给出人类可读的理由，不能黑箱拒绝
        reasons = []
        if ctx.path_sensitivity == "system":
            reasons.append("影响系统关键路径")
        if ctx.deviation_score > 0.8:
            reasons.append("偏离常规命令模式")
        return "；".join(reasons) or "风险较高，建议人工确认"

Guardian 相比纯规则的优势在于能理解上下文和意图。规则只能机械匹配"是否含 rm"，Guardian 能综合"这次 rm 是否符合本次任务模式、目标路径是否敏感、与历史命令是否一致"做判断。它的短板是可解释性——AI 决策必须给出理由，否则用户无法信任，这也是上面代码强调 _explain 的原因。

D2 生产避坑：AI 审批的风险与缓解

风险 1：AI 被绕过（对抗攻击）

Guardian 本身是 LLM，可能被精心构造的命令描述误导（如把恶意命令包装成"清理临时文件"）。缓解：不要让 Guardian 只看 LLM 的文字描述，必须看实际命令和参数，并对参数做独立校验。Guardian 是规则的补充，不能替代硬规则。

风险 2：延迟影响体验

每次审批都调 LLM 推理，延迟可能达数百毫秒，在快速连续操作时拖慢体验。优化：分层决策——规则能定的不调 Guardian，只有 unknown 或高风险才升级给 Guardian。Guardian 只处理 10% 的疑难决策。

风险 3：模型一致性

Guardian 用的模型若升级，审批行为可能变化，导致同样的命令昨天允许今天拒绝。建议对 Guardian 决策做版本化 + 回归测试，关键命令有固定的 golden 决策集，升级后必须通过。

★ 本章小结

Guardian 是 Codex 安全体系的「观察者」角色。要点回顾：

  • 旁路而非内联：Guardian 作为本地代理观察流量，不增加执行延迟，故障不影响 Agent。
  • 解决「合法但可疑」：沙箱管「能不能访问」，Guardian 管「该不该访问」，捕获白名单内的异常行为。
  • 基于行为模式分类：同样是 npm registry，GET 是正常 install，大 POST 可能是可疑 publish，分类器看模式不看域名。
  • 四档 action：AutoApprove / RequireApproval / Deny / AuditOnly，比二态更精细，AuditOnly 保留追溯不打断流程。
  • 学习 + 建议：观察用户多次批准的模式，建议加入 auto-approve，变更始终需用户确认。
  • 命令归一化：统计行为时把参数变体归并，避免每个变体独立计数达不到阈值。
  • 可读审计报告：按类别汇总网络访问，陌生端点高亮，让 Agent 行为可追溯。
