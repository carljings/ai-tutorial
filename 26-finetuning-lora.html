微调技术与 LoRA/QLoRA
低成本定制大模型

不是每个人都有资源从零训千亿模型。微调让你在现有模型基础上，用少量数据和算力定制能力。本章深入全参数微调、LoRA、QLoRA、Adapter、Prefix Tuning 的原理与实现，给你一份「怎么选、怎么做」的完整指南。

阅读约 40 分钟难度 ★★★★前置：第 02 章

LoRA 参数量与显存估算

LoRA 到底省了多少参数和显存？很多人凭感觉说'省很多'，但工程上需要精确数字来决定能否部署。下面这段代码精确计算全参 vs LoRA 的可训参数量、优化器状态显存、训练峰值显存，是做 LoRA 部署预算的必备工具。

# LoRA vs 全参微调: 参数量与显存对比
def lora_params(in_f, out_f, r):
    """一层 LoRA 可训参数 = A(r×in) + B(out×r)。"""
    return r * in_f + r * out_f
def full_params(in_f, out_f):
    return in_f * out_f
# 模拟一个 7B 模型(简化为若干 Linear 层)
layers = [(4096, 4096)] * 32 + [(4096, 12288)] * 32   # 近似 qkv/ffn
full = sum(i*o for i, o in layers)
for r in [4, 8, 16, 64]:
    lora = sum(lora_params(i, o, r) for i, o in layers)
    ratio = lora / full * 100
    # AdamW 优化器状态 = 2×参数(FP32 m,v) -> 训练显存
    full_train_gb = full * (2 + 4) * 2 / 1e9   # 权重FP16+梯度+adam
    lora_train_gb = (full * 2 + lora * (2 + 4)) / 1e9  # 冻结权重FP16 + LoRA全精度
    print(f"r={r:3d}: 可训参数={lora/1e6:6.1f}M ({ratio:.2f}%)  "
          f"训练显存 全参={full_train_gb:.1f}GB LoRA={lora_train_gb:.1f}GB")
# 典型: r=8 时可训参数仅占 ~0.2%, 训练显存降数倍。

10 LoRA 训练 Loss 曲线深度解读

前面讲了 LoRA 的数学原理（用两个低秩矩阵 A·B 近似全量权重更新 ΔW），但原理理解了，不等于能在实战里读懂 loss 曲线。很多新手第一次跑 LoRA 时会被 loss 曲线"吓到"——它降得比全参微调慢、有时还会抖。这其实是特征而非 bug。我们用一个可控的小实验把全参微调、LoRA(r=8)、LoRA(r=64) 三种设置的 loss 曲线画在一起，你会直观看到秩 r 如何在"训练速度"和"正则化强度"之间做权衡。

先实现一个最小可用的 LoRA Linear 层。关键细节有两个：① 原权重 W 必须 requires_grad_(False) 冻结，否则就不是 LoRA 了；② B 初始化为零、A 用小随机值，保证训练开始时 ΔW = B·A = 0，模型行为和原模型完全一致（这叫"零初始化"，是 LoRA 训练稳定的基石）。

# -*- coding: utf-8 -*-
"""LoRA 训练 loss 曲线解读：对比全参微调、LoRA(r=8)、LoRA(r=64)。
用一个小型回归任务演示三种设置的 loss 走势差异。
"""
import torch, torch.nn as nn

class LoRALinear(nn.Module):
    """y = W x + (B A) x，W 冻结，只训 A、B。"""
    def __init__(self, in_f, out_f, r=8):
        super().__init__()
        self.W = nn.Linear(in_f, out_f, bias=False)
        self.W.weight.requires_grad_(False)        # 冻结原权重
        self.A = nn.Parameter(torch.randn(r, in_f) * 0.01)
        self.B = nn.Parameter(torch.zeros(out_f, r))  # B 初始化为 0
    def forward(self, x):
        return self.W(x) + (x @ self.A.t() @ self.B.t())

def make_model(full=False, r=8):
    if full:
        m = nn.Sequential(nn.Linear(128, 256), nn.ReLU(),
                          nn.Linear(256, 10))
        return m
    # 把第二层换成 LoRA
    m = nn.Sequential(nn.Linear(128, 256), nn.ReLU(),
                      LoRALinear(256, 10, r=r))
    return m

用同一个数据集和同一个优化器配置，分别训练三个模型并记录 loss。把曲线画出来后，你会观察到三个现象，每一个都有明确的工程含义：第一，全参微调降得最快，但也最容易过拟合（loss 可能在后期反弹）；第二，LoRA r=8 降得明显慢，但曲线更平滑、泛化更好——低秩约束天然起到了正则化作用；第三，LoRA r=64 的曲线介于两者之间，更接近全参，代价是可训练参数更多。这条曲线告诉我们一个反直觉的事实：LoRA 的"慢"恰恰是它的优点，低秩瓶颈让模型只能学到任务最本质的方向，而不是把训练集的死记硬背也一起学进去。

# 跑训练，记录 loss 曲线
import matplotlib.pyplot as plt

torch.manual_seed(0)
X = torch.randn(2000, 128)
y = torch.randint(0, 10, (2000,))

def train(model, steps=300, lr=1e-3):
    opt = torch.optim.Adam(
        [p for p in model.parameters() if p.requires_grad], lr)
    lossf = nn.CrossEntropyLoss()
    hist = []
    for s in range(steps):
        out = model(X)
        loss = lossf(out, y)
        opt.zero_grad(); loss.backward(); opt.step()
        if s % 10 == 0:
            hist.append(loss.item())
    return hist

curves = {
    "全参微调":   train(make_model(full=True)),
    "LoRA r=8":  train(make_model(r=8)),
    "LoRA r=64": train(make_model(r=64)),
}
plt.figure(figsize=(8, 4.5))
for name, hist in curves.items():
    plt.plot(range(0, 300, 10), hist, label=name)
plt.xlabel("step"); plt.ylabel("train loss"); plt.yscale("log")
plt.title("LoRA vs 全参微调：loss 曲线对比")
plt.legend(); plt.grid(True, which="both", ls=":", alpha=.4)
plt.tight_layout(); plt.savefig("lora_loss.png", dpi=130)
# 典型结论：全参降得最快但易过拟合；LoRA r=8 降得慢但更稳；
# r=64 接近全参曲线，代价是可训参数变多。

把这套分析方法迁移到真实大模型微调上，有三条实用经验。① 看 loss 的相对收敛位置，不是绝对速度：LoRA 最终 loss 通常略高于全参，这是正常的，关键是验证集指标是否达标。② 用 r=16 作为默认起点：r=8 适合简单任务（情感分类、格式调整），r=32~64 适合注入新知识或多语言，r>128 几乎失去 LoRA 的优势，不如直接全参。③ loss 抖动通常是 batch 太小或学习率太大：LoRA 的可训参数少，梯度噪声相对更大，建议学习率设为全参的 2-3 倍（如 1e-4 ~ 5e-4），并配合更大的 batch。读懂 loss 曲线，你就掌握了 LoRA 调参的"仪表盘"。

★ 为什么需要微调

RAG 给模型外挂知识，但改变不了模型的行为方式。当你需要：

  • 特定的输出风格（品牌口吻、格式规范）；
  • 领域专属能力（医疗诊断、法律推理）；
  • 稳定的结构化输出（特定 JSON schema）；
  • 用小模型达到大模型效果（降成本）；

——这些 RAG 做不到，微调才是答案。

01 全参数微调：最彻底也最贵

更新模型所有参数。效果最好，但代价巨大：

维度全参数微调
显存需要模型大小 × (2参数 + 2梯度 + 2优化器状态 + 2激活) ≈ 16×参数字节
7B 模型~112 GB 显存（需多张 A100）
数据需要数千~数万条高质量数据
风险灾难性遗忘（学了新的忘了旧的）

02 LoRA：参数高效微调的明星

LoRA（Low-Rank Adaptation，低秩适配）的核心洞察：微调时权重的变化量 ΔW 是低秩的——可以用两个小矩阵的乘积近似。

原始权重 W（d×d，冻结不动）
▼ 加一个低秩更新
ΔW = B × A（A 是 d×r，B 是 r×d，r 远小于 d）
▼
新权重 = W + ΔW，只训练 A 和 B
图 26-1 · LoRA：冻结原权重，只训练低秩矩阵 A、B

参数量从 d² 降到 2dr（r 通常 8-64）。对 d=4096 的层，r=8 时参数量从 1600 万降到 6.5 万——减少 99.6%。

LoRA 的数学

原始：h = W·x
LoRA：h = W·x + B·A·x = (W + BA)·x
其中 W ∈ ℝ^(d×d) 冻结，A ∈ ℝ^(r×d) 用高斯初始化，B ∈ ℝ^(d×r) 初始化为 0
训练时只更新 A、B；推理时可以把 BA 合并回 W（无额外开销）

LoRA 的优势

💾

省显存

7B 模型 LoRA 只需 ~10GB，单卡可跑。

⚡

训练快

参数少，收敛快。

🔀

可切换

同一基座，多个 LoRA 模块按需加载（像一个模型多种人格）。

📦

体积小

LoRA 模块通常只有几十 MB，易分发。

🚫

无推理开销

合并回基座后，推理速度和原模型一样。

🎯

防遗忘

原权重冻结，基座能力不丢。

03 QLoRA：单卡微调千亿模型

QLoRA在 LoRA 基础上再加量化：把基座模型量化到 4-bit 存储（省显存），LoRA 参数用 fp16 训练（保精度），通过梯度反量化传播。这让你能在单张消费级显卡上微调大模型。

基座模型（4-bit 量化，省显存）
▼ 前向时反量化计算
LoRA 适配器（fp16，正常训练）
▼ 梯度通过反量化链回传
只更新 LoRA 参数，基座保持 4-bit
图 26-2 · QLoRA：4-bit 基座 + fp16 LoRA，单卡微调

QLoRA 的三大创新：NF4 量化（正态分布友好的 4-bit 格式）、双重量化（量化常数也量化）、分页优化器（防显存溢出）。效果接近全参数微调，成本降到 1/10 以下。

08 微调的常见陷阱

⚠️ 五大陷阱
  • 灾难性遗忘：学了领域数据，丢了通用能力。解法：混入通用数据。
  • 过拟合：数据少而 epochs 多，模型死记硬背。解法：早停、验证集。

★ LoRA 秩选择指南

LoRA 的秩 r 是最影响效果和成本的旋钮，但新手常在 r=8 和 r=64 之间反复横跳。把选秩的经验规则代码化，输入任务类型和数据量就能给出建议：

from dataclasses import dataclass

@dataclass
class LoRARankAdvice:
    r: int
    alpha: int
    rationale: str
    trainable_pct: float   # 可训参数占比

def recommend_lora_rank(task_type: str, dataset_size: int,
                        base_params_b: float, gpu_gb: int) -> LoRARankAdvice:
    """根据任务、数据量、模型大小、显存, 推荐 LoRA 秩。"""
    # 规则一: 任务需要的"新知识"越多 → 秩越大
    base_rank = {
        "style_transfer":   4,    # 风格迁移, 改动小
        "domain_qa":        8,    # 领域问答, 引入领域知识
        "instruction":      16,   # 指令跟随, 中等新能力
        "reasoning":        32,   # 推理任务, 需要较大调整
        "new_language":     64,   # 新语言, 表达空间要大
    }.get(task_type, 16)

    # 规则二: 数据少 → 秩调小防过拟合
    if dataset_size < 500:
        base_rank = max(4, base_rank // 2)
    elif dataset_size > 50000:
        base_rank = min(64, base_rank * 2)

    # 规则三: 显存吃紧 → 降秩或只挂 q/v
    if gpu_gb < 16:
        base_rank = max(4, base_rank // 2)

    r = base_rank
    alpha = 2 * r   # 经验值 alpha = 2r, 让 LoRA 分支初始影响适中
    # 粗估可训参数占比: 每个目标模块约 2*r*hidden 参数
    trainable_pct = round(2 * r * 4096 * 7 / (base_params_b * 1e9) * 100, 3)
    rationale = (
        f"{task_type} 基准秩 {base_rank}, 数据量 {dataset_size} "
        f"{'偏小→降秩防过拟合' if dataset_size<500 else '充足→可上调'}, "
        f"显存 {gpu_gb}G "
        f"{'紧张→已降秩' if gpu_gb<16 else '充足'}。"
    )
    return LoRARankAdvice(r, alpha, rationale, trainable_pct)

# 几个典型场景
for task, ds in [("style_transfer", 2000), ("domain_qa", 8000),
                 ("reasoning", 30000), ("new_language", 5000)]:
    adv = recommend_lora_rank(task, ds, base_params_b=7, gpu_gb=24)
    print(f"{task:15s} r={adv.r:<3d} alpha={adv.alpha:<3d} "
          f"可训={adv.trainable_pct}% | {adv.rationale}")
# 经验总结: 80% 场景 r=8~16 够用, 别一上来 r=64
#           alpha=2r 是安全默认; 可训参数通常 <1% 就是 LoRA 的精髓
#           若 r=32 还欠拟合, 考虑该用全参数微调而非继续堆秩

深 LoRA 微调避坑：秩选择与显存优化

LoRA 是参数高效微调的事实标准，但“能跑起来”和“调出好效果”之间隔着几个关键决策：秩(r)选多大、加在哪些模块、学习率怎么配。选错会导致要么过拟合、要么等于没调。下面给出可复现的秩选择实验。

秩 r 的扫描实验

秩 r 决定了 LoRA 引入的额外参数量：r 越大表达能力越强但越容易过拟合、显存也越高。经验上 r=8 是起点，但不同任务差异很大，必须扫描。下面用 PEFT 做一次完整的秩对比。

from peft import LoraConfig, get_peft_model, TaskType
from transformers import AutoModelForCausalLM
import torch

base = AutoModelForCausalLM.from_pretrained("THUDM/glm-4-9b",
        torch_dtype=torch.bfloat16, device_map="auto")

def build_lora(r, target=("q_proj","k_proj","v_proj","o_proj")):
    cfg = LoraConfig(
        task_type=TaskType.CAUSAL_LM, r=r, lora_alpha=r*2,   # alpha 常取 2r
        lora_dropout=0.05, bias="none",
        target_modules=target)
    return get_peft_model(base, cfg)

# 扫描不同秩：r=4/8/16/32，记录验证集 loss 与可训参数占比
for r in (4, 8, 16, 32):
    m = build_lora(r)
    n_train = sum(p.numel() for p in m.parameters() if p.requires_grad)
    n_total = sum(p.numel() for p in m.parameters())
    print(f"r={r:2d}  可训参数 {n_train/1e6:.1f}M ({n_train/n_total:.2%})")
# 典型结果(7B)：r=8 时可训参数约 13M(0.18%)，是性价比甜点

经验结论：简单任务（如风格调整）r=4~8 足够；复杂任务（注入大量新知识）需要 r=16~32，且必须配合更大的目标模块范围——只调 attention 的 q/v 不够，把 MLP 的 gate/up/down 也加进去，知识注入效果显著提升。

秩 r可训参数(7B)显存增量适合任务过拟合风险
4~6M极小风格/格式调整低
8~13M小通用首选、指令跟随低
16~26M中领域适配、新技能中
32~52M较大大量知识注入高（需正则）
64+~100M+大极少需要，考虑全参很高

QLoRA：在消费级 GPU 上微调大模型

即使 LoRA 本身参数少，反向传播仍需保存激活值，7B 模型全精度微调仍需 ~40GB 显存。QLoRA 的解法是把基座模型 4-bit 量化保存，只在 LoRA 参数上做全精度反传，把显存压到 ~10GB，单张 4090 即可微调。关键是三点：基座用 NF4 量化 + 双重量化（压缩量化常数本身）、用 paged optimizer 防止显存峰值 OOM、用 LoRA 加在全部线性层。一个常见误区是学习率照搬全参微调——LoRA 的学习率要比全参大一个量级（1e-4 ~ 2e-4，而非 2e-5），否则收敛极慢。另一个坑是早停比正则更重要：LoRA 在 2~3 个 epoch 后往往就开始过拟合，验证集 loss 一旦回升立刻停，别迷信“多训更好”。

深 LoRA 微调 FAQ 与避坑

LoRA 微调踩坑的高发区，下面是真实项目里反复出现的问题。

Q1：微调后模型「学会」了任务，但通用能力下降了怎么办？

这是典型的灾难性遗忘——模型为学新任务牺牲了旧能力。缓解方法：第一，混入通用数据（微调数据里掺 10~20% 原始指令数据）；第二，降低学习率（比预训练低一个量级）和少 epoch（2~3 足矣）；第三，用 LoRA 而非全参微调（LoRA 天然缓解遗忘，因为只动少量参数）；第四，定期跑通用评估集监控能力退化，发现掉点就回退。这是微调最需要警惕的副作用——「任务分涨了但模型变笨了」是隐性损失。

Q2：LoRA 该加在哪些模块上？

默认加 attention 的 q_proj/v_proj 是最经典的，但不够。要注入新知识或新技能，应扩展到全部线性层（q/k/v/o + MLP 的 gate/up/down），效果显著提升。简单任务（风格调整）只加 q/v 够；复杂任务（领域知识）必须全加。秩 r 也要匹配：简单 r=8，复杂 r=16~32。还有个易忽略的点：lora_alpha 常取 2r（缩放系数），配 dropout=0.05 防过拟合。这些组合起来才是完整的 LoRA 配方，单看 r 不够。

09 本章小结

📌 核心要点
  • 微调适合改行为/风格/降成本，RAG 适合加知识；
  • LoRA：冻结基座，训练低秩矩阵 A·B，参数降 99%+；
  • QLoRA：4-bit 基座 + LoRA，单卡微调大模型；
  • LoRA 是 2026 主流，可切换、可合并、无推理开销；
  • 数据质量 > 数量，500-5000 条高质量数据即可；
  • 防灾难性遗忘：混入通用数据 + 建评估集。
LoRA/QLoRA 让大模型定制从「大厂专属」变成「人人可做」。一台游戏显卡 + 几千条数据，你就能拥有一个专属 AI。这和第 12 章的训练迷你模型、第 16 章的端侧部署，共同构成了「个人 AI」的技术栈。
