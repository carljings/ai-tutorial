Continue codebase 索引
@codebase 检索与 tf-idf + 向量混合检索

当你在 Continue 的 chat 里输入 @codebase 这个 bug 在哪，它会检索整个代码库，找出最相关的几段代码塞进 context。这个能力听起来简单——不就是 embedding + 向量搜索吗？但真实的工程实现要复杂得多：纯向量检索对代码这种「关键词密集」的文本效果有限，Continue 用的是 tf-idf 关键词检索 + 向量语义检索的 混合 方案，还要处理增量更新和大型仓库的性能。本章我们拆解这套检索系统。

★ 一、为什么纯向量检索对代码不够好

向量检索（embedding + cosine similarity）是 RAG 的主流方案：把文档切成块，每块算 embedding 存进向量数据库，查询时算 query 的 embedding，找最相似的块。这对自然语言文档效果很好，但对代码有个先天弱点。

代码里大量信息是 精确的标识符——类名、函数名、变量名。如果用户问「AuthService 在哪里被调用」，最相关的是包含字面量 "AuthService" 的代码。但向量检索把 "AuthService" 变成一个向量后，可能和 "LoginService"、"UserService" 这些语义相近但字面不同的代码相似度更高——这恰恰是错的，用户要的是字面匹配。

纯关键词检索（tf-idf/BM25）则相反：它精确匹配字面量，"AuthService" 只匹配 "AuthService"。但它不懂语义——用户问「登录相关代码」，关键词检索找不到不含「登录」二字但实际是 login 函数的代码。

Continue 的洞察是：两种方法各有盲区，组合起来最互补。向量检索擅长语义相似（「登录」匹配 login 函数），关键词检索擅长精确匹配（"AuthService" 匹配字面）。混合检索同时跑两种，合并结果，兼顾语义和精度。

★ 二、分块策略：代码不能按字数切

检索的第一步是把代码库切成「块」（chunk），每块作为一个检索单元。自然语言 RAG 通常按固定字数（如 500 字）切，但代码不能这么切——把一个函数从中间切断，两半都失去意义。

Continue 用的是 语法感知的分块：用 tree-sitter（和 Aider 的 repo map 用的一样，参见 163 章）解析代码，按函数/类的边界切。每个函数是一个块，每个类是一个块（或类的块包含其方法）。

// core/indexing/chunking.ts（简化）
import { getParser } from "tree-sitter-languages";

export interface CodeChunk {
  filepath: string;
  startLine: number;
  endLine: number;
  content: string;
  // 元信息：函数名、类名等
  name?: string;
  type?: "function" | "class" | "method";
}

export function chunkFile(filepath: string, content: string): CodeChunk[] {
  const lang = inferLanguage(filepath);
  const parser = getParser(lang);
  const tree = parser.parse(content);
  const lines = content.split("\n");

  const chunks: CodeChunk[] = [];

  function walk(node: any) {
    if (isDefinitionNode(node)) {  // function_definition, class_definition 等
      const startLine = node.startPosition.row;
      const endLine = node.endPosition.row;
      // 太小的定义（<3 行）合并到父块
      if (endLine - startLine >= 3) {
        const nameNode = node.childByFieldName?.("name");
        chunks.push({
          filepath,
          startLine,
          endLine,
          content: lines.slice(startLine, endLine + 1).join("\n"),
          name: nameNode?.text,
          type: node.type.replace("_definition", "").replace("_declaration", ""),
        });
        return;  // 不再递归进这个定义（它的内部已经被这个块覆盖）
      }
    }
    for (const child of node.children || []) walk(child);
  }

  walk(tree.rootNode);

  // 如果整个文件没有定义（纯脚本），按 50 行粗切
  if (chunks.length === 0) {
    for (let i = 0; i < lines.length; i += 50) {
      chunks.push({
        filepath,
        startLine: i,
        endLine: Math.min(i + 49, lines.length - 1),
        content: lines.slice(i, i + 50).join("\n"),
      });
    }
  }
  return chunks;
}

注意两个细节。第一，小定义不单独成块：不到 3 行的 getter/setter 单独成块意义不大（embedding 信息量不足），让它们留在父块里。第二，无定义文件粗切兜底：配置文件、shell 脚本没有函数定义，按 50 行粗切，至少保证能被检索到。这种「语法优先 + 兜底粗切」的策略，保证了大多数代码文件都有合理的分块。

★ 三、向量索引：embedding 与存储

分块后，每个块算一个 embedding 向量。Continue 支持多种 embedding provider——云端（OpenAI text-embedding-3）或本地（transformers.js 跑 all-MiniLM-L6-v2）。本地 embedding 的好处是不花钱、不外传代码，Continue 默认走本地。

// core/indexing/vectorIndex.ts（简化）
export class VectorIndex {
  private items: IndexItem[] = [];
  private vectors: Float32Array[] = [];
  private embeddingsProvider: EmbeddingsProvider;

  async addChunks(chunks: CodeChunk[]) {
    // 批量算 embedding（比逐个快）
    const texts = chunks.map(c => this.formatForEmbedding(c));
    const embeddings = await this.embeddingsProvider.embed(texts);

    for (let i = 0; i < chunks.length; i++) {
      this.items.push({
        filepath: chunks[i].filepath,
        startLine: chunks[i].startLine,
        endLine: chunks[i].endLine,
        content: chunks[i].content,
        name: chunks[i].name,
      });
      this.vectors.push(embeddings[i]);
    }
  }

  // embedding 时把元信息也带上，提高检索质量
  private formatForEmbedding(chunk: CodeChunk): string {
    // 例：函数块格式化为 "function AuthService.login in auth.py:\n<代码>"
    const tag = chunk.type ? `${chunk.type} ${chunk.name || ""}` : "";
    const path = chunk.filepath;
    return `${tag} in ${path}:\n${chunk.content}`;
  }

  async search(query: string, topK: number): Promise<SearchResult[]> {
    const queryVec = await this.embeddingsProvider.embed([query]);
    const scores = this.vectors.map(v => cosineSimilarity(queryVec[0], v));
    // 取 topK
    const indices = argsort(scores).slice(-topK).reverse();
    return indices.map(i => ({
      item: this.items[i],
      score: scores[i],
    }));
  }
}

function cosineSimilarity(a: Float32Array, b: Float32Array): number {
  let dot = 0, normA = 0, normB = 0;
  for (let i = 0; i < a.length; i++) {
    dot += a[i] * b[i];
    normA += a[i] * a[i];
    normB += b[i] * b[i];
  }
  return dot / (Math.sqrt(normA) * Math.sqrt(normB));
}

注意 formatForEmbedding 这一步——它把块的元信息（函数名、类型、路径）拼在代码前面再算 embedding。这是个小但重要的优化：embedding 模型看到「function AuthService.login」前缀，能让这个块的向量更聚焦于「AuthService 的 login 函数」语义，而不是泛泛的代码。

★ 四、tf-idf 索引：关键词精确匹配

除了向量索引，Continue 还维护一份 tf-idf 索引，用于关键词检索。tf-idf（Term Frequency - Inverse Document Frequency）衡量一个词对某个文档的重要性——在当前块出现频率高（tf 高）、在所有块里少见（idf 高）的词，权重高。

// core/indexing/tfidfIndex.ts（简化）
export class TFIDFIndex {
  private docs: string[] = [];              // 每个块的文本
  private termFreqs: Map<string, number>[] = []; // 每个块的词频
  private docFreq: Map<string, number> = new Map(); // 每个词出现在多少块
  private N = 0;                            // 总块数

  addChunk(text: string) {
    const docId = this.docs.length;
    this.docs.push(text);
    // 分词（代码分词：按 camelCase 和 snake_case 拆，加下划线边界）
    const tokens = tokenizeCode(text);
    const freq = new Map<string, number>();
    for (const tok of tokens) {
      freq.set(tok, (freq.get(tok) || 0) + 1);
    }
    this.termFreqs.push(freq);
    // 更新 document frequency
    for (const term of freq.keys()) {
      this.docFreq.set(term, (this.docFreq.get(term) || 0) + 1);
    }
    this.N++;
  }

  search(query: string, topK: number): Array<{docId: number, score: number}> {
    const queryTokens = tokenizeCode(query);
    const scores: number[] = new Array(this.N).fill(0);

    for (const term of queryTokens) {
      const df = this.docFreq.get(term) || 0;
      if (df === 0) continue;
      // idf = log(N / df)，词越稀有 idf 越高
      const idf = Math.log((this.N + 1) / (df + 1));
      // 对每个包含该词的文档，累加 tf * idf
      for (let i = 0; i < this.N; i++) {
        const tf = this.termFreqs[i].get(term) || 0;
        if (tf > 0) {
          // 归一化 tf（除以文档长度）
          const normalizedTf = tf / this.docs[i].length;
          scores[i] += normalizedTf * idf;
        }
      }
    }
    return argsort(scores).slice(-topK).reverse()
      .filter(i => scores[i] > 0)
      .map(i => ({ docId: i, score: scores[i] }));
  }
}

// 代码专用分词：把 camelCase 和 snake_case 都拆开
function tokenizeCode(text: string): string[] {
  return text
    .replace(/([a-z])([A-Z])/g, "$1 $2")   // camelCase -> camel Case
    .replace(/[_\-\.\/]/g, " ")              // snake_case / path -> 拆开
    .toLowerCase()
    .split(/\s+/)
    .filter(t => t.length > 1);             // 丢单字符
}

这里最值得注意是 tokenizeCode——它把 camelCase 和 snake_case 都拆开。这解决了代码检索的一个大问题：「AuthService」的查询要能匹配到「auth_service」（同一个概念不同命名风格）。拆开后两者都包含 "auth" 和 "service" 两个 token，tf-idf 就能匹配上。这种「分词器要适配代码命名习惯」的细节，是代码检索比自然语言检索更 tricky 的地方。

★ 五、混合检索的合并策略

向量检索和 tf-idf 检索各返回一份 topK 结果，怎么合并？Continue 用的是 分数归一化 + 加权融合：

// core/indexing/hybridSearch.ts（简化）
export async function hybridSearch(
  query: string,
  vectorIndex: VectorIndex,
  tfidfIndex: TFIDFIndex,
  topK: number,
  options: { vectorWeight?: number; keywordWeight?: number } = {}
): Promise<SearchResult[]> {
  const vw = options.vectorWeight ?? 0.5;
  const kw = options.keywordWeight ?? 0.5;

  // 并行跑两种检索
  const [vectorResults, keywordResults] = await Promise.all([
    vectorIndex.search(query, topK * 2),
    Promise.resolve(tfidfIndex.search(query, topK * 2)),
  ]);

  // 分数归一化到 [0,1]
  const normalize = (results: Array<{score: number}>) => {
    if (results.length === 0) return results;
    const max = Math.max(...results.map(r => r.score));
    const min = Math.min(...results.map(r => r.score));
    const range = max - min || 1;
    return results.map(r => ({ ...r, score: (r.score - min) / range }));
  };

  const normVec = normalize(vectorResults);
  const normKw = normalize(keywordResults);

  // 按 docId/item 合并分数
  const merged = new Map<string, SearchResult>();
  for (const r of normVec) {
    const key = `${r.item.filepath}:${r.item.startLine}`;
    merged.set(key, { ...r, score: r.score * vw });
  }
  for (const r of normKw) {
    const key = `${r.item.filepath}:${r.item.startLine}`;
    const existing = merged.get(key);
    if (existing) {
      existing.score += r.score * kw;  // 两种都命中，加权累加
    } else {
      merged.set(key, { ...r, score: r.score * kw });
    }
  }

  // 排序取 topK
  return Array.from(merged.values())
    .sort((a, b) => b.score - a.score)
    .slice(0, topK);
}

这里有个关键设计：一个块如果同时被两种检索命中，分数是加权累加的。这意味着「既语义相关又关键词匹配」的块会排得最高。这正是我们想要的——比如查询「AuthService.login」，包含 "AuthService" 字面量且语义上和 login 相关的块，会得到双倍加分，稳居 top。而只命中一种的块排后面。这种「两种证据互证」的策略，比单检索的精度高得多。

★ 六、增量更新：文件改了不用全量重建

代码库在持续变化——用户每改一个文件，索引要不要重建？全量重建一个中型仓库可能要几分钟，每次改动都重建不可接受。Continue 的做法是 增量更新：只重新索引改动的文件。

// core/indexing/incremental.ts（简化）
export class IncrementalIndex {
  private fileHashes: Map<string, string> = new Map();  // path -> 内容 hash

  async updateForChangedFiles(changedFiles: string[]) {
    for (const filepath of changedFiles) {
      const content = await readFile(filepath);
      const hash = md5(content);

      // 没变就跳过
      if (this.fileHashes.get(filepath) === hash) continue;
      this.fileHashes.set(filepath, hash);

      // 删除这个文件的所有旧块
      this.vectorIndex.removeByFile(filepath);
      this.tfidfIndex.removeByFile(filepath);

      // 重新分块并加入
      const chunks = chunkFile(filepath, content);
      await this.vectorIndex.addChunks(chunks);
      for (const c of chunks) {
        this.tfidfIndex.addChunk(c.content);
      }
    }
  }
}

用 hash 判断文件是否真的变了——保存时的 touch 不触发重建。删旧加新保证索引里不会有过期块。这种增量策略让索引始终保持最新，且每次只花毫秒级（单文件）而非分钟级（全量）。

★ 本章小结

@codebase 是 Continue 让 Agent 真正理解项目的核心能力。要点回顾：

  • 混合检索的必要性：向量擅长语义、关键词擅长精确匹配，代码检索两者都需要，组合互补。
  • 语法感知分块：用 tree-sitter 按函数/类边界切，小定义合并、无定义文件粗切兜底。
  • 向量索引：embedding 时把元信息（函数名/类型/路径）拼在前面，本地 transformer.js 默认。
  • tf-idf 索引：代码专用分词器拆 camelCase 和 snake_case，解决命名风格不一致的匹配问题。
  • 合并策略：分数归一化 + 加权融合，同时被两种检索命中的块分数累加，精度最高。
  • 增量更新：hash 判断文件变更，删旧加新，单文件毫秒级而非全量分钟级。
