第 01 讲

AI 到底是啥？从死规则到会生成的机器

🎯 本讲核心要点
  • 学习AI最怕只学名词，今天知道一个新词，明天又被另一个新词带跑
  • 在本册主题中，人工智能、机器学习、深度学习、大语言模型会不断出现

学习AI最怕只学名词，今天知道一个新词，明天又被另一个新词带跑。更稳的方法是：每学一个概念，都问它解决什么问题、依赖什么数据、输出什么结果、如何评估、失败时怎么办。

在本册主题中，人工智能、机器学习、深度学习、大语言模型会不断出现。它们不是彼此孤立的知识点，而是构成AI系统的几个层次：底层是数据和计算，中间是模型和算法，上层是应用、流程、评估和治理。你可以先不记住所有细节，但要知道每个概念在系统里站在哪个位置。

一、先建立直觉

理解“AI到底是什么：从规则系统到生成式AI”时，可以先用一个简单问题开场：如果把AI看成一个助手，它到底需要什么才能完成任务？它需要输入，需要背景资料，需要可调用的工具，需要判断结果好坏的标准，也需要在高风险动作前停下来让人确认。这个直觉非常重要，因为它能帮你避免把所有问题都归结为“模型不够强”。

以“个人知识管理”为例，表面上用户只是提出一个问题，实际系统背后要处理的是资料来源、用户权限、输出格式、失败兜底和人工接管。一个能上线的AI应用，通常不是单个提示词就能解决的，而是把模型、知识、工具、工作流和评估机制组合在一起。

初学阶段不必急着追求复杂架构。真正值得训练的能力，是把问题拆小：哪些部分可以用普通程序解决，哪些部分适合模型理解语言，哪些部分需要检索资料，哪些部分需要调用外部系统。这个拆解能力，比盲目堆技术名词更值钱。

二、核心概念拆解

第一层是输入。输入可能是一句话、一张图片、一份表格、一段录音，也可能是一批历史订单。输入越模糊，模型越容易猜；输入越结构化，系统越容易稳定运行。所以在设计任何AI任务时，都要先写清楚输入字段、允许的格式、缺失信息的处理方式。

第二层是知识。知识可能来自模型训练阶段，也可能来自你后来接入的文档、数据库和业务规则。对于会变化的政策、合同、课程、产品说明，不应该期待模型凭记忆回答，而应该让系统先检索最新资料。这也是RAG知识库思想的核心：回答之前先找依据。

第三层是行动。只会回答问题的模型，价值主要体现在内容生成和辅助判断；能调用工具的Agent，才可能查询订单、创建日程、生成文件、提交工单或更新状态。但行动能力越强，风险越高，所以必须把工具权限、确认机制和审计日志一起设计进去。

第四层是评估。AI系统的输出经常带有不确定性，不能只靠“看起来像对的”来判断。你需要准备样例、标准答案、失败案例和业务指标，把系统表现变成可以讨论、可以复盘、可以迭代的数据。

三、操作流程

  1. 把需求写成一句清楚的问题陈述。
  2. 列出用户、输入、输出、约束、风险和成功指标。
  3. 准备十到三十条真实样例，作为评估集的雏形。
  4. 先做最小可用原型，不要一开始就追求大而全。
  5. 记录每次错误，把错误按检索错误、生成错误、工具错误、权限错误分类。
  6. 把有效改动写成版本记录，避免“今天调好、明天忘记为什么好”。

四、案例：把概念放进业务里

假设我们要为“个人知识管理”做一个AI助手。第一步不是选择模型，而是采访真实用户：他们每天最常问什么，哪些问题最浪费时间，哪些错误最不能接受，哪些资料是权威来源。只有把这些问题问清楚，后面的技术选择才有依据。

如果用户问题主要是规则查询，例如“能不能报销”“什么时候选课”“如何申请售后”，系统优先考虑RAG。资料需要按版本管理，回答需要引用来源。如果用户任务包含多步操作，例如“查资料、生成方案、创建提醒、同步系统”，就要考虑Agent或固定工作流。

再看“人力资源培训”。它可能不需要复杂Agent，只需要一个稳定的表单流程和一个总结模型。比如会议纪要自动化，流程可以固定为：上传录音或文字、抽取议题、整理待办、确认负责人、发送摘要。这个场景重点是格式和审核，不一定需要模型自由规划。

五、常见误区

✗ 没有定义输入、输出和成功标准，就急着选工具
✗ 只追求“看起来高级”的架构，忽略最小可用方案
✗ 把训练集、验证集和测试集混在一起，导致评估失真
✗ 只关注平均准确率，不看高风险错误和边界样本
✗ 把私有数据随意放进系统，没有权限、脱敏和审计设计

六、练习

  • 练习1：用三句话解释“AI到底是什么：从规则系统到生成式AI”：第一句说明它解决什么问题，第二句说明它需要什么输入，第三句说明如何判断结果好坏。
  • 练习2：选择一个你熟悉的场景，把它拆成“用户问题、可用资料、可用工具、输出格式、失败兜底、人工确认”六栏。这个练习看起来简单，但它是AI项目设计的基本功。
  • 练习3：围绕“设计一个个人AI学习助手：能记录目标、检索课程笔记、生成练习题并安排复习。”写一份小型方案，至少包括目标用户、核心功能、数据来源、评估方法和风险清单。不要写空泛口号，要写到别人可以照着做原型的程度。

七、本讲小结

本讲最重要的结论是：AI到底是什么：从规则系统到生成式AI不是一个孤立知识点，而是AI系统设计的一块拼图。学习时不要只问“它是什么”，还要追问“什么时候用、怎么用、如何验证、失败怎么办”。当你能用这四个问题审视每个技术概念时，你已经从工具使用者开始向AI方案设计者迈进。

第二层是知识。知识可能来自模型训练阶段，也可能来自你后来接入的文档、数据库和业务规则。对于会变化的政策、合同、课程、产品说明，不应该期待模型凭记忆回答，而应该让系统先检索最新资料。这也是RAG知识库思想的核心：回答之前先找依据。

第三层是行动。只会回答问题的模型，价值主要体现在内容生成和辅助判断；能调用工具的Agent，才可能查询订单、创建日程、生成文件、提交工单或更新状态。但行动能力越强，风险越高，所以必须把工具权限、确认机制和审计日志一起设计进去。

第四层是评估。AI系统的输出经常带有不确定性，不能只靠“看起来像对的”来判断。你需要准备样例、标准答案、失败案例和业务指标，把系统表现变成可以讨论、可以复盘、可以迭代的数据。

三、操作流程

  1. 把需求写成一句清楚的问题陈述。
  2. 列出用户、输入、输出、约束、风险和成功指标。
  3. 准备十到三十条真实样例，作为评估集的雏形。
  4. 先做最小可用原型，不要一开始就追求大而全。
  5. 记录每次错误，把错误按检索错误、生成错误、工具错误、权限错误分类。
  6. 把有效改动写成版本记录，避免“今天调好、明天忘记为什么好”。

四、案例：把概念放进业务里

五、常见误区

✗ Prompt 写得很长，却没有给模型明确的角色、目标、格式和约束
✗ RAG 只做上传文档，不做清洗、切分、检索评估和引用设计
✗ Agent 接了太多工具，没有动作权限分级，调试时很容易变成黑箱
✗ 项目上线后不记录日志、不收集失败案例，导致系统无法持续改进
✗ 把模型输出当作最终事实，不做来源核验

六、练习

七、本讲小结

理解“AI Agent前言：让AI调用工具完成任务”时，可以先用一个简单问题开场：如果把AI看成一个助手，它到底需要什么才能完成任务？它需要输入，需要背景资料，需要可调用的工具，需要判断结果好坏的标准，也需要在高风险动作前停下来让人确认。这个直觉非常重要，因为它能帮你避免把所有问题都归结为“模型不够强”。

以“法律合同初筛”为例，表面上用户只是提出一个问题，实际系统背后要处理的是资料来源、用户权限、输出格式、失败兜底和人工接管。一个能上线的AI应用，通常不是单个提示词就能解决的，而是把模型、知识、工具、工作流和评估机制组合在一起。

初学阶段不必急着追求复杂架构。真正值得训练的能力，是把问题拆小：哪些部分可以用普通程序解决，哪些部分适合模型理解语言，哪些部分需要检索资料，哪些部分需要调用外部系统。这个拆解能力，比盲目堆技术名词更值钱。

二、核心概念拆解

第一层是输入。输入可能是一句话、一张图片、一份表格、一段录音，也可能是一批历史订单。输入越模糊，模型越容易猜；输入越结构化，系统越容易稳定运行。所以在设计任何AI任务时，都要先写清楚输入字段、允许的格式、缺失信息的处理方式。

第二层是知识。知识可能来自模型训练阶段，也可能来自你后来接入的文档、数据库和业务规则。对于会变化的政策、合同、课程、产品说明，不应该期待模型凭记忆回答，而应该让系统先检索最新资料。这也是RAG知识库思想的核心：回答之前先找依据。

第三层是行动。只会回答问题的模型，价值主要体现在内容生成和辅助判断；能调用工具的Agent，才可能查询订单、创建日程、生成文件、提交工单或更新状态。但行动能力越强，风险越高，所以必须把工具权限、确认机制和审计日志一起设计进去。

第四层是评估。AI系统的输出经常带有不确定性，不能只靠“看起来像对的”来判断。你需要准备样例、标准答案、失败案例和业务指标，把系统表现变成可以讨论、可以复盘、可以迭代的数据。

三、操作流程

  1. 把需求写成一句清楚的问题陈述。
  2. 列出用户、输入、输出、约束、风险和成功指标。
  3. 准备十到三十条真实样例，作为评估集的雏形。
  4. 先做最小可用原型，不要一开始就追求大而全。
  5. 记录每次错误，把错误按检索错误、生成错误、工具错误、权限错误分类。
  6. 把有效改动写成版本记录，避免“今天调好、明天忘记为什么好”。

四、案例：把概念放进业务里

假设我们要为“法律合同初筛”做一个AI助手。第一步不是选择模型，而是采访真实用户：他们每天最常问什么，哪些问题最浪费时间，哪些错误最不能接受，哪些资料是权威来源。只有把这些问题问清楚，后面的技术选择才有依据。

如果用户问题主要是规则查询，例如“能不能报销”“什么时候选课”“如何申请售后”，系统优先考虑RAG。资料需要按版本管理，回答需要引用来源。如果用户任务包含多步操作，例如“查资料、生成方案、创建提醒、同步系统”，就要考虑Agent或固定工作流。

再看“学习计划制定”。它可能不需要复杂Agent，只需要一个稳定的表单流程和一个总结模型。比如会议纪要自动化，流程可以固定为：上传录音或文字、抽取议题、整理待办、确认负责人、发送摘要。这个场景重点是格式和审核，不一定需要模型自由规划。

五、常见误区

✗ 项目上线后不记录日志、不收集失败案例，导致系统无法持续改进
✗ 把模型输出当作最终事实，不做来源核验
✗ 没有定义输入、输出和成功标准，就急着选工具
✗ 只追求“看起来高级”的架构，忽略最小可用方案
✗ 把训练集、验证集和测试集混在一起，导致评估失真

六、练习

  • 练习1：用三句话解释“AI Agent前言：让AI调用工具完成任务”：第一句说明它解决什么问题，第二句说明它需要什么输入，第三句说明如何判断结果好坏。
  • 练习2：选择一个你熟悉的场景，把它拆成“用户问题、可用资料、可用工具、输出格式、失败兜底、人工确认”六栏。这个练习看起来简单，但它是AI项目设计的基本功。
  • 练习3：围绕“设计一个个人AI学习助手：能记录目标、检索课程笔记、生成练习题并安排复习。”写一份小型方案，至少包括目标用户、核心功能、数据来源、评估方法和风险清单。不要写空泛口号，要写到别人可以照着做原型的程度。

七、本讲小结

本讲最重要的结论是：AI Agent前言：让AI调用工具完成任务不是一个孤立知识点，而是AI系统设计的一块拼图。学习时不要只问“它是什么”，还要追问“什么时候用、怎么用、如何验证、失败怎么办”。当你能用这四个问题审视每个技术概念时，你已经从工具使用者开始向AI方案设计者迈进。
