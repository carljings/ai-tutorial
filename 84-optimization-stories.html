★ 二、案例 1：推理速度优化（7.5 倍提升）

背景：一个电商客服聊天机器人，用 HuggingFace transformers 的原生 generate 方法跑 Llama-3-8B。用户反馈「回复太慢，等得花儿都谢了」。

2.1 优化前的瓶颈分析

# 优化前的配置和性能
from transformers import AutoModelForCausalLM, AutoTokenizer
import time

model = AutoModelForCausalLM.from_pretrained("llama-3-8b", torch_dtype="fp16")
tokenizer = AutoTokenizer.from_pretrained("llama-3-8b")

def generate_naive(prompt):
    inputs = tokenizer(prompt, return_tensors="pt")
    start = time.time()
    # 原生 generate：逐 token 生成，没有 batch 优化
    outputs = model.generate(**inputs, max_new_tokens=200)
    elapsed = time.time() - start
    tokens = outputs.shape[1] - inputs.input_ids.shape[1]
    speed = tokens / elapsed
    print(f"速度: {speed:.1f} tokens/s")
    return tokenizer.decode(outputs[0])

# 基准测试结果：
# 速度: 20 tokens/s
# 首 token 延迟: 2.5 秒
# 单请求显存: 18 GB

# 问题诊断：
# 1. 原生 generate 没有 KV Cache 优化
# 2. 没有 batch，每次只处理一个请求
# 3. FP16 没有量化
# 4. 没有连续批处理

2.2 优化步骤

# 优化步骤 1：切换到 vLLM（最大的 free lunch）
from vllm import LLM, SamplingParams

llm = LLM(
    model="llama-3-8b",
    # 核心优化 1：PagedAttention，高效管理 KV Cache
    # 核心优化 2：Continuous Batching，支持并发请求
    tensor_parallel_size=1,
    gpu_memory_utilization=0.9,
    max_model_len=4096,
)

def generate_vllm(prompts):
    sampling = SamplingParams(temperature=0.3, max_tokens=200)
    start = time.time()
    outputs = llm.generate(prompts, sampling)
    elapsed = time.time() - start
    total_tokens = sum(len(o.outputs[0].token_ids) for o in outputs)
    print(f"速度: {total_tokens/elapsed:.1f} tokens/s")

# 单请求测试：55 tokens/s（提升 2.75 倍）
# 批量请求测试（batch=8）：120 tokens/s（吞吐再翻倍）

# 优化步骤 2：INT8 量化
llm = LLM(
    model="llama-3-8b",
    quantization="awq",           # AWQ 量化
    dtype="float16",
    gpu_memory_utilization=0.9,
)
# 量化后：显存从 18GB 降到 10GB，速度提升到 75 tokens/s（单请求）

# 优化步骤 3：Prompt 缓存（客服场景的 system prompt 固定）
# vLLM 自动启用 prefix caching
llm = LLM(model="llama-3-8b", enable_prefix_caching=True)
# 重复的 system prompt 不重新计算，TTFT 从 2.5s 降到 0.8s

# 优化步骤 4：Speculative Decoding（用小模型加速）
from vllm import LLM
llm = LLM(
    model="llama-3-8b",
    speculative_model="llama-3-1b",  # 小模型做草稿
    num_speculative_tokens=4,
)
# 小模型预测 + 大模型验证，速度再提升 40%

2.3 最终效果

# 优化前后的完整对比
speed_comparison = {
    "指标": ["单请求速度", "并发吞吐(8)", "首token延迟", "显存占用"],
    "优化前": ["20 tok/s", "20 tok/s", "2.5s", "18 GB"],

# 优化 2：分块处理，不要把全文塞进一个 prompt
def summarize_long_document(text):
    chunks = split_text(text, chunk_size=2000)  # 按章节切
    # 先分别摘要每个块
    chunk_summaries = [llm.generate(f"总结：{chunk}") for chunk in chunks]
    # 再把块的摘要合并做总摘要
    final_summary = llm.generate(f"综合总结：{' '.join(chunk_summaries)}")
    return final_summary
    # Map-Reduce 思想：把大问题分解成小问题

# 优化 3：用 RAG 替代全文塞入 prompt

    # prompt 从 70 万 token 缩短到 2000 token

★ 四、案例 3：成本优化（降 85%）

背景：一个 AI 写作助手，月活 5000 用户，全部用 GPT-4，每月 API 费用 $12000，入不敷出。

4.1 成本分析

# 优化前的成本分析
cost_analysis_before = {
    "月活用户": 5000,

    "平均输入": 1500 tokens,

# 输出 tokens = 250000 × 500 = 125M

# 输出成本 = 125 × $60 = $7,500
# 总成本 = $18,750（实际有部分免费额度等，约 $12,000）
# 单用户成本 = $2.4/月，而订阅费只有 $10/月，毛利太低

4.2 优化策略

# 优化 1：模型路由（最大头）
class ModelRouter:
    def route(self, request):
        # 简单任务（续写、改写）用便宜的 GPT-4o-mini
        if request.type in ["continue", "rewrite", "grammar"]:
            return "gpt-4o-mini"  # $0.15/$0.60 per 1M
        # 复杂任务（创意写作、深度分析）才用 GPT-4
        return "gpt-4o"           # $2.5/$10 per 1M

# 效果：70% 的请求路由到 mini，输入成本降 90%

# 优化 2：Prompt 缓存（重复内容不重复计费）
# 很多请求有相同的 system prompt（约 500 token）
# 用 OpenAI 的 prompt caching 功能，缓存部分自动降价
# 效果：约 30% 的输入 token 被缓存，成本减半

# 优化 3：响应缓存（相同问题不重复调用）
from hashlib import md5
response_cache = {}

def cached_generate(prompt):
    key = md5(prompt.encode()).hexdigest()
    if key in response_cache:
        return response_cache[key]  # 缓存命中，零成本
    response = llm.generate(prompt)
    response_cache[key] = response
    return response
# 效果：约 15% 的请求命中缓存，完全免费

# 优化 4：压缩 prompt（去掉冗余）
# 原始 system prompt（500 token）有很多示例和说明
# 精简后只有 200 token，省 60% 输入成本

4.3 最终效果

★ 六、四个案例的共同经验

  • 先 benchmark 再优化：不测量就不知道瓶颈在哪。每个优化都量化效果。
  • 从最大杠杆开始：速度优化先上 vLLM，成本优化先做模型路由。80% 的收益来自 20% 的手段。
  • 组合拳才有效：单一优化效果有限，多种手段叠加才有质变。
  • 优化有天花板：prompt 优化的精度天花板约 90%，突破要靠微调。
  • 权衡是常态：速度 vs 质量、成本 vs 效果、通用 vs 专业，永远在做平衡。

★ 本章小结

这一章通过四个真实优化案例展示了 LLM 工程的优化方法论。核心要点：

  • 速度优化：vLLM 是最大 free lunch（2.75 倍），叠加量化、缓存、Speculative Decoding 可达 7.5 倍。
  • 内存优化：流式处理 + 分块 + RAG + 量化，内存降 75%。
  • 成本优化：模型路由是最大杠杆（降 65%），叠加缓存可达 85%。
  • 精度优化：Few-shot + CoT 提升到 90%，微调突破到 94%。
  • 共同经验：先 benchmark、抓最大杠杆、打组合拳、认知天花板、接受权衡。
