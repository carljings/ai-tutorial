高级编程实战
用 AI 做代码审查、Bug 定位、重构与测试生成

AI 辅助编程早就不是「自动补全」那么简单了。今天的 LLM 能做代码审查、定位隐蔽 Bug、重构遗留代码、生成测试用例——这些都是高级工程师的核心工作。但工具再强，关键在于你会不会用。这一章把四个场景的实战 prompt 模板和完整工作流给你，让你把 AI 用成 10x 工程师的杠杆，而不是复制粘贴的拐杖。

阅读约 30 分钟 难度 ★★★

★ 一、AI 编程的正确姿势

很多人用 AI 编程还停留在「帮我写个函数」的阶段，这是对 AI 能力的巨大浪费。真正高效的工作流是让 AI 参与到编程的每一个环节：理解需求、设计方案、写代码、审查、测试、重构。但要做到这点，你必须学会写高质量的 prompt。

一个好的编程 prompt 有四个要素：明确角色（让 AI 扮演资深工程师）、充足上下文（相关代码、约束、目标）、具体任务（不要含糊）、输出格式（结构化输出便于使用）。接下来四个场景，我们逐一实战。

★ 二、代码审查：比人工更细致

代码审查（Code Review）是保证代码质量的关键环节，但人工审查容易疲劳、遗漏、流于形式。AI 审查的优势是稳定、全面、可以覆盖大量边界情况。下面是一个经过实战验证的审查 prompt 模板。

# AI 代码审查的 prompt 模板（生产级）
CODE_REVIEW_PROMPT = """你是资深 {language} 工程师，请审查以下代码。

## 审查维度（逐项检查）
1. **正确性**：逻辑是否正确？有无边界条件遗漏？空值/异常处理是否完整？
2. **安全性**：有无 SQL 注入、XSS、敏感信息泄露、权限绕过风险？
3. **性能**：有无 N+1 查询、不必要的循环、内存泄漏、O(n²) 可优化处？
4. **可读性**：命名是否清晰？复杂度是否过高？是否需要拆分？
5. **最佳实践**：是否符合 {language} 的惯用法和框架规范？

## 待审查代码
```{language}
{code}
```

## 上下文（可选）
业务背景：{context}
已知约束：{constraints}

## 输出格式
按严重程度分级：
- 🔴 严重（必须修复）：...
- 🟡 建议（应该修复）：...
- 🟢 优化（可以改进）：...

每个问题给出：位置、问题描述、修复建议（含代码示例）。"""

# 使用示例
def review_code(code, language="python", context="", constraints=""):
    prompt = CODE_REVIEW_PROMPT.format(
        language=language, code=code,
        context=context, constraints=constraints
    )
    response = call_llm(prompt, model="gpt-4o")
    return response

# 待审查的代码（故意埋了几个问题）
buggy_code = '''
def get_user_orders(user_id, db):
    conn = db.connect()
    cursor = conn.cursor()
    # 获取用户所有订单
    cursor.execute(f"SELECT * FROM orders WHERE user_id = {user_id}")
    orders = cursor.fetchall()
    result = []
    for order in orders:
        # 获取每个订单的商品
        cursor.execute(f"SELECT * FROM items WHERE order_id = {order[0]}")
        items = cursor.fetchall()
        result.append({"order": order, "items": items})
    return result
'''

review = review_code(buggy_code, language="python",
                     context="电商系统，db 是 MySQL 连接",
                     constraints="需要支持高并发")
# AI 会指出：SQL 注入风险、N+1 查询、连接未关闭、无异常处理

这段代码里埋了四个问题：SQL 注入（f-string 拼接）、N+1 查询（循环里查数据库）、连接泄漏（没 close）、无异常处理。一个好的 AI 审查能一次性全部指出，并给出修复代码。人工审查很可能只发现一两个。

★ 三、Bug 定位：像侦探一样推理

定位 Bug 是最耗时的开发活动。AI 的优势在于它能快速分析大量代码，找出人类容易忽略的因果链。关键是给 AI 提供足够的线索，让它做结构化推理。

# Bug 定位的 prompt 模板（侦探式推理）
BUG_HUNT_PROMPT = """你是调试专家。请帮我定位以下 Bug。

## Bug 现象
{symptom}

## 复现步骤
{reproduction_steps}

## 相关代码
{relevant_code}

## 已知的错误日志/堆栈
{error_log}

## 我的猜测（可选）
{hypothesis}

## 分析方法
请用「假设-验证」的方法逐步推理：
1. 列出所有可能的原因（按可能性排序）
2. 对每个原因，指出如何在代码中验证
3. 给出最可能的根因和修复方案

## 输出格式
**根因分析**：
**证据链**：
**修复方案**（含代码）：
**预防建议**（如何避免同类 Bug）："""

# 实战案例：一个诡异的内存泄漏
bug_case = {
    "symptom": "服务运行 6-8 小时后 OOM（内存溢出），重启后恢复",
    "reproduction_steps": "压测环境跑 6 小时必现，但短时间测试无法复现",
    "relevant_code": '''
class ResponseCache:
    def __init__(self):
        self.cache = {}  # 简单的内存缓存

    def get(self, key):
        return self.cache.get(key)

    def set(self, key, value):
        self.cache[key] = value  # 存入缓存

# 在请求处理中
def handle_request(request):
    cache_key = request.url + str(time.time())  # 每次时间不同
    cache.set(cache_key, process(request))
''',
    "error_log": "MemoryError after 6h, RSS grows linearly",
    "hypothesis": "我觉得是数据库连接没释放",
}

# AI 的分析会指出：
# 根因：cache_key 包含 time.time()，每次请求都不同，缓存无限增长
# 这是一个「无界缓存」的经典 Bug，不是数据库连接问题
# 修复：改用 LRU 缓存或限制大小

这个案例展示了 AI 定位 Bug 的价值：它能跳出开发者的思维定势。开发者怀疑是数据库连接，但 AI 通过分析代码发现真正的根因是缓存 key 设计导致内存无限增长。这种「跳出假设」的能力，是 AI 调试的核心价值。

★ 四、重构：让遗留代码重获新生

重构是工程师最不想做但又必须做的事。AI 重构的优势是耐心和全面——它能处理那些人类看一眼就想跑的超长函数。关键是 prompt 要明确重构的目标和约束。

# 重构的 prompt 模板（保守且安全）
REFACTOR_PROMPT = """你是资深工程师，请重构以下代码。

## 重构目标（按优先级）
{goals}

## 硬约束（不可违反）
1. **不改变外部行为**：函数签名、输入输出、副作用必须保持一致
2. **小步前进**：每次只做一个改进，便于 review
3. **保留原有注释和业务逻辑**

## 待重构代码
```
{code}
```

## 输出格式
对每个改动：
- **改动描述**：做了什么、为什么
- **改动前**：（代码片段）
- **改动后**：（代码片段）
- **测试验证**：怎么确认行为没变

最后给出完整的重构后代码。"""

# 实战案例：重构一个 300 行的「上帝函数」

4.1 重构的安全保障

AI 重构最大的风险是「改对了结构但改错了行为」。所以一定要配合测试：重构前先跑一遍测试记录基准，重构后再跑一遍，结果必须一致。没有测试的代码不要让 AI 大规模重构——先让它生成测试，再重构。

★ 五、测试生成：覆盖你想不到的边界

写测试是开发者最常偷懒的事。AI 生成测试的价值不只是省时间，更在于它能想到人类想不到的边界情况——空值、并发、超大输入、非法格式。

# 测试生成的 prompt 模板（追求高覆盖率）
TEST_GEN_PROMPT = """你是测试工程师，请为以下函数生成完整的单元测试。

## 待测函数
```
{code}
```

## 函数说明
- 功能：{description}
- 输入约束：{input_constraints}
- 已知的边界情况：{known_edge_cases}

## 测试要求
1. **正常路径**：覆盖主要使用场景
2. **边界值**：空值、零值、最大值、最小值
3. **异常输入**：非法类型、格式错误、超长字符串
4. **并发安全**（如适用）：多线程调用
5. **性能边界**：超大输入下的行为

## 输出格式
使用 {test_framework} 框架，每个测试用例包含：
- 描述性的测试名（说明测试意图）
- Arrange-Act-Assert 结构
- 必要的 mock 和 fixture

## 覆盖率目标
行覆盖率 > 90%，分支覆盖率 > 80%"""

★ 六、完整的 AI 编程工作流

把前面四个场景串起来，就是一个完整的 AI 辅助开发工作流：

# 完整的 AI 辅助开发工作流
class AIDevelopmentWorkflow:
    """把 AI 融入开发的每一个环节。"""

    def step1_understand(self, requirement):
        """让 AI 帮你理解需求，列出实现要点和风险。"""
        prompt = f"分析这个需求，列出实现步骤、技术选型、潜在风险：\n{requirement}"
        return call_llm(prompt)

    def step2_design(self, requirement, codebase_context):
        """让 AI 设计方案，对比多个选项。"""
        prompt = f"""基于现有代码：{codebase_context}
        为这个需求设计 2-3 个实现方案，对比优缺点：{requirement}"""
        return call_llm(prompt)

    def step3_implement(self, design, context):
        """让 AI 写代码，但你要 review 每一行。"""
        prompt = f"按这个设计实现代码，遵循现有代码风格：\n{design}\n上下文：{context}"
        return call_llm(prompt)

    def step4_review(self, code):
        """让 AI 审查代码（前面讲过）。"""
        return review_code(code)

    def step5_test(self, code):
        """让 AI 生成测试（前面讲过）。"""
        return generate_tests(code)

    def step6_refactor(self, code):
        """让 AI 重构（前面讲过）。"""
        return refactor_code(code)

    def step7_document(self, code):
        """让 AI 写文档。"""
        prompt = f"为这段代码生成清晰的 docstring 和使用说明：\n{code}"
        return call_llm(prompt)

# 关键认知：AI 是你的「配对编程伙伴」，不是替代品
# 每一步你都要参与决策、验证结果、承担责任

★ 七、AI 编程的反模式：别这么用

AI 编程用不好会适得其反。几个常见的反模式：

  • 无脑复制粘贴：不看 AI 生成的代码就直接提交。后果：引入 Bug、安全漏洞、不符合团队规范。
  • 过度依赖：简单的问题也问 AI，丧失独立思考能力。
  • 上下文不足：给 AI 零散的代码片段，让它猜全貌。结果一定不好。
  • 用 AI 写、不测就上：AI 生成的代码必须有测试覆盖，否则就是定时炸弹。
  • 让 AI 做架构决策：架构决策要考虑业务、团队、历史，AI 的全局视野有限。

深 AI 辅助 Bug 定位与测试生成

AI 在编程上的两个高价值场景是Bug 定位和测试生成——它们都不需要 AI 写出完美生产代码，只需提供有价值的线索和草稿，人工再精修。这两个场景的可靠性门槛比“AI 直接写功能”低得多。

基于堆栈与日志的 Bug 定位

给 AI 喂堆栈、错误日志和相关代码，让它给出“可能原因 + 排查路径”，比让人肉眼翻代码快得多。关键是控制上下文，只给相关的。

BUG_PROMPT = '''你在帮定位一个 Bug。给你：错误堆栈、相关日志、相关代码片段。
请给出：
1. 最可能的根因(按可能性排序，最多3个)
2. 每个根因的证据(引用具体日志/代码行)
3. 验证方法(如何确认是哪个根因)
4. 修复方向(不要直接写完整代码，给思路)
如果信息不足，列出还需要看什么。不要编造没有的证据。'''

def locate_bug(stacktrace, logs, code_snippets):
    context = f"堆栈:\n{stacktrace}\n\n日志:\n{logs}\n\n代码:\n{code_snippets}"
    r = client.chat.completions.create(model="glm-4-plus", temperature=0.0,
        messages=[{"role":"system","content":BUG_PROMPT},{"role":"user","content":context}])
    return r.choices[0].message.content
# 经验：给 AI 相关代码(而非全仓) + 真实日志，定位准确率很高
# 它最强的不是猜根因，而是"告诉你接下来该看什么"，加速排查

AI 在 Bug 定位上最强的能力其实是“缩小搜索范围”——根据堆栈和日志，告诉你最该去看哪几个文件、哪几行，这比直接给答案更有用，因为它不会编造，却能大幅缩短人的排查时间。

测试生成：让 AI 补盲区

AI 生成测试的最大价值不是“省去写测试的力气”，而是覆盖人想不到的边界。人会下意识测“正常路径”，AI 反而更可能想到极端值、空集合、并发这类边界。用法是：给它被测函数的签名和实现，让它列出所有边界场景再生成测试，而不是直接生成测试代码。关键约束是要求它每个断言都要解释为什么期望这个结果——这能过滤掉“为了凑数而写的无意义断言”。生成的测试必须人工校验断言正确性——AI 可能算错期望值，让测试“通过但实际没验证任何东西”，这是最隐蔽的坑。一个高效流程是AI 生成测试骨架 → 人工补关键断言 → 跑覆盖率看盲区 → 再让 AI 补的循环。最后，用真实 Bug 倒推测试：每修一个 Bug，让 AI 生成一个回归测试，长期积累下来，测试集会越来越能防住历史踩过的坑，这是测试集自我进化最实际的路径。

深 AI 调试/测试 FAQ 与实战

AI 辅助调试和测试的核心是「提供线索而非替你判断」，这些问题最实用。

Q1：AI 给的 Bug 定位建议不准，怎么办？

给足相关上下文。AI 定位不准多半是信息不足——只给了错误信息没给相关代码/日志。改进：给完整堆栈+相关日志+相关代码片段+你的猜测；让 AI「先列出可能根因和验证方法，不要直接给结论」——它最强的其实是缩搜索范围而非猜答案；多轮对话逼近（按它的建议查，反馈结果，迭代）。AI 在「告诉你接下来看什么」上比「直接给答案」强，把它当排查向导而非预言机。给它越相关上下文，定位越准。

Q2：AI 生成的测试能信吗？

骨架可用，断言要人工校验。AI 生测试常有的问题：断言期望值算错（测试「通过但没验证任何东西」）、只测正常路径漏边界、为了凑数写无意义断言。正确用法：让 AI 先列边界场景再生成测试（它反而更可能想到极端值）；每个断言要求解释为什么期望这个值（过滤凑数断言）；人工校验断言正确性（最重要，别盲信）；跑覆盖率看盲区再补。AI 测试的最大价值是覆盖人想不到的边界，但要人来保证断言对。

★ 高级编码范式的边界与适用场景

高级编码范式（如函数式编程、响应式流、宏与元编程、依赖注入等）是强大的工具，但它们都有明确的适用边界。一个常见的反模式是"为了用而用"——团队引入某个范式仅仅因为它时髦，而非因为它解决了真实问题。判断是否应当引入某个范式的核心标准是：它能否让代码的"不变量"更显式、让"变化点"更隔离、让"测试"更容易写。如果引入后反而增加了认知负担却没有换来这些收益，那就是过度设计。

范式的选择还受团队成熟度制约。一个人员流动频繁、平均经验较浅的团队，强行采用高度抽象的范式（如大量使用自由单子、复杂的类型级编程），会显著降低代码的可维护性，因为新人理解成本过高。在这种场景下，"无聊但清晰"的命令式代码反而更优。反之，一个稳定且经验丰富的核心团队，则可以从高度抽象的范式中获得长期红利。因此范式选型本质上是"团队 context 的函数"，没有普适答案。

最后，要警惕范式之间的"隐性强耦合"。当一个项目同时混用过多范式（例如函数式 + 响应式 + 注解驱动的 AOP + 元编程），调试时会极难定位问题，因为执行流被分散在多个抽象层。建议每个项目明确"主导范式 + 受限的辅助范式"，并对辅助范式的使用范围做明确约束（例如只在边界层用响应式，领域层保持纯函数）。这种克制本身就是高级工程能力的体现。

★ 本章小结

这一章我们把 AI 辅助编程的四个高级场景实战了一遍。核心要点：

  • 代码审查：用结构化 prompt 让 AI 按正确性、安全性、性能、可读性、最佳实践五个维度审查，比人工更稳定全面。
  • Bug 定位：用「假设-验证」式推理 prompt，让 AI 跳出你的思维定势找根因，特别适合疑难 Bug。
  • 重构：明确目标和约束（不改外部行为），小步前进，配合测试验证。
  • 测试生成：让 AI 覆盖正常路径 + 边界值 + 异常输入 + 并发 + 安全，它能想到人类想不到的角落。
  • 完整工作流：理解需求 → 设计 → 实现 → 审查 → 测试 → 重构 → 文档，每一步都用 AI 做 leverage。
  • 反模式：不 review 就提交、过度依赖、上下文不足、不测试就上线、让 AI 做架构决策。

AI 编程的真正价值不是替代你写代码，而是放大你的能力——让你能处理更大的代码库、更复杂的问题、更高的质量标准。下一章我们会看开源社区的运营，理解如何用 AI 工具管理一个活跃的开源项目。
