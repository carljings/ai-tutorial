Token Embedding
从 ID 到语义向量的查表魔法

分词把文本变成了 token ID，但「猫」是 234、「狗」是 567——这些数字本身没有语义，234 和 567 的差并不代表「猫和狗的语义差」。Embedding 层的使命，就是把这些无意义的 ID，变成捕获了语义关系的稠密向量。本章讲清 embedding 的查表本质、为什么能学到语义、维度怎么选、位置 embedding 怎么组合，以及 tied embedding（输入输出权重共享）这个省参数又提效果的精巧设计。

阅读约 26 分钟 难度 ★★★★ 前置：第 107 章

★ Embedding 就是查表

Embedding 层的实现极其朴素：一个大矩阵，每行对应一个 token ID，查表就是把某行取出来。

Embedding 矩阵 E：形状 (vocab_size, d_model)
输入 token ID：t（一个整数）
输出 embedding：E[t]，即矩阵的第 t 行（一个 d_model 维向量）

就这样。没有复杂的计算，就是按行索引。这个矩阵 E 是可学习的参数，训练时通过反向传播更新每一行。训练收敛后，E 的每一行就成了对应 token 的语义向量。

词表：[猫=234, 狗=567, 鱼=890, ...]
Embedding 矩阵 E (vocab × d_model)
▼ 查表：取第 234 行
「猫」→ [0.2, -0.5, 0.8, ...] (d_model 维向量)
图 108-1 · Embedding 就是按 token ID 查矩阵的行
embedding_basics.py —— Embedding 的本质
import torch
import torch.nn as nn

# Embedding 层 = 一个可学习的矩阵
vocab_size = 1000
d_model = 64
embedding = nn.Embedding(vocab_size, d_model)

# 它内部就是一个 (vocab, d_model) 的矩阵
print("权重形状:", embedding.weight.shape)  # (1000, 64)

# 「查表」= 按索引取行
token_ids = torch.tensor([234, 567, 890])  # 三个 token
vectors = embedding(token_ids)
print("输出形状:", vectors.shape)  # (3, 64) —— 每个 ID 一行

# 手动验证：embedding(token) 等于直接索引权重矩阵
manual = embedding.weight[token_ids]
print("和手动索引一致:", torch.allclose(vectors, manual))  # True

就这么简单。Embedding 层没有非线性，没有矩阵乘，就是按 ID 取行。它之所以能学到语义，是因为这个矩阵是和整个模型一起训练的——通过反向传播，语义相近的 token 会被推到向量空间相近的位置。

01 为什么查表能学到语义

这是初学者最困惑的点：一个按行索引的操作，怎么就「学到了猫和狗都是动物」？

答案在分布式表示（distributed representation）和任务驱动的学习。关键洞察：语义相似的 token，在语言任务里会出现在相似的上下文中。「猫」和「狗」都会出现在「养了一只___」「___在叫」这样的上下文里。模型要预测下一个 token 时，「猫」和「狗」的 embedding 必须长得相似，才能在相同上下文里给出合理的预测。

语义相似 ⟺ 上下文分布相似 ⟺ Embedding 向量相似
这是 word2vec 的「分布假设」：一个词的意义由它的上下文决定。

训练过程会自然地把「猫」和「狗」的 embedding 推近，把「猫」和「汽车」的 embedding 推远——因为后者上下文分布差异大。所以 embedding 矩阵学到的，是基于上下文统计的语义关系，不是人手工编码的。

💡 著名的向量算术

word2vec 发现 embedding 学出了惊人的线性关系：vec("国王") − vec("男") + vec("女") ≈ vec("女王")。vec("巴黎") − vec("法国") + vec("中国") ≈ vec("北京")。这说明 embedding 空间里，「首都-国家」这类关系被编码成了方向。这是分布式表示的力量——语义关系自发地从数据中涌现。

02 Token Embedding + Position Embedding

第 96 章讲过，Transformer 需要位置信息。位置信息怎么和 token embedding 组合？有两种做法：

做法一：相加（原始 Transformer）

输入 = TokenEmbedding(t) + PositionEmbedding(pos)

位置编码（正弦余弦或可学习）和 token embedding 维度相同，直接相加。前面讲过，高维空间里两者近似正交，加法不破坏彼此的信息。

做法二：RoPE（现代主流）

输入 = TokenEmbedding(t)，然后在 Q、K 上应用 RoPE 旋转

RoPE 不在输入层加位置，而是在注意力的 Q、K 上做旋转（第 97 章）。所以现代大模型（Llama、GLM）的 embedding 层只做 token embedding，不加 position embedding——位置信息由后续的 RoPE 注入。

方案位置注入位置代表模型
正弦余弦（相加）输入层 embedding 相加原始 Transformer
可学习 PE（相加）输入层 embedding 相加BERT, GPT-2
RoPE（旋转）每层注意力的 Q/KLlama, GLM, Qwen
ALiBi（偏置）注意力分数加偏置BLOOM, MPT

03 维度 d_model 怎么选

embedding 的维度 d_model 是核心超参，影响整个模型：

  • 太小（128、256）：表达力不足，语义区分不开。早期小模型用。
  • 适中（768、1024、4096）：BERT-base 是 768，GPT-2 small 是 768，Llama-7B 是 4096。
  • 很大（8192、12288）：GPT-3 是 12288。表达力强，但参数和计算都涨。

经验法则：d_model 大致和模型规模的开立方成正比。模型参数量 N，d_model ∝ N^(1/3)（配合层数和头数也 ∝ N^(1/3)，这是「计算最优缩放」的结论）。所以 7B 模型 d_model≈4096，70B 模型 d_model≈8192。

embedding 矩阵的参数 = vocab_size × d_model。这是模型参数的重要组成部分。Llama-2-7B：32000 × 4096 = 1.3 亿（占 7B 的 1.9%）。规模变大时，embedding 占比反而下降（因为 FFN 和注意力参数增长更快）。

04 Tied Embedding：输入输出权重共享

Transformer 有两个地方用到词表维度的矩阵：

  • 输入端（Embedding 层）：E_in，形状 (vocab, d_model)。把 token ID 变成向量。
  • 输出端（LM Head，分类头）：E_out，形状 (d_model, vocab)。把最后的隐藏向量变成对每个词的预测分数（logits）。

这两个矩阵形状互为转置，且语义上应该相关（「猫」的输入向量和输出分数应该都对齐到「猫」这个概念）。Weight Tying（权重共享）的做法：让 E_out = E_in^T，共享同一组参数。

Tied: logits = hidden @ E_in^T   （输出投影直接用 embedding 矩阵的转置）
Untied: logits = hidden @ E_out   （输出投影是独立的矩阵）

好处：①省参数——vocab × d_model 的参数从两份变成一份（Llama-7B 省 1.3 亿）；②效果往往更好——输入和输出被迫对齐到同一个语义空间，理论上更一致。

早期 Transformer、GPT-2、早期 Llama 都用 tied embedding。但现代大模型（Llama 2、GLM、Qwen）大多改成 untied（不共享）。原因：

  • 大模型不在乎那点参数：untied 多的参数占比很小，但表达力更强。
  • 输入和输出的最优空间其实不同：输入 embedding 要便于「被注意力处理」，输出 head 要便于「分类到词表」。强行共享可能限制两者各自优化。
  • 实验显示 untied 略好：大模型上 untied 的最终质量略高于 tied。
tied_vs_untied.py —— 对比两种实现
import torch
import torch.nn as nn

class TiedModel(nn.Module):
    """权重共享：输出 head 用 embedding 矩阵的转置。"""
    def __init__(self, vocab_size, d_model):
        super().__init__()
        self.embedding = nn.Embedding(vocab_size, d_model)
        # 没有 LM head，直接用 embedding.weight 做输出

    def forward(self, token_ids, hidden_states):
        # token_ids 用于查 embedding
        emb = self.embedding(token_ids)
        # 输出 logits：直接用 embedding 矩阵转置
        logits = hidden_states @ self.embedding.weight.T  # 共享！
        return emb, logits

class UntiedModel(nn.Module):
    """权重不共享：embedding 和 LM head 是两个独立矩阵。"""
    def __init__(self, vocab_size, d_model):
        super().__init__()
        self.embedding = nn.Embedding(vocab_size, d_model)
        self.lm_head = nn.Linear(d_model, vocab_size, bias=False)  # 独立！

    def forward(self, token_ids, hidden_states):
        emb = self.embedding(token_ids)
        logits = self.lm_head(hidden_states)  # 用独立的 head
        return emb, logits

# 参数对比
vocab, d = 32000, 4096
tied = TiedModel(vocab, d)
untied = UntiedModel(vocab, d)
tied_p = sum(p.numel() for p in tied.parameters())
untied_p = sum(p.numel() for p in untied.parameters())
print(f"Tied 参数:   {tied_p:,}")      # 131,072,000
print(f"Untied 参数: {untied_p:,}")    # 262,144,000 —— 多一倍 embedding 参数
print(f"Untied 多: {(untied_p - tied_p)/1e6:.0f}M")

05 Embedding 怎么初始化

Embedding 矩阵的初始化影响训练起步。常见做法：

  • 随机小值：正态分布 N(0, 0.02²) 或均匀分布。GPT 系列用 N(0, 0.02²)。
  • 缩放到合理范围：除以 √d_model，让初始 embedding 的方差和位置编码相当（避免相加时一方淹没另一方）。
  • 预训练初始化：从 word2vec、GloVe 这些预训练向量初始化。早期做法，现代 LLM 不用了（数据足够多，自己学更好）。

关键：初始 embedding 是随机的，语义是训练过程中逐渐「长出来」的。没有先验知识，全靠数据驱动。

06 Embedding 之后的处理

查完 embedding，通常还会做两件事：

  1. 乘以 √d_model（原论文做法）：把 embedding 的尺度放大到和位置编码相当，避免相加后被位置编码淹没。现代用 RoPE 的模型一般不做这步（RoPE 在 Q/K 上旋转，不涉及 embedding 相加）。
  2. Dropout：随机把一些维度置零，防过拟合。现代大模型常省略（数据多，不需要）。

然后 embedding 就进入第一个 Transformer block（注意力 + FFN），开始层层加工。

07 输出端：从隐藏向量到词表概率

经过几十层 Transformer 后，最后一个 token 的隐藏向量 h（d_model 维）要变成「下一个 token 是哪个」的预测。流程：

logits = h @ E_out   （d_model → vocab_size，得到每个词的分数）
probs = softmax(logits)   （归一化成概率）
next_token = argmax(probs) 或按概率采样

这里的 E_out 就是上一节讲的 LM head（tied 或 untied）。本质是把隐藏向量和每个词的 embedding 做点积——点积最大说明隐藏向量和那个词的语义最匹配，预测就是它。

这给了 embedding 一个双重的语义：输入端「猫」查表得到向量，输出端把隐藏向量和「猫」的向量做点积。如果 tied，两者用同一个矩阵，「查」和「被查」共享一套语义坐标。

08 子词的 embedding 有什么特别的

现代 LLM 用子词分词，embedding 学的是子词的语义，不是整词。这带来一个有趣现象：

  • 「un」+「happy」=「unhappy」的语义：子词的 embedding 会被注意力组合出整词的语义。比如「unhappiness」被切成 ['un','happiness']，注意力层要学会把「否定」（un）和「幸福」（happiness）组合成「不幸福」。
  • 罕见词靠子词组合：没见过的词被拆成子词，每个子词的 embedding 都学过，组合起来能得到合理的语义表示。这就是子词分词泛化性好的原因。
  • 词缀的语义被单独学到：「re_」（再）、「un_」（否定）、「_ly」（副词）这些词缀的 embedding 会学到稳定的语义，组合时发挥规则作用。

所以 embedding 不只是「背单词」，它学到的是语言的组成单元及其组合规律。这也是为什么子词 + embedding 比字符级或整词级都强。

09 Embedding 的参数规模与压缩

embedding 矩阵是模型里「长而扁」的参数块（vocab × d_model）。对大词表模型（vocab=12.8 万），它占的参数不小：

Llama-3-8B：vocab=128k, d_model=4096 → embedding 参数 = 5.2 亿（占 8B 的 6.5%）
GPT-4（推测）：vocab=100k, d_model=12288 → embedding 参数 = 12 亿

量化时 embedding 是重点压缩对象。常见做法：①embedding 用比其他层更激进的量化（如 INT8 甚至 INT4）；②词表分片到多卡（Vocab Parallel Embedding）；③对罕见 token 的 embedding 做低秩分解（少见但有探索）。

📌 词表大小影响的不只是 embedding

词表越大，embedding 参数越多，但每个参数能分到的训练数据变少（总 token 数固定时，分给每个 token 的更新少了）。所以词表大的模型，罕见 token 的 embedding 学得不充分。这是词表不能无限大的另一个原因——除了参数，还有「数据不足以训练好每个 embedding」的问题。

10 分词器和 embedding 的协同

分词器（第 107 章）和 embedding 是协同设计的。分词器决定了词表，embedding 学的是词表里每个 token 的向量。两者必须匹配：

  • 分词器变了，embedding 必须重训：词表变了，embedding 矩阵的行数变了，旧的 embedding 用不了。这就是为什么「换分词器」是个大改动。
  • 词表的语义粒度决定 embedding 的学习难度：如果「自然语言处理」被切成 5 个 token，embedding 只需学 5 个子词；如果切成 1 个整词 token，embedding 要单独学这个整词（且数据可能不够）。
  • 多语言词表的 embedding 要平衡：中文 token 多了，英文就少（总词表固定），反之亦然。embedding 参数要在语言间分配。

这就是为什么大模型的训练流程是：①先确定分词器（在语料上训 BPE）；②确定词表；③再训 embedding 和模型。分词器一旦定下，整个训练周期内不变。

11 本章小结

Embedding 核心
  • 本质：按 token ID 查矩阵的行
  • 语义从训练中涌现（分布式表示）
  • 相似 token → 相似向量（上下文分布决定）
  • d_model 选 ∝ N^(1/3)
Tied vs Untied
  • Tied：输入输出共享，省参数
  • Untied：独立，表达力更强
  • 早期模型 tied，现代大模型 untied
  • 分词器定了，embedding 才能训

到此，Transformer 的输入端——从原始文本到可处理的向量——就完整讲完了：分词把文本切成 token ID，embedding 把 ID 变成语义向量，位置信息（RoPE）在后续注意力里注入。这条流水线是所有 LLM 的起点。

回顾从第 94 章到本章的 15 篇内容，我们把 Transformer 的每一个核心组件都拆透了：注意力的数学、QKV 投影、位置编码（正弦余弦/RoPE/ALiBi）、多头与 GQA、KV Cache、Flash Attention、FFN 的演进、层归一化、Softmax 变体、长上下文工程、分词、Embedding。这些组件拼在一起，就是一个完整的、能理解语言、能生成文本的现代大模型。下一章我们会把这些组件组装成一个完整的 Decoder-only 架构（GPT/GLM 的形态），看它们怎么协同工作。
