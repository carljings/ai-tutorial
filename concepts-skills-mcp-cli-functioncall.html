Skills / MCP / CLI / Function Call 到底什么区别

这四个词总让人犯晕——它们都在讲"让 AI 做更多事"，但本质完全不同。这一章用类比、对比表、完整故事和代码，把它们彻底讲清。读完你会明白：它们不在一个层面上，是配合关系，不是竞争关系。

概念辨析深入浅出★ 必读

为什么这四个概念总让人犯晕

如果你刚开始接触 AI agent，几乎一定会被这四个词绕进去：Skills、MCP、CLI、Function Call。它们听起来都在讲"让 AI 做更多事"，文档里又经常混着用，到底谁是谁？这一章就用最直白的方式把它们彻底讲清——不堆术语，只讲本质。读完你会明白：这四个根本不在一个层面上，把它们放一起比较就像拿"菜谱""厨具""手""切菜动作"来比，它们是配合关系，不是竞争关系。

先把结论亮出来，后面逐个展开：Function Call 是 AI "开口要东西"的能力；CLI 是"命令行"这种操作界面；MCP 是连接外部系统的"标准插头"；Skills 是教 AI "怎么做某类复杂任务"的知识包。一个负责说、一个负责界面、一个负责接、一个负责懂。下面我们用一个统一的场景把它们串起来。

💡 要点

一句话记忆：Function Call = 嘴（说要什么） · CLI = 手的动作（敲命令） · MCP = 插头（接外部系统） · Skills = 脑子里的经验（知道怎么做）。

先认识这四位"选手"

在深入之前，先给每个人画个速写，建立初步印象。不用记细节，只要感受到它们各管一摊就行。

Function Call（函数调用）：AI 学会"点菜"

最早的 AI 只会聊天——你说一句它回一句，它没办法真的"去做"任何事。Function Call 的出现改变了一切：它让 AI 能在回答里"附带一个结构化的请求"，比如告诉系统"帮我调用查天气这个函数，参数是城市=上海"。系统收到这个请求，真的去查天气，把结果喂回给 AI，AI 再据此回答你。

打个比方：以前 AI 是个只会动嘴的顾问，你说"今天出门带伞吗"，它只能说"建议你查查天气"。有了 Function Call，它变成了能"按下服务铃"的顾问——它会按一下铃（发起函数调用），让服务员（系统）去查天气，查完回来它再告诉你"带伞，今天有雨"。这个"按铃"的能力，就是 Function Call。

注意：Function Call 本身不执行任何操作，它只是 AI 表达"我要调用某个功能"的一种约定格式。真正执行的是系统侧的代码。AI 是点菜的顾客，系统是厨房；Function Call 是那张点菜单。

CLI（命令行）：人和机器的"文字对话框"

CLI 是 Command Line Interface 的缩写，就是那个黑底白字、靠敲命令操作的界面。ls、cd、git、npm 这些都是 CLI 命令。CLI 不是 AI 专属的概念——它从计算机诞生起就存在，是最古老也最强大的人机交互方式之一。

在 AI agent 的语境里，CLI 之所以重要，是因为很多 agent（比如 Codex、Aider、OpenCode）就运行在命令行里，而且它们执行任务时往往通过"敲命令"来完成——AI 决定要跑个测试，它就调用 bash 这个工具执行 npm test 这样的 CLI 命令。所以 CLI 既是 agent 的"工作台"（agent 本身在命令行里跑），也是 agent 的"操作方式"（通过执行命令来干活）。

类比：CLI 就像是厨房里的"操作台"——所有切、炒、煮的动作都在这上面发生。AI agent 站在这个操作台前，决定执行什么操作。

MCP（模型上下文协议）：AI 世界的"USB 标准"

MCP 是 Model Context Protocol 的缩写。它解决的问题是：每接一个外部系统（数据库、Slack、GitHub、文件系统），都要单独写一套对接代码，太累了。MCP 定义了一套统一协议——只要外部系统按这个协议封装成"MCP server"，任何支持 MCP 的 AI agent 都能直接连上用它，不用为每个 agent 单独适配。

类比：MCP 就像 USB 接口标准。USB 出现之前，鼠标是圆口、键盘是 PS/2、打印机是并口，每个设备一种线。USB 统一了接口，任何设备只要做成 USB 的，任何电脑只要有 USB 口就能用。MCP 之于 AI agent，就是 USB 之于电脑——一个"插上就能用"的统一标准。

具体来说，一个 MCP server 把自己提供的"能力"（查询数据、发消息、读文件）按 MCP 协议暴露成一组"工具"，agent 通过 MCP 协议发现这些工具、调用它们。从 agent 的视角看，不管背后是 PostgreSQL 还是 Slack，调用方式都一样——这就是标准化的力量。

Skills（技能）：AI 的"操作手册"

Skills 是最近兴起的概念，代表"打包好的、可复用的任务执行知识"。一个 Skill 本质上是一个文件夹，里面有一份 SKILL.md 说明书（告诉 AI 这类任务该怎么做）加上模板、示例等辅助文件。当 AI 遇到相关任务时，加载对应的 Skill，就"学会了"这套做事方法。

类比：Skills 就像是新员工入职时发的"SOP 操作手册"。手册里写着"处理客户投诉分五步：倾听→记录→分类→回应→归档"。新员工（AI）原本不知道怎么做，但读了手册就按这套流程走。不同的 Skill 就是不同的手册——代码审查手册、写周报手册、部署流程手册。

💡 要点

关键洞察：这四个不在同一层面。Function Call 是"语言"（怎么对话）、CLI 是"地点"（在哪干）、MCP 是"通道"（怎么接）、Skills 是"知识"（怎么干）。它们是配合关系，不是替代关系。

用一个完整故事把它们串起来

抽象对比容易忘，我们用一个具体任务把四个概念串成一条线。任务是这样的：用户对 AI agent 说"帮我看下这个月的销售数据，写份月报发到团队群"。这看似一句话，背后四个概念全用上了。

第一步：Function Call 让 AI "开口"

AI 接到这个任务，它知道自己需要两样东西：销售数据、发消息的能力。但它自己没有这些能力——它只是个语言模型。于是它用 Function Call 发起请求："请调用 query_sales 函数，参数 month=本月"。这个"调用 query_sales"的请求，就是 Function Call。AI 在这里扮演点菜的角色，Function Call 是点菜单。

第二步：MCP 提供"连接外部系统"的通道

query_sales 这个函数背后是什么？是公司的销售数据库。AI 怎么连这个数据库？通过 MCP。有人把销售数据库封装成了一个 MCP server，按 MCP 协议暴露了 query_sales 这个工具。AI agent 通过 MCP 协议连上这个 server，就能调用它——不用关心背后是 MySQL 还是 MongoDB，MCP 把这些差异屏蔽了。MCP 在这里扮演 USB 的角色，让 AI 能"插上"数据库。

第三步：CLI 可能是执行动作的方式

数据拿到了，AI 要生成月报。它可能调用一个生成报告的工具——而这个工具底层可能是通过 CLI 执行的，比如跑一个 pandoc 命令把 Markdown 转成 PDF。或者 AI 要把报告发到团队群，它调用的发消息工具，底层可能是执行一个 curl 命令调用 webhook。这里 CLI 是"执行动作的操作台"——很多 agent 的工具，底层就是执行各种命令行命令。

第四步：Skills 教 AI "怎么做月报"

但 AI 怎么知道"月报该包含哪些章节、用什么格式、数据怎么呈现"？这需要领域知识。如果 agent 装了"月报生成"这个 Skill，AI 会先加载它（读 SKILL.md），里面写着"月报结构：概述→数据总览→亮点→问题→下月计划，用公司模板，数据用表格呈现"。AI 按这套"操作手册"来组织报告。Skills 在这里扮演 SOP 手册的角色，让 AI 从"乱写一气"变成"按规范来"。

💡 要点

串起来看：Skills 教 AI 怎么做（知识）→ Function Call 是 AI 表达需求的格式（语言）→ MCP 提供连接数据库的通道（集成）→ CLI 是执行具体命令的操作台（界面）。一个任务，四个概念各司其职，缺一不可。

最容易搞混的两对关系

despite 四者不在同一层面，初学者还是容易在两处混淆。我们把这两处单独拎出来讲透。

混淆一：Function Call vs MCP

很多人问："Function Call 不也是调用工具吗？MCP 不也是提供工具吗？它俩啥区别？"区别在于层次：Function Call 是 AI"怎么表达要调用工具"的格式约定（一张点菜单的格式），MCP 是"工具怎么被封装和发现"的集成标准（厨房怎么接入外卖平台）。

具体说：AI 通过 Function Call 说"我要调用 query_sales"——这是 Function Call 的作用。但 query_sales 这个工具从哪来、怎么连数据库、怎么返回结果——这些是 MCP server 负责的。没有 MCP，你得自己手写一个 query_sales 的实现并注册给 AI；有了 MCP，你只要起一个 MCP server，AI 自动发现并调用。Function Call 是"点菜的格式"，MCP 是"菜从哪来的供应链"。

换言之：MCP server 提供的工具，最终也是通过 Function Call 被 AI 调用的。它们不是竞争，是上下游——MCP 负责"工具的供给"，Function Call 负责"工具的消费"。

混淆二：MCP vs Skills

另一对常被搞混："MCP 和 Skills 不都是扩展 AI 能力吗？"是，但扩展的是完全不同的东西。MCP 扩展的是"能连什么"（access）——连数据库、连 Slack、连 GitHub。Skills 扩展的是"会怎么做"（know-how）——怎么做代码审查、怎么写月报。

一个更能说明问题的角度：MCP 提供能力，但 AI 可能不知道什么时候用、怎么用好。比如 MCP 给了一个"查数据库"的工具，但 AI 不知道该查哪些表、怎么 Join、结果怎么解读才算"对"。Skills 补的就是这一块——它教 AI 在特定场景下，该怎么组合使用这些工具。MCP 是"给 AI 一把锤子"，Skills 是"教 AI 这把锤子该敲哪里、怎么敲"。

所以它们是配合关系：Skills 里写的操作步骤，经常会用到 MCP 提供的工具。比如"月报技能"的步骤里会写"调用 query_sales 拿数据"——这个 query_sales 就是 MCP server 提供的。Skills 是方法，MCP 是手段。

💡 要点

终极记忆法：把 AI agent 想象成一个新员工。Function Call 是它跟你沟通用的语言；CLI 是它工作的电脑终端；MCP 是公司给它开通的各种系统账号（邮箱、CRM、数据库）；Skills 是你发给它的岗位操作手册。语言、终端、账号、手册——缺哪个它都干不好活。

代码层面的对照

最后用代码把它们各自的"样子"展示出来，建立具象认知。你会发现它们在代码里形态完全不同，根本不会搞混。

Function Call 的样子：一段 JSON

// AI 返回的 Function Call 请求（嵌在回复消息里）
{
  "role": "assistant",
  "tool_calls": [{
    "id": "call_abc123",
    "type": "function",
    "function": {
      "name": "query_sales",
      "arguments": "{"month": "2026-06"}"
    }
  }]
}
// 这就是 Function Call —— AI 用 JSON 表达"我要调用 query_sales"

CLI 的样子：一条终端命令

# AI 决定生成 PDF，底层执行 CLI 命令
$ pandoc report.md -o report.pdf --template=company.tex

# AI 决定发消息，底层可能是
$ curl -X POST https://hooks.slack.com/xxx \
    -d '{"text":"本月销售月报已生成"}'

# CLI 就是这些在终端里敲的命令

@server.tool()
def query_customers(region: str) -> dict:
    """查询某区域的客户列表。"""
    return db.query(f"SELECT * FROM customers WHERE region='{region}'")

# server.run() 后，任何 MCP 兼容的 agent 都能发现并调用这两个工具
# 这就是 MCP —— 一个按协议暴露工具的运行时服务

Skills 的样子：一个文件夹

monthly-report/              ← 技能文件夹
├── SKILL.md                 ← 入口说明书（AI 只读这个就懂怎么做）
│   ---
│   name: monthly-report
│   description: 按公司规范生成本月销售月报...
│   ---
│   # 月报生成流程
│   1. 调用 query_sales 拿数据（这个工具由 MCP 提供）
│   2. 按模板组织：概述→数据→亮点→问题→计划
│   3. 生成 PDF 并发送
├── template.tex             ← 公司月报模板（按需加载）
└── examples/
    └── sample.md            ← 范例（按需加载）
# 这就是 Skills —— 一个装着"怎么做"知识

看，四者在代码里的形态天差地别：Function Call 是 JSON 消息、CLI 是终端命令、MCP 是 server 进程、Skills 是文件夹。记住了这些"长相"，就不会再混淆了。

它们如何协同：一张架构图

把四者放在一张架构图里，配合关系一目了然。想象一个完整的请求处理流程：

用户："帮我写本月销售月报"
   │
   ▼
┌─────────────────────────────────────────────┐
│  AI Agent（大脑）                            │
│                                             │
│  ① 加载 Skills「monthly-report」            │  ← Skills：教 AI 怎么做
│     "月报要含概述/数据/亮点/问题/计划"        │
│                                             │
│  ② Function Call: query_sales(month=06)     │  ← Function Call：AI 开口要
│     │                                       │
│     ▼                                       │
│  ③ MCP Server「sales-db」执行查询           │  ← MCP：连接外部数据库
│     │                                       │
│     ▼                                       │
│  ④ 底层执行: psql -c "SELECT..."            │  ← CLI：实际敲的命令
│     │                                       │
│     ▼ 返回数据                              │
│  ⑤ AI 按技能指引组织报告                    │
│  ⑥ Function Call: send_slack(...)           │
│     └→ MCP Server「slack」发送              │
└─────────────────────────────────────────────┘
   │
   ▼
用户收到："月报已生成并发送 ✓"
💡 要点

从图里能清楚看到：Skills 在最上层指挥"怎么做"，Function Call 是 AI 每一步表达需求的格式，MCP 在中间层连接外部系统，CLI 在最底层执行具体命令。自上而下：知识→表达→连接→执行。

常见疑问快答

Q：学会了 Function Call 是不是就不用 MCP 了？

不是。Function Call 只是"格式"，你得有工具让它调用才行。如果你自己写工具实现，确实可以不用 MCP。但每接一个系统就写一套，太累。MCP 让你用现成的 server，省去重复造轮子。所以它们是配合：MCP 提供工具池，Function Call 是调用工具的格式。

Q：Skills 和"写一个好的 system prompt"有什么区别？

本质上 Skills 就是一种"按需加载的、模块化的 prompt"。区别在于工程：system prompt 是常驻的（一直占 token），Skills 是按需的（用到才加载）。你可以把 100 个 Skills 的简介都放进 system prompt，但只在用到某个时才加载它的完整内容——这就是上一章讲的"渐进式披露"。Skills 是 prompt 工程的工程化升级。

Q：CLI 要被 AI agent 淘汰吗？

恰恰相反——AI agent 让 CLI 更重要了。因为 agent 执行任务时大量依赖命令行工具（git、npm、grep、curl），CLI 是 agent 的"手"。而且 agent 本身很多就是 CLI 程序（Codex、Aider、OpenCode）。所以 AI 时代 CLI 不但没消失，反而成了人机协作的核心界面。

Q：我该先学哪个？

建议顺序：先理解 Function Call（它是基础，不懂这个后面的都悬）→ 再学 MCP（它是当前最热的集成标准，实用价值高）→ 然后看 Skills（它是更上层的方法论封装）→ CLI 作为通用技能随时补。如果你是开发者，Function Call + MCP 是必学的硬技能；如果你是应用者，Skills 能让你快速给 agent 装上专业能力。

本章小结

Function Call、CLI、MCP、Skills 是扩展 AI agent 能力的四种机制，但它们分属不同层面，是配合而非竞争。Function Call 是 AI 表达需求的"语言"，CLI 是执行操作的"界面"，MCP 是连接外部系统的"标准插头"，Skills 是封装做事方法的"知识包"。一个完整的 agent 任务往往四者协同：Skills 教怎么做，Function Call 表达每一步要什么，MCP 连接外部系统提供能力，CLI 在底层执行具体命令。

记住那个终极类比：把 AI agent 当新员工。Function Call 是它说话的方式，CLI 是它的电脑终端，MCP 是给它开的系统账号，Skills 是发的操作手册。四样齐备，它才能独立干活。分清了这四者，你再看任何 agent 的文档都不会犯晕——因为你已经建立了"哪个概念属于哪个层面"的清晰坐标系。
