深度分块
Late Chunking、Contextual Retrieval 与 parent-child

上一章的分块策略有个根本缺陷：块被从文档里切出来后，就丢失了"它属于哪篇文档、在哪个上下文里"的信息。一个孤立的块" revenues grew 20%"，embedding 模型根本不知道这是哪家公司、哪个季度。本章介绍三种解决"上下文丢失"的前沿方案——Late Chunking、Contextual Retrieval、parent-child 分块——它们是 2024 年 RAG 效果提升的重要来源。

★ 分块的根本病：上下文剥离

传统的分块流程是"先切再 embed"：把文档切成块，每个块独立送进 embedding 模型。问题在于，embedding 模型是 transformer，它的力量来自 self-attention——句子的每个 token 通过 attention 互相参照来生成表示。当你把一个孤立的块喂进去，它只能看到块内的 token，看不到这个块在原文档里的位置、所属章节、上文提到的主语。

举个例子，一篇财报里有这么一段：

Acme Corp 2024 年 Q3 财报显示，公司营收同比增长 20%，达到 50 亿美元。
云服务业务表现尤其亮眼，贡献了 60% 的营收增长。研发投入同比增加 15%。

如果按句切，第二块"云服务业务表现尤其亮眼，贡献了 60% 的营收增长"被独立 embed 时，模型不知道这是 Acme Corp、是 Q3、是 2024 年。用户问"Acme Corp Q3 云服务增长多少"，这个块的 embedding 和查询的相似度会偏低——因为查询里有"Acme Corp"，而块的 embedding 里没有这个信息。

这就是"上下文剥离"问题，是传统分块精度的天花板。本章三种方案从不同角度解决它。

★ Late Chunking：先 embed 整篇，再切

Jina AI 在 2024 年提出的 Late Chunking 思路非常巧妙：把整个文档先一次性喂进 embedding 模型，让 transformer 的 attention 把全局上下文融入每个 token 的表示，然后再按块边界切这些 token 表示，每块做 mean pooling 得到块向量。

关键点在于"切晚一点"——传统是"先切再 embed"，Late Chunking 是"先 embed 再切"。这样每个块的向量里，因为 attention 已经把全文信息融入了，自然就包含了"我是 Acme Corp Q3 的云服务块"这种上下文。

前提条件：embedding 模型必须支持长输入（8K token 以上），否则整篇文档塞不进去。jina-embeddings-v2-base（8K）、bge-m3（8K）都支持。所以 Late Chunking 是长上下文 embedding 模型成熟后才可行的方案。

# Late Chunking 的概念实现（用 jina-embeddings-v2 为例）
from transformers import AutoModel, AutoTokenizer
import torch
import numpy as np

model_name = "jinaai/jina-embeddings-v2-base-en"
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModel.from_pretrained(model_name, trust_remote_code=True)

doc = """Acme Corp 2024 Q3 报告：营收同比增长 20%，达 50 亿美元。
云服务业务贡献了 60% 的营收增长。研发投入同比增加 15%。"""

# 1. 先按句确定块的 token 边界（不实际切文本）
sentences = doc.split("\n")
chunk_starts = []   # 每块在 token 序列里的起始位置
offset = 1          # 跳过 [CLS]
boundaries = []
for s in sentences:
    tokens = tokenizer(s, add_special_tokens=False)["input_ids"]
    boundaries.append((offset, offset + len(tokens)))
    offset += len(tokens) + 1   # +1 for 换行/分隔 token

# 2. 关键：整篇文档一次性喂进模型，得到每个 token 的表示
inputs = tokenizer(doc, return_tensors="pt", truncation=True, max_length=8192)
with torch.no_grad():
    outputs = model(**inputs)
token_embs = outputs.last_hidden_state[0]   # (seq_len, hidden)

# 3. 在 token 表示层面切块 + mean pooling
chunk_vectors = []
for start, end in boundaries:
    chunk_vec = token_embs[start:end].mean(dim=0)
    chunk_vectors.append(chunk_vec.numpy())
chunk_vectors = np.stack(chunk_vectors)
chunk_vectors = chunk_vectors / np.linalg.norm(chunk_vectors, axis=1, keepdims=True)

print(f"块数: {len(chunk_vectors)}, 每块维度: {chunk_vectors.shape[1]}")
# 每个块向量都"知道"自己是 Acme Corp Q3 的内容，因为整篇经过 attention 了

★ Contextual Retrieval：用 LLM 给每块加上下文

Anthropic 在 2024 年提出的 Contextual Retrieval 是另一个角度的解法：分块照常，但 embed 之前，先用一个 LLM 给每个块生成一段简短的上下文摘要，前置到块里，再 embed。

具体做法：对每个块，让 LLM 读完整文档后回答"这个块在文档里的上下文是什么"，把 LLM 的回答加到块前面。比如对上面的"云服务业务..."块，LLM 会生成："这段话出自 Acme Corp 2024 年 Q3 财报，描述云服务业务的表现。"然后把这段话 + 原块一起 embed。

代价是每块要调一次 LLM（百万块要百万次调用，成本不小）。Anthropic 的实验显示这能把检索失败率降低 49%（配合 rerank 再降 67%）。优化手段是用 prompt caching（同一文档的块共享缓存）把成本压到原来的 1/10。

# Contextual Retrieval：用 LLM 给每块加上下文
from anthropic import Anthropic
import anthropic

client = anthropic.Anthropic()   # 设置 ANTHROPIC_API_KEY

CONTEXTUAL_PROMPT = """
{document}

Here is the chunk we want to situate within the whole document:

{chunk}

Please give a short succinct context to situate this chunk within the overall document
for the purposes of improving search retrieval of the chunk. Answer only with the
succinct context and nothing else. 用中文回答。"""

def add_context_to_chunk(doc, chunk, model="claude-3-5-haiku-latest"):
    resp = client.messages.create(
        model=model,
        max_tokens=200,
        messages=[{"role": "user",
                   "content": CONTEXTUAL_PROMPT.format(document=doc, chunk=chunk)}],
        # 利用 prompt caching 缓存 document 部分，降低重复成本
        extra_headers={"anthropic-beta": "prompt-caching-2024-07-31"},
    )
    context = resp.content[0].text.strip()
    return f"{context}\n\n{chunk}"      # 上下文前置 + 原块

# 对所有块处理
chunks = recursive_split(doc, chunk_size=300)
contextualized = [add_context_to_chunk(doc, c) for c in chunks]
# 再用 contextualized 列表去 embed，召回质量显著提升

★ parent-child 分块：召回小块，返回大块

第三种方案是结构性的：把文档做两层分块——粗粒度的 parent 块（如 1000 字）和细粒度的 child 块（如 200 字）。child 块用于 embedding 和检索（小块语义聚焦、向量质量高），但检索命中后返回的是它所属的 parent 块（大块上下文完整，喂 LLM 时信息全）。

这个思路的精髓是"分离检索粒度和返回粒度"：小块利于精确召回（用户问一个具体细节，小块 embedding 更聚焦），大块利于 LLM 理解（喂回去时上下文不缺失）。LlamaIndex 的 AutoMergingRetriever、LangChain 的 ParentDocumentRetriever 都实现了这个策略。

# parent-child 分块：LangChain ParentDocumentRetriever
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.retrievers import ParentDocumentRetriever
from langchain.storage import InMemoryStore
from langchain_community.vectorstores import Chroma
from langchain_community.embeddings import HuggingFaceEmbeddings
from langchain_core.documents import Document

embeddings = HuggingFaceEmbeddings(model_name="BAAI/bge-small-zh-v1.5")

# child splitter：小块，用于检索
child_splitter = RecursiveCharacterTextSplitter(chunk_size=200, chunk_overlap=30)
# parent splitter：大块，用于返回
parent_splitter = RecursiveCharacterTextSplitter(chunk_size=1000, chunk_overlap=100)

# 向量库存 child 块；InMemoryStore 存 parent_id → parent 块的映射
vectorstore = Chroma(embedding_function=embeddings)
store = InMemoryStore()

retriever = ParentDocumentRetriever(
    vectorstore=vectorstore,
    docstore=store,
    child_splitter=child_splitter,
    parent_splitter=parent_splitter,
)

# 添加文档：自动按 parent 切，每个 parent 再切成 child，child 入向量库
docs = [Document(page_content=long_doc_text, metadata={"source": "report.pdf"})]
retriever.add_documents(docs)

# 检索：内部用 child 块匹配，但返回 parent 块
results = retriever.invoke("云服务营收增长多少")
# results 里是 1000 字的 parent 块，包含完整的 Acme Corp 上下文
print(len(results[0].page_content))   # 约 1000，而非 200

★ 三种方案对比与选型

三种"上下文增强"方案对比
┌──────────────────────┬──────────────┬──────────────┬─────────────────────┐
│ 方案                 │ 召回提升     │ 成本         │ 适用                │
├──────────────────────┼──────────────┼──────────────┼─────────────────────┤
│ Late Chunking        │ 中（10-15%） │ 低（无LLM）  │ 长文档，模型支持8K  │
│ Contextual Retrieval │ 高（30-50%） │ 高（每块LLM）│ 高价值场景，预算足  │

└──────────────────────┴──────────────┴──────────────┴─────────────────────┘

实战建议：

  • 预算敏感的通用场景：parent-child 是 ROI 最高的，实现简单、效果稳定，建议作为默认。
  • 长文档 + 长上下文 embedding 模型：加 Late Chunking，几乎零成本再涨一截。
  • 高价值低频场景（如医疗、法律）：上 Contextual Retrieval，配合 prompt caching 控成本。
  • 三者可叠加：parent-child + Late Chunking + Contextual Retrieval 能叠加，但收益边际递减，按需取。

★ Contextual Retrieval 的成本控制实战

Contextual Retrieval 效果好但成本吓人——每个块调一次 LLM 生成上下文，百万块的语料要百万次 LLM 调用，按 GPT-4 价格算要几千到几万美元。这让很多人望而却步。但其实有几招能把成本压到可承受范围。

Prompt Caching 是最大杠杆。Anthropic 和 OpenAI 都支持缓存 prompt 的前缀部分。Contextual Retrieval 的 prompt 结构是"[长文档] + [当前块] + [生成上下文要求]"，其中"[长文档]"对同一文档的所有块是相同的。开启 caching 后，同一文档的 N 个块只算 1 次文档 token 费用 + N 次小块费用，成本降到原来的 1/10 甚至更低。

用便宜模型。生成上下文不需要顶级推理能力，用 Haiku/GPT-4o-mini 这种小模型即可，成本是 Sonnet/GPT-4 的 1/10。实测小模型生成的上下文摘要质量足够用于检索增强。

批量处理。把同一文档的多个块攒成一次 LLM 调用（一个 prompt 里列多个块，让 LLM 逐个生成上下文），减少调用次数。一次处理 10 个块，调用量降到 1/10。

选择性应用。不是所有块都需要 Contextual Retrieval。短文档（一块就够）不需要；高度自洽的块（如独立 FAQ 条目）不需要。只对"脱离上下文就难理解"的块应用，能省 50-70%。

# 带 caching 的 Contextual Retrieval（Anthropic）
def contextual_with_cache(doc, chunks, client):
    results = []
    for chunk in chunks:
        resp = client.messages.create(
            model="claude-3-5-haiku-latest",   # 便宜模型
            max_tokens=150,
            messages=[{"role": "user", "content": CONTEXTUAL_PROMPT.format(
                document=doc, chunk=chunk)}],
            # 关键：缓存 doc 部分
            extra_headers={"anthropic-beta": "prompt-caching-2024-07-31"},
            system=[{"type": "text", "text": doc,    # 文档作为缓存前缀
                     "cache_control": {"type": "ephemeral"}}]
        )
        results.append(f"{resp.content[0].text}

{chunk}")
    return results
# 带 caching + haiku，成本约为无缓存 Sonnet 的 1/30

组合这些手段，百万块语料的 Contextual Retrieval 成本能从几千美元压到几十到一百美元，进入可承受范围。ROI 计算：如果它能把你 RAG 系统的失败率降 30%，省下的客服/人工纠错成本远高于这个 LLM 费用。

★ Late Chunking 的工程落地细节

Late Chunking 听起来优雅，但工程落地有几个细节要处理。

块边界怎么定。Late Chunking 是"先 embed 整篇再切"，但仍要决定在哪些 token 位置切。最稳的做法是先用传统分块器（递归/语义）确定句子的边界，然后在 token embedding 层面按这些句子边界切。直接在 token 层面硬切（如每 128 token 切一刀）会切断句子，违背"语义边界"的初衷。

长文档的 token 限制。即便 bge-m3 支持 8K token，一份 50 页财报可能 3 万 token 塞不进。对策：先按章节切到大块（每块 < 8K），每个大块内做 Late Chunking 得到子块向量。这是"两阶段 Late Chunking"。

mean pooling 的细节。Late Chunking 对每个块的 token 向量做 mean pooling 得到块向量。但要注意：pooling 时应该排除特殊 token（[CLS]、[SEP]、padding），且对长块要考虑注意力衰减（中间 token 权重低）。jina 的实现里用了 attention-weighted pooling 缓解这个问题。

查询端要不要 Late。查询通常很短（几十 token），不需要 Late Chunking，正常 embed 即可。Late Chunking 只用于文档端。

# 两阶段 Late Chunking：先章节切，再段内 Late Chunk
def two_stage_late_chunking(long_doc, max_tokens=7000):
    # 第一阶段：按章节/段落切成大块（< max_tokens）
    sections = split_by_section(long_doc, max_tokens=max_tokens)
    all_chunk_vecs = []
    for section in sections:
        # 第二阶段：对每个 section 做 Late Chunking
        token_vecs = encode_full(section)          # 整 section embed
        # 用句子边界定块边界
        sent_boundaries = find_sentence_token_boundaries(section)
        for start, end in sent_boundaries:
            chunk_vec = token_vecs[start:end].mean(dim=0)
            all_chunk_vecs.append(chunk_vec)
    return all_chunk_vecs

这些细节决定了 Late Chunking 在生产里是"效果好"还是"效果一般"。它的理论很美，但工程化需要细致处理，这也是为什么它还没有成为默认方案——多数团队还在用更简单的 parent-child。

★ 上下文增强方法的组合与选型矩阵

Late Chunking、Contextual Retrieval、parent-child 三种方法不是互斥的，可以组合。但组合越多越复杂，要按 ROI 选。给一个选型矩阵。

单方法选型：

  • 预算极敏感、追求简单 → parent-child（实现最简单、效果稳定）。
  • 长文档 + 长上下文 embedding 模型 → + Late Chunking（零 LLM 成本）。
  • 高价值低频场景 → Contextual Retrieval（效果最强但贵）。

组合策略：

  • parent-child + Late Chunking：child 块用 Late Chunking 编码（带全文上下文），返回 parent。这是"上下文感知 + 大块返回"的强组合，几乎零 LLM 成本。
  • parent-child + Contextual Retrieval：child 块加上下文摘要后再 embed，命中返回 parent。效果最强但成本最高。
  • 三者全上：边际收益递减，不推荐除非极致精度场景。

    • 选型矩阵（按场景 × 预算）
┌──────────────┬─────────────┬──────────────┬──────────────────┐
│ 场景\预算    │ 低          │ 中           │ 高               │
├──────────────┼─────────────┼──────────────┼──────────────────┤
│ 短文档       │ 递归分块    │ +parent-child│ +Contextual      │
│ 长文档       │ 递归+Late   │ parent+Late  │ parent+Contextual│
│ 高价值(法律) │ -           │ Contextual   │ 全套+rerank      │
│ 高频(客服)   │ 递归分块    │ +parent-child│ +Late Chunking   │
└──────────────┴─────────────┴──────────────┴──────────────────┘

    评估组合效果。组合方法要用 RAGAS（140 篇）评估，确认提升且没回退。常见情况：A 方法单独提升 10%，B 方法单独提升 8%，A+B 组合提升 12%（不是 18%，因为有重叠收益）。所以组合的边际收益要实测，不能简单相加。

    # 组合方法的 A/B 评估
methods = {
    "baseline": naive_chunk_embed,
    "parent-child": parent_child_embed,
    "parent+Late": parent_child_with_late_chunking,
    "parent+Contextual": parent_child_with_contextual,
}
for name, fn in methods.items():
    chunks = fn(corpus)
    index = build_index(chunks)
    recall = eval_recall(index, eval_set)
    faith = eval_faithfulness(index, eval_set)
    print(f"{name:25s}: recall={recall:.3f} faithfulness={faith:.3f}")
# 看 recall 涨幅是否值得增加的复杂度/成本

    实施节奏。不要一上来就上最复杂的组合。从 parent-child 开始（最简单），评估效果，按瓶颈再加。如果 parent-child 后瓶颈在"块脱离上下文难理解"，加 Late Chunking 或 Contextual；如果瓶颈在别的（如检索召回、rerank），先解决那个。增量演进比一步到位务实。

    这个选型思维——"按瓶颈选方法、增量演进、实测验证"——适用于 RAG 的所有优化环节。盲目堆方法不仅浪费，还可能因为复杂度引入新问题。

    ★ Contextual Retrieval 的成本控制实战

    Contextual Retrieval 用 LLM 给每块加上下文，成本可能爆炸。几个实战控制方法。

    成本测算。100 万块 × 平均 200 token 上下文 × Haiku 价格（$0.25/百万输入 token）≈ $50。看着不贵，但实际每块要"块原文 + 已生成上下文"作为输入，输出上下文，往返 token 更多。实测 100 万块要 $150-300。对大型知识库（千万块）是 $1500-3000，要规划预算。

    Prompt Caching 降本。Claude/GPT 的 prompt caching 让重复前缀几乎免费。Contextual Retrieval 的 prompt 前缀（指令 + 文档背景）可以缓存——同一文档的所有块共享缓存，第二次起几乎免费。实测能降 70-90% 成本。这是 Anthropic 官方推荐的做法。

    # 用 prompt caching 的 Contextual Retrieval
def contextual_with_cache(doc, chunks, cached_client):
    # 文档背景作为缓存前缀（同一文档的块共享）
    cache_prefix = [{"type": "text", "text": f"文档背景：
{doc[:8000]}",
                     "cache_control": {"type": "ephemeral"}}]
    enriched = []
    for chunk in chunks:
        msg = cache_prefix + [{"type": "text",
            "text": f"给下面这段加一句上下文：
{chunk}"}]
        resp = cached_client.messages.create(model="claude-haiku",
            max_tokens=50, messages=[{"role": "user", "content": msg}])
        enriched.append(f"{resp.content[0].text}
{chunk}")
    return enriched
# 同文档的块第二次起命中缓存，成本降 90%

    批量处理。一次给 LLM 多个块，让它批量加上下文，减少 API 调用次数。比如一次处理 10 个块，往返开销摊薄。但要注意上下文窗口限制和 LLM 对多任务的质量下降——批量太多质量会掉，经验是 5-10 块一批。

    选择性增强。不是所有块都需要加上下文——标题、表格、明确段落自解释的块不需要。只给"脱离上下文难理解"的块加（中间段落、碎片化块）。用规则或小模型预测哪些块需要，能省 40-60% 成本。

    增量增强。新加入的块加上下文即可，老块不必重做。但 embedding 模型升级时要全量重做（上下文是模型无关的文本，但 embedding 编码会变）。

    ★ 本章小结

    • 传统分块的病是"上下文剥离"：块独立 embed 时丢失了文档全局信息，这是精度天花板。
    • Late Chunking（Jina）：整篇先 embed 让 attention 融入全局，再切 token 表示；需要长上下文 embedding 模型，零 LLM 成本。
    • Contextual Retrieval（Anthropic）：每块用 LLM 生成上下文摘要前置，再 embed；效果最强但每块一次 LLM 调用，用 prompt caching 控成本。
    • parent-child 分块：小块检索、大块返回，分离检索粒度和返回粒度；实现简单、ROI 高，建议作为默认。
    • 选型：parent-child 通用首选，长文档加 Late Chunking，高价值场景上 Contextual Retrieval，三者可叠加。

    下一篇我们把这些零件组装起来，看 RAG 架构的演进——从 Naive RAG 到 Advanced RAG 再到 Modular RAG。

    import numpy as np
from dataclasses import dataclass
from typing import List, Optional

@dataclass
class PerformanceMetrics:
    latency_ms: float
    throughput_ops: float
    memory_mb: float
    accuracy: float

def benchmark(fn, iterations=100):
    """通用性能基准测试"""
    import time
    t0 = time.time()
    for _ in range(iterations):
        result = fn()
    elapsed = time.time() - t0
    return PerformanceMetrics(
        latency_ms=elapsed * 1000 / iterations,
        throughput_ops=iterations / elapsed,
        memory_mb=0,  # 实际中用 psutil 获取
        accuracy=0.0  # 根据具体任务计算
    )

# 示例：对比两种方案的性能
def approach_a(data):
    """方案A：直接计算"""
    return np.softmax(data @ data.T / np.sqrt(data.shape[-1]), axis=-1)

def approach_b(data):
    """方案B：分块计算（省内存）"""
    chunk_size = 64
    results = []
    for i in range(0, data.shape[0], chunk_size):
        chunk = data[i:i+chunk_size]
        results.append(np.softmax(chunk @ data.T / np.sqrt(data.shape[-1]), axis=-1))
    return np.vstack(results)

data = np.random.randn(512, 768)
metrics_a = benchmark(lambda: approach_a(data))
metrics_b = benchmark(lambda: approach_b(data))
print(f"方案A: {metrics_a.latency_ms:.1f}ms, {metrics_a.throughput_ops:.0f} ops/s")
print(f"方案B: {metrics_b.latency_ms:.1f}ms, {metrics_b.throughput_ops:.0f} ops/s")

    常见问题FAQ

    Q: 如何选择合适的方案？ 先用方案A跑通验证正确性，再根据实际瓶颈（内存/延迟/精度）切换到方案B或C。永远不要过早优化。

    Q: 生产环境最大的坑是什么？ 过度工程化——在需求不明确时投入大量精力做复杂架构。建议MVP先行，收集反馈后迭代。

    Q: 如何评估效果？ 定义明确的成功指标（准确率/延迟/成本），用golden set做回归测试，A/B测试验证改进效果。至少100个样本才有统计显著性。
