模型量化全景
从 FP16 到 INT4 的精度与效率博弈

一个 70B 参数的模型在 FP16 下要 140GB 显存，量化到 INT4 只要 35GB——这意味着原本需要 4 张 A100 的模型，现在一张消费级 4090 就能跑。量化是大模型平民化的关键杠杆。本章从最朴素的线性量化讲起，一路覆盖 GPTQ、AWQ、GGUF 三大主流方案的原理差异，用 bitsandbytes 的真实代码带你完成一次 INT8/INT4 量化，并分析精度损失从何而来。

阅读约 28 分钟 难度 ★★★★★

★ 一、为什么必须量化

大模型的显存占用主要由两部分构成：模型权重和 KV cache。权重部分，每个参数在 FP16（半精度浮点）下占 2 字节，一个 7B 模型就要 14GB，一个 70B 模型要 140GB。这远超单张消费级显卡的容量（RTX 4090 是 24GB）。

量化的目标就是把每个参数的存储位宽从 16 位压缩到 8 位甚至 4 位。从 FP16 到 INT8，显存直接砍半；到 INT4，再砍一半，只剩原来的四分之一。这意味着 7B 模型能在 8GB 显存上跑，70B 模型能在一张 40GB 显卡上跑。量化是大模型走向端侧、走向消费硬件的必经之路。

1.1 量化的三重收益

除了省显存，量化还有两个附带收益。第一，计算加速：整数运算（INT8 的 GEMM）比浮点运算快，硬件支持更好（尤其是 Ampere 之后的 GPU 有专门的 INT8 Tensor Core）。第二，能耗降低：访存量减少，内存带宽压力小，能效比提升。这三者叠加，使得量化在端侧部署、高并发服务里几乎是必选项。

但天下没有免费午餐，量化的代价是精度损失。把连续的浮点数映射到有限的整数格点，必然会丢失信息。如何在「显存省一半」和「精度不掉太多」之间找到平衡，就是各种量化算法要解决的核心问题。

★ 二、线性量化：最朴素的方案

最简单的量化是线性量化（也叫对称量化或 min-max 量化）。它的做法是：找到权重的最大绝对值 max_abs，把区间 [-max_abs, max_abs] 线性映射到整数范围 [-127, 127]（INT8）或 [-7, 7]（INT4）。具体公式是：量化值 q = round(w / scale)，其中 scale = max_abs / 127。反量化时 w ≈ q * scale。

2.1 从零实现线性量化

import torch
import numpy as np

def linear_quantize(w, n_bits=8):
    """
    对称线性量化。返回 (量化后的整数权重, scale)。
    """
    qmax = 2 ** (n_bits - 1) - 1   # INT8: 127, INT4: 7
    max_abs = w.abs().max()
    scale = max_abs / qmax
    q = torch.round(w / scale).clamp(-qmax, qmax).to(torch.int8 if n_bits==8 else torch.int8)
    return q, scale

def linear_dequantize(q, scale):
    return q.float() * scale

# 测试：构造一个权重矩阵，量化再反量化，看误差
torch.manual_seed(0)
w = torch.randn(256, 256) * 0.1     # 模拟一个权重矩阵

q8, scale8 = linear_quantize(w, 8)
w8 = linear_dequantize(q8, scale8)
err8 = (w - w8).abs().mean()

q4, scale4 = linear_quantize(w, 4)
w4 = linear_dequantize(q4, scale4)
err4 = (w - w4).abs().mean()

print(f"INT8 平均量化误差: {err8:.6f}")
print(f"INT4 平均量化误差: {err4:.6f}")
print(f"INT4 误差是 INT8 的 {err4/err8:.1f} 倍")

你会看到 INT4 的量化误差远大于 INT8。这就是为什么 INT8 量化基本无痛，而 INT4 量化需要专门的算法来控制误差——朴素线性量化到 INT4 通常会让模型效果明显下降。

2.2 朴素量化的两个痛点

朴素线性量化有两个致命弱点。第一，离群值（outlier）会撑大 max_abs，导致大部分正常权重的量化粒度变粗、误差变大。比如一个权重矩阵里 99% 的值在 [-0.1, 0.1]，但有一个值是 2.0，那么 scale 被这个 2.0 拉大，其他值的精度就被牺牲了。

第二，逐层误差累积。每一层的量化误差会在网络里逐层传播放大，最后几层的输出可能完全失真。所以工业级量化要么逐 channel 量化（让每个通道有独立 scale），要么用校准数据集调整 scale，要么干脆用算法搜索最优量化方式——这就催生了 GPTQ、AWQ 这些方法。

★ 三、GPTQ：基于二阶信息的后训练量化

GPTQ（Generalized Post-Training Quantization）是 2022 年提出的一种 INT4 量化方法，能在几乎不掉点的前提下把模型压到 4bit。它的核心思想是：用一小批校准数据，逐层地、按列地量化权重，并用二阶信息（Hessian 矩阵）来补偿量化误差。

3.1 GPTQ 的直觉

把一层权重的每一列想象成一根柱子，GPTQ 不是一次性量化所有柱子，而是一列一列来：量化第 j 列后，立刻计算这列的量化误差对输出的影响（用 Hessian 衡量），然后微调还没量化的后续列，去抵消这个误差。这就好比砌墙时每砌一块砖都拿水平仪校一下，发现歪了就调整后面的砖，保证整面墙不斜。

它的数学基础是 OBQ（Optimal Brain Quantization），通过最小化量化前后输出的均方误差来求解每列的最优量化值。GPTQ 的贡献是把 OBQ 做得足够快、足够稳，能扩展到千亿参数模型。

3.2 GPTQ 的使用特点

GPTQ 量化需要一份校准数据集（通常用几百到几千条通用文本），量化过程较慢（7B 模型要几分钟，70B 要小时级），但量化后的模型可以离线保存，推理时直接加载。它的优势是精度保持好，尤其在 4bit 下表现稳定。劣势是量化过程依赖校准数据和较多计算资源，不适合快速试验。

GPTQ 模型在社区里非常流行，Hugging Face 上能找到大量 GPTQ 量化版本的开源模型，配合 vLLM、AutoGPTQ 等框架使用。

★ 四、AWQ：激活感知的权重量化

AWQ（Activation-aware Weight Quantization）是 MIT-IBM 团队 2023 年提出的方法。它的核心洞察是：不是所有权重都同等重要，重要性取决于它对应的激活值大小。那些常被大激活值「激活」的权重，量化误差会被放大传播，需要重点保护。

4.1 AWQ 的核心思想

AWQ 用校准数据跑一遍模型，统计每个权重通道对应的激活幅度。然后给那些「激活大」的通道乘上一个保护系数 s（> 1），让它们在量化时被分配到更精细的整数格点；同时给激活小的通道乘上 1/s 来抵消。这样既保护了关键通道，又不改变整体数值范围。

和 GPTQ 相比，AWQ 的量化过程更快（不需要复杂的二阶计算，只需要搜索最佳缩放系数），且推理时不需要反量化的额外计算——它直接把权重存成量化形式，推理时即时反量化。这让 AWQ 在速度上往往优于 GPTQ。

4.2 AWQ vs GPTQ 的对比

两者的精度都很接近 FP16，但在不同场景下各有优势。GPTQ 在 perplexity 指标上通常略好一点（尤其 3bit 极限量化时），AWQ 在推理速度和显存友好性上更好。工程选择上：如果追求极致精度选 GPTQ，如果追求部署易用性和速度选 AWQ。两者都有成熟的框架支持（AutoGPTQ、llm-awq）。

★ 五、GGUF：端侧部署的事实标准

GGUF（前身是 GGML）是 llama.cpp 生态使用的模型格式和量化方案。它的设计目标是让大模型能在 CPU、Mac、手机等没有高端 GPU 的设备上运行。GGUF 的一大特色是提供了极其丰富的量化粒度选择。

5.1 GGUF 的量化档位

GGUF 用 Q4_K_M、Q5_K_M、Q8_0 这样的命名表示不同的量化方案。以最常用的 Q4_K_M 为例，它用 4bit 量化大部分权重，但对重要的层用更高的位宽保护，平均下来每参数约 4.8 bit。这种「混合位宽」策略在显存和精度之间取得了很好平衡。

常见的 GGUF 档位：Q4_0（最快但精度略损）、Q4_K_M（速度精度平衡，最常用）、Q5_K_M（精度更高，显存稍大）、Q8_0（接近 FP16 精度）。在 llama.cpp 里，你可以用 convert 脚本把任意 HF 模型转成这些档位。

5.2 GGUF 的核心优势

GGUF 最大的优势是CPU 友好。它的量化格式针对 x86 AVX2/AVX512、ARM NEON 等 CPU 指令集做了极致优化，在纯 CPU 上也能跑到可用的速度（7B 模型在普通笔记本上约 5 到 10 token/s）。这让它成为端侧部署、私有化部署的首选。配合 llama.cpp 的 server 模式，可以在任何机器上轻松搭一个本地大模型服务。

★ 六、实战：用 bitsandbytes 做 INT8/INT4 量化

理论讲够了，现在上手实操。bitsandbytes 是 Hugging Face 生态里最易用的量化库，一行代码就能把模型量化成 INT8 或 INT4（NF4）。下面演示完整的量化加载和推理流程。

6.1 INT8 量化

from transformers import AutoModelForCausalLM, AutoTokenizer, BitsAndBytesConfig
import torch

model_id = "bigscience/bloom-560m"   # 用小模型演示，便于 CPU 跑通

# INT8 量化配置
bnb_config_8bit = BitsAndBytesConfig(
    load_in_8bit=True,
    # 可选：llm_int8_threshold 控制哪些层用更高精度，默认 6.0
)

tokenizer = AutoTokenizer.from_pretrained(model_id)
model_8bit = AutoModelForCausalLM.from_pretrained(
    model_id, quantization_config=bnb_config_8bit, device_map="auto"
)

# 对比显存占用
model_fp = AutoModelForCausalLM.from_pretrained(model_id, torch_dtype=torch.float16)
mem_fp = sum(p.numel() * p.element_size() for p in model_fp.parameters()) / 1e9
mem_8bit = sum(p.numel() * p.element_size() for p in model_8bit.parameters()) / 1e9
print(f"FP16 显存: {mem_fp:.2f} GB")
print(f"INT8 显存: {mem_8bit:.2f} GB")
print(f"节省: {(1 - mem_8bit/mem_fp)*100:.0f}%")

# 用量化模型推理
inputs = tokenizer("The capital of France is", return_tensors="pt").to(model_8bit.device)
out = model_8bit.generate(**inputs, max_new_tokens=10)
print(tokenizer.decode(out[0], skip_special_tokens=True))

你会看到 INT8 模型的显存大约是 FP16 的一半，推理输出质量几乎无差异。bitsandbytes 的 INT8 量化是混合精度——对异常值用 FP16、对常规值用 INT8，这就是它能保持精度的秘密。

6.2 INT4（NF4）量化 + QLoRA 微调

# INT4 量化配置（NF4 是 QLoRA 论文提出的 4bit 格式，专为正态分布权重设计）
bnb_config_4bit = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",          # NF4 格式
    bnb_4bit_compute_dtype=torch.float16,  # 计算时反量化到 FP16
    bnb_4bit_use_double_quant=True,      # 对 scale 本身也量化，再省一点显存
)

model_4bit = AutoModelForCausalLM.from_pretrained(
    model_id, quantization_config=bnb_config_4bit, device_map="auto"
)

mem_4bit = sum(p.numel() * p.element_size() for p in model_4bit.parameters()) / 1e9
print(f"INT4 显存: {mem_4bit:.2f} GB")
print(f"相比 FP16 节省: {(1 - mem_4bit/mem_fp)*100:.0f}%")

# 关键：4bit 模型可以直接配合 QLoRA 微调
# 冻结量化权重，只训练低秩 LoRA adapter
from peft import LoraConfig, get_peft_model, TaskType

lora_config = LoraConfig(
    r=8, lora_alpha=16, lora_dropout=0.05,
    target_modules=["query_key_value"],   # Bloom 的注意力模块名
    task_type=TaskType.CAUSAL_LM,
)
model_4bit = get_peft_model(model_4bit, lora_config)
model_4bit.print_trainable_parameters()
# 输出类似：trainable params: 1,572,864 || all params: 562,227,840 || trainable%: 0.28%

这段代码演示了 QLoRA 的核心：把大模型量化到 4bit 节省显存，再在上面挂一个低秩 LoRA adapter 做微调。可训练参数只占 0.28%，但效果接近全参数微调。这是在单卡上微调大模型的标准方案。

★ 七、精度损失从何而来

量化后模型效果会下降一点，理解这个损失从哪来，能帮你判断「这个模型能不能量化」。

  • 权重表示误差：最直接的误差。4bit 只有 16 个格点，权重分布到这些格点上必然有舍入。正态分布的权重用 NF4 能最小化这个误差（NF4 的格点按正态分布密度排布）。
  • 异常值保护不足：某些权重值偏大，是模型效果的关键。朴素量化会牺牲它们。GPTQ/AWQ 都通过特殊机制保护这些值。
  • 激活量化误差：除了权重，激活值也可以量化（W8A8 方案）。但激活的分布比权重更不规则（长尾更严重），量化更难，精度损失更大，所以现在主流是只量化权重（W4A16）。
  • 层间误差累积：每一层的误差向后传播。深网络、小模型对量化更敏感。这也是为什么 70B 模型 INT4 几乎无痛，而 1B 模型 INT4 可能明显掉点。

一个经验法则：模型越大，量化越安全。大模型的参数冗余度高，少量精度损失可以靠其他参数补偿。小模型参数紧绷，量化损失更敏感。所以通常 7B 以上的模型 INT4 才比较稳。

★ 八、如何选择量化方案

面对众多方案，工程上的选择可以参考这个决策树。

  • 只想快速体验、不在意极致精度：用 bitsandbytes 的 INT8（load_in_8bit），最简单，质量几乎无损。
  • 显存紧张、要跑大模型：用 bitsandbytes 的 NF4 或 AWQ 的 INT4，显存省四分之三。
    • 要部署生产服务、追求速度和精度平衡：用 GPTQ 或 AWQ 的 4bit 模型，配 vLLM 推理。
  • 要在 CPU、Mac、手机上跑：用 GGUF 格式（Q4_K_M），配 llama.cpp。
  • 要微调但显存不够：用 NF4 + QLoRA，单卡微调大模型。

最后提醒：量化不是免费的，部署前务必在评测集上对比量化前后的指标（perplexity、下游任务准确率），确认损失在可接受范围内再上线。不同模型对量化的敏感度差异很大，不能盲目套用。

★ 本章小结

这一章我们把模型量化的全景梳理了一遍。要点回顾：

  • 量化的意义：把权重从 FP16 压到 INT8/INT4，显存省一半到四分之三，附带计算加速和能耗降低，是大模型落地的关键。
  • 朴素线性量化：用 max_abs 把浮点映射到整数格点，简单但有离群值和误差累积两个痛点。
  • GPTQ：基于二阶信息（Hessian）逐列量化并补偿误差，精度保持好，适合生产部署。
  • AWQ：激活感知，保护大激活对应的权重通道，速度快、易用。
  • GGUF：端侧 CPU 部署的事实标准，混合位宽，支持 Q4_K_M 等多档位。
  • bitsandbytes：最易用的库，INT8 几乎无痛，NF4+QLoRA 让单卡微调大模型成为可能。
  • 精度损失：来自权重舍入、异常值保护不足、激活量化、层间累积；模型越大越抗量化。

掌握了量化，你就能让大模型跑在各种硬件上。下一章我们进入一个全新的架构话题——MoE 混合专家。它从另一个维度优化大模型：不是压参数，而是让每个 token 只激活一小部分参数，实现「参数多但计算少」的稀疏激活。
