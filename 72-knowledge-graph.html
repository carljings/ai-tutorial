知识图谱与 GraphRAG
从 Neo4j 实操到图增强检索的进阶之路

RAG 的痛点是「只懂相似、不懂关系」——向量检索能找到语义相近的文本，但抓不住实体间的逻辑关系。「张三是李四的导师」这种关系，纯向量检索很难召回。知识图谱（Knowledge Graph）补上了这一块。这一章从知识图谱的构建流程、Neo4j 图数据库实操，到微软 GraphRAG 的前沿方案，让你理解什么时候该用向量检索，什么时候必须上图谱。

阅读约 28 分钟 难度 ★★★★

★ 一、纯向量检索的局限

向量检索（RAG 的基础）的核心逻辑是「语义相似」——把文本编码成向量，用余弦相似度找最相关的片段。这对「找相关内容」很有效，但对「找关系」很无力。

举一个真实场景：用户问「和特斯拉在同一个供应链里的公司有哪些」。纯向量检索会找和「特斯拉供应链」语义相近的文本片段，但它不理解「供应链」是一种关系，也不知道哪些公司通过哪些零部件和特斯拉关联。这种多跳关系推理，正是知识图谱的强项。

# 向量检索 vs 图谱检索的差异
query = "张三的学生里，谁现在在 Google 工作？"

# 向量检索（RAG）的问题：
# 1. 检索到包含"张三""学生""Google"的文本片段
# 2. 但这些信息可能分散在不同文档里
# 3. 无法做"张三→学生→在Google工作"的多跳推理
retrieved = vector_db.search(query, top_k=5)
# 结果：可能是一些提到张三和Google的零散段落，答非所问

# 知识图谱的优势：
# 图里存的是结构化的三元组：(张三)-[指导]->(李四)
#                            (李四)-[就职于]->(Google)
# 一次图遍历就能精确回答这个问题
graph_query = """
MATCH (teacher {name: '张三'})-[:ADVISES]->(student)-[:WORKS_AT]->(company {name: 'Google'})
RETURN student.name
"""
# 结果：精确返回符合条件的人名

★ 二、知识图谱的基本概念

知识图谱用「实体-关系-实体」的三元组来表示世界。实体是节点（人、公司、地点），关系是边（就职于、指导、位于）。这种结构天然适合表达复杂的关系网络。

  • 实体（Entity）：图中的节点，如「张三」「Google」「北京」。
  • 关系（Relation）：节点间的有向边，如「就职于」「位于」「指导」。
  • 属性（Property）：节点和边的附加信息，如「张三.年龄=35」「就职于.起始时间=2020」。
  • 本体（Ontology）：定义实体类型和关系类型的 schema，如「人可以就职于公司，但不能就职于城市」。

★ 三、知识图谱的构建流程

构建知识图谱是最大的工程挑战。从非结构化文本里抽取实体和关系，传统方法用 NLP（命名实体识别 + 关系抽取），现在 LLM 让这个过程大大简化。

# 用 LLM 从文本构建知识图谱
from openai import OpenAI
import json

client = OpenAI()

def extract_triples(text: str) -> list:
    """用 LLM 从文本抽取三元组（实体-关系-实体）。"""
    prompt = f"""从以下文本中抽取知识三元组。

文本：{text}

抽取规则：
1. 识别文本中的实体（人名、公司名、地点、概念等）
2. 识别实体间的关系
3. 输出 JSON 数组，每个元素是一个三元组

输出格式：
[
  {{"subject": "实体1", "relation": "关系", "object": "实体2"}},
  ...
]

示例：
文本："张三于2020年加入Google，担任高级工程师，他的导师是李四。"
输出：
[
  {{"subject": "张三", "relation": "就职于", "object": "Google"}},
  {{"subject": "张三", "relation": "职位", "object": "高级工程师"}},
  {{"subject": "张三", "relation": "导师是", "object": "李四"}}
]"""

    resp = client.chat.completions.create(
        model="gpt-4o",
        response_format={"type": "json_object"},
        messages=[{"role": "user", "content": prompt}],
        temperature=0,
    )
    return json.loads(resp.choices[0].message.content)["triples"]

# 从一段新闻构建图谱
news = "马斯克于2004年投资特斯拉，担任CEO。特斯拉总部位于得州奥斯汀，"
      "主要生产电动汽车。马斯克同时是SpaceX的创始人。"
triples = extract_triples(news)
# 输出：
# [
#   {"subject": "马斯克", "relation": "投资", "object": "特斯拉"},
#   {"subject": "马斯克", "relation": "担任", "object": "CEO"},
#   {"subject": "特斯拉", "relation": "总部位于", "object": "奥斯汀"},
#   {"subject": "特斯拉", "relation": "生产", "object": "电动汽车"},
#   {"subject": "马斯克", "relation": "创立", "object": "SpaceX"},
# ]

3.1 完整构建流水线

# 知识图谱构建的完整流水线
class KnowledgeGraphBuilder:
    def __init__(self, graph_db):
        self.db = graph_db  # Neo4j 连接
        self.entity_resolver = EntityResolver()

    def build_from_documents(self, documents: list):
        """从一批文档构建知识图谱。"""
        all_triples = []
        for doc in documents:
            # 1. 实体关系抽取
            triples = extract_triples(doc)
            # 2. 实体消歧（合并指向同一实体的不同表述）
            triples = self.entity_resolver.resolve(triples)
            # 3. 写入图数据库
            for t in triples:
                self.upsert_triple(t)
            all_triples.extend(triples)
        return all_triples

    def upsert_triple(self, triple):
        """写入三元组（存在则更新，不存在则创建）。"""
        cypher = """
        MERGE (s:{subj_type} {name: $subject})
        MERGE (o:{obj_type} {name: $object})
        MERGE (s)-[r:{relation}]->(o)
        """
        self.db.run(cypher, **triple)

    def validate_with_ontology(self, triples):
        """用本体校验三元组的合法性。"""
        # 确保关系类型符合 schema
        # 例如："人"不能"位于"（只有"公司"才能"位于"）
        valid = [t for t in triples if self.ontology.is_valid(t)]
        return valid

★ 四、Neo4j 实操：图数据库入门

Neo4j 是最主流的图数据库，用 Cypher 查询语言。它的数据模型直观、查询表达力强，是知识图谱落地的首选。

# Neo4j 实操：从连接到查询
from neo4j import GraphDatabase

class Neo4jKG:
    def __init__(self, uri, user, password):
        self.driver = GraphDatabase.driver(uri, auth=(user, password))

    def create_entity(self, name, entity_type, properties=None):
        """创建实体节点。"""
        query = f"CREATE (n:{entity_type} {{name: $name}})"
        if properties:
            props = ", ".join(f"n.{k} = ${k}" for k in properties)
            query = query[:-1] + ", " + props + "}"
        with self.driver.session() as session:
            session.run(query, name=name, **(properties or {}))

    def create_relation(self, subject, relation, object_):
        """创建关系。"""
        query = f"""
        MATCH (s {{name: $subject}}), (o {{name: $object}})
        MERGE (s)-[:{relation}]->(o)
        """
        with self.driver.session() as session:
            session.run(query, subject=subject, object=object_)

    def find_multi_hop(self, start_entity, relation_chain):
        """多跳查询：知识图谱的核心能力。"""
        # 例如：找"马斯克"投资的公司里，哪些在得州
        query = """
        MATCH path = (p:Person {name: $start})-[:INVESTS_IN]->(c:Company)-[:LOCATED_IN]->(loc:State {name: 'Texas'})
        RETURN c.name, loc.name
        """
        with self.driver.session() as session:
            return list(session.run(query, start=start_entity))

    def shortest_path(self, entity_a, entity_b):
        """找两个实体间的最短路径（关系发现）。"""
        query = """
        MATCH path = shortestPath(
          (a {name: $a})-[*..5]-(b {name: $b})
        )
        RETURN [node IN nodes(path) | node.name] AS path
        """
        with self.driver.session() as session:
            result = session.run(query, a=entity_a, b=entity_b)
            return result.single()["path"]

# 使用示例
kg = Neo4jKG("bolt://localhost:7687", "neo4j", "password")
kg.create_entity("马斯克", "Person", {"age": 53})
kg.create_entity("特斯拉", "Company", {"industry": "汽车"})
kg.create_relation("马斯克", "INVESTS_IN", "特斯拉")

# 多跳查询：谁通过供应链和特斯拉关联？
path = kg.shortest_path("宁德时代", "特斯拉")
# 可能返回：["宁德时代", "供应电池", "Model 3", "属于", "特斯拉"]

★ 五、GraphRAG：微软的前沿方案

2024 年微软提出的 GraphRAG 是知识图谱增强 RAG 的里程碑方案。它解决了传统 RAG 的两个痛点：无法回答全局性问题（如「这本书的主要主题是什么」）和无法做多跳推理。

# GraphRAG 的工作流程（对比传统 RAG）
graphrag_pipeline = {
    "第 1 步：索引（离线）": """
    1. 把文档切分成 chunks
    2. 用 LLM 从每个 chunk 抽取实体和关系（前面讲过）
    3. 构建知识图谱
    4. 用图聚类算法（如 Leiden）把相关实体聚成「社区」
    5. 用 LLM 为每个社区生成摘要（这就是「层级摘要」）
    """,

    "第 2 步：查询（在线）": """
    GraphRAG 支持两种查询模式：

    A. 局部搜索（Local Search）：
       - 适合具体的、指向特定实体的问题
       - "马斯克投资了哪些公司？"
       - 从图里找到相关实体及其邻居，结合原文回答

    B. 全局搜索（Global Search）：
       - 适合抽象的、跨文档的问题
       - "这些文档讨论的主要趋势是什么？"
       - 遍历所有社区摘要，用 map-reduce 方式汇总答案
    """,
}

★ 六、混合检索：向量 + 图谱

GraphRAG 虽强，但成本高。更实用的方案是「向量检索 + 图谱检索」的混合模式：用向量找相关文本，用图谱补充关系信息。两者互补，效果最好。

# 混合检索的实现
class HybridRetriever:
    """向量检索 + 图谱检索的混合方案。"""
    def __init__(self, vector_db, graph_db):
        self.vector_db = vector_db
        self.graph_db = graph_db

    def retrieve(self, query: str, top_k: int = 5):
        # 1. 向量检索：找语义相关的文本片段
        vector_results = self.vector_db.search(query, top_k=top_k)

        # 2. 实体抽取：从 query 里提取关键实体
        entities = extract_entities_from_query(query)

        # 3. 图谱检索：找这些实体的关系信息
        graph_context = []
        for entity in entities:
            # 找实体的邻居和关系
            neighbors = self.graph_db.get_neighbors(entity, depth=2)
            graph_context.append(format_neighbors(entity, neighbors))

        # 4. 融合：把向量结果和图谱信息一起喂给 LLM
        context = f"""
        相关文档（向量检索）：
        {format_vector_results(vector_results)}

        实体关系（图谱检索）：
        {chr(10).join(graph_context)}
        """
        return context

# 混合检索的优势：
# 1. 向量检索提供「语义相关」的内容
# 2. 图谱检索提供「结构化关系」的信息
# 3. 两者互补，能回答更复杂的问题
# 4. 比纯 GraphRAG 成本低很多

★ 七、什么场景该用知识图谱

  • 关系密集型场景：社交网络、供应链、组织架构。这些天然是图结构。
  • 多跳推理：「A 的合作伙伴投资了哪些公司」这种需要多步推理的查询。
  • 全局分析：「这批文档的主要趋势是什么」需要跨文档的抽象。
  • 知识溯源：需要精确知道答案来自哪些结构化关系。

反过来，如果你的场景只是简单的「找相关文档回答问题」，传统 RAG 就够了，不必上图谱——知识图谱的构建和维护成本不低。

深 GraphRAG 实战：从图构建到增强检索

GraphRAG 把知识图谱引入 RAG，能解决传统向量 RAG 在多跳推理、全局总结上的短板。但它工程复杂度高，并非所有场景都该用。下面给出落地路径和与向量 RAG 的选型。

知识图谱的构建与查询

GraphRAG 的第一步是把文档转成图（实体+关系）。下面用 Neo4j 演示构建和查询的核心流程。

from neo4j import GraphDatabase

driver = GraphDatabase.driver("bolt://localhost:7687", auth=("neo4j","password"))

def build_graph_from_text(tx, text):
    """用 LLM 抽取实体与关系，写入图(概念性)。"""
    entities_relations = llm_extract(text)
    # entities_relations 例: {"entities":["GLM","Z.ai"], "relations":[("GLM","开发商","Z.ai")]}
    for e in entities_relations["entities"]:
        tx.run("MERGE (n:Entity {name: $name})", name=e)
    for subj, rel, obj in entities_relations["relations"]:
        tx.run('''MERGE (a:Entity {name:$s}) MERGE (b:Entity {name:$o})
                  MERGE (a)-[:RELATION {type:$r}]->(b)''', s=subj, r=rel, o=obj)

def multi_hop_query(tx, question):
    """多跳查询：向量 RAG 难以回答的'谁开发了GLM，它属于哪家公司'这类问题。"""
    cypher = """MATCH (n:Entity {name:'GLM'})-[:RELATION]->(m)
                RETURN n.name, m.name LIMIT 10"""
    return tx.run(cypher).data()

with driver.session() as s:
    s.execute_write(build_graph_from_text, "GLM由Z.ai开发...")
    print(s.execute_read(multi_hop_query, "GLM的开发商"))

何时该上 GraphRAG

GraphRAG 不是“更高级的 RAG”，而是解决特定问题的 RAG，多数场景向量 RAG 就够了，盲目上图反而增加复杂度。该考虑 GraphRAG 的信号：第一，用户问题常涉及多跳关系（“A 的供应商的竞争对手是谁”），向量检索召回不准；第二，需要全局性总结（“总结整个行业的主要玩家和关系”），向量 RAG 只能找片段、难综观全局；第三，数据本身强结构化（已有图谱、或关系是核心价值，如金融股权、医疗药物相互作用）。即便如此，推荐混合 RAG——向量检索找相关片段 + 图谱补关系，两者互补，而非二选一。落地的坑主要有三个：图谱构建质量（LLM 抽实体关系会出错，要人工抽检校准）、图谱更新（文档变了图要同步更新，比向量库索引维护麻烦）、查询转换（把自然语言问题转成 Cypher 是难点，需好的 NL2Graph 模块）。所以 GraphRAG 要在收益明显大于复杂度成本时才上，先验证“向量 RAG 真的答不好这类问题”，再投入。

深 GraphRAG FAQ 与选型

GraphRAG 不是万能药，这些问题决定该不该用。

Q1：我的 RAG 效果不好，换 GraphRAG 能解决吗？

不一定，先诊断。效果不好原因多：chunking 不合理、embedding 差、召回不准、prompt 没用好上下文。这些用 GraphRAG 也救不了。该考虑 GraphRAG 的信号：用户问题常涉及多跳关系（「A 的供应商的竞争对手」）、需全局总结、数据本身强结构化。即便如此，推荐混合 RAG（向量+图互补）而非二选一。盲目上图增加复杂度（图构建/更新/查询转换都难）却未必更好。先验证「向量 RAG 真的答不好这类问题」，再投入 GraphRAG。

Q2：GraphRAG 的图怎么构建和维护？

构建：用 LLM 从文档抽取实体和关系（结构化输出），写入图数据库（Neo4j）。坑是抽取会出错（幻觉关系），要人工抽检校准。维护更难：文档变了图要同步更新（比向量索引麻烦）、实体消歧（同名不同实体、异名同实体）。查询转换是另一难点：把自然语言转 Cypher 需要 NL2Graph 模块。所以 GraphRAG 落地成本高，要收益明显大于复杂度才上。建议先小规模验证（一个子领域），跑通构建-查询-维护全流程再扩大。

★ 本章小结

这一章我们覆盖了知识图谱从构建到应用的完整链路。核心要点：

  • 纯向量的局限：向量检索擅长找语义相似的内容，但不擅长找关系和多跳推理。知识图谱补上了关系维度。
  • 构建流程：LLM 让实体关系抽取大大简化，流水线是「文档 → 抽取三元组 → 实体消歧 → 写入图库 → 本体校验」。
  • Neo4j 实操：用 Cypher 做多跳查询、最短路径发现，这是图谱的核心价值。
  • GraphRAG：微软的方案，通过社区聚类和层级摘要，支持全局查询和多跳推理，代价是索引成本高。
  • 混合检索：向量 + 图谱互补，比纯 GraphRAG 实用，适合大多数生产场景。
  • 适用场景：关系密集、多跳推理、全局分析、知识溯源才需要图谱。简单 QA 用传统 RAG。
